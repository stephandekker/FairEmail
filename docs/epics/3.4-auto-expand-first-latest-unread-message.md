# Epic: Auto-Expand First / Latest Unread Message

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.4** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Auto-expand first / latest unread message in a conversation, with collapse/expand controls per message."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

When a user opens a conversation (thread) containing multiple messages, they need to see the most relevant message body immediately — without manually clicking through each message. At the same time, they need a way to reveal or hide individual messages to manage visual clutter and focus their attention.

This feature solves two problems simultaneously:

1. **Smart initial focus.** The application uses heuristics to determine which single message in the conversation is most likely what the user wants to read, and expands it automatically. This eliminates a click from the most common workflow (opening a conversation to read the latest reply).

2. **Per-message expand/collapse controls.** Every message in the conversation has a toggle that lets the user expand it (revealing its full body, attachments, and actions) or collapse it (showing only the header summary). This keeps long threads manageable while preserving access to any message in the chain.

Together, these behaviors make conversation-mode reading efficient for both short threads (one message auto-expands, nothing else to do) and long threads (the right message is pre-expanded, and the user can selectively reveal others as needed).

## 2. Goals

- **G1.** Automatically expand the single most relevant message when a user navigates to a conversation, requiring zero interaction in the common case.
- **G2.** Provide a clear, per-message expand/collapse toggle so the user can manually reveal or hide any message in the conversation.
- **G3.** Make the auto-expand heuristic predictable and user-configurable, so that the application's choice matches the user's mental model.
- **G4.** Support both a single-expand mode (only one message expanded at a time) and a multi-expand mode (several messages open simultaneously) to accommodate different reading styles.
- **G5.** Integrate cleanly with mark-as-read behavior so that expanding a message can optionally trigger read-state changes.

## 3. Non-Goals

- **NG1.** This epic does **not** define how conversation threading itself works (grouping, sort order within a thread, cross-account merging). Threading is governed by feature 3.2.
- **NG2.** This epic does **not** define the message body rendering pipeline (HTML sanitization, image blocking, font controls). It only specifies when and whether a message body becomes visible.
- **NG3.** This epic does **not** prescribe the visual design of the collapsed or expanded message states — only the behavioral contract (what is shown, when, and under what conditions).
- **NG4.** This epic does **not** cover expand/collapse of quoted text within a single message body. That is a separate concern within the compose/reply-quoting feature.
- **NG5.** This epic does **not** address folder-list or navigation-pane collapsing. "Collapse" here refers exclusively to individual messages within a conversation view.

## 4. Glossary

| Term | Definition |
|---|---|
| **Conversation** | A set of related messages grouped into a single thread by the application's threading logic. Displayed as a vertical stack of messages. |
| **Collapsed message** | A message displayed in a compact header-only form: sender, date, subject summary, flags — but with the full message body hidden. |
| **Expanded message** | A message displayed with its full body visible: formatted content, inline images, attachments, and per-message action buttons. |
| **Auto-expand** | The application's decision to expand a specific message automatically when the user first navigates to a conversation, without requiring a manual click. |
| **Expand toggle** | A per-message UI control (button or clickable region) that switches a message between collapsed and expanded states. |
| **Single-expand mode** | A behavior mode where expanding one message automatically collapses any previously expanded message, so at most one message is expanded at a time. |
| **Multi-expand mode** | A behavior mode where multiple messages may remain expanded simultaneously. |
| **Unread message** | A message that has not been marked as read (seen). |
| **Starred message** | A message the user has flagged/starred for attention. |
| **Duplicate message** | A copy of a message that exists in another folder (e.g. an archived copy). Duplicates are excluded from auto-expand heuristics. |

## 5. Personas

- **P1. The quick-reply reader.** Opens a conversation to read and respond to the latest reply. Wants the newest unread message pre-expanded so they can start reading immediately.
- **P2. The thread reviewer.** Opens long threads to review context. Wants to expand several messages selectively while keeping others collapsed for orientation.
- **P3. The inbox-zero triager.** Scans conversations quickly, often just reading the auto-expanded message and archiving. Needs the heuristic to pick the right message on the first try.
- **P4. The mobile/bandwidth-conscious user.** On a metered or slow connection, does not want large messages auto-downloaded just because they were auto-expanded.

## 6. User Stories

### 6.1 Auto-expand on navigation

> **US-1.** As any user, when I open a conversation that contains exactly one unread message, I want that message to be automatically expanded so I can read it without an extra click.

> **US-2.** As any user, when I open a conversation where all messages are read, I want the most recent message (or the first message, depending on my preference) to be automatically expanded so the conversation does not appear blank.

> **US-3.** As any user, when I open a conversation that contains exactly one starred message and no unread messages, I want the starred message to be automatically expanded, since it is likely the one I flagged for attention.

> **US-4.** As any user, when I open a conversation that contains multiple unread messages, I want **no** message to be auto-expanded, so that I do not accidentally skip over unread messages by only reading the one that was expanded.

> **US-5.** As any user, when I open a conversation containing only a single message, I want that message to be automatically expanded regardless of its read state, since there is no ambiguity about what to show.

> **US-6.** As any user, I want duplicate messages (archived copies), trashed messages, and draft messages to be excluded from the auto-expand heuristic, so that these secondary copies do not interfere with the selection of the primary message to expand.

### 6.2 Per-message expand/collapse controls

> **US-7.** As any user viewing a conversation, I want each message to have a visible expand/collapse toggle (e.g. a chevron icon or clickable header region) so that I can reveal or hide any message's body on demand.

> **US-8.** As any user, I want the expand/collapse toggle to provide clear visual feedback of the current state (expanded vs. collapsed) so I always know which messages are open.

> **US-9.** As a thread reviewer, I want to be able to expand a message by clicking anywhere on its collapsed header (not just a small icon), so that the target area is large and easy to hit.

### 6.3 Single-expand vs. multi-expand mode

> **US-10.** As a quick-reply reader who finds long threads distracting, I want a mode where expanding one message automatically collapses any other expanded message, so I only ever see one message body at a time (single-expand mode).

> **US-11.** As a thread reviewer who needs to cross-reference multiple messages, I want a mode where I can have several messages expanded simultaneously without them collapsing each other (multi-expand mode).

> **US-12.** As any user, I want a setting to choose between single-expand and multi-expand modes, and I want my choice to persist across sessions.

### 6.4 Expand-all option

> **US-13.** As a thread reviewer, I want an option to automatically expand all read messages when I open a conversation, so that the entire thread is visible without manual interaction.

> **US-14.** As any user, when all messages are expanded simultaneously, I want to understand that per-message action buttons (such as reply) may be contextually ambiguous, since it may be unclear which message the action applies to.

### 6.5 Collapse behaviors

> **US-15.** As any user, I want a setting that automatically collapses a message when I mark it as read or unread, so that state changes move me along the thread without extra clicks.

> **US-16.** As any user, when I press the back/escape key while multiple messages are expanded, I want all expanded messages to collapse (returning me to the thread overview) rather than immediately leaving the conversation, so I get a chance to re-orient before navigating away.

### 6.6 Auto-expand and mark-as-read interaction

> **US-17.** As any user, I want a per-account setting that controls whether expanding a message automatically marks it as read, so I can choose between "expand = read" (for quick triage) and "expand ≠ read" (for careful review).

> **US-18.** As any user, I want the auto-expand logic to respect the same mark-as-read setting, so that auto-expanding a message on navigation does not mark it read if I have disabled that behavior.

### 6.7 Bandwidth-aware auto-expand

> **US-19.** As a bandwidth-conscious user on a metered connection, I do not want messages to auto-expand if their body has not been downloaded and exceeds a configurable size threshold, so that I am not surprised by large downloads.

> **US-20.** As a bandwidth-conscious user, when a message would auto-expand but is too large to download on a metered connection, I want to see a clear indication that the message exists but was not expanded due to size, so I understand why the conversation appears empty or why a different message was chosen.

### 6.8 Configuration

> **US-21.** As any user, I want a master toggle to enable or disable auto-expand entirely, so that I can opt out of the feature if I prefer to always choose manually.

> **US-22.** As any user, I want a setting to control whether the "first" (earliest) message in a read conversation is auto-expanded, so that I can match my preference for reading order (chronological vs. reverse-chronological).

## 7. Functional Requirements

### 7.1 Auto-expand heuristic

- **FR-1.** The application shall apply auto-expand logic when the user navigates to a conversation view. The decision shall be made once on entry, not continuously re-evaluated.
- **FR-2.** The auto-expand heuristic shall select at most one message to expand, using the following priority order:
  1. If the conversation contains exactly one message: expand it.
  2. If the conversation contains exactly one unread message (excluding duplicates, trashed, and drafts): expand it.
  3. If the conversation contains no unread messages but exactly one starred message: expand the starred message.
  4. If the conversation contains no unread messages and the "expand first when read" setting is enabled: expand the first (or last, respecting sort order) message.
  5. If none of the above conditions are met (e.g. multiple unread messages): expand nothing.
- **FR-3.** Duplicate messages (copies in archive or other folders), trashed messages, and draft messages shall be excluded from the counting logic in FR-2.
- **FR-4.** When the heuristic selects a message to expand, the expansion shall occur automatically without user interaction.
- **FR-5.** The user shall be able to disable auto-expand entirely via a global setting. When disabled, conversations always open with all messages collapsed.

### 7.2 Per-message expand/collapse

- **FR-6.** Every message displayed in a conversation view shall have an expand/collapse toggle control.
- **FR-7.** The toggle control shall be accessible both as a dedicated button and by clicking the message's header region.
- **FR-8.** The toggle shall provide immediate visual feedback: the control's icon or appearance shall clearly indicate whether the message is currently expanded or collapsed.
- **FR-9.** Expanding a message shall reveal its full rendered body, inline attachments, attachment list, and per-message action buttons.
- **FR-10.** Collapsing a message shall hide its body, attachments, and action buttons, showing only the header summary (sender, date, subject line, flag indicators).

### 7.3 Single-expand mode

- **FR-11.** When single-expand mode is enabled (the default), expanding a message shall automatically collapse any other currently expanded message in the same conversation.
- **FR-12.** The transition from one expanded message to another shall be seamless — the user sees the old message collapse and the new message expand in one logical action.

### 7.4 Multi-expand mode

- **FR-13.** When multi-expand mode is enabled, expanding a message shall not affect the expanded/collapsed state of any other message.
- **FR-14.** The user may have any number of messages expanded simultaneously in multi-expand mode.

### 7.5 Expand-all option

- **FR-15.** A setting shall exist to automatically expand all read (seen) messages when opening a conversation. This is independent of the auto-expand heuristic for unread messages.
- **FR-16.** When expand-all is enabled, it shall operate in multi-expand mode regardless of the single-expand setting.

### 7.6 Collapse behaviors

- **FR-17.** A setting shall exist that automatically collapses a message when the user changes its read/unread state (marks it read or marks it unread).
- **FR-18.** When multiple messages are expanded and the user presses the back/escape key, all expanded messages shall collapse (returning to a thread-overview state) rather than immediately exiting the conversation. A second back/escape shall navigate away from the conversation.

### 7.7 Bandwidth-aware expansion

- **FR-19.** On a metered connection, the auto-expand heuristic shall not expand a message whose body has not yet been downloaded if the message size exceeds a user-configurable threshold.
- **FR-20.** When a message is skipped due to FR-19, the user shall see a visual indication that the message body is available for manual download/expansion.
- **FR-21.** On an unmetered connection, or when the message body is already cached locally, FR-19 does not apply — the message may be auto-expanded normally.

### 7.8 Mark-as-read on expand

- **FR-22.** A per-account setting shall control whether expanding a message (whether by auto-expand or manual toggle) automatically marks it as read.
- **FR-23.** When this setting is enabled, the message shall be marked as read at the moment of expansion.
- **FR-24.** When this setting is disabled, expanding a message shall have no effect on its read state.
- **FR-25.** This setting shall apply equally to auto-expanded messages and manually expanded messages.

### 7.9 Settings persistence

- **FR-26.** All expand/collapse preference settings (auto-expand on/off, expand-first-when-read, expand-all, single-expand mode, collapse-on-mark, collapse-on-back) shall persist across application restarts.
- **FR-27.** The mark-as-read-on-expand setting shall be configurable independently per account.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Expanding or collapsing a message shall complete its visual transition in under 200ms, so the interaction feels instant.
- **NFR-2. Scroll stability.** When a message is expanded or collapsed, the viewport shall adjust so that the user does not lose their reading position. The expanded message's header should remain visible after expansion.
- **NFR-3. Accessibility.** The expand/collapse toggle shall be keyboard-accessible, shall have a screen-reader label indicating the current state ("expanded" / "collapsed"), and shall respect the application-wide theme.
- **NFR-4. Predictability.** The auto-expand heuristic shall produce the same result given the same conversation state. Users shall be able to predict which message will expand after learning the rules.
- **NFR-5. Performance at scale.** The feature shall remain responsive for conversations with at least 100 messages. Expand-all mode may degrade gracefully (lazy-rendering bodies) for very long threads.
- **NFR-6. Offline operation.** For messages whose bodies are already cached, expand/collapse shall work fully offline with no network dependency.

| AC-2 | Opening a conversation with multiple unread messages does not auto-expand any message; all messages appear collapsed. |

## 10. Open Questions

- **OQ-1. "First" vs. "latest" ambiguity.** The feature name says "first / latest unread" but the actual behavior depends on the conversation's sort order. When messages are sorted newest-first, the "first" visible message is the most recent. When sorted oldest-first, it is the earliest. The auto-expand heuristic respects sort order, but should the setting description explicitly reference sort order or use absolute terms ("oldest" / "newest")?

- **OQ-2. Starred-message priority.** The heuristic expands a single starred message only when there are zero unread messages. Should there be a user option to always prefer starred messages over the "expand first" fallback, even when no unread messages exist and multiple starred messages are present?

- **OQ-3. Multi-expand and reply button.** When multiple messages are expanded simultaneously, the per-message reply button appears on each. The source application notes that this creates ambiguity about "which message am I replying to." Should the conversation-level reply action (which always replies to the latest) be visually distinguished from per-message reply buttons in multi-expand mode?

- **OQ-4. Expand-all performance boundary.** The expand-all option on a conversation with 50+ messages could be resource-intensive. Should there be a configurable cap (e.g. "expand all, up to N messages") or should the application always expand all regardless of count, relying on lazy rendering?

- **OQ-5. Interaction with search navigation.** When the user arrives at a conversation via a search result pointing to a specific message, should that specific message be auto-expanded (overriding the normal heuristic), or should the standard heuristic apply? The source application expands the search-targeted message; this should be confirmed as desired behavior.

- **OQ-6. Draft messages in conversation.** Draft messages are excluded from auto-expand counting, but if a user opens a conversation containing only a draft (e.g. a reply-in-progress), should the draft auto-expand into the compose editor, or should it remain collapsed with an indicator?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. No auto-expand when multiple unread.** The deliberate choice to expand nothing when multiple unread messages exist prevents a subtle usability trap: if the application expanded the latest unread, users might read it, reply, and never notice older unread messages further up the thread. Forcing the user to manually expand ensures awareness of all unread messages.

- **N-2. Single-expand as default.** Single-expand mode (only one message open at a time) is the default because it keeps the conversation view focused and reduces cognitive load. Multi-expand is available as an opt-in for power users who need to cross-reference.

- **N-3. Mark-as-read is per-account, not global.** Different accounts serve different purposes. A user might want their work account to auto-mark-on-expand (for fast triage) but their personal account to leave messages unread until explicitly marked (for careful reading). Per-account granularity supports this without global compromise.

- **N-4. Bandwidth gating is a safety net.** The metered-connection size check exists to prevent surprising data usage when the user is on a mobile/limited connection. On unmetered connections, the auto-expand proceeds normally regardless of size, because the cost of a download is negligible.

- **N-5. Collapse-on-back as orientation aid.** The behavior of collapsing all messages on the first back-press (rather than immediately exiting) gives the user a "thread overview" state — a chance to see all headers at once before deciding to leave. This is especially valuable in long threads where the user may have scrolled deep into one message and lost spatial context.

- **N-6. Header click expands.** Making the entire collapsed header a click target (not just a small icon) follows Fitts's law: larger targets are faster to acquire. This is particularly important for conversations with many messages, where rapid expand/collapse switching is common.
