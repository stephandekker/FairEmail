# Epic: IMAP IDLE (Push)

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **7.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"IMAP IDLE (push). With multi-folder IDLE, server capability detection, and automatic fallback to polling."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email users expect their client to reflect new mail within seconds of its arrival on the server — not minutes later when a periodic check runs. The IMAP protocol provides the IDLE extension (RFC 2177) for exactly this purpose: the client holds a long-lived connection open on a folder and the server pushes notifications when the folder's state changes.

However, real-world conditions complicate the simple picture:

1. **Not all servers support IDLE.** Some providers (notably certain Yahoo, AOL, and legacy hosts) do not advertise the IDLE capability.
2. **One IDLE session monitors one folder.** A user with many folders needs either multiple concurrent IDLE sessions or a strategy for deciding which folders get real-time push and which fall back to periodic polling.
3. **Long-lived connections die silently.** Firewalls, NATs, mobile networks, and server-side timeouts can sever an IDLE connection without the client receiving a clean error. The client must detect staleness and recover.
4. **Network and power budgets matter.** Maintaining many open connections consumes system resources and, on laptops running on battery, may prevent power-saving states.

The IMAP IDLE feature exists to give users near-instant mail delivery while gracefully handling every combination of server capability, network reliability, folder count, and power constraint — transparently, with no user intervention in the common case.

## 2. Goals

- **G1.** Deliver new-mail notifications to the user within seconds of the message arriving on the server, for every folder the user has opted into real-time synchronization.
- **G2.** Maintain concurrent IDLE sessions on multiple folders across multiple accounts, so that push coverage is not limited to a single Inbox.
- **G3.** Detect server IDLE capability at connection time and automatically select push (IDLE) or poll for each account/folder accordingly.
- **G4.** Recover from connection failures, silent drops, and transient network loss without user intervention, resuming push as soon as conditions allow.
- **G5.** Automatically tune connection keep-alive intervals to the narrowest reliable window for each account, minimizing unnecessary traffic while preventing silent disconnects.
- **G6.** Provide the user with meaningful feedback when push is degraded or unavailable — without requiring them to understand the underlying protocol.

## 3. Non-Goals

- **NG1.** This epic does **not** define what happens after a new message is detected (notification display, badge updates, message download) — only that the detection occurs promptly.
- **NG2.** This epic does **not** prescribe how connections are pooled or multiplexed at the transport level — only the user-visible and contract-level behavior of the push subsystem.
- **NG3.** Full RFC 5465 NOTIFY support (server-side multi-folder notifications via a single connection) is **not** required, though capability detection for it is in scope (see §7.2).
- **NG4.** Scheduling or frequency of full folder synchronization (beyond the change-detection trigger) is out of scope; this epic covers only the "something changed" signal and its reliability.
- **NG5.** OAuth token lifecycle management is out of scope as a standalone concern — this epic only requires that token expiry does not silently kill an IDLE session without recovery.

## 4. Glossary

| Term | Definition |
|---|---|
| **IDLE** | An IMAP protocol extension (RFC 2177) allowing the server to push folder-change notifications to the client over a held-open connection. |
| **Push** | The delivery mode where the server notifies the client of changes in near-real-time via IDLE. |
| **Poll / Polling** | The delivery mode where the client periodically connects and checks for changes on a schedule (e.g. every 15 minutes). |
| **Keep-alive** | A periodic signal (typically a NOOP command) sent by the client to prevent the connection from being silently dropped by intermediaries. |
| **Keep-alive interval** | The time between successive keep-alive signals. Must be shorter than the most aggressive timeout along the network path. |
| **Capability** | A feature advertised by the IMAP server in its CAPABILITY response (e.g. IDLE, NOTIFY, UTF8=ACCEPT). |
| **NOOP** | An IMAP command that does nothing but elicit a server response, used to verify the connection is alive and to trigger any pending notifications. |
| **Folder listener** | A logical subscription within the client that fires when the server reports a message-count change, flag change, or expunge on a monitored folder. |
| **Auto-optimization** | The process by which the client automatically switches an account from push to poll when server behavior indicates that IDLE is unreliable or counterproductive. |
| **Still-here message** | A server-initiated unsolicited response sent during IDLE to confirm the connection is alive. Some servers send these very frequently, indicating poor IDLE support. |

## 5. Personas

- **P1. The always-current user.** Has 1–3 accounts, expects mail to appear the instant it arrives, and considers even a 60-second delay unacceptable. Uses push on all Inboxes.
- **P2. The multi-folder monitor.** Has a work account with 5–10 active project folders plus a personal account. Wants push on all actively-watched folders, not just the Inbox.
- **P3. The battery-conscious laptop user.** Uses the client on a laptop that is frequently on battery power. Wants near-instant delivery when plugged in, but accepts slightly longer delays to preserve battery life.
- **P4. The legacy-server user.** Uses a provider or self-hosted server that does not support IDLE, or supports it unreliably. Expects the client to work correctly regardless, falling back to polling without manual intervention.
- **P5. The set-and-forget user.** Does not know or care what IDLE means. Expects the client to "just work" — delivering mail quickly when possible and never leaving them in a broken state they must debug.

## 6. User Stories

### 6.1 Automatic push activation

> **US-1.** As a set-and-forget user, when I add an account whose server supports IDLE, I want push to be activated automatically for my Inbox without any manual configuration, so that I get near-instant delivery out of the box.

> **US-2.** As any user, when I enable synchronization on an additional folder, I want the application to activate push on that folder automatically (if the server supports IDLE), so that I do not have to separately enable push per folder.

> **US-3.** As any user, when my server does not support IDLE, I want the application to automatically fall back to periodic polling at a reasonable default interval, so that I still receive mail without errors or manual setup.

### 6.2 Multi-folder IDLE

> **US-4.** As a multi-folder monitor, I want push to be active on every folder I have marked for synchronization (not just the Inbox), so that I am notified of changes in project folders, shared folders, and other non-Inbox locations.

> **US-5.** As any user, I want the ability to designate specific folders as "poll only" even when the server supports IDLE, so that I can reduce resource usage for low-priority folders without disabling synchronization entirely.

> **US-6.** As any user, I want each folder's push-vs-poll mode to be independently configurable, so that I can have push on my Inbox and important project folders but polling on archive or bulk-mail folders.

### 6.3 Connection keep-alive and reliability

> **US-7.** As any user, I want the application to automatically keep IDLE connections alive by sending periodic signals, so that firewalls and NATs do not silently drop my connections.

> **US-8.** As any user, when a keep-alive signal fails or times out, I want the application to reconnect and resume push without my intervention, so that I do not silently miss new mail.

> **US-9.** As any user, I want the application to automatically discover the optimal keep-alive interval for each account (the longest interval that still prevents drops), so that I do not have to guess a value or tune it manually.

> **US-10.** As any user, if the application detects that a keep-alive interval is too long (repeated connection drops), I want it to shorten the interval automatically until a stable value is found, so that push remains reliable without manual tuning.

### 6.4 Failure detection and recovery

> **US-11.** As any user, when network connectivity is lost, I want the application to detect the loss promptly and attempt reconnection as soon as connectivity returns, so that push resumes without manual action.

> **US-12.** As any user, when the server closes an IDLE connection unexpectedly, I want the application to reconnect with a brief backoff delay and resume push, so that transient server issues do not require my attention.

> **US-13.** As any user, when repeated reconnection attempts fail, I want the application to notify me that push is degraded for a specific account, so that I understand why mail may be delayed and can investigate if I choose.

> **US-14.** As any user, when connectivity is restored after a period of disconnection, I want the application to perform a full check on all push-enabled folders before resuming IDLE, so that I do not miss messages that arrived while disconnected.

### 6.5 Auto-optimization and fallback

> **US-15.** As any user, when the server's behavior during IDLE indicates poor support (e.g. sending "still here" signals every few minutes, or repeatedly dropping the connection), I want the application to automatically switch that account to polling mode and inform me, so that I get reliable delivery even from poorly-behaved servers.

> **US-16.** As any user, when the application auto-optimizes an account from push to poll, I want the polling interval to default to a reasonable value (e.g. 15 minutes), so that I continue receiving mail at a predictable cadence.

> **US-17.** As any user, I want the ability to override auto-optimization and force push mode for an account if I believe the server issue is transient, so that I retain control.

### 6.6 Power and network awareness

> **US-18.** As a battery-conscious user, I want the application to respect system power-saving modes (e.g. reducing connection frequency or switching to polling when the system signals that power conservation is active), so that background email checking does not drain my battery unnecessarily.

> **US-19.** As any user, I want the application to differentiate between metered and unmetered network connections, allowing me to configure whether push is active on metered connections, so that I can avoid unexpected data usage.

> **US-20.** As any user, when the network type changes (e.g. switching from wired to wireless, or from unmetered to metered), I want the application to re-evaluate push eligibility and reconnect if necessary, so that my push configuration is always respected.

### 6.7 User-facing configuration

> **US-21.** As any user, I want to configure the keep-alive interval per account, so that I can accommodate unusual server or network environments.

> **US-22.** As any user, I want a global setting to choose between push mode (IDLE) and poll mode (with configurable interval), so that I can make a blanket choice for all accounts if I prefer.

> **US-23.** As any user, I want the application to display the current push/poll status for each account (e.g. "Push active", "Polling every 15 min", "Disconnected — retrying"), so that I can verify at a glance whether real-time delivery is working.

> **US-24.** As any user, I want the ability to enable or disable automatic keep-alive tuning, so that I can lock in a manually chosen interval if I know what works for my server.

### 6.8 Provider-specific behavior

> **US-25.** As a user of a provider known to have unreliable or absent IDLE support (e.g. certain Yahoo or AOL configurations), I want the application to default such accounts to polling without attempting IDLE first, so that initial setup is smooth.

> **US-26.** As a user of a provider whose user-created folders do not reliably support IDLE (e.g. certain implementations where only the Inbox supports it), I want the application to default non-Inbox folders to polling for that provider, while still using IDLE on the Inbox.

## 7. Functional Requirements

### 7.1 Capability detection

- **FR-1.** On each new connection to an IMAP server, the application shall query the server's advertised capabilities and determine whether IDLE is supported.
- **FR-2.** The detected IDLE capability shall be recorded persistently per account, so that the application can make informed decisions before the next connection is fully established.
- **FR-3.** If the server does not advertise IDLE, the application shall not attempt to issue IDLE commands on that connection.
- **FR-4.** The application shall also detect the NOTIFY capability (RFC 5465) for future use, recording it alongside other capabilities, even if multi-folder NOTIFY is not currently exploited.

### 7.2 Push mode (IDLE) operation

- **FR-5.** For each folder that is (a) marked for synchronization, (b) not marked as poll-only, and (c) on a server that supports IDLE, the application shall maintain a dedicated, long-lived IDLE session.
- **FR-6.** Each IDLE session shall monitor the folder for: new message arrival, message removal/expunge, and flag changes (read/unread, flagged, etc.).
- **FR-7.** When the server signals a change during IDLE, the application shall immediately trigger a synchronization action for that folder to fetch or update the affected messages.
- **FR-8.** Multiple IDLE sessions shall be maintained concurrently — one per eligible folder — across all configured accounts.
- **FR-9.** The number of concurrent IDLE sessions shall be limited only by the folders the user has opted into synchronization; there shall be no arbitrary cap below the user's configured scope.

### 7.3 Poll mode operation

- **FR-10.** For each folder that is (a) marked for synchronization and (b) either on a server without IDLE support or explicitly marked as poll-only, the application shall check for changes on a periodic schedule.
- **FR-11.** The default polling interval shall be 15 minutes. The user shall be able to configure this per account.
- **FR-12.** Individual folders may have a polling frequency multiplier (poll factor) that causes them to be checked less frequently than the account's base interval (e.g. every second or third cycle), to reduce traffic for low-priority folders.
- **FR-13.** A poll cycle shall also be triggered opportunistically during each keep-alive check on accounts using push, so that poll-only folders on otherwise-push-capable accounts are checked at least as often as the keep-alive interval.

### 7.4 Keep-alive mechanism

- **FR-14.** For each active IDLE connection, the application shall send a keep-alive signal (NOOP or equivalent) at a configurable interval to prevent the connection from being silently dropped.
- **FR-15.** The default keep-alive interval shall be 15 minutes, consistent with RFC 2177's recommendation that clients not hold an IDLE longer than 29 minutes without a refresh.
- **FR-16.** The minimum keep-alive interval shall be no less than 9 minutes, to avoid excessive network chatter.
- **FR-17.** The application shall optionally verify the connection is alive (by checking for a server response to the keep-alive signal) and treat a missing response as a connection failure.
- **FR-18.** Keep-alive timing shall be driven by a system-level scheduling mechanism (alarm or timer) rather than an in-process sleep, so that the application can be suspended between keep-alive events without missing the next one.

### 7.5 Automatic keep-alive tuning

- **FR-19.** The application shall track keep-alive failures per account. If a configurable threshold of consecutive failures is reached (default: 3), the application shall reduce the keep-alive interval by a step (default: 2 minutes) and retry.
- **FR-20.** Interval reduction shall continue step-wise until the interval reaches the configured minimum (FR-16), at which point no further reduction is possible.
- **FR-21.** Once a stable interval is found (no failures over a sustained period), the application shall mark the interval as "confirmed" and stop tuning unless failures recur.
- **FR-22.** The user shall be able to disable automatic tuning and lock in a manually chosen interval.

### 7.6 Auto-optimization (push-to-poll fallback)

- **FR-23.** If the server sends unsolicited "still here" notifications more frequently than a threshold (default: within 3 minutes of entering IDLE), the application shall treat this as evidence of poor IDLE support.
- **FR-24.** If IDLE repeatedly fails to maintain a stable connection despite keep-alive tuning, or the keep-alive interval has been reduced below a viability threshold (e.g. 12 minutes), the application shall recommend or automatically switch the account to polling.
- **FR-25.** When auto-optimization triggers, the application shall: (a) switch the account to polling at a default interval (15 minutes), (b) log the reason, and (c) surface a notification or status indicator to the user.
- **FR-26.** The user shall be able to override auto-optimization and force push mode for any account.
- **FR-27.** Auto-optimization shall be a per-account decision. Folders on other accounts shall not be affected.

### 7.7 Connection failure and recovery

- **FR-28.** When an IDLE connection is lost (timeout, server close, network error), the application shall attempt to reconnect after a brief delay.
- **FR-29.** Reconnection attempts shall use exponential backoff, starting from a short initial delay and increasing with each consecutive failure, up to a configurable maximum.
- **FR-30.** On successful reconnection, the application shall perform a full folder check before re-entering IDLE, to catch any messages that arrived during the disconnection window.
- **FR-31.** If reconnection fails for an extended period, the application shall surface an account-level error notification to the user indicating that push is not operational.
- **FR-32.** Recovery shall be attempted automatically whenever network connectivity is restored (detected via system network-change events).

### 7.8 Network and power awareness

- **FR-33.** The application shall monitor system power-saving state and, when the system enters a power-saving mode (e.g. laptop suspend, aggressive power saver), shall allow IDLE connections to lapse gracefully rather than fighting to maintain them, resuming push when normal power state returns.
- **FR-34.** The application shall distinguish between metered and unmetered network connections. The user shall be able to configure per account whether push (or polling) is permitted on metered connections.
- **FR-35.** When the active network changes (e.g. interface switch, metered status change), the application shall re-evaluate all active connections and reconnect or disconnect as the new conditions require.
- **FR-36.** The application shall expose a user-facing setting to exempt specific accounts from power-saving restrictions, allowing push to remain active even in power-saving mode for high-priority accounts.

### 7.9 Per-folder push/poll control

- **FR-37.** Each folder shall have an independent, persistent setting controlling whether it uses push (IDLE) or poll when its server supports IDLE. The default shall be push.
- **FR-38.** The user shall be able to switch any folder to poll-only mode via the folder's properties, without affecting other folders on the same account.
- **FR-39.** Provider-specific defaults may set certain folder types (e.g. user-created folders on specific providers) to poll-only by default, while still allowing the user to override to push.

### 7.10 Status and observability

- **FR-40.** The application shall maintain and expose a per-account connection status indicating the current state of push: connected and idle, connecting, polling, disconnected (with reason), or optimized-to-poll.
- **FR-41.** The application shall log significant connection lifecycle events (connect, disconnect, keep-alive failure, interval tuning, auto-optimization trigger) for diagnostic purposes, accessible to the user via a log or debug view.
- **FR-42.** The application shall record per-account metrics: current keep-alive interval, whether the interval is confirmed stable, and the timestamp of the last successful server interaction.

### 7.11 Token and credential refresh

- **FR-43.** When an account uses token-based authentication (e.g. OAuth), the application shall monitor token expiry relative to the keep-alive schedule. If the token will expire before the next scheduled keep-alive, the application shall refresh the token proactively or expedite the keep-alive cycle.
- **FR-44.** A token refresh failure shall be treated as a connection error and shall trigger the same recovery path (FR-28 through FR-32), with the addition of surfacing an authentication error to the user if refresh continues to fail.

## 8. Non-Functional Requirements

- **NFR-1. Latency.** Under normal network conditions, the delay between a message arriving on the server and the application detecting it via IDLE shall be under 5 seconds.
- **NFR-2. Scalability.** The application shall support at least 10 accounts with up to 20 push-enabled folders each (200 concurrent IDLE sessions) without degrading system performance to the point of user-noticeable lag.
- **NFR-3. Resource efficiency.** Each IDLE session shall consume minimal CPU when idle (effectively zero between keep-alive events) and minimal memory (one open connection per folder, no active processing loop).
- **NFR-4. Reliability.** Over a 24-hour period on a stable network, the application shall maintain push connectivity for at least 99% of the time (cumulative downtime from recovery cycles ≤ ~15 minutes).
- **NFR-5. Graceful degradation.** Loss of push capability (server issue, network issue, power saving) shall never result in *no* mail delivery — polling shall always serve as the safety net.
- **NFR-6. Transparency.** The user shall always be able to determine the current delivery mode (push vs. poll) and connection health for each account, without needing protocol-level knowledge.
- **NFR-7. Network fairness.** Keep-alive signals and polling cycles shall be scheduled to avoid thundering-herd effects (e.g. all accounts reconnecting simultaneously after a network restoration).
- **NFR-8. Security.** IDLE connections shall use the same transport security (TLS) as regular IMAP connections. No credentials or tokens shall be logged in user-accessible diagnostics.

## 10. Open Questions

- **OQ-1. NOTIFY capability exploitation.** The application detects RFC 5465 NOTIFY capability but does not currently exploit it for multi-folder monitoring over a single connection. Should a future iteration support NOTIFY as an optimization (reducing connection count), or is per-folder IDLE sufficient indefinitely? This may depend on server adoption rates.
- **OQ-2. Provider-specific poll defaults.** The source application defaults certain providers' user-created folders to poll-only (e.g. Gmail label folders). The canonical list of such providers and the rationale for each should be documented per provider. Is this list user-extensible?
- **OQ-3. Keep-alive interval ceiling.** RFC 2177 suggests clients should not IDLE longer than 29 minutes without a fresh IDLE command. The source application uses 15 minutes as the default. Should the user be allowed to set values above 29 minutes (at their own risk), or should the UI enforce the RFC ceiling?
- **OQ-4. Interaction with system suspend/hibernate.** On a laptop that suspends to RAM, IDLE connections will die silently. The current design reconnects on wake. Should the application also schedule a system wake-up timer (where supported) to perform periodic checks during suspend, or is "check on wake" sufficient?
- **OQ-5. Concurrent connection limits.** Some IMAP servers impose per-user connection limits. With many push-enabled folders, the application may exhaust such limits. Should the application detect "too many connections" errors and automatically fall back some folders to polling, or should this be left to the user to resolve?
- **OQ-6. Outlook IDLE bug.** The source application's documentation references a known IDLE bug in Microsoft Outlook/Exchange IMAP servers. Should the application maintain a server-behavior quirks database that automatically adjusts IDLE behavior for known-buggy implementations, or should this be handled purely via auto-optimization?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. One IDLE session per folder, not one per account.** The source application opens a separate long-lived connection for each push-enabled folder. This is a deliberate trade-off: more connections consume more resources, but each folder gets independent, immediate notification. The alternative — cycling through folders on a single connection — would introduce latency proportional to the number of folders. The per-folder model is the only way to achieve sub-5-second detection across all monitored folders simultaneously.

- **N-2. Keep-alive via scheduled alarm, not in-process timer.** Keep-alive signals are scheduled using system-level alarms rather than in-process sleep loops. This allows the application process to be suspended between events (saving CPU and battery) while still guaranteeing timely wake-up for the next keep-alive. An in-process timer would require the process to remain running continuously.

- **N-3. Auto-tuning direction is always downward.** The keep-alive interval only decreases automatically (never increases). This is conservative by design: once a shorter interval is found to work, lengthening it risks re-introducing drops. The user can manually increase the interval, but the auto-tuner will not do so on its own. The interval is marked "confirmed" once stable, preventing unnecessary further reduction.

- **N-4. Auto-optimization threshold: "still here" within 3 minutes.** If a server sends unsolicited keep-alive messages to the client within 3 minutes of entering IDLE, the source application interprets this as the server being configured to terminate IDLE sessions very aggressively — meaning the overhead of maintaining IDLE (constant re-establishment) exceeds the benefit over simple polling. The 3-minute threshold was chosen empirically.

- **N-5. Poll is the universal safety net.** Every code path that can cause IDLE to fail ultimately lands on polling as the fallback. There is no state in which a synchronized folder receives *neither* push *nor* poll. This guarantee is the foundation of the "set and forget" user experience: the user never needs to know whether they are on push or poll, because mail always arrives.

- **N-6. Per-folder poll factor for traffic shaping.** Rather than offering a separate poll interval per folder (which would be confusing), the source application uses a "poll factor" — a multiplier on the account's base keep-alive interval. A factor of 1 means "check every cycle", 2 means "every other cycle", etc. This keeps the configuration simple while still allowing low-priority folders to generate less traffic.

- **N-7. Network change triggers full reconnection.** When the active network interface changes (e.g. Wi-Fi to Ethernet, or gaining/losing metered status), the application tears down all IDLE sessions and re-establishes them. This is intentionally aggressive: a network change may mean a new IP address, new NAT state, or new firewall rules, any of which could leave existing connections in an indeterminate state. A clean reconnection is more reliable than attempting to probe whether existing connections survived.

- **N-8. Exponential backoff with ceiling.** Reconnection after failure uses exponential backoff (doubling the delay each attempt) with a maximum ceiling. This prevents both thundering-herd reconnection storms (when many accounts lose connectivity simultaneously) and runaway resource consumption from tight retry loops against an unreachable server.
