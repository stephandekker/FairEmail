# Epic: Local Contact Store

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **12.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Local contact store. All sent/received addresses are remembered with first-seen / last-seen / count, used for autocomplete and grouping."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email client that only knows about contacts through an external address book forces the user to maintain that address book proactively. In practice, most people correspond with far more addresses than they ever formally save as contacts. Every time the user sends or receives a message, the email addresses involved carry implicit relationship information: who the user talks to, how often, and how recently.

The Local Contact Store exists to capture that implicit information automatically. Every address the application encounters during normal email use — whether as a sender, a recipient, or a CC — is recorded locally together with lightweight metadata: the first time the address was seen, the most recent time, and how many times it has appeared. This data feeds two primary consumers:

1. **Autocomplete.** When composing a message, the application can suggest recipients ranked by frequency and recency, drawing from addresses the user has actually corresponded with rather than requiring a pre-populated address book.
2. **Grouping and organisation.** Contacts can be assigned to user-defined groups, marked as favorites for priority treatment, or flagged as ignored to suppress them from suggestions — giving the user lightweight contact management without the overhead of a full-featured address book application.

The store is strictly local. No contact data is transmitted to any external service. The feature complements — but does not replace — integration with the desktop's system address book; the two sources coexist and can be used independently or together.

## 2. Goals

- **G1.** Automatically learn email addresses from the user's sent and received mail, without requiring any manual action.
- **G2.** Maintain per-contact metadata (first-seen, last-seen, times-contacted) sufficient to rank suggestions by frequency and recency.
- **G3.** Feed the compose autocomplete with relevant, frequency-ranked suggestions that improve over time as the user corresponds.
- **G4.** Allow the user to organise contacts into named groups for quick retrieval and bulk addressing.
- **G5.** Provide a simple contact management interface: view, search, edit, delete, favorite, and ignore individual contacts.
- **G6.** Respect user privacy: all contact data stays local, collection can be disabled per direction (sent/received), and stale contacts are automatically purged.

## 3. Non-Goals

- **NG1.** This epic does **not** define a full-featured contact/address-book application. The local store is a lightweight learning cache, not a replacement for a dedicated contact manager.
- **NG2.** This epic does **not** define synchronisation of contacts to or from any external service, cloud provider, or protocol (CardDAV, LDAP, etc.).
- **NG3.** This epic does **not** define system address-book integration (reading from or writing to the desktop's contact store). That is covered by feature 12.2.
- **NG4.** This epic does **not** define avatar resolution from external sources (Gravatar, Libravatar, BIMI, favicons). Those are covered by feature 12.8. The local store may *cache* an avatar reference, but the resolution logic is out of scope.
- **NG5.** This epic does **not** define contact group expansion in the compose fields (inserting all members of a group as recipients). That is covered by feature 12.3.
- **NG6.** This epic does **not** define per-contact aliases, notes, or colour assignments. Those are covered by feature 12.4.

## 4. Glossary

| Term | Definition |
|---|---|
| **Local contact** | An email address that the application has automatically recorded from sent or received mail, together with its associated metadata. |
| **Contact type** | A classification that distinguishes how the address was encountered. The primary types are: *sent-to* (the user sent mail to this address) and *received-from* (the user received mail from this address). Additional types exist for junk/blocked and whitelisted addresses. |
| **Times contacted** | A counter recording how many times the address has been seen in messages processed by the application. |
| **First contacted** | The timestamp of the earliest message in which the address was observed. |
| **Last contacted** | The timestamp of the most recent message in which the address was observed. |
| **Contact state** | A per-contact flag indicating one of three states: *default* (normal), *favorite* (prioritised in suggestions and given prominent treatment), or *ignored* (suppressed from autocomplete suggestions). |
| **Contact group** | A user-defined label that can be assigned to one or more contacts, used for organising and filtering the contact list. |
| **Junk contact** | An address explicitly marked by the user (or by a rule) as a blocked/junk sender. Junk contacts are stored separately from regular contacts and are used to inform spam filtering, not autocomplete. |
| **No-junk contact** | An address explicitly whitelisted by the user, overriding any junk classification. Acts as an inverse of the junk flag. |
| **Purge** | The automatic deletion of contacts that have not been seen recently and have been contacted fewer than a configured number of times. |

## 5. Personas

- **P1. The everyday correspondent.** Sends and receives email regularly across a handful of accounts. Expects the compose field to suggest people they have recently emailed without having to maintain an address book.
- **P2. The high-volume user.** Deals with hundreds of distinct addresses per month (support queues, mailing lists, community groups). Needs stale contacts to be cleaned up automatically so that autocomplete remains relevant.
- **P3. The privacy-conscious user.** Wants full control over what the application remembers: may disable learning from received mail entirely, or configure aggressive purge intervals.
- **P4. The organiser.** Wants to group contacts into categories (e.g. "Project X", "Family", "Vendors") for quick lookup and to keep the contact list manageable.
- **P5. The multi-account user.** Has several email accounts and expects contacts learned from one account to remain scoped to that account, so that work and personal suggestions do not bleed into each other.

## 6. User Stories

### 6.1 Automatic contact learning

> **US-1.** As an everyday correspondent, when I send an email, I want all recipient addresses (To, Cc, Bcc) to be automatically recorded as local contacts, so that I can find them quickly next time I compose.

> **US-2.** As an everyday correspondent, when I receive an email, I want the sender's address to be automatically recorded as a local contact, so that the application learns who writes to me.

> **US-3.** As a multi-account user, I want contacts learned from a given account to be associated with that account, so that suggestions reflect the context of the account I am composing from.

> **US-4.** As any user, when I send or receive a message involving an address that already exists in the local store, I want the existing contact's last-contacted time and times-contacted count to be updated rather than a duplicate being created, so that frequency data accumulates over time.

> **US-5.** As any user, I want the application to skip recording addresses that are clearly non-personal (e.g. no-reply addresses, mailer-daemon addresses), so that the contact list is not polluted with unaddressable entries.

> **US-6.** As any user, I want messages in system folders such as Drafts, Trash, and Spam not to generate new contacts or update existing contact metadata, so that discarded or junk mail does not contaminate my contact store.

### 6.2 Autocomplete

> **US-7.** As any user, when I start typing in the To, Cc, or Bcc field of a new message, I want the application to suggest matching contacts from the local store — matching on both name and email address — so that I can select a recipient without typing the full address.

> **US-8.** As any user, I want autocomplete suggestions to be ranked so that favorites appear first, followed by frequently and recently contacted addresses, so that the most relevant suggestions are at the top.

> **US-9.** As any user, I want contacts whose state is "ignored" to be excluded from autocomplete suggestions, so that addresses I have deliberately suppressed do not reappear.

> **US-10.** As a multi-account user, I want the option to restrict autocomplete suggestions to contacts associated with the account I am currently composing from, so that suggestions are contextually relevant.

> **US-11.** As any user, I want autocomplete to deduplicate suggestions when the same email address appears as both a sent-to and received-from contact, showing it only once with the combined frequency, so that the suggestion list is clean.

> **US-12.** As any user, I want to control whether autocomplete draws from sent-to contacts, received-from contacts, or both, so that I can tune suggestion sources to my preference.

### 6.3 Contact management

> **US-13.** As any user, I want to view a list of all my local contacts, showing each contact's name, email address, times contacted, and last-contacted date, so that I can see who the application has learned about.

> **US-14.** As any user, I want to search the contact list by name or email address, so that I can find a specific contact quickly.

> **US-15.** As any user, I want to edit a contact's name, email, type, and group assignment, so that I can correct or organise entries the application has learned.

> **US-16.** As any user, I want to delete individual contacts, so that I can remove addresses I no longer want remembered.

> **US-17.** As any user, I want to bulk-clear all contacts of a given type (e.g. all sent-to contacts, or all blocked contacts) for a specific account, so that I can start fresh if the store becomes cluttered.

### 6.4 Favorite and ignore states

> **US-18.** As any user, I want to mark a contact as a "favorite", so that it appears at the top of autocomplete suggestions and receives any other priority treatment the application offers (e.g. application shortcuts).

> **US-19.** As any user, I want to mark a contact as "ignored", so that it is excluded from autocomplete suggestions without being deleted.

> **US-20.** As any user, I want to cycle a contact's state between default, favorite, and ignored from the contact list, so that I can quickly adjust its treatment.

### 6.5 Contact groups

> **US-21.** As an organiser, I want to assign a group label to any contact, so that I can categorise contacts by project, relationship, or any other criterion.

> **US-22.** As an organiser, I want to view my contacts filtered by group, so that I can see all members of a category at once.

> **US-23.** As an organiser, I want the application to show a list of all groups with the number of contacts in each, so that I can see what groupings I have created.

### 6.6 Junk / blocked sender management

> **US-24.** As any user, when I mark a sender as blocked or junk, I want the sender's address to be recorded as a junk contact, so that the application can use this information for future filtering.

> **US-25.** As any user, I want to view a separate list of blocked/junk contacts, so that I can review and manage who I have blocked.

> **US-26.** As any user, I want to whitelist an address (mark as "no-junk") to override any junk classification, so that I can rescue addresses that were incorrectly blocked.

> **US-27.** As any user, I want to clear all blocked contacts for a specific account in one action, so that I can reset my block list.

### 6.7 Automatic cleanup

> **US-28.** As a high-volume user, I want the application to automatically purge contacts that have not been seen for a configurable period and have been contacted fewer than a configurable number of times, so that old, rarely-used addresses do not clutter my store.

> **US-29.** As any user, I want favorite contacts to be exempt from automatic purge, so that contacts I have explicitly prioritised are never silently removed.

> **US-30.** As any user, I want junk/blocked contacts to be exempt from the frequency-based purge, so that my block list is not eroded by age alone.

### 6.8 Privacy controls

> **US-31.** As a privacy-conscious user, I want to disable contact learning from sent messages independently of learning from received messages, so that I have fine-grained control over what the application remembers.

> **US-32.** As a privacy-conscious user, I want all contact data to remain strictly local and never be transmitted to any external service, so that my correspondence patterns are not exposed.

### 6.9 Import and export

> **US-33.** As any user, I want to export my local contacts to a standard interchange format (vCard), so that I can back them up or import them into another application.

> **US-34.** As any user, I want to import contacts from a vCard file into the local store, preserving group assignments, so that I can restore a backup or seed the store from an external source.

## 7. Functional Requirements

### 7.1 Contact record

- **FR-1.** Each local contact shall store at minimum: the associated account, the contact type (sent-to, received-from, junk, no-junk), the email address, an optional display name, an optional group label, a times-contacted counter, a first-contacted timestamp, a last-contacted timestamp, and a state (default / favorite / ignored).
- **FR-2.** The combination of account, type, and email address shall be unique — the application shall not create duplicate contacts for the same address within the same account and type.
- **FR-3.** A contact may optionally store a reference to the sending identity that was used when the user sent mail to that address, so that autocomplete can infer which identity to use for a given recipient.

### 7.2 Automatic learning

- **FR-4.** When the user sends a message, the application shall create or update a sent-to contact for each recipient address (To, Cc, Bcc) in the context of the sending account.
- **FR-5.** When the application processes a newly received message, it shall create or update a received-from contact for the sender's address (preferring Reply-To if present, otherwise From) in the context of the receiving account.
- **FR-6.** When creating a new contact, the application shall initialise times-contacted to 1, first-contacted to the message timestamp, and last-contacted to the message timestamp.
- **FR-7.** When updating an existing contact, the application shall increment times-contacted by 1, preserve the earliest first-contacted value, and update last-contacted to the message timestamp if it is more recent.
- **FR-8.** When updating an existing contact whose name or group is currently empty, the application shall fill in the name or group from the message headers if available.
- **FR-9.** The application shall not create or update contacts from messages in Drafts, Trash, or Spam/Junk folders.
- **FR-10.** The application shall skip addresses that match well-known non-personal patterns (no-reply, mailer-daemon, postmaster, and similar) when learning contacts.
- **FR-11.** The application shall detect self-sent messages (where the sender matches one of the account's own identities) and handle them appropriately — recording the recipients as sent-to contacts but not recording the sender as a received-from contact.
- **FR-12.** Contact learning from sent messages and contact learning from received messages shall each be independently enableable or disableable via user preference.

### 7.3 Autocomplete

- **FR-13.** The compose autocomplete shall query the local contact store for contacts matching the user's input against both the email address and the display name, using a case-insensitive substring match.
- **FR-14.** Autocomplete results shall be sorted by: (1) favorite state first, (2) contacts with a known avatar or photo before those without, (3) times-contacted descending, (4) last-contacted descending.
- **FR-15.** Contacts with state "ignored" shall be excluded from autocomplete results.
- **FR-16.** The user shall be able to configure whether autocomplete draws from sent-to contacts only, received-from contacts only, or both.
- **FR-17.** The user shall be able to configure whether autocomplete is scoped to the current composing account or spans all accounts.
- **FR-18.** When the same email address exists as both a sent-to and a received-from contact, autocomplete shall present it as a single entry, merging the frequency data.

### 7.4 Contact management interface

- **FR-19.** The application shall provide a contact list view showing all local contacts for a given account, with columns for: avatar/photo, name, email, type indicator (sent-to / received-from / junk / no-junk), times contacted, last contacted, and favorite/ignored state.
- **FR-20.** The contact list shall be searchable by name and email address.
- **FR-21.** The user shall be able to edit a contact's email, name, type, and group from the contact list.
- **FR-22.** The user shall be able to delete individual contacts from the contact list.
- **FR-23.** The user shall be able to bulk-clear all contacts of one or more types for a given account.
- **FR-24.** The user shall be able to cycle a contact's state through default → favorite → ignored → default from the contact list.
- **FR-25.** Junk/blocked contacts shall be viewable as a separate filtered list or tab, showing the count of blocked addresses per account.

### 7.5 Contact groups

- **FR-26.** Each contact may belong to at most one group (a string label).
- **FR-27.** The application shall provide a view of all defined groups with the count of contacts in each.
- **FR-28.** The user shall be able to filter the contact list to a single group.
- **FR-29.** Group labels shall be preserved during vCard import and export.

### 7.6 Automatic purge

- **FR-30.** The application shall periodically purge contacts that satisfy **all** of the following conditions: (a) last-contacted is older than a user-configurable threshold (default: one month), (b) times-contacted is below a user-configurable threshold (default: one), (c) state is not "favorite", and (d) type is sent-to or received-from (not junk or no-junk).
- **FR-31.** The purge shall run as part of the application's regular maintenance cycle, not on a user-triggered action (though the user may trigger maintenance manually).
- **FR-32.** The purge thresholds (age and frequency) shall be configurable in the application's settings.

### 7.7 Import and export

- **FR-33.** The application shall support exporting the local contact list to the vCard format.
- **FR-34.** The application shall support importing contacts from vCard files, creating new contacts or updating existing ones.
- **FR-35.** The contact type (sent-to, received-from, etc.) shall be preserved across export and import using an application-specific extension field in the vCard.

### 7.8 Cascading deletion

- **FR-36.** When an account is deleted from the application, all contacts associated with that account shall be deleted.

## 8. Non-Functional Requirements

- **NFR-1. Performance.** Autocomplete suggestions shall appear within 200 ms of the user's keystroke, even with tens of thousands of stored contacts.
- **NFR-2. Scale.** The contact store shall handle at least 100,000 contacts across all accounts without degradation in query or display performance.
- **NFR-3. Storage efficiency.** The store shall use appropriate indexing to support fast lookups by email, by frequency, by recency, and by state, without excessive storage overhead.
- **NFR-4. Privacy.** No contact data shall leave the local machine. External avatar lookups (Gravatar, favicon, etc.) are governed by separate features and preferences, not by the contact store itself.
- **NFR-5. Data integrity.** The uniqueness constraint (account + type + email) shall be enforced at the storage level, not merely in application logic, to prevent duplicates under any race condition.
- **NFR-6. Graceful degradation.** If the contact store is empty (new installation, contacts cleared, learning disabled), autocomplete shall degrade gracefully — returning no suggestions rather than failing — and shall recover as soon as contacts are learned or imported.
- **NFR-7. Accessibility.** The contact list and all management controls shall be fully navigable via keyboard and compatible with screen readers.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|

| AC-2 | After receiving a message from a new address, the sender's address appears as a received-from contact with times-contacted = 1. |
| AC-3 | Sending a second message to the same address increments times-contacted to 2 and updates last-contacted, while first-contacted remains unchanged. |

| AC-6 | A contact marked as "ignored" does not appear in autocomplete suggestions. |
| AC-7 | With "suggest from sent-to contacts" disabled, sent-to contacts do not appear in autocomplete. With "suggest from received-from contacts" disabled, received-from contacts do not appear. |
| AC-8 | With account-scoped autocomplete enabled, composing from Account A shows only contacts associated with Account A. |
| AC-9 | The same email address appearing as both sent-to and received-from shows as a single autocomplete entry. |
| AC-10 | Addresses matching no-reply patterns (e.g. noreply@example.com) are not recorded as contacts. |
| AC-11 | Messages processed from the Drafts, Trash, or Spam folders do not create or update contacts. |
| AC-12 | Editing a contact's name and group in the contact management interface persists across application restart. |
| AC-13 | Deleting an account removes all associated contacts from the store. |
| AC-14 | After the purge interval elapses, a contact with times-contacted below the threshold and last-contacted older than the threshold is automatically removed — unless it is a favorite. |
| AC-15 | A favorite contact is never removed by automatic purge, regardless of age or frequency. |
| AC-16 | Exporting contacts to vCard and re-importing them produces the same set of contacts with the same type, name, email, and group. |

| AC-19 | Autocomplete returns results within 200 ms on a store containing 50,000 contacts. |

## 10. Open Questions

- **OQ-1. Group cardinality.** The source application allows each contact to belong to at most one group. Should the desktop application support multiple group memberships per contact (tagging rather than categorisation), or is the single-group model sufficient?
- **OQ-2. Cross-account contact merging.** Currently contacts are scoped per-account. Should the application offer a "global" autocomplete mode that merges contacts across all accounts, deduplicating by email? The source application supports this via a preference, but the interaction with per-account scoping and identity inference may need design attention.
- **OQ-3. Identity inference from contact history.** The source application stores which sending identity was last used to contact a given address. Should the desktop application use this to pre-select the From identity when composing to that address from the autocomplete? If so, what happens when the identity no longer exists?
- **OQ-4. Folder reference on contacts.** The source application stores a reference to the "last used folder" on each contact. The purpose of this field is not entirely clear — it may be used for move-to suggestions or context display. The desktop application should clarify whether this field is needed and, if so, what user-visible behavior it supports.
- **OQ-5. Archive folder exclusion.** The source application excludes Drafts, Trash, and Spam from contact learning but does not appear to exclude Archive. Should contact learning from Archive folders be suppressed (since archived messages have already been processed once)?
- **OQ-6. Purge defaults.** The default purge threshold is one month and one contact. For a desktop application that may run less frequently than a phone, should the default age threshold be longer (e.g. three months)?
- **OQ-7. vCard extension interoperability.** The source application uses a custom extension field in vCard to preserve the contact type. Should the desktop application adopt the same extension, define its own, or use a standard vCard field?
- **OQ-8. Suggest-received default.** The source application defaults "suggest from received" to off, while "suggest from sent" defaults to on. This means a fresh installation only learns from outgoing mail. Should the desktop application adopt the same conservative default, or enable both directions by default for a richer out-of-box experience?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Per-account scoping.** Contacts are scoped to the account from which they were learned. This is a deliberate design choice: a user with a work account and a personal account does not want work colleagues appearing as suggestions when composing from their personal account. The optional "suggest across all accounts" mode exists as an opt-in escape hatch, not as the default.
- **N-2. Separate types rather than a single merged record.** The source application stores a sent-to contact and a received-from contact as two separate records even when the email address is the same. This allows independent tracking of outbound vs. inbound frequency and lets the user disable one direction without affecting the other. Autocomplete merges them at query time for display, but the underlying data remains separate.
- **N-3. Three-state contact state.** The favorite/ignored/default trichotomy covers the three most common user intentions without introducing a complex tagging or scoring system. "Favorite" is the positive signal ("always suggest this person first"), "ignored" is the negative signal ("never suggest this person"), and "default" is the absence of either signal — which lets frequency and recency do the ranking.
- **N-4. Junk contacts as a distinct type.** Blocked/junk addresses are stored as a separate contact type rather than as a flag on an existing contact. This ensures that blocking a sender does not interfere with any existing sent-to or received-from record for the same address, and that the junk list can be managed (viewed, cleared) independently.
- **N-5. Conservative learning defaults.** The source application defaults to learning from sent mail but not from received mail. The rationale is that sent-to addresses reflect deliberate user choices, while received-from addresses include unsolicited mail, mailing lists, and automated senders. Enabling received-from learning is a user opt-in.
- **N-6. Purge protects favorites.** Automatic purge never removes favorite contacts regardless of age or frequency. This ensures that the user's explicit curation is never overridden by the maintenance heuristic.
- **N-7. Avatar reference, not avatar data.** The contact record stores a reference (URI) to an avatar, not the image data itself. This keeps the contact store lightweight and delegates the actual avatar resolution, caching, and display to the avatar subsystem (feature 12.8).
