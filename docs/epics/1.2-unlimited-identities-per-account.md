# Epic: Unlimited Identities per Account

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.2** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Unlimited identities per account. Each account can host multiple 'From' identities (alias, display name, signature, reply-to, BCC, signing/encryption defaults, default folder)."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email account often serves as a conduit for more than one sender address. A user may own aliases on the same domain (`jane@example.com`, `support@example.com`), addresses on vanity domains that route through a single mailbox, or simply want to present different display names and signatures depending on context (work vs. personal, individual vs. role). In all these cases the user authenticates to the same incoming-mail server, but needs to control the outgoing "From" line, reply-to address, automatic BCC, signature, and encryption posture independently for each address they send as.

The Identities feature exists to model this. An **identity** is the outgoing persona associated with a single "From" address. Each account may host an unlimited number of identities, each with its own:

- display name and email address,
- independent outgoing-mail server credentials (the same or different from the account's),
- signature (rich text or plain),
- default reply-to, CC, and BCC addresses,
- signing and encryption defaults,
- sender-address flexibility rules (catch-all, plus-addressing, sub-addressing),
- visual color tag,
- sensitivity and receipt preferences.

One identity per account is designated **primary** and serves as the default "From" whenever the user starts a new message from that account without further context.

## 2. Goals

- **G1.** Allow each account to host an unlimited number of identities, each independently configurable.
- **G2.** Give every identity its own outgoing-mail server connection settings, so that different identities on the same account can send through different servers when necessary.
- **G3.** Provide a clear primary-identity concept per account so that the application can always resolve "which identity should I use?" without ambiguity.
- **G4.** Match incoming messages to the correct identity automatically when replying, so that the user's outgoing address is contextually appropriate without manual selection.
- **G5.** Let each identity carry its own signature, encryption posture, and addressing defaults, so that switching identity also switches all associated sending behavior.
- **G6.** Support sender-address flexibility (plus-addressing, catch-all, regex-based matching) so that users with wildcard or sub-addressed mailboxes can reply from the exact address a message was sent to.
- **G7.** Make identity management discoverable and efficient — creation, editing, copying, and deletion should be straightforward even for users with many identities.

## 3. Non-Goals

- **NG1.** This epic does **not** define how incoming mail is fetched, stored, or synchronized; identities govern *outgoing* behavior only.
- **NG2.** This epic does **not** define per-identity incoming folders. Folder assignment (Inbox, Sent, Drafts, Trash, Archive) remains an account-level concern. (See §11 for rationale.)
- **NG3.** Server-side alias management is out of scope. This feature models locally configured sending personas; it does not create or modify aliases on the mail server.
- **NG4.** Shared identities across accounts are out of scope. Each identity belongs to exactly one account. If the user wants the same address on two accounts, they create two identities.
- **NG5.** This epic does **not** prescribe the signature editor's formatting capabilities — only that each identity has an independent signature. The editor itself is a composition concern (see feature 4.5).

## 4. Glossary

| Term | Definition |
|---|---|
| **Account** | A configured connection to a single mail provider, with credentials, server settings, and its own folder tree. |
| **Identity** | A sending persona within an account: a "From" address together with its display name, outgoing-mail server credentials, signature, and sending defaults. An account may have many identities. |
| **Primary identity** | The one identity per account that is used by default when composing from that account without further context. Exactly one identity per account holds this designation. |
| **Primary account** | The one account designated as the system-wide default for operations where no account is implied (e.g. composing from the Unified Inbox). |
| **Display name** | An optional human-readable name presented alongside the email address in the "From" header (e.g. `Jane Doe <jane@example.com>`). |
| **Signature** | A block of text (optionally rich text) appended to outgoing messages composed under a given identity. |
| **Reply-to address** | An address placed in the `Reply-To` header, directing replies to a different address than the "From". |
| **Envelope sender** | The address used in the SMTP envelope (`MAIL FROM`), which may differ from the visible "From" header. |
| **Sender-extra** | A mode in which the user may edit the sender address at compose time, typically to use plus-addressing (`user+tag@domain`) or sub-domain addressing. |
| **Catch-all** | A server configuration where all addresses at a domain are delivered to a single mailbox, making regex-based identity matching useful. |
| **Composable identity** | An identity that is eligible for use in the composer: the identity is enabled, its owning account is enabled, and the account has a Drafts folder configured. |

## 5. Personas

- **P1. The single-alias user.** Has one account and one identity. May never touch identity settings beyond the initial setup. Expects the application to "just work" with the address they entered during account creation.
- **P2. The multi-alias user.** Has one account but sends from several addresses (personal, role address, department alias). Needs to switch identities when composing and expects replies to be matched to the correct alias automatically.
- **P3. The catch-all domain owner.** Owns a domain where every address routes to one mailbox. Wants to reply from the exact address each message was sent to, using wildcard or regex matching rather than creating a separate identity for every possible address.
- **P4. The multi-account power user.** Has several accounts, each with one or more identities. Needs the composer to select the right identity across accounts and wants clear visual cues about which identity is active.
- **P5. The security-conscious sender.** Requires different signing/encryption configurations for different identities — e.g. one identity signs with an OpenPGP key, another with an S/MIME certificate, a third sends unsigned.

## 6. User Stories

### 6.1 Identity lifecycle

> **US-1.** As any user, when I finish adding a new account, I want the application to create a default identity for that account automatically (using the account's email address, name, and outgoing-mail server settings), so that I can start sending immediately without manual identity setup.

> **US-2.** As any user, I want to create additional identities for an existing account, so that I can send from multiple "From" addresses through the same mailbox.

> **US-3.** As a multi-alias user, I want a quick way to create a new identity by copying an existing one and changing only the email address, so that I do not have to re-enter server credentials and other settings each time.

> **US-4.** As any user, I want to edit any property of an existing identity at any time (display name, email address, server credentials, signature, defaults), so that I can correct mistakes or adapt to changes.

> **US-5.** As any user, I want to delete an identity I no longer need, so that it does not clutter the identity list or appear in the composer.

> **US-6.** As any user, I want to enable or disable an identity without deleting it, so that I can temporarily take an identity out of rotation and restore it later with all its settings intact.

> **US-7.** As any user, when I delete an account, I expect all identities belonging to that account to be deleted automatically, so that no orphaned identities remain.

### 6.2 Primary identity

> **US-8.** As any user, I want exactly one identity per account to be designated as primary, so that the application always has a clear default for that account.

> **US-9.** As any user, I want to change which identity is primary for an account via a simple action (e.g. a context menu or a toggle in the identity editor), so that I can switch my default sending address easily.

> **US-10.** As any user, when the first identity is created for an account, I want it to be marked as primary automatically, so that I do not have to set it manually.

### 6.3 Identity selection during composition

> **US-11.** As any user, when I compose a new message from a specific account, I want the composer to pre-select that account's primary identity, so that I get a sensible default.

> **US-12.** As any user, when I compose from the Unified Inbox (or any cross-account context), I want the composer to pre-select the primary identity of the primary account, so that there is always a deterministic default.

> **US-13.** As any user, I want the composer to present a list of all composable identities (across all enabled accounts), so that I can switch to any identity before or during composition.

> **US-14.** As any user, I want each entry in the identity selector to show the display name, email address, owning account name, and a visual indicator for the primary identity, so that I can distinguish identities at a glance.

> **US-15.** As a multi-alias user, I want the identity selector to show visual hints when an identity has a default CC, BCC, or a reply-to address that differs from the "From" address, so that I am reminded of the side-effects of choosing that identity.

### 6.4 Automatic identity matching on reply

> **US-16.** As any user, when I reply to a message, I want the application to automatically select the identity whose email address matches the address the original message was sent to, so that my reply comes from the same address the sender wrote to.

> **US-17.** As any user, I want the matching to check the To, CC, BCC, and delivery-envelope headers of the original message (in that priority order), so that the application finds the right identity even when I was CC'd or BCC'd.

> **US-18.** As any user, when an exact address match exists, I want it to take precedence over a partial or regex match, so that the most specific identity wins.

> **US-19.** As any user, when no identity matches the original message's addressing at all, I want the application to fall back to the account's primary identity (or the system-wide primary identity if the account is ambiguous), so that composition can always proceed.

### 6.5 Sender-address flexibility

> **US-20.** As a catch-all domain owner, I want to enable a "sender extra" mode on an identity that lets me edit the sender address at compose time, so that I can reply from the exact address each message was sent to without pre-creating an identity for every variant.

> **US-21.** As a catch-all domain owner, I want to define a regex pattern that controls which sender-address variations are allowed for an identity, so that the application can auto-match incoming addresses to this identity even when the local part differs.

> **US-22.** As any user with sender-extra enabled, I want to be able to edit the display name at compose time as well (optionally), so that I can adapt the "From" line fully.

> **US-23.** As any user with sender-extra enabled, I want to be able to edit the reply-to name at compose time as well (optionally), so that the reply-to header can be adjusted to match.

### 6.6 Outgoing-mail server per identity

> **US-24.** As any user, I want each identity to have its own outgoing-mail (SMTP) server configuration — host, port, encryption mode, username, password, and client certificate — so that different identities on the same account can send through different servers if needed.

> **US-25.** As any user, when I create a new identity by copying an existing one, I want the outgoing-mail server settings to be copied as well, so that I only need to change what differs.

> **US-26.** As any user, I want the identity editor to offer automatic configuration lookup based on the identity's email domain, so that outgoing-mail server settings can be populated without manual entry when the provider is known.

> **US-27.** As any user, I want to test the outgoing-mail server connection from the identity editor before saving, so that I can verify credentials without sending a real message.

> **US-28.** As any user, when I change the account's password, I want identities that share the same outgoing-mail server username to have their passwords updated automatically, so that I do not have to update each identity individually.

### 6.7 Signatures

> **US-29.** As any user, I want each identity to have its own independent signature (rich text or plain text), so that different sending personas can have different sign-offs.

> **US-30.** As any user, I want to edit an identity's signature from within the identity editor, opening a dedicated signature editing screen, so that I have adequate space and formatting tools.

> **US-31.** As any user, when I switch identities in the composer, I want the signature block in the draft to update to reflect the newly selected identity's signature, so that the outgoing message is always consistent with the chosen identity.

### 6.8 Reply-to, CC, and BCC defaults

> **US-32.** As any user, I want to set a default reply-to address per identity, so that replies to messages I send from that identity are directed to a different address than the "From".

> **US-33.** As any user, I want to set default CC and BCC addresses per identity, so that every message sent from that identity automatically includes those recipients.

> **US-34.** As any user, when I compose with an identity that has default CC or BCC addresses, I want those addresses to be pre-populated in the composer, and I want to be able to remove them before sending if needed.

### 6.9 Encryption and signing defaults

> **US-35.** As a security-conscious sender, I want each identity to have independent defaults for whether messages should be signed, encrypted, or both, so that my encryption posture can vary by identity.

> **US-36.** As a security-conscious sender, I want each identity to specify its preferred encryption method (OpenPGP or S/MIME), so that the correct cryptographic system is used automatically.

> **US-37.** As a security-conscious sender, I want to associate a specific signing key (OpenPGP key or S/MIME certificate) with each identity, so that the application uses the correct key without prompting me each time.

> **US-38.** As a security-conscious sender, I want S/MIME identities to support separate certificates for signing and encryption, so that I can use split-key configurations required by some organizations.

### 6.10 Envelope sender

> **US-39.** As a power user, I want to set an envelope sender address per identity that differs from the visible "From" address, so that bounce messages and delivery notifications are routed to a different mailbox.

### 6.11 Visual identity

> **US-40.** As any user, I want to assign a color to each identity, so that messages composed from or matched to that identity are visually tagged throughout the application.

> **US-41.** As any user, I want the identity's color to be used in the message list, the composer, and the identity selector, so that I can tell at a glance which identity is in play.

### 6.12 Receipt and sensitivity defaults

> **US-42.** As any user, I want to set per-identity defaults for requesting read receipts or delivery notifications, so that receipt behavior can vary by identity.

> **US-43.** As any user, I want to set a per-identity default sensitivity level (normal, personal, private, company-confidential), so that messages sent from certain identities carry the appropriate sensitivity header.

### 6.13 Identity list management

> **US-44.** As any user, I want to see all identities for an account in a single list, clearly showing which is primary, which are enabled, and any errors, so that I can manage them efficiently.

> **US-45.** As any user, I want to reorder identities in the list, so that my most-used identities appear first in the composer's identity selector.

## 7. Functional Requirements

### 7.1 Identity data

- **FR-1.** An identity shall have, at minimum: a name, an email address, a link to its owning account, and a complete set of outgoing-mail server connection parameters (host, port, encryption mode, username, password).
- **FR-2.** An identity shall optionally have: a display name (separate from the identity name), a color, a signature, a reply-to address, default CC addresses, default BCC addresses, an envelope sender address, encryption defaults, a signing key reference, a receipt-request preference, a sensitivity level, and sender-extra configuration.
- **FR-3.** All identity properties shall be independently editable after creation.
- **FR-4.** Identity data shall persist across application restarts.
- **FR-5.** There shall be no application-imposed upper limit on the number of identities per account.

### 7.2 Identity–account relationship

- **FR-6.** Every identity shall belong to exactly one account. An identity cannot exist without an owning account.
- **FR-7.** Deleting an account shall delete all of its identities.
- **FR-8.** An identity shall not be transferable between accounts after creation.

### 7.3 Default identity on account creation

- **FR-9.** When a new account is created, the application shall automatically create one identity for that account, populated with the account's email address, name, and outgoing-mail server credentials.
- **FR-10.** This automatically created identity shall be marked as primary.

### 7.4 Primary identity

- **FR-11.** Exactly one identity per account shall be designated as the primary identity at any given time.
- **FR-12.** The user shall be able to change which identity is primary via the identity list or the identity editor.
- **FR-13.** Setting a new primary identity shall automatically clear the primary flag from the previously primary identity in the same account.

### 7.5 Composable identity filtering

- **FR-14.** An identity shall be considered *composable* (eligible for use in the composer) if and only if all of the following are true: (a) the identity is enabled, (b) the owning account is enabled, (c) the owning account has a Drafts folder configured.
- **FR-15.** The composer's identity selector shall display only composable identities.
- **FR-16.** Composable identities shall be sorted by: account primary status (primary account first), then account display order, then account name; within the same account: identity primary status (primary first), then display name, then name, then email address.

### 7.6 Automatic identity matching on reply

- **FR-17.** When the user replies to a message, the application shall attempt to select the identity whose email address matches an address found in the original message's headers, checking in priority order: To, CC, BCC, From, then delivery-envelope headers (Delivered-To, Envelope-To, Original-To, X-Delivered-To, X-Original-To).
- **FR-18.** An exact email-address match shall take precedence over a partial or regex-based match.
- **FR-19.** If sender-extra regex matching is enabled on an identity, the regex shall be applied to the local part of candidate addresses (the part before the `@`). The domain part shall always require an exact, case-insensitive match.
- **FR-20.** If no identity matches any header, the application shall fall back to the account's primary identity; if the account is ambiguous, to the system-wide primary identity.

### 7.7 Sender-address flexibility

- **FR-21.** Each identity shall have an optional "sender extra" flag. When enabled, the composer shall allow the user to edit the sender address before sending.
- **FR-22.** Each identity shall have an optional "sender extra name" flag. When enabled alongside sender-extra, the composer shall also allow editing the display name.
- **FR-23.** Each identity shall have an optional "reply extra name" flag. When enabled, the composer shall allow editing the reply-to display name.
- **FR-24.** Each identity shall have an optional regex pattern that defines which sender-address variations are considered valid matches for that identity during automatic identity selection.
- **FR-25.** When a regex pattern is defined and the `@` symbol is included in the pattern, the regex shall match against the full address; otherwise it shall match against the local part only, with the domain always required to match exactly.

### 7.8 Outgoing-mail server per identity

- **FR-26.** Each identity shall carry its own outgoing-mail server configuration, independent of other identities in the same account.
- **FR-27.** The identity editor shall offer automatic server-configuration lookup based on the identity's email domain, using the application's bundled provider database.
- **FR-28.** The identity editor shall provide a connection-test function that verifies the outgoing-mail server credentials before saving.
- **FR-29.** When the user changes an account's password, the application shall offer to update the passwords of all identities in that account that share the same outgoing-mail server username.

### 7.9 Signatures

- **FR-30.** Each identity shall have an independent signature field, supporting rich text.
- **FR-31.** When the user switches identities in the composer, the application shall replace the signature block in the draft with the newly selected identity's signature.

### 7.10 Reply-to, CC, BCC, and envelope sender

- **FR-32.** Each identity shall optionally specify a reply-to address. When set, outgoing messages shall include a `Reply-To` header with that address.
- **FR-33.** Each identity shall optionally specify one or more default CC addresses. These shall be pre-populated in the CC field of the composer when that identity is selected.
- **FR-34.** Each identity shall optionally specify one or more default BCC addresses. These shall be pre-populated in the BCC field of the composer when that identity is selected.
- **FR-35.** Each identity shall optionally specify an envelope sender address. When set, the SMTP envelope (`MAIL FROM`) shall use that address instead of the identity's visible "From" address.
- **FR-36.** Default CC and BCC addresses shall be pre-populated but user-removable before sending.

### 7.11 Encryption and signing defaults

- **FR-37.** Each identity shall independently specify: (a) whether outgoing messages should be signed by default, (b) whether outgoing messages should be encrypted by default, (c) the preferred encryption method (OpenPGP or S/MIME).
- **FR-38.** Each identity shall optionally reference a specific signing key (an OpenPGP key identifier or an S/MIME certificate alias).
- **FR-39.** For S/MIME, each identity shall support separate certificate references for signing and encryption.
- **FR-40.** The composer shall apply the selected identity's encryption and signing defaults when a new draft is created, and shall update them when the user switches identities.

### 7.12 Receipt and sensitivity defaults

- **FR-41.** Each identity shall optionally specify a default receipt-request type (none, delivery notification, read receipt, or both).
- **FR-42.** Each identity shall optionally specify a default sensitivity level (normal, personal, private, company-confidential).
- **FR-43.** These defaults shall be applied to new drafts composed under that identity and updated when the user switches identities.

### 7.13 Visual identity

- **FR-44.** Each identity shall optionally have a user-assigned color.
- **FR-45.** The identity color, when set, shall be used in the message list, the composer, and the identity selector to visually tag messages associated with that identity.
- **FR-46.** If no identity color is set, the application shall fall back to the account color.

### 7.14 Identity creation helpers

- **FR-47.** The application shall provide a "copy identity" action that duplicates an existing identity's settings into a new identity, allowing the user to change only what differs (typically the email address).
- **FR-48.** The application shall provide a shortcut to create a new alias identity from an existing identity, pre-filling all server and account settings.

### 7.15 Encoding options

- **FR-49.** Each identity shall optionally specify encoding preferences (e.g. Unicode encoding, octet-stream MIME) that override the application-wide defaults for messages sent from that identity.

## 8. Non-Functional Requirements

- **NFR-1. No identity limit.** The application shall impose no hard upper limit on the number of identities per account. The feature shall remain usable with at least 50 identities across 20 accounts.
- **NFR-2. Responsiveness.** Switching identities in the composer (including signature replacement and encryption-default update) shall complete in well under one second.
- **NFR-3. Credential security.** Identity passwords shall be stored with the same protection as account passwords (encrypted at rest). Passwords shall not be exposed in logs, exports (unless explicitly included in an encrypted backup), or error messages.
- **NFR-4. Offline operation.** All identity management operations (create, edit, delete, reorder) shall work offline. Outgoing messages queued offline shall retain the identity they were composed with and send through that identity's server when connectivity returns.
- **NFR-5. Consistency.** When a message is sent, the "From", "Reply-To", signature, encryption, and all other identity-governed properties shall reflect the identity that was selected at send time, even if the identity's settings were changed between drafting and sending.
- **NFR-6. Accessibility.** All identity-related controls (identity selector, editor fields, primary toggle) shall be keyboard-navigable with screen-reader labels and shall respect the application-wide theme.
- **NFR-7. Privacy.** Identity data (especially passwords and signing keys) shall never leave the local machine except through the user's own outgoing-mail server connections or an explicitly requested encrypted backup.

## 10. Open Questions

- **OQ-1. Per-identity Sent folder.** The current implementation stores Sent, Drafts, Trash, and Archive folder assignments at the account level, not the identity level. Some users with multiple domains on one account may want sent mail filed into different folders per identity. Should a future iteration add per-identity folder overrides, or is account-level assignment sufficient?
- **OQ-2. Identity ordering.** The sort order of identities in the composer's selector follows a fixed algorithm (primary first, then alphabetical by display name / name / email). Should the user be able to define a custom order, or is the current sort sufficient?
- **OQ-3. Sender-extra regex complexity.** The regex matching feature is powerful but may be difficult for non-technical users to configure correctly. Should the application offer a simplified mode (e.g. "match any address at this domain") alongside the full regex option?
- **OQ-4. Password sync scope.** When the user changes an account password, identities sharing the same outgoing-mail server username have their passwords auto-updated. Should the scope of this auto-update be configurable (e.g. opt-in per identity), or is the current behavior — all matching identities — always correct?
- **OQ-5. Interaction between identity color and account color.** When both are set, the identity color takes precedence. Should there be a way to explicitly inherit the account color (i.e. "no override") vs. having no color at all, or is the current fallback (identity color → account color → no color) sufficient?
- **OQ-6. Display-name vs. identity-name distinction.** An identity has both a "name" (used as a label in the identity list) and an optional "display name" (used in the "From" header). This two-field model may confuse users who expect a single "name" field. Should the distinction be preserved, simplified, or better explained in the UI?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Per-identity SMTP, not per-account.** Each identity carries its own full outgoing-mail server configuration. This is intentional and necessary: a user may have aliases on different domains that each require sending through a different server (e.g. a corporate relay for work aliases, a personal server for personal ones). Even when all identities share the same server, storing credentials per-identity means password changes can be applied per-identity, and a misconfigured identity does not break others.
- **N-2. Folders remain account-level.** Drafts, Sent, Trash, and Archive folders are assigned at the account level, not the identity level. This simplifies the model considerably: the account's folder tree is a property of the incoming-mail server, and identities — which are outgoing-side concepts — should not dictate incoming-side storage. The high-level feature description mentions "default folder" as an identity property; in practice, the only folder interaction is that an identity requires its account to have a Drafts folder in order to be composable.
- **N-3. Exactly one primary per account.** The application enforces a single primary identity per account (and a single primary account system-wide). This avoids ambiguity in every "which identity should I use?" decision. The primary is a tie-breaker, not a restriction — the user can always override it in the composer.
- **N-4. Identity matching priority order.** The To → CC → BCC → From → delivery-envelope header priority is deliberate. The To field is the strongest signal ("this message was addressed directly to this identity"), while delivery-envelope headers are a last resort (they may be rewritten by intermediaries). This order minimizes false matches.
- **N-5. Regex matches local part only by default.** The sender-extra regex is applied to the local part (before the `@`) unless the regex itself contains `@`, in which case it matches the full address. This design avoids accidental cross-domain matches — the domain must always match exactly — while still allowing full-address patterns for advanced users who need them.
- **N-6. Copy-identity workflow.** The "create alias" / "copy identity" shortcut is an important usability affordance. Without it, creating a second identity requires re-entering all outgoing-mail server fields, which is error-prone and tedious. The copy workflow pre-fills everything and lets the user change only the email address, which is the common case.
- **N-7. Composable-identity filter.** An identity is hidden from the composer if it or its account is disabled, or if the account lacks a Drafts folder. This is a safety measure: sending requires a Drafts folder for auto-save, and an identity whose account is disabled should not be offered for composition. The filter is applied at query time so the composer never presents an unusable option.
