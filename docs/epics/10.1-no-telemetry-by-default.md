# Epic: No Telemetry by Default

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **10.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"No analytics, no remote crash reporting; opt-in error reporting via Bugsnag if user explicitly enables it."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Users of email clients handle some of the most sensitive data in their digital lives — credentials, personal correspondence, financial information, legal communications. Any telemetry or crash-reporting system that phones home without explicit consent represents both a privacy violation and a trust barrier.

This feature exists to guarantee that the application collects and transmits **zero data** about the user, their behavior, or their errors to any external service — unless the user deliberately, affirmatively opts in. The default posture is total silence: no analytics, no usage metrics, no crash reports, no heartbeat pings, no background network traffic to vendor endpoints.

When a user *does* opt in, the application provides a narrowly-scoped error-reporting channel that transmits only crash and exception data, anonymized via a random identifier unlinked to any account or device identity, with automatic expiry of stored reports. The user can revoke consent at any time, and the application immediately ceases transmission.

This stance differentiates the application from mainstream email clients that bundle analytics by default, and is a core part of its privacy-first identity.

## 2. Goals

- **G1.** Ensure that a freshly installed application, with no user interaction beyond account setup, sends zero bytes to any analytics, telemetry, or error-reporting endpoint.
- **G2.** Provide a clear, user-initiated path to opt in to anonymous error reporting, so that users who wish to help improve the application can do so.
- **G3.** Ensure that the scope of data transmitted when error reporting is enabled is minimal, anonymous, and time-bounded.
- **G4.** Allow the user to revoke error-reporting consent at any time, with immediate effect.
- **G5.** Make the privacy stance visible and verifiable — the user should be able to confirm that no telemetry is active without relying solely on the developer's word.

## 3. Non-Goals

- **NG1.** This epic does **not** cover network traffic required for core email operations (IMAP/SMTP/POP3 connections, DNS, certificate validation). Only *vendor-directed* telemetry and reporting traffic is in scope.
- **NG2.** This epic does **not** define the mechanism for local-only crash logging (writing crash data to a file on the user's machine). Local logging is not telemetry and is always permitted.
- **NG3.** This epic does **not** require the application to block third-party libraries from making network calls unrelated to telemetry (e.g. font fetching, system update checks). Only application-initiated reporting is in scope.
- **NG4.** This epic does **not** prescribe a specific error-reporting vendor or protocol — only the behavioral contract around consent, data scope, and anonymity.
- **NG5.** This epic does **not** cover in-app feedback forms, manual bug-report submission, or "send debug info to developer" actions that the user initiates on a per-incident basis. Those are distinct voluntary disclosure actions, not background telemetry.

## 4. Glossary

| Term | Definition |
|---|---|
| **Telemetry** | Any automatic, periodic, or event-driven transmission of usage data, behavioral metrics, or performance counters from the application to an external service operated by the developer or a third party. |
| **Analytics** | A subset of telemetry focused on tracking user behavior patterns — screens visited, features used, session duration, click paths, etc. |
| **Crash reporting** | Automatic capture and transmission of exception/crash data (stack traces, error context) to an external service when the application encounters an unhandled error. |
| **Error reporting** | Broader than crash reporting; includes warnings, caught exceptions, and abnormal-state notifications in addition to fatal crashes. |
| **Opt-in** | A consent model where the feature is disabled by default and requires an explicit, affirmative user action to enable. Contrasted with opt-out, where the feature is enabled by default and requires action to disable. |
| **Anonymous identifier** | A randomly-generated, non-deterministic identifier used solely to correlate error reports from the same installation. It is not derived from any device hardware identifier, account credential, email address, or personally-identifiable information. |
| **Report expiry** | The automatic deletion of submitted error reports after a defined retention period, without requiring user action. |

## 5. Personas

- **P1. The privacy-conscious user.** Chose this application specifically because of its no-telemetry stance. Expects zero outbound data by default and would consider any unrequested transmission a breach of trust.
- **P2. The helpful contributor.** Understands that error reports help developers fix bugs. Willing to opt in to anonymous reporting, but wants to know exactly what is sent and wants the ability to stop at any time.
- **P3. The verifier.** Technically sophisticated user who audits network traffic or reviews source code to confirm that privacy claims are truthful. Expects the application's behavior to match its stated policy exactly.
- **P4. The default user.** Does not read settings, does not change defaults, and may not even be aware that error reporting exists. This user must be protected by the default-off posture without any action on their part.

## 6. User Stories

### 6.1 Default state — no telemetry

> **US-1.** As a default user, when I install and use the application without changing any settings, I want zero data about my usage, crashes, or errors to be sent to any external service, so that my privacy is protected without requiring any action from me.

> **US-2.** As a privacy-conscious user, I want the application to contain no analytics infrastructure whatsoever — not merely disabled analytics, but the complete absence of usage tracking, behavioral metrics, and session recording — so that there is no hidden capability that could be enabled without my knowledge.

> **US-3.** As any user, when the application crashes or encounters an error while error reporting is disabled, I want the crash information to be recorded only locally on my machine (if at all), so that I can choose to share it manually if I wish but nothing is sent automatically.

### 6.2 Opt-in to error reporting

> **US-4.** As a helpful contributor, I want a clearly-labeled setting in the application's preferences that allows me to enable anonymous error reporting, so that I can make an informed, deliberate choice to help improve the application.

> **US-5.** As any user, when I encounter an error and error reporting is not enabled, I want the application to briefly inform me that error reporting exists and could help improve the application, without being coercive or repetitive, so that I am aware of the option but not pressured into it.

> **US-6.** As any user, when I am prompted about error reporting, I want to be able to decline, and once I decline I want the application to stop asking me, so that I am not nagged.

> **US-7.** As a helpful contributor, when I am prompted about error reporting, I want to be able to access information explaining what data will be sent, how it is anonymized, and how long it is retained, so that I can make an informed decision before consenting.

### 6.3 Behavior when error reporting is enabled

> **US-8.** As a user who has opted in, when the application encounters an error or abnormal state, I want it to send a report containing only technical error information (stack traces, error context, application version) — never my email content, credentials, contacts, or personal information — so that my privacy is maintained even while helping.

> **US-9.** As a user who has opted in, I want error reports to be associated only with a random anonymous identifier that cannot be linked to my email accounts, my device identity, or my real-world identity, so that reports cannot be used to track or profile me.

> **US-10.** As a user who has opted in, I want submitted error reports to be automatically deleted from the remote service after a bounded retention period (no longer than one month), so that my data does not accumulate indefinitely.

> **US-11.** As a user who has opted in, I want to be able to view my anonymous identifier within the application, so that I can reference it if I ever want to request manual deletion of my reports or communicate with the developer about a specific issue.

### 6.4 Revoking consent

> **US-12.** As any user who previously opted in, I want to be able to disable error reporting at any time via the same setting I used to enable it, so that my consent is always revocable.

> **US-13.** As any user who disables error reporting, I want transmission of error data to cease immediately — no queued reports should be sent after I disable the setting — so that revocation has instant effect.

### 6.5 Transparency and verifiability

> **US-14.** As a verifier, I want the application's privacy policy to explicitly document that no data is sent to the developer except when error reporting is explicitly enabled, so that I have a written commitment I can hold the developer accountable to.

> **US-15.** As a verifier, I want the network endpoints used for error reporting (when enabled) to be documented, so that I can independently audit network traffic against the application's claims.

> **US-16.** As any user, I want the application's public documentation (FAQ, privacy policy) to clearly state the default-off posture and what opting in entails, so that the privacy stance is not hidden in technical details.

## 7. Functional Requirements

### 7.1 Default state

- **FR-1.** On a fresh installation with no user configuration changes, the application shall make zero network connections to any analytics, telemetry, or error-reporting endpoint.
- **FR-2.** The application shall contain no analytics or behavioral-tracking subsystem — neither active nor dormant. The absence of analytics shall be architectural, not merely configurational.
- **FR-3.** When the application crashes or encounters an unhandled exception while error reporting is disabled, it shall handle the error locally (log to a local file or pass to the operating system's default crash handler) and shall not transmit any data to an external service.

### 7.2 Error reporting opt-in control

- **FR-4.** The application shall provide a single, clearly-labeled boolean setting (on/off) for error reporting, located in the application's general or miscellaneous settings area.
- **FR-5.** The default value of this setting shall be **off** (disabled).
- **FR-6.** Changing this setting shall take effect immediately — enabling it begins transmission on the next error; disabling it stops transmission immediately.
- **FR-7.** The setting shall persist across application restarts.

### 7.3 Non-coercive awareness prompt

- **FR-8.** When the application encounters an error while error reporting is disabled, it may display a non-modal, dismissible notification informing the user that error reporting is available and would help improve the application.
- **FR-9.** This notification shall include a way to access more information (e.g. a link to the FAQ or an informational dialog) and a way to enable reporting directly.
- **FR-10.** If the user dismisses or declines the notification, the application shall record that the user has been asked and shall not display the notification again for that user.
- **FR-11.** The notification shall never block the user's workflow or require interaction before the user can continue using the application.

### 7.4 Data scope when reporting is enabled

- **FR-12.** When error reporting is enabled, the application shall transmit only: (a) exception type and stack trace, (b) application version and build information, (c) basic operating environment information (OS version, architecture), and (d) the anonymous identifier.
- **FR-13.** The application shall **never** transmit, under any configuration: email message content, email headers, account credentials, email addresses, contact information, folder names, server hostnames, IP addresses, hardware serial numbers, or any other personally-identifiable information.
- **FR-14.** The anonymous identifier shall be a randomly-generated value, created locally, not derived from any hardware or account identifier, and stored only locally on the user's machine.
- **FR-15.** The error-reporting service's own telemetry/analytics features (if any) shall be explicitly disabled in the application's configuration of that service.

### 7.5 Report retention and deletion

- **FR-16.** Error reports submitted to the remote service shall be subject to automatic deletion after a retention period not exceeding one calendar month.
- **FR-17.** The user shall be able to request manual deletion of their reports (by referencing their anonymous identifier) at any time.

### 7.6 Identifier visibility

- **FR-18.** When error reporting is enabled, the application shall make the user's anonymous identifier visible within a diagnostic or "about" screen, so the user can reference it.
- **FR-19.** When error reporting is disabled, the anonymous identifier need not be displayed (since no reports are being sent).

### 7.7 Documentation requirements

- **FR-20.** The application's privacy policy shall explicitly state that no data is transmitted to the developer or any third party except when error reporting is explicitly enabled by the user.
- **FR-21.** The application's FAQ or help documentation shall include an entry explaining: what error reporting is, what data is sent, how it is anonymized, how long reports are retained, and how to enable or disable it.
- **FR-22.** The network endpoints used for error report submission shall be documented in the privacy policy or FAQ.

## 8. Non-Functional Requirements

- **NFR-1. Zero network overhead by default.** With error reporting disabled, the application shall generate exactly zero bytes of network traffic to any vendor or analytics endpoint. This is not "minimal" — it is zero.
- **NFR-2. Immediate effect.** Enabling or disabling error reporting shall take effect within the current application session, without requiring a restart.
- **NFR-3. No degradation.** The application shall function identically whether error reporting is enabled or disabled. No features shall be gated behind enabling telemetry.
- **NFR-4. Minimal data.** Even when enabled, the volume of data transmitted shall be the minimum necessary to identify and diagnose the error. No extra context, screenshots, logs, or behavioral data shall be included.
- **NFR-5. Anonymity.** It shall be computationally infeasible to link an anonymous identifier to a real user identity, device, or account using only the data available to the error-reporting service.
- **NFR-6. Auditability.** The application's source code shall be structured such that the absence of telemetry and the opt-in nature of error reporting are independently verifiable by code review or network traffic analysis.
- **NFR-7. No silent re-enablement.** No application update, configuration migration, or account change shall silently re-enable error reporting once the user has disabled it or declined it.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|
| AC-1 | A fresh installation with default settings generates zero network connections to any analytics, telemetry, or error-reporting endpoint (verifiable by network traffic monitoring). |
| AC-2 | The error-reporting setting exists in the application preferences, is clearly labeled, and defaults to off. |
| AC-3 | Enabling the error-reporting setting and then triggering an error results in a report being transmitted to the documented endpoint. |
| AC-4 | Disabling the error-reporting setting and then triggering an error results in no report being transmitted. |
| AC-5 | After declining the error-reporting prompt, the prompt does not appear again regardless of how many subsequent errors occur. |
| AC-6 | Error reports contain no email content, credentials, account identifiers, device hardware identifiers, or personally-identifiable information (verifiable by inspecting transmitted payloads). |
| AC-7 | The anonymous identifier shown in the application matches the identifier attached to transmitted reports. |
| AC-8 | Reports older than one month are automatically purged from the remote service. |
| AC-9 | The privacy policy and FAQ accurately describe the default-off posture, the data scope, and the retention policy. |
| AC-10 | No application feature is degraded, hidden, or restricted when error reporting is disabled. |
| AC-11 | The error-reporting service's own telemetry features are confirmed disabled (no meta-telemetry about telemetry). |
| AC-12 | When error reporting is disabled, local crash information is still written to a local log file, accessible to the user for manual submission if desired. |
| AC-13 | Toggling error reporting on and then off again results in immediate cessation of report transmission — no queued or delayed reports are sent after disabling. |

## 10. Open Questions

- **OQ-1. Error-reporting service status.** The source application historically used a third-party error-reporting service but removed the integration in a later version (retaining only stub entry points). For the desktop application, should the error-reporting capability be (a) implemented with a new service, (b) restored with the original service, or (c) deferred entirely? The behavioral contract (opt-in, anonymous, time-bounded) remains valid regardless.
- **OQ-2. Local crash log scope.** When error reporting is disabled and a crash occurs, how much information should be written to the local crash log? The source application writes crash logs only for beta/non-store builds when reporting is off. Should the desktop application always write local crash logs, or only in development/beta modes?
- **OQ-3. Prompt trigger conditions.** The source application shows the error-reporting prompt only on certain error conditions (specifically in the message list view). Should the desktop application prompt on any error, or only on specific classes of errors? Over-prompting risks annoyance; under-prompting risks the user never discovering the option.
- **OQ-4. Anonymous identifier lifecycle.** Should the anonymous identifier be regenerated periodically, or remain stable for the lifetime of the installation? A stable identifier helps correlate reports from the same user for debugging; a rotating identifier provides stronger anonymity. The source application uses a single stable identifier.
- **OQ-5. Deletion request mechanism.** FR-17 requires that the user can request deletion by referencing their identifier. What is the mechanism — an in-app button, an email to the developer, a web form? The source application documents email-based deletion requests.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Architectural absence of analytics, not just configuration.** The source application does not contain a dormant analytics system that is merely switched off. There is no analytics code, no tracking identifiers, no behavioral event emission, no session recording. This is a deliberate architectural choice: it is harder to accidentally enable what does not exist. The desktop application should maintain this stance — do not include an analytics subsystem "for future use."

- **N-2. Opt-in, never opt-out.** The source application treats error reporting as strictly opt-in. The user must take an affirmative action to enable it. This is not merely a GDPR compliance measure — it reflects the application's identity and user promise. Even in jurisdictions where opt-out would be legally permissible, the application uses opt-in.

- **N-3. Single ask, then silence.** The awareness prompt is shown at most once. If the user declines, the application records this and never asks again. This avoids the "nag screen" pattern common in applications that really want the user to enable telemetry. The tradeoff is that users who might have opted in later will never be prompted again — but they can still find the setting in preferences.

- **N-4. Service-level telemetry disabled.** When the source application used a third-party error-reporting service, it explicitly disabled that service's own telemetry and session-tracking features. This is important: many error-reporting services collect meta-telemetry (how often the app starts, session duration, etc.) by default. The application explicitly opts out of these features even within the error-reporting service.

- **N-5. Anonymous identifier is not a device fingerprint.** The identifier used for error-report correlation is a random value generated and stored locally. It is deliberately not derived from any hardware identifier, MAC address, hostname, or account information. This means that reinstalling the application produces a new identifier, and there is no way for the error-reporting service to correlate across installations.

- **N-6. Privacy stance as product differentiator.** The no-telemetry-by-default posture is not merely a feature — it is a core value proposition of the application. Documentation, the README, and the privacy policy all prominently advertise it. Any erosion of this stance (even well-intentioned, such as adding opt-out analytics "to understand feature usage") would undermine the application's positioning and user trust.
