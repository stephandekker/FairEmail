# Epic: Inline Attachment Preview

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.9** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Inline preview for images, audio, PDF, EML and barcodes."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

When reading an email that includes attachments, the user benefits enormously from being able to see a meaningful preview of those attachments without leaving the message view or opening a separate application. A recipient scanning a message with three photos, a boarding-pass PDF, and a forwarded `.eml` conversation should be able to glance at all of them in context — understanding what is attached and whether it warrants further action — before committing to opening, downloading, or saving anything.

The Inline Attachment Preview feature provides this capability for the five most common attachment families found in email:

1. **Images** — rendered directly within the message body (when embedded) or as thumbnails in a media gallery below the message.
2. **Audio** — a compact inline player that lets the user listen without switching applications.
3. **PDF documents** — a thumbnail of the first page, giving a visual hint of the content.
4. **Embedded email messages (EML/RFC 822)** — the attached message's headers and body rendered inline, so the user can read the forwarded conversation in place.
5. **Barcodes and QR codes** — image attachments that contain machine-readable codes are automatically decoded and the textual content is displayed, saving the user from having to scan the screen with a separate tool.

Each preview type is independently toggleable, and a master "show thumbnails" switch governs the media gallery as a whole. All previews are generated locally from already-downloaded attachment data — no additional network requests are introduced beyond what attachment download already requires.

## 2. Goals

- **G1.** Let the user see a useful representation of image, audio, PDF, EML, and barcode attachments without leaving the message view.
- **G2.** Make every preview type independently toggleable so users can enable only the previews they find useful and disable those they consider distracting or resource-heavy.
- **G3.** Generate all previews locally from downloaded attachment data, introducing no extra network traffic.
- **G4.** Keep previews lightweight — thumbnails, first-page renders, and decoded text rather than full interactive viewers — so that message scrolling remains fluid even with many attachments.
- **G5.** Present previews in a consistent, visually cohesive gallery area below the message body, with embedded images additionally appearing inline within the HTML body where the sender placed them.

## 3. Non-Goals

- **NG1.** Full interactive viewers (multi-page PDF reader, audio equalizer, image editor) are **not** in scope. The preview is a glance, not a replacement for opening the attachment in a dedicated application.
- **NG2.** Streaming or progressive download of attachments is not a goal. Previews require the attachment data to be locally available first.
- **NG3.** Preview of arbitrary file types (spreadsheets, word-processor documents, archives, executables) is not a goal. Only the five families listed above are covered.
- **NG4.** Video preview is a related but separate concern (covered by its own toggle). This epic mentions it only where behavior is shared with the media gallery.
- **NG5.** Remote-image loading policy (blocking/allowing remote images, tracker detection) is governed by the External Image Confirmation feature (3.10), not by this epic. This epic concerns only locally-available attachment data.

## 4. Glossary

| Term | Definition |
|---|---|
| **Attachment** | A discrete file carried by an email message, identified by MIME type, filename, size, and disposition (inline or regular). |
| **Inline image** | An image attachment referenced from within the HTML message body (typically via a content-ID reference), intended by the sender to appear at a specific location in the message text. |
| **Regular attachment** | An attachment that is not referenced from the message body — it appears only in the attachment list and (if eligible) in the media gallery. |
| **Media gallery** | A horizontally scrollable strip of thumbnail previews displayed below the message body, showing eligible image, PDF, video, and audio attachments. |
| **Thumbnail** | A small, scaled-down rendering of an attachment used in the media gallery. For images: a proportionally scaled version. For PDFs: a render of the first page. For audio: a styled play-button card. |
| **Embedded message** | An attachment of type `message/rfc822` (an `.eml` file) — a complete email message nested inside the parent message. |
| **Barcode decoding** | The process of scanning an image attachment for machine-readable codes (QR, Code 128, EAN, UPC, Aztec, etc.) and extracting the encoded text. |
| **Content-ID (CID)** | A MIME header that assigns a unique identifier to an attachment, allowing the HTML body to reference it with a `cid:` URI so the image appears inline at that location. |

## 5. Personas

- **P1. The visual scanner.** Receives many image-heavy emails (e-commerce confirmations, newsletters, photo shares). Wants to see images in context without extra clicks.
- **P2. The audio reviewer.** Receives voice memos or podcast clips via email. Wants to listen quickly without opening a media player.
- **P3. The document triager.** Receives invoices, tickets, and contracts as PDF attachments. A thumbnail of the first page helps them identify the right document at a glance.
- **P4. The forwarded-thread reader.** Receives emails with attached `.eml` conversations. Wants to read the attached thread inline rather than saving and opening it separately.
- **P5. The traveller.** Receives boarding passes, event tickets, and payment confirmations containing QR codes or barcodes. Wants the code content extracted automatically so they can act on it (copy a link, see a confirmation number) without a separate scanner.
- **P6. The minimalist.** Finds inline previews distracting or resource-heavy. Wants to disable some or all preview types while keeping basic attachment listing intact.

## 6. User Stories

### 6.1 Inline images in the message body

> **US-1.** As a visual scanner, when I open a message that contains embedded images (placed by the sender at specific locations in the message text), I want those images to appear at the positions the sender intended, so that the message reads naturally as a visual document.

> **US-2.** As any user, I want a global setting to control whether embedded images are shown automatically or only on demand, so that I can choose between convenience and bandwidth/privacy conservation.

> **US-3.** As any user, when embedded images are not shown automatically, I want a per-message control to reveal them, so that I can selectively show images for messages I trust.

> **US-4.** As any user, I want embedded images to be scaled to fit the message viewport width, so that oversized images do not break the layout or require horizontal scrolling.

> **US-5.** As any user, I want image orientation metadata to be respected, so that photos taken in portrait or landscape display correctly regardless of how the file was encoded.

### 6.2 Image thumbnails in the media gallery

> **US-6.** As a visual scanner, when a message has image attachments (whether or not they are embedded in the body), I want to see small thumbnails of each image in a gallery strip below the message, so that I can quickly identify what is attached.

> **US-7.** As any user, I want image thumbnails to be generated from the locally downloaded attachment data, not fetched from a remote server, so that no extra network requests are made.

> **US-8.** As any user, I want a master toggle that controls whether the media gallery (thumbnails for all attachment types) is shown, so that I can hide the entire gallery area if I prefer a cleaner reading view.

### 6.3 Audio preview

> **US-9.** As an audio reviewer, when a message has an audio attachment, I want to see a compact player card in the media gallery showing the filename, duration, and a play/stop control, so that I can listen to the clip without leaving the message view.

> **US-10.** As an audio reviewer, when I tap the play control, I want playback to start immediately from the locally downloaded file and I want the card to update with playback progress, so that I have basic transport control inline.

> **US-11.** As any user, I want a toggle to enable or disable the inline audio player independently of other preview types, so that I can suppress audio previews if I find them unnecessary.

### 6.4 PDF preview

> **US-12.** As a document triager, when a message has a PDF attachment, I want to see a thumbnail of the first page in the media gallery, so that I can visually identify the document without opening it.

> **US-13.** As any user, I want a toggle to enable or disable PDF thumbnails independently of other preview types, so that I can suppress them if I find them slow or unnecessary.

### 6.5 Embedded email (EML) preview

> **US-14.** As a forwarded-thread reader, when a message has an attached `.eml` file (an embedded email message), I want the attached message's key headers (sender, recipients, date, subject) and body to be rendered inline below the parent message, so that I can read the forwarded conversation without opening it in a separate viewer.

> **US-15.** As any user, I want the embedded message to be visually separated from the parent message (e.g. by a horizontal rule and a styled header), so that there is no confusion about where the parent message ends and the attached message begins.

> **US-16.** As any user, I want the body of the embedded message to go through the same safety sanitization as any other message (script stripping, remote-image blocking, link confirmation), so that an embedded message cannot bypass the application's security policies.

### 6.6 Barcode and QR code decoding

> **US-17.** As a traveller, when a message contains an image attachment that encodes a barcode or QR code, I want the application to automatically decode the image and display the encoded text below the thumbnail, so that I can read the content without a separate scanning tool.

> **US-18.** As a traveller, when the decoded content is a URI (web link, email address, phone number), I want it displayed as a clickable link, so that I can act on it directly.

> **US-19.** As any user, I want a toggle to enable or disable barcode decoding independently of other preview types, so that I can suppress the decoding if I do not need it or find it distracting.

### 6.7 Gallery-level controls

> **US-20.** As any user, I want save and share controls accessible from the media gallery, so that I can quickly save or forward an attachment I have identified via its preview.

> **US-21.** As any user, when I click on a thumbnail in the media gallery (for types other than audio), I want the attachment to open in the system's default application for that type, so that the inline preview serves as a launching point for deeper inspection.

### 6.8 Performance and resource management

> **US-22.** As any user, I want thumbnails and previews to load asynchronously and not block message display, so that I can start reading the message text immediately even if attachments are still being processed.

> **US-23.** As any user on a slow or metered connection, I want image loading to be throttled, so that large image attachments do not saturate my bandwidth while I am reading.

> **US-24.** As any user, I want generated thumbnails to be cached locally, so that re-opening the same message does not regenerate previews from scratch.

## 7. Functional Requirements

### 7.1 Supported attachment types

- **FR-1.** The application shall provide inline preview for five families of attachment:
  - (a) Images: at minimum JPEG, PNG, GIF (including animated), WebP, BMP, and SVG; optionally HEIC/HEIF and AVIF where the platform supports them.
  - (b) Audio: at minimum MP3, WAV, OGG, M4A, and MIDI.
  - (c) PDF documents.
  - (d) Embedded email messages (`message/rfc822` / `.eml` files).
  - (e) Barcodes and QR codes embedded in image attachments (all common symbologies: QR, Code 128, EAN, UPC, Aztec, Data Matrix, etc.).

### 7.2 Inline images (embedded in message body)

- **FR-2.** When a message's HTML body references an attachment via a content-ID (CID) URI, the application shall render that image at the location specified in the HTML, provided the attachment data is locally available and the user has permitted inline image display.
- **FR-3.** A global boolean setting ("show inline images") shall control whether embedded images are rendered automatically. The default shall be **off** (images not shown until the user explicitly requests them).
- **FR-4.** When the global setting is off, a per-message action ("show images") shall allow the user to reveal embedded images for that specific message.
- **FR-5.** Inline images shall be scaled to fit within the viewport width. No horizontal scrolling shall be required to view an inline image.
- **FR-6.** Image orientation metadata (EXIF rotation/flip) shall be respected when rendering.
- **FR-7.** Animated images (e.g. animated GIF) shall animate when a separate "animate images" setting is enabled; otherwise, only the first frame shall be displayed.
- **FR-8.** SVG images shall be rendered but shall be subject to a maximum file-size cap to prevent denial-of-service from maliciously large vector files.
- **FR-9.** Inline images that are not yet downloaded shall display a placeholder indicating that an image will appear once the data is available.

### 7.3 Media gallery

- **FR-10.** Below the message body (and above the full attachment list), the application shall display a horizontally scrollable media gallery containing thumbnail previews of eligible attachments.
- **FR-11.** A master boolean setting ("show thumbnails") shall control whether the media gallery is displayed. The default shall be **on**.
- **FR-12.** The media gallery shall not appear for messages in the Spam/Junk folder, regardless of settings, as a safety measure.
- **FR-13.** The gallery shall include controls to save all displayed attachments or share them, accessible without opening individual attachments.

### 7.4 Image thumbnails

- **FR-14.** Non-inline image attachments shall appear in the media gallery as proportionally scaled thumbnails.
- **FR-15.** Thumbnails shall be generated locally from the downloaded attachment file; no remote thumbnail service shall be used.
- **FR-16.** Images shall be downsampled to a reasonable display size to prevent excessive memory consumption. A maximum decoded bitmap size cap shall be enforced.

### 7.5 Audio preview

- **FR-17.** Audio attachments shall appear in the media gallery as a player card showing the filename, file size, and a play/stop control.
- **FR-18.** Tapping the play control shall begin playback from the locally downloaded file. Playback progress shall be visually indicated and updated at least once per second.
- **FR-19.** Tapping the control again (or reaching the end of the file) shall stop playback.
- **FR-20.** A boolean setting ("show audio player") shall independently enable or disable the audio preview. The default shall be **on**.
- **FR-21.** Audio preview shall only be available when the master "show thumbnails" setting is also enabled.

### 7.6 PDF preview

- **FR-22.** PDF attachments shall appear in the media gallery as a rendered thumbnail of the first page.
- **FR-23.** The thumbnail shall maintain the aspect ratio of the original page and shall be rendered at a fixed display width (not the full page resolution).
- **FR-24.** A boolean setting ("show PDF thumbnails") shall independently enable or disable the PDF preview. The default shall be **on**.
- **FR-25.** PDF preview shall only be available when the master "show thumbnails" setting is also enabled.

### 7.7 Embedded email (EML) preview

- **FR-26.** When a message has one or more `message/rfc822` attachments, the application shall parse each attached message and render it inline below the parent message body.
- **FR-27.** The rendered preview shall include: sender ("From"), recipients ("To", "Cc"), date, subject, and the full message body.
- **FR-28.** The preview shall be visually distinguished from the parent message by a clear separator (e.g. horizontal rule) and a styled header identifying it as an attached message.
- **FR-29.** The body of the embedded message shall pass through the same HTML sanitization pipeline as any top-level message, including script removal, remote-image blocking, and link-safety checks.
- **FR-30.** Inline images within the embedded message shall be rendered if the embedded message's own CID references are resolvable and the user has permitted inline images.

### 7.8 Barcode and QR code decoding

- **FR-31.** When the media gallery displays an image thumbnail, the application shall attempt to decode any barcode or QR code present in the image.
- **FR-32.** If decoding succeeds, the decoded text shall be displayed below the image thumbnail in the gallery.
- **FR-33.** If the decoded text is a recognized URI scheme (HTTP/HTTPS, `mailto:`, `tel:`, etc.), it shall be rendered as a clickable, visually distinct link.
- **FR-34.** If the decoded text is not a URI, it shall be rendered as styled (bold) plain text.
- **FR-35.** A boolean setting ("show barcode content") shall independently enable or disable barcode decoding. The default shall be **on**.
- **FR-36.** Barcode preview shall only be available when the master "show thumbnails" setting is also enabled.

### 7.9 Settings hierarchy

- **FR-37.** The settings shall form a two-level hierarchy:
  - **Level 1 (master):** "Show thumbnails" — controls whether the media gallery is displayed at all.
  - **Level 2 (per-type):** "Show PDF thumbnails", "Show audio player", "Show barcode content" — each independently controls whether that specific preview type appears within the gallery. These settings have no effect when the master setting is off.
- **FR-38.** The "Show inline images" setting shall be independent of the media-gallery master switch, because it governs rendering within the HTML message body rather than in the gallery.

### 7.10 Interaction with attachment list

- **FR-39.** The media gallery is a complement to, not a replacement for, the traditional attachment list. The full attachment list (showing filename, size, MIME type, and download/open/save actions) shall remain available regardless of gallery settings.
- **FR-40.** A per-message toggle ("show inline attachments") shall allow the user to show or hide inline image attachments within the attachment list, independent of whether they are rendered in the message body.

## 8. Non-Functional Requirements

- **NFR-1. Asynchronous loading.** All preview generation (thumbnail creation, PDF rendering, audio metadata extraction, barcode decoding) shall occur asynchronously and shall not block message body display.
- **NFR-2. Memory safety.** Image decoding shall enforce a maximum decoded bitmap size cap. Images exceeding this cap shall be downsampled rather than rejected, so that a preview is still shown.
- **NFR-3. File-size guardrails.** SVG rendering shall enforce a maximum source-file-size cap to prevent excessive CPU or memory use from adversarial vector content.
- **NFR-4. Caching.** Generated thumbnails and decoded barcode results shall be cached locally so that re-opening a message does not repeat the work.
- **NFR-5. Bandwidth awareness.** On slow or metered connections, image loading shall be throttled (e.g. via concurrency limits) so that attachment previews do not saturate the connection while the user is reading.
- **NFR-6. Graceful degradation.** If a preview cannot be generated (corrupt file, unsupported codec, decoding failure), the attachment shall still appear in the attachment list with its filename and size — the user is never left without a way to access the file.
- **NFR-7. Security.** Embedded-message previews shall apply the same content-security policies as top-level messages. No preview type shall execute scripts, load remote resources beyond what the user has permitted, or bypass link-safety checks.
- **NFR-8. Accessibility.** Thumbnails shall have descriptive alternative text (at minimum the filename). Audio playback controls shall be keyboard-reachable and labelled for screen readers. Barcode-decoded text shall be selectable and copyable.
- **NFR-9. Privacy.** All preview generation shall be performed locally. No attachment data shall be sent to any external service for preview rendering or barcode decoding.

| AC-19 | A message with a corrupt or unreadable attachment still shows that attachment in the attachment list with its filename and size; the gallery gracefully omits the unrenderable preview. |
| AC-20 | All preview settings (master toggle and per-type toggles) persist across application restarts. |

## 10. Open Questions

- **OQ-1. EML preview toggle.** Unlike images, audio, PDF, and barcodes, embedded email (EML) preview does not appear to have its own independent on/off toggle in the current source application. Should it? Or is the assumption that EML rendering is always on when an `.eml` attachment is present?
- **OQ-2. Barcode decoding scope.** The current implementation decodes barcodes from image attachments shown in the media gallery. Should it also attempt decoding of embedded (CID) images that appear inline in the message body, or only gallery images?
- **OQ-3. Multiple barcodes in one image.** If a single image attachment contains multiple barcodes, should all decoded values be shown, or only the first? The current behavior appears to decode only the first match.
- **OQ-4. Audio playback scope.** The current inline player is minimal (play/stop with progress). Should seek, volume control, or playback-speed adjustment be in scope, or are those deferred to the external media player?
- **OQ-5. SVG file-size cap value.** A maximum file-size cap is enforced for SVG rendering. The appropriate cap value for a desktop environment (which typically has more memory than a mobile device) is a design decision to be made at implementation time.
- **OQ-6. Gallery ordering.** When a message has a mix of images, PDFs, and audio, what is the canonical ordering of items in the media gallery? The current behavior appears to follow attachment order as declared in the MIME structure. Should the user be able to reorder or group by type?
- **OQ-7. Interaction between inline images and the gallery.** An image that is both embedded (CID-referenced) in the body and listed as an attachment could appear in both the message body and the media gallery. The current behavior appears to exclude inline images from the gallery. Should this be configurable?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Two distinct display surfaces.** Inline images appear *within* the HTML message body at the location the sender chose (via CID references). All other previews — and non-inline image thumbnails — appear in a separate *media gallery strip* below the body. These are deliberately different surfaces: the HTML body is the sender's layout; the gallery is the application's layout. Mixing them (e.g. inserting a PDF thumbnail into the HTML flow) would break the sender's intended reading order.

- **N-2. Master + per-type toggle hierarchy.** The two-level settings hierarchy (master "show thumbnails" plus per-type toggles) exists because some users want no gallery at all (master off), while others want the gallery but find specific types distracting (e.g. barcode decoding is noisy if every screenshot triggers a scan). The master switch avoids the need to individually disable every type.

- **N-3. Inline images default to off.** The "show inline images" setting defaults to off because embedded images are a vector for tracking pixels and remote-content loading. Users who want automatic display opt in explicitly; the safe default is to show a placeholder and let the user decide per message.

- **N-4. Gallery suppressed in Spam.** The media gallery is suppressed for messages in Spam/Junk folders as a defense-in-depth measure. Even though previews are generated from local data, rendering images from spam increases the attack surface (malformed images, adversarial barcode content) and provides no user value for messages the user has already classified as unwanted.

- **N-5. EML rendered inline, not in the gallery.** Embedded email messages are rendered as full inline content (headers + body) rather than as a thumbnail in the gallery. This is because an email message's value is its text, which cannot be meaningfully represented as a small thumbnail. The inline rendering lets the user read the forwarded conversation without any extra interaction.

- **N-6. Barcode decoded text as actionable content.** Decoded barcode content is displayed as a clickable link when it matches a known URI scheme, and as bold plain text otherwise. This makes the common case (a QR code containing a URL) immediately actionable while keeping non-URL content (serial numbers, plain text) visible but inert.

- **N-7. No remote services for preview generation.** All preview generation — image scaling, PDF page rendering, audio metadata extraction, barcode decoding — is performed locally. This is a privacy-by-design choice consistent with the application's principle that no user data leaves the device without explicit consent.
