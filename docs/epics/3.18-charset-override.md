# Epic: Charset Override

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.18** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Charset override when a message declares an incorrect charset."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email messages carry a declared character encoding (charset) in their content-type headers — for example `text/plain; charset=ISO-8859-1` or `text/html; charset=UTF-8`. When a sender's mail software declares the wrong charset, the recipient sees garbled text: accented characters replaced by question marks, Chinese text decoded as Latin gibberish, or Cyrillic rendered as Western European symbols.

This problem is widespread. Common causes include:

- Legacy or misconfigured mail servers that stamp every message as `US-ASCII` or `ISO-8859-1` regardless of the actual encoding.
- Messages that declare `UTF-8` but actually contain Windows-1252 "smart quotes" and other legacy characters.
- Messages with no charset declaration at all, forcing the client to guess.
- HTML email whose MIME headers say one charset but whose `<meta>` tag says another.
- Messages in East Asian or Cyrillic scripts whose encoding is declared as a Western European charset.

The Charset Override feature provides a two-pronged solution:

1. **Automatic detection and correction**: the application uses heuristic charset detection to identify and silently fix common misdeclarations when a message is first loaded, so that most users never see garbled text.
2. **Manual override**: when automatic detection fails or produces an unsatisfactory result, the user can explicitly select any charset from the system's full repertoire, causing the message body to be re-decoded from the raw bytes with the chosen encoding.

Together, these mechanisms ensure that the user can always read a message, even when the sender's software got the encoding wrong.

## 2. Goals

- **G1.** Automatically detect and correct the most common charset misdeclarations so that the vast majority of incorrectly-encoded messages render correctly without user intervention.
- **G2.** Provide a manual charset override action on any individual message, giving the user full control when automatic detection is insufficient.
- **G3.** Present the full list of system-supported charsets as override choices, plus an "Automatic" option to revert to the application's default detection behavior.
- **G4.** Warn the user when a message declares a charset that the application does not recognize, so they understand why the text may appear garbled and can take corrective action.
- **G5.** Handle gracefully the many real-world charset aliases, misspellings, and non-standard names found in the wild (e.g. `x-unknown`, `8bit`, `NONE`, `latin`, `CHARSET`, names with embedded quotes or spaces).

## 3. Non-Goals

- **NG1.** This epic does **not** define a global default-charset preference. The override is per-message only.
- **NG2.** This epic does **not** cover charset handling for outgoing (composed) messages. Composition charset is governed by the compose subsystem.
- **NG3.** This epic does **not** specify how attachment filenames are decoded when they use RFC 2231 encoded parameters — only the message body text is in scope.
- **NG4.** This epic does **not** define the internal detection algorithm or its tuning — only the user-facing contract (detect automatically; allow manual override; warn on unknown charsets).
- **NG5.** This epic does **not** cover the "alternative view" toggle (plain ↔ HTML). That is a separate feature that happens to share the same re-download mechanism.

## 4. Glossary

| Term | Definition |
|---|---|
| **Charset (character encoding)** | A mapping between byte sequences and characters. Common charsets include UTF-8, ISO-8859-1, Windows-1252, GBK, Shift_JIS, KOI8-R, etc. |
| **Declared charset** | The charset stated in the message's MIME content-type header (e.g. `charset=UTF-8`). |
| **Actual charset** | The charset in which the message body bytes were truly encoded by the sender. May differ from the declared charset. |
| **Meta charset** | A charset declared inside the HTML body itself via a `<meta>` tag, which may differ from or supplement the MIME-level declared charset. |
| **Charset detection** | A heuristic or statistical process that examines the raw byte content to infer the most likely actual charset. |
| **Charset override** | The user's explicit selection of a charset to use when decoding a particular message, bypassing both the declared charset and automatic detection. |
| **Unknown charset** | A charset name declared by a message that the application cannot map to any known encoding. |
| **Charset alias** | An alternate or non-standard name for a charset (e.g. `latin` for ISO-8859-1, `UTF8` for UTF-8, `8bit` for a raw byte stream). |

## 5. Personas

- **P1. The monolingual Western user.** Reads email almost entirely in Latin-script languages. Rarely encounters charset problems but occasionally receives a message with garbled "smart quotes" or accented characters. Expects the application to just fix it.
- **P2. The multilingual user.** Reads email in multiple scripts (e.g. Latin, Cyrillic, Chinese, Japanese). Regularly encounters messages with incorrect or missing charset declarations. Needs to manually override charsets when the automatic detection guesses wrong.
- **P3. The power user / diagnostician.** Wants to understand *why* a message is garbled. May try multiple charsets to identify the correct one. Values the ability to revert to "Automatic" after experimenting.
- **P4. The recipient of legacy mail.** Receives mail from old or misconfigured systems that consistently mis-declare charsets. Needs per-message correction since there is no sender-side fix.

## 6. User Stories

### 6.1 Automatic charset correction

> **US-1.** As any user, when I open a message whose declared charset is incorrect but whose actual encoding is a common one (e.g. the message says ISO-8859-1 but the bytes are valid UTF-8), I want the application to detect this mismatch and render the message in the correct charset automatically, so that I see readable text without having to do anything.

> **US-2.** As any user, when I open an HTML message whose MIME header declares one charset (or no charset) but whose HTML `<meta>` tag declares a different charset, I want the application to reconcile the two declarations intelligently, so that the message renders correctly.

> **US-3.** As any user, when I open a plain-text message with no charset declaration, I want the application to detect the most likely charset from the message bytes and render accordingly, rather than defaulting to a charset that produces garbled output.

> **US-4.** As any user, when a message declares a charset that the application does not recognize (e.g. a misspelling or proprietary name), I want the application to (a) fall back to a reasonable default so the message is still readable, and (b) show me a warning indicating that the declared encoding is unsupported.

> **US-5.** As any user, when a message declares UTF-8 but actually contains Windows-1252 byte sequences (e.g. curly quotes, em-dashes), I want the application to detect and repair this common mismatch so those characters display correctly.

### 6.2 Manual charset override

> **US-6.** As any user, when I see a garbled message, I want to open a per-message menu and select a different charset from a list, so that the message body is re-decoded with the charset I choose.

> **US-7.** As any user, after selecting a manual charset override, I want the message body to be re-downloaded from the server and re-decoded with my chosen charset, so that the override works from the raw bytes — not from an already-garbled intermediate.

> **US-8.** As any user, I want the charset selection list to include an "Automatic" option at the top, so that I can revert to the application's default detection behavior after experimenting with manual overrides.

> **US-9.** As any user, I want the charset selection list to present every charset that the system supports, sorted alphabetically, so that I can find obscure or regional charsets when needed.

> **US-10.** As a power user, I want to be able to try different charsets on the same message repeatedly (selecting one, viewing the result, selecting another) without any limit, so that I can experiment until I find the correct encoding.

### 6.3 Availability and scope

> **US-11.** As any user, I want the charset override action to be available on messages retrieved via any protocol (IMAP, POP3), so that the feature works regardless of my account type.

> **US-12.** As any user, I want the charset override to apply only to the specific message I act on, not to other messages or future messages, so that overriding one message does not cause side effects elsewhere.

> **US-13.** As any user, I want the charset override action to be accessible from the message's extended action menu (the "more" or context menu shown while viewing a message), in the same area as other diagnostic and advanced actions.

### 6.4 Warnings

> **US-14.** As any user, when a message declares a charset that the application cannot recognize at all, I want to see a visible warning (e.g. "Unsupported encoding: <name>") so that I understand why the message may be garbled and know that a manual override might help.

## 7. Functional Requirements

### 7.1 Automatic charset detection and correction

- **FR-1.** When the message body is decoded, the application shall attempt to detect charset mismatches between the declared charset and the actual byte content.
- **FR-2.** If the declared charset is ISO-8859-1 (or empty / US-ASCII) and the byte content is valid UTF-8, the application shall silently re-decode the content as UTF-8.
- **FR-3.** If the declared charset is ISO-8859-1 (or empty) and the byte content is not valid UTF-8, the application shall run a statistical charset detection algorithm and, if a result is obtained with a recognized charset, re-decode accordingly.
- **FR-4.** If the declared charset is UTF-8 and the byte content contains sequences characteristic of Windows-1252 data that was incorrectly round-tripped through UTF-8, the application shall attempt to repair these sequences for plain-text content.
- **FR-5.** For HTML content, if the MIME-level charset is ISO-8859-1 or empty, the application shall examine the HTML `<meta>` tags for a charset declaration and, if found, use it to re-decode the content — subject to validation that the meta-declared charset is consistent with the byte content.
- **FR-6.** The application shall validate meta-charset declarations before applying them: if the meta says UTF-8 but the bytes are not valid UTF-8, the meta declaration shall be ignored.
- **FR-7.** Automatic detection shall handle UTF-16 content, including detecting little-endian byte order when no BOM is present, and detecting cases where UTF-16 is declared but the content is actually UTF-8.
- **FR-8.** Automatic detection and correction shall apply only when no manual override is in effect for the message. When the user has selected a manual override, automatic detection shall be bypassed entirely.

### 7.2 Unknown and non-standard charset handling

- **FR-9.** The application shall maintain a mapping of common non-standard charset aliases to their canonical equivalents (e.g. `UTF8` → UTF-8, `latin` → ISO-8859-1, `8bit` → ISO-8859-1, `x-unknown` → ISO-8859-1, `NONE` → ISO-8859-1, names containing `1252` → Windows-1252, names containing `GB2312` → GB2312).
- **FR-10.** Charset names shall be normalized before lookup: surrounding quotes stripped, trailing content after the first space truncated, Unicode normalization applied, and case-insensitive matching used.
- **FR-11.** When a message declares a charset that cannot be resolved to any known encoding even after alias mapping and normalization, the application shall (a) fall back to ISO-8859-1 for decoding and (b) add a user-visible warning to the message indicating the unsupported charset name.

### 7.3 Manual charset override

- **FR-12.** The application shall provide a "Charset" (or "Encoding") action in the per-message extended action menu.
- **FR-13.** The charset override action shall be available when the message can be re-downloaded from the server (i.e. the message has a server identifier, or is a POP3 message that can be re-fetched).
- **FR-14.** Activating the charset override action shall present a list of all charsets supported by the system, sorted alphabetically by name.
- **FR-15.** The list shall include an "Automatic" entry at the top, which clears any manual override and reverts to the application's default detection behavior.
- **FR-16.** Selecting a charset from the list shall trigger a re-download of the message body from the server, decoding it with the selected charset instead of the declared or detected charset.
- **FR-17.** When a manual override charset is selected, the raw message bytes shall be decoded directly using the chosen charset, bypassing all automatic detection and correction logic.
- **FR-18.** After a charset override is applied, the message's locally stored body, preview text, and detected language shall all be updated to reflect the new decoding.
- **FR-19.** Selecting the "Automatic" entry shall re-download and re-decode the message body using the normal automatic detection logic, effectively undoing any previous manual override.
- **FR-20.** The charset override shall reset the message view state (scroll position, rendered dimensions) so that the newly decoded content is displayed fresh.

### 7.4 Scope and persistence

- **FR-21.** A charset override shall apply to exactly one message. It shall not affect any other message, nor create any lasting charset preference for the sender, folder, or account.
- **FR-22.** The result of a charset override (the re-decoded body) shall be persisted locally, so that the user does not need to re-apply the override every time they view the message — unless the body is re-synced from the server for other reasons (in which case the automatic detection logic would apply again).

## 8. Non-Functional Requirements

- **NFR-1. Transparency of automatic correction.** Automatic charset correction shall be silent — no user confirmation or notification shall be required. The user shall simply see correctly rendered text.
- **NFR-2. Responsiveness.** Selecting a charset override shall initiate a re-download; the message body shall update as soon as the download completes, within the normal network round-trip time. No additional delay beyond the re-download shall be introduced.
- **NFR-3. Completeness of charset list.** The manual override list shall include every charset that the platform supports, not a curated subset. The application shall not filter or hide charsets it considers unlikely.
- **NFR-4. Graceful degradation.** If a selected override charset produces unreadable output, the user shall be able to select a different charset or revert to "Automatic" without any lasting damage to the message.
- **NFR-5. Protocol independence.** The charset override feature shall work identically for messages retrieved over IMAP and POP3.
- **NFR-6. No data loss.** A charset override shall never modify the message on the server. The override is a local re-interpretation of the raw bytes only.
- **NFR-7. Accessibility.** The charset override action and the charset selection list shall be reachable via keyboard and shall have appropriate screen-reader labels.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|
| AC-1 | A message declared as `charset=ISO-8859-1` but encoded in UTF-8 displays correctly without user intervention — accented characters, emoji, and multi-byte sequences all render properly. |
| AC-2 | A message declared as `charset=UTF-8` but containing Windows-1252 "smart quote" byte sequences (e.g. 0x93/0x94 for curly double quotes) renders the quotes correctly in the plain-text view. |
| AC-3 | A message with no charset declaration whose body contains Cyrillic KOI8-R text is rendered readably via automatic detection (not as Latin gibberish). |
| AC-4 | An HTML message whose MIME header says `charset=ISO-8859-1` but whose `<meta>` tag says `charset=windows-1251` renders correctly using the meta-declared charset. |
| AC-5 | A message declaring a nonsense charset name (e.g. `charset=UseInqueCodePage`) falls back to a readable rendering and shows a warning indicating the unsupported encoding name. |
| AC-6 | Opening the charset override menu on a message displays an "Automatic" entry followed by an alphabetically sorted list of all system-supported charsets. |
| AC-7 | Selecting a charset from the override list causes the message body to update and display text decoded with the chosen charset. |
| AC-8 | After manually overriding to a wrong charset (producing garbled text), selecting "Automatic" re-decodes the message using the application's default detection, restoring the original readable rendering. |
| AC-9 | Overriding the charset on one message does not affect the rendering of any other message in the same folder or conversation. |
| AC-10 | The charset override action is available on both IMAP and POP3 messages. |
| AC-11 | A message declared as UTF-16 (without BOM) whose bytes are in little-endian order renders correctly without user intervention. |
| AC-12 | The user can apply multiple different charset overrides to the same message in succession, each time seeing the body re-rendered with the newly selected encoding. |
| AC-13 | Charset names with non-standard casing, embedded quotes, or trailing spaces (e.g. `"utf-8"`, `UTF8`, `windows-1252 `) are resolved to their canonical charsets without error. |

## 10. Open Questions

- **OQ-1. Persistence of manual override across re-sync.** When the user manually overrides a charset and the message body is later re-synced from the server (e.g. due to a folder re-sync or database rebuild), the manual override is lost and automatic detection takes over again. Should the application remember the user's manual charset choice per message and re-apply it on re-sync? The current behavior does not persist the override choice itself — only its result.
- **OQ-2. Indicator for active override.** There is currently no visible indicator on a message that a manual charset override is in effect. Should the UI show a badge, icon, or note when the displayed body was decoded with a user-selected charset rather than the declared or auto-detected one?
- **OQ-3. Per-sender or per-domain charset preference.** Some users regularly receive mail from the same misconfigured sender. Should the application support a "remember this charset for messages from this sender/domain" feature, or is per-message override sufficient?
- **OQ-4. Charset detection for non-Latin scripts.** The automatic detection algorithm uses a native statistical detector. For some language families (e.g. Chinese GB18030 detection on non-Chinese locales), the detector may produce false positives. The current behavior suppresses GB18030 detection results when the user's locale is not Chinese. Is this locale-gating appropriate, or should the user's locale have no influence on detection?
- **OQ-5. Interaction with encrypted messages.** If a message is encrypted (PGP or S/MIME), the charset override action presumably applies to the decrypted body. The current behavior does not explicitly address this case. Should charset override be available for encrypted messages, and if so, at what stage of the decryption pipeline?
- **OQ-6. Scope of re-download.** Applying a charset override re-downloads the entire message body from the server. For very large messages or slow connections, this may be slow. Should the application cache the raw bytes locally to allow charset re-decoding without a network round-trip?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Automatic detection is the primary mechanism; manual override is the escape hatch.** The application invests significant effort in automatic charset detection and correction — checking for UTF-8 validity, examining HTML meta tags, running statistical detection, repairing UTF-8/Windows-1252 confusion — so that most users never need to use the manual override. The manual override exists for the long tail of cases where heuristics fail.

- **N-2. Override bypasses all heuristics.** When the user selects a manual charset, the raw message bytes are decoded directly with that charset and no automatic correction is applied. This is intentional: the user is asserting "I know the correct encoding, trust me." The "Automatic" option restores the full heuristic pipeline.

- **N-3. Re-download rather than re-decode.** A charset override triggers a fresh download of the message body from the server rather than re-decoding the locally cached body. This ensures the override works from the original raw bytes, which may have been transformed during the initial decoding (e.g. converted from ISO-8859-1 bytes to a Java string). Re-downloading is the only way to guarantee a clean re-interpretation.

- **N-4. Broad alias mapping for resilience.** The application maps a wide range of non-standard, misspelled, and shorthand charset names to their canonical equivalents. This is a deliberate choice to maximize interoperability with the messy reality of email charset declarations, where senders routinely declare charsets like `NONE`, `UNKNOWN`, `8bit`, `latin`, or even `CHARSET`. The fallback for truly unresolvable names is ISO-8859-1, chosen because it is a single-byte encoding that can represent any byte value without throwing an error — ensuring the message body is always displayable, even if incorrectly.

- **N-5. HTML meta-charset as a secondary signal.** For HTML messages, the application checks `<meta>` tags only when the MIME-level charset is absent, US-ASCII, or ISO-8859-1 — i.e., when the MIME header provides no useful information. The meta charset is validated against the actual byte content before use (e.g., a meta claiming UTF-8 is rejected if the bytes are not valid UTF-8). This conservative approach avoids cases where an HTML template has a hardcoded `<meta charset="UTF-8">` but the actual content was re-encoded by a mailing list processor into a different charset.

- **N-6. Statistical detection scoped to recognized charsets.** The automatic detection engine returns a charset name and a reliability indicator. The application accepts results for a defined set of common and less-common charsets (Western European, Central European, Cyrillic, CJK, etc.) and rejects or warns on unusual results. This prevents the detector from steering the application toward exotic encodings that are more likely to be false positives than genuine detections.
