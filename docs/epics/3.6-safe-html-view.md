# Epic: Safe HTML View

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.6** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Sanitized HTML rendering with scripts, remote fonts, form actions and unsafe styling stripped."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email messages are one of the most common vectors for phishing, tracking, and remote code execution. Raw HTML email can contain executable scripts, tracking pixels, remote fonts that fingerprint the reader, form elements that harvest credentials, and CSS constructs that leak information or disrupt readability. A competent email client must render messages in a way that preserves the sender's intended formatting while neutralizing every class of embedded threat.

The Safe HTML View is the application's **default message rendering mode**. Every incoming HTML message passes through a sanitization pipeline before it reaches the screen. The pipeline strips or transforms dangerous constructs, normalizes styling for readability, adapts colors for the active theme (including dark mode), and blocks remote content unless the user explicitly allows it. The result is a "reformatted" view that is safe to display without user intervention.

A separate "original" (full HTML) view exists for cases where the sanitized rendering is insufficient; Safe HTML View does not replace or remove that escape hatch, but it is the view users see by default — and the view that protects them.

## 2. Goals

- **G1.** Render every HTML email safely by default, with no user action required, by stripping or neutralizing all executable, tracking, and information-leaking constructs before display.
- **G2.** Preserve legitimate formatting — colors, font choices, text sizes, alignment, lists, tables, inline structure — to the greatest extent possible without re-introducing risk.
- **G3.** Give the user fine-grained control over which categories of styling are preserved (colors, fonts, sizes, alignment, backgrounds) so that they can trade fidelity for uniformity.
- **G4.** Adapt sanitized message colors to the application's active theme (light, dark, or high-contrast) so that messages remain readable regardless of the user's display preferences.
- **G5.** Detect and neutralize tracking pixels, replacing them with a visible indicator so the user knows tracking was attempted.
- **G6.** Provide a clearly discoverable toggle to switch any message from the safe reformatted view to the original HTML view (and back), with optional per-sender and per-domain memory for that choice.
- **G7.** Ensure the rendering environment itself is locked down: no script execution, no cookie storage, no local file access, no form submission — regardless of what the sanitization pipeline may have missed.

## 3. Non-Goals

- **NG1.** This epic does **not** define how remote images are fetched, cached, or proxied — only that they are blocked by default in the safe view and that the user can allow them. Remote-image handling is covered by feature 3.10 (External-image confirmation).
- **NG2.** This epic does **not** define link-click behavior (confirmation dialogs, tracking-parameter stripping, punycode warnings). That is feature 3.12 (Link confirmation & sanitization).
- **NG3.** This epic does **not** define the "original message view" (full unsanitized HTML rendering). That is feature 3.7 (Original / raw message view). This epic only requires that the toggle to that view exists and is discoverable.
- **NG4.** This epic does **not** define tracker-image blocklist management (sourcing, updating, or configuring blocklists). That is feature 3.11 (Tracker-image detection). This epic consumes the blocklist verdict but does not manage the list.
- **NG5.** This epic does **not** prescribe how the sanitized output is rendered on screen (choice of rendering engine, zoom controls, scroll behavior). It specifies only what the rendering engine must **not** be allowed to do and what the sanitized HTML must look like when it arrives at the renderer.
- **NG6.** Plain-text reformatting (feature 3.8) is a separate concern; this epic covers HTML-to-sanitized-HTML, not HTML-to-plain-text.

## 4. Glossary

| Term | Definition |
|---|---|
| **Sanitization pipeline** | The processing stage that transforms raw HTML email into safe, renderable HTML by removing, transforming, or neutralizing dangerous constructs. |
| **Reformatted view** | The default message display mode, which shows the output of the sanitization pipeline. Synonymous with "safe view". |
| **Original view** | An alternative display mode that renders the message's HTML with minimal transformation, bypassing most of the sanitization pipeline. Not covered by this epic except as a toggle target. |
| **Allowlist** | The set of HTML tags, attributes, and CSS properties that the sanitization pipeline permits to pass through. Everything not on the allowlist is removed. |
| **Tracking pixel** | A tiny or invisible image (typically 1x1 pixel) embedded in an email to report back to the sender when the message is opened. |
| **Remote content** | Any resource (image, font, stylesheet) referenced by URL that would require a network fetch to display. |
| **Force-light mode** | A per-message override that renders the sanitized message against a light background even when the application is in a dark theme. |
| **Hidden element** | An HTML element whose styling sets it to invisible (`display: none`, `visibility: hidden`, or equivalent). Some senders use hidden elements to embed tracking content or preheader text. |
| **Inline image** | An image embedded directly in the message body (typically via a `cid:` reference to an attached file), as opposed to a remote image fetched over the network. |
| **Viewport** | The logical display area in which the sanitized HTML is rendered. The sanitization pipeline normalizes viewport constraints to ensure the user can always zoom and scroll. |

## 5. Personas

- **P1. The cautious reader.** Uses the default settings; expects the application to protect them from phishing and tracking without any manual configuration. Never switches to the original view.
- **P2. The visual fidelity user.** Wants messages to look "as the sender intended" — prefers background colors, custom fonts, and branded layouts to be preserved. Enables all optional styling categories but still relies on script and form stripping.
- **P3. The privacy maximalist.** Disables text colors, background colors, custom fonts, and font sizes so that every message looks the same — uniform plain styling. Wants tracking pixels visibly flagged.
- **P4. The dark-mode user.** Runs the application in a dark or high-contrast theme and expects message colors to be adapted so that dark text on a dark background never occurs.
- **P5. The power user.** Occasionally needs the original HTML for messages that render poorly after sanitization (e.g. complex newsletters). Uses the per-sender "always show original" shortcut to avoid repeated toggling.

## 6. User Stories

### 6.1 Default safe rendering

> **US-1.** As a cautious reader, when I open any HTML message, I want it to be rendered through the sanitization pipeline by default, so that I am protected without having to take any action.

> **US-2.** As any user, when I view a sanitized message, I want it to look reasonably close to the sender's intended layout — preserving headings, paragraphs, lists, bold/italic, text color, and basic structure — so that the message remains legible and useful.

> **US-3.** As any user, I want the sanitization to happen before the message reaches the rendering engine, so that even if the renderer has vulnerabilities, dangerous content never arrives at it.

### 6.2 Script and executable content removal

> **US-4.** As any user, I want all script elements, event-handler attributes, and script-bearing URL protocols to be completely removed from the message before rendering, so that no code can execute in my mail client.

> **US-5.** As any user, I want the rendering engine itself to have scripting disabled as a defense-in-depth measure, so that any construct that survives sanitization still cannot execute.

### 6.3 Form neutralization

> **US-6.** As any user, I want form elements (forms, buttons, text areas, select menus) to be removed or rendered inert, so that I cannot accidentally submit data to an attacker's server from within a message.

> **US-7.** As any user, I want checkbox and radio-button inputs to be rendered as visual indicators only (preserving their checked state for display) but to be non-interactive and unable to submit data, so that styled surveys and receipts remain readable without being actionable.

### 6.4 Remote font blocking

> **US-8.** As a privacy-conscious user, I want all references to remote fonts — whether via external stylesheets, `@font-face` rules, or `@import` directives — to be stripped, so that no network request is made to load a font and my IP address is not leaked to a font-hosting service.

> **US-9.** As a visual fidelity user, I want the sanitizer to preserve `font-family` declarations that reference locally available fonts, so that the sender's font choice is honored when the font is already installed on my system.

### 6.5 CSS and style sanitization

> **US-10.** As any user, I want only a defined set of safe CSS properties to be preserved in inline styles, and all other CSS properties to be silently removed, so that no CSS construct can leak information, overlay content, or disrupt the UI.

> **US-11.** As any user, I want external stylesheets (linked via `<link>` tags) to be stripped entirely, so that no remote CSS is fetched.

> **US-12.** As a visual fidelity user, I want embedded style blocks (in-document CSS rules) to be parsed, validated property by property against the same allowlist used for inline styles, and then applied to matching elements, so that well-structured HTML emails retain their class-based styling.

> **US-13.** As any user, I want `!important` flags on CSS properties to be stripped, so that message styling cannot override the application's own UI styling.

### 6.6 Selective styling controls

> **US-14.** As any user, I want independent on/off preferences for each of the following styling categories, so that I can choose exactly how much of the sender's formatting I see:
> - Text colors
> - Background colors
> - Font sizes
> - Font families
> - Text alignment

> **US-15.** As a privacy maximalist, when I disable all optional styling categories, I want every message to render with the application's default font, size, color, and alignment, so that all messages look uniform and no sender-controlled styling is visible.

> **US-16.** As a visual fidelity user, when I enable all optional styling categories, I want the sanitized view to preserve as much of the sender's visual design as safety allows.

### 6.7 Dark-mode color adaptation

> **US-17.** As a dark-mode user, when I view a sanitized message, I want the pipeline to adjust text and background colors for adequate contrast against my dark background, so that I never see dark text on a dark background or light text on a light background.

> **US-18.** As a dark-mode user, I want a per-message "force light background" toggle that renders the sanitized message against a white background even in dark mode, so that I can see color-dependent content (charts, branded layouts) as the sender intended.

> **US-19.** As any user in a high-contrast or monochrome theme, I want background colors from the message to be suppressed automatically, so that the high-contrast experience is not broken by sender styling.

### 6.8 Tracking pixel detection and replacement

> **US-20.** As any user, I want the sanitization pipeline to detect tracking pixels — images that are invisible, extremely small (a few pixels or fewer), or hosted on known tracking domains — and replace them with a visible indicator icon, so that I know when a sender is attempting to track whether I opened the message.

> **US-21.** As any user, I want a preference to disable automatic tracking-pixel detection, so that if I do not want the detection behavior (or it causes false positives), I can turn it off.

### 6.9 Image handling in the safe view

> **US-22.** As any user, I want remote images to be blocked by default in the safe view, with optional placeholders shown in their place, so that no network request is made without my consent.

> **US-23.** As any user, I want inline (embedded) images to be displayable via a preference, so that I can see images that are part of the message body without allowing remote fetches.

> **US-24.** As any user, I want images whose declared dimensions are unreasonably small (a few pixels or less on either axis) to be removed entirely as likely tracking constructs, so that invisible images do not waste space or trigger network requests.

> **US-25.** As any user, I want CSS background images to be converted to regular inline images and then subjected to the same blocking and tracking rules as any other image, so that background-image tricks cannot bypass image blocking.

### 6.10 Structural HTML transformations

> **US-26.** As any user, I want HTML tables to be transformed into a linear block layout in the reformatted view, so that messages designed with table-based layouts reflow correctly in narrow or variable-width display areas.

> **US-27.** As any user, I want deprecated or non-standard formatting elements (e.g. `<font>`, `<center>`) to be converted to their modern equivalents with appropriate styling, so that older emails still render correctly.

### 6.11 Hidden element handling

> **US-28.** As a privacy maximalist, I want a preference that reveals elements the sender marked as hidden (`display: none` or equivalent), so that I can inspect content that was intended to be invisible (preheader text, hidden tracking markup, or concealed phishing content).

> **US-29.** As any user, when hidden-element revelation is disabled (the default), I want hidden elements to remain hidden, so that the message looks as the sender laid it out.

### 6.12 Size limits and truncation

> **US-30.** As any user, I want the sanitization pipeline to truncate excessively large HTML messages at a reasonable threshold, so that a maliciously or accidentally oversized message cannot consume unbounded memory or freeze the display.

> **US-31.** As any user, when a message is truncated, I want a visible indication that not all content is shown, so that I know the message continues beyond what I see.

### 6.13 Viewport normalization

> **US-32.** As any user, I want the sanitization pipeline to normalize or override viewport restrictions embedded in the message HTML (such as disabling user zoom or locking scale), so that I can always zoom and scroll freely regardless of what the sender specified.

### 6.14 Toggle to original view

> **US-33.** As any user, I want a clearly labeled button on each message to switch between the safe reformatted view and the original HTML view, so that I can fall back to the full rendering when the sanitized version is insufficient.

> **US-34.** As a power user, I want per-sender and per-domain "always show original" and "always show images" preferences, so that trusted senders' messages are shown in their full form without repeated toggling.

> **US-35.** As any user, when I switch to the original view, I want a confirmation prompt (dismissible, with a "remember for this sender" option), so that I do not accidentally expose myself to unsanitized content.

### 6.15 Rendering engine lockdown

> **US-36.** As any user, I want the rendering engine to have scripting, local file access, and cookie storage disabled regardless of the sanitization pipeline's output, so that defense in depth protects me even if a sanitization bypass exists.

## 7. Functional Requirements

### 7.1 Sanitization pipeline

- **FR-1.** Every HTML message shall be processed through the sanitization pipeline before being displayed in the reformatted (default) view. No unsanitized HTML shall reach the rendering engine in this mode.
- **FR-2.** The sanitization pipeline shall operate on the HTML document structure (parsed DOM), not on raw text, so that encoding tricks and malformed markup are handled correctly.
- **FR-3.** The pipeline shall apply an allowlist of permitted HTML tags. Tags not on the allowlist shall be removed; their text content shall be preserved where appropriate (e.g. removing `<font>` preserves its inner text).
- **FR-4.** The pipeline shall apply an allowlist of permitted attributes per tag. Attributes not on the allowlist shall be silently removed.

### 7.2 Script removal

- **FR-5.** All `<script>` elements shall be removed entirely (tag and contents).
- **FR-6.** All event-handler attributes (`onclick`, `onload`, `onerror`, `onmouseover`, and all other `on*` attributes) shall be removed from every element.
- **FR-7.** All `javascript:` protocol references in any attribute (notably `href`) shall be removed or the containing attribute shall be cleared.
- **FR-8.** The rendering engine shall have script execution disabled as a separate, independent safeguard.

### 7.3 Form neutralization

- **FR-9.** `<form>`, `<textarea>`, `<button>`, and `<select>` elements shall be removed entirely.
- **FR-10.** `<input>` elements may be retained for display purposes only, limited to their `type` and visual-state attributes (e.g. `checked`). No `name`, `value`, `action`, or submission-related attributes shall be preserved.
- **FR-11.** No `action`, `method`, or `formaction` attribute shall survive on any element.

### 7.4 Remote font blocking

- **FR-12.** External stylesheet references (`<link rel="stylesheet">`) shall be removed entirely.
- **FR-13.** `@import` rules within embedded style blocks shall not be processed or followed.
- **FR-14.** `@font-face` rules shall not be processed, ensuring no remote font download occurs.
- **FR-15.** `font-family` declarations referencing locally installed fonts shall be preserved (subject to the user's font-family preference toggle).

### 7.5 CSS property allowlist

- **FR-16.** The pipeline shall maintain an allowlist of CSS properties. Only properties on this list shall be preserved in inline styles and in applied class-based rules. All other CSS properties shall be silently removed.
- **FR-17.** The following CSS property categories shall be allowlisted (each individually toggleable by user preference where noted):
  - **Text color** (`color`) — toggleable; default: on
  - **Background color** (`background-color`, `background`) — toggleable; default: off
  - **Font size** (`font-size`) — toggleable; default: on
  - **Font family** (`font-family`) — toggleable; default: on
  - **Text alignment** (`text-align`) — toggleable; default: on
  - **Font weight and style** (`font-weight`, `font-style`) — always allowed
  - **Text decoration** (`text-decoration`, `text-decoration-line`) — limited to `line-through` and `underline`
  - **Text transform** (`text-transform`) — always allowed
  - **Display and visibility** (`display`, `visibility`) — limited and validated
  - **Margin and padding** (`margin`, `padding` and their directional variants) — allowed with value validation
  - **Border** (left and right borders only) — limited
  - **List style** (`list-style-type`) — allowed
  - **White space** (`white-space`) — limited to `pre` and `pre-wrap`
- **FR-18.** `!important` flags shall be stripped from all CSS property values.

### 7.6 Embedded style block processing

- **FR-19.** When class-based CSS parsing is enabled (default: on), embedded `<style>` blocks shall be parsed, and each rule shall be resolved against the allowlisted properties before being applied to matching elements as inline styles.
- **FR-20.** When class-based CSS parsing is disabled, embedded `<style>` blocks shall be removed and no class-based styling shall be applied.

### 7.7 Dark-mode color adaptation

- **FR-21.** When the application is in a dark theme, the pipeline shall evaluate the luminance of each text color and background color in the sanitized output and adjust them to maintain adequate contrast against the dark background.
- **FR-22.** When a text color is present without a corresponding background color (or vice versa), the pipeline shall infer the missing color to avoid contrast failures.
- **FR-23.** In a monochrome or high-contrast theme, background colors from the message shall be suppressed.
- **FR-24.** A per-message "force light background" toggle shall be available. When enabled in a dark theme, the message shall be rendered against a white background with the original (light-theme) color adjustments.

### 7.8 Tracking pixel handling

- **FR-25.** The pipeline shall identify likely tracking pixels using at least the following heuristics:
  - Images with declared width or height of zero or near-zero (a few pixels or less)
  - Images with a computed surface area below a small threshold
  - Images with CSS that constrains their maximum dimension to one pixel or less
  - Images hosted on domains present in a known-tracker list
  - Images bearing CSS class names associated with known tracking beacons
- **FR-26.** Detected tracking pixels shall be replaced with a small, visible indicator icon and descriptive alt text (e.g. "Tracking image").
- **FR-27.** The original image source of a detected tracking pixel shall be preserved in a non-rendering attribute so that it can be inspected by the user if desired.
- **FR-28.** A user preference shall control whether automatic tracking-pixel detection is enabled (default: on).

### 7.9 Image handling

- **FR-29.** Remote images shall not be loaded in the safe view unless the user has explicitly allowed images for the message, sender, or domain.
- **FR-30.** When remote images are blocked and image placeholders are enabled (default: on), a placeholder shall be shown in place of each blocked image.
- **FR-31.** Inline (embedded) images shall be displayable via a user preference (default: off).
- **FR-32.** CSS `background-image` declarations with URL references shall be converted into regular image elements and subjected to the same blocking and tracking rules.
- **FR-33.** Images with empty source attributes shall be removed.
- **FR-34.** Data-URI images (base64-encoded) shall be allowed through the sanitizer.

### 7.10 Structural transformations

- **FR-35.** HTML tables shall be transformed into a linear block layout so that table-based email designs reflow correctly at any display width.
- **FR-36.** Deprecated formatting elements (`<font>`, `<center>`, `<big>`, and similar) shall be converted to semantically neutral elements with equivalent inline styling.
- **FR-37.** Address-type elements shall be converted to actionable links where appropriate (e.g. geographic links).

### 7.11 Hidden elements

- **FR-38.** By default, elements with `display: none`, `visibility: hidden`, or equivalent styling shall remain hidden in the rendered output.
- **FR-39.** A user preference (default: off) shall reveal hidden elements with a visual distinction (e.g. strikethrough or muted styling), so that users who want to inspect concealed content can do so.

### 7.12 Size limits

- **FR-40.** The sanitization pipeline shall enforce a maximum input size for the reformatted view. Messages exceeding this limit shall be truncated.
- **FR-41.** A separate, larger maximum input size shall apply when the user switches to the original view.
- **FR-42.** When truncation occurs, a visible indicator shall be appended to the rendered output so the user knows the message was cut short.

### 7.13 Viewport normalization

- **FR-43.** The pipeline shall normalize viewport meta tags to ensure user zooming and scrolling are always permitted, regardless of what the message specified.
- **FR-44.** Restrictive viewport attributes (minimum/maximum scale locks, user-scalable=no) shall be removed or overridden.

### 7.14 View toggle

- **FR-45.** Each message shall provide a toggle control to switch between the safe reformatted view and the original HTML view.
- **FR-46.** When switching to the original view, the application shall prompt for confirmation (default: on), with an option to remember the choice for the sender or domain.
- **FR-47.** Per-sender and per-domain preferences for "always show original" and "always show images" shall be stored persistently and applied automatically on future messages from the same sender or domain.

### 7.15 Rendering engine lockdown

- **FR-48.** The rendering engine shall have script execution disabled.
- **FR-49.** The rendering engine shall have local file access disabled.
- **FR-50.** The rendering engine shall have cookie storage disabled.
- **FR-51.** The rendering engine shall have attribution/tracking APIs disabled.

### 7.16 Auto-linking

- **FR-52.** Plain-text URLs and email addresses in the sanitized output shall be automatically converted to clickable links.
- **FR-53.** Auto-linking shall be rate-limited per element to prevent denial-of-service from messages containing thousands of URL-like strings.

## 8. Non-Functional Requirements

- **NFR-1. Performance.** Sanitization of a typical HTML message (under 100 KB) shall complete in well under one second on a modern system, so that message display does not feel sluggish.
- **NFR-2. Bounded resource use.** The pipeline shall enforce size limits and element-count bounds so that a maliciously crafted message cannot cause excessive memory consumption or processing time.
- **NFR-3. Defense in depth.** Rendering-engine restrictions (no scripts, no cookies, no file access) shall be enforced independently of the sanitization pipeline, so that a bypass in one layer does not compromise the user.
- **NFR-4. Determinism.** The same HTML input with the same user preferences shall produce the same sanitized output, so that messages do not change appearance on re-open.
- **NFR-5. Graceful degradation.** When sanitization removes constructs that the sender relied on for layout (e.g. complex table nesting, CSS grid), the result shall still be readable — degraded but not garbled.
- **NFR-6. Accessibility.** Sanitized output shall preserve semantic HTML structure (headings, lists, links, alt text) so that screen readers and keyboard navigation work correctly on the rendered message.
- **NFR-7. Theme consistency.** Color adaptations shall respect the application's active theme (light, dark, high-contrast, monochrome) and shall never produce unreadable contrast ratios.

## 10. Open Questions

- **OQ-1. Allowlist completeness.** The current set of allowed CSS properties was built incrementally. A systematic review should determine whether any safe, commonly used properties are missing (e.g. `line-height`, `word-spacing`, `letter-spacing`) and whether any currently allowed properties have edge-case risks.
- **OQ-2. Mixed content policy.** The rendering engine currently allows mixed HTTP/HTTPS content. This means that if the user enables remote images, images served over plain HTTP will load even if the surrounding context is HTTPS. Whether this should be restricted to HTTPS-only is a security/usability trade-off that has not been explicitly decided.
- **OQ-3. Size-limit tuning.** The reformatted-view truncation threshold and the original-view threshold are set at fixed values (approximately 100 KB and 1 MB respectively), with scaling based on available system memory. Whether these defaults are optimal — and whether the user should be able to configure them — is unresolved.
- **OQ-4. Table-to-block transformation fidelity.** Converting tables to linear block layouts works well for simple newsletter layouts but may garble complex multi-column tables that carry tabular data (not layout). Whether the pipeline should attempt to detect data tables and preserve them as tables is an open design question.
- **OQ-5. `display` and `visibility` property handling.** The current allowlist permits limited values of `display` and `visibility`. The exact set of allowed values and their interaction with the hidden-element-reveal preference deserves explicit documentation to avoid unintended information hiding.
- **OQ-6. Per-element-type attribute allowlists.** The current allowlist permits a broad set of tags with a shared attribute policy derived from a standard "relaxed" safety profile plus extensions. Whether any tags need more restrictive or more permissive attribute sets than the baseline should be reviewed.
- **OQ-7. Auto-linking false positives.** Automatic conversion of URL-like text to clickable links can produce false positives in technical messages (log dumps, code snippets). Whether auto-linking should be suppressible per message or per context is not decided.
- **OQ-8. Algorithmic dark-mode rendering.** The platform's built-in algorithmic darkening feature is enabled alongside the pipeline's own color adjustments when in dark mode. Whether these two mechanisms ever conflict (double-inverting colors) should be tested and documented.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Sanitize before render, not during.** The pipeline transforms the HTML document *before* handing it to the rendering engine, rather than relying on the rendering engine's own security restrictions. This is deliberate: it means the rendered document is structurally safe regardless of rendering-engine bugs or configuration oversights. The rendering-engine lockdown (no scripts, no cookies, no file access) is a second line of defense, not the primary one.

- **N-2. Allowlist, not blocklist.** The pipeline uses an allowlist of safe tags, attributes, and CSS properties rather than a blocklist of dangerous ones. This is the only defensible approach: new attack vectors appear regularly, and a blocklist can never be complete. An allowlist ensures that unknown constructs are removed by default.

- **N-3. Background colors off by default.** The `background-color` preference defaults to off. This is intentional: background colors are the most common source of readability problems (white text on sender's dark background becomes invisible on the user's light background, or vice versa in dark mode). Requiring the user to opt in reduces support requests and improves the out-of-box reading experience.

- **N-4. Inline images off by default.** Even embedded (non-remote) images are off by default. This is a conservative choice motivated by the fact that `cid:` images are still sender-controlled content, and some messages embed very large or very numerous images that degrade performance.

- **N-5. Per-sender memory is opt-in, not automatic.** The application does not automatically learn a user's view-mode preference from behavior. The user must explicitly choose "always show original for this sender" via a prompt or a setting. This avoids the case where a single accidental tap permanently changes the security posture for a sender.

- **N-6. CSS class parsing on by default.** Embedded `<style>` blocks are parsed and applied by default because many modern HTML emails rely entirely on class-based styling. Stripping classes would make most newsletter and transactional emails unreadable. The safety comes from the property-level allowlist, not from ignoring classes entirely.

- **N-7. Tables converted to blocks.** The decision to convert HTML tables into block-level elements is driven by the fact that a large majority of HTML emails use tables for layout, not data. Converting them ensures the message reflows correctly at any display width and avoids horizontal scrolling. The trade-off is that genuine data tables lose their tabular structure in the reformatted view; users who need the table layout can switch to the original view.

- **N-8. Tracking pixel replacement, not removal.** Detected tracking pixels are replaced with a visible icon rather than silently removed. This makes the user aware that tracking was attempted, which supports informed decisions about the sender's trustworthiness. Silent removal would protect privacy equally but would leave the user unaware.
