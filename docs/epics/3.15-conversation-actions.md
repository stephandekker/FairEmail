# Epic: Conversation Actions

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.15** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Reply, reply-all, forward, forward-as-attachment, redirect/bounce, edit-as-new, resend."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

When a user reads a message, the most common next step is to *act on it* — respond to the sender, loop in additional people, pass it along, or re-use the message as the basis for a new composition. These actions form a small but critical family of operations collectively referred to as "conversation actions."

Each action differs in:

- **Who receives** the resulting message (original sender, all parties, a new audience, a mailing list, the bounce address).
- **How the original content is carried** (quoted in-body, attached as a raw file, sent verbatim).
- **Whether thread continuity is preserved** (stays in the same conversation, or starts fresh).
- **What headers are generated** (reply threading headers, resend headers, DSN envelope).

Getting these semantics right is essential to interoperability — email threading, DKIM/SPF validation, and recipient expectations all depend on the correct headers and envelope being produced for each action type.

## 2. Goals

- **G1.** Provide a complete set of response actions covering all standard email workflows: direct reply, group reply, mailing-list reply, forwarding (inline and as attachment), redirecting/bouncing, editing-as-new, and resending.
- **G2.** Make the most common actions (reply, reply-all, forward) immediately accessible with minimal clicks, while keeping less common actions (bounce, resend, edit-as-new) discoverable but not cluttering the primary UI.
- **G3.** Allow the user to configure which action the primary reply button performs, so power users can optimize for their most frequent workflow.
- **G4.** Preserve RFC-compliant threading: replies and forwards that continue a conversation must set appropriate threading headers; actions that create a new message must not falsely link to the original thread.
- **G5.** Handle identity selection intelligently — default to the identity that received the message, with the ability to override before sending.
- **G6.** Respect per-action preferences for quoting, signatures, subject prefixes, and attachment handling without requiring per-message configuration.

## 3. Non-Goals

- **NG1.** This epic does **not** define the compose editor itself (rich text, plain text, formatting toolbar, draft management). Those belong to a separate composition epic.
- **NG2.** This epic does **not** define attachment handling beyond specifying which attachments are carried over by each action type. Inline image management, resize logic, and attachment UI belong elsewhere.
- **NG3.** This epic does **not** define notification-based reply (quick-reply from a desktop notification). That belongs to the notifications epic.
- **NG4.** This epic does **not** define encryption or signing behavior beyond noting that reply can inherit encryption settings from the original. The encryption epic governs details.
- **NG5.** Template/snippet insertion is a separate concern. This epic notes that templates can be invoked *in conjunction with* a reply action, but does not define the template system itself.

## 4. Glossary

| Term | Definition |
|---|---|
| **Reply** | A response directed to the original sender (or the address in the Reply-To header), with the original message quoted and threading headers linking it to the conversation. |
| **Reply-all** | A reply directed to the original sender *and* all other recipients (To + CC), minus the user's own addresses. |
| **Reply-to-list** | A reply directed to the mailing list's posting address (derived from the List-Post header), bypassing the original sender and CC list. |
| **Forward** | A new message carrying the original message's body (quoted) and all attachments, directed to recipients chosen by the user. |
| **Forward-as-attachment** | A forward where the entire original message is attached as a raw `.eml` file rather than quoted inline. |
| **Redirect / Bounce** | Sending a Delivery Status Notification (DSN) to the envelope sender (Return-Path), informing them that the message could not be or was not delivered. |
| **Edit-as-new** | Creating a brand-new draft pre-filled with the content, recipients, and attachments of an existing message, but with no threading link to the original. |
| **Resend** | Re-transmitting an existing message largely verbatim (original headers preserved), per RFC 2822 §3.6.6. |
| **Threading headers** | The `In-Reply-To` and `References` headers that link a message to its parent and conversation. |
| **Identity** | A configured "From" persona (display name, email address, signature) associated with an account. |
| **Bounce address** | The address in the Return-Path header, designating where delivery failures should be reported. |
| **DSN** | Delivery Status Notification — a machine-readable bounce report per RFC 3464. |
| **Subject prefix** | The conventional string prepended to a subject on reply ("Re:") or forward ("Fwd:"). |

## 5. Personas

- **P1. The conversationalist.** Replies frequently to individuals and small groups. Expects reply and reply-all to be one click away. Values correct quoting and threading.
- **P2. The delegator.** Forwards messages regularly to colleagues or external parties. Needs fast forwarding to frequently-used recipients and expects all attachments to come along.
- **P3. The mailing-list participant.** Interacts with discussion lists. Needs reply-to-list to avoid accidentally mailing individuals when intending to address the list.
- **P4. The administrator.** Manages shared mailboxes or support queues. Uses resend to re-transmit messages on behalf of the original sender, and redirect/bounce to reject unwanted mail at the envelope level.
- **P5. The drafter.** Re-uses previously sent messages as starting points for new ones. Uses edit-as-new to create fresh compositions pre-filled from old messages without creating a threaded reply.

## 6. User Stories

### 6.1 Reply

> **US-1.** As a conversationalist, when I reply to a message, I want the "To" field populated with the original sender's reply-to address (or From address if no Reply-To exists), so that my response reaches the right person.

> **US-2.** As any user, when I reply, I want the original message body quoted below my cursor position, so that context is preserved for the recipient.

> **US-3.** As any user, when I reply, I want the subject prefixed with "Re:" (deduplicated if already present), so that threading conventions are maintained.

> **US-4.** As any user, when I reply, I want the correct threading headers set automatically, so that my reply appears in the same conversation thread in all compliant mail clients.

> **US-5.** As any user, when I reply to a message I myself sent (a self-reply), I want the "To" field populated with the original recipients rather than my own address, so that the reply goes to the people I was corresponding with.

> **US-6.** As any user, when I select text in a message before replying, I want only the selected text to be quoted (not the entire message body), so that I can respond to a specific part of the message.

### 6.2 Reply-all

> **US-7.** As a conversationalist, when I reply-all, I want the "To" field set to the original sender and the "CC" field populated with all other original recipients minus my own addresses, so that everyone in the conversation receives my response.

> **US-8.** As any user, I want the reply-all option to be hidden or disabled when there are no additional recipients beyond the sender, so that the UI does not offer a pointless action.

### 6.3 Reply-to-list

> **US-9.** As a mailing-list participant, when a message has a List-Post header, I want a "Reply to list" action that sends my reply to the list address only, so that I address the list without accidentally emailing individuals.

> **US-10.** As any user, I want the reply-to-list action to be visible only when the message actually belongs to a mailing list (has List-Post header), so that irrelevant actions do not clutter the menu.

### 6.4 Forward

> **US-11.** As a delegator, when I forward a message, I want the "To" field empty so I can specify recipients, the subject prefixed with "Fwd:", and the original body quoted below my text, so that recipients see the forwarded content clearly attributed.

> **US-12.** As any user, when I forward a message, I want all of the original message's attachments (both inline and regular) included automatically, so that recipients receive the full context.

> **US-13.** As a delegator, I want a "forward to" shortcut that shows my recently-used forwarding recipients, so that I can forward to frequent contacts in one step without typing their address.

> **US-14.** As any user, I want the option to configure whether a forwarded message remains linked to the original conversation thread or starts a new thread, so that I can control how the forward appears in my sent folder and the recipient's inbox.

### 6.5 Forward-as-attachment

> **US-15.** As any user, I want to forward a message as a complete `.eml` file attachment rather than quoting it inline, so that the recipient receives the original message with all headers intact (useful for spam reporting or forensic purposes).

> **US-16.** As any user, I want the option to forward an entire conversation (all messages in the thread) as attachments in a single message, so that I can share a complete discussion without manually selecting each message.

> **US-17.** As any user, I want to be informed if the raw message must be downloaded from the server before it can be attached, and I want a progress indicator during that download, so that I understand any delay.

### 6.6 Redirect / Bounce (DSN)

> **US-18.** As an administrator, I want to send a hard bounce (DSN) to the envelope sender, so that the sending system receives a standards-compliant delivery failure notification.

> **US-19.** As any user, I want the bounce action to be available only when the message has a Return-Path header and when the feature has been explicitly enabled in settings, so that accidental misuse is prevented.

> **US-20.** As any user, I want to be warned that sending bounces may affect my email provider's sending reputation, so that I understand the consequences before acting.

> **US-21.** As any user, I want the bounce to be suppressed when the Return-Path matches my own address, so that I cannot accidentally create a bounce loop to myself.

### 6.7 Edit-as-new

> **US-22.** As a drafter, when I use edit-as-new on an existing message, I want a fresh draft created with the original's recipients, subject, body, and attachments — but with no threading link to the original and a new message identifier, so that I can re-use old messages as templates without creating false conversation links.

> **US-23.** As any user, when I edit-as-new, I want the "From" identity set to the identity that originally sent the message (if it matches one of my configured identities), so that the re-use is seamless.

> **US-24.** As any user, when I edit-as-new, I want no signature automatically appended (since the body already contains the original content including any original signature), so that I have full control over the content.

### 6.8 Resend

> **US-25.** As an administrator, I want to resend a message with its original headers preserved (including original From, Date, Subject, and Message-ID), so that the message appears to recipients as if it came from the original sender.

> **US-26.** As any user, I want the resend action to be available only when the message headers have been downloaded, and I want a clear visual indicator when headers are not yet available, so that I understand why the action cannot be performed.

> **US-27.** As any user, I want to be warned that resent messages are likely to fail DKIM/SPF/DMARC checks and may be marked as spam by the recipient's server, so that I have realistic expectations.

> **US-28.** As any user, when I resend, I want no signature appended and no subject prefix added, so that the message remains faithful to the original.

> **US-29.** As any user, I want resend to start a new thread (not linked to the original conversation), so that the resent message stands on its own.

### 6.9 Read receipt

> **US-30.** As any user, when a message requests a read receipt (via Disposition-Notification-To header), I want a "Send read receipt" action that transmits an MDN to the requesting address, so that I can acknowledge receipt when appropriate.

> **US-31.** As any user, I want read receipt sending to be manual by default (not automatic), with an option to configure automatic sending, so that my privacy is preserved unless I explicitly opt in.

### 6.10 Reply button configuration

> **US-32.** As a power user, I want to configure what the primary reply button does on single-click (e.g., open a menu, reply to sender, reply-all, forward, or another action), so that my most common action is always one click away.

> **US-33.** As a power user, I want to configure what the primary reply button does on long-press as a secondary action, so that I have two actions immediately accessible.

> **US-34.** As any user, I want the reply button's icon to visually reflect its configured action (reply icon, reply-all icon, forward icon, etc.), so that I can tell at a glance what it will do.

## 7. Functional Requirements

### 7.1 Action availability and triggering

- **FR-1.** The application shall provide the following conversation actions: reply, reply-all, reply-to-list, forward, forward-as-attachment, redirect/bounce, edit-as-new, resend, and send-read-receipt.
- **FR-2.** All actions shall be accessible from a reply/action menu when viewing a message.
- **FR-3.** The primary reply button shall support two configurable trigger gestures: single-click and long-press, each independently mappable to any action or to "show menu."
- **FR-4.** The reply button icon shall change to reflect the configured single-click action.
- **FR-5.** Conditional actions shall only be shown when their preconditions are met:
  - Reply-to-list: message has a List-Post header.
  - Send read receipt: message has a Disposition-Notification-To header.
  - Redirect/bounce: message has a Return-Path header, the Return-Path is not the user's own address, and the feature is enabled in settings.
  - Resend: message headers are available (with a disabled/dimmed state when not yet downloaded).
- **FR-6.** All actions (except forward-as-attachment when the raw message must be fetched) shall be available offline for messages whose content has been downloaded.

### 7.2 Reply and Reply-all

- **FR-7.** Reply shall set the "To" field to the address in the Reply-To header; if absent, the From header.
- **FR-8.** Reply-all shall set "To" to the original sender and "CC" to all original To and CC recipients, excluding any of the user's own configured identity addresses.
- **FR-9.** When replying to a message sent by one of the user's own identities (self-reply), the "To" field shall be set to the original recipients and the "CC" to any original CC recipients — not to the user's own address.
- **FR-10.** The subject shall be prefixed with "Re:" (or a configured alternative prefix). If the subject already carries the prefix, it shall not be duplicated (configurable: allow duplication, or show a reply count such as "Re[3]:").
- **FR-11.** The `In-Reply-To` header shall be set to the Message-ID of the original message; the `References` header shall carry the conversation's full reference chain.
- **FR-12.** The original message body shall be quoted below the user's composition area, preceded by a reply header line (e.g., "On [date], [sender] wrote:"). The format of this header line shall be configurable (separate line, extended format, or custom template with placeholders for sender, recipient, date, subject).
- **FR-13.** If the user has selected specific text in the message before invoking reply, only the selected text shall be quoted.
- **FR-14.** The application shall offer a configurable option to remove signatures (content below "-- " separator) from quoted text.
- **FR-15.** The application shall offer a configurable limit on quote depth (removing deeply nested quotes to prevent quote bloat).
- **FR-16.** The user shall be able to configure whether the composition cursor starts above the quote (top-posting) or below it (bottom-posting).

### 7.3 Reply-to-list

- **FR-17.** Reply-to-list shall set the "To" field to the address extracted from the List-Post header.
- **FR-18.** Reply-to-list shall set the "From" field to the address that received the message (the To address from the original), so that the list recognizes the subscriber.
- **FR-19.** Threading headers shall be set as for a normal reply.

### 7.4 Forward

- **FR-20.** Forward shall present an empty "To" field for the user to fill in.
- **FR-21.** The subject shall be prefixed with "Fwd:" (or a configured alternative prefix), with the same deduplication rules as "Re:".
- **FR-22.** The original message body shall be quoted (same quoting rules as reply).
- **FR-23.** All attachments from the original message (both inline images and regular file attachments) shall be copied to the new draft.
- **FR-24.** The application shall offer a configurable option for whether the forward creates a new conversation thread or remains linked to the original.
- **FR-25.** The application shall maintain a list of recently-used forwarding recipients and offer a "forward to" shortcut that presents this list for one-step forwarding.
- **FR-26.** When forwarding remains linked to the original thread, the `In-Reply-To` and `References` headers shall be set. When starting a new thread, no threading headers shall be set.

### 7.5 Forward-as-attachment

- **FR-27.** Forward-as-attachment shall create a new compose window with the original message attached as a complete `.eml` file (full RFC 2822 content including all headers).
- **FR-28.** The user shall have the choice of attaching a single message or all messages in the conversation.
- **FR-29.** If the raw message content is not available locally, the application shall download it from the server, showing a progress indicator, before attaching.
- **FR-30.** The application shall offer a configurable option to automatically confirm the download without prompting each time.

### 7.6 Redirect / Bounce

- **FR-31.** Redirect/bounce shall generate a Delivery Status Notification (DSN) per RFC 3464 directed to the bounce address (Return-Path header value).
- **FR-32.** The feature shall be gated behind an explicit opt-in setting (e.g., an "experimental features" or "advanced" toggle).
- **FR-33.** The action shall be suppressed when the Return-Path matches any of the user's own identity addresses.
- **FR-34.** Before sending, the user shall be presented with a warning about potential reputation impact.

### 7.7 Edit-as-new

- **FR-35.** Edit-as-new shall create a new draft with: (a) original To/CC/BCC recipients, (b) original subject (no prefix), (c) original body (not quoted — presented as editable content), (d) all original attachments.
- **FR-36.** The new draft shall have a fresh message identifier and no threading headers linking it to the original.
- **FR-37.** The "From" identity shall be set to the identity that originally sent the message, if it matches a configured identity; otherwise the account's default identity.
- **FR-38.** No signature shall be automatically appended.

### 7.8 Resend

- **FR-39.** Resend shall create a draft that preserves the original message's headers (From, To, CC, Subject, Date, Message-ID) as they were.
- **FR-40.** The user may edit the body and attachments before sending, but by default they are transmitted as-is.
- **FR-41.** No subject prefix, signature, or default CC/BCC shall be applied.
- **FR-42.** A new conversation thread identifier shall be generated (the resent message does not link to the original conversation).
- **FR-43.** The action shall be disabled (shown in a dimmed state) when message headers have not yet been downloaded. The user shall be able to trigger a header download from a related menu.
- **FR-44.** Messages addressed to "undisclosed-recipients:" shall not be resendable.
- **FR-45.** A warning shall inform the user that DKIM/SPF/DMARC validation is likely to fail for resent messages.

### 7.9 Read receipt (MDN)

- **FR-46.** Send-read-receipt shall generate a Message Disposition Notification (MDN) per RFC 3798, directed to the address in the Disposition-Notification-To header.
- **FR-47.** Sending read receipts shall be manual (user-initiated) by default.
- **FR-48.** A global preference shall allow the user to opt into automatic receipt sending.
- **FR-49.** The user shall be able to choose between standard and legacy MDN formats.

### 7.10 Identity selection

- **FR-50.** For reply, reply-all, and reply-to-list: the "From" identity shall default to the identity whose address matches a recipient address on the original message. If no match, fall back to the account's default identity.
- **FR-51.** For forward and forward-as-attachment: the "From" identity shall default to the account that holds the message being forwarded.
- **FR-52.** For all actions: the user shall always be able to override the pre-selected identity before sending.

### 7.11 Signature handling

- **FR-53.** Signatures shall be applied by default to reply, reply-all, reply-to-list, and forward actions. Each of these shall have an independent per-action toggle (e.g., "include signature on reply" vs. "include signature on forward").
- **FR-54.** For reply actions, a "signature only on first reply" option shall suppress the signature on follow-up replies within the same thread.
- **FR-55.** Signature placement shall be configurable: before quoted text, after quoted text, or at the very end of the message.
- **FR-56.** Resend and edit-as-new shall never append a signature automatically.

### 7.12 Encryption inheritance

- **FR-57.** When replying to an encrypted message, the reply shall default to the same encryption mode (sign, encrypt, or both) if the user has a matching key/certificate. This shall be controlled by a global preference.
- **FR-58.** Forward, edit-as-new, and resend shall not inherit encryption settings automatically.

### 7.13 Image resize on reply

- **FR-59.** When replying (not forwarding), inline images from the original message may be automatically reduced in resolution to limit message size. This behavior shall be controlled by a configurable option.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Invoking any conversation action shall open the compose window in under one second for messages whose content is already available locally.
- **NFR-2. Standards compliance.** All generated messages shall conform to the relevant RFCs: RFC 5322 (message format), RFC 2822 §3.6.6 (resent headers), RFC 3464 (DSN), RFC 3798 (MDN), RFC 2183 (content disposition for attachments).
- **NFR-3. Interoperability.** Threading headers produced by reply and forward actions shall result in correct thread grouping in all major mail clients (Gmail, Outlook, Thunderbird, Apple Mail).
- **NFR-4. Offline capability.** All actions except forward-as-attachment (when raw content must be fetched) shall work offline, queuing the send operation for when connectivity returns.
- **NFR-5. Data integrity.** Forward-as-attachment shall produce a byte-accurate copy of the original message (no re-encoding, no header stripping, no content modification).
- **NFR-6. Accessibility.** All action buttons and menu items shall have descriptive labels, be reachable via keyboard shortcuts, and be compatible with screen readers.
- **NFR-7. Safety.** Destructive or reputation-affecting actions (bounce, resend) shall be gated behind opt-in settings and shall display explicit warnings before execution.

## 10. Open Questions

- **OQ-1. Alternative prefix localization.** The application offers configurable alternative prefixes for "Re:" and "Fwd:" — are these intended to support non-English locale conventions (e.g., "AW:" in German, "SV:" in Swedish), or are they purely a power-user customization? The current implementation provides a single alternative per prefix type, which may not cover all locale needs.

- **OQ-2. Bounce vs. redirect semantics.** The feature labeled "hard bounce" sends a DSN (delivery failure notification), which is technically different from a "redirect" (re-sending with Resent-* headers to a new recipient). The high-level feature list mentions "redirect/bounce" as if they are synonyms or a single feature. In the implementation, only DSN-style bounce exists; there is no "redirect to a different recipient" action. Should a true redirect (resend to a new address with Resent-To/Resent-From headers) be considered a separate action?

- **OQ-3. Resend header fidelity.** When resending, should the application add Resent-From, Resent-To, Resent-Date, and Resent-Message-ID headers per RFC 2822 §3.6.6, or does it send the message exactly as-is? The implications for DKIM verification differ. The current behavior should be verified against the RFC.

- **OQ-4. Forward-as-attachment for POP3.** The raw message download for forward-as-attachment requires the message to still exist on the server. For POP3 accounts configured to delete after download, this may fail silently. Should the application cache raw messages locally, or clearly communicate the limitation?

- **OQ-5. Reply-all suppression threshold.** Should reply-all be hidden only when the recipient count equals zero (beyond the sender), or should there be a warning when replying-all to very large recipient lists (e.g., 50+ addresses)?

- **OQ-6. Bounce loop detection.** Some servers (notably Outlook.com) respond to bounce messages with their own bounces, creating loops. Beyond suppressing self-addressed bounces, should the application detect and prevent potential bounce storms (e.g., by refusing to bounce a message that is itself a DSN)?

- **OQ-7. Resend and delivery/read receipts.** When resending a message that originally requested a delivery or read receipt, should the receipt go to the original sender or to the resending user? The current behavior may be ambiguous.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Two-gesture reply button.** The primary reply button supports both single-click and long-press, each independently configurable. This avoids the trade-off between "reply is always one click" and "I need a menu to choose between reply and forward." Users who always reply can set single-click to reply; users who vary can set single-click to open the menu. The button icon updates to reflect the single-click binding, providing a visual cue.

- **N-2. Conditional menu items.** Actions with narrow applicability (reply-to-list, send receipt, bounce, resend) are shown only when their preconditions are met. This keeps the common-case menu short (reply, reply-all, forward, edit-as-new) while ensuring advanced actions remain discoverable when relevant.

- **N-3. Experimental gating for dangerous actions.** Bounce and resend are gated behind an opt-in "experimental" or "advanced" setting. Both actions carry reputation risk (bounce) or deliverability risk (resend DKIM failure). Requiring explicit opt-in prevents casual misuse while still serving power users.

- **N-4. Forward thread linkage is configurable.** Some users expect forwards to appear in the same thread as the original (useful for "FYI" forwards within a team). Others expect a clean break. The `forward_new` preference satisfies both camps. The default (linked) matches the behavior of most desktop clients.

- **N-5. Self-reply detection.** When a user replies to their own sent message, the naive behavior (set To = From) would address the reply to themselves. The application detects this case and reverses the logic — setting To to the original recipients. This handles the common "reply to continue a thread I started" workflow without manual address editing.

- **N-6. Signature suppression on follow-up replies.** In long threads, repeatedly appending a signature to every reply creates visual noise. The "signature only on first reply" option suppresses it on subsequent replies within the same thread. This mimics the behavior of several enterprise mail clients and is preferred by users with lengthy signatures.

- **N-7. No signature on edit-as-new and resend.** Edit-as-new copies the original body verbatim (which may already contain a signature). Resend transmits the message as-is. Appending a signature in either case would corrupt the intended content. The application correctly omits it.

- **N-8. Quote depth limiting.** Long email threads accumulate deeply nested quotes ("quote of a quote of a quote"). The quote-limit feature strips excessive nesting to keep reply drafts manageable. This is a space optimization that most users will never notice but that prevents pathological message sizes in long threads.

- **N-9. Fast-forward-to as a workflow accelerator.** The "forward to" menu of recent recipients exists because some users forward many messages per day to a small set of destinations (e.g., a ticketing system, a colleague, a filing address). Rather than requiring them to type or autocomplete the address each time, the application remembers recent forward targets and offers one-step forwarding.
