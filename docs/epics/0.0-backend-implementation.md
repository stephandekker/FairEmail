# Epic: Backend Implementation

> Driven by [docs/architecture.md](../architecture.md). This epic is **not** reverse-engineered from the FairEmail Android source — it is a green-field architectural decision for the Linux desktop application.
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — every other epic that touches mail content, account state, or sync depends on the storage and sync foundation defined here. v1 cannot ship without it.
>
> This document captures **what** the backend must do, not **how** it must be implemented in detail. It describes the contracts, capabilities, and behaviors the storage layer, credential layer, and sync engine must satisfy, and is intended to drive design and implementation work in subsequent stages without prescribing them line-by-line.

---

## 1. Background & Purpose

The Linux desktop port currently has placeholder backends. Account metadata, settings, and account ordering are persisted as JSON files in the user data directory; passwords are written to those files in plain text; and every protocol-touching service (IMAP probe, SMTP probe, inbound test, connection test, folder sync, notification channel) is a mock implementation that simulates success or failure based on hardcoded patterns. There is no message storage of any kind — no database, no `.eml` files, no folder cache, no sync engine.

Building any user-visible behavior beyond the account setup wizard requires replacing this scaffolding with a real backend. The replacement is governed by [docs/architecture.md](../architecture.md), which mandates a hybrid storage model: **SQLite as the primary structured store, with raw `.eml` files on disk as the source of truth for message content**. Credentials move out of plain-text JSON into the OS keychain (`libsecret`). All network I/O moves out of the UI thread into a dedicated sync engine that owns IMAP and SMTP, refreshes OAuth tokens, and writes to SQLite; the UI reads from SQLite and listens for change notifications.

The work is foundational. Without it:

- Accounts can be created in the wizard but not used (no real connection ever happens).
- Folders cannot be enumerated or synchronized.
- Messages cannot be received, stored, read, searched, threaded, flagged, moved, or sent.
- Notifications, offline operation, full-text search, conversation threading, and every other downstream epic has nothing to build on.

This epic does **not** describe new user-visible features. It describes the storage, credential, and sync substrate that all of the user-visible features in epics 1.x–12.x assume.

## 2. Goals

- **G1.** Replace the JSON file stores (accounts, settings, order) with a single SQLite database, in a way that existing wizard flows and tests continue to work without modification.
- **G2.** Move account credentials (passwords, OAuth tokens) out of plain-text JSON into the OS keychain.
- **G3.** Provide a content-addressed `.eml` storage layer that the sync engine writes to and the UI reads from, with deduplication and cleanup.
- **G4.** Implement real IMAP and SMTP connectivity to replace the mock checkers, so that the existing account-setup test-connection flow validates against live servers.
- **G5.** Provide a sync engine that owns all network I/O for an account: connect, authenticate, refresh tokens, fetch headers, fetch bodies, IDLE, send mail. The UI must never block on the network.
- **G6.** Define a schema for messages, folders, and per-mailbox state that supports incremental sync via CONDSTORE/QRESYNC (RFC 7162) from day one.
- **G7.** Provide full-text search over subjects and bodies via SQLite FTS5, with the index maintained automatically as messages arrive and depart.
- **G8.** Provide change notification from the sync engine to the UI, so that arriving mail and changed flags propagate without polling.
- **G9.** Maintain the existing service-trait shape (`AccountStore`, `SettingsStore`, `OrderStore`, `ImapChecker`, `SmtpChecker`, `InboundTester`, `ConnectionTester`, `FolderSyncService`, `NotificationChannelManager`) so that current UI consumers continue to compile against the same interfaces and so that mock implementations remain available for tests.

## 3. Non-Goals

- **NG1.** This epic does **not** implement JMAP (RFC 8620/8621) clients. The schema and data model must be designed so that JMAP can be added later without restructuring, but the actual JMAP client and protocol code are out of scope. (See architecture.md, "Plan for JMAP".)
- **NG2.** This epic does **not** implement Gmail-specific APIs (Gmail API for labels, search) or Microsoft Graph (for modern Outlook accounts). The credential and provider model must accommodate them, but the integrations themselves are separate epics.
- **NG3.** This epic does **not** implement OAuth 2.0 sign-in flows (covered by epic 1.5). Credential *storage* (keychain) is in scope; *acquisition* (the OAuth dance itself) is not.
- **NG4.** This epic does **not** define user-visible offline-operation behavior — the queueing semantics, replay rules, and conflict-resolution policies belong to epic 7.5. This epic provides only the storage foundation that 7.5 builds on.
- **NG5.** This epic does **not** implement IMAP IDLE polling/scheduling policy (covered by epic 7.1). The sync engine must support IDLE; *when* to enter or leave IDLE is decided in 7.1.
- **NG6.** This epic does **not** implement search UI or query language (covered by epics 5.x). It provides the FTS5 index; the search experience is layered on top.
- **NG7.** This epic does **not** define per-folder sync settings or system-folder hiding (covered by epics 2.x). It provides the folder/sync schema; the user-visible policies are layered on top.
- **NG8.** **POP3 message storage** is deferred. The schema must not preclude POP3 (a `protocol` column on accounts and a flag on the inbound connection suffice), but the sync engine's first protocol implementation is IMAP. POP3 sync is a follow-up.
- **NG9.** Multi-device synchronization beyond what IMAP/JMAP themselves provide is out of scope. The application is single-machine; there is no application-level cloud sync of state.
- **NG10.** Database encryption-at-rest (e.g. SQLCipher) is out of scope for this epic. The keychain protects credentials; message-content protection is a future hardening epic.

## 4. Glossary

| Term | Definition |
|---|---|
| **Primary store** | The SQLite database that holds parsed headers, folder state, flags, MODSEQ values, FTS5 index, and pointers to `.eml` files. |
| **Content store** | The on-disk content-addressed tree of `.eml` files. Each file is named by the SHA-256 hash of its raw bytes and sharded into subdirectories. |
| **Sync engine** | The background component that owns each account's network I/O and writes to the primary store. Runs in a separate task/thread from the UI. |
| **Sync session** | A connected, authenticated, ready-to-fetch state with a single mail server for a single account. |
| **MODSEQ** | The "modification sequence" value advertised by IMAP servers supporting CONDSTORE (RFC 7162). Used for incremental sync — "give me everything that changed since MODSEQ N". |
| **CONDSTORE** | IMAP extension for conditional store operations and per-message MODSEQ. |
| **QRESYNC** | IMAP extension layered on CONDSTORE that allows resync of an entire folder by MODSEQ + UID set. |
| **UID** | IMAP unique identifier for a message within a folder. Stable for the lifetime of the folder under normal IMAP semantics. |
| **UIDVALIDITY** | IMAP folder identifier that changes when UIDs are no longer valid (e.g. folder recreated). The sync engine must invalidate cached UIDs on UIDVALIDITY change. |
| **Keychain** | The OS credential store: `libsecret` (Secret Service API) on Linux, Keychain on macOS, DPAPI / Credential Manager on Windows. v1 targets Linux only. |
| **Change notification** | An in-process event emitted by the sync engine and consumed by the UI when relevant primary-store rows change (new message, flag changed, folder added, etc.). |
| **WAL** | SQLite Write-Ahead Logging journal mode, which allows readers and writers to operate concurrently without blocking each other. |
| **FTS5** | SQLite's full-text-search extension, version 5. Used as a virtual table joined to `messages` for search. |
| **EML file** | A single message stored as raw RFC 5322 bytes — the wire format. The on-disk source of truth for content. |
| **Content-addressed** | Stored under a name derived from the cryptographic hash of its bytes. Two identical messages share one file. |
| **Mock implementation** | A trait implementation that simulates I/O without touching the network or disk, used for tests and the current placeholder UI. |
| **Pending operation** | A user action (move, flag, delete, send) that has been accepted by the UI and persisted but not yet committed to the server. |

## 5. Personas

For an infrastructure epic, "personas" are the actors that interact with the backend rather than the backend itself.

- **P1. The end user.** Doesn't know or care what the backend looks like. Cares only that mail arrives, search works, the app is responsive offline, and credentials are not exposed if the disk is read.
- **P2. The UI developer.** Reads from the primary store, writes user actions to it, listens for change notifications. Never opens a network socket. Wants a stable, well-typed API.
- **P3. The sync engine developer.** Owns network I/O. Reads queued operations from the primary store, talks IMAP/SMTP, writes results back. Wants clear contracts about queue ordering, retries, and conflict semantics.
- **P4. The protocol-extension developer.** Adds JMAP, Gmail API, or Microsoft Graph support later. Wants the schema and engine boundaries to make it possible to plug in a new protocol without rewriting storage.
- **P5. The packager.** Ships the application on Linux distributions. Wants the database, content store, and keychain interactions to follow XDG conventions and not surprise system administrators.
- **P6. The migrating user.** Already has accounts created via the wizard and stored as JSON. Expects accounts to keep working after the upgrade — automatic migration to SQLite, automatic credential move to keychain, no manual reconfiguration.

## 6. User Stories

These describe what becomes possible — for end users, the UI, and the sync engine — once the backend is in place. They are organized to support being broken into independently implementable user stories.

### 6.1 Storage foundation

> **US-1.** As a UI developer, I want the application to open a single SQLite database file at startup with WAL mode and foreign keys enabled, so that all subsequent storage operations have a consistent connection.

> **US-2.** As a packager, I want the database file to live under `XDG_DATA_HOME` (e.g. `~/.local/share/fairmail/fairmail.db`), with the path overridable for tests, so that the file follows Linux conventions and tests can use a temporary directory.

> **US-3.** As a developer, I want a schema-migration mechanism that records the current schema version and applies pending migrations on startup, so that schema changes ship safely between releases.

> **US-4.** As a developer, I want the schema to define tables for accounts, identities, folders, messages, message–folder associations (for IMAP labels / Gmail-style multi-folder semantics), pending operations, and per-mailbox sync state (UIDVALIDITY, HIGHESTMODSEQ), with foreign keys and indexes appropriate to the access patterns, so that subsequent epics have a stable substrate.

> **US-5.** As a developer, I want an FTS5 virtual table over message subject and indexed body text, kept in sync with the `messages` table via triggers or explicit upserts, so that search epics can layer on without restructuring.

### 6.2 Migrating existing JSON stores

> **US-6.** As a migrating user, I want my existing `accounts.json`, `settings.json`, and `order.json` to be imported automatically into the SQLite database the first time the new backend runs, with the JSON files preserved on disk but no longer authoritative, so that I do not lose accounts or preferences across the upgrade.

> **US-7.** As a UI developer, I want the existing `AccountStore`, `SettingsStore`, and `OrderStore` services to keep their public traits but switch to a SQLite-backed implementation, so that the wizard, the account list, and other consumers compile and run unchanged.

> **US-8.** As a developer, I want the JSON-backed implementations of those stores to remain available behind a feature flag or as in-memory test doubles, so that unit tests do not require a real database.

### 6.3 Credentials in the OS keychain

> **US-9.** As an end user, I want my mail account passwords and OAuth tokens stored in the system keychain (libsecret on Linux), not in plain text on disk, so that someone reading my data files cannot recover them.

> **US-10.** As a migrating user, I want my existing plain-text credentials in `accounts.json` to be moved into the keychain on the first run of the new backend, with the plain-text values cleared from the database, so that I do not have to re-enter them.

> **US-11.** As an end user, if the keychain is locked or unavailable when I open the app, I want a clear, recoverable error rather than a silent failure or a crash, so that I know to unlock the keychain or report the problem.

> **US-12.** As a developer, I want a `CredentialStore` trait with a libsecret-backed implementation and an in-memory mock for tests, so that account-related tests do not require a running keychain daemon.

### 6.4 EML content store

> **US-13.** As a developer, I want the application to maintain a content-addressed directory tree (e.g. `~/.local/share/fairmail/messages/<aa>/<bb>/<sha256>.eml`) where each `.eml` file is the raw RFC 5322 bytes of one message, so that two identical messages share one file and the filesystem is the source of truth for message content.

> **US-14.** As a developer, I want writes to the content store to be atomic (write-to-temp + rename) and idempotent, so that crashes mid-write do not leave partial files and re-fetching the same message is harmless.

> **US-15.** As a developer, I want a `ContentStore` trait with a filesystem-backed implementation and an in-memory mock, so that sync tests can run without disk.

> **US-16.** As a developer, I want a deletion path that removes a `.eml` file when no `messages` row references it any longer, so that storage does not leak forever.

> **US-17.** As an end user, if my SQLite database is lost or corrupted, I want the application to be able to rebuild the index by re-parsing the `.eml` files, so that my mail content is recoverable from the filesystem alone.

### 6.5 Real IMAP

> **US-18.** As an end user, when I press "Test Connection" in the manual or wizard account screens with IMAP settings, I want the application to actually open a TCP connection to the configured host and port, perform the chosen TLS handshake, authenticate, list folders, and report results — replacing the mock that always returned hardcoded answers.

> **US-19.** As a developer, I want the `ImapChecker`, `InboundTester`, and `ConnectionTester` traits to be implemented against a real IMAP client crate (e.g. `async-imap`), with the existing mocks retained for tests, so that production runs against the network and tests stay fast and offline.

> **US-20.** As a developer, I want the IMAP client to detect CONDSTORE/QRESYNC support and downgrade gracefully when a server lacks them, so that incremental sync works against modern servers and falls back correctly on older ones.

> **US-21.** As a developer, I want the sync engine to fetch full message bytes into the content store, parse headers into the primary store, and link the two by SHA-256, so that the full pipeline (fetch → store → index) is in place.

> **US-22.** As an end user, I want my IMAP folders to enumerate after a successful test, with well-known folder roles (Inbox, Sent, Drafts, Trash, Spam, Archive) detected from server-advertised attributes, so that the wizard's folder-role assignment step has real data to display.

### 6.6 Real SMTP

> **US-23.** As an end user, when I press "Test Connection" on an SMTP identity screen, I want the application to actually open a connection to the SMTP server, perform the chosen TLS handshake, authenticate, query EHLO capabilities (including advertised maximum message size), and report results.

> **US-24.** As a developer, I want the `SmtpChecker` trait to be implemented against a real SMTP client crate (e.g. `lettre`), with the mock retained for tests, so that the wizard's SMTP test is real.

> **US-25.** As an end user, when I send a message, I want the sync engine to deliver it via SMTP using the configured identity's settings and credentials, append the sent message to the IMAP Sent folder, and store it in the local content store with appropriate flags, so that sending works end-to-end.

### 6.7 Sync engine

> **US-26.** As a UI developer, I want a sync engine running as a background task that owns each account's network state, accepts commands ("sync now", "send this draft", "mark these messages read") via a queue persisted in the primary store, and writes outcomes back to the primary store, so that the UI is never blocked on I/O.

> **US-27.** As a developer, I want the sync engine to maintain per-folder `UIDVALIDITY` and `HIGHESTMODSEQ` and to use them on each connection to fetch only what changed since the last session (CONDSTORE/QRESYNC), so that re-sync is cheap and bandwidth-friendly.

> **US-28.** As an end user, I want new mail to appear without me pressing refresh: the sync engine maintains an IMAP IDLE connection per account where supported, falling back to periodic polling otherwise, so that arriving messages surface promptly.

> **US-29.** As a UI developer, I want a change-notification mechanism (e.g. a `glib`-friendly broadcast channel or signal) that the sync engine emits when primary-store rows change and that the UI can subscribe to per-table or per-folder, so that views update in response to background changes.

> **US-30.** As a developer, I want pending operations (move, flag, delete, send) to be persisted in a queue table before the sync engine attempts them, so that an unexpected shutdown does not lose user actions.

> **US-31.** As a developer, I want transient sync failures (network down, timeout, 5xx) to be retried with backoff, and permanent failures (auth rejected, message gone, quota exceeded) to be surfaced to the user via a status mechanism rather than silent retry, so that the user understands when something needs their attention.

### 6.8 Notifications

> **US-32.** As an end user, when new mail arrives, I want a desktop notification via the freedesktop notification protocol (or equivalent), grouped per account and respecting per-account preferences, so that I know mail came in without watching the app.

> **US-33.** As a developer, I want the `NotificationChannelManager` trait to be implemented against a real freedesktop / D-Bus notification client (e.g. `notify-rust`), with the mock retained for tests.

### 6.9 Folder sync service

> **US-34.** As an end user, when I create or rename a folder in the application, I want the change to be reflected on the IMAP server, with the sync engine performing the protocol operation and the UI reflecting the result, so that folder management works end-to-end.

> **US-35.** As a developer, I want `FolderSyncService` to be backed by the sync engine: the trait method enqueues an operation rather than performing it inline, and the result returns when the engine completes the operation.

### 6.10 Observability and operability

> **US-36.** As a developer, I want a per-account "connection log" (already referenced by the existing UI) that captures protocol-level events — connection attempts, TLS results, capabilities advertised, MODSEQ values, errors — and that the user can inspect from a dialog, so that diagnosis of misbehaving servers is possible without external tools.

> **US-37.** As a developer, I want a single way to inspect the database state from the command line (e.g. by opening the SQLite file in any tool), with no encryption-at-rest in this epic, so that diagnosis is straightforward.

> **US-38.** As a packager, I want the database, content store, and any cache directories to follow `XDG_DATA_HOME` and `XDG_CACHE_HOME` conventions and to be relocatable via environment variables, so that distribution-specific layouts are honored.

## 7. Functional Requirements

### 7.1 Database setup

- **FR-1.** The application shall open a single SQLite database file at startup. The default path shall be `$XDG_DATA_HOME/fairmail/fairmail.db` (or `~/.local/share/fairmail/fairmail.db` if `XDG_DATA_HOME` is unset). The path shall be overridable via environment variable for tests.
- **FR-2.** The database connection shall be opened with WAL journal mode (`PRAGMA journal_mode=WAL`), foreign keys enabled (`PRAGMA foreign_keys=ON`), and `synchronous=NORMAL` (or stronger).
- **FR-3.** Schema changes shall be applied via a versioned migration system that records the current schema version (e.g. via PRAGMA `user_version` or a `schema_migrations` table). Migrations shall run automatically on startup before any application code accesses the database.
- **FR-4.** The schema shall include, at minimum, the following tables, with indexes appropriate to the documented access patterns (per-table column lists are subject to design refinement):
    - `accounts` — id (UUID), display name, color, category, protocol, host, port, encryption mode, username, security flags (allow-insecure, DANE, DNSSEC, etc.), poll interval, sync flags, created/modified timestamps. **No password or token columns.**
    - `identities` — id, account_id (FK), email address, display name, SMTP host/port/encryption/username/realm, advanced flags (use-IP-in-EHLO, custom EHLO, login-before-send), max-message-size cache.
    - `folders` — id, account_id (FK), name, full path, attributes bitmask, role (Inbox/Sent/Drafts/Trash/Spam/Archive/Other), sync settings (subscribed, sync enabled), `uidvalidity`, `highestmodseq`, last sync timestamp.
    - `messages` — id, account_id (FK), uid, modseq, message-id (RFC 5322), in-reply-to, references, from/to/cc/bcc (parsed, normalized), subject, date received, date sent, flags bitmask, size, content_hash (SHA-256 of `.eml`), thread id.
    - `message_folders` — message_id (FK), folder_id (FK) — the many-to-many table that supports IMAP labels / Gmail-style multi-folder semantics.
    - `messages_fts` — FTS5 virtual table over subject and body indexed text, content-rowid linked to `messages`.
    - `pending_operations` — id, account_id (FK), kind (move/flag/delete/send/...), payload (JSON or typed columns), state (pending/in-flight/failed), retry count, last error, created timestamp.
    - `sync_state` — per-account/per-folder counters that don't naturally live on `folders`, e.g. last full sync, capability cache.
    - `settings` — application-level key/value preferences (replaces `settings.json`).
    - `account_order` — UUIDs in user-defined display order (replaces `order.json`), either as a column on `accounts` or a separate table.
- **FR-5.** All foreign keys shall be declared with `ON DELETE CASCADE` where the dependent rows have no meaning without the parent (e.g. `message_folders.account_id`), and with `ON DELETE RESTRICT` where deletion would leave the system in an unrecoverable state.

### 7.2 Migration of existing JSON stores

- **FR-6.** On first run with the SQLite backend present, if `accounts.json`, `settings.json`, or `order.json` exists in the previously-used data directory, the application shall import their contents into the corresponding SQLite tables atomically (a single transaction) before the UI is allowed to read.
- **FR-7.** After successful import, each JSON file shall be renamed (e.g. `accounts.json.migrated`) but not deleted, so that the user has a recovery path if the migration is wrong.
- **FR-8.** Plain-text credentials present in the imported `accounts.json` shall be written into the OS keychain as part of the same migration, and the credential field on the new `accounts` row shall be left empty.
- **FR-9.** Migration shall be idempotent: running it again with no JSON files present shall be a no-op; running with already-imported JSON files (suffix `.migrated` present) shall be a no-op.
- **FR-10.** A migration failure (e.g. keychain unavailable) shall abort the migration cleanly, leave the JSON files untouched, and surface a clear error.

### 7.3 Credential store

- **FR-11.** A `CredentialStore` trait shall encapsulate read, write, and delete of secrets keyed by a stable identifier (e.g. account UUID + role suffix such as `"<uuid>:imap-password"`, `"<uuid>:smtp-password"`, `"<uuid>:oauth-refresh-token"`).
- **FR-12.** The default implementation on Linux shall use the Secret Service API via `libsecret` (e.g. via the `secret-service` or `keyring` crate). Secrets shall be stored in a dedicated collection or with attributes that identify them as belonging to FairMail, so that they can be enumerated for cleanup on account delete.
- **FR-13.** When deleting an account, all credentials keyed to that account in the keychain shall be deleted as part of the same transaction; failure to delete from the keychain shall not leave dangling rows in `accounts`.
- **FR-14.** When the keychain is unavailable (locked, daemon not running), the application shall not crash. The error shall propagate to the caller, who shall display a user-readable message (e.g. "Cannot read account password: system keychain is locked.").
- **FR-15.** Credentials shall never be written to logs, error messages, or the connection-log dialog. Log redaction shall be enforced at the trait boundary.

### 7.4 EML content store

- **FR-16.** A `ContentStore` trait shall encapsulate `put(bytes) -> ContentHash`, `get(hash) -> Option<Bytes>`, `delete(hash)`, and `exists(hash) -> bool`.
- **FR-17.** The default implementation shall write each message to a path of the form `<root>/<aa>/<bb>/<full-sha256>.eml`, where `aa` and `bb` are the first two hex bytes of the SHA-256, sharding files into directories of bounded size.
- **FR-18.** Writes shall be atomic: the implementation shall write to a temporary file in the same directory and rename into place, so that a crash mid-write cannot leave a partial `.eml` file at the canonical path.
- **FR-19.** `put` shall be idempotent: writing the same bytes twice shall return the same hash without duplicating storage.
- **FR-20.** `delete` shall remove a file only when the database has confirmed no `messages` row still references it. The deletion sequence shall be: (a) delete the row, (b) check the reference count is zero, (c) delete the file.
- **FR-21.** A "rebuild from content" code path shall be available (operator-invoked, not automatic) that walks the content store and re-parses each file into the database, so that the index can be reconstructed if the database is lost.

### 7.5 Real IMAP

- **FR-22.** The application shall depend on a single Rust IMAP client implementation (e.g. `async-imap`) and shall not maintain a hand-rolled IMAP parser unless an unavoidable gap forces it.
- **FR-23.** The `ImapChecker`, `InboundTester`, and `ConnectionTester` traits shall each have a real implementation backed by the IMAP client. The mock implementations shall remain in the codebase for use as test doubles.
- **FR-24.** The IMAP client shall negotiate TLS per the encryption mode chosen on the account: implicit SSL/TLS, STARTTLS, or none. Insecure-connection acceptance, DANE, DNSSEC, and pinned-certificate behaviors shall honor the per-account flags.
- **FR-25.** On a successful connection, the client shall query server capabilities and persist a capability cache in `sync_state`, including: `IDLE`, `CONDSTORE`, `QRESYNC`, `UTF8=ACCEPT`, advertised max message size where present, and authentication mechanisms.
- **FR-26.** The folder enumeration step (`LIST` / `XLIST`) shall return folders with their attributes and special-use markers (e.g. `\Sent`, `\Drafts`), which are then mapped to the role column on `folders`.
- **FR-27.** Message fetching shall, for each message, fetch the raw RFC 5322 bytes (`BODY[]` or equivalent), write them to the content store, parse headers (using a parser such as `mail-parser`), and insert the resulting `messages` row in a single transaction with the `message_folders` row.
- **FR-28.** Incremental sync shall use CONDSTORE+QRESYNC where advertised: store and pass `MODSEQ`/`UIDVALIDITY`, fetch only changes since the last `HIGHESTMODSEQ`. On `UIDVALIDITY` change, the engine shall invalidate cached UIDs for that folder and re-fetch.
- **FR-29.** IDLE shall be entered when supported and the folder is the currently-watched folder (for v1, the inbox of each account; the policy of which folders to IDLE on is refined in epic 7.1).
- **FR-30.** Every protocol-level event (connect, TLS handshake outcome, login result, list folders count, fetch range, IDLE enter/exit, errors) shall be appended to the per-account connection log.

### 7.6 Real SMTP

- **FR-31.** The application shall depend on a single Rust SMTP client implementation (e.g. `lettre`) and shall not maintain hand-rolled SMTP code unless unavoidable.
- **FR-32.** The `SmtpChecker` trait shall have a real implementation. The mock shall remain available for tests.
- **FR-33.** The SMTP client shall negotiate TLS per the identity's encryption mode (SSL/TLS, STARTTLS, none). Insecure-connection acceptance shall honor the per-identity flag.
- **FR-34.** On the successful test path, the client shall record the server's advertised maximum message size from EHLO and update the identity's `max_message_size_cache`.
- **FR-35.** When sending a real message, the client shall: (a) authenticate, (b) submit the message, (c) on success, append a copy to the IMAP Sent folder via the inbound connection, (d) write the message bytes to the content store and the parsed metadata to `messages`, (e) update or remove the queued send operation in `pending_operations`.
- **FR-36.** The SMTP "login before send" option (FR-54 of epic 1.4) shall be honored: when enabled, the engine shall verify the inbound IMAP login succeeds before attempting submission.

### 7.7 Sync engine

- **FR-37.** The sync engine shall run on a runtime separate from the GTK main loop (e.g. a `tokio` runtime on a dedicated worker thread, or a `glib::MainContext` bound to a background thread). It shall not invoke GTK widgets directly.
- **FR-38.** Per account, the sync engine shall own at most one IMAP connection in IDLE/idle-poll mode and at most one short-lived SMTP connection per send. Connection lifetime, reconnection, and backoff shall be the engine's responsibility, not the UI's.
- **FR-39.** User actions (mark read, move, delete, send) shall be delivered to the engine by inserting rows into `pending_operations`. The engine shall poll/listen for new rows and process them in insertion order per account, batching where the protocol allows (e.g. multiple `STORE` flag updates in one IMAP command).
- **FR-40.** On a transient failure (network error, timeout, 4xx-equivalent), the operation shall be re-queued with exponential backoff (e.g. 5s, 30s, 2m, 10m, 1h, capped). On a permanent failure (auth rejected, message gone, quota), the operation row shall be marked failed and surfaced to the UI status area.
- **FR-41.** A "sync now" command shall be available from the UI per account, which the engine handles by performing one full reconciliation pass against the server.
- **FR-42.** Change notifications shall be emitted to the UI when: a new message lands, an existing message's flags change, a message is moved, a folder is added or removed, an account's sync state changes, or a pending operation transitions to failed. The notification shall name the account and folder (or rows) affected so the UI can scope refreshes.

### 7.8 Search index (FTS5)

- **FR-43.** The `messages_fts` virtual table shall index, at minimum, the `subject` field and a derived `body_text` field (plain-text rendering of the message body) per message.
- **FR-44.** Insert, update, and delete on `messages` shall keep `messages_fts` consistent, either via SQLite triggers or via explicit upserts in the same transaction.
- **FR-45.** This epic does **not** define a search API or query syntax for the UI — only that the index is present and consistent. Search-feature epics consume it.

### 7.9 Notification channel manager

- **FR-46.** The `NotificationChannelManager` trait shall have a real implementation backed by the freedesktop notification protocol (e.g. via `notify-rust` or direct D-Bus). The mock shall remain available for tests.
- **FR-47.** The real implementation shall present per-account notification grouping where the desktop environment supports it, and shall respect per-account notification-enabled and per-folder notification-enabled flags from `accounts` / `folders`.
- **FR-48.** This epic provides only the channel/dispatcher; the user-visible content, click behavior, and reply/quick-action support are the concern of epic 8.1.

### 7.10 Backwards compatibility

- **FR-49.** Existing service traits (`AccountStore`, `SettingsStore`, `OrderStore`, `ImapChecker`, `SmtpChecker`, `InboundTester`, `ConnectionTester`, `FolderSyncService`, `NotificationChannelManager`) shall keep their current public method signatures wherever possible. Where a trait method is genuinely incompatible with a real implementation, the change shall be additive (new methods alongside, with deprecations if needed) and called out in the user-story description that introduces it.
- **FR-50.** Mock implementations of every trait shall continue to compile and pass existing unit tests.

### 7.11 Configuration

- **FR-51.** The database file path, content-store root, and any cache directories shall be configurable via environment variables (e.g. `FAIRMAIL_DATA_DIR`, `FAIRMAIL_CACHE_DIR`) for tests and packaging, defaulting to `$XDG_DATA_HOME/fairmail/` and `$XDG_CACHE_HOME/fairmail/`.
- **FR-52.** Connection timeouts, retry-backoff caps, IDLE-renewal interval, and similar tunables shall live in code constants for v1, with no user-facing knobs unless an existing epic requires one (e.g. polling interval in epic 7.2).

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** The UI shall never block on the network. Any user action that requires server I/O shall return to the event loop within tens of milliseconds; the I/O proceeds asynchronously in the sync engine.
- **NFR-2. Durability.** Once a user action has been accepted by the UI (e.g. a flag toggle), it shall be persisted in `pending_operations` (or directly in `messages`) before the call returns, so that an immediate crash does not lose the action.
- **NFR-3. Concurrency.** SQLite shall be accessed in WAL mode such that a single writer (the sync engine) and multiple readers (UI views) do not contend. Long-running writes shall be broken into smaller transactions.
- **NFR-4. Recoverability.** A corrupt or missing primary store shall not lose user mail content: re-running with the existing content store shall reconstruct the index. A corrupt content-store file shall be detectable (hash mismatch) and replaceable by re-fetch.
- **NFR-5. Credential safety.** Plain-text credentials shall never be written to the database, log files, the connection-log dialog, exported account files (without explicit user opt-in), or any user-readable on-disk artifact other than the OS keychain itself.
- **NFR-6. Test friendliness.** Every storage and protocol component shall have a mock or in-memory implementation that allows the rest of the system to be unit-tested without touching the network, the disk, or the keychain.
- **NFR-7. Migration safety.** The first run of the new backend on a system with existing JSON-format data shall not lose accounts, settings, ordering, or credentials. The original files shall be preserved (with a `.migrated` suffix) for at least one release as a recovery path.
- **NFR-8. Standards compliance.** IMAP, SMTP, and FTS5 usage shall conform to the relevant RFCs and SQLite documentation; vendor-specific quirks shall be opt-in (e.g. Gmail-specific extensions are not enabled by default).
- **NFR-9. Observability.** Every protocol-level operation shall be loggable to the per-account connection log at a verbosity level the user can change for diagnosis.
- **NFR-10. Linux platform conformance.** Database, content store, and cache paths shall follow XDG Base Directory conventions. The keychain integration shall use the Secret Service API rather than a custom credential file.
- **NFR-11. No blocking calls on the UI thread.** No `std::thread::sleep`, synchronous I/O, or synchronous SQLite write longer than a few milliseconds shall execute on the GTK main loop. All longer work shall be marshalled to the sync engine.

## 10. Open Questions

- **OQ-1. Async runtime.** The application currently uses `glib::MainContext` for UI tasks. The sync engine clearly needs heavier async infrastructure; the obvious choices are (a) a `tokio` runtime on a dedicated thread with channels to the main loop, (b) `async-std`, (c) `glib`-only with `glib::MainContext::spawn_blocking`. The IMAP and SMTP crate ecosystem in 2026 strongly favors `tokio`; this should be confirmed and committed to early.
- **OQ-2. Keychain crate.** Two viable options on Linux: the `keyring` crate (cross-platform, abstract) and the `secret-service` crate (Linux-specific, lower-level). `keyring` is friendlier; `secret-service` exposes more control over collections and attributes. A choice needs to be made and recorded.
- **OQ-3. Body-text extraction for FTS.** Indexing `body_text` requires deciding what "body text" means: the plain-text part if present, otherwise HTML stripped to text, otherwise nothing. A strategy and implementation library (e.g. `mail-parser` for parsing, `html2text` for HTML stripping) need to be selected.
- **OQ-4. Schema strictness.** Should the schema commit fully to JMAP-style normalization (separate `email_addresses`, `mailboxes`, `threads` tables) from day one, or start IMAP-shaped (folders + flat headers) and refactor later? The architecture document suggests modeling close to JMAP, but the practical migration cost from a v1 IMAP-shaped schema to a JMAP-shaped one needs to be estimated.
- **OQ-5. Threading.** RFC 5322 `In-Reply-To`/`References` is the simplest correct threading rule, but Gmail and Microsoft return their own thread IDs that disagree with it. Should `thread_id` be derived (deterministic, simple, sometimes wrong on Gmail) or server-supplied where available (correct on Gmail, undefined on plain IMAP)? Threading details belong to epic 3.2 — but the **column** must be defined in this epic.
- **OQ-6. Content-store cleanup cadence.** Calling `delete` on the content store on every message delete creates many small filesystem operations. Should there be a periodic GC pass instead of synchronous file removal? If so, how is "no row references this hash" detected efficiently?
- **OQ-7. Per-account database vs. shared database.** Architecture.md implies a single database file. With multiple accounts, a long-running sync on one account must not block UI reads of another. WAL mode mitigates this, but a single-writer constraint still applies. Confirm WAL is sufficient or revisit one-DB-per-account.
- **OQ-8. Backwards-compatibility window.** How many releases must the JSON-import path remain in place? Removing it later simplifies code but breaks anyone upgrading from a very old version.
- **OQ-9. Test infrastructure for IMAP/SMTP.** Real-server tests require a server. Options: bundle a local Dovecot/Postfix in CI, mock the IMAP/SMTP wire protocol with a small in-process server, or rely on a test account on a public provider. The choice influences how much of the protocol code is realistically testable.
- **OQ-10. POP3 deferral confirmation.** This epic defers POP3 message sync to a follow-up. Epic 1.4 lists POP3 settings as in-scope. Confirm that POP3 wizard fields can be saved (and thus must round-trip through the new schema) even though POP3 message fetching is deferred.

## 11. Design Notes & Rationale

These notes capture the *why* behind decisions in this epic, so that future implementers can judge edge cases without re-deriving them.

- **N-1. Hybrid storage.** The decision to keep `.eml` files on disk separately from the SQLite index is deliberate. The database is fast and queryable but is also the most common thing to corrupt; the filesystem of plain RFC 5322 files is portable, greppable, backupable, and rebuildable. Treating the filesystem as the source of truth for content and the database as the source of truth for state allows recovery in either direction.
- **N-2. Single database, single writer.** A single SQLite file serves all accounts to keep operations atomic across accounts (e.g. moving a message between accounts in a future feature) and to keep backups simple. WAL mode permits concurrent reads from the UI while the sync engine writes; the UI must never write directly.
- **N-3. Mocks remain.** The existing mock implementations of every trait are not deleted. They are valuable both as test doubles and as contract documentation of what each trait must do. Real implementations are added alongside; a feature flag or wiring decision in `main.rs` selects which is used at runtime.
- **N-4. Credentials never in the database.** Putting credentials in the database makes backup, export, and replication footguns dramatic. A backup of the database should be safe to share with a sysadmin for diagnosis. The keychain is the only place plaintext credentials live at rest.
- **N-5. Sync engine as a separate runtime, not a separate process.** Architecture.md mentions process separation as the gold standard; v1 ships with a separate runtime within the same process to avoid the cost of designing IPC. The trait boundaries are drawn so that the engine could be moved to a separate process later without rewriting callers.
- **N-6. CONDSTORE/QRESYNC from day one.** Adding MODSEQ tracking after the fact requires a schema migration and a re-sync of every account. Designing it in from the start is a small cost up-front and a large saving later. Servers that don't advertise CONDSTORE simply use a different fetch path.
- **N-7. JMAP planned, not built.** Architecture.md correctly notes JMAP is a saner protocol and adoption is growing. Building JMAP support without a clear test target (Fastmail) and without IMAP first risks delivering nothing usable. The schema must accommodate JMAP — multi-folder per message, MODSEQ-equivalent state, push semantics — but the JMAP client itself is a follow-up epic.
- **N-8. Linux-only keychain.** v1 targets Linux. The `CredentialStore` trait abstracts the underlying mechanism so that macOS Keychain or Windows DPAPI can be added later without disturbing callers. v1 ships only the `libsecret` implementation.
- **N-9. Body-text indexing precision.** Indexing the rendered text of a message gives users the search results they expect, at the cost of choosing a renderer. The renderer choice is local: it can be replaced later without re-fetching content (the `.eml` file is intact); only the FTS index needs rebuilding.
- **N-10. Migration is one-way.** Once accounts have moved to SQLite + keychain, downgrading to the JSON store is unsupported. The `.migrated` JSON files are kept as a recovery path, not as a fall-back the application reads at runtime.

---

*This is a green-field architecture epic, not a reverse-engineering of the FairEmail Android source. Where the existing Rust services already have a well-defined behavior (e.g. atomic JSON file writes), that behavior is preserved in spirit but reimplemented atop SQLite and the keychain.*
