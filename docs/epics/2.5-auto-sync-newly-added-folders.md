# Epic: Auto-Sync Newly Added Folders

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **2.5** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Auto-sync newly added folders."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[COULD]` — nice to have; ship if time allows, otherwise post-v1.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

When a user creates a new folder on the mail server — whether through the email client itself, another client, a web interface, or a server-side rule — the application discovers the folder during its next folder-list synchronization cycle. By default, a newly discovered user folder does **not** synchronize: the user must manually open the folder's properties and enable synchronization before its messages will be fetched.

For users who routinely create folders (e.g. to organize projects, teams, or clients) this default is an annoyance: every new folder requires a manual step before it becomes useful. The Auto-Sync Newly Added Folders feature provides an opt-in global setting that changes this default. When enabled, any **newly discovered user-created folder** is automatically enrolled in synchronization with notifications enabled, so that the user does not have to touch each folder individually.

The feature is deliberately conservative:

- It is **disabled by default**, because enabling synchronization for every new folder may be unwanted for users with large or shared mailboxes.
- It applies **only to user-created folders** — system folders (Inbox, Sent, Drafts, Trash, Spam, Archive) already have their own hard-coded synchronization defaults.
- It respects the **parent-folder inheritance** mechanism: when a new folder is a child of an existing user folder, it first inherits its parent's sync settings, and then the auto-sync override (if enabled) ensures synchronization and notifications are on.

## 2. Goals

- **G1.** Provide a single, global, opt-in toggle that automatically enables synchronization and notifications for newly discovered user-created folders.
- **G2.** Eliminate the manual step of enabling sync on every new folder for users who want all their folders actively synchronized.
- **G3.** Ensure the feature is safe to enable: it must not affect system folders, must not retroactively change existing folders, and must not apply when the account has no pre-existing user folders (to avoid premature mass-sync on first-time account setup).
- **G4.** Integrate cleanly with the existing parent-folder inheritance mechanism, acting as a final override rather than replacing inheritance.

## 3. Non-Goals

- **NG1.** This epic does **not** define a per-account auto-sync toggle. The setting is global (application-wide). Per-account granularity is out of scope.
- **NG2.** This epic does **not** change the default synchronization behavior for system folders (Inbox, Sent, Drafts, Trash, Spam, Archive). Those defaults are governed by the per-folder sync settings feature (epic 2.3).
- **NG3.** This epic does **not** provide auto-sync for folders that already exist when the setting is first enabled. It applies only to folders discovered **after** the setting is turned on. Retroactive bulk-enable is a separate concern.
- **NG4.** This epic does **not** define the sync window (days to sync, days to keep) for auto-synced folders. Those values come from the folder's type-based defaults or from parent-folder inheritance.
- **NG5.** This epic does **not** cover shared/public IMAP namespace folders. Shared folder synchronization is governed by a separate setting.

## 4. Glossary

| Term | Definition |
|---|---|
| **Folder-list synchronization** | The periodic process by which the application queries the mail server for the current list of folders and reconciles it with the locally known folder list — adding, updating, or removing folders as needed. |
| **User folder** | A folder that is not one of the well-known system roles (Inbox, Sent, Drafts, Trash, Spam, Archive, Outbox). Typically created by the user or by server-side rules. |
| **System folder** | A folder with a recognized role: Inbox, Sent, Drafts, Trash, Spam, Archive, Outbox. System folders have hard-coded synchronization defaults that this feature does not override. |
| **Synchronization** | The process of fetching message headers and bodies from the server for a given folder, keeping the local copy up to date. A folder with synchronization disabled is known to the application but its contents are not actively fetched. |
| **Parent-folder inheritance** | A mechanism whereby a newly discovered child folder copies certain settings (synchronization on/off, download on/off, sync window, keep window, notifications) from its parent folder, if both are user-type folders. |
| **Selectable folder** | A folder that can hold messages (as opposed to a namespace container or hierarchy-only node that exists solely to contain child folders). |
| **Poll mode** | A synchronization mode where the application periodically queries the server for changes, as opposed to push mode (where the server notifies the application of changes in real time). |

## 5. Personas

- **P1. The folder organizer.** Regularly creates new folders to sort projects, clients, or topics. Expects every new folder to "just work" — syncing and notifying — without manual intervention.
- **P2. The shared-mailbox user.** Has access to a large shared IMAP namespace where folders are created frequently by others. Does **not** want every new folder auto-synced, because most are irrelevant. Leaves the setting off.
- **P3. The set-and-forget user.** Has a stable folder structure and rarely creates new folders. The default (off) suits them fine; they may never discover this setting.
- **P4. The multi-device user.** Creates folders from a web client or another device and expects them to start syncing in this application without having to open folder properties.

## 6. User Stories

### 6.1 Enabling the feature

> **US-1.** As any user, I want a clearly labeled toggle in the synchronization settings that controls whether newly discovered user folders are automatically synchronized, so that I can opt in or out of this behavior globally.

> **US-2.** As any user, I want the toggle to be **off by default**, so that enabling synchronization for new folders is an explicit, informed choice.

### 6.2 Automatic synchronization of new folders

> **US-3.** As a folder organizer (P1), when I create a new folder on the server (from any client) and the application discovers it during the next folder-list sync, I want that folder to have synchronization and notifications automatically enabled, so that I do not have to configure each folder manually.

> **US-4.** As a multi-device user (P4), when I create a folder from a web interface and my desktop client discovers it, I want it to start syncing immediately (on the next sync cycle), so that I can begin using it right away without switching to the desktop client's settings.

> **US-5.** As any user, when auto-sync is enabled and a new folder is discovered as a child of an existing user folder, I want the new folder to first inherit its parent's sync settings (sync window, keep window, download preference) and then have synchronization and notifications forced on, so that the folder behaves consistently with its siblings while also being actively synced.

### 6.3 Scope restrictions

> **US-6.** As any user, I want the auto-sync feature to apply **only** to user-created folders, so that system folders (Inbox, Sent, Drafts, Trash, Spam, Archive) continue to use their own well-defined defaults regardless of this setting.

> **US-7.** As any user, I want the auto-sync feature to apply only to folders that are **newly discovered** — not to folders that already existed when the setting was enabled — so that turning the setting on does not trigger a flood of synchronization for long-dormant folders.

> **US-8.** As any user, I want the auto-sync feature to apply only when the account already has at least one existing user folder, so that during initial account setup the application does not mass-enable sync for a large pre-existing folder tree before I have had a chance to review it.

> **US-9.** As any user, I want the auto-sync feature to apply only to **selectable** folders (folders that can hold messages), so that namespace containers and hierarchy-only nodes are not affected.

### 6.4 Interaction with other settings

> **US-10.** As any user, I want the auto-sync setting to be independent of the "synchronize folder list" setting, the "synchronize shared folders" setting, and per-folder sync settings, so that each concern remains separately controllable.

> **US-11.** As any user, when auto-sync enables synchronization for a newly discovered folder, I want the application to immediately queue a sync operation for that folder, so that messages begin appearing without waiting for the next scheduled poll interval.

### 6.5 Manual override

> **US-12.** As any user, after auto-sync has enabled synchronization on a new folder, I want to be able to manually disable synchronization on that folder through the normal folder properties, so that auto-sync is a starting default, not a lock.

## 7. Functional Requirements

### 7.1 Global setting

- **FR-1.** The application shall provide a global boolean setting, presented in the synchronization options area, that controls whether newly discovered user folders are automatically enrolled in synchronization.
- **FR-2.** The setting shall default to **disabled** (off).
- **FR-3.** The setting shall be persisted across application restarts.

### 7.2 Trigger conditions

- **FR-4.** During each folder-list synchronization cycle, for every folder on the server that does not yet exist in the local folder list, the application shall evaluate whether auto-sync applies.
- **FR-5.** Auto-sync shall be applied to a newly discovered folder if and only if **all** of the following conditions are true:
  1. The global auto-sync setting is enabled.
  2. The folder's detected type is **user** (not a system folder role such as Inbox, Sent, Drafts, Trash, Spam, Archive).
  3. The folder is **selectable** (capable of holding messages).
  4. The account already contains at least one existing user-type folder in the local folder list prior to this synchronization cycle.
- **FR-6.** If a folder that would normally be classified as a system role (e.g. Drafts, Sent) is instead classified as a user folder because a system folder of that role already exists for the account, auto-sync shall still apply (it follows the effective type, not the original type).

### 7.3 Applied settings

- **FR-7.** When auto-sync applies to a newly discovered folder, the application shall set:
  - Synchronization: **enabled**
  - Notifications: **enabled**
- **FR-8.** All other folder settings (sync window, keep window, download preference, poll mode, poll interval, classification settings) shall be determined by the standard defaulting and inheritance mechanisms — auto-sync does not override them.

### 7.4 Interaction with parent-folder inheritance

- **FR-9.** Parent-folder inheritance shall be applied **before** the auto-sync override. The order of operations for a newly discovered user folder shall be:
  1. Set type-based defaults (sync window, keep window, download, poll mode, etc.).
  2. Apply provider-specific overrides if applicable.
  3. If the folder has a parent user folder in the local folder list, inherit the parent's synchronization, download, sync window, keep window, and notification settings.
  4. If the auto-sync conditions (FR-5) are met, force synchronization to enabled and notifications to enabled.
- **FR-10.** This ordering means that auto-sync can override a parent's "synchronization disabled" setting, ensuring the new folder syncs even if its parent does not.

### 7.5 Immediate sync

- **FR-11.** When auto-sync enables synchronization on a newly discovered folder, the application shall immediately queue a synchronization operation for that folder, so that message fetching begins without waiting for the next scheduled poll.

### 7.6 Non-retroactivity

- **FR-12.** Enabling the auto-sync setting shall not modify the synchronization state of any folder that already exists in the local folder list. It affects only folders discovered in subsequent folder-list synchronization cycles.

### 7.7 Protocol scope

- **FR-13.** Auto-sync shall apply only to accounts using a protocol that supports server-side folders (i.e. IMAP or equivalent). Accounts using protocols without a server-side folder concept (e.g. POP3) are unaffected.

### 7.8 Manual override

- **FR-14.** After auto-sync has applied to a folder, the user shall be able to change any of the folder's settings (including disabling synchronization) through the normal folder-properties interface. Auto-sync does not prevent or re-apply changes.

## 8. Non-Functional Requirements

- **NFR-1. Performance.** The auto-sync evaluation adds negligible overhead to the folder-list synchronization cycle — it is a single conditional check per newly discovered folder.
- **NFR-2. Safety.** The feature shall never enable synchronization for system folders, for non-selectable folders, or for folders in accounts that have no pre-existing user folders. These guards prevent accidental mass-synchronization during initial account setup.
- **NFR-3. Idempotency.** If folder-list synchronization runs multiple times and the folder already exists locally, the auto-sync logic shall not re-evaluate or re-apply. It applies only at the moment of initial folder creation.
- **NFR-4. Discoverability.** The setting shall be located alongside related folder-synchronization settings (e.g. "synchronize folder list", "synchronize shared folders") so that users can find it in a logical place.
- **NFR-5. Locality.** The setting is a local user preference. It shall not be transmitted to the server, shall not affect other clients, and shall not introduce any additional network traffic beyond the synchronization it triggers.

## 10. Open Questions

- **OQ-1. Per-account granularity.** The current implementation is a single global toggle. Should a per-account override be offered (e.g. "auto-sync new folders for this account only")? Users with a mix of personal and shared mailboxes might want auto-sync on one but not the other. This is deferred to a future enhancement.
- **OQ-2. Notification default.** Auto-sync currently enables both synchronization **and** notifications. Should notifications be a separate choice, or should the user be able to auto-sync without auto-notify? The source application pairs them; whether this is the ideal default is worth revisiting.
- **OQ-3. First-run guard rationale.** The guard that suppresses auto-sync when no user folders yet exist for the account prevents mass-sync on initial setup. However, it also means the **first** user folder discovered for an account will never be auto-synced (because at the moment it is discovered, the "at least one user folder" condition is not yet met). This is an intentional conservatism, but its subtlety may confuse users who enable the setting and then add a fresh account. The behavior should be documented or reconsidered.
- **OQ-4. Shared namespace interaction.** If the "synchronize shared folders" setting is also enabled, should auto-sync apply to newly discovered shared-namespace user folders, or only to personal-namespace user folders? The source application applies the same logic to both namespaces, but the desirability of auto-syncing shared folders is less clear.
- **OQ-5. Retroactive bulk-enable.** Users who enable the setting after having many unsynchronized user folders may expect it to apply retroactively. Should a companion "enable sync on all existing user folders" action be offered? This is out of scope for this epic but is a natural companion feature.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Disabled by default.** The feature defaults to off because many IMAP accounts have large numbers of user folders (labels, filters, shared namespaces) and enabling synchronization for all of them would consume bandwidth, storage, and battery. The user must consciously opt in.
- **N-2. User folders only.** System folders (Inbox, Sent, Drafts, Trash, Spam, Archive) have carefully tuned defaults — some sync, some don't; some download bodies, some don't; some notify, some don't. Auto-sync must not override these curated defaults. It applies only to user folders, whose out-of-the-box default is "do not synchronize".
- **N-3. Inheritance then override.** The ordering — type defaults, then provider specials, then parent inheritance, then auto-sync — is deliberate. Parent inheritance gives the folder sensible sync-window and download settings that match its siblings. Auto-sync then forces synchronization and notifications on, regardless of what the parent's synchronization state was. This means a folder nested under a non-syncing parent will still sync if auto-sync is enabled, which matches user intent ("I want all new folders to sync").
- **N-4. Notifications paired with sync.** When auto-sync turns on synchronization, it also turns on notifications. The reasoning is that a user who wants folders auto-synced likely also wants to know when new messages arrive in them. If they don't, they can disable notifications per-folder after the fact. Pairing them avoids the silent-accumulation problem where a folder syncs but the user never notices new mail.
- **N-5. "At least one user folder" guard.** During initial account setup, the application discovers all of the account's folders at once. Without the guard, enabling auto-sync would immediately enable synchronization for potentially hundreds of user folders on the first sync — an expensive and likely unwanted operation. The guard ensures that auto-sync only kicks in for folders discovered **after** the initial folder tree has been established.
- **N-6. Global, not per-account.** The setting is a single global toggle rather than per-account. This keeps the settings surface simple. Users who need per-account control can achieve an approximation by manually disabling sync on unwanted folders after they appear, or by not enabling the setting and manually opting in per folder.
