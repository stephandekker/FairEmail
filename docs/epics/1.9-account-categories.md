# Epic: Account Categories

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.9** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Group accounts under user-defined category labels in the navigation drawer."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

A user who manages several email accounts — personal, work, side-projects, family — quickly finds that a flat list of accounts in the navigation pane becomes unwieldy. Scrolling past eight or twelve accounts to reach the right one is slow and error-prone, and the lack of structure makes it hard to form a mental map of "which accounts belong together."

Account Categories solve this by letting the user assign a free-form label (a *category*) to each account. The navigation pane, the unified inbox, and identity-selection surfaces can then group items by category, turning the flat list into a shallow hierarchy that mirrors the user's own mental model — for example "Work", "Personal", "Newsletters", or "Shared".

Categories are intentionally lightweight: there is no separate category-management screen, no rigid taxonomy, and no predefined set. A category exists as long as at least one account carries that label; it vanishes when the last account is re-labelled or cleared. This keeps the feature zero-overhead for users who do not need it, while giving power users a meaningful organisational tool.

## 2. Goals

- **G1.** Allow the user to assign a free-form text label (category) to any account.
- **G2.** Group accounts by category in the navigation pane, so that related accounts appear together under a shared heading.
- **G3.** Optionally group unified-inbox entries (per-category unified inboxes) in the navigation pane, so each category has its own merged inbox entry.
- **G4.** Optionally group messages in the unified inbox by category, so the user can visually distinguish which category a block of messages belongs to.
- **G5.** Provide autocomplete when assigning categories, drawing from categories already in use, to encourage consistent naming.
- **G6.** Keep the feature zero-cost for users who never assign categories — no extra UI, no changed behaviour, no configuration required.

## 3. Non-Goals

- **NG1.** This epic does **not** define a dedicated category-management screen (create / rename / reorder / delete categories as first-class entities). Categories are an emergent property of account labels.
- **NG2.** This epic does **not** provide nested or hierarchical categories (categories within categories). The grouping is a single level deep.
- **NG3.** This epic does **not** change synchronisation, notification, or polling behaviour based on category. Categories are a display-and-organisation concern only.
- **NG4.** This epic does **not** provide per-category notification rules or per-category sync schedules. Those remain per-account and per-folder.
- **NG5.** This epic does **not** define category-based filter rules for messages. Categories group accounts, not messages.

## 4. Glossary

| Term | Definition |
|---|---|
| **Account** | A configured connection to a single mail provider, with credentials, server settings, and its own folder tree. |
| **Category** | A user-defined text label assigned to an account, used to group accounts visually in the navigation pane and optionally in the unified inbox. An account may have zero or one category. |
| **Uncategorised account** | An account with no category assigned. Uncategorised accounts are displayed without a grouping header and sort before or alongside categorised accounts. |
| **Navigation pane** | The primary sidebar listing accounts, folders, and unified-inbox entries. |
| **Unified inbox** | The cross-account merged inbox view (see epic 3.1). |
| **Per-category unified inbox** | An optional navigation entry that merges only the inboxes of accounts sharing the same category, giving each category its own unified stream. |
| **Identity** | A "From" persona (alias, display name, signature) associated with an account. |

## 5. Personas

- **P1. The minimalist.** Has one or two accounts and no need for grouping. Never assigns a category; the feature is invisible.
- **P2. The multi-role user.** Has four to six accounts split across work, personal, and perhaps a volunteer or side-project context. Assigns two or three categories to bring order to the navigation pane.
- **P3. The heavy organiser.** Has ten or more accounts (shared mailboxes, aliases, team accounts) and uses categories both in the navigation pane and in the unified inbox to keep context clear.
- **P4. The identity-conscious user.** Cares about quickly finding the right sending identity. Benefits from identities being grouped by their account's category when composing.

## 6. User Stories

### 6.1 Assigning a category to an account

> **US-1.** As any user, I want to type a category label into a text field on the account-editing screen, so that I can assign my account to a group of my choosing.

> **US-2.** As any user, when I start typing a category, I want to see autocomplete suggestions drawn from categories I have already used on other accounts, so that I can keep my naming consistent without having to remember exact strings.

> **US-3.** As any user, I want to clear the category field to remove an account from any group, so that I can revert to the ungrouped state at any time.

> **US-4.** As any user, I want category assignment to take effect immediately after saving the account, without restarting the application, so that I can verify my grouping right away.

### 6.2 Navigation pane grouping

> **US-5.** As any user with categories enabled, I want accounts in the navigation pane to be sorted first by category (alphabetically, case-insensitive), then by my chosen account order within each category, so that related accounts cluster together.

> **US-6.** As any user, I want uncategorised accounts to appear in a natural position (before or among categorised accounts), so that they are not hidden or lost.

> **US-7.** As any user, I want a setting to enable or disable category-based grouping in the navigation pane, so that I can turn the feature on only when I have enough accounts to benefit from it.

### 6.3 Per-category unified inboxes in the navigation pane

> **US-8.** As a multi-role user, I want the navigation pane to optionally show a separate unified-inbox entry for each category, so that I can open a merged inbox for just my "Work" accounts or just my "Personal" accounts.

> **US-9.** As any user, when I open a per-category unified inbox, I want to see only messages from inboxes belonging to accounts in that category, so that the view is scoped to the context I chose.

> **US-10.** As any user, I want the per-category unified inbox entries to show aggregated unread counts for the accounts in that category, so that I can see at a glance which context needs attention.

> **US-11.** As any user, if all accounts in a category share the same color, I want the per-category unified inbox entry to display that color, so that the visual language is consistent.

### 6.4 Message grouping in the unified inbox

> **US-12.** As a heavy organiser, I want an option to group messages in the unified inbox by account category, so that I see category headings separating blocks of messages and can mentally switch context as I scroll.

> **US-13.** As any user, when category grouping is enabled in the unified inbox, I want messages from uncategorised accounts to appear without a category header, so that they are not artificially grouped under a label I did not assign.

> **US-14.** As any user, I want the category-grouping option to be independent of the navigation-pane grouping option, so that I can use one without the other.

### 6.5 Identities grouped by category

> **US-15.** As an identity-conscious user, I want the identity list to visually separate identities by their owning account's category, so that when I choose a "From" identity I can quickly find the right context.

### 6.6 Discoverability

> **US-16.** As any user, I want a brief explanatory note near the category-grouping setting telling me where to assign categories to accounts (in the account editing screen), so that I can find the setting without guessing.

## 7. Functional Requirements

### 7.1 Category data

- **FR-1.** Each account shall have an optional, free-form text field for its category. The field may be empty (null), meaning the account is uncategorised.
- **FR-2.** Category text shall support sentence-case input with autocorrect.
- **FR-3.** The application shall maintain a dynamic list of all distinct, non-empty categories currently assigned to any account. This list shall be used for autocomplete suggestions.
- **FR-4.** Autocomplete shall activate after a single character is typed.
- **FR-5.** Category values shall be persisted durably and survive application restart, re-login, and account re-synchronisation.
- **FR-6.** Category values shall be included in settings export/import, so that a user migrating to a new machine retains their category assignments.

### 7.2 Category lifecycle

- **FR-7.** A category comes into existence implicitly when the first account is assigned that label. No explicit "create category" action is needed.
- **FR-8.** A category ceases to exist implicitly when no account carries that label any longer. No explicit "delete category" action is needed.
- **FR-9.** Renaming a category is accomplished by editing each account that carries the old label and typing the new one. There is no bulk-rename operation.

### 7.3 Navigation pane behaviour

- **FR-10.** A global setting ("group by account category") shall control whether the navigation pane groups accounts by category. The default shall be **off**.
- **FR-11.** When grouping is enabled, accounts shall be sorted primarily by category name (case-insensitive alphabetical), then by the user's chosen account ordering (manual order, then primary-account flag, then account name).
- **FR-12.** Uncategorised accounts shall sort to the beginning of the list (before any named category).
- **FR-13.** When grouping is enabled, the navigation pane may display visual separators or headings between categories.

### 7.4 Per-category unified inboxes

- **FR-14.** A global setting ("show per-category inboxes in the navigation pane") shall control whether per-category unified inbox entries appear. The default shall be **off**.
- **FR-15.** When enabled, the navigation pane shall display one unified-inbox entry per category, listing only the inbox folders of accounts assigned to that category.
- **FR-16.** Each per-category unified inbox entry shall display aggregated counts: total messages, unread messages, and messages not yet seen by the user.
- **FR-17.** If all inbox folders within a category share the same user-assigned color, the per-category entry shall display that color. If colors differ, no single color shall be shown.
- **FR-18.** Opening a per-category unified inbox shall display a message list filtered to only the inbox folders of accounts in that category, behaving in all other respects like the global unified inbox (same actions, same sort/filter options, same threading).
- **FR-19.** System folders other than inbox (Drafts, Sent, Trash, etc.) and user-created folders shall not be deduplicated or grouped by category in the navigation pane — category grouping applies only to inbox entries.

### 7.5 Message grouping in the unified inbox

- **FR-20.** A global setting ("group messages by account category") shall control whether messages in the unified inbox are visually grouped under category headings. The default shall be **off**.
- **FR-21.** When enabled, messages shall be clustered under their owning account's category. A heading row showing the category name shall precede each cluster.
- **FR-22.** Messages from uncategorised accounts shall not display a category heading row.
- **FR-23.** Category grouping in the message list shall coexist with date-based grouping: within a category cluster, date headers may still appear if date grouping is also enabled.
- **FR-24.** This setting shall be independent of the navigation-pane grouping setting (FR-10) and the per-category unified inbox setting (FR-14).

### 7.6 Identity grouping

- **FR-25.** The identity-selection list shall support visual separation of identities by their owning account's category, using decorators or dividers to delineate groups.

### 7.7 Interaction with other features

- **FR-26.** Category assignment shall have **no effect** on synchronisation behaviour. All accounts synchronise according to their own per-account and per-folder settings regardless of category.
- **FR-27.** Category assignment shall have **no effect** on notification behaviour. Notifications remain governed by per-account and per-folder notification settings.
- **FR-28.** Category assignment shall have **no effect** on unified-inbox membership. A folder's membership in the global unified inbox is controlled independently (see epic 3.1).
- **FR-29.** When a per-category unified inbox is opened and the user initiates a search, the search shall be scoped to the inboxes of accounts in that category only.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Changing a category on an account shall be reflected in the navigation pane and unified inbox within one second under normal load.
- **NFR-2. Scale.** The feature shall remain usable with at least 20 accounts distributed across at least 10 distinct categories.
- **NFR-3. Zero-cost default.** When no account has a category assigned and all category-related settings are off (the default), the feature shall introduce no visible UI elements, no extra processing, and no behavioural changes.
- **NFR-4. Consistency.** Category sort order shall be consistent across the navigation pane, the unified inbox category headers, and the per-category unified inbox entries.
- **NFR-5. Accessibility.** Category headings in the navigation pane and message list shall be reachable via keyboard, carry appropriate screen-reader labels, and respect the application-wide theme.
- **NFR-6. Privacy.** Category labels shall be stored locally only. They shall not be transmitted to the mail server or any third party.

## 10. Open Questions

- **OQ-1. Sort position of uncategorised accounts.** The source application sorts uncategorised accounts (null category) to the beginning of the list. Should this be configurable — for example, allowing uncategorised accounts to sort to the end — or is "first" the only sensible default?
- **OQ-2. Bulk rename.** Renaming a category currently requires editing each account individually. Should a bulk-rename operation be provided (e.g. right-click a category heading → "Rename category"), or is the current implicit approach sufficient?
- **OQ-3. Category ordering.** Categories are sorted alphabetically. Should the user be able to define a custom sort order for categories (e.g. drag-to-reorder), or is alphabetical sufficient?
- **OQ-4. Collapsibility of category groups.** Should category groups in the navigation pane be independently collapsible (hiding all accounts in a category), or is the current flat grouping with visual separators sufficient?
- **OQ-5. Gmail "categories" confusion.** Gmail uses the term "categories" for its own server-side classification (Social, Updates, Promotions, Forums). The in-repo FAQ addresses user confusion about this. The desktop application should consider whether to use a different term (e.g. "account group") to avoid ambiguity, or whether the current terminology is clear enough in context.
- **OQ-6. Per-category unified inbox for non-inbox folders.** The current implementation only creates per-category entries for inbox folders. Should other unified folder types (e.g. unified Drafts, unified Sent) also be groupable by category, or is inbox-only the right scope?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Emergent categories, not managed entities.** Categories are not first-class objects with their own identity, order, or metadata. They are simply string labels on accounts. This is a deliberate simplicity trade-off: there is no category-management screen to build, no orphan-category cleanup to worry about, and no schema complexity. The cost is that renaming requires editing each account and there is no explicit ordering — but these are rare operations for a feature whose primary value is visual grouping in the navigation pane.

- **N-2. Three independent settings.** The feature is controlled by three independent boolean settings: (1) group accounts by category in the navigation pane, (2) show per-category unified inbox entries in the navigation pane, and (3) group messages by category in the unified inbox. Each can be enabled independently of the others. This gives the user fine-grained control: a user who wants grouped accounts but not per-category inboxes can have that; a user who wants category headers in the unified inbox but a flat account list can have that too.

- **N-3. Category is display-only.** A core design principle is that categories affect **only** how items are displayed and grouped. They do not gate synchronisation, notifications, filter rules, or any other functional behaviour. This means a user can freely experiment with categories without risk of breaking their mail flow.

- **N-4. Color aggregation heuristic.** When creating a per-category unified inbox entry, the application uses a simple heuristic for the entry's color: if every inbox in the category has the same user-assigned color, the entry inherits that color; otherwise no color is shown. This avoids inventing a "category color" concept while still providing a useful visual cue for the common case where a user colors all their work accounts blue.

- **N-5. Autocomplete as the consistency mechanism.** Rather than enforcing category names from a predefined list, the application uses autocomplete (triggering after a single character) to nudge the user toward reusing existing category names. This is intentionally soft: the user can always type a new category. The trade-off is that typos can create unintended categories, but the cost of a typo is low (just edit the account again) and the benefit of zero-overhead category creation is high.

- **N-6. Default off.** All three category-related settings default to off. This ensures that the feature is invisible to users who do not need it and introduces no cognitive overhead for the common case of one or two accounts. The feature only becomes visible when the user both assigns categories to accounts and enables the relevant display settings.
