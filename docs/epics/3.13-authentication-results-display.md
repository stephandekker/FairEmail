# Epic: Authentication Results Display

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.13** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"SPF / DKIM / DMARC / ARC / DNSSEC / MTA-STS / DANE indicators per message; warning banner on failure."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email authentication is a layered defense against spoofing, phishing, and man-in-the-middle interception. Several complementary standards exist — SPF, DKIM, DMARC, ARC — each verifying a different aspect of message origin and integrity. Additional transport-level protections — TLS, DNSSEC, DANE — ensure that the connection between mail servers was itself secure and authenticated.

Most of this verification is performed by the receiving mail server, which records its findings in a standard header (`Authentication-Results`, RFC 8601) attached to each message. An email client can parse these results and present them to the user in a meaningful way, giving visibility into whether a message's claimed origin is trustworthy.

The Authentication Results Display feature exists to:

1. Surface the outcome of server-side authentication checks (SPF, DKIM, DMARC, SMTP AUTH) as a per-message visual indicator, so that users can see at a glance whether a message has been authenticated.
2. Warn the user prominently when any authentication check has explicitly **failed**, because failure is a strong signal that the message may be spoofed or tampered with.
3. Provide a detailed breakdown of individual check results on demand, so that technically literate users can understand exactly which checks passed, failed, or were absent.
4. Optionally perform client-side DKIM and ARC verification as a fallback when the mail server does not supply authentication results.
5. Surface supplementary trust signals — TLS transport security, MX record validity, sender/reply-domain consistency, and blocklist status — alongside the core authentication results.
6. Support per-connection DNSSEC and DANE enforcement at the account and identity level, ensuring that the connection to the mail server is itself cryptographically authenticated.

The feature deliberately does **not** attempt to filter or quarantine messages based on authentication results — that is the server's responsibility. Its role is informational: making the existing trust signals visible and actionable for the user.

## 2. Goals

- **G1.** Show a clear, color-coded icon on each message in the message list indicating whether authentication checks passed, partially passed, or failed.
- **G2.** Display a prominent warning when any authentication method has explicitly failed.
- **G3.** Provide a detailed per-method breakdown (DKIM, SPF, DMARC, SMTP AUTH, TLS, MX) accessible by interacting with the authentication indicator.
- **G4.** Fall back to client-side DKIM and ARC verification when the server does not supply authentication results.
- **G5.** Surface additional trust signals — sender/reply-domain mismatch, blocklist membership, and "via" domain — as part of the authentication detail view.
- **G6.** Allow the user to configure which checks are active and how results are displayed, without requiring technical knowledge to benefit from the default settings.
- **G7.** Support DNSSEC and DANE enforcement as optional per-account and per-identity settings for users whose providers support these standards.

## 3. Non-Goals

- **NG1.** This epic does **not** cover message filtering or spam classification based on authentication results. Authentication results may inform rules (covered in the rules epic), but the display feature itself takes no automatic action.
- **NG2.** This epic does **not** define how messages are fetched or synchronized — only how authentication metadata extracted during synchronization is presented to the user.
- **NG3.** BIMI logo display (feature 3.14) is a separate feature that builds on authentication results but is out of scope here.
- **NG4.** This epic does **not** cover OpenPGP or S/MIME signature verification. Those are cryptographic message signatures applied by the sender, not server-side authentication results.
- **NG5.** This epic does **not** prescribe how authentication data is stored, queried, or indexed — only that it is persisted per message and available for display.

## 4. Glossary

| Term | Definition |
|---|---|
| **SPF** (Sender Policy Framework) | A DNS-based mechanism that lets a domain owner declare which mail servers are authorized to send on its behalf. The receiving server checks the sending server's IP against the domain's SPF record. |
| **DKIM** (DomainKeys Identified Mail) | A cryptographic signing scheme where the sending server attaches a digital signature to the message header; the receiving server verifies the signature against a public key published in DNS. |
| **DMARC** (Domain-based Message Authentication, Reporting & Conformance) | A policy framework that builds on SPF and DKIM, requiring "alignment" — i.e. that the domain authenticated by SPF or DKIM matches the domain in the visible From address. |
| **ARC** (Authenticated Received Chain) | An extension that preserves authentication results across intermediary mail servers (e.g. mailing lists, forwarding services) that would otherwise break DKIM signatures. |
| **SMTP AUTH** | Authentication of the sending client to the SMTP submission server (the `auth` result in the Authentication-Results header). |
| **TLS** (Transport Layer Security) | Encryption of the connection between mail servers. A TLS indicator means the message was transported securely. |
| **DNSSEC** (DNS Security Extensions) | Cryptographic signing of DNS records, allowing clients to verify that DNS responses have not been tampered with. |
| **DANE** (DNS-based Authentication of Named Entities) | A mechanism that publishes TLS certificate information in DNSSEC-signed DNS records (TLSA records), allowing a client to verify the mail server's TLS certificate against DNS rather than relying solely on certificate authorities. |
| **MTA-STS** (Mail Transfer Agent Strict Transport Security) | A policy mechanism for mail domains to declare that they support TLS and that senders should refuse to deliver mail over unencrypted connections. |
| **MX record** | A DNS record that identifies the mail server responsible for accepting email on behalf of a domain. |
| **Authentication-Results header** | A message header (RFC 8601) added by the receiving mail server that records the outcomes of SPF, DKIM, DMARC, and other authentication checks. |
| **Alignment** | In DMARC context, the requirement that the domain authenticated by SPF or DKIM matches the domain in the message's visible From address. |
| **Blocklist** | A third-party list of IP addresses or domains known to send spam. |
| **Native DKIM verification** | Client-side DKIM signature verification performed by the email application itself, as a fallback when the server does not provide authentication results. |
| **Native ARC verification** | Client-side ARC chain verification, used to recover DKIM results for messages that passed through whitelisted intermediaries. |

## 5. Personas

- **P1. The security-aware user.** Understands what SPF, DKIM, and DMARC are; wants to see which checks passed and which failed for each message, and wants the ability to enable advanced checks (native DKIM, DANE, strict alignment).
- **P2. The everyday user.** Does not know what these acronyms mean, but wants a simple visual signal that a message is trustworthy — and a clear warning when it is not.
- **P3. The domain administrator.** Receives DMARC aggregate reports and wants to inspect them to monitor their domain's authentication posture.
- **P4. The privacy-conscious user.** Uses forwarding services or aliasing providers (e.g. AnonAddy) where standard DKIM signatures break; needs ARC-based fallback verification to avoid false warnings.

## 6. User Stories

### 6.1 Warning on authentication failure

> **US-1.** As an everyday user, when I receive a message where DKIM, SPF, or DMARC authentication has explicitly failed, I want to see a prominent warning icon on that message in the message list, so that I am alerted to a potentially spoofed message without needing to understand the underlying protocols.

> **US-2.** As any user, when I tap or click the warning icon, I want to see a brief explanation of which authentication method(s) failed, so that I can decide how to treat the message.

> **US-3.** As any user, when SMTP AUTH has explicitly failed, I want that failure included in the warning, so that I am aware that the sending client was not properly authenticated to its mail server.

### 6.2 Authentication status indicator

> **US-4.** As a security-aware user, I want to see a graduated authentication status icon on every message in the message list — not just on failures — so that I can distinguish fully authenticated messages from partially authenticated ones at a glance.

> **US-5.** As a security-aware user, I want the icon to reflect a composite score: the more authentication methods that pass, the stronger the visual signal (e.g. different icon shapes and colors for no authentication, partial authentication, complete authentication, and complete authentication plus TLS).

> **US-6.** As any user, I want the icon to be color-coded: a verified/green color when all expected checks pass (complete authentication), and a neutral/gray color when authentication is partial, so that I do not need to interpret icon shapes.

### 6.3 Detailed authentication breakdown

> **US-7.** As a security-aware user, when I interact with the authentication indicator on a message, I want to see a detailed per-method breakdown showing pass (✓), fail (✗), or absent/unknown (–) for each of: TLS, DKIM, SPF, DMARC, SMTP AUTH, and MX — so that I can understand exactly what was verified.

> **US-8.** As a security-aware user, when native DKIM verification is enabled, I want to see which domain(s) signed the message and whether the signer domain matches the sender domain (alignment), so that I can assess whether the signature is meaningful.

> **US-9.** As any user, when the domain in the SPF envelope (SMTP MAIL FROM) differs from the visible From domain, I want to see a "Via: [domain]" indicator, so that I am aware the message was sent through an intermediary.

> **US-10.** As any user, when the reply-to domain differs from the sender domain, I want to see a warning noting the discrepancy, so that I am alerted to a common phishing technique.

> **US-11.** As any user, when the sending server's IP address is on a known spam blocklist, I want to see an "on blocklist" indicator, so that I have an additional signal about the message's trustworthiness.

### 6.4 Sender domain in authentication detail

> **US-12.** As any user, when I view the authentication detail for a message, I want to see the sender's domain displayed prominently at the top, so that I can immediately identify which domain the authentication results pertain to.

### 6.5 Client-side DKIM and ARC verification

> **US-13.** As a user whose mail server does not add Authentication-Results headers, I want to enable client-side (native) DKIM verification so that I still get authentication indicators for my messages.

> **US-14.** As a user who receives mail through forwarding services or mailing lists, I want the application to perform ARC chain verification against a configurable whitelist of trusted intermediaries, so that messages forwarded by trusted services are not falsely flagged as unauthenticated.

> **US-15.** As a security-aware user, I want to configure the ARC intermediary whitelist (including a "trust all" wildcard option), so that I can control which forwarding services are trusted.

### 6.6 DMARC alignment

> **US-16.** As a security-aware user, I want the application to evaluate DMARC alignment — i.e. whether the DKIM signer domain or SPF-authenticated domain matches the visible From domain — so that the green/verified indicator reflects genuine alignment, not just a passing DKIM check from an unrelated domain.

> **US-17.** As a security-aware user, I want the option to enforce **strict** DMARC alignment (exact domain match, without fallback to organizational domain or SPF pass), so that I can apply a more conservative trust model.

### 6.7 MX record verification

> **US-18.** As a security-aware user, I want to enable MX record verification, which checks whether the sender's or reply-to's domain has valid MX records pointing to a mail server, so that I am warned about messages from domains that cannot receive replies (a common indicator of disposable or spoofed addresses).

### 6.8 DNSSEC and DANE

> **US-19.** As a security-aware user, I want to enable DNSSEC enforcement on a per-account and per-identity basis, so that DNS lookups for my mail server connections are cryptographically validated.

> **US-20.** As a security-aware user, I want to enable DANE enforcement on a per-account and per-identity basis, so that my mail server's TLS certificate is validated against DNSSEC-signed TLSA records rather than relying solely on certificate authorities.

> **US-21.** As any user, when my provider or DNS configuration does not support DNSSEC or DANE, I want the enforcement options to be hidden or disabled, so that I am not confused by settings that cannot function.

### 6.9 DMARC aggregate report viewer

> **US-22.** As a domain administrator, when I receive a DMARC aggregate report (XML attachment), I want to open it in a dedicated viewer that parses and displays the report in a readable format — showing the reporting organization, date range, published policy, and per-source-IP authentication results (SPF and DKIM outcomes, policy evaluation dispositions).

> **US-23.** As a domain administrator viewing a DMARC report, I want the viewer to look up the domain's published DMARC, SPF, and BIMI DNS records, so that I can cross-reference the report against my current DNS configuration.

### 6.10 Configuration

> **US-24.** As any user, I want authentication warning display to be enabled by default, so that I benefit from authentication checks without needing to find and enable a setting.

> **US-25.** As any user, I want the detailed authentication status indicator (graduated icon beyond just failure warnings) to be available as an opt-in setting, so that users who want more detail can enable it without cluttering the default experience.

> **US-26.** As any user, I want the individual underlying checks (server-side authentication, TLS verification, reply-domain check, MX verification, blocklist check) to be independently configurable in the receive settings, so that I can enable or disable each signal according to my needs.

> **US-27.** As any user, I want native DKIM verification, native ARC verification, the ARC whitelist, and strict alignment mode to be available as advanced/debug-level settings, so that power users can access them without exposing complexity to casual users.

## 7. Functional Requirements

### 7.1 Authentication result extraction

- **FR-1.** During message synchronization, the application shall extract authentication results from the standard `Authentication-Results` header (RFC 8601), the `ARC-Authentication-Results` header (RFC 8617), and any recognized provider-specific equivalents (e.g. forwarding-service authentication-results headers).
- **FR-2.** For each of SPF, DKIM, DMARC, and SMTP AUTH, the application shall map the result value to one of three states: **pass** (true), **fail** (false), or **unknown/absent** (null). The mapping shall treat "pass" as pass; "fail", "policy", and "permerror" as fail; "none", "neutral", "temperror", and unrecognized values as unknown.
- **FR-3.** The extracted authentication states shall be persisted per message and shall survive application restarts and re-synchronization.
- **FR-4.** The application shall extract the SMTP MAIL FROM domain from the SPF result for use in "via" domain display (see FR-18).

### 7.2 Native DKIM verification (client-side fallback)

- **FR-5.** When native DKIM verification is enabled, the application shall verify DKIM signatures on incoming messages by retrieving the signer's public key from DNS and validating the cryptographic signature.
- **FR-6.** Native DKIM verification shall not be performed on outgoing messages or messages sent by the user's own identities.
- **FR-7.** When native DKIM verification succeeds, the application shall record which domain(s) signed the message, and shall display this information in the authentication detail view.
- **FR-8.** When native DKIM verification is enabled, the DMARC alignment check shall compare the DKIM signer domain against the envelope domains (Return-Path, From, and optionally SMTP MAIL FROM) to determine whether the signature is aligned with the visible sender.

### 7.3 Native ARC verification

- **FR-9.** When native ARC verification is enabled and no DKIM signers are found, the application shall attempt to validate the ARC chain on the message according to RFC 8617 (collect ARC sets, verify sequence integrity, check chain validation status values, verify the most recent ARC-Message-Signature).
- **FR-10.** ARC verification shall only accept signers whose domain appears on a user-configurable whitelist of trusted intermediaries. A wildcard option shall be available to trust all ARC signers.
- **FR-11.** The default ARC whitelist shall include commonly used forwarding and aliasing services.

### 7.4 Warning display (authentication failure)

- **FR-12.** When the "show authentication warnings" setting is enabled (default: enabled) and any of the enabled authentication checks has explicitly **failed**, the application shall display a warning icon on the message in the message list. The icon shall use a visually distinct warning treatment (e.g. a flag shape and a warning color).
- **FR-13.** The warning shall be triggered by any of: DKIM fail, SPF fail, DMARC fail, SMTP AUTH fail, reply-domain mismatch (if the reply-domain check is enabled), MX fail (if the MX check is enabled), or blocklist hit (if the blocklist check is enabled).
- **FR-14.** When the user interacts with the warning icon, the application shall display a summary listing which specific method(s) failed (e.g. "DKIM, SPF authentication failed").

### 7.5 Authentication status indicator (graduated display)

- **FR-15.** When the "show authentication status indicator" setting is enabled (default: disabled), the application shall display a graduated authentication icon on each message, reflecting the composite authentication strength.
- **FR-16.** The composite score shall be computed as follows:
  - Count the number of passing checks among DKIM, SPF, and DMARC (0–3).
  - If DMARC passes **and** at least one of DKIM or SPF passes (alignment), treat the score as 3 (complete authentication) regardless of individual counts.
  - If SMTP AUTH passes, treat the score as 3 (complete authentication).
  - If TLS verification is enabled and TLS passed, add 1 to the score.
- **FR-17.** The icon shall have the following visual states, in ascending order of authentication strength:
  1. **Flag** (warning color): authentication explicitly failed (shown only when failures are detected; see FR-12).
  2. **Question mark** (neutral color): no authentication data available (no results for any method).
  3. **Bad shield** (neutral color): score 0 — authentication data exists but nothing passed.
  4. **Partial shield** (neutral color): score 1 — one method passed.
  5. **Partial shield** (neutral color): score 2 — two methods passed.
  6. **Good shield** (verified color): score 3 — complete authentication (all core methods or alignment).
  7. **Verified badge** (verified color): score 4 — complete authentication plus TLS.
- **FR-18.** The verified/green color shall be used only when the score is 3 or higher; all lower scores shall use a neutral/gray color.

### 7.6 Authentication detail view

- **FR-19.** When the user interacts with the authentication indicator (whether in warning or indicator mode), the application shall display a detail view containing:
  - The sender's domain, displayed prominently.
  - A list of failed authentication methods (if any), with an "authentication failed" message.
  - If the indicator mode is enabled: a per-method breakdown showing pass (✓), fail (✗), or absent/unknown (–) for each active check (TLS if enabled, DKIM, SPF, DMARC, SMTP AUTH if present, MX if enabled).
  - If native DKIM is enabled and signer domains are available: "Signed by: [domain(s)]" with aligned domains visually highlighted.
  - If the sender's IP is on a blocklist: an "on blocklist" notice.
  - If the SMTP MAIL FROM domain differs from the visible From domain: a "Via: [domain]" notice.
  - If the reply-to domain differs from the sender domain: a domain-mismatch warning.

### 7.7 Signer domain display in message headers

- **FR-20.** When native DKIM verification is enabled and signer domains are available, the message header area (expanded view) shall display the signer domain(s).
- **FR-21.** Signer domains that are aligned with the sender's envelope domain shall be visually distinguished (e.g. highlighted in the link/accent color). Non-aligned signer domains shall be shown in a warning color.
- **FR-22.** The signer domain display shall be visible by default when the message headers are expanded and either the addresses are shown in full or the signer is not aligned (to draw attention to the mismatch).

### 7.8 DMARC aggregate report viewer

- **FR-23.** The application shall provide a dedicated viewer for DMARC aggregate report attachments (XML format per RFC 7489).
- **FR-24.** The viewer shall parse and display: the reporting organization, report ID, date range, published DMARC policy (domain, DKIM alignment mode, SPF alignment mode, policy, subdomain policy, percentage), and per-record details (source IP with reverse DNS, message count, policy disposition, DKIM and SPF evaluation results including domain, result, and selector).
- **FR-25.** The viewer shall perform supplementary DNS lookups for the reported domain's DMARC record (`_dmarc.[domain]`), SPF record, and BIMI record (`default._bimi.[domain]`), displaying the current DNS values alongside the report data.

### 7.9 DNSSEC and DANE enforcement

- **FR-26.** Each account and each identity shall have independent, optional boolean settings for enforcing DNSSEC and DANE on their mail server connections.
- **FR-27.** When DNSSEC is enforced for an account or identity, the application shall validate that DNS responses for that connection's server hostname are DNSSEC-signed. Connection shall fail if validation fails.
- **FR-28.** When DANE is enforced for an account or identity, the application shall retrieve TLSA records for the mail server and verify the server's TLS certificate against them. Connection shall fail if DANE verification fails.
- **FR-29.** The DNSSEC and DANE options shall only be available when the runtime environment supports the required DNS capabilities. When unavailable, the options shall be hidden from the user.
- **FR-30.** DANE enforcement shall imply TLS enforcement — if DANE is enabled, the connection must not fall back to an unencrypted transport.

### 7.10 TLS transport indicator

- **FR-31.** The application shall record whether each incoming message was transported over TLS (based on server-provided metadata or headers).
- **FR-32.** When TLS verification display is enabled, the TLS status shall be included in both the composite authentication score (FR-16) and the detailed breakdown (FR-19).

### 7.11 Sender domain checks

- **FR-33.** When the reply-domain check is enabled (default: enabled), the application shall compare the domain of the sender address to the domain of the reply-to address during synchronization and persist the result per message.
- **FR-34.** When the MX check is enabled (default: disabled), the application shall perform a DNS MX lookup for the sender's or reply-to's domain during synchronization and persist the result per message.
- **FR-35.** When the blocklist check is enabled (default: disabled), the application shall check the sending server's IP address against configured blocklists during synchronization and persist the result per message.

### 7.12 Configuration surface

- **FR-36.** The display settings shall include:
  - A toggle for "Show authentication warnings" (default: enabled). Controls whether the failure warning icon appears.
  - A toggle for "Show authentication status indicator" (default: disabled). Controls the graduated indicator. This toggle shall be dependent on the warnings toggle being enabled.
  - A visual legend showing the meaning of each indicator level (no authentication, partial authentication, complete authentication, +TLS).
- **FR-37.** The receive/synchronization settings shall include:
  - A toggle for "Check message authentication" (default: enabled). Controls whether DKIM, SPF, and DMARC results are extracted from server headers.
  - A toggle for "Check TLS" (default: disabled). Controls whether TLS transport status is checked and displayed.
  - A toggle for "Check reply domain" (default: enabled). Controls whether sender/reply-domain mismatch is flagged.
  - A toggle for "Check MX" (default: disabled). Controls whether MX record existence is verified.
  - A toggle for "Check blocklist" (default: disabled). Controls whether sending-server IP blocklist checking is performed.
- **FR-38.** Advanced/debug settings shall include:
  - A toggle for "Native DKIM verification" (default: disabled).
  - A toggle for "Native ARC verification" (default: enabled when native DKIM is active).
  - A text field for the ARC intermediary whitelist.
  - A toggle for "Strict DMARC alignment" (default: disabled).
- **FR-39.** Per-account and per-identity advanced settings shall include:
  - A checkbox for "Enforce DNSSEC" (default: disabled; hidden when not supported).
  - A checkbox for "Enforce DANE" (default: disabled; hidden when DNSSEC is not available).

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Authentication indicators shall render in the message list without perceptible delay; the underlying data is pre-computed during synchronization, not at display time.
- **NFR-2. Resource efficiency.** Native DKIM and ARC verification, MX lookups, and blocklist checks all increase network traffic and processing time during synchronization. Each shall be disabled by default and shall include user-facing documentation of its resource impact.
- **NFR-3. Accuracy.** The application shall not display a pass indicator for a check that the server reported as "fail", "policy", or "permerror", and shall not display a fail indicator for a check the server reported as "pass". The three-state model (pass/fail/unknown) shall faithfully reflect the server's report.
- **NFR-4. Graceful degradation.** When a mail server does not add an `Authentication-Results` header (which is optional per RFC), the application shall display "no authentication data" (unknown state) rather than a misleading pass or fail. Native DKIM verification serves as an opt-in fallback for this case.
- **NFR-5. Offline availability.** Authentication results are persisted per message and shall be fully available offline. DNSSEC/DANE enforcement applies at connection time only and does not affect offline reading.
- **NFR-6. Accessibility.** All authentication icons shall have screen-reader labels (e.g. "Show authentication result"). The detail view shall be accessible via keyboard interaction.
- **NFR-7. Privacy.** Authentication checking shall not introduce network traffic beyond what synchronization already requires, except for the opt-in features that explicitly perform additional DNS lookups (native DKIM, MX check, blocklist check, DNSSEC/DANE). These opt-in features shall document their network implications.
- **NFR-8. Compatibility.** The feature shall correctly parse Authentication-Results headers from all major mail providers, including those that use non-standard header names for forwarding/aliasing services.

## 10. Open Questions

- **OQ-1. MTA-STS surface.** The high-level feature list mentions MTA-STS alongside DNSSEC and DANE, but the source application does not appear to have an explicit MTA-STS verification or indicator beyond general TLS enforcement. Should MTA-STS policy retrieval and display be a first-class feature, or is the current behavior (implicit via TLS enforcement preferences) sufficient?
- **OQ-2. DNSSEC/DANE availability constraints.** In the source application, DNSSEC and DANE are available only in the GitHub release variant (not the Play Store variant) and require a minimum OS version. For a desktop application, should DNSSEC and DANE be universally available, or are there equivalent distribution or runtime constraints to consider?
- **OQ-3. Native DKIM battery/data warning.** The source application warns that native DKIM verification increases data and battery usage. On a desktop application with persistent power and broadband, should native DKIM verification be promoted from a debug-level setting to a standard setting, or even enabled by default?
- **OQ-4. ARC whitelist default contents.** The source application includes a default ARC whitelist of common forwarding services. The canonical list of default entries and the criteria for inclusion should be documented and maintained as a living list.
- **OQ-5. Blocklist selection.** The source application supports checking sending-server IPs against blocklists, but the specific blocklists used and their update mechanism are not fully specified in this epic. The list of blocklists and their refresh cadence should be decided during design.
- **OQ-6. DMARC report viewer scope.** The DMARC aggregate report viewer in the source application is a dedicated screen that parses XML attachments. Should the desktop application also support DMARC failure (forensic) reports (RFC 7489 §7.3), or only aggregate reports?
- **OQ-7. Authentication results from AnonAddy-style headers.** The source application recognizes a provider-specific `X-AnonAddy-Authentication-Results` header. The full list of recognized non-standard authentication-results header names, and the policy for adding new ones, should be decided during design.
- **OQ-8. Interaction between authentication display and encryption indicators.** When a message is both encrypted (OpenPGP/S/MIME) and has authentication results, how should the two sets of indicators coexist visually? The source application uses separate icon positions, but the interaction deserves explicit design consideration.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Server-side first, client-side as fallback.** The source application's philosophy is that authentication verification is the mail server's job. The application's primary role is to *display* server-provided results, not to replicate the server's work. Native DKIM/ARC verification exists as a fallback for servers that do not provide Authentication-Results headers, and is deliberately off by default and placed in advanced settings, reflecting its higher resource cost and the principle that the server should be doing this work.

- **N-2. Three-state model (pass/fail/unknown).** Each authentication method maps to a tri-state boolean: true (pass), false (fail), or null (unknown/absent). This is important because "unknown" and "fail" are meaningfully different — an absent SPF result means the server did not check (or the domain has no SPF record), while a failed SPF result means the server checked and the check failed. The UI distinguishes these cases: "fail" triggers a warning, "unknown" does not.

- **N-3. DMARC alignment as the trust threshold.** The green/verified indicator requires DMARC pass combined with at least one of DKIM or SPF passing (i.e. DMARC alignment). A message that passes DKIM and SPF individually but fails DMARC alignment does not get the green indicator. This reflects the industry consensus that alignment is the meaningful trust signal — an unaligned DKIM signature proves the message was signed, but not necessarily by the claimed sender.

- **N-4. SMTP AUTH equivalence.** When SMTP AUTH passes, the application treats it as equivalent to full authentication (score 3). This is because SMTP AUTH means the sending client authenticated to the submission server, which is a strong trust signal for messages where the user's own server adds this header.

- **N-5. Cascading dependencies in display settings.** The "Show authentication status indicator" toggle depends on "Show authentication warnings" being enabled. This prevents a confusing state where the graduated indicator is on but the failure warning is off. The underlying checks in receive settings are independently configurable, so a user can enable authentication display but disable individual checks (e.g. show warnings for DKIM/SPF/DMARC failures but not for MX failures).

- **N-6. Default-on for warnings, default-off for indicator.** Authentication warnings (the failure flag) are on by default because they surface a clear, actionable signal (something is wrong). The graduated indicator is off by default because it adds visual complexity to every message and is most useful for users who understand the underlying protocols.

- **N-7. DNSSEC and DANE as connection-level enforcement.** DNSSEC and DANE are fundamentally different from the per-message authentication checks. They protect the *connection* to the mail server, not individual messages. They are therefore configured per-account and per-identity (where the connection is established), not in the display settings. Their enforcement is binary (connect or refuse), not graduated.

- **N-8. Strict alignment as an opt-in.** The default DMARC alignment check uses "relaxed" alignment: the DKIM signer domain and the From domain need only share the same organizational (root) domain. Strict alignment requires an exact match. The relaxed default is appropriate because many legitimate senders use subdomains for outbound mail (e.g. `mail.example.com` signing for `example.com`). Strict alignment is available for users who want a more conservative trust model.
