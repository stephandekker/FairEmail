# Epic: OAuth2 Sign-In

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.5** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"First-class OAuth flows for Gmail, Outlook/Microsoft 365, Yahoo, AOL, Yandex, Mail.ru, etc. Tokens are refreshed automatically."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Traditional email authentication requires the user to supply their mailbox password directly to the email client. This has two problems: (a) the user must trust the client with their actual account credentials, and (b) many major email providers — Google, Microsoft, Yahoo — have deprecated or outright blocked password-based IMAP/SMTP access in favor of OAuth2 token-based authentication.

OAuth2 sign-in addresses both problems. The user authenticates directly with their email provider in a browser or system-provided web view; the provider issues a time-limited access token and a long-lived refresh token to the email client. The client never sees the user's password. Tokens are scoped to only the permissions the client needs (mail access), and the provider can revoke them at any time without affecting the user's account password.

For the application to be viable with the most popular email services, OAuth2 is not optional — it is the **primary and sometimes only** way to authenticate. The feature must cover the full lifecycle: initial authorization, token storage, automatic refresh before expiry, re-authorization when tokens are revoked, and graceful degradation when OAuth is unavailable (e.g. for providers that do not support it, or for distribution channels where OAuth credentials are not available).

## 2. Goals

- **G1.** Provide a one-click, browser-based OAuth2 sign-in flow for every supported provider, integrated into the quick-setup wizard.
- **G2.** Refresh access tokens automatically and transparently, so that the user never has to re-authenticate under normal circumstances.
- **G3.** Support all major providers that mandate or strongly prefer OAuth2: Gmail, Outlook/Office 365, Yahoo, AOL, Yandex, Mail.ru, Fastmail, and NetEase 163.
- **G4.** Support multi-tenant environments (e.g. organizational Office 365 tenants) without requiring the user to understand OAuth internals.
- **G5.** Allow power users to define custom OAuth configurations for providers not in the bundled list.
- **G6.** Degrade gracefully when OAuth is not available: fall back to password-based authentication and inform the user clearly.
- **G7.** Protect the user's authorization session against cross-site request forgery and session hijacking.

## 3. Non-Goals

- **NG1.** This epic does **not** define how mail is fetched or synchronized once authenticated — only how the credential is obtained, stored, and refreshed.
- **NG2.** This epic does **not** prescribe the credential storage mechanism (keyring, encrypted database, etc.) — only that tokens must be stored durably and securely.
- **NG3.** Provider-specific APIs beyond authentication (e.g. a vendor's proprietary send-mail API or contacts API) are out of scope except where a provider **requires** a non-IMAP/SMTP pathway as part of the OAuth grant (see §11, Design Note N-5 on vendor graph APIs).
- **NG4.** Implementing an OAuth2 authorization server is not a goal; the application is always an OAuth2 client.
- **NG5.** Single sign-on (SSO) federation between accounts is not a goal; each account authenticates independently.
- **NG6.** OAuth1 or OAuth1a support is not a goal.

## 4. Glossary

| Term | Definition |
|---|---|
| **OAuth2** | An authorization framework (RFC 6749) that allows a third-party application to obtain limited access to a service on behalf of the user, without the user sharing their password with the application. |
| **Authorization code flow** | The OAuth2 grant type used for interactive sign-in: the user authenticates in a browser, receives an authorization code, and the application exchanges it for tokens. |
| **Access token** | A short-lived credential (typically 1 hour) that the application presents to the mail server to prove it is authorized to act on the user's behalf. |
| **Refresh token** | A long-lived credential issued alongside the access token, used to obtain a new access token when the current one expires, without requiring the user to re-authenticate. |
| **Scope** | A permission string that limits what the access token can do (e.g. read/write mail via IMAP, send via SMTP). |
| **PKCE** | Proof Key for Code Exchange (RFC 7636) — a security extension that prevents authorization code interception attacks, especially important for public (non-confidential) clients. |
| **Redirect URI** | The URI to which the provider redirects the browser after the user authorizes; the application intercepts this redirect to receive the authorization code. |
| **Tenant** | In multi-tenant identity systems (e.g. Microsoft Entra ID / Azure AD), the organizational directory that governs which users can sign in. A personal account uses a common/shared tenant; an organizational account uses the organization's tenant ID. |
| **Token refresh** | The process of using a refresh token to obtain a new access token without user interaction. |
| **Provider** | A bundled configuration record that maps an email domain to its server settings, OAuth endpoints, scopes, and client credentials. |
| **Custom OAuth** | A user-imported provider configuration that supplies OAuth endpoints and credentials for a provider not included in the bundled list. |
| **Jump page** | An intermediary web page hosted by the application's publisher that performs the OAuth redirect on behalf of the application, used when the native redirect mechanism is unavailable or untrusted. |
| **On-device account** | A system-level account (e.g. a Google account registered with the operating system) whose tokens are managed by the OS account manager rather than by the application directly. |

## 5. Personas

- **P1. The mainstream Gmail/Outlook user.** Has one or two accounts with a major provider. Expects sign-in to "just work" — tap the provider, authenticate in the browser, done. Does not know or care what OAuth is.
- **P2. The multi-account professional.** Has personal Gmail, an organizational Office 365 account (with a custom tenant), and perhaps a Yahoo or Yandex account. Needs each account to authenticate through the correct provider flow, and needs the organizational account to respect tenant restrictions.
- **P3. The privacy-conscious user.** Wants to understand what permissions are being granted, prefers a privacy-focused browser for the OAuth flow, and wants assurance that tokens are stored securely and that the application cannot access more than mail.
- **P4. The self-hoster / power user.** Runs their own mail server with an identity provider that supports OAuth2 (e.g. Keycloak, Authelia). Needs to supply custom OAuth endpoints and credentials.
- **P5. The distribution-restricted user.** Runs a build of the application obtained from a distribution channel that does not have OAuth credentials (e.g. a community-maintained package). Must fall back to password authentication or supply their own OAuth configuration.

## 6. User Stories

### 6.1 Initial sign-in via OAuth

> **US-1.** As a mainstream user, when I add a new account and select a supported provider (e.g. Gmail, Outlook), I want the application to open my browser to the provider's sign-in page, so that I can authenticate without giving the application my password.

> **US-2.** As any user, after I authenticate in the browser and approve the requested permissions, I want the browser to redirect back to the application automatically, so that the sign-in completes without manual steps.

> **US-3.** As any user, after the redirect, I want the application to verify that my credentials work by testing the mail server connection, and then create my account, so that I know the setup succeeded before I leave the wizard.

> **US-4.** As a multi-account professional with an organizational Office 365 account, I want the application to let me specify my organization's tenant identifier, so that the OAuth flow authenticates against the correct directory.

> **US-5.** As any user, I want the application to request only the minimum permissions necessary to access my mail (read, write, send), so that I am not granting broader access than needed.

### 6.2 Token lifecycle

> **US-6.** As any user, I want the application to refresh my access token automatically before it expires, so that my mail keeps flowing without any action from me.

> **US-7.** As any user, if my refresh token is revoked by the provider (e.g. I changed my password, or an administrator revoked access), I want the application to notify me clearly and offer a one-click path to re-authorize, so that I can restore access quickly.

> **US-8.** As any user, I want the application to retry token refresh gracefully in the face of transient network errors, rather than immediately declaring my account broken.

> **US-9.** As any user, I want the most recently valid access token to continue being used for in-flight operations even while a refresh is in progress, so that brief refresh delays do not interrupt active synchronization.

### 6.3 Provider selection and discovery

> **US-10.** As a mainstream user, when I enter my email address in the setup wizard, I want the application to detect my provider and offer the OAuth sign-in option automatically, so that I do not have to search a provider list.

> **US-11.** As any user, I want to see the list of providers that support OAuth sign-in, clearly distinguished from password-only providers, so that I can make an informed choice.

> **US-12.** As any user, when the provider I selected supports OAuth, I want the OAuth flow to be the default and recommended path, so that I am guided toward the more secure option.

### 6.4 Custom OAuth

> **US-13.** As a self-hoster, I want to import a configuration file that defines my own OAuth endpoints, scopes, and client credentials, so that I can use OAuth with my private identity provider.

> **US-14.** As a self-hoster, after importing a custom OAuth configuration, I want it to appear in the setup wizard alongside the bundled providers, so that the sign-in experience is identical.

### 6.5 Fallback to password authentication

> **US-15.** As a distribution-restricted user whose build does not include OAuth credentials, I want the application to inform me that OAuth is unavailable for my build and guide me to use password or app-password authentication instead.

> **US-16.** As any user whose provider supports both OAuth and password authentication, I want the ability to choose password authentication if I prefer, so that I am not forced into OAuth.

> **US-17.** As any user, I want the ability to convert an existing OAuth-authenticated account to password authentication (and vice versa), without losing my account settings, folders, or local message store.

### 6.6 Re-authorization

> **US-18.** As any user, I want to be able to re-authorize an existing account (re-run the OAuth flow) from the account settings, so that I can fix token problems without deleting and re-adding the account.

> **US-19.** As any user, when re-authorization succeeds, I want the new tokens to replace the old ones seamlessly, preserving all my account settings, folder structure, rules, and identities.

### 6.7 Browser and security

> **US-20.** As a privacy-conscious user, I want the OAuth flow to open in my preferred system browser (or a privacy-focused browser), not in an embedded web view controlled by the application, so that my credentials are entered in a trusted context.

> **US-21.** As any user, I want the OAuth session to be protected against cross-site request forgery, so that a malicious page cannot complete the authorization on my behalf.

> **US-22.** As any user, I want the authorization session to expire after a reasonable timeout (e.g. 20 minutes) if I do not complete it, so that an abandoned browser tab cannot be used later.

### 6.8 Multiple identities and shared mailboxes

> **US-23.** As an Outlook user with access to a shared mailbox, I want to authenticate with my own credentials and then access the shared mailbox, so that I can send and receive on behalf of the shared address.

> **US-24.** As a user with multiple identities (aliases) on one account, I want the OAuth sign-in to authenticate the account once and make all my identities available, so that I do not have to authorize separately for each alias.

### 6.9 Error communication

> **US-25.** As any user, when the OAuth flow fails (provider error, network error, permission denied), I want a clear, actionable error message — not a raw protocol error — so that I know what to do next.

> **US-26.** As any user, when the provider returns a rate-limit or temporary-block error during token refresh, I want the application to tell me what happened and suggest waiting or re-authorizing, rather than silently failing.

## 7. Functional Requirements

### 7.1 Supported providers

- **FR-1.** The application shall ship with a bundled provider database that includes OAuth2 configurations for at least: Gmail, Outlook/Office 365, Yahoo, AOL, Yandex, Mail.ru, Fastmail, and NetEase 163.
- **FR-2.** Each bundled OAuth provider configuration shall specify: authorization endpoint, token endpoint, redirect URI, required scopes, and any provider-specific parameters (e.g. tenant placeholder, consent prompt, offline-access flag).
- **FR-3.** The provider database shall be extensible by the user through import of a custom configuration file (see §7.6).

### 7.2 Authorization flow

- **FR-4.** The application shall implement the OAuth2 authorization code flow with PKCE for all providers.
- **FR-5.** The authorization request shall be opened in the user's system browser or a system-provided secure browser surface (e.g. a sandboxed browser tab), never in an application-embedded web view.
- **FR-6.** The application shall include a cryptographic state parameter in every authorization request and shall validate it on the redirect response, rejecting any response with a missing or mismatched state.
- **FR-7.** The application shall enforce a session timeout on the authorization flow (no longer than 20 minutes). If the redirect is not received within this window, the session shall be invalidated.
- **FR-8.** On receiving a valid authorization code, the application shall exchange it for an access token and a refresh token.
- **FR-9.** If the token exchange does not return a refresh token, the application shall treat the authorization as failed and inform the user, since automatic refresh will not be possible.
- **FR-10.** For providers that support or require a tenant identifier (e.g. Microsoft), the application shall provide a user-facing field to enter the tenant, and shall substitute the tenant value into the endpoint URLs.
- **FR-11.** The application shall support a jump page (an intermediary web redirect hosted by the publisher) as a fallback redirect mechanism for environments where native deep-link redirection is unavailable.

### 7.3 Token storage

- **FR-12.** Access tokens, refresh tokens, and token expiry timestamps shall be stored durably and survive application restarts.
- **FR-13.** Tokens shall be stored with at least the same level of protection as account passwords (i.e. encrypted at rest if the application supports credential encryption).
- **FR-14.** Each account shall store its own independent token state; tokens are never shared between accounts, even if they belong to the same provider.

### 7.4 Token refresh

- **FR-15.** The application shall refresh the access token automatically before it expires, without any user interaction.
- **FR-16.** Token refresh shall be attempted proactively (before expiry) rather than reactively (after a server rejection), to minimize authentication failures during active synchronization.
- **FR-17.** If a token refresh fails due to a transient error (network timeout, server error), the application shall retry with backoff before declaring the token unusable.
- **FR-18.** If a token refresh fails due to a permanent error (refresh token revoked, user consent withdrawn), the application shall mark the account as requiring re-authorization and notify the user.
- **FR-19.** While a token refresh is in progress, the application shall serialize concurrent refresh attempts for the same provider to avoid redundant requests and rate-limit triggers.
- **FR-20.** A successful token refresh shall update the stored token state atomically, so that no connection attempt uses a partially-updated credential.

### 7.5 Account setup integration

- **FR-21.** When the user enters an email address in the quick-setup wizard and the detected provider supports OAuth, the application shall present OAuth as the default and recommended authentication method.
- **FR-22.** After successful authorization and token acquisition, the application shall test the IMAP (or POP3) and SMTP connections using the new token to verify that the credentials are functional.
- **FR-23.** If the connection test succeeds, the application shall create the account, discover folders, and create a default sending identity, using information extracted from the authorization response (e.g. the user's email address and display name from identity claims in the token).
- **FR-24.** If the connection test fails despite a valid token, the application shall display a provider-specific error message and suggest corrective action (e.g. enabling IMAP access in provider settings, checking tenant security policies).
- **FR-25.** The application shall support updating an existing account's tokens by re-running the OAuth flow from the account settings, without deleting the account.

### 7.6 Custom OAuth

- **FR-26.** The application shall allow the user to import a provider configuration file that defines custom OAuth2 endpoints, scopes, client credentials, and server settings.
- **FR-27.** Imported custom providers shall be available in the setup wizard alongside bundled providers.
- **FR-28.** The import mechanism shall be accessible from the application's advanced or debug settings.

### 7.7 Fallback and conversion

- **FR-29.** When OAuth credentials are not available for a provider (e.g. on a distribution channel that lacks client registrations), the application shall clearly indicate that OAuth is unavailable and guide the user to password-based setup.
- **FR-30.** The application shall support converting an account from OAuth to password authentication and from password to OAuth, preserving all account settings, folder state, rules, identities, and local messages.

### 7.8 Browser selection

- **FR-31.** The application shall prefer privacy-focused browsers for the OAuth flow when available, and shall allow the user to configure which browser is used.
- **FR-32.** The application shall warn the user if the selected or default browser is known to have compatibility issues with OAuth redirect handling.
- **FR-33.** If no suitable browser is available, the application shall fall back to its own embedded secure browser surface with a clear indication that the context has changed.

### 7.9 Identity extraction

- **FR-34.** When the authorization response includes identity information (e.g. email address, display name, or username encoded in an identity token), the application shall extract and use this information to pre-fill the account and identity configuration.
- **FR-35.** For providers that do not include identity information in the token, the application shall fetch it from a provider-specific user-info endpoint if one is available.
- **FR-36.** If neither method yields a usable email address, the application shall prompt the user to enter it before completing setup.

### 7.10 Provider-specific behavior

- **FR-37.** For providers that require an explicit consent prompt on every authorization (e.g. Gmail's `prompt=consent`), the application shall include the required parameter.
- **FR-38.** For providers that require explicit offline-access flags to issue a refresh token (e.g. Gmail's `access_type=offline`), the application shall include the required parameter.
- **FR-39.** For providers that offer both IMAP/SMTP access and a proprietary send-mail API via a broader OAuth grant, the application shall request the broader grant when the provider's SMTP reliability is improved by using the proprietary pathway (see §11, N-5). The user need not be aware of this distinction.
- **FR-40.** For providers that support shared mailbox access (e.g. Outlook delegated mailboxes), the application shall accept a shared-mailbox identifier during setup and use the appropriate username syntax when authenticating to the mail server.

## 8. Non-Functional Requirements

- **NFR-1. Latency.** The token refresh shall complete within the duration of a normal connection setup, so that the user does not perceive a delay when mail is being fetched.
- **NFR-2. Reliability.** Token refresh shall tolerate transient network failures of up to 90 seconds before declaring a permanent failure.
- **NFR-3. Security — no embedded credentials in UI.** Access tokens and refresh tokens shall never be displayed to the user in plain text under normal operation. A debug or diagnostic mode may expose token metadata (expiry time, scopes) but not the token value itself.
- **NFR-4. Security — CSRF protection.** Every authorization request shall include a cryptographic state parameter validated on response. A missing or incorrect state shall abort the flow.
- **NFR-5. Security — session expiry.** An incomplete authorization session shall become invalid after at most 20 minutes.
- **NFR-6. Security — minimal scopes.** The application shall request the narrowest set of OAuth scopes sufficient for mail access (read, write, send). It shall not request profile, contacts, calendar, or other scopes unless they are bundled inseparably by the provider with the mail scope.
- **NFR-7. Concurrency.** Concurrent token refresh requests for the same account shall be serialized so that only one refresh is in flight at a time, avoiding rate-limit penalties.
- **NFR-8. Privacy.** The OAuth flow shall not transmit any information to the application's publisher or to any third party beyond the email provider itself, except when the jump page redirect is used (and in that case, only the authorization code transits the publisher's server — no tokens).
- **NFR-9. Accessibility.** The OAuth provider selection, tenant input, and error messages shall be fully navigable by keyboard, announced by screen readers, and styled according to the application-wide theme.
- **NFR-10. Offline resilience.** If the application is offline when a token refresh is due, it shall continue using the last valid access token for any cached/local operations and shall refresh as soon as connectivity is restored.

## 10. Open Questions

- **OQ-1. Jump page trust model.** The application uses a publisher-hosted jump page as a fallback redirect mechanism. What is the trust model for this page? Should the user be informed when the jump page is in use vs. a native redirect? Should self-hosters be able to run their own jump page?
- **OQ-2. On-device account integration.** The source application has (or historically had) support for using the operating system's account manager to obtain tokens for on-device Google accounts, bypassing the browser flow entirely. The desktop equivalent would be integration with GNOME Online Accounts or KDE's KAccounts. Should this be a goal? The source codebase shows this path exists but has been partially disabled at various points.
- **OQ-3. Multiple Google accounts.** The source application's FAQ notes that when using OAuth with multiple Google accounts, other Google accounts may need to be logged out of the browser first. Is this a fundamental limitation of Google's consent screen, or can it be mitigated by passing a login hint? The source passes the user's email as a hint, but the FAQ suggests this does not always work.
- **OQ-4. Graph API for sending.** The source application has a variant of the Outlook OAuth configuration that uses a vendor-specific graph API for sending mail instead of SMTP. This blurs the line between "authentication" and "transport." Should this be treated as part of the OAuth epic, or split into a separate provider-integration epic? (Currently captured minimally in FR-39.)
- **OQ-5. Yahoo/AOL dual registrations.** The source application maintains two separate OAuth client registrations for Yahoo (and a separate one for AOL, which shares Yahoo's infrastructure). It is unclear whether both registrations are active or whether one is a legacy fallback. This should be clarified before the bundled provider list is finalized.
- **OQ-6. Token storage encryption scope.** FR-13 requires token storage to be "at least as protected as passwords." If the application supports a master-password or encrypted-database feature, should OAuth tokens be stored within that encrypted boundary, or in a system keyring? Both approaches have trade-offs (portability vs. OS integration).
- **OQ-7. NetEase 163 OAuth uses HTTP.** The bundled provider configuration for NetEase 163 specifies non-HTTPS (plain HTTP) authorization and token endpoints. This is a potential security concern. Should the application warn the user or refuse to use non-HTTPS OAuth endpoints?
- **OQ-8. Browser compatibility list.** The source application excludes certain browser versions from the OAuth flow due to known redirect-handling bugs (e.g. older Samsung Browser). How should the desktop application maintain this compatibility list, and how often should it be updated?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. OAuth as default, not exclusive.** For providers that support OAuth, the source application presents OAuth as the default and recommended path in the setup wizard, but it does not prevent the user from choosing password authentication via the "Other provider" manual-setup path. This is intentional: some organizational environments block third-party OAuth but permit app passwords, and some users simply prefer password auth. OAuth is the **recommended** path, not a locked gate.

- **N-2. PKCE for all providers.** The source application uses PKCE (Proof Key for Code Exchange) for the authorization code flow regardless of whether the provider requires it. This is a defense-in-depth measure: even if a provider does not mandate PKCE, using it prevents authorization code interception attacks at no cost. Future implementations should maintain this posture.

- **N-3. One refresh at a time per provider.** The source application serializes token refresh requests per provider (not merely per account) using a lock. This prevents a scenario where multiple accounts on the same provider (e.g. two Gmail accounts) simultaneously attempt to refresh and trigger rate limiting. The lock ensures only one refresh per provider is in flight at any moment.

- **N-4. Mandatory refresh token.** If the token exchange response does not include a refresh token, the source application treats the entire authorization as failed. This is a deliberate contract: without a refresh token, the application would need to re-launch the browser flow every time the access token expires (typically every hour), which is unacceptable for a background-syncing email client. Providers that do not issue refresh tokens (or that require special parameters to do so) must have those parameters configured in the provider record.

- **N-5. Vendor graph API as an OAuth variant.** For Outlook, the source application maintains a separate OAuth configuration that requests additional scopes to access the vendor's proprietary mail-sending API (beyond standard SMTP). This is used because some organizational tenants have disabled SMTP authentication but permit the proprietary API. From the user's perspective, this is invisible — they see the same "Outlook" option and the application silently uses whichever send path the token enables. This is modeled as a provider variant, not a separate feature.

- **N-6. Consent and offline-access parameters.** Gmail requires `prompt=consent` and `access_type=offline` to guarantee a refresh token is returned. Yandex requires `force_confirm=true`. These are provider-specific quirks baked into the provider database, not user-configurable settings. If a future provider has similar requirements, they should be added to that provider's configuration record.

- **N-7. Distribution-channel gating.** OAuth client credentials are registered with each email provider by the application's publisher. Some providers (notably Google) require that the entity responsible for a distribution channel be the same entity that registered the OAuth client. Community-maintained builds (e.g. the F-Droid build in the source application) therefore cannot use the publisher's OAuth credentials. The application must detect this situation at runtime and hide the OAuth option for affected providers, rather than letting the user attempt an OAuth flow that will fail.

- **N-8. Shared mailbox encoding.** For Outlook shared mailboxes, the source application encodes the shared mailbox as part of the username (e.g. `shared@domain\user@domain`) rather than as a separate configuration field. This exploits the fact that the IMAP server interprets the backslash-separated format. It is a provider-specific convention, not a general mechanism.

- **N-9. User-info fallback.** Most providers encode the user's email address in the identity token (as a JWT claim). Mail.ru is an exception: it does not, so the source application makes a separate HTTP request to Mail.ru's user-info endpoint to retrieve the email address after token exchange. The provider configuration flags whether this step is needed (`askAccount` flag). Future providers should be checked for this behavior.
