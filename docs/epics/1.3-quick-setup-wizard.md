# Epic: Quick Setup Wizard

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.3** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Quick setup wizard. Guided onboarding that detects the provider from the email address and fills in IMAP/SMTP servers automatically using a bundled provider database."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Configuring an email account in a traditional email client requires the user to know — or discover — their provider's incoming (IMAP/POP3) and outgoing (SMTP) server hostnames, ports, encryption modes, and authentication method. Most people do not know these details and should not need to. Friction at this stage is the single largest barrier to adoption: a user who cannot get past setup never sees the rest of the application.

The Quick Setup Wizard exists to reduce the setup of a new account to three pieces of information the user *does* know: their **display name**, their **email address**, and their **password**. From the email address alone the wizard infers the provider and fills in all server settings automatically, using a cascading chain of detection strategies: a bundled offline provider database, DNS-based discovery, an online autoconfig database, vendor-specific autodiscovery protocols, and — as a last resort — port scanning. The user presses a single button, waits for a connectivity check, and arrives at a working account.

The wizard is deliberately a separate, streamlined path from the full manual setup. It is optimized for the common case (known providers, standard ports, password or OAuth authentication) and hands off to the manual setup path for everything else.

## 2. Goals

- **G1.** Enable a new user to add a working email account by providing only their name, email address, and password — no server details required.
- **G2.** Automatically detect the correct provider and server settings from the email address with a high success rate across common providers.
- **G3.** Verify connectivity to both the incoming and outgoing servers *before* saving the account, so the user never lands in a broken state.
- **G4.** Provide clear, actionable feedback when auto-detection or connectivity fails, including provider-specific guidance (e.g. "you may need an app-specific password").
- **G5.** Serve as the default and recommended onboarding path for new users — the first thing a new user sees after installation.
- **G6.** Offer a graceful escape hatch to full manual setup at any point, so that edge cases do not leave the user stranded.
- **G7.** Support OAuth-based sign-in for providers that offer it, as a seamless alternative to password entry.

## 3. Non-Goals

- **NG1.** This epic does **not** define the manual setup path (host, port, encryption, auth-method fields). That is feature 1.4.
- **NG2.** This epic does **not** define the OAuth sign-in flow itself. OAuth token acquisition, refresh, and storage are the concern of feature 1.5. The wizard's responsibility is limited to triggering the OAuth flow when the detected provider supports it.
- **NG3.** This epic does **not** prescribe how detected settings are persisted, how accounts are stored, or what synchronization subsystem is started afterward — only that a usable account exists when the wizard completes.
- **NG4.** Editing, re-ordering, or deleting accounts after creation is out of scope.
- **NG5.** Identity management (aliases, multiple From addresses) beyond the single default identity created during setup is out of scope.
- **NG6.** POP3 accounts are not a target of the quick setup path. POP3 entries may exist in the provider database, but the wizard's primary flow targets IMAP + SMTP.

## 4. Glossary

| Term | Definition |
|---|---|
| **Provider** | A recognized email service with known server settings (e.g. Gmail, Outlook, Yahoo, Fastmail). A provider may cover one or many email domains. |
| **Provider database** | A bundled, offline catalogue of providers with their IMAP/SMTP/OAuth settings and domain-matching rules. Ships with the application and is available without network access. |
| **Domain detection** | The process of mapping an email domain (the part after `@`) to a provider entry, using the bundled database, DNS records, an online autoconfig database, vendor autodiscovery, or port scanning. |
| **Connectivity check** | An end-to-end test that connects to the incoming server, authenticates, enumerates folders, then connects to the outgoing server and authenticates — all before the account is saved. |
| **App-specific password** | A secondary password generated by a provider (e.g. Google, Microsoft) when the user has two-factor authentication enabled, because the provider disallows the primary password for third-party clients. |
| **Proprietary provider** | An email provider that does not support standard IMAP/SMTP access (e.g. ProtonMail, Tutanota). These providers are explicitly rejected by the wizard with a clear explanation. |
| **ISPDB** | The Internet Service Provider Database maintained by the Thunderbird project — an online, community-curated autoconfig source queried when the bundled database has no match. |
| **Autodiscovery** | Vendor-specific configuration protocols (e.g. Microsoft Autodiscover) that return server settings for a given email address. |
| **DNS SRV discovery** | RFC 6186-based lookup of `_imap._tcp`, `_imaps._tcp`, `_submission._tcp`, and `_submissions._tcp` DNS SRV records to find server hostnames and ports. |
| **Score** | An internal confidence ranking assigned to each candidate provider result, with bundled-database matches scoring highest and port-scan guesses scoring lowest. Higher-scored candidates are preferred. |
| **Username type** | The form of the username expected by the server: the full email address, the local part only (before `@`), or a provider-specific template. |

## 5. Personas

- **P1. The everyday user.** Uses a mainstream provider (Gmail, Outlook, Yahoo, iCloud). Expects setup to "just work" with their email and password. Has no idea what IMAP or SMTP means.
- **P2. The self-hoster.** Runs their own mail server on a custom domain. The server is correctly configured with DNS SRV or autoconfig records. Expects the wizard to discover settings automatically.
- **P3. The small-provider user.** Uses a regional ISP or niche provider that may not be in the bundled database. Needs the wizard to try harder (MX lookups, ISPDB, port scanning) or to fail gracefully and offer manual setup.
- **P4. The security-conscious user.** Has two-factor authentication enabled and must use app-specific passwords or OAuth. Needs the wizard to tell them *why* their regular password was rejected and *what* to do instead.
- **P5. The multi-account user.** Already has one or more accounts and is adding another. Expects the wizard to be equally fast the second time.

## 6. User Stories

### 6.1 Launching the wizard

> **US-1.** As a new user opening the application for the first time, I want to be presented with the quick setup wizard immediately, so that I can add my first account without navigating through settings.

> **US-2.** As an existing user, I want to reach the quick setup wizard from the account-addition flow, so that adding a second or third account is just as easy as the first.

### 6.2 Entering credentials

> **US-3.** As any user, I want to enter my display name, email address, and password on a single screen, so that setup feels fast and simple.

> **US-4.** As any user, I want the password field to support show/hide toggling, so that I can verify what I typed without re-entering it.

> **US-5.** As any user, if my password contains leading/trailing whitespace or control characters, I want a visible warning before I proceed, so that I can catch copy-paste errors early.

> **US-6.** As any user, I want the wizard to validate that all three fields are non-empty and that the email address is syntactically valid *before* attempting a network check, so that I get instant feedback on obvious mistakes.

### 6.3 Provider detection

> **US-7.** As an everyday user with a well-known provider, I want the wizard to detect my provider from my email address and fill in all server settings without any action on my part, so that I never see a hostname or port number.

> **US-8.** As a self-hoster, I want the wizard to discover my server settings via DNS records, an online autoconfig database, or vendor autodiscovery when my domain is not in the bundled database, so that I do not have to switch to manual setup.

> **US-9.** As any user, if my email domain belongs to a proprietary provider that does not support standard IMAP/SMTP (e.g. ProtonMail, Tutanota), I want the wizard to tell me clearly and immediately that my provider is not compatible, rather than failing with a cryptic connection error.

> **US-10.** As any user, I want to see real-time progress while the wizard is trying to detect my provider (e.g. "Looking up DNS records...", "Checking autoconfig..."), so that I know the application is working and not frozen.

### 6.4 Connectivity check

> **US-11.** As any user, after clicking the check button, I want the wizard to test connectivity to both the incoming (IMAP) and outgoing (SMTP) servers and authenticate with my credentials, so that I know the account will work before it is saved.

> **US-12.** As any user, I want the connectivity check to discover the available server-side folders (Inbox, Sent, Drafts, Trash, Spam, Archive) during the IMAP test, so that the account is correctly configured from the start.

> **US-13.** As any user, if the IMAP or SMTP server presents a certificate that is not trusted by the system trust store, I want the wizard to show me the certificate fingerprint and the hostnames it covers, so that I can make an informed decision about whether to trust it.

> **US-14.** As any user, I want the wizard to try multiple username formats (full email address, local part only, provider-specific template) if the first authentication attempt fails, so that provider quirks are handled transparently.

### 6.5 Error handling and guidance

> **US-15.** As any user, if the connectivity check fails because of an authentication error, I want the wizard to show a clear, non-technical explanation and, when applicable, a link to the provider's documentation on app-specific passwords or two-factor setup.

> **US-16.** As any user, if the provider's entry in the database indicates that an app-specific password is required, I want the wizard to include that hint prominently in the error message, so that I know what to do next.

> **US-17.** As any user using an Outlook, Hotmail, or Live domain, I want the wizard to show provider-specific guidance about modern authentication requirements or app passwords, so that I am not confused by Microsoft-specific restrictions.

> **US-18.** As any user, if auto-detection fails entirely (no settings could be determined), I want to see an informative message and a clear path to manual setup, so that I am not stuck.

> **US-19.** As any user, I want a link to provider-specific documentation or the application's FAQ directly from the error screen, so that I can troubleshoot without leaving the application.

### 6.6 Saving the account

> **US-20.** As any user, after a successful connectivity check, I want to review a summary of the detected configuration — including which server-side folders were found — before the account is saved, so that I can confirm the setup looks correct.

> **US-21.** As any user, when I confirm the save, I want the wizard to create both the incoming account and the outgoing (sending) identity in one step, so that I can immediately send and receive mail.

> **US-22.** As any user adding their first account, I want that account to be designated as the primary account automatically, so that composing a new message works without further configuration.

> **US-23.** As any user, after the account is saved, I want synchronization to begin immediately and the application to navigate me to a view where I can see my incoming mail, so that the setup feels complete.

### 6.7 Re-authentication

> **US-24.** As any user whose existing account has stopped working (e.g. expired password, revoked token), I want the wizard to offer an option to re-authorize the existing account rather than creating a duplicate, so that I keep my folder structure, rules, and settings.

> **US-25.** As any user re-authorizing, I want only the credentials to be updated — not the folders, sync settings, or identity — so that my existing configuration is preserved.

### 6.8 Escape to manual setup

> **US-26.** As any user, I want a clearly visible button to switch to full manual setup from the wizard at any time, so that I am never forced to rely on auto-detection.

### 6.9 Privacy

> **US-27.** As any user, I want the wizard to display links to the relevant privacy policies (the application's own and any third-party autoconfig service used) before I begin, so that I understand what data is shared during detection.

### 6.10 Provider-specific behavior

> **US-28.** As a user of a provider that supports OAuth, I want the wizard to offer OAuth sign-in as an option after detection, so that I can authenticate without entering my password into the application.

> **US-29.** As a user of a provider with known quirks (e.g. keep-alive requirements, TLS version ceilings, partial-fetch limitations, NOOP-based idle), I want the wizard to apply those provider-specific tuning parameters automatically when the account is created, so that the account works reliably without me knowing about these details.

## 7. Functional Requirements

### 7.1 Wizard entry point

- **FR-1.** On first launch with no configured accounts, the application shall present the quick setup wizard as the default and only initial screen.
- **FR-2.** When the user initiates "Add account" from settings or the navigation pane, the quick setup wizard shall be the default path, with manual setup available as a secondary option.

### 7.2 Input fields and validation

- **FR-3.** The wizard shall present three input fields on a single screen: display name, email address, and password.
- **FR-4.** The password field shall support visibility toggling (show/hide).
- **FR-5.** Before initiating any network operation, the wizard shall validate:
  - (a) Display name is non-empty.
  - (b) Email address is non-empty and matches a standard email-address pattern.
  - (c) Password is non-empty.
- **FR-6.** If the password contains leading/trailing whitespace or non-printable characters, the wizard shall display a warning (not a blocking error) to alert the user to possible copy-paste artifacts.
- **FR-7.** The wizard shall require network connectivity before starting the check and shall display a clear message if the network is unavailable.

### 7.3 Provider detection

- **FR-8.** The application shall ship a bundled provider database containing at least 150 provider entries, each with: a display name, one or more domain-matching patterns, incoming-server settings (host, port, encryption mode), outgoing-server settings (host, port, encryption mode), and optional provider-specific flags.
- **FR-9.** The provider database shall be available offline and shall not require a network connection to consult.
- **FR-10.** The wizard shall detect the provider by applying the following strategies in order of preference and confidence:
  1. **Bundled database domain match** — match the email domain against the provider entries' domain patterns.
  2. **DNS NS record match** — look up the domain's name-server records and match against known provider patterns; if a match is found, attempt RFC 6186 SRV discovery first, falling back to the matched provider entry.
  3. **DNS MX record match** — look up the domain's MX records and match against known provider MX patterns or root-domain equivalences.
  4. **RFC 6186 DNS SRV discovery** — query `_imaps._tcp`, `_imap._tcp`, `_submissions._tcp`, and `_submission._tcp` SRV records for the domain.
  5. **ISPDB (online autoconfig database)** — query the Thunderbird project's Internet Service Provider Database for the domain.
  6. **Vendor autodiscovery** — attempt vendor-specific autodiscovery protocols (e.g. Microsoft Autodiscover) for the domain.
  7. **Port scanning** — as a last resort, attempt connections to well-known IMAP/SMTP ports on the MX host or the domain itself.
- **FR-11.** Each detection strategy shall produce a candidate with a confidence score. When multiple candidates are found, the wizard shall prefer the highest-scoring candidate. Bundled-database matches shall always score higher than network-discovered candidates.
- **FR-12.** When a network-discovered candidate's server hostname matches a bundled provider entry, the wizard shall replace the network-discovered settings with the bundled entry's settings (preserving the discovery score), so that provider-specific flags and documentation are available.
- **FR-13.** If the email domain belongs to a known proprietary provider that does not support standard IMAP/SMTP, the wizard shall immediately reject the address with a clear, user-friendly message explaining the incompatibility — without attempting any network-based detection.
- **FR-14.** The wizard shall display real-time progress during detection, indicating which strategy is currently being attempted (e.g. "Looking up DNS...", "Checking autoconfig...", "Scanning ports on [host]...").

### 7.4 Provider database fields

- **FR-15.** Each provider entry in the bundled database shall support at minimum the following fields:
  - (a) Unique identifier and display name.
  - (b) One or more domain-matching patterns (regular expressions).
  - (c) One or more MX-matching patterns (for providers whose customers use custom domains).
  - (d) Incoming server: hostname, port, encryption mode (implicit TLS or STARTTLS).
  - (e) Outgoing server: hostname, port, encryption mode.
  - (f) Username type: full email address, local part only, or a provider-specific template.
  - (g) Keep-alive interval (override for the default poll/idle timer).
  - (h) Whether idle keep-alive requires NOOP commands.
  - (i) Whether partial message fetching is supported.
  - (j) Maximum TLS version (for providers with known TLS limitations).
  - (k) Whether an app-specific password is typically required.
  - (l) A link to provider-specific documentation or FAQ entry.
  - (m) Localized documentation snippets (e.g. instructions for enabling app passwords).
  - (n) OAuth configuration (if the provider supports OAuth): authorization endpoint, token endpoint, scopes, redirect URI, and provider-specific parameters.
  - (o) Display order / priority for UI presentation.
  - (p) An enabled/disabled flag (so that entries can be shipped but deactivated).
- **FR-16.** The application shall support loading an additional, user-supplied provider file that augments or overrides the bundled database, so that administrators or power users can add custom providers without modifying the application.

### 7.5 Connectivity check

- **FR-17.** The wizard shall perform a connectivity check consisting of:
  1. Connect to the incoming (IMAP) server using the detected settings.
  2. Authenticate using the user's credentials.
  3. Enumerate the available server-side folders, detecting system-folder roles (Inbox, Sent, Drafts, Trash, Spam, Archive).
  4. Connect to the outgoing (SMTP) server using the detected settings.
  5. Authenticate using the user's credentials.
  6. Query the outgoing server's maximum message size, if advertised.
- **FR-18.** If the provider entry specifies multiple possible username types, the wizard shall try each in turn until authentication succeeds or all have been exhausted.
- **FR-19.** If the server presents a certificate not trusted by the system trust store, the wizard shall:
  - (a) Display the certificate's fingerprint and the DNS names it covers.
  - (b) Visually highlight any mismatch between the certificate's DNS names and the server hostname.
  - (c) Allow the user to accept the certificate (by fingerprint) for this account.
  - (d) If the user accepts, retry the connection using the accepted fingerprint.
- **FR-20.** The accepted certificate fingerprint shall be stored with the account so that future connections to the same server are trusted without re-prompting.

### 7.6 Error handling

- **FR-21.** On authentication failure, the wizard shall display a user-friendly error message. If the provider entry indicates that app-specific passwords are typically required, the error message shall include that hint and a link to the provider's app-password setup page (if available).
- **FR-22.** For Outlook / Hotmail / Live domains, the wizard shall display a provider-specific warning about modern authentication requirements.
- **FR-23.** On detection failure (no settings could be determined), the wizard shall display a message explaining that the provider could not be auto-detected and shall offer a one-click path to manual setup.
- **FR-24.** On any error, the wizard shall offer a link to provider-specific documentation (if available in the provider entry) and a general support/FAQ link.
- **FR-25.** Error messages shall be non-technical. Raw server error strings shall not be shown to the user by default (a "show details" expansion may be offered for power users).

### 7.7 Account creation

- **FR-26.** On successful connectivity check, the wizard shall present a review screen showing:
  - (a) The provider name and account name.
  - (b) The detected system folders (Inbox, Drafts, Sent, Trash, Spam, Archive) with indicators for which were found.
  - (c) An editable account name field.
- **FR-27.** On confirmation, the wizard shall create:
  - (a) An incoming-mail account with the detected IMAP settings, credentials, and provider-specific tuning parameters (keep-alive interval, NOOP flag, partial-fetch flag, TLS ceiling, certificate fingerprint if accepted).
  - (b) A sending identity with the detected SMTP settings, the user's display name, email address, credentials, and the outgoing server's maximum message size.
- **FR-28.** The first account created shall be designated as the primary account. Subsequent accounts shall not override the primary designation.
- **FR-29.** If any standard system folders (Drafts, Sent, Archive, Trash, Spam) are missing on the server, the wizard shall create them in the account's namespace so that the application has a complete folder structure from the start.
- **FR-30.** The wizard shall configure default synchronization for the new account's system folders:
  - (a) **Inbox**: synchronize and download messages, with push/idle if the server supports it.
  - (b) **Drafts, Sent, Archive**: synchronize and download messages, polled periodically.
  - (c) **Trash, Spam**: synchronize (polled) but do not download message bodies by default.
  - (d) **User-created folders**: do not synchronize by default.
- **FR-31.** After saving, the wizard shall trigger an immediate synchronization cycle and navigate the user to a view where incoming messages will appear.

### 7.8 Re-authentication of existing accounts

- **FR-32.** The wizard shall offer a visible option to re-authorize an existing account (e.g. "Authorize existing account again") rather than creating a new one.
- **FR-33.** When re-authorizing, the wizard shall update only the credentials (password or OAuth token) and the synchronization-enabled flag. Folder structure, sync settings, identities, rules, and all other account properties shall be preserved.
- **FR-34.** The wizard shall match an existing account by username and incoming-server protocol type.

### 7.9 Manual-setup escape hatch

- **FR-35.** A clearly labeled button to switch to manual setup shall be visible on the wizard's main screen at all times.
- **FR-36.** Switching to manual setup shall carry over any information the user has already entered (name, email, password) so that the user does not have to re-type it.

### 7.10 Privacy

- **FR-37.** The wizard's initial screen shall display links to the application's privacy policy and to the privacy policy of any third-party autoconfig service used during detection (e.g. the Thunderbird ISPDB / Mozilla Privacy Policy).
- **FR-38.** The wizard shall not transmit the user's password to any third-party service. Passwords are sent only to the user's own mail server during the connectivity check.

### 7.11 OAuth integration

- **FR-39.** When the detected provider entry includes an OAuth configuration and OAuth is enabled for that provider, the wizard shall offer OAuth-based sign-in as an option.
- **FR-40.** Providers known to support OAuth include at minimum: Gmail, Outlook/Office 365, Yahoo, AOL, Yandex, Mail.ru, and Fastmail.
- **FR-41.** When OAuth is used, the wizard shall store the resulting tokens instead of the user's password and shall configure the account for token-based authentication.

## 8. Non-Functional Requirements

- **NFR-1. Speed.** The entire wizard flow — from entering credentials to a saved, syncing account — shall complete in under 30 seconds for a provider present in the bundled database on a broadband connection. Bundled-database lookups themselves shall be effectively instantaneous (well under one second).
- **NFR-2. Offline provider lookup.** The bundled provider database shall be usable without any network access, so that domain-to-provider matching and pre-populating settings works even when connectivity is intermittent. The connectivity check itself obviously requires a network.
- **NFR-3. Detection success rate.** For the top 20 email providers by global market share, the wizard shall detect correct settings from the bundled database alone, without relying on network-based discovery.
- **NFR-4. Provider database size.** The bundled database shall contain at least 150 provider entries, covering major global and regional providers.
- **NFR-5. Extensibility.** The provider database format shall be documented and shall support user-supplied additions without modifying the application binary, so that corporate or niche deployments can add their own providers.
- **NFR-6. Localization.** Error messages, hints, and provider-specific documentation snippets shall support localization into multiple languages.
- **NFR-7. Security.** The wizard shall never transmit the user's password to any server other than the user's own IMAP and SMTP servers. Certificate validation shall follow the system trust store by default; exceptions shall require explicit user consent.
- **NFR-8. Accessibility.** All wizard screens, buttons, progress indicators, error messages, and certificate-review dialogs shall be navigable by keyboard and screen reader, and shall respect the application-wide theme.

## 10. Open Questions

- **OQ-1. Provider database update cadence.** The bundled database is a point-in-time snapshot. Should the application check for provider-database updates independently of application updates? If so, from what source and how often? The source application bundles the list at build time.
- **OQ-2. Port-scan safety.** The last-resort port-scanning strategy sends the user's password to whatever server answers on standard IMAP/SMTP ports. The source application has a disabled-by-default code path that would also try connecting to the mail-exchange host's TLS port (443) to discover hostnames from the certificate. This path is explicitly flagged as "unsafe: the password could be sent to an unrelated email server." The current wizard does not use this path, but the concern extends to any port-scan-based strategy. Should the wizard warn the user before attempting port scanning, or should port scanning require explicit opt-in?
- **OQ-3. Proprietary-provider list maintenance.** The list of proprietary domains (ProtonMail, Tutanota, etc.) that are rejected immediately is hardcoded. Should this list be part of the provider database (so it can be updated) or remain a static deny-list?
- **OQ-4. OAuth provider completeness.** Some providers have OAuth entries that are marked "debug-only" (enabled only in development builds). The criteria for promoting a provider's OAuth support from debug to production are unclear. Which providers should ship with OAuth enabled?
- **OQ-5. Certificate-acceptance UX.** The source application allows accepting an untrusted certificate by fingerprint, which is stored permanently. Should there be an expiry or periodic re-prompt for accepted certificates, to guard against silently trusting a compromised cert indefinitely?
- **OQ-6. Multiple-candidate disambiguation.** When multiple detection strategies produce different candidates (e.g. DNS SRV points to one server, ISPDB points to another), the wizard currently selects by score. Should the user be shown multiple options and asked to choose, or is highest-score-wins the right policy?
- **OQ-7. Synchronization defaults for re-authorized accounts.** FR-33 says re-authorization preserves existing settings, but if the user had previously disabled synchronization (which is re-enabled during re-auth), this may surprise the user. Should re-authorization prompt for whether to re-enable sync?
- **OQ-8. Missing-folder creation policy.** The wizard creates missing system folders (Drafts, Sent, etc.) on the server. Some servers may not permit folder creation, or the user may prefer server-side folder names in a different language. Should folder creation be optional or configurable?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Three fields, not two.** The wizard collects display name alongside email and password. Many competing wizards defer the display name to after account creation, but collecting it upfront means the sending identity is fully configured at the end of the wizard — the user can send mail immediately without an additional settings trip.
- **N-2. Check before save.** The wizard performs a full connectivity check (IMAP authenticate + folder enumerate + SMTP authenticate) *before* the account is persisted. This "validate-then-save" pattern means the user never encounters a broken account in their account list. The trade-off is that the wizard is slightly slower than a "save-then-background-check" approach, but the reliability benefit is significant.
- **N-3. Cascading detection with score-based ranking.** The detection chain is ordered by confidence: bundled database > DNS NS > DNS MX > RFC 6186 SRV > ISPDB > vendor autodiscovery > port scan. Each level produces a numeric score, and the wizard always prefers the highest-scored candidate. Bundled-database matches always outrank network-discovered ones because the bundled entries carry provider-specific tuning (keep-alive, NOOP, TLS ceiling, documentation) that network discovery cannot provide.
- **N-4. Bundled database replaces network-discovered settings.** When a network-discovered candidate's hostname matches a bundled provider entry, the wizard *replaces* the network-discovered settings with the bundled entry. This ensures that provider-specific flags (e.g. partial-fetch disabled for Outlook, NOOP keep-alive for Yahoo) are applied even when the domain was discovered via MX or ISPDB rather than direct domain match.
- **N-5. Proprietary-provider rejection.** Providers like ProtonMail and Tutanota are rejected before any network request is made. This avoids wasting the user's time on a detection chain that would inevitably fail, and avoids the confusing error messages that would result from trying to connect to servers that do not speak standard IMAP.
- **N-6. System-folder creation.** The wizard creates missing system folders (Drafts, Sent, Archive, Trash, Spam) on the server using the IMAP namespace prefix. This is aggressive but pragmatic: without a Sent folder, sent-message filing fails silently; without a Drafts folder, draft saving fails. Creating them ensures the application works correctly from the start. The folder names follow the server's namespace conventions (e.g. "[Gmail]/" prefix for Gmail).
- **N-7. Provider-specific tuning is invisible.** Parameters like keep-alive interval, NOOP mode, partial-fetch support, maximum TLS version, and IP-preference are applied silently based on the provider entry. The user never sees these settings during quick setup. They exist to work around known server quirks and are the reason the bundled database is preferred over generic auto-discovery.
- **N-8. Re-authorization vs. new account.** The wizard includes an explicit "re-authorize existing account" toggle. Without this, a user whose password has changed would have to delete and re-create the account, losing all folder configuration, rules, and local messages. The re-auth path updates only credentials and the sync-enabled flag, preserving everything else.
