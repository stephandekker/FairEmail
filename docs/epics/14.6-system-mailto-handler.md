# Epic: System mailto: Handler

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **14.6** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"System mailto: handler — registration."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Desktop email clients are expected to integrate with the operating system so that clicking a `mailto:` link in a web browser, document viewer, or any other application opens the email client's compose window with the relevant fields pre-populated. This integration is a baseline expectation — without it, the email client feels disconnected from the rest of the desktop environment.

The System mailto: Handler feature covers two concerns:

1. **Registration** — the application advertises itself to the desktop environment as capable of handling `mailto:` URIs, so the system routes those URIs to it instead of (or in addition to) other installed email clients.
2. **URI consumption** — when a `mailto:` URI arrives, the application parses it according to RFC 6068 and opens a compose window with the extracted fields pre-filled, ready for the user to review and send.

The feature must work transparently whether the application is already running or needs to be launched cold, and regardless of how many accounts the user has configured.

## 2. Goals

- **G1.** Allow the application to register itself as the system's preferred handler for `mailto:` URIs, so that the user can designate it as the default email client for link handling.
- **G2.** Parse incoming `mailto:` URIs fully and correctly per RFC 6068, pre-populating all supported compose fields.
- **G3.** Open a compose window immediately upon receiving a `mailto:` URI, with minimal user friction.
- **G4.** Select a sensible default sending identity/account when the URI does not specify one, following the same rules as composing from the Unified Inbox.
- **G5.** Handle edge cases gracefully: no configured accounts, malformed URIs, cold-start scenarios, and concurrent compose requests.

## 3. Non-Goals

- **NG1.** This epic does **not** cover outbound sharing of content (e.g. "Share via email" from another application with file attachments attached externally). That is a distinct sharing/send integration concern.
- **NG2.** This epic does **not** cover the `attachment` or `attach` parameter sometimes seen in non-standard `mailto:` extensions. Only parameters defined in RFC 6068 are in scope.
- **NG3.** This epic does **not** define how the compose window itself works — only that it opens with the correct fields pre-filled. The compose experience is governed by the Rich Text Editor epic (4.1) and related composition epics.
- **NG4.** This epic does **not** cover registration as a handler for other URI schemes (e.g. `mid:`, `cid:`).
- **NG5.** This epic does **not** specify the mechanism by which the system associates the application with `mailto:` URIs — only that the application provides the necessary metadata for the system to do so.

## 4. Glossary

| Term | Definition |
|---|---|
| **mailto: URI** | A URI conforming to RFC 6068 that encodes an email composition request: one or more recipients, and optionally a subject, body, CC, BCC, and additional headers. |
| **URI handler / default application** | The application the operating system invokes when the user activates a `mailto:` link in any context (browser, file manager, document, terminal, etc.). |
| **Compose window** | The application's interface for drafting and sending a new email message. |
| **Primary account** | The account designated by the user as the fallback for any composition initiated from a context where no specific account is implied. |
| **Identity** | A configured sender profile (email address, display name) associated with an account. A single account may have multiple identities. |
| **Cold start** | The scenario where the application is not currently running and must be launched by the system in order to handle the URI. |
| **RFC 6068** | The IETF standard defining the `mailto:` URI scheme, superseding RFC 2368. |

## 5. Personas

- **P1. The browser user.** Clicks `mailto:` links on web pages and expects the email client to open a compose window with the recipient already filled in.
- **P2. The document worker.** Clicks email addresses embedded in PDFs, office documents, or help dialogs, expecting seamless handoff to the compose window.
- **P3. The multi-account user.** Has several configured accounts and wants the system to pick a reasonable default when a `mailto:` link provides no hint about which account to use.
- **P4. The power user.** Encounters complex `mailto:` links (multiple recipients, pre-filled subject and body) generated by web applications, ticketing systems, or mailing-list interfaces and expects all fields to transfer correctly.
- **P5. The new user.** Has installed the application but not yet configured an account; clicks a `mailto:` link and needs clear feedback that setup is required before composing.

## 6. User Stories

### 6.1 Registration

> **US-1.** As any user, I want the application to declare itself as a candidate handler for `mailto:` URIs during or after installation, so that the operating system can offer it as a choice (or use it as the default) when a `mailto:` link is activated.

> **US-2.** As any user, I want to be able to set this application as my default `mailto:` handler through the system's default-application settings, so that I do not need to configure anything inside the email client itself to enable this.

> **US-3.** As any user, I want the application to remain registered as a `mailto:` handler across application upgrades and restarts, so that my default-application preference is durable.

### 6.2 Basic URI handling

> **US-4.** As a browser user, when I click a `mailto:` link that contains only a recipient address (e.g. `mailto:alice@example.com`), I want the application to open a compose window with that address in the To field and all other fields empty, so that I can start typing immediately.

> **US-5.** As a power user, when I click a `mailto:` link that includes a subject and body (e.g. `mailto:alice@example.com?subject=Hello&body=Hi%20there`), I want the compose window to open with To, Subject, and Body all pre-filled, so that I only need to review and send.

> **US-6.** As a power user, when I click a `mailto:` link with CC and BCC recipients, I want those fields to be pre-populated in the compose window, so that the link author's intended distribution is respected.

> **US-7.** As a power user, when I click a `mailto:` link with multiple To recipients (comma-separated per RFC 6068), I want all recipients to appear in the To field, so that group-addressed links work correctly.

### 6.3 Account and identity selection

> **US-8.** As a multi-account user, when a `mailto:` link provides no hint about which account to use, I want the application to default to my primary account's primary identity, so that I get a predictable From address without manual intervention.

> **US-9.** As a multi-account user, I want to be able to change the From identity in the compose window after it opens from a `mailto:` link, so that I retain full control over which account sends the message.

> **US-10.** As a multi-account user, when the application can infer the appropriate account from context (e.g. a custom header hint in the URI, or a prior correspondence history with the recipient), I want it to pre-select that account, so that the From address matches the conversation context.

### 6.4 Error and edge-case handling

> **US-11.** As a new user who has not yet configured any account, when I click a `mailto:` link I want the application to inform me clearly that no sending account is available and guide me toward account setup, rather than silently failing or crashing.

> **US-12.** As any user, when a `mailto:` URI is malformed or contains unrecognized parameters, I want the application to open a compose window with whatever fields it could parse successfully, ignoring the rest gracefully, so that partial information is not lost.

> **US-13.** As any user, when I click a `mailto:` link while the application is not running, I want the application to launch and open the compose window without requiring me to navigate through the main interface first, so that the experience is seamless.

### 6.5 Security and sanitization

> **US-14.** As any user, I want the application to sanitize recipient addresses extracted from `mailto:` URIs (removing invisible characters, normalizing whitespace), so that visually-deceptive addresses do not trick me into emailing the wrong person.

> **US-15.** As any user, I want the body text from a `mailto:` URI to be treated as plain text (not as executable markup), so that a malicious link cannot inject formatted content or scripts into my compose window.

> **US-16.** As any user, I want the application to clearly display all pre-filled fields (especially BCC) so that I can review what the link author intended before sending, preventing hidden-recipient surprises.

### 6.6 Convenience integration

> **US-17.** As any user, I want a "send to self" shortcut that composes a message addressed to my own primary email address, so that I can quickly send myself notes or links without manually entering my address.

## 7. Functional Requirements

### 7.1 Registration

- **FR-1.** The application shall declare support for handling `mailto:` URIs in its system-integration metadata, so that the operating system's default-application mechanism can discover it.
- **FR-2.** The declaration shall cover at minimum the `mailto:` scheme with no restriction on host or path, so that all valid `mailto:` URIs are routable to the application.
- **FR-3.** Registration shall persist across application restarts and upgrades without user re-intervention.

### 7.2 URI parsing

- **FR-4.** The application shall parse `mailto:` URIs conforming to RFC 6068, extracting at minimum: To (one or more addresses), CC, BCC, Subject, and Body.
- **FR-5.** Multiple To addresses separated by commas shall each be placed into the To field as separate recipients.
- **FR-6.** Percent-encoded characters in all fields shall be decoded correctly per RFC 3986.
- **FR-7.** The Body parameter shall be treated as plain text. Line breaks encoded as `%0D%0A` or `%0A` shall be rendered as line breaks in the compose window.
- **FR-8.** Unrecognized or unsupported header parameters in the URI shall be silently ignored; their presence shall not prevent the compose window from opening with the fields that were successfully parsed.
- **FR-9.** An empty or absent `mailto:` path (e.g. `mailto:?subject=Test`) shall open a compose window with an empty To field and other parsed fields pre-filled.

### 7.3 Compose window behavior

- **FR-10.** Upon receiving a valid `mailto:` URI, the application shall open a compose window (or bring the application to the foreground with a new compose window) within a reasonable time, regardless of whether the application was already running.
- **FR-11.** The compose window shall be pre-populated with all fields extracted from the URI: To, CC, BCC, Subject, and Body.
- **FR-12.** All pre-filled fields shall be editable by the user before sending.
- **FR-13.** The compose window opened via `mailto:` shall behave identically to a compose window opened from within the application in all other respects (signature insertion, draft saving, attachment support, identity switching, etc.).

### 7.4 Account and identity selection

- **FR-14.** When the URI provides no account hint, the application shall select the user's primary account and its primary identity as the default From address.
- **FR-15.** When the URI contains a recognized hint indicating a specific account (e.g. a custom header parameter), the application shall honor that hint if the referenced account exists and is composable.
- **FR-16.** When smart identity matching is enabled and the application has prior correspondence history with the To recipient, the application may suggest the identity that previously communicated with that recipient.
- **FR-17.** The user shall always be able to change the selected identity/account from within the compose window before sending.

### 7.5 No-account error handling

- **FR-18.** If no accounts are configured (or no account has a composable identity), the application shall display a clear, non-technical error message indicating that an account must be set up before email can be sent.
- **FR-19.** The error state shall not crash the application or leave an unusable blank window.

### 7.6 Sanitization

- **FR-20.** Email addresses extracted from the URI shall be sanitized: leading/trailing whitespace removed, invisible Unicode characters (zero-width spaces, etc.) stripped.
- **FR-21.** Body text shall be escaped to prevent interpretation as markup. Any HTML or script content in the body parameter shall be rendered as literal text, not executed or formatted.

### 7.7 Send-to-self shortcut

- **FR-22.** The application shall provide a quick-action or shortcut that composes a message with the user's own primary email address pre-filled in the To field, bypassing the need to type or paste the address.
- **FR-23.** The send-to-self action shall use the user's primary identity and follow the same compose flow as a standard `mailto:` invocation.

## 8. Non-Functional Requirements

- **NFR-1. Launch latency.** When the application is already running, a `mailto:` URI shall result in a visible compose window within one second. On cold start, the compose window shall appear within the application's normal startup time plus one second.
- **NFR-2. Reliability.** The handler shall not crash or hang on any syntactically valid URI, regardless of field length or character encoding.
- **NFR-3. Concurrency.** If multiple `mailto:` URIs are received in rapid succession (e.g. user clicks several links), each shall open its own compose window or tab without losing any.
- **NFR-4. Security.** No URI content shall be executed as code or markup. The handler shall be resistant to URI-based injection attacks.
- **NFR-5. Accessibility.** The compose window opened via `mailto:` shall be fully keyboard-navigable and screen-reader compatible, consistent with the rest of the application.
- **NFR-6. Privacy.** Handling a `mailto:` URI shall not trigger any network requests beyond those required for normal compose operations (e.g. fetching a signature from a server). The URI content shall not be logged or transmitted to third parties.
- **NFR-7. Graceful degradation.** If the URI contains parameters the application does not support (e.g. a hypothetical `attachment` parameter), the unsupported parameters shall be silently dropped without affecting the supported fields.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|

## 10. Open Questions

- **OQ-1. In-Reply-To header.** The source application supports a non-standard `in-reply-to` header parameter in `mailto:` URIs, allowing a link to create a reply-context. Should the desktop application support this extension, or treat it as an unrecognized header and ignore it? If supported, how should the compose window indicate that the message is "in reply to" something when the original message may not be in the local store?
- **OQ-2. Account-hint parameter.** The source application recognizes a custom `aid` header parameter that pre-selects a specific account. This is an internal mechanism with no cross-application standard. Should the desktop application support an equivalent (e.g. a configuration key or identity-email-based hint), or rely solely on smart identity matching?
- **OQ-3. Maximum URI length.** RFC 6068 places no hard limit on URI length, but operating systems and browsers may truncate very long URIs. Should the application document a maximum supported length, or attempt to handle arbitrarily long URIs?
- **OQ-4. Default vs. opt-in registration.** Should the application register itself as a `mailto:` handler automatically on first run (and let the user remove it via system settings), or should it prompt the user to opt in? The source application registers automatically.
- **OQ-5. XDG compliance.** On Linux desktops, `mailto:` handler registration typically involves a `.desktop` file with a `MimeType=x-scheme-handler/mailto` entry and `xdg-mime` / `xdg-settings` integration. The specific mechanism is an implementation detail, but the question of whether the application should *actively* call `xdg-settings set default-url-scheme-handler mailto` on first run (claiming default status) or passively declare support and let the user choose is a UX decision.
- **OQ-6. Body format ambiguity.** RFC 6068 specifies that the body parameter is plain text. However, some web applications generate `mailto:` links with HTML in the body. Should the application offer a user preference to interpret body content as HTML in addition to the default plain-text treatment, or always enforce plain-text rendering?

## 11. Design Notes & Rationale

- **N-1. Registration is passive metadata.** The source application registers its `mailto:` handling capability declaratively (in its manifest), not by actively claiming a system default. This is the correct model for a well-behaved desktop application: declare capability, let the user or system choose. The application should not forcibly override another application's default status.
- **N-2. Cold-start compose.** The source application supports opening directly into compose mode when launched via a `mailto:` URI — the user does not land on the inbox and then get a compose window. This "compose-first" cold start path is essential for the feature to feel integrated rather than bolted on.
- **N-3. Sanitization of invisible characters.** The source application explicitly strips zero-width spaces from email addresses, noting compatibility with specific applications (e.g. Discord) that sometimes insert them. This is a pragmatic defense against copy-paste artifacts and homograph-style attacks.
- **N-4. Body as plain text with line-break preservation.** The source application converts body text to its internal compose format while preserving line breaks and escaping any markup. This ensures that `mailto:` links generated by web applications (which often include multi-line instructions) render readably, while preventing injection.
- **N-5. Send-to-self as a distinct entry point.** The source application provides a separate shortcut for "send to self" that pre-fills the user's own address. This is a convenience feature layered on top of the `mailto:` handler infrastructure — it uses the same compose path but with a hardcoded recipient. It exists because emailing oneself is a common quick-capture pattern.
- **N-6. Identity selection follows unified-inbox rules.** The account/identity selection logic for `mailto:` compose is the same as for composing from the Unified Inbox: primary account first, then smart matching if enabled. This consistency means the user only learns one mental model for "where does my outgoing mail come from when context is ambiguous?"
