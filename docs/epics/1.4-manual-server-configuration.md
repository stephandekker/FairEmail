# Epic: Manual Server Configuration

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.4** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Full IMAP/POP3 + SMTP setup with host, port, encryption (SSL/TLS, STARTTLS, plain), authentication method, certificate selection, and 'Test connection' before saving."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Not every email provider is known in advance. Users with self-hosted mail servers, corporate on-premise infrastructure, or niche providers need the ability to configure every aspect of the server connection by hand. Even users whose provider *is* in the bundled database sometimes need to override auto-detected settings — for example, to point at an internal hostname, use a non-standard port, select a specific encryption mode, or supply a client certificate for mutual TLS.

The Manual Server Configuration feature is the application's full-control path for creating and editing mail accounts. It exposes every parameter the application needs to connect to a mail server — protocol, host, port, encryption, authentication method, credentials, and client certificate — for both the inbound (IMAP or POP3) and outbound (SMTP) legs of the connection. It also provides a **Test Connection** operation that validates the configuration against the live server *before* the account is persisted, so that the user gets immediate, actionable feedback on misconfiguration.

This feature is complementary to the Quick Setup Wizard (feature 1.3). Where the wizard fills fields automatically from a provider database, Manual Server Configuration lets the user fill or override every field themselves. The two flows may also cooperate: the manual screen offers a domain-based auto-config lookup that pre-populates fields, which the user can then adjust before testing and saving.

## 2. Goals

- **G1.** Allow any user to create a fully functional email account by supplying raw server parameters, without relying on a bundled provider database or auto-discovery.
- **G2.** Expose every connection-relevant parameter (protocol, host, port, encryption mode, authentication type, credentials, client certificate, security options) for both inbound and outbound servers.
- **G3.** Provide a non-destructive "Test Connection" operation that validates the configuration against the live server and reports success, capabilities, or failure with actionable diagnostics.
- **G4.** Support IMAP, POP3, and SMTP protocols with their respective defaults, port conventions, and protocol-specific options.
- **G5.** Let the user edit any existing account's server configuration at any time after creation, using the same interface.
- **G6.** Offer a domain-based auto-config shortcut within the manual screen that pre-fills fields from provider metadata or DNS-based discovery, while keeping all fields user-editable.
- **G7.** Guide the user from inbound configuration through to outbound (SMTP) identity creation in a natural, sequential flow.

## 3. Non-Goals

- **NG1.** This epic does **not** define the Quick Setup Wizard (feature 1.3) — only the manual path. The auto-config shortcut within the manual screen is in scope; the separate guided wizard is not.
- **NG2.** This epic does **not** define OAuth sign-in flows (feature 1.5). OAuth-authenticated accounts use a different entry point; manual configuration handles password-based and client-certificate-based authentication.
- **NG3.** This epic does **not** specify per-folder sync settings, notification settings, or folder management — only the server connection and credential parameters needed to establish the account.
- **NG4.** This epic does **not** define identity fields unrelated to SMTP connectivity (signatures, default encryption, reply-to, CC/BCC defaults, sender extras). Those belong to the identity/compose features.
- **NG5.** This epic does **not** prescribe the bundled provider database format or its update mechanism — only that manual configuration can optionally consume provider metadata to pre-fill fields.
- **NG6.** Global TLS-hardening preferences (minimum protocol version, cipher suites, certificate transparency) are application-wide settings, not per-account manual-configuration concerns. This epic assumes they exist but does not define them.

## 4. Glossary

| Term | Definition |
|---|---|
| **Account** | A configured connection to a single mail provider, with credentials, server settings, and its own folder tree. Each account has exactly one inbound (IMAP or POP3) connection. |
| **Identity** | A sending configuration associated with an account. Contains an email address, display name, and its own SMTP connection settings. An account may have multiple identities. |
| **Inbound connection** | The IMAP or POP3 connection used to retrieve mail from the server. |
| **Outbound connection** | The SMTP connection used to send mail. Configured per identity, not per account. |
| **Encryption mode** | The TLS strategy for the connection: **Implicit SSL/TLS** (connection is encrypted from the first byte), **STARTTLS** (connection begins in cleartext and upgrades via the STARTTLS command), or **None** (no encryption). |
| **Client certificate** | A user-supplied X.509 certificate and private key used for mutual TLS authentication (the server verifies the client's identity in addition to the client verifying the server's). |
| **Certificate pinning** | Accepting a specific server certificate by its fingerprint, bypassing the system trust store. Used when the server presents a self-signed or otherwise untrusted certificate that the user explicitly approves. |
| **DANE** | DNS-based Authentication of Named Entities (RFC 6698). Verifies the server's TLS certificate against TLSA records published in DNSSEC-signed DNS. |
| **DNSSEC** | DNS Security Extensions. Cryptographically validates DNS responses to prevent spoofing. A prerequisite for DANE. |
| **Test Connection** | A user-initiated operation that connects to the configured server, authenticates, and reports success or failure without persisting any data. |
| **Auto-config** | A domain-based lookup that attempts to discover server settings from a bundled provider database, DNS SRV records (RFC 6186), MX records, or well-known auto-configuration endpoints, and pre-fills the manual fields with the results. |
| **Realm** | An optional authentication scope parameter used by some SASL mechanisms (e.g. NTLM domain). |
| **Provider** | A known email service whose connection parameters are stored in the bundled provider database. Selecting a provider pre-fills host, port, and encryption fields. |

## 5. Personas

- **P1. The self-hoster.** Runs their own mail server. Knows the hostname, port, and encryption requirements. May use a self-signed certificate. Needs full control over every field.
- **P2. The corporate user.** Their IT department has provided specific IMAP/SMTP settings, possibly with a client certificate for mutual TLS. Needs to enter them exactly as specified.
- **P3. The power user.** Has several accounts across different providers. Comfortable with server settings but appreciates auto-config as a time-saver, and expects to be able to override any auto-filled value.
- **P4. The newcomer.** Less familiar with email protocols. Benefits from sensible defaults (port auto-fill on encryption change, domain-based auto-config) and clear error messages from the test-connection step.
- **P5. The migrating user.** Switching from another email client. Has their server settings written down or exported. Needs to enter them once and confirm they work.

## 6. User Stories

### 6.1 Account creation — inbound server

> **US-1.** As any user, I want to create a new account by selecting the protocol (IMAP or POP3) and entering the server hostname, port, encryption mode, username, and password, so that the application can connect to my mail server.

> **US-2.** As any user, when I select an encryption mode (SSL/TLS, STARTTLS, or None), I want the port field to auto-fill with the conventional default for that protocol and encryption combination, so that I only need to change the port if my server uses a non-standard one.

> **US-3.** As a self-hoster, I want to be able to allow insecure (self-signed, expired, or hostname-mismatched) server certificates by enabling an explicit "allow insecure connections" option, so that I can connect to servers that do not have a publicly trusted certificate.

> **US-4.** As a corporate user, I want to select a client certificate from my system's certificate store and associate it with this account, so that the server can verify my identity via mutual TLS.

> **US-5.** As any user, I want an optional authentication realm field for servers that require a SASL domain or realm parameter, so that I can authenticate against domain-scoped accounts (e.g. NTLM domain authentication).

> **US-6.** As a security-conscious user, I want to enable DANE and/or DNSSEC verification for this account's connection, so that the server's certificate is validated against DNS-published TLSA records.

> **US-7.** As any user, I want to give my account a display name, an optional color, and an optional category, so that I can identify it easily in the navigation and folder list.

### 6.2 Domain-based auto-config within manual setup

> **US-8.** As any user, I want to enter my email domain and press an "Auto-config" button to have the application look up server settings and pre-fill the host, port, and encryption fields, so that I do not have to look up standard settings for well-known providers.

> **US-9.** As any user, after auto-config pre-fills the fields, I want every field to remain fully editable, so that I can override any incorrect or non-standard value before testing.

> **US-10.** As any user, if auto-config fails (provider not found, DNS error), I want a clear message telling me that discovery failed, so that I know I need to enter the settings manually.

### 6.3 Provider selection

> **US-11.** As any user, I want to optionally select a known provider from a dropdown to pre-fill server fields, with a "Custom" option that leaves all fields blank for fully manual entry.

> **US-12.** As any user, when a provider is selected, I want the port hint and encryption mode to reflect that provider's recommended settings, while still being overridable.

### 6.4 Test connection — inbound

> **US-13.** As any user, I want to press a "Test Connection" button that validates my inbound server configuration by connecting to the server, authenticating, and enumerating folders, so that I know the settings work before I save.

> **US-14.** As any user, on a successful inbound test I want to see the list of folders discovered on the server, with well-known folder types (Inbox, Sent, Drafts, Trash, Spam, Archive) auto-detected, so that I can verify the server responded correctly and assign folder roles.

> **US-15.** As any user, on a successful inbound test I want to be informed whether the server supports push (IDLE), so that I understand whether the application will use push delivery or periodic polling for this account.

> **US-16.** As any user, on a failed inbound test I want an actionable error message that identifies the cause (wrong hostname, wrong port, wrong credentials, certificate error, protocol mismatch, etc.), so that I can correct the configuration.

> **US-17.** As a self-hoster, when the test fails due to an untrusted server certificate, I want the application to display the certificate's fingerprint and offer me the option to trust that specific certificate for future connections, so that I can pin a self-signed certificate without enabling blanket insecure mode.

> **US-18.** As any user, I want the test to run in the background with a visible progress indicator and with input fields disabled during the test, so that I know an operation is in progress and cannot accidentally modify settings mid-test.

### 6.5 Folder role assignment

> **US-19.** As any user, after a successful inbound test, I want dropdown selectors for each well-known folder role (Drafts, Sent, Archive, Trash, Spam) pre-populated with the auto-detected folder, so that I can confirm or override the assignment.

> **US-20.** As any user, if the server does not advertise a folder for a given role, I want the corresponding selector to default to "not set" rather than guessing, so that I am not misled.

### 6.6 Saving the account

> **US-21.** As any user, after a successful test I want to save the account, persisting all connection settings and folder role assignments.

> **US-22.** As any user, if I attempt to save without having tested and synchronization is enabled, I want the application to automatically perform a connection check at save time, so that invalid configurations are caught even if I skipped the manual test.

> **US-23.** As any user, after saving a new inbound account I want the application to guide me to the outbound (SMTP) identity configuration screen, so that I can complete setup of sending capability in a natural sequential flow.

### 6.7 Outbound (SMTP) identity configuration

> **US-24.** As any user, I want to configure the SMTP server for each identity independently from the inbound server, specifying host, port, encryption mode, username, password, and optionally a client certificate and realm, so that I can use different outbound credentials or servers if needed.

> **US-25.** As any user, I want SMTP port defaults to follow the same convention as inbound: auto-fill based on encryption mode (465 for SSL/TLS, 587 for STARTTLS, 25 for None).

> **US-26.** As any user, I want a domain-based auto-config for SMTP that works the same way as for the inbound server, so that I can look up SMTP settings by domain.

> **US-27.** As any user, I want a "Test Connection" for the SMTP server that connects, authenticates, and reports the server's advertised maximum message size if available, so that I know sending will work before I save.

> **US-28.** As any user, I want the identity form to also collect the email address, display name, and the association with the inbound account, so that the sending identity is fully defined in one place.

> **US-29.** As any user, I want the SMTP username and password to default to the same credentials as the associated inbound account, but to be independently overridable for servers that require separate outbound credentials.

### 6.8 POP3-specific settings

> **US-30.** As a POP3 user, I want a "Leave messages on server" option (defaulting to on) that controls whether fetched messages are deleted from the server, so that I can choose between keeping a server copy and freeing server space.

> **US-31.** As a POP3 user, when "Leave on server" is disabled, I want a "Delete locally when removed from client" option, so that I can control whether local deletions propagate to the server.

> **US-32.** As a POP3 user, I want a "Maximum messages" field that limits how many messages are retained locally, so that I can cap storage usage on my device.

> **US-33.** As a POP3 user, I want a "Leave deleted messages" option that prevents automatic removal of messages I have deleted, so that I retain full local control over my message store.

### 6.9 Editing an existing account

> **US-34.** As any user, I want to open any existing account's configuration and change any server parameter (host, port, encryption, credentials, certificate, security options), test the updated configuration, and save, so that I can respond to server migrations or credential rotations.

> **US-35.** As any user, when I change the password on an existing account, I want the option to propagate the new password to all identities (SMTP configurations) associated with that account, so that I do not have to update each identity individually.

### 6.10 Advanced inbound options

> **US-36.** As a power user, I want to configure the polling interval for this account, so that I can balance responsiveness against resource usage.

> **US-37.** As a power user, I want options to restrict synchronization to unmetered networks only and/or to VPN-only connections, so that I can control when and how the application connects.

> **US-38.** As any user, I want to enable or disable UTF-8 / internationalized email support for this connection, so that the application communicates correctly with servers that support (or do not support) non-ASCII in protocol commands.

### 6.11 SMTP advanced options

> **US-39.** As a power user, I want to control whether the SMTP EHLO command uses my IP address or a custom hostname, so that I can satisfy servers with strict EHLO policies or avoid leaking my IP.

> **US-40.** As a power user, I want a "Login before send" option that verifies the inbound (IMAP) account is accessible before sending via SMTP, so that I can catch credential issues before a send attempt.

## 7. Functional Requirements

### 7.1 Protocol selection

- **FR-1.** The application shall support creating accounts with IMAP or POP3 as the inbound protocol, selectable by the user at account creation time.
- **FR-2.** The outbound protocol shall always be SMTP; no user selection is needed.
- **FR-3.** The inbound protocol selection shall determine which protocol-specific options are displayed (e.g. POP3 shows leave-on-server options; IMAP does not).

### 7.2 Inbound server fields

- **FR-4.** The inbound configuration screen shall present the following user-editable fields: **hostname**, **port**, **encryption mode**, **username**, **password**, **client certificate** (optional), and **authentication realm** (optional).
- **FR-5.** The encryption mode shall be selectable from exactly three options: **SSL/TLS** (implicit), **STARTTLS** (upgrade), and **None** (unencrypted).
- **FR-6.** When the user changes the encryption mode, the port field shall auto-fill with the conventional default for the selected protocol and encryption combination:
  - IMAP: SSL/TLS → 993, STARTTLS → 143, None → 143
  - POP3: SSL/TLS → 995, STARTTLS → 110, None → 110
- **FR-7.** The port auto-fill shall replace the current value only if the current value matches one of the known defaults (or is empty); a user-entered non-default port shall not be overwritten.
- **FR-8.** The password field shall have a visibility toggle allowing the user to reveal or hide the entered text.
- **FR-9.** The client certificate selector shall open the system certificate store and allow the user to choose a certificate. The selected certificate's name shall be displayed. A clear action shall allow removing the selection.
- **FR-10.** The realm field shall be optional and shall default to empty. When provided, it shall be passed to the authentication mechanism as the SASL realm or NTLM domain parameter.

### 7.3 Security options

- **FR-11.** The configuration screen shall include an **"Allow insecure connections"** toggle (default off). When enabled, the application shall accept server certificates that would otherwise fail validation (self-signed, expired, hostname mismatch).
- **FR-12.** When "Allow insecure connections" is enabled, the DANE option shall be automatically disabled, as DANE verification is incompatible with insecure certificate acceptance.
- **FR-13.** The configuration screen shall include a **DANE** toggle (default off). When enabled, the application shall verify the server's certificate against TLSA records in DNSSEC-signed DNS.
- **FR-14.** The configuration screen shall include a **DNSSEC** toggle (default off). When enabled, the application shall require DNSSEC-validated DNS resolution for the server hostname.
- **FR-15.** When a connection test fails due to an untrusted certificate, the application shall display the server certificate's fingerprint and offer a "Trust this certificate" action. Accepting this action shall pin the fingerprint for this account, bypassing the system trust store for future connections to this server.
- **FR-16.** The "None" encryption mode shall carry a prominent visual warning (e.g. warning color, icon) indicating that the connection will be unencrypted.

### 7.4 Validation

- **FR-17.** The hostname field shall be required (non-empty) for all accounts with synchronization enabled.
- **FR-18.** The username field shall be required (non-empty) unless "Allow insecure connections" is enabled.
- **FR-19.** The password field shall be required (non-empty) unless at least one of the following is true: (a) "Allow insecure connections" is enabled, or (b) a client certificate is selected.
- **FR-20.** Leading and trailing whitespace shall be trimmed from the hostname and username before use.
- **FR-21.** If the password contains leading/trailing whitespace or control characters, the application shall display a non-blocking warning (the password is still accepted).
- **FR-22.** The port field shall accept only numeric input (maximum 5 digits).

### 7.5 Domain-based auto-config

- **FR-23.** The manual configuration screen shall provide a domain input field and an "Auto-config" button.
- **FR-24.** On activation, auto-config shall attempt to discover server settings in the following order: (1) bundled provider database by domain match, (2) DNS NS record lookup, (3) DNS SRV records per RFC 6186, (4) MX record lookup, (5) well-known auto-configuration XML endpoints, (6) port scanning as a last resort.
- **FR-25.** On success, auto-config shall populate the hostname, port, and encryption mode fields with the discovered values. All other fields (username, password, certificate, security options) shall remain unchanged.
- **FR-26.** All auto-filled fields shall remain user-editable after population.
- **FR-27.** On failure, a clear error message shall be displayed; no fields shall be modified.
- **FR-28.** Auto-config shall run in the background with a progress indicator; the auto-config button and domain field shall be disabled during the operation.

### 7.6 Provider selection

- **FR-29.** The manual configuration screen shall offer a provider dropdown containing known providers from the bundled database, plus a "Custom" option.
- **FR-30.** Selecting a provider shall pre-fill the hostname, port, and encryption mode with that provider's recommended settings. Selecting "Custom" shall leave all fields as-is (empty for new accounts, unchanged for existing accounts).
- **FR-31.** Provider-specific guidance (e.g. "Use an app-specific password" for providers that require it) shall be displayed as contextual help text when a corresponding provider is selected.

### 7.7 Test connection — inbound

- **FR-32.** The manual configuration screen shall provide a "Test Connection" button that initiates a live connection test using the currently entered parameters.
- **FR-33.** The test shall: (a) resolve the hostname via DNS (with DNSSEC if enabled), (b) establish a connection using the selected encryption mode, (c) authenticate using the supplied credentials and optional client certificate, (d) enumerate the server's folder list (for IMAP), and (e) disconnect cleanly.
- **FR-34.** During the test, a progress indicator shall be visible and all input fields and buttons shall be disabled to prevent concurrent modification.
- **FR-35.** On success, the test shall:
  - Display the list of discovered folders (IMAP only) with auto-detected folder types.
  - Report whether the server supports IDLE (push). If IDLE is not supported, display a warning that the application will fall back to periodic polling.
  - Report whether the server supports UTF-8.
  - Make the folder-role selectors and Save button visible.
- **FR-36.** On failure, the test shall display a diagnostic error message. At minimum, the following failure categories shall be distinguished with specific guidance:
  - DNS resolution failure (unknown host)
  - Connection refused / timeout
  - TLS/SSL handshake failure (certificate error, protocol mismatch)
  - Authentication failure (wrong credentials)
  - Protocol mismatch (e.g. POP3 port used for IMAP)
  - Untrusted certificate (with fingerprint and trust option per FR-15)
- **FR-37.** When connected to a provider that has documentation or a help link, the error display shall include a link to that provider's setup documentation.

### 7.8 Folder role assignment

- **FR-38.** After a successful inbound test (IMAP only), the application shall display dropdown selectors for each well-known folder role: Drafts, Sent, Archive, Trash, and Spam.
- **FR-39.** Each selector shall be pre-populated with the folder that best matches that role based on server-advertised folder attributes and naming conventions. If no match is found, the selector shall default to "not set".
- **FR-40.** The user shall be able to override any auto-detected assignment by selecting a different folder from the dropdown.

### 7.9 Saving

- **FR-41.** A "Save" button shall persist all connection settings, credentials, security options, folder role assignments, and display properties (name, color, category) as a new or updated account.
- **FR-42.** If synchronization is enabled and the connection parameters have changed (or the account is new), and the user has not already run a successful test in this session, the save operation shall automatically perform a connection check. If the check fails, the save shall be aborted and the error displayed.
- **FR-43.** On saving a new account, if the user has opted to create an identity (sending configuration), the application shall navigate to the identity/SMTP configuration screen with the new account pre-selected.
- **FR-44.** When saving an existing account with a changed password, the application shall offer to propagate the new password to all identities associated with that account.

### 7.10 Outbound (SMTP) identity configuration

- **FR-45.** The identity configuration screen shall present the following SMTP fields: **hostname**, **port**, **encryption mode**, **username**, **password**, **client certificate** (optional), and **authentication realm** (optional).
- **FR-46.** Encryption mode, port auto-fill, security options (insecure, DANE), client certificate selection, and validation rules shall follow the same conventions as the inbound configuration, with SMTP-specific port defaults: SSL/TLS → 465, STARTTLS → 587, None → 25.
- **FR-47.** The identity form shall also collect: **email address**, **display name**, and **associated account** (selected from a dropdown of existing inbound accounts).
- **FR-48.** The SMTP username and password shall default to the associated account's inbound credentials but shall be independently editable.
- **FR-49.** A domain-based auto-config for SMTP shall be available, functioning identically to the inbound auto-config but discovering SMTP settings.
- **FR-50.** The identity configuration screen shall provide a "Test Connection" button that connects to the SMTP server, authenticates, queries the server's advertised maximum message size, and reports success or failure.
- **FR-51.** On successful SMTP test, if the server reports a maximum message size, that value shall be displayed and optionally stored as the identity's size limit.

### 7.11 SMTP advanced options

- **FR-52.** The identity screen shall include a **"Use IP address in EHLO"** toggle (default on). When enabled, the application shall use the device's IP address in the SMTP EHLO command instead of a hostname.
- **FR-53.** The identity screen shall include a **custom EHLO hostname** field, used when "Use IP" is disabled and a custom value is provided.
- **FR-54.** The identity screen shall include a **"Login before send"** toggle. When enabled, the application shall verify the inbound (IMAP/POP3) account is accessible before each SMTP send attempt.

### 7.12 POP3-specific settings

- **FR-55.** When the inbound protocol is POP3, the configuration screen shall display the following additional options:
  - **Leave messages on server** (default: on)
  - **Delete locally when removed from client** (only enabled when "Leave on server" is off)
  - **Leave deleted messages** (default: on)
  - **Leave on device** (default: on)
  - **Maximum messages** (optional integer; limits the number of messages retained locally)
- **FR-56.** Changing the "Leave on server" option on an existing account shall trigger a re-evaluation of the local message store at next synchronization.

### 7.13 Account display properties

- **FR-57.** The configuration screen shall include fields for: **account display name** (defaults to the username/email if not provided), **category** (optional, free-text), **color** (optional color picker), and **avatar** (optional image).
- **FR-58.** The display name shall be used throughout the application to identify this account in navigation, message lists, and account selectors.

### 7.14 Editing existing accounts

- **FR-59.** All fields that are configurable at account creation shall also be editable on an existing account.
- **FR-60.** The "Test Connection" and "Save" operations shall be available when editing, with the same validation and test behavior as at creation time.
- **FR-61.** Changing an existing account's server configuration shall not discard previously fetched messages or folder state; only the connection parameters shall be updated.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** The connection test shall begin within one second of pressing the "Test Connection" button. The test shall time out and report failure if the server does not respond within a reasonable period (on the order of tens of seconds), rather than hanging indefinitely.
- **NFR-2. Security defaults.** All security options shall default to the most secure setting: SSL/TLS encryption pre-selected, "Allow insecure connections" off, DANE off (opt-in), client certificate optional. The application shall never silently downgrade security.
- **NFR-3. Credential safety.** Passwords shall be masked by default in the UI. Credentials shall not be logged in normal operation. Passwords stored locally shall be protected by whatever credential-storage mechanism the application provides (e.g. encrypted database, system keyring).
- **NFR-4. Accessibility.** All fields, buttons, toggles, dropdowns, and error messages shall be reachable via keyboard, labeled for screen readers, and shall respect the application-wide theme (light/dark/high-contrast).
- **NFR-5. Offline resilience.** Account configuration (reading/editing the form, changing fields) shall be fully available offline. Only the "Test Connection" and "Auto-config" operations require network access; the application shall display a clear message if the network is unavailable.
- **NFR-6. Error clarity.** Error messages from failed connection tests shall be specific enough for the user to diagnose and correct the problem without external help in the common cases (wrong host, wrong port, wrong password, certificate issue, protocol mismatch).
- **NFR-7. Idempotent save.** Saving an account with unchanged parameters shall have no side effects — no re-sync triggered, no folder state reset, no credential re-validation.
- **NFR-8. Scale.** The manual configuration feature shall function correctly for users with up to 20 or more accounts, with no degradation in the account editor UI.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|
| AC-1 | A user can create a new IMAP account by entering host, port (or accepting the auto-filled default), encryption mode, username, and password, pressing "Test Connection", seeing the folder list, and pressing "Save". The account appears in the navigation and begins synchronizing. |
| AC-2 | A user can create a new POP3 account with the same flow. POP3-specific options (leave on server, max messages) are visible and configurable. |
| AC-3 | Changing the encryption mode from SSL/TLS to STARTTLS auto-fills the port from 993 to 143 (IMAP) or from 995 to 110 (POP3). Changing to None also updates the port. A user-entered non-default port is not overwritten. |
| AC-4 | Selecting "None" as the encryption mode displays a visible warning indicating the connection will be unencrypted. |

| AC-6 | A user can select a client certificate from the system store, and the certificate name is displayed in the configuration. The certificate is used during the connection test and subsequent connections. |
| AC-7 | When the connection test fails due to an untrusted certificate, the certificate fingerprint is displayed and a "Trust" option is available. After trusting, re-testing succeeds. |
| AC-8 | The connection test displays a progress indicator while running and disables all input fields. On completion, fields are re-enabled. |

| AC-10 | A successful IMAP test reports whether the server supports IDLE. If not, a warning about polling fallback is displayed. |

| AC-16 | SMTP port auto-fills to 465 for SSL/TLS, 587 for STARTTLS, and 25 for None. |
| AC-17 | Editing an existing account's password and choosing to propagate updates the password for all associated identities. |

## 10. Open Questions

- **OQ-1. Password propagation scope.** When saving an existing account with a changed password and propagating to identities, should the propagation apply only to identities whose SMTP password previously matched the old account password, or to all identities unconditionally? The source application propagates unconditionally; this may be surprising if the user intentionally has a different SMTP password on some identities.
- **OQ-2. Auto-config discovery order and timeout.** The source application uses a multi-stage discovery process (provider DB → DNS NS → SRV → MX → autoconfig XML → port scan). What should the total timeout be, and should the user be able to see which stage is being attempted? The source application shows only a progress spinner with no stage indication.
- **OQ-3. Certificate pinning lifetime.** When a user trusts a specific certificate fingerprint, is that trust permanent or should it expire? The source application stores the fingerprint indefinitely with no expiration. A certificate rotation on the server would cause a new trust prompt, but there is no mechanism to remind the user to review pinned certificates.
- **OQ-4. POP3 folder discovery.** POP3 does not support server-side folders. The source application does not display a folder list after a POP3 test. Should the test success screen for POP3 display any confirmation beyond "connection succeeded"?
- **OQ-5. SMTP test depth.** The source application's SMTP test connects and authenticates but does not send a test message. Should there be an option to send a test message to oneself to verify end-to-end delivery?
- **OQ-6. Concurrent editing.** If the user opens two account configuration screens simultaneously (e.g. in a multi-window setup), what happens if both are saved? The source application does not appear to guard against concurrent edits to the same account.
- **OQ-7. Provider-specific warnings.** The source application shows provider-specific help text (e.g. "use an app-specific password" for certain providers). Should the set of providers with special warnings be enumerable and documented, or treated as an opaque database?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Inbound and outbound are configured separately.** The inbound (IMAP/POP3) server is an account-level concern; the outbound (SMTP) server is an identity-level concern. This separation exists because a single account may have multiple identities (aliases), each potentially using a different SMTP server. The manual configuration flow reflects this: the user first configures the inbound account, then is guided to create at least one identity with its own SMTP settings.
- **N-2. Test before save, not test-on-save.** The primary flow encourages the user to test explicitly before saving: the Save button and folder selectors are hidden until the test succeeds. However, if the user bypasses testing (e.g. by navigating directly to save), a connection check is performed at save time as a safety net. This dual approach balances user experience (immediate feedback) with data integrity (no broken accounts saved).
- **N-3. Port auto-fill heuristic.** The port field auto-fills on encryption mode change, but only when the current value is empty or matches a known default. This prevents overwriting a user's intentional non-standard port while still providing the convenience of sensible defaults for the common case.
- **N-4. Certificate pinning as an escape hatch.** Certificate pinning (trusting a specific fingerprint) is deliberately presented only in response to a failed connection test, not as a proactive setting. This ensures the user sees the actual certificate before trusting it, and discourages pinning without understanding the implications.
- **N-5. "Allow insecure" is account-scoped, not global.** The insecure-connections toggle applies only to the account being configured, not application-wide. This allows a user to accept a self-signed certificate on one server while maintaining strict validation on all others.
- **N-6. POP3 as a first-class protocol.** POP3 is not a second-class citizen in the configuration flow. It has its own dedicated screen with protocol-specific options (leave on server, max messages), and the same encryption, authentication, and security options as IMAP. The application treats POP3 and IMAP as peers, differing only in protocol-specific capabilities.
- **N-7. SMTP credentials default to inbound.** Defaulting the SMTP username and password to the inbound account's credentials covers the overwhelmingly common case where the same credentials work for both. Making the fields independently editable covers the less common case where they differ. This avoids both unnecessary duplication for the common case and inflexibility for the uncommon case.
- **N-8. Auto-config is advisory, not authoritative.** Both the domain-based auto-config and the provider dropdown pre-fill fields but never lock them. Every auto-filled value is editable. This reflects the reality that provider databases can be outdated or incomplete, and the user's specific server configuration may differ from the provider's standard offering.
