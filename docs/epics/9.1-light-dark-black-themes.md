# Epic: Light / Dark / Black Themes

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **9.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Light / dark / black themes — with auto-switch following the system theme (GNOME / KDE / freedesktop color-scheme)."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

A desktop email client is a long-session application — users leave it open all day, often in varying lighting conditions. The ability to switch between a light appearance (for bright environments or user preference), a dark appearance (for dim environments, reduced eye strain, and reduced power consumption on OLED displays), and a true-black appearance (maximizing contrast and battery savings on OLED screens) is a basic expectation of modern desktop software.

Beyond a manual toggle, modern desktop environments (GNOME, KDE Plasma, freedesktop-compliant compositors) expose a system-wide color-scheme preference that changes automatically based on time of day, ambient light, or user schedule. A well-integrated desktop application should respect this signal and transition between its light and dark appearances automatically, so the user configures their preference once (at the OS level) and every application follows.

The Themes feature exists to provide:
1. A curated set of color schemes (primary/accent color pairings) so the user can personalize the application's appearance.
2. Three brightness variants — light, dark, and true-black — for every color scheme.
3. An automatic mode that follows the desktop environment's color-scheme preference, switching between light and dark/black at the system's direction.
4. Content-specific overrides so that the application chrome and the message content area can be themed independently when needed.

## 2. Goals

- **G1.** Provide at least three brightness variants (light, dark, true-black) for every supported color scheme.
- **G2.** Provide an automatic mode that follows the system color-scheme preference (freedesktop `color-scheme`, GNOME/KDE night mode) without requiring manual intervention.
- **G3.** Offer multiple color schemes (primary/accent pairings) so users can personalize the application beyond brightness alone.
- **G4.** Allow the message content area (original HTML messages and the composer) to use a different brightness than the application chrome, so that readability is preserved regardless of overall theme.
- **G5.** Ensure all theme transitions are seamless — the user never sees partially-rendered states or inconsistent coloring between application regions.
- **G6.** Respect accessibility constraints: maintain sufficient contrast ratios in every variant, and ensure interactive elements remain distinguishable.

## 3. Non-Goals

- **NG1.** Fully user-customizable per-element color picking (arbitrary hex/RGB for individual UI elements) is not a goal. The user selects from curated schemes.
- **NG2.** Per-account or per-folder theming (e.g. "make my work account blue and my personal account green") is not a goal. Theming is application-wide.
- **NG3.** Theming of third-party notification content or system-tray indicators beyond what the desktop environment natively supports is not a goal.
- **NG4.** Animated transitions between themes (cross-fades, morphing) are not a goal. An instantaneous switch is acceptable.
- **NG5.** Exposing a theme editor or theme file import/export mechanism is not a goal.
- **NG6.** Dynamically deriving a color scheme from the user's desktop wallpaper or accent color is out of scope for the initial release (see Open Questions for future consideration).

## 4. Glossary

| Term | Definition |
|---|---|
| **Color scheme** | A curated pairing of a primary color and an accent color that together define the application's personality. Examples: Blue/Orange, Red/Green, Yellow/Purple, Grey/Steel Blue, Solarized, Black & White. |
| **Brightness variant** | One of three modes applied on top of a color scheme: *light* (bright backgrounds, dark text), *dark* (dim backgrounds, light text), or *black* (true-black backgrounds, light text). |
| **Theme** | The combination of a color scheme and a brightness variant — e.g. "Blue/Orange Dark" or "Solarized Light". |
| **Auto mode** | A brightness-variant setting that delegates the light-vs-dark decision to the desktop environment's color-scheme preference, switching automatically when the system switches. |
| **System color-scheme** | The desktop environment's advertised preference for light or dark appearance (freedesktop `org.freedesktop.appearance color-scheme`, GNOME `color-scheme`, KDE color scheme settings). |
| **Application chrome** | All UI elements that are part of the application itself: toolbars, sidebars, message list, folder tree, dialogs, menus, status bar, cards, etc. |
| **Message content area** | The region that renders the body of an email message (original HTML or plain-text) or the compose editor. This area may follow a different brightness than the chrome. |
| **True-black** | A background color of pure black (#000000), as distinct from the dark grey used in the standard dark variant. Optimized for OLED displays. |
| **Beige mode** | An optional warm-tinted background for cards and message list items in the light variant, reducing harshness of pure white. |

## 5. Personas

- **P1. The day/night automatic user.** Has their desktop set to switch appearance with time of day or ambient light. Wants the email client to follow along without any manual action.
- **P2. The always-dark user.** Prefers dark mode at all times regardless of system setting. Wants to lock the theme to dark or black and never see a light flash.
- **P3. The accessibility-conscious user.** Needs high contrast or specific color pairings to accommodate a visual impairment. Wants a curated scheme that guarantees readability.
- **P4. The OLED power-saver.** Uses a true-black theme to maximize battery life or reduce light bleed on an OLED display. Specifically wants pure-black backgrounds, not dark grey.
- **P5. The content-reader.** Reads many HTML newsletters and formatted messages; wants the message content area to always render on a light background for fidelity, even when the rest of the application is dark.

## 6. User Stories

### 6.1 Selecting a theme

> **US-1.** As any user, I want to open a theme-selection interface from the application's settings or main menu, so that I can browse and choose my preferred color scheme and brightness variant.

> **US-2.** As any user, I want to see a preview or immediate application of my theme choice, so that I can evaluate how the application looks before committing (or so that the change takes effect immediately without requiring a restart).

> **US-3.** As any user, I want to choose independently: (a) a color scheme and (b) a brightness variant (light / dark / black / auto), so that I can pair any color palette with any brightness level.

> **US-4.** As any user, I want an option to reverse the primary and accent colors of my chosen color scheme, so that I can emphasize the accent as the dominant color if I prefer.

### 6.2 Auto-switching with system theme

> **US-5.** As a day/night automatic user, I want to select an "auto" or "system" brightness mode that follows my desktop environment's light/dark preference, so that the application switches themes automatically when my system does.

> **US-6.** As a day/night automatic user, when my system switches from light to dark (or vice versa), I want the application to transition within a few seconds without requiring me to close or restart it, so that the experience is seamless.

> **US-7.** As a day/night automatic user in auto mode, I want to control whether the dark variant used is standard dark or true-black, so that auto-switching respects my OLED preference.

> **US-8.** As any user, if my desktop environment does not expose a color-scheme preference (or the application cannot detect it), I want the application to fall back to a sensible default (light) rather than crash or render incorrectly.

### 6.3 True-black (OLED) variant

> **US-9.** As an OLED power-saver, I want a true-black variant that uses a pure-black background instead of dark grey, so that pixels are fully off and battery is conserved.

> **US-10.** As an OLED power-saver, I want the true-black variant available both as a fixed choice and as the dark side of auto mode, so that I am not forced to choose between auto-switching and OLED optimization.

### 6.4 Message content theming

> **US-11.** As a content-reader, I want an option to force the original message view to always use a light background, regardless of the application's overall theme, so that HTML messages and images render as their authors intended.

> **US-12.** As a content-reader, I want an option to force the message composer to always use a light background, so that I can see how my formatting will appear to recipients on light backgrounds.

> **US-13.** As any user, when no content-override is active, I want the message content area to follow the application's current theme (including its dark/black mode), so that there is no jarring contrast between chrome and content.

> **US-14.** As any user viewing a dark-themed message content area, I want the application to algorithmically darken HTML content that was designed for light backgrounds, so that text remains readable and images remain visible.

### 6.5 Light-variant refinements

> **US-15.** As a user of the light variant, I want an option to use a warm beige background for message cards and list items instead of pure white, so that the interface is less harsh in bright environments.

> **US-16.** As a user of the light variant, I want the option to disable the beige tint and use a standard white background, so that I can choose whichever I find more comfortable.

### 6.6 Theme persistence and scope

> **US-17.** As any user, I want my theme choice to persist across application restarts, so that I configure it once and it stays.

> **US-18.** As any user, I want the theme to apply uniformly to all application windows (main window, compose window, settings dialogs, pop-out message views), so that the experience is consistent.

### 6.7 Contrast and accessibility

> **US-19.** As an accessibility-conscious user, I want every curated color scheme to guarantee sufficient contrast between text and background in all brightness variants, so that I can trust any scheme to be readable.

> **US-20.** As an accessibility-conscious user, I want interactive elements (buttons, links, toggles) to remain visually distinguishable from static content in every theme, so that I can navigate the interface without confusion.

> **US-21.** As any user, I want links in dark/black themes to use adjusted colors (not the standard blue used in light themes) so that they remain legible against dark backgrounds.

## 7. Functional Requirements

### 7.1 Color schemes

- **FR-1.** The application shall provide at least six curated color schemes, each defining a primary color and an accent color.
- **FR-2.** Each color scheme shall be available in all three brightness variants: light, dark, and true-black.
- **FR-3.** Each color scheme shall offer a "reversed" option that swaps the roles of primary and accent color.
- **FR-4.** A monochrome/neutral scheme (e.g. Black & White, Grey) shall be available for users who prefer no color.
- **FR-5.** A warm-toned scheme (e.g. Solarized) shall be available for users who prefer reduced blue-light exposure.

### 7.2 Brightness variants

- **FR-6.** The **light** variant shall use bright backgrounds (white or near-white) with dark foreground text and controls.
- **FR-7.** The **dark** variant shall use dim grey backgrounds with light foreground text and controls.
- **FR-8.** The **true-black** variant shall use pure-black backgrounds with light foreground text and controls. The distinction from the dark variant must be visually obvious (backgrounds are fully black, not dark grey).
- **FR-9.** Every UI surface in the application (toolbars, sidebars, dialogs, menus, cards, floating panels, splash/loading screens) shall respect the active brightness variant.

### 7.3 Auto mode (system-following)

- **FR-10.** The application shall offer an "auto" / "system" brightness mode that determines whether to use the light or dark/black variant based on the desktop environment's advertised color-scheme preference.
- **FR-11.** When in auto mode, the application shall monitor for system color-scheme changes at runtime and apply the corresponding variant without requiring user action or application restart.
- **FR-12.** When in auto mode, the user shall be able to specify whether the dark side of the auto switch uses the standard dark variant or the true-black variant.
- **FR-13.** If the system does not expose a color-scheme preference, auto mode shall default to the light variant.
- **FR-14.** The application shall support detecting the color-scheme preference from at least the following desktop environments: GNOME, KDE Plasma, and any freedesktop-portal-compliant compositor.

### 7.4 Message content overrides

- **FR-15.** The application shall provide an option ("force light for original messages") to render the HTML/plain-text message view with a light background regardless of the application's brightness variant.
- **FR-16.** The application shall provide an option ("force light for composer") to render the message composition area with a light background regardless of the application's brightness variant.
- **FR-17.** When no content override is active and the application is in a dark or black variant, the message content renderer shall apply algorithmic darkening to HTML content designed for light backgrounds, preserving legibility of text and visibility of images.
- **FR-18.** A per-message override (e.g. a toolbar button or menu item) shall allow the user to toggle the content area between light and themed rendering for the currently viewed message, without changing the global default.

### 7.5 Visual consistency and derived colors

- **FR-19.** The application shall derive all semantic UI colors (read/unread indicators, verification badges, encryption indicators, warning markers, folder-color accents, account-color stripes, separators, etc.) from the active theme's color scheme and brightness variant, ensuring visual harmony.
- **FR-20.** The application shall adjust link colors in dark/black variants to maintain sufficient contrast against dark backgrounds (links shall not use the same hue and lightness as in the light variant).
- **FR-21.** Toolbar icon colors and status-bar icon colors shall adapt to the luminance of the toolbar background, ensuring legibility in both light and dark variants.
- **FR-22.** When the light variant is active and the beige option is enabled, card backgrounds shall use a warm off-white tint. When beige is disabled, standard white shall be used.

### 7.6 Theme selection interface

- **FR-23.** The application shall provide a theme-selection interface accessible from the main menu or settings.
- **FR-24.** The theme-selection interface shall allow independent selection of: (a) color scheme, (b) brightness variant (light / dark / black / auto), (c) color reversal option, (d) auto-mode dark variant (dark vs. black), (e) content-area overrides (force light for messages, force light for composer).
- **FR-25.** Theme changes shall take effect immediately upon confirmation; the user shall not need to restart the application manually.
- **FR-26.** If the application must reload its interface to apply a theme change, this shall happen automatically and preserve the user's navigation state (current view, scroll position) as closely as possible.

### 7.7 Persistence

- **FR-27.** The user's theme selection (scheme, variant, reversal, auto-dark preference, content overrides, beige preference) shall persist across application restarts.
- **FR-28.** Theme preferences shall be stored as local user configuration only; no network communication shall be required to apply or change themes.

### 7.8 Scope of application

- **FR-29.** The active theme shall apply uniformly to all application windows, dialogs, and pop-out views.
- **FR-30.** The splash or loading screen shown at application startup shall respect the user's last-known brightness variant, avoiding a light flash for dark-mode users or a dark flash for light-mode users.

## 8. Non-Functional Requirements

- **NFR-1. Performance.** Theme switching (whether user-initiated or system-triggered) shall complete within two seconds, with no visible rendering artifacts or half-applied states.
- **NFR-2. Contrast compliance.** All text in every theme variant shall meet at least WCAG AA contrast ratio (4.5:1 for normal text, 3:1 for large text) against its immediate background.
- **NFR-3. Completeness.** No UI element shall be left unstyled or shall inherit a hard-coded color that conflicts with the active theme. All custom-drawn elements shall participate in theming.
- **NFR-4. Stability.** The auto-switch shall not flicker or oscillate if the system color-scheme signal changes rapidly (e.g. during a transition animation). The application shall debounce or coalesce rapid changes.
- **NFR-5. Graceful degradation.** On desktop environments that do not expose a color-scheme preference, all manual theme features shall work normally; only auto-switching is unavailable, and the user shall be informed.
- **NFR-6. Keyboard accessibility.** The theme-selection interface shall be fully navigable via keyboard. Theme changes shall not break keyboard focus or tab order in the rest of the application.
- **NFR-7. Screen-reader compatibility.** Theme names, variant labels, and toggle states shall be announced to screen readers. Active theme state shall be programmatically determinable.

## 10. Open Questions

- **OQ-1. Wallpaper-derived color scheme.** The source application supports a "Material You" mode that derives colors from the system wallpaper. Should the desktop version support a similar feature (e.g. deriving an accent color from the desktop wallpaper or system accent color), or is this deferred to a future enhancement? The source restricts this to newer OS versions.
- **OQ-2. Number and identity of curated schemes.** The source application provides approximately eight color schemes. Should the desktop version ship with the same set, a subset, or an expanded set? This depends on design capacity and testing burden.
- **OQ-3. Widget / panel applet theming.** If the application exposes desktop widgets or panel applets, should those follow the application theme, the system theme, or have their own independent setting? The source application has a separate day/night option for widgets.
- **OQ-4. Auto-mode detection priority.** On systems where multiple color-scheme signals exist (e.g. freedesktop portal *and* a GNOME-specific setting), which takes precedence? Should the application expose a preference, or should it use a fixed detection order?
- **OQ-5. Algorithmic darkening quality.** When applying algorithmic darkening to HTML message content in dark mode, how aggressively should colors be inverted? Some messages contain brand colors or images that look wrong when inverted. Should the user have a per-message escape hatch beyond "force light"? The source application relies on the platform's built-in algorithmic darkening.
- **OQ-6. Theme and notification integration.** Should notification pop-ups or system-tray tooltip previews respect the application theme, or should they always follow the system's notification styling? This likely depends on the desktop environment's constraints.
- **OQ-7. Rapid system-switch debouncing.** FR-11 requires runtime monitoring. If the system color-scheme signal flickers (e.g. during a GNOME animation), how long should the debounce window be? The source application relies on OS-level activity lifecycle rather than explicit debouncing.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Three variants, not two.** The source application distinguishes dark (dark grey backgrounds, ~#121316) from true-black (pure #000000). This is not cosmetic — on OLED displays, true-black pixels are physically off, yielding measurable power savings and deeper contrast. The distinction is worth preserving for the desktop version, especially as OLED laptop displays and monitors become more common.
- **N-2. Auto mode delegates only brightness, not scheme.** The system color-scheme preference determines only light-vs-dark. The user's chosen color scheme (Blue/Orange, Solarized, etc.) remains unchanged across the switch. This means auto mode requires storing two preferences: the scheme and the auto-dark-variant (dark or black).
- **N-3. Content-area independence.** The decision to allow the message content area to differ from the chrome theme is driven by a practical concern: many HTML emails contain hard-coded light-background assumptions (white text-shadow, transparent PNGs designed for white, CSS that assumes a light canvas). Forcing these into a dark renderer can produce illegible or ugly results. The "force light" option is an escape valve for readability, not a cosmetic preference.
- **N-4. Immediate application, not deferred.** The source application applies theme changes by tearing down and recreating the current screen. This ensures no stale-color artifacts but means a brief visual reset. The key contract is that the user never needs to manually close and reopen the application. Whether the desktop version achieves this through recreation, live style re-application, or another mechanism is an implementation choice — the requirement is instantaneity.
- **N-5. Curated schemes, not arbitrary.** The source application explicitly documents that "it is not feasible to add for each color combination a predefined theme." Instead, schemes are curated to guarantee contrast, harmony, and accessibility. Each scheme is tested in all three brightness variants. This editorial constraint is load-bearing for accessibility (NFR-2) and should be retained.
- **N-6. Beige as a light-mode refinement.** The beige card background in light mode exists because pure-white cards on a white background create a flat, harsh appearance. The warm tint adds depth and reduces eye fatigue. It is on by default in the source application but can be disabled by the user.
- **N-7. Splash-screen consistency.** The source application sets the splash screen's appearance based on the stored theme preference so that there is no "flash of wrong color" at startup. This is a small but meaningful polish point — a dark-mode user should never be greeted by a white flash.
