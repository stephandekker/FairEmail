# Epic: Background Daemon (System Tray Icon)

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **8.9** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Persistent process running as a user systemd unit, with status icon showing connection / unread state — equivalent of the Android foreground service."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

A desktop email client that supports push-style delivery (via IMAP IDLE) must maintain persistent connections to one or more mail servers. Unlike a web browser that the user opens and closes at will, an email client is expected to be "always on" — receiving messages, executing queued operations, and surfacing notifications even when the user is not actively looking at the application window.

On Android, FairEmail achieves this through a *foreground service* — a long-running background process that the operating system will not aggressively terminate, signaled to the user by a persistent status-bar notification. On a Linux desktop, the equivalent is a **background daemon process** managed by the user's session (e.g. a user-level service unit), combined with a **system tray icon** (status notifier) that communicates the daemon's state at a glance.

The daemon is the backbone of the application's real-time behavior: without it, there is no push mail, no queued-operation replay, no background notification delivery, and no offline resilience. The system tray icon is the daemon's face — it tells the user whether the application is connected, how many accounts are being monitored, whether operations are pending, and whether the network is suitable. It also provides a quick entry point into the main application window.

## 2. Goals

- **G1.** Provide a long-lived background process that maintains persistent connections to all enabled mail accounts, independent of whether the main application window is open.
- **G2.** Survive session events (window close, workspace switch, screen lock) without interrupting mail synchronization or notification delivery.
- **G3.** Present a system tray icon that communicates the daemon's operational state (connected, waiting, error) and aggregate status (monitored account count, pending operation count, network suitability) at a glance.
- **G4.** Allow the user to control the daemon's lifecycle — start, stop, and restart — without requiring the main application window.
- **G5.** Ensure the daemon starts automatically when the user logs in, so that mail is available from the moment the desktop session begins.
- **G6.** Minimize resource consumption (CPU, memory, network, power) when idle, so that the daemon can run indefinitely without measurable impact on system performance.
- **G7.** Degrade gracefully when the network is unavailable, resuming normal operation automatically when connectivity returns.

## 3. Non-Goals

- **NG1.** This epic does **not** define the synchronization protocol, connection management, or IMAP IDLE behavior — those are covered by the synchronization and connectivity epics (7.x). The daemon is the *host process* for those subsystems; it does not define their internal behavior.
- **NG2.** This epic does **not** define the content or behavior of new-message notifications — those are covered by the notifications epic (8.1). The daemon is the process that *delivers* notifications; the notification content, grouping, and actions are specified elsewhere.
- **NG3.** This epic does **not** prescribe the process supervision mechanism (whether the daemon is managed by a session manager, a service unit, or a self-daemonizing process). It specifies the behavioral contract only.
- **NG4.** Per-account connection indicators in the main application window (folder-tree icons, account-list badges) are not part of this epic. This epic covers only the system tray icon and daemon lifecycle.
- **NG5.** Scheduling synchronization to specific time windows (work hours, weekends) is covered by epic 7.3. This epic assumes the daemon respects whatever schedule is configured but does not define the scheduling rules themselves.

## 4. Glossary

| Term | Definition |
|---|---|
| **Daemon** | A long-running background process that persists across application window open/close cycles, responsible for maintaining mail server connections, executing queued operations, and delivering notifications. |
| **System tray icon** | A small icon displayed in the desktop environment's notification area (system tray, status notifier area, or panel indicator) that represents the daemon's presence and state. |
| **Monitoring** | The state in which the daemon is actively maintaining connections to one or more mail accounts and listening for new messages. |
| **Monitored account** | An account that is enabled for synchronization, is not on-demand-only, has at least one folder configured, and is not marked for deletion. |
| **Pending operation** | A queued action (send, move, copy, delete, flag change, etc.) that has been committed locally but not yet executed on the remote server. |
| **Network suitability** | Whether the current network connection meets the user's configured requirements for synchronization (e.g. unmetered only, VPN required). |
| **Autostart** | The ability of the daemon to launch automatically when the user's desktop session begins, without manual intervention. |
| **Global sync toggle** | A single application-wide setting that enables or disables all background synchronization. When disabled, the daemon stops and the tray icon reflects the inactive state. |
| **Poll mode** | An alternative to continuous push connections where the daemon wakes periodically to check for new messages, rather than maintaining a persistent IMAP IDLE connection. |

## 5. Personas

- **P1. The always-on user.** Wants mail to arrive instantly, notifications to fire in real time, and the tray icon to confirm that everything is working — all without the main window being open.
- **P2. The resource-conscious user.** Runs many applications and wants the daemon to be invisible in terms of CPU and memory. Wants to verify at a glance that the daemon is not stuck in a retry loop or consuming excessive resources.
- **P3. The intermittent user.** Prefers to check mail on their own schedule. Wants to be able to stop the daemon entirely (no background connections, no tray icon) and start it again when ready.
- **P4. The multi-account user.** Has many accounts and wants the tray icon to summarize the aggregate state: how many accounts are connected, whether any have errors, and how many operations are pending.
- **P5. The newcomer.** Has just installed the application and expects it to "just work" — the daemon should start automatically, the tray icon should appear, and mail should arrive without any manual configuration of background services.

## 6. User Stories

### 6.1 Daemon lifecycle

> **US-1.** As any user, when I close the main application window, I want the daemon to continue running in the background so that I keep receiving new mail and notifications without having to leave the window open.

> **US-2.** As any user, when I log in to my desktop session, I want the daemon to start automatically so that mail is being synchronized from the moment my session begins.

> **US-3.** As any user, when the daemon is running, I want to see a system tray icon so that I have visual confirmation that background synchronization is active.

> **US-4.** As an intermittent user, I want to be able to stop the daemon entirely from a menu or setting, so that no background connections are maintained and no system resources are consumed when I do not want mail synchronization.

> **US-5.** As any user, when I re-enable synchronization after having stopped it, I want the daemon to start again and the tray icon to reappear, without requiring me to log out or reboot.

> **US-6.** As any user, if the daemon process is terminated unexpectedly (e.g. by the OS reclaiming memory or by a crash), I want it to restart automatically so that mail synchronization resumes without my intervention.

### 6.2 System tray icon — status display

> **US-7.** As any user, I want the tray icon to indicate at a glance whether the daemon is actively monitoring accounts, so that I can tell whether my mail is being synchronized.

> **US-8.** As a multi-account user, I want the tray icon or its tooltip to show how many accounts are currently being monitored, so that I can verify all my accounts are active.

> **US-9.** As any user, when there are pending operations (messages queued to send, moves not yet executed on the server, etc.), I want the tray icon or its tooltip to indicate the count of pending operations, so that I know work is in progress.

> **US-10.** As any user, when the network is unsuitable for synchronization (disconnected, metered when metered sync is disabled, etc.), I want the tray icon or its tooltip to indicate that the daemon is waiting for a suitable connection, so that I understand why new mail is not arriving.

> **US-11.** As any user, when an account encounters an authentication or connection error, I want the tray icon to reflect that something requires attention, so that I am not left wondering why mail has stopped arriving.

### 6.3 System tray icon — interactions

> **US-12.** As any user, I want to click (or activate) the tray icon to open the main application window, so that I can quickly access my mail from the tray.

> **US-13.** As any user, I want the tray icon to provide a context menu (e.g. via right-click) with at least the ability to open the main window, manually trigger a sync, and quit the application (including the daemon), so that I have basic control without opening the full UI.

### 6.4 Relationship to global sync toggle

> **US-14.** As any user, when I disable the global synchronization toggle, I want the daemon to stop, all server connections to close, and the tray icon to either disappear or switch to an "inactive" state, so that I have a single, clear control to turn off all background activity.

> **US-15.** As any user, when I re-enable the global synchronization toggle, I want the daemon to restart and the tray icon to return to its "monitoring" state, so that background synchronization resumes.

### 6.5 Poll mode vs. continuous mode

> **US-16.** As a resource-conscious user, I want to be able to switch from continuous push (IMAP IDLE) to periodic polling at a configurable interval, so that the daemon can release persistent connections between polls and reduce resource usage.

> **US-17.** As any user, regardless of whether I am in continuous or poll mode, I want the tray icon to remain visible and to reflect the current state, so that I always know the daemon is running.

### 6.6 Schedule-aware behavior

> **US-18.** As any user who has configured a synchronization schedule (e.g. work hours only), I want the daemon to respect the schedule — stopping synchronization outside the configured window and resuming when the window opens — and I want the tray icon to reflect whether synchronization is currently active or paused due to schedule.

### 6.7 Daemon and notifications

> **US-19.** As any user, I want the daemon to be the process responsible for delivering desktop notifications for new mail, so that notifications arrive even when the main window is closed.

> **US-20.** As any user, I want the tray icon's presence and the delivery of new-mail notifications to be independent concerns: I should be able to have the tray icon visible without receiving notifications for every folder, and vice versa.

### 6.8 Quitting the application

> **US-21.** As any user, when I choose to "quit" the application (as opposed to simply closing the window), I want both the main window and the daemon to terminate, the tray icon to disappear, and all server connections to close, so that I can fully exit when I choose to.

> **US-22.** As any user, I want a clear distinction between "close window" (daemon continues, tray icon remains) and "quit application" (everything stops), so that I do not accidentally stop mail synchronization by closing the window.

## 7. Functional Requirements

### 7.1 Daemon lifecycle

- **FR-1.** The application shall provide a background daemon process that runs independently of the main application window.
- **FR-2.** The daemon shall start automatically when the user's desktop session begins (autostart), unless the user has explicitly disabled autostart.
- **FR-3.** The daemon shall continue running when the main application window is closed, and shall stop only when (a) the user explicitly quits the application, (b) the user disables the global synchronization toggle, or (c) there are no enabled accounts to monitor and no pending operations to execute.
- **FR-4.** When the daemon has no enabled accounts to monitor and no pending operations, it shall wait a brief grace period (on the order of seconds) before stopping, to avoid rapid start/stop cycling during account configuration changes.
- **FR-5.** If the daemon process terminates unexpectedly, the session manager or equivalent mechanism shall restart it automatically.
- **FR-6.** The daemon shall return to full operational state after an unexpected restart without requiring user interaction.
- **FR-7.** The user shall be able to enable or disable autostart via a discoverable application setting.

### 7.2 System tray icon — presence

- **FR-8.** When the daemon is running, it shall display a system tray icon in the desktop environment's notification area.
- **FR-9.** When the daemon stops (whether by user action, schedule, or lack of accounts), the tray icon shall either disappear or switch to a visually distinct "inactive" state.
- **FR-10.** The tray icon shall be compatible with the major Linux desktop notification area protocols (freedesktop StatusNotifierItem and the legacy system tray protocol).

### 7.3 System tray icon — status communication

- **FR-11.** The tray icon or its tooltip shall communicate the following aggregate information:
  - (a) The number of accounts currently being monitored.
  - (b) The number of pending operations, if any.
  - (c) Whether the daemon is waiting for a suitable network connection.
- **FR-12.** When the number of monitored accounts is zero but the daemon is running (e.g. only processing pending operations), the tray icon or tooltip shall indicate this distinct state.
- **FR-13.** The tray icon shall update its displayed state promptly (within seconds) when the underlying state changes (account connects/disconnects, operation completes, network changes).
- **FR-14.** The tray icon shall use a minimal, non-intrusive visual style (low-priority, no sound, no animation beyond state changes) so that it does not compete with new-mail notifications for the user's attention.

### 7.4 System tray icon — interactions

- **FR-15.** Activating (clicking) the tray icon shall open or bring to focus the main application window.
- **FR-16.** The tray icon shall provide a context menu with at minimum:
  - (a) Open / show the main application window.
  - (b) Quit the application (terminate both window and daemon).
- **FR-17.** Activating the tray icon when the main window is already focused may minimize or hide the window, at the discretion of the desktop environment's conventions.

### 7.5 Global synchronization toggle

- **FR-18.** The application shall provide a single, global toggle that enables or disables all background synchronization. This toggle shall be accessible from both the main application settings and (optionally) the tray icon's context menu.
- **FR-19.** When the global toggle is disabled:
  - (a) The daemon shall close all server connections.
  - (b) The daemon shall stop monitoring all accounts.
  - (c) The tray icon shall reflect the inactive state.
  - (d) No new-mail notifications shall be delivered.
  - (e) Pending operations shall remain queued but shall not be executed until the toggle is re-enabled.
- **FR-20.** When the global toggle is re-enabled, the daemon shall resume monitoring all previously enabled accounts and begin executing any queued operations.
- **FR-21.** The global toggle default shall be **enabled**, so that new users receive mail without additional configuration.

### 7.6 Account-level conditions

- **FR-22.** The daemon shall monitor an account only if all of the following conditions are met:
  - (a) The global synchronization toggle is enabled.
  - (b) The account's own synchronization setting is enabled.
  - (c) The account is not configured as on-demand-only.
  - (d) The account has at least one folder configured.
  - (e) The account is not marked for deletion.
- **FR-23.** The daemon shall continue running to execute pending operations for an account even if that account does not meet all monitoring conditions, provided the account's synchronization setting is enabled and it is not marked for deletion.
- **FR-24.** Changes to account-level settings shall take effect on the daemon immediately (within seconds) without requiring a manual restart.

### 7.7 Network awareness

- **FR-25.** The daemon shall monitor network connectivity changes and respond automatically:
  - (a) When a suitable network becomes available, the daemon shall establish or re-establish server connections.
  - (b) When the network becomes unsuitable or unavailable, the daemon shall update the tray icon to reflect the waiting state and shall not attempt connections until suitability is restored.
- **FR-26.** The definition of "suitable network" shall respect user preferences (e.g. sync on metered networks, sync on unmetered networks only).

### 7.8 Connection error handling

- **FR-27.** When a server connection fails, the daemon shall retry with increasing back-off intervals to avoid draining system resources on persistent errors.
- **FR-28.** The tray icon or its tooltip shall indicate when one or more accounts are in an error or back-off state.
- **FR-29.** Authentication errors shall be surfaced to the user via a separate, higher-priority notification mechanism (not just the tray icon tooltip), so that credential problems are not silently ignored.

### 7.9 Poll mode

- **FR-30.** The user shall be able to configure the daemon to use periodic polling instead of continuous push connections, with a configurable interval.
- **FR-31.** When in poll mode, the daemon shall wake at the configured interval, synchronize, and then release server connections until the next poll.
- **FR-32.** The tray icon shall remain visible in poll mode and shall reflect the daemon's state (idle between polls, actively syncing during a poll).

### 7.10 Schedule integration

- **FR-33.** When a synchronization schedule is configured (covered by epic 7.3), the daemon shall respect it: pausing synchronization outside the scheduled window and resuming when the window opens.
- **FR-34.** During a schedule-imposed pause, the tray icon shall indicate the paused/scheduled state so that the user understands why synchronization is not active.

### 7.11 Quit vs. close

- **FR-35.** Closing the main application window shall **not** terminate the daemon. The tray icon shall remain, and synchronization shall continue.
- **FR-36.** An explicit "Quit" action (from the application menu, the tray icon context menu, or a keyboard shortcut) shall terminate both the main window and the daemon, remove the tray icon, and close all server connections.

## 8. Non-Functional Requirements

- **NFR-1. Resource efficiency.** The daemon, when idle (no active synchronization, connections established and waiting), shall consume negligible CPU and minimal memory, comparable to other well-behaved background services.
- **NFR-2. Startup speed.** The daemon shall reach its monitoring state (connections initiated to all enabled accounts) within a few seconds of being started, not counting server response time.
- **NFR-3. Resilience.** The daemon shall tolerate transient failures (network drops, server timeouts, DNS failures) without crashing. It shall log errors, apply back-off, and recover automatically when conditions improve.
- **NFR-4. Session integration.** The daemon shall integrate with the desktop session lifecycle: starting on login (if autostart is enabled), and terminating cleanly on logout or shutdown without leaving orphan processes or stale lock files.
- **NFR-5. Desktop environment compatibility.** The system tray icon shall be visible and functional on at least GNOME (via the AppIndicator/StatusNotifierItem extension), KDE Plasma, and XFCE, covering the most common Linux desktop environments.
- **NFR-6. Unobtrusive presence.** The tray icon shall not produce sounds, badges, or animations that compete with new-mail notifications. Its purpose is passive status display, not active alerting.
- **NFR-7. Accessibility.** The tray icon's tooltip and context menu shall be accessible to screen readers. State changes shall be expressible as text, not only as icon variations.
- **NFR-8. Privacy.** The daemon shall not transmit any data other than standard mail protocol traffic to the user's configured servers. No telemetry, heartbeats, or status reports shall be sent to any third party.

## 10. Open Questions

- **OQ-1. Tray icon visual vocabulary.** What distinct icon states should be defined (e.g. connected, partially connected, all disconnected, error, paused, polling)? Should these be distinct icons, overlays on a base icon, or expressed solely through the tooltip? This is a design decision deferred to the UI specification stage.
- **OQ-2. Unread count on the tray icon.** The source application's foreground notification shows monitored-account count and pending-operation count, but does **not** show an aggregate unread message count. The high-level feature description mentions "unread state." Should the tray icon display an unread count badge, or is that the responsibility of the notification system (epic 8.1)? This needs a design decision.
- **OQ-3. Context menu richness.** Beyond "Open" and "Quit," should the tray context menu include additional actions such as "Synchronize now," "Pause synchronization," "Compose new message," or per-account status entries? The source application's foreground notification has no action buttons; a desktop tray icon typically affords richer interaction. The appropriate menu contents should be decided during design.
- **OQ-4. Multiple-instance prevention.** Should the daemon enforce single-instance semantics (only one daemon per user session), and if so, how should a second launch attempt behave — silently exit, signal the existing instance, or show an error? The source application is single-instance by virtue of Android's activity/service model.
- **OQ-5. Tray icon when no desktop environment tray is available.** Some minimal desktop environments or window managers do not provide a notification area. Should the daemon still run (headless, no tray icon), or should it require a tray to start? The source application's foreground notification is mandatory on Android 8+; on Linux, the tray icon may be optional.
- **OQ-6. Graceful shutdown on session end.** When the desktop session ends (logout, shutdown), should the daemon attempt to complete in-progress operations (e.g. a message being sent) before terminating, or should it terminate immediately and rely on the operation queue to resume on next start? The source application uses a 10-second grace period before self-termination.
- **OQ-7. Tray icon interaction conventions across DEs.** Click behavior varies across desktop environments (single-click vs. double-click to activate, left-click vs. middle-click). Should the application follow the host DE's conventions, or define its own? This is a desktop integration concern to be resolved during implementation.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. The daemon is the application's heartbeat.** In the source application, the foreground service is not optional — it is the mechanism that keeps mail flowing. Every other real-time feature (push notifications, queued operations, IMAP IDLE) depends on the service being alive. The daemon in the desktop application plays the same foundational role: it is not a "nice to have" but the prerequisite for the application's core value proposition.

- **N-2. Minimal, non-intrusive notification.** The source application's foreground notification is deliberately configured at the lowest possible priority: no sound, no vibration, no badge, hidden on the lock screen, not clearable by the user. It exists only because Android requires it for a foreground service. The desktop equivalent (the system tray icon) should follow the same philosophy — present but unobtrusive. It is a status indicator, not an alerting mechanism.

- **N-3. Separation of monitoring notification from mail notifications.** The source application maintains a strict separation between the foreground service notification (channel: "service," importance: minimum) and new-mail notifications (channel: "notification," importance: high). These are independently configurable. The desktop application should maintain this separation: the tray icon is the daemon's status indicator; new-mail notifications are a separate concern governed by the notifications epic (8.1).

- **N-4. Grace period before shutdown.** The source application waits approximately 10 seconds after the last account becomes unnecessary before stopping the service. This prevents rapid stop-start cycling when the user is making configuration changes (e.g. toggling accounts on and off in quick succession). The desktop daemon should implement a similar grace period.

- **N-5. Sticky restart semantics.** The source application's service is configured to restart automatically if killed by the OS (equivalent of "restart on failure" in a service unit). This is critical for reliability — users should not have to manually restart their email client after a transient system resource pressure event.

- **N-6. Aggregate status, not per-account detail.** The source application's foreground notification shows a single aggregate view: "Monitoring N account(s)" and "M operation(s) pending." It does not list individual account names or per-account connection states in the notification itself. Per-account detail is available in the main application UI. The tray icon should follow the same pattern — aggregate summary in the tooltip, detailed status in the main window.

- **N-7. Operations keep the daemon alive.** Even when no accounts are enabled for monitoring, the source application keeps the service running if there are pending operations (messages queued to send, moves to execute). The daemon should follow the same rule: it runs as long as there is work to do, not just as long as there are accounts to watch.

- **N-8. Opening the main window from the tray.** In the source application, tapping the foreground notification opens the unified inbox view. The tray icon should follow the same convention — activating it opens the main window to the user's default view (typically the unified inbox).
