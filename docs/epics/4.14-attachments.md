# Epic: Attachments

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **4.14** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Add files, images (with optional resize / EXIF strip / HEIC re-encode), inline images, vCards, pre-existing message attachments, or 'attach this thread as EML'. Collapsible attachment list."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

When composing an email, users frequently need to include supplementary content beyond the message body: documents, photos, contact cards, or even entire prior messages. The attachment system must provide straightforward mechanisms to add, manage, and present these items — while also protecting user privacy (location data in photos, personal metadata) and respecting the practical constraints of email delivery (file size limits, interoperability across mail clients).

A well-designed attachment system distinguishes between two fundamentally different embedding modes: **inline images** that appear within the flow of the message text, and **regular attachments** that appear as a downloadable list after the message body. Both modes are first-class citizens, and the user must always understand which mode an item is in.

On the reading side, attachments must be presented clearly, downloadable on demand, and safely openable — with appropriate warnings for potentially dangerous content.

## 2. Goals

- **G1.** Allow users to attach arbitrary files to a compose message via a standard file picker.
- **G2.** Allow users to add images either as inline (embedded in the message body) or as regular file attachments, with clear visual distinction.
- **G3.** Provide optional image processing at attach time: resizing, privacy-sensitive metadata removal, and re-encoding from modern formats to widely supported ones.
- **G4.** Support attaching vCards (contact cards), pre-existing attachments from other messages, and entire message threads as EML files.
- **G5.** Present attachments in a collapsible list that stays manageable even when many items are attached.
- **G6.** Provide clear feedback about attachment state: processing progress, size, potential issues (long filenames, dangerous file types, missing downloads).
- **G7.** Remind the user when the message body mentions attachments but none have been added.
- **G8.** Respect encryption state — when a message is encrypted, all attachments are encrypted together as a unit.

## 3. Non-Goals

- **NG1.** This epic does **not** define how attachments are stored locally or synchronized with the server — only how they are presented to and managed by the user during compose and reading.
- **NG2.** Cloud storage integration (e.g. linking to files in remote storage rather than attaching them) is out of scope.
- **NG3.** Virus/malware scanning of attachments is not an in-app responsibility (though integration with external scanning services may be noted).
- **NG4.** Automatic compression of non-image attachments (e.g. auto-zipping folders) is out of scope — though manual zipping of individual attachments is in scope.
- **NG5.** Drag-to-reorder of inline images within the message body is covered by the inline image management epic (4.17), not here.

## 4. Glossary

| Term | Definition |
|---|---|
| **Attachment** | Any file appended to an email message, delivered to the recipient as a downloadable item separate from the message body. |
| **Inline image** | An image embedded within the message body via a content reference, rendered in-place when the recipient reads the message. Distinguished from a regular attachment by its disposition. |
| **Disposition** | The declared intent for how an attached item should be presented: either INLINE (rendered in the body) or ATTACHMENT (listed separately for download). |
| **EXIF / metadata** | Exchangeable Image File Format data embedded in photographs — includes GPS coordinates, timestamps, camera serial numbers, and other potentially privacy-sensitive information. |
| **HEIC / HEIF** | High Efficiency Image Container — a modern image format that offers better compression than JPEG but is not universally supported by all email clients and operating systems. |
| **vCard** | A standard file format (.vcf) for electronic business/contact cards, containing name, email, phone, and other contact fields. |
| **EML** | A standard file format for email messages (RFC 5322), allowing an entire message — headers, body, and attachments — to be saved or forwarded as a single file. |
| **Content-ID (CID)** | A unique identifier that links an inline attachment to its reference point in the HTML message body. |
| **TNEF** | Transport Neutral Encapsulation Format — a proprietary format used by some mail systems to bundle attachments; the application unpacks these for display. |

## 5. Personas

- **P1. The everyday user.** Attaches one or two documents or photos per message; expects the process to be simple and familiar (file picker, drag-and-drop).
- **P2. The privacy-conscious user.** Shares photos but wants GPS coordinates, serial numbers, and original filenames stripped before sending.
- **P3. The power user.** Forwards complex threads as EML, re-attaches files from prior messages, manages many attachments per draft, and uses inline images for formatted newsletters or reports.
- **P4. The mobile-bandwidth user.** Needs image resizing to keep message sizes manageable, especially when attaching high-resolution photographs.
- **P5. The recipient.** Receives messages with attachments and needs to preview, download, and open them safely.

## 6. User Stories

### 6.1 Adding file attachments

> **US-1.** As any user, I want to attach one or more files from my file system to the current draft via a file picker, so that I can share documents with my recipients.

> **US-2.** As any user, I want to drag files from my file manager into the compose window to attach them, so that I can add attachments without navigating a picker dialog.

> **US-3.** As any user, when I share files from another application into the email client, I want those files to appear as attachments in a new or existing draft, so that the standard system sharing mechanism works as expected.

### 6.2 Adding images

> **US-4.** As any user, when I select an image to add, I want to choose whether it should be inserted inline (embedded in the message body) or attached as a regular file, so that I control how the recipient sees it.

> **US-5.** As any user, I want the option to resize images at attach time by choosing a maximum pixel dimension, so that I can reduce the message size without using an external image editor.

> **US-6.** As any user, I want a "limit width only" resize option that constrains only the horizontal dimension while preserving aspect ratio, so that tall images are not excessively shrunk.

> **US-7.** As a privacy-conscious user, I want the option to strip privacy-sensitive metadata (GPS coordinates, timestamps, serial numbers, artist information, original filename) from images before they are attached, so that I do not inadvertently disclose my location or device identity.

> **US-8.** As any user, when I attach a HEIC/HEIF image, I want the application to re-encode it into a widely supported format (JPEG/PNG) if I choose to resize it, so that my recipients can view the image regardless of their platform.

> **US-9.** As any user, I want a global setting that controls whether the image options dialog (resize, privacy strip) is shown every time I add an image, so that I can either always be prompted or skip the dialog if my preferences are fixed.

> **US-10.** As any user, I want a setting to automatically resize images in quoted/replied text, so that replies to image-heavy messages do not balloon in size.

### 6.3 Adding other content types

> **US-11.** As any user, I want to attach a vCard (contact card) to a message, so that I can share contact information in a standard format.

> **US-12.** As a power user, I want to attach the entire current conversation thread as a single EML file, so that I can forward a complete discussion history as a self-contained message file.

> **US-13.** As a power user, when forwarding or replying to a message that has attachments, I want the application to carry over the relevant attachments (all for forwards, only inline images for replies), so that context is preserved without manual re-attachment.

> **US-14.** As any user, I want a dialog or prompt when forwarding/replying to a message with attachments, asking whether to download and include them, so that I have explicit control over what is carried forward.

> **US-15.** As any user, I want to take a photo directly from the compose window and have it attached immediately, so that I can capture and send in one flow.

> **US-16.** As any user, I want to record an audio clip directly from the compose window and have it attached, so that I can include voice notes without leaving the application.

### 6.4 Managing the attachment list in compose

> **US-17.** As any user, I want the attachment list in the compose view to be collapsible, so that when I have many attachments the compose area is not overwhelmed and I can focus on writing.

> **US-18.** As any user, I want the collapsed state to show a summary (total count of attachments), so that I know how many items are queued even when the list is hidden.

> **US-19.** As any user, I want each attachment in the list to show: a type icon, the filename, the file size, and whether it is inline or a regular attachment (visually distinguished), so that I can quickly assess what I have attached.

> **US-20.** As any user, I want to remove a single attachment from the list, so that I can correct mistakes without starting over.

> **US-21.** As any user, I want a "remove all attachments" action, so that I can clear the entire list in one step.

> **US-22.** As a power user, I want to rename an attachment before sending, so that I can give it a more meaningful filename for the recipient.

> **US-23.** As a power user, I want to zip an individual non-inline attachment from the compose list, so that I can compress a large file without leaving the application.

> **US-24.** As any user, I want the attachment list to persist its collapsed/expanded state across compose sessions, so that my preference is remembered.

### 6.5 Warnings and validation

> **US-25.** As any user, if my message body contains words like "attached" or "attachment" but I have not added any attachments, I want to be warned before sending, so that I do not forget to include a referenced file.

> **US-26.** As any user, if an attachment has a filename longer than 60 characters, I want to see a warning that some mail clients may not handle it correctly, so that I can shorten the name if desired.

> **US-27.** As any user, if I attach a file with a potentially dangerous extension (executables, scripts, etc.), I want to see a warning, so that I am aware the recipient's mail system may block or flag it.

> **US-28.** As any user, if the total message size (body plus attachments) is very large, I want a warning indicating that some providers may reject the message, so that I can resize images or remove attachments before attempting to send.

### 6.6 Attachments and encryption

> **US-29.** As a user who encrypts messages (PGP or S/MIME), I want all attachments to be encrypted as part of the encrypted message container, so that the attachments are as protected as the message body.

> **US-30.** As a user who signs messages, I want all attachments to be included in the cryptographic signature, so that recipients can verify the integrity of the full message including attachments.

### 6.7 Viewing and downloading attachments (reading side)

> **US-31.** As a recipient, I want to see a list of all attachments on a received message, with filename, type icon, and file size, so that I know what is available to download.

> **US-32.** As a recipient, I want inline images to be automatically shown within the message body (subject to my remote-content settings), so that the message renders as the sender intended.

> **US-33.** As a recipient, I want to download individual attachments on demand or download all attachments at once, so that I control bandwidth usage.

> **US-34.** As a recipient, I want to see download progress and be notified of download failures with the option to retry, so that I am not left wondering about the state of a download.

> **US-35.** As a recipient, I want to open an attachment using the system's default application for that file type (via the standard "Open with…" mechanism), so that I do not need to manually save and locate the file.

> **US-36.** As a recipient, I want to save an attachment to a location of my choice on disk, so that I can archive it.

> **US-37.** As a recipient, I want to share an attachment directly to another application without saving it to disk first, so that I can forward content efficiently.

> **US-38.** As a recipient, when I receive a message with an EML attachment, I want to be able to open it as a nested message within the application, so that I can read forwarded threads without an external tool.

> **US-39.** As a recipient, when I receive a TNEF-encapsulated message, I want the application to unpack and display the embedded attachments normally, so that I can access the content without specialized tools.

> **US-40.** As a recipient, I want dangerous file type warnings shown before I open a potentially unsafe attachment, so that I am protected from accidental execution of harmful files.

> **US-41.** As a recipient, I want the option to view the contents of a zipped attachment without extracting it, so that I can inspect what is inside before downloading the full archive.

## 7. Functional Requirements

### 7.1 Attachment sources

- **FR-1.** The compose interface shall provide a mechanism to attach one or more arbitrary files via the system file picker.
- **FR-2.** The compose interface shall provide a mechanism to add images, with the user choosing between inline and attachment disposition.
- **FR-3.** The compose interface shall provide a mechanism to capture a photo and attach it directly.
- **FR-4.** The compose interface shall provide a mechanism to record audio and attach it directly.
- **FR-5.** The compose interface shall support drag-and-drop of files into the compose area.
- **FR-6.** The compose interface shall accept files shared from other applications via the system share mechanism.
- **FR-7.** The compose interface shall provide a mechanism to attach a vCard.
- **FR-8.** The compose interface shall provide a mechanism to attach the current conversation thread as a single EML file.
- **FR-9.** When forwarding a message, the application shall copy all non-encrypted attachments from the original message to the new draft.
- **FR-10.** When replying to a message, the application shall copy only inline images (those referenced by Content-ID in the body) from the original message to the reply draft.
- **FR-11.** When forwarding or replying to a message with attachments that have not yet been downloaded, the application shall prompt the user to download them before including them.

### 7.2 Image processing

- **FR-12.** When adding an image, if image options are enabled, the application shall present a dialog offering: (a) inline vs. attach, (b) resize with selectable target pixel dimensions, (c) limit-width-only toggle, (d) remove privacy-sensitive metadata.
- **FR-13.** The image options dialog shall be suppressible via a "don't show again" preference, and re-enableable from settings.
- **FR-14.** Resizing shall support at minimum: JPEG, PNG, WebP, and HEIC/HEIF input formats.
- **FR-15.** When a HEIC/HEIF image is resized, it shall be re-encoded into a universally supported format (JPEG or PNG).
- **FR-16.** When the privacy-strip option is enabled, the application shall remove: GPS coordinates (latitude, longitude, altitude, speed), timestamps, image descriptions, artist information, camera/lens serial numbers, XMP data, user comments, and shall replace the original filename with a generic sequential identifier.
- **FR-17.** Resizing shall use integer scaling factors and preserve aspect ratio.
- **FR-18.** A global setting shall allow automatic resizing of images in replied/quoted text without user intervention per message.
- **FR-19.** The set of supported resize target dimensions shall be configurable in send settings, with a sensible default (e.g. 1440 pixels maximum dimension).

### 7.3 Attachment list (compose)

- **FR-20.** The compose view shall display an attachment list that can be collapsed to a single summary line (showing the total attachment count) or expanded to show all items.
- **FR-21.** The collapsed/expanded state shall persist as a user preference across compose sessions.
- **FR-22.** Each item in the expanded list shall display: a type-appropriate icon, the filename, the file size, a disposition indicator (inline items visually distinguished — e.g. dimmed), and a progress indicator during processing.
- **FR-23.** Each item shall support individual removal.
- **FR-24.** A bulk "remove all attachments" action shall be available.
- **FR-25.** Each item shall support renaming.
- **FR-26.** Non-inline items shall support in-place zipping (compress to a zip archive).
- **FR-27.** EML attachments shall offer a "save as EML" action from their context menu.
- **FR-28.** Image attachments shall offer an "edit image" action for basic adjustments.

### 7.4 Warnings and validation

- **FR-29.** The application shall maintain a list of keywords (configurable per locale) that suggest the user intends to attach a file (e.g. "attached", "attachment", "enclosed"). If any keyword is present in the message body and no attachments have been added, the application shall warn the user before sending.
- **FR-30.** Filenames exceeding 60 characters shall trigger a visible warning citing interoperability concerns.
- **FR-31.** The application shall maintain a list of known dangerous file extensions. Attaching a file with such an extension shall trigger a warning.
- **FR-32.** When the combined message size (body + attachments) exceeds a configurable threshold, the application shall display a size warning. Provider-specific limits (e.g. 25 MB for common providers) shall be documented.
- **FR-33.** If attachments required for forwarding have not been fully downloaded, the application shall display a clear indicator and prevent sending until the downloads complete or the user removes the incomplete items.

### 7.5 Encryption interaction

- **FR-34.** When a message is signed and/or encrypted (PGP or S/MIME), all attachments shall be included within the signed/encrypted container — not sent separately.
- **FR-35.** Encryption-related attachments (signature files, encrypted containers) shall be tracked separately and shall not appear as user-visible regular attachments.

### 7.6 Reading-side behavior

- **FR-36.** The message viewer shall display all non-inline attachments in a list showing: type icon, filename, file size, and download state (not downloaded / downloading / available).
- **FR-37.** Inline images shall be rendered within the message body at their content-reference positions, subject to the user's remote-content and auto-download settings.
- **FR-38.** The user shall be able to toggle visibility of inline images independently of regular attachments.
- **FR-39.** Individual attachments shall be downloadable on demand; a "download all" action shall also be available.
- **FR-40.** Downloaded attachments shall be openable via the system's default application handler.
- **FR-41.** Downloaded attachments shall be saveable to a user-chosen location.
- **FR-42.** Downloaded attachments shall be shareable to other applications via the system share mechanism.
- **FR-43.** EML attachments shall be openable as nested messages within the application.
- **FR-44.** TNEF-encapsulated content shall be automatically unpacked and its embedded attachments presented as normal attachments.
- **FR-45.** Zipped attachments shall support previewing their contents list without extraction.
- **FR-46.** Attachments with dangerous file extensions shall trigger a warning before opening.
- **FR-47.** Download progress shall be displayed per attachment, and failed downloads shall offer a retry action with a visible error message.

### 7.7 Shared behavior

- **FR-48.** A setting shall control whether files shared into the application from external sources create a new draft or are appended to the currently open draft.
- **FR-49.** The application shall detect and correctly handle the MIME type of all attached files, guessing the extension if one is not provided and sanitizing filenames to prevent path-traversal attacks.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Adding an attachment (file pick to list appearance) shall complete in under two seconds for files below 10 MB under normal conditions. Image resize operations shall show progress feedback if they exceed one second.
- **NFR-2. Scale.** The compose attachment list shall remain usable with at least 50 attachments without degrading compose performance.
- **NFR-3. Privacy.** Metadata stripping shall be thorough and verifiable — no residual GPS, serial number, or XMP data shall survive the strip operation. Stripped filenames shall reveal no information about the original file path.
- **NFR-4. Interoperability.** Attachments shall be encoded using standard MIME (RFC 2045–2049) with dispositions per RFC 2183. Inline images shall use Content-ID references. The output shall be consumable by all major mail clients.
- **NFR-5. Safety.** The application shall never automatically execute or open an attachment without explicit user action. Dangerous-extension warnings shall not be dismissible permanently (each occurrence requires acknowledgement).
- **NFR-6. Offline.** Attachment management in compose (add, remove, rename, reorder) shall work fully offline. Only the actual send requires connectivity.
- **NFR-7. Accessibility.** All attachment list items, action buttons, and dialogs shall be keyboard-navigable and carry appropriate screen-reader labels.

## 10. Open Questions

- **OQ-1. Provider-specific size limits.** The application documents Gmail (25 MB), Outlook (20 MB), and Yahoo (25 MB) limits in its FAQ but does not enforce hard per-provider caps programmatically. Should the desktop application detect the provider and apply provider-specific hard limits, or continue with advisory warnings only?
- **OQ-2. Auto-resize default behavior.** The source application offers automatic resizing of images in reply text as a setting, but it is unclear whether this defaults to on or off for new installations. The desktop application should define a clear default.
- **OQ-3. HEIC re-encoding scope.** Re-encoding currently applies only when the user explicitly resizes. Should HEIC images always be re-encoded to JPEG/PNG on attach (even without resize) to maximize recipient compatibility, or should the original format be preserved when no resize is requested?
- **OQ-4. Drag-and-drop inline vs. attach.** When a user drags an image into the compose body (as opposed to the attachment area), should it default to inline insertion or regular attachment? The source application's behavior here may vary by context.
- **OQ-5. Audio recording format.** The source application delegates audio recording to the system recorder. Should the desktop application specify a preferred audio format (e.g. OGG, MP3, WAV) for maximum email-client compatibility, or accept whatever the system recorder produces?
- **OQ-6. Attachment deduplication.** If a user adds the same file twice (identical content, same name), should the application warn about duplication, silently allow it, or prevent it? The source application appears to allow duplicates.
- **OQ-7. External storage for large attachments.** The source application had (and later removed for policy reasons) an option to store attachments on external storage. The desktop application has no equivalent policy constraint — should it offer configurable attachment storage location?
- **OQ-8. Zip preview depth.** The source application shows the contents of zip files. Should nested zips (zip within zip) also be previewable, or only the top level?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Inline vs. attachment is explicit.** The source application always asks the user to choose between inline and attachment when adding an image. This is intentional: silently choosing one mode would confuse users who expect the other, and the distinction has real consequences for how recipients see the message. The dialog can be suppressed by users who always want the same choice, but the default is to ask.
- **N-2. Privacy strip is opt-in per image.** Metadata removal is not automatic — it requires the user to check the option in the image dialog (or configure it as a default in settings). This avoids the surprise of silently modifying files and lets users who intentionally share location data (e.g. "here's where I took this photo") do so without friction.
- **N-3. Resize uses whole-number scaling factors.** Rather than resampling to an exact pixel target (which can produce interpolation artifacts), the source application rounds to the nearest whole-number divisor of the original dimensions. This preserves sharpness at the cost of not hitting the exact target size.
- **N-4. Inline images are dimmed in the list.** The visual dimming of inline attachments in the list is a deliberate UX choice: it signals "this item is embedded in the body above, not a separate downloadable" without removing it from the list entirely. Users still need to see inline items listed (for removal, renaming, or disposition change).
- **N-5. Attachment reminder is keyword-based.** The "did you forget to attach?" reminder uses a simple locale-specific keyword list scanned against the message body. This is intentionally naive — false positives (e.g. "see the attached document" when the user means a previous email) are acceptable because the prompt is easily dismissed. Machine-learning-based detection is out of scope.
- **N-6. Forward carries all; reply carries only inline.** This asymmetry reflects real-world expectations: forwarding means "pass along everything", while replying means "respond to the content" — carrying all attachments on a reply would balloon thread sizes unnecessarily. Only images that are visually part of the message body (inline, referenced by CID) are preserved on reply.
- **N-7. Collapsible attachment list default.** The list defaults to expanded when few attachments are present and remembers the user's explicit collapse/expand action. This balances visibility (new users see what they have attached) with manageability (power users with many attachments collapse it).
- **N-8. Dangerous-extension list matches common provider behavior.** The list of flagged extensions mirrors what major mail providers (Gmail, Outlook) block or warn about, ensuring the user is warned about files that will likely be blocked on the receiving end regardless of local safety.
