# Epic: Tracker-Image Detection

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.11** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Heuristic detection of tracking pixels using bundled blocklists (Disconnect, AdGuard, etc.) — blocked images are reported."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[SHOULD]` — important, planned for v1 but acceptable to slip to v1.x if pressed.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Marketing emails, newsletters, and many routine messages contain invisible or near-invisible images — commonly called "tracking pixels" — whose sole purpose is to inform the sender when (and sometimes where, on which device, and how often) a message is opened. This is a significant privacy concern: the user never consents to being tracked, and the images convey no useful content.

A privacy-respecting email client should detect these tracking images and prevent them from phoning home, while still making the user aware that tracking was attempted. Detection relies on two complementary strategies:

1. **Heuristic detection** — identifying images that are invisible, extremely small (a few pixels or fewer), or styled to be hidden, regardless of their hosting domain.
2. **Blocklist-based detection** — checking the hosting domain of each image against curated, community-maintained lists of known tracker domains.

Both strategies work together: heuristics catch novel or self-hosted trackers that no list covers, and blocklists catch trackers that are delivered at normal image sizes or that evade simple size-based heuristics.

Detected tracking images are suppressed (not loaded, or replaced with a visible indicator) and the user is informed of the detection — so that tracking is blocked transparently rather than silently.

## 2. Goals

- **G1.** Detect tracking images in incoming email messages using a combination of size/visibility heuristics and curated domain blocklists, and prevent those images from being loaded (and therefore from reporting back to the sender).
- **G2.** Replace each detected tracking image with a clearly visible indicator so the user knows tracking was attempted and can inspect the original URL if desired.
- **G3.** Support multiple, independently togglable blocklist sources so that the user can choose the level of protection they prefer.
- **G4.** Keep blocklists up to date via periodic automatic downloads, while also supporting manual refresh.
- **G5.** Allow per-category granularity within blocklists so that the user can selectively permit categories they consider acceptable (e.g. social media widgets) while blocking others (e.g. analytics).
- **G6.** Integrate URL-parameter stripping so that even when an image or link is permitted, known tracking parameters are removed before any network request is made.

## 3. Non-Goals

- **NG1.** This epic does **not** cover the decision of whether to load external images at all — that is the domain of the external-image confirmation feature (feature 3.10). Tracker detection operates *within* the set of images that would otherwise be loaded or displayed.
- **NG2.** This epic does **not** define link-click tracking protection (e.g. redirect unwinding or bounce-tracking mitigation) except insofar as blocklist domain checks also apply to link destinations.
- **NG3.** This epic does **not** cover general ad-blocking or content filtering beyond tracker detection.
- **NG4.** This epic does **not** prescribe the internal format or storage mechanism of blocklist data — only the user-visible behavior and update contract.
- **NG5.** This epic does **not** require the application to inform the *sender* that tracking was blocked.

## 4. Glossary

| Term | Definition |
|---|---|
| **Tracking pixel** | An image embedded in an email whose primary purpose is to signal the sender when the message is opened. Typically 1×1 pixel or invisible, but may be larger when hosted on known tracker domains. |
| **Tracking image** | Synonym for tracking pixel; used when the image may not literally be one pixel in size but still serves a tracking function. |
| **Blocklist** | A curated, externally maintained list of domains and/or URL patterns known to serve tracking content. |
| **Blocklist category** | A thematic grouping within a blocklist (e.g. "Analytics", "Advertising", "Social Media") that the user can enable or disable independently. |
| **Heuristic detection** | Identification of a tracking image based on its intrinsic properties (dimensions, CSS styling, known beacon class names) rather than its hosting domain. |
| **Tracking indicator** | A small, visible icon that replaces a suppressed tracking image in the rendered message, informing the user that a tracker was detected. |
| **Tracking parameter** | A URL query-string parameter whose purpose is to carry a tracking identifier (e.g. click IDs, campaign tags, subscriber tokens). Removing it does not affect the destination resource. |
| **Filter list** | A text-based rule file (in Adblock-style syntax) specifying URL patterns and parameters to strip. |
| **Auto-update** | Periodic, unattended re-download of blocklists and filter lists to keep them current. |

## 5. Personas

- **P1. The privacy-conscious user.** Wants maximum protection from email tracking; enables all blocklists and heuristics; appreciates being informed when tracking is blocked.
- **P2. The casual user.** Has the feature enabled by default and rarely thinks about it; expects reasonable protection without configuration.
- **P3. The newsletter reader.** Subscribes to many marketing emails and accepts that some tracking exists; wants fine-grained control so they can permit certain categories (e.g. social media embeds) while blocking analytics.
- **P4. The power user.** Wants to inspect exactly which URLs were blocked, understand why, and override on a per-message basis when needed.

## 6. User Stories

### 6.1 Detection and suppression

> **US-1.** As any user, when I open a message that contains a tracking pixel (an image that is 1×1 pixel, invisible, or extremely small), I want the application to detect it and prevent it from loading, so that my open is not reported to the sender.

> **US-2.** As any user, when I open a message that contains an image hosted on a known tracker domain (per an enabled blocklist), I want the application to suppress that image regardless of its size, so that sophisticated trackers that use normal-sized images are also blocked.

> **US-3.** As any user, when a tracking image is suppressed, I want to see a visible indicator icon in its place, so that I know tracking was attempted and was blocked.

> **US-4.** As a power user, when I see a tracking indicator in a message, I want to be able to tap or click it to see the original URL that was blocked, so that I can inspect the tracker's identity.

> **US-5.** As any user, I want heuristic detection and blocklist detection to operate together, so that a tracker is caught if it matches *either* criterion.

### 6.2 Blocklist management

> **US-6.** As any user, I want the application to ship with at least one bundled tracker-protection blocklist enabled by default, so that I have protection from the moment I start using the application.

> **US-7.** As a privacy-conscious user, I want to enable or disable each blocklist source independently, so that I can choose my preferred level of protection.

> **US-8.** As a newsletter reader, I want to enable or disable individual categories within a blocklist (e.g. permit "Social Media" but block "Analytics" and "Advertising"), so that I get granular control without all-or-nothing.

> **US-9.** As any user, I want blocklists to be updated automatically on a regular schedule (e.g. weekly) without my intervention, so that new trackers are caught as they appear.

> **US-10.** As any user, I want to be able to trigger a manual download of blocklists at any time, so that I can force an update if I suspect a list is stale.

> **US-11.** As any user, I want to see when the blocklists were last updated, so that I can judge whether my protection is current.

### 6.3 URL-parameter stripping

> **US-12.** As any user, when I click a link in an email, I want known tracking parameters to be stripped from the URL before my browser is opened, so that the destination site does not receive a tracking token tied to my email address.

> **US-13.** As any user, I want URL-parameter stripping to be governed by a separate, updatable filter list, so that new tracking parameters can be handled without a software update.

> **US-14.** As any user, I want parameter stripping to operate independently of image blocking, so that I can have links cleaned even if I choose to permit all images.

### 6.4 Settings and control

> **US-15.** As any user, I want a master toggle for tracking-image detection, so that I can disable the entire feature if I choose.

> **US-16.** As any user, I want the feature to be enabled by default, so that I am protected without having to discover and enable a setting.

> **US-17.** As any user, I want all tracker-detection settings grouped in a dedicated privacy section of the application preferences, so that they are easy to find and understand.

> **US-18.** As a power user, I want separate toggles for: (a) heuristic pixel detection, (b) blocklist-based image detection, and (c) URL-parameter stripping, so that I can enable each capability independently.

### 6.5 Interaction with external-image loading

> **US-19.** As any user, when I have chosen to load external images for a message (via the external-image confirmation feature), I still want tracking-image detection to suppress known trackers, so that allowing images does not disable privacy protection.

> **US-20.** As any user, when images are blocked entirely (because I have not confirmed external-image loading), tracker detection is moot — but if I later confirm loading, tracker detection should apply at that point.

## 7. Functional Requirements

### 7.1 Heuristic detection

- **FR-1.** The application shall detect tracking images by inspecting each image element's declared dimensions (width and height attributes), computed CSS dimensions, and visibility properties.
- **FR-2.** An image shall be classified as a tracking pixel if its surface area (width × height) is at or below a defined small-image threshold (on the order of a few dozen square pixels).
- **FR-3.** An image shall be classified as a tracking pixel if its CSS styling renders it invisible (e.g. zero dimensions, hidden overflow, opacity zero, display none) while still causing a network request.
- **FR-4.** An image shall be classified as a tracking pixel if it carries a well-known beacon class name or structural pattern used by recognized email-tracking services.
- **FR-5.** Heuristic detection shall operate on the message's HTML structure before images are loaded, so that a tracking pixel is never fetched.

### 7.2 Blocklist-based detection

- **FR-6.** The application shall support at least two independent blocklist sources (e.g. a domain-based tracker-protection list and a URL-pattern-based filter list).
- **FR-7.** Each blocklist source shall be individually enableable and disableable by the user.
- **FR-8.** For blocklists that define categories, the user shall be able to enable or disable each category independently.
- **FR-9.** An image whose hosting domain matches any enabled blocklist entry shall be classified as a tracking image, regardless of its size or styling.
- **FR-10.** The application shall also apply blocklist domain checks to link destinations (not only images), when the relevant blocklist-for-links toggle is enabled.
- **FR-11.** A hardcoded, application-maintained list of universally recognized tracking domains shall serve as a baseline that is always active when the master tracking-detection toggle is enabled, independent of any downloadable blocklist.

### 7.3 Replacement and reporting

- **FR-12.** When an image is classified as a tracking image (by either heuristic or blocklist), its source URL shall be replaced with a locally generated indicator icon before the message is rendered.
- **FR-13.** The indicator icon shall be a clearly distinguishable, fixed-size graphic (e.g. a small warning or location pin icon) that does not resemble normal message content.
- **FR-14.** The original (blocked) URL shall be preserved in association with the indicator, so that the user can inspect it on demand.
- **FR-15.** When the user activates (clicks/taps) a tracking indicator, the application shall display the original tracking URL — clearly labeled as a tracking image — and allow the user to choose whether to open it or dismiss.
- **FR-16.** The alt-text of the replacement image shall be set to a human-readable label (e.g. "Tracking image") for accessibility.

### 7.4 URL-parameter stripping

- **FR-17.** The application shall support a filter-list format that defines URL parameters to strip, including pattern-based and regex-based parameter matching.
- **FR-18.** When a user clicks a link in an email, the application shall remove all matching tracking parameters from the URL before passing it to the system for opening.
- **FR-19.** Parameter stripping shall respect exception rules defined in the filter list (e.g. specific domains excluded from a given rule).
- **FR-20.** Parameter stripping shall be togglable independently of image-tracker detection.

### 7.5 Blocklist updates

- **FR-21.** The application shall support automatic periodic downloading of updated blocklists and filter lists (default interval: weekly).
- **FR-22.** Auto-update shall be individually togglable per list source.
- **FR-23.** The application shall provide a manual "download now" action for each list source, accessible from the privacy settings.
- **FR-24.** The application shall store and display the timestamp of the most recent successful download for each list source.
- **FR-25.** If a download fails (network error, malformed data), the application shall continue using the previously cached version of the list and shall not disable protection.
- **FR-26.** On first launch (before any download has occurred), the application shall use a bundled snapshot of each list so that protection is available immediately.

### 7.6 Settings

- **FR-27.** A master toggle ("Detect tracking images") shall enable or disable all tracker-detection behavior. It shall default to **enabled**.
- **FR-28.** A separate toggle shall control whether blocklist domain checks apply to images. It shall default to **disabled** (heuristic detection alone is the baseline).
- **FR-29.** A separate toggle shall control whether blocklist domain checks apply to links. It shall default to **enabled**.
- **FR-30.** A separate toggle shall control URL-parameter stripping via the filter list. It shall default to **enabled**.
- **FR-31.** Per-category toggles shall be available for each blocklist that defines categories.
- **FR-32.** All tracker-detection settings shall be grouped within the application's privacy preferences section.

### 7.7 Interaction with other features

- **FR-33.** Tracker detection shall operate as a post-processing step on the sanitized HTML of a message, after general HTML sanitization but before final rendering.
- **FR-34.** Tracker detection shall apply regardless of whether external images are permitted for the message — if images are permitted, trackers within that set are still suppressed.
- **FR-35.** Tracker detection results (indicator icons, preserved URLs) shall be part of the rendered message's persistent display state, so that the user sees consistent indicators on every view of the same message without re-detection.

## 8. Non-Functional Requirements

- **NFR-1. Performance.** Tracker detection (both heuristic and blocklist lookup) shall complete within a negligible fraction of the message rendering time — the user shall not perceive any delay attributable to detection.
- **NFR-2. Memory efficiency.** Blocklist data structures shall be loaded once at application startup and shared across all message renderings, not re-parsed per message.
- **NFR-3. Offline operation.** Tracker detection shall function fully offline using cached blocklist data. No network connectivity shall be required for the detection step itself.
- **NFR-4. Update resilience.** A corrupted or empty downloaded list shall never degrade protection below the bundled baseline snapshot.
- **NFR-5. Privacy.** The feature shall not introduce any outbound network requests beyond the scheduled blocklist downloads. Detection is local; results are never reported to any external service.
- **NFR-6. Accessibility.** Tracking indicators shall carry appropriate alt-text and be navigable by assistive technology, so that visually impaired users are informed of blocked trackers.
- **NFR-7. Extensibility.** Adding a new blocklist source in the future shall require only configuration (URL, format, category structure), not architectural changes to the detection pipeline.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|

| AC-3 | A message containing a normal-sized image hosted on a domain *not* in any blocklist and *not* matching heuristic criteria displays the image normally (no false positive). |
| AC-4 | Clicking/tapping a tracking indicator displays the original blocked URL, clearly labeled as a tracking image. |

| AC-7 | With auto-update enabled, blocklists are refreshed within the configured interval (e.g. weekly) without user intervention. |
| AC-8 | A manual "download now" action fetches the latest list and updates the "last updated" timestamp displayed in settings. |
| AC-9 | If the network is unavailable during a scheduled update, the previously cached list remains in use and detection continues to function. |

| AC-11 | A link containing known tracking parameters (e.g. click IDs, campaign tokens) opens in the browser with those parameters stripped, while the base URL and non-tracking parameters remain intact. |

| AC-13 | Tracker detection operates correctly regardless of whether external images are globally permitted or individually permitted for a message. |

## 10. Open Questions

- **OQ-1. Scope of "known beacon class names."** The heuristic detection references specific CSS class names used by certain email-tracking services. The canonical list of such class names should be maintained — but it is unclear whether this list should be user-extensible or strictly application-maintained. The source application hardcodes a small set.
- **OQ-2. Interaction with cached/inline images.** If a tracking image is embedded as a `cid:` (content-ID) attachment rather than fetched remotely, heuristic detection based on size still applies, but the privacy harm is arguably lower (no network request). Should `cid:` images matching heuristic criteria still be replaced with an indicator, or only flagged informatively?
- **OQ-3. Hardcoded baseline vs. blocklist overlap.** The application maintains a small hardcoded list of universally known tracker domains as a baseline. If this list overlaps with entries in a downloadable blocklist, should the hardcoded entries be removable by disabling the blocklist category? The current behavior suggests the hardcoded list is unconditional (always active when the master toggle is on).
- **OQ-4. Debounce / redirect-unwinding scope.** The source application bundles a list of known tracking-redirect patterns (where a URL redirects through a tracker before reaching the final destination). The boundary between "tracker-image detection" and "link-redirect protection" is fuzzy; this epic has included URL-parameter stripping but left full redirect unwinding out of scope. A future epic may be needed for comprehensive link-tracking protection.
- **OQ-5. User override per message.** Should the user be able to say "show all blocked trackers in this specific message" as a one-time override (analogous to "load images for this message")? The source application does not appear to provide this; it is unclear whether it should.
- **OQ-6. Notification of detection count.** Should the application surface a count of blocked trackers per message (e.g. "3 trackers blocked") in the message header or status area, or is the presence of indicator icons in the body sufficient? The source application relies on in-body indicators only.

## 11. Design Notes & Rationale

- **N-1. Defense in depth: heuristics plus blocklists.** Neither approach alone is sufficient. Heuristics miss trackers that use normal-sized images (e.g. a 200×1 image or a branded footer containing a tracking beacon). Blocklists miss self-hosted or novel tracker domains. The two together provide substantially better coverage than either alone. This is why they are exposed as independent toggles — a user who objects to false positives from blocklists can rely on heuristics alone, while a user who wants maximum coverage enables both.
- **N-2. Blocklist for images defaults to off; for links defaults to on.** This asymmetry reflects a risk/benefit tradeoff: blocking an image that turns out to be legitimate content is visually disruptive (the user sees an indicator where they expected a picture), whereas stripping a tracking parameter from a link is invisible to the user and has no downside unless the parameter is required for functionality (which is rare for tracking parameters, by definition). The heuristic approach (which is very rarely wrong for images) is the default image protection; blocklist-for-images is opt-in for users who want stronger coverage and accept the small risk of false positives.
- **N-3. Indicator rather than silent suppression.** The design deliberately shows the user *that* tracking was blocked, rather than silently removing the image with no trace. This serves trust: the user can verify the feature is working, can inspect what was blocked, and is not surprised by missing content. It also enables informed overrides if the user wants to load a specific "tracker" that they judge to be benign.
- **N-4. Weekly auto-update cadence.** Tracker domains change, but not so rapidly that daily updates are justified. Weekly strikes a balance between freshness and resource consumption (bandwidth, battery, background processing). The user can always force a manual update if they need immediate freshness.
- **N-5. Bundled snapshot for day-zero protection.** Because the application cannot guarantee network access on first run, and because tracker detection is a privacy feature whose absence is harmful, a snapshot of each list is bundled with the application. This ensures protection is *never* zero, even before the first successful download.
