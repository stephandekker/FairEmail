# Epic: Offline Storage and Operations

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **7.5** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Offline storage and operations. All actions queue locally and replay when the connection returns."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email client that relies on constant network connectivity is fragile — connections drop, laptops sleep, servers go down, and users travel. To remain useful in all conditions, the application must give the user the experience of a fully local mailbox: actions take effect immediately in the UI, messages that have been fetched remain readable, and nothing the user does while offline is lost.

The Offline Storage and Operations feature ensures that:

1. Every user action (flag, move, delete, reply, send, etc.) is accepted instantly regardless of connectivity, stored locally, and replayed against the mail server as soon as a connection is available.
2. Previously synchronized messages remain readable without a network connection.
3. The transition between online and offline states is invisible to the user under normal conditions — no explicit "go offline" or "go online" modes.

This feature is foundational. Without it, every other feature in the application (Unified Inbox, rules, search, compose) would degrade or break when the network is unavailable.

## 2. Goals

- **G1.** Accept every user action instantly and persist it locally, regardless of current connectivity, so that the user never waits on the network for their mailbox to respond.
- **G2.** Replay all locally queued actions faithfully and in a deterministic order when connectivity is restored, so that the server state converges with the user's intent.
- **G3.** Keep previously downloaded message content (bodies, attachments, headers) available for offline reading, so that the user can continue working without a connection.
- **G4.** Handle transient and permanent server errors gracefully, with automatic retries for transient failures and clear user-visible reporting for permanent ones.
- **G5.** Make the queue of pending operations visible and manageable, so that the user retains control and can cancel or inspect any action that has not yet been committed to the server.
- **G6.** Resolve conflicts (e.g. a message deleted on the server while a local action targets it) without data loss or user confusion.

## 3. Non-Goals

- **NG1.** This epic does **not** define the synchronization schedule, polling interval, or push mechanism; it assumes that *some* connectivity event eventually triggers replay. The scheduling concern belongs to features 7.1 (IMAP IDLE) and 7.2 (configurable poll interval).
- **NG2.** This epic does **not** prescribe the local storage format, schema, or caching engine — only the behavioral contracts around what is stored and when.
- **NG3.** Full offline search indexing is out of scope; that belongs to feature 5.2 (local full-text search index). This epic requires only that already-fetched messages be *readable* offline, not necessarily *searchable*.
- **NG4.** Network-aware behavior (metered vs. unmetered, VPN detection) is out of scope; that belongs to feature 7.4. This epic treats connectivity as a binary: available or unavailable.
- **NG5.** This epic does **not** address initial synchronization of a newly added account — only the offline behavior once messages have been fetched at least once.

## 4. Glossary

| Term | Definition |
|---|---|
| **Operation** | A single discrete action (e.g. mark-read, move, send) that the user has initiated and that must eventually be committed to the mail server. |
| **Operation queue** | The ordered, persistent local store of all operations that have not yet been successfully committed to the server. |
| **Replay** | The process of executing queued operations against the mail server when connectivity becomes available. |
| **Optimistic UI update** | The practice of immediately updating the local display state to reflect the user's intent, before the server has confirmed the action. |
| **Transient failure** | A server or network error that may resolve on its own (timeout, temporary unavailability, rate limit). |
| **Permanent failure** | A server error that will not resolve by retrying (message not found, permission denied, quota exceeded, folder deleted). |
| **Local content** | Message bodies, headers, and attachment files that have been downloaded and stored locally for offline access. |
| **Pending operation** | An operation that is waiting in the queue and has not yet been attempted or has failed transiently and is awaiting retry. |
| **Failed operation** | An operation that has exhausted its retry attempts or encountered a permanent error. |

## 5. Personas

- **P1. The commuter.** Works on email during a daily train ride with intermittent connectivity. Expects to flag, move, delete, and reply to messages without interruption; expects all actions to "just work" when they get to the office.
- **P2. The traveler.** Works offline for extended periods (flights, remote locations). Needs to compose and queue outbound mail, and read all previously synced messages.
- **P3. The triage-heavy user.** Processes hundreds of messages in short bursts; needs bulk actions (mark all read, move batch to folder) to be instant regardless of server latency.
- **P4. The power user.** Wants to see what is pending, cancel a mistaken move before it reaches the server, and understand why an operation failed.

## 6. User Stories

### 6.1 Immediate local acceptance of actions

> **US-1.** As any user, when I perform an action on a message (mark read, flag, move, delete, archive, etc.), I want the UI to reflect the change instantly, so that I am not blocked by network latency or disconnection.

> **US-2.** As any user, when I compose and send a message while offline, I want the message to be accepted into an outgoing queue and delivered automatically when connectivity returns, so that I can write mail at any time.

> **US-3.** As any user, when I perform a bulk action on multiple messages, I want all of them to be queued and reflected in the UI immediately, so that triage remains fast.

### 6.2 Automatic replay on reconnection

> **US-4.** As a commuter, when my device reconnects to the network, I want all queued actions to be replayed against the server automatically without any manual intervention, so that my mailbox reaches the intended state without me thinking about it.

> **US-5.** As any user, I want operations to be replayed in a deterministic, priority-aware order, so that dependencies between actions are respected (e.g. a "move" completes before a "flag" on the moved message).

> **US-6.** As any user, I want the replay process to batch similar operations together where possible (e.g. marking 50 messages as read in one server exchange), so that replay completes quickly even after a long offline session.

### 6.3 Offline reading

> **US-7.** As a traveler, I want to read the full body and view attachments of any message that was previously downloaded, without needing a network connection, so that my reading flow is uninterrupted.

> **US-8.** As any user, I want message previews and metadata (sender, subject, date, flags) to always be available locally for messages within my configured sync window, so that the message list is always usable.

> **US-9.** As any user, when I attempt to open a message whose body has not been downloaded, I want a clear indication that the content is unavailable offline rather than a blank or broken view, so that I understand the limitation.

### 6.4 Error handling and retries

> **US-10.** As any user, I want transient failures (timeouts, temporary server errors) to be retried automatically a reasonable number of times, so that I am not bothered by normal network hiccups.

> **US-11.** As any user, when an operation fails permanently (e.g. the target message was deleted on the server, the destination folder no longer exists, quota exceeded), I want to be notified clearly, so that I can decide what to do.

> **US-12.** As any user, I want failed operations to remain visible in the queue with their error reason, so that I can understand what happened and take corrective action.

### 6.5 Queue visibility and management

> **US-13.** As a power user, I want to view all pending operations in an operations screen, showing operation type, target message/folder, account, and current status, so that I know what is waiting to happen.

> **US-14.** As any user, I want to cancel (delete) a pending operation before it has been committed to the server, so that I can undo a mistake without it ever reaching the server.

> **US-15.** As any user, I want to cancel a queued outgoing message (undo send) while it is still pending in the queue, recovering it as a draft, so that I can fix mistakes.

> **US-16.** As a power user, I want to clear operations in bulk by type (e.g. clear all failed operations, clear all pending fetch operations), so that I can manage a backlog efficiently.

### 6.6 Conflict resolution

> **US-17.** As any user, when an operation targets a message that no longer exists on the server, I want the operation to be silently abandoned without error noise for non-destructive actions (e.g. marking a deleted message as read), so that normal conflicts do not clutter my attention.

> **US-18.** As any user, when a destructive conflict occurs (e.g. I moved a message offline but the destination folder was deleted on the server), I want to be informed so that I can decide what to do with the affected message.

### 6.7 Persistence across restarts

> **US-19.** As any user, I want all pending operations to survive application restarts, system reboots, and crashes, so that nothing I did while offline is lost.

> **US-20.** As any user, I want locally cached message content to persist across application restarts, so that offline reading does not depend on the application session.

## 7. Functional Requirements

### 7.1 Operation creation and local state

- **FR-1.** When the user initiates any mail action, the system shall immediately create a persistent operation record and update the local message state to reflect the user's intent (optimistic UI update).
- **FR-2.** The local UI state (read/unread, flagged/unflagged, deleted, moved, etc.) shall always reflect the user's most recent action, regardless of whether the server has confirmed it.
- **FR-3.** Operation records shall persist in durable local storage and survive application restart, crash, and system reboot.
- **FR-4.** Each operation shall record: the operation type, the target entity (message or folder), the owning account, any arguments (e.g. destination folder, flag color), and the creation timestamp.

### 7.2 Supported operation types

- **FR-5.** The following user actions shall be queueable as operations:
  - Mark message as read / unread
  - Mark message as answered / forwarded
  - Flag / unflag a message (with optional color)
  - Add or remove keywords / labels
  - Move a message to another folder
  - Copy a message to another folder
  - Delete a message
  - Download message body
  - Download message attachment(s)
  - Download message headers / raw source
  - Send a composed message
  - Subscribe / unsubscribe a folder
  - Purge deleted messages from a folder
  - Report message as spam or phishing
  - Edit message subject (where server supports)

- **FR-6.** For protocols with limited server-side capability (e.g. POP3), operations that have no server-side effect shall execute locally only, updating the local state without attempting server communication.

### 7.3 Operation ordering and priority

- **FR-7.** Each operation shall be assigned a priority level that determines its execution order relative to other pending operations.
- **FR-8.** Priority ordering shall ensure that:
  - Content retrieval (body, attachment download) executes at highest priority, so the user can read messages as soon as connectivity allows.
  - Message creation and deletion execute at high priority.
  - State changes (read, flag, keyword) execute at normal priority.
  - Bulk folder operations (sync, purge, expunge) execute at lower priority.
  - Outgoing mail (send) executes at normal priority.
- **FR-9.** Within the same priority level, operations shall execute in creation order (first-in, first-out).

### 7.4 Replay and execution

- **FR-10.** When network connectivity becomes available, the system shall automatically begin processing the operation queue without user intervention.
- **FR-11.** Operations shall be executed against the mail server in priority order, then creation order within each priority level.
- **FR-12.** Where multiple operations of the same type target different messages in the same folder (e.g. 20 mark-as-read operations), the system shall batch them into a single server exchange where the protocol supports it.
- **FR-13.** The batch size shall be bounded (configurable upper limit) to prevent excessively large server commands. If a batch is rejected, the system shall fall back to smaller batches or individual execution.
- **FR-14.** A successfully executed operation shall be removed from the queue.
- **FR-15.** Multiple accounts shall be able to process their operation queues in parallel, independently.

### 7.5 Retry and error handling

- **FR-16.** Each operation shall maintain a try counter, incremented on each execution attempt.
- **FR-17.** On transient failure (network timeout, temporary server error, rate limit), the system shall retry the operation automatically, up to a configured maximum number of attempts.
- **FR-18.** A brief delay shall be introduced between retry attempts to avoid hammering a struggling server.
- **FR-19.** On permanent failure (message not found on server, folder not found, quota exceeded, permission denied, read-only folder), the operation shall be marked as failed immediately without further retries.
- **FR-20.** Failed operations shall remain in the queue with their error details recorded, visible to the user, until explicitly dismissed or deleted.
- **FR-21.** When a batched group of operations fails, the system shall retry them individually to isolate the specific failing operation(s).

### 7.6 Conflict resolution

- **FR-22.** If an operation targets a message that no longer exists on the server, the operation shall be abandoned. For non-destructive operations (mark read, flag), this shall be silent. For destructive or structural operations (move, copy), the user shall be notified.
- **FR-23.** If a move or copy operation targets a folder that no longer exists on the server, the operation shall fail permanently and the user shall be notified.
- **FR-24.** If the server's current message state already matches the operation's intent (e.g. message is already flagged), the operation shall be treated as successfully completed.
- **FR-25.** Temporary local message copies created during move operations shall be cleaned up if the operation fails permanently, preventing orphaned messages.

### 7.7 Duplicate suppression

- **FR-26.** If multiple consecutive identical operations are queued for the same message (same type, same arguments), the system shall suppress the duplicates and execute only one.

### 7.8 Queue management

- **FR-27.** The application shall provide a dedicated operations view showing all pending and failed operations, with columns for: operation type, target message/folder, owning account, status (pending/executing/failed), error message (if failed), and creation time.
- **FR-28.** The user shall be able to delete individual operations from the queue.
- **FR-29.** The user shall be able to delete operations in bulk, filtered by type (e.g. all failed, all fetch, all move, all flag, all delete, all send).
- **FR-30.** When a queued operation is deleted, any optimistic UI state it introduced shall be reverted to reflect the server's last known state.
- **FR-31.** Deleting a queued send operation shall recover the message as a draft in the user's drafts folder.
- **FR-32.** The application shall display a count of pending operations and failed operations (e.g. in a status indicator or badge).

### 7.9 Local content storage for offline reading

- **FR-33.** Message metadata (sender, recipients, subject, date, size, flags, preview text) shall be stored locally for all messages within the configured sync window and shall be readable offline.
- **FR-34.** Message bodies shall be stored locally once downloaded, and shall remain readable offline until explicitly purged or until the message leaves the configured retention window.
- **FR-35.** Attachments shall be stored locally once downloaded, and shall remain readable offline.
- **FR-36.** Messages whose body has not been downloaded shall be clearly indicated in the UI, and attempting to open them offline shall present an informative message rather than a blank or error state.
- **FR-37.** The message list (with previews, sender, subject, date, flags) shall always be usable offline for synced messages.

### 7.10 Seamless online/offline transition

- **FR-38.** The application shall not require explicit "go offline" or "go online" user actions. The transition shall be automatic and transparent.
- **FR-39.** When offline, all features that depend only on local data (reading synced messages, navigating folders, viewing attachments, composing drafts, performing triage actions) shall continue to function normally.
- **FR-40.** When connectivity returns, queue replay shall begin automatically without user prompting.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Any user action (flag, move, delete, compose) shall complete its local acceptance and UI update in well under one second, regardless of connectivity state or queue depth.
- **NFR-2. Durability.** No queued operation shall be lost due to application crash, unexpected termination, or system reboot. Operations must be durably persisted before the UI acknowledges the action to the user.
- **NFR-3. Scalability.** The operation queue shall handle at least 1,000 pending operations without degradation of UI responsiveness or replay performance.
- **NFR-4. Convergence.** After replay completes without permanent errors, the server state shall match the user's intended state as expressed by their sequence of actions.
- **NFR-5. Idempotency.** Replaying an operation that has already taken effect on the server (e.g. due to a communication failure after the server applied it but before confirmation was received) shall not produce duplicate or contradictory results.
- **NFR-6. Transparency.** The user shall always be able to determine (a) whether they are currently online or offline, (b) how many operations are pending, and (c) whether any operations have failed — via a persistent, non-intrusive status indicator.
- **NFR-7. Data integrity.** Local optimistic state shall never diverge permanently from server state. After a successful full sync cycle, local and server states shall be consistent.
- **NFR-8. Storage efficiency.** Local message content storage shall respect the user's configured retention settings (days to keep, days to download) and shall not grow unbounded.

| AC-3 | With no network connection, composing and sending a message places it in an outgoing queue; the message is delivered when connectivity returns. |

| AC-5 | A previously downloaded message body is readable offline, including inline images that were fetched at download time. |

| AC-8 | An operation that fails due to a transient error (e.g. timeout) is retried automatically and succeeds on a subsequent attempt without user action. |
| AC-9 | An operation that fails permanently (e.g. target message deleted on server) is marked as failed with an error message visible in the operations view. |
| AC-10 | Deleting a queued move operation reverts the message to its original folder in the local view. |
| AC-11 | Deleting a queued send operation recovers the message as a draft. |

| AC-14 | Queued operations survive application crash: force-terminating the application and restarting it shows the same operations still pending. |
| AC-15 | After an extended offline period (100+ queued operations), replay completes within a reasonable time by batching similar operations. |

## 10. Open Questions

- **OQ-1. Retry ceiling configuration.** The source application uses a fixed maximum retry count. Should this be user-configurable, or is a sensible fixed default sufficient for the desktop application?
- **OQ-2. Batch size tuning.** The source application batches similar operations in groups of up to 20. Should this batch size be configurable, or auto-tuned based on server responsiveness?
- **OQ-3. Conflict notification granularity.** When a non-destructive operation is silently abandoned (e.g. flagging a server-deleted message), should the user be informed at all? The source application silently drops these — is that appropriate for power users who want full transparency?
- **OQ-4. Operation ordering vs. user intent.** The priority system ensures content downloads happen before state changes. Are there edge cases where a user would want a different ordering (e.g. "send this message before downloading those 50 bodies")?
- **OQ-5. POP3 operation limitations.** For POP3 accounts, many operations (flag, mark read, keywords) have no server-side representation and execute locally only. Should the user be informed that these actions will not persist if they switch clients, or is this an acceptable silent limitation?
- **OQ-6. Storage retention vs. offline availability.** The configured retention window (days to keep) may purge messages a user expects to still have offline. Should there be a "pin for offline" mechanism that overrides retention for specific messages?
- **OQ-7. Queue replay and metered connections.** Should replay of large operations (attachment downloads, bulk syncs) be deferred on metered connections? This overlaps with feature 7.4 (network-aware sync) — the boundary between the two features needs clarification.
- **OQ-8. Undo-send window.** The source application allows cancelling a send while it is still queued. Should there be an explicit configurable undo-send grace period (distinct from feature 4.9's send delay), or is "cancellable while queued" sufficient?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Optimistic UI is non-negotiable.** The source application never makes the user wait for a server round-trip to see the effect of their action. This is the core UX contract: the local state *is* the user's truth; the server is a durable backup that will eventually converge. Any design that introduces "pending" spinners or "waiting for server" states for basic mail actions would be a regression.

- **N-2. Operations are first-class persistent entities.** Operations are not ephemeral in-memory tasks — they are durable records with their own lifecycle (created → executing → completed/failed). This ensures crash safety and allows the operations view to function as an audit log of what the application intends to do.

- **N-3. Priority ordering prevents starvation.** Without priorities, a large batch of low-value operations (e.g. hundreds of folder syncs) could starve time-sensitive actions (e.g. sending an urgent reply). The priority system ensures the user's direct actions are honored first.

- **N-4. Batch-then-fallback strategy.** The source application optimistically batches similar operations for efficiency, but falls back to individual execution if a batch is rejected. This is a pragmatic approach: most servers handle batches well, but edge cases (message UIDs that changed, partial failures) require individual retry. Designs should preserve this two-phase approach.

- **N-5. Silent abandonment for non-destructive conflicts.** When the user flags a message offline and that message is deleted on the server before replay, the flag operation is silently dropped. This is intentional: bothering the user about a flag on a message they can no longer see adds noise without value. Only conflicts with user-visible consequences (lost moves, failed sends) warrant notification.

- **N-6. Protocol-aware operation routing.** Not all protocols support all operations server-side. POP3 notably lacks server-side flags, keywords, and move semantics. The source application handles this by routing unsupported operations to local-only execution rather than failing them. This is transparent to the user — their flags and labels "work" on POP3 accounts, they just don't sync to the server.

- **N-7. Queue cleanup restores truth.** When a user deletes a pending operation, the application reverts the optimistic UI state. This ensures that "cancel this action" truly means undo — the message returns to its pre-action state in the UI. Without this, deleting an operation could leave the UI showing a state that neither the server nor any pending operation supports.
