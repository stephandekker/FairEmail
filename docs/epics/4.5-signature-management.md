# Epic: Signature Management

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **4.5** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Signature management. Per-identity signatures (HTML or plain), with placement (above/below quote, before/after reply text), variables (name, email, date), and disable-on-reply option."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email signatures are a fundamental part of professional and personal correspondence. Users expect to define one or more signatures — typically containing their name, title, contact details, or a legal disclaimer — and have those signatures automatically appended to outgoing messages.

In a multi-identity email client, different "From" personas often require different signatures: a work identity might carry a corporate disclaimer and job title, while a personal identity might carry only a name or nothing at all. The signature feature must therefore be *per-identity*, not per-account or application-wide.

Beyond simple insertion, users have strong opinions about *where* the signature appears relative to quoted text when replying or forwarding. Top-posters expect the signature between their new text and the quote; bottom-posters expect it at the very end. Some users want the signature only on new messages and not on replies at all, or only on the first reply in a thread to avoid stacking duplicate signatures. The feature must accommodate all of these workflows.

Finally, signatures may contain dynamic content — the sender's name, email address, or the current date — inserted via simple variable placeholders, allowing a single signature template to adapt to the context in which it is used.

## 2. Goals

- **G1.** Allow each identity to have its own independent signature, authored in either rich text (HTML) or plain text (raw HTML source editing).
- **G2.** Give the user fine-grained control over *when* a signature is inserted: independently for new messages, replies, and forwards.
- **G3.** Provide configurable *placement* of the signature relative to the message body and any quoted text.
- **G4.** Support a "reply-once" mode that adds the signature only to the first reply in a conversation thread, preventing signature stacking.
- **G5.** Support variable placeholders in signatures that are expanded at send time (or insertion time) with contextual values such as the sender's name, email address, and date.
- **G6.** Allow the user to override the signature decision on a per-message basis in the compose window.
- **G7.** Optionally prepend the Usenet signature separator (`-- `) before the signature, per RFC 3676.

## 3. Non-Goals

- **NG1.** This epic does **not** cover reply templates / saved snippets ("answers"), which are a separate feature with their own placeholder system.
- **NG2.** This epic does **not** define the rich text editing toolbar or its capabilities; those belong to the rich text editor epic (4.1). The signature editor re-uses whatever rich text editing surface the application provides.
- **NG3.** This epic does **not** prescribe how signatures are stored or synchronized — only that they are durable and per-identity.
- **NG4.** Signature management for shared / team mailboxes is out of scope.
- **NG5.** Automatic signature selection based on recipient, domain, or other heuristics is out of scope. The signature is always determined by the chosen identity.

## 4. Glossary

| Term | Definition |
|---|---|
| **Identity** | A "From" persona associated with an account. Each identity has its own display name, email address, reply-to address, and signature. A single account may have multiple identities. |
| **Signature** | A block of text (plain or HTML) associated with an identity, automatically inserted into outgoing messages. |
| **Signature placement** | The position where the signature is inserted relative to the user's new text and any quoted/forwarded material. |
| **Quoted text / reference** | The original message text included when replying to or forwarding a message, typically visually indented or prefixed. |
| **Signature separator** | The conventional `-- ` (two dashes followed by a space) line that precedes a signature per Usenet/RFC 3676 convention. Used by some clients to automatically strip signatures from quoted text. |
| **Reply-once mode** | A setting that causes the signature to be added only to the first outbound message in a conversation thread, suppressing it on subsequent replies. |
| **Variable placeholder** | A token embedded in a signature template (e.g. `$name$`, `$email$`, `$date$`) that is replaced with a contextual value when the signature is inserted into a message. |

## 5. Personas

- **P1. The professional user.** Has 2–3 identities (work, personal, freelance), each with a distinct HTML signature containing name, title, phone, and company logo. Expects the signature on every new message and reply, placed between their text and the quote.
- **P2. The minimalist.** Uses one identity with a short plain-text signature (just a name). Wants it on new messages only, not on replies or forwards, to keep threads clean.
- **P3. The mailing-list participant.** Posts to technical mailing lists where Usenet conventions are expected. Wants the `-- ` separator, signature at the very bottom, and reply-once mode so that the signature does not accumulate in long threads.
- **P4. The power user.** Manages many identities and frequently switches between them in the compose window. Expects the signature to update instantly when the identity changes, and wants to toggle it off for individual messages when it is not appropriate.
- **P5. The template user.** Uses variable placeholders in their signature so that one template works across contexts — e.g. including the current date in a legal disclaimer.

## 6. User Stories

### 6.1 Creating and editing a signature

> **US-1.** As any user, I want to create a signature for each of my identities through the identity settings, so that each "From" persona can have its own signature.

> **US-2.** As a professional user, I want to author my signature in a rich text editor with formatting (bold, italic, links, images, colors, fonts), so that my signature can match my corporate branding.

> **US-3.** As a power user, I want to switch to a raw HTML editing mode in the signature editor, so that I can write or paste arbitrary HTML that the visual editor might not support.

> **US-4.** As any user, I want a full-screen preview of my signature, so that I can see how it will look before saving.

> **US-5.** As any user, I want to be warned that switching between the visual editor and raw HTML mode may alter the formatting, so that I do not lose work unexpectedly.

> **US-6.** As any user, I want to insert images into my signature (e.g. a company logo), so that my signature can include visual branding.

> **US-7.** As any user, I want to validate my signature's HTML for syntax errors before saving, so that I can catch mistakes early.

### 6.2 Controlling when signatures are inserted

> **US-8.** As any user, I want to independently control whether a signature is added to new messages, replies, and forwarded messages, via three separate toggles in the send settings, so that I can tailor signature behavior to each action type.

> **US-9.** As a minimalist, I want to disable signatures on replies and forwards while keeping them on new messages, so that threads stay clean.

> **US-10.** As a mailing-list participant, I want a "reply-once" option that adds the signature only to the first reply in a thread, so that my signature does not stack up in long conversations.

> **US-11.** As any user, when reply-once mode is active and I have already sent a reply in a thread, I want the signature to be suppressed on subsequent replies in that same thread automatically.

### 6.3 Signature placement

> **US-12.** As any user, I want to choose where my signature is placed from among at least three options: (a) above my text, (b) below my text but before the quoted material, or (c) at the very bottom of the message after the quoted material.

> **US-13.** As a top-poster, I want the default placement to put my signature below my reply text but before the quote, so that the recipient sees my message and signature without scrolling past the quoted thread.

> **US-14.** As a bottom-poster who writes replies below the quoted text, I want the signature to appear after the quoted material at the very bottom, consistent with my reply position.

### 6.4 Signature separator (Usenet convention)

> **US-15.** As a mailing-list participant, I want an option to prepend the standard `-- ` separator line before my signature, so that RFC 3676-compliant clients can automatically identify and strip my signature from quoted text.

> **US-16.** As any user, when the Usenet separator is enabled, I want the signature to be placed at the very bottom of the message (after any quoted material), consistent with Usenet conventions.

### 6.5 Variable placeholders

> **US-17.** As a template user, I want to use placeholders in my signature for at least: the sender's full name, first name, last name, email address, current date (with configurable format), and day of the week, so that one signature template adapts to context.

> **US-18.** As any user, I want placeholders to be expanded when the signature is inserted into a message, so that the final message contains the resolved values.

### 6.6 Per-message override in the compose window

> **US-19.** As any user, I want a toggle in the compose window that lets me enable or disable the signature for the current message, overriding the default setting, so that I have final control.

> **US-20.** As any user, I want to preview the current signature content from within the compose window, so that I can verify what will be sent.

> **US-21.** As a power user, when I change the identity in the compose window, I want the signature to update to match the newly selected identity, so that the correct signature is always used.

### 6.7 Signature removal from quoted text

> **US-22.** As any user, I want an option to automatically strip recognized signatures from the quoted text when replying, so that quoted material does not accumulate stale signatures from previous messages in the thread.

### 6.8 Font consistency

> **US-23.** As any user, I want the signature to inherit the compose font setting, so that the signature visually matches the rest of my message.

## 7. Functional Requirements

### 7.1 Signature association

- **FR-1.** Each identity shall have exactly one signature, which may be empty (no signature) or a block of HTML content.
- **FR-2.** The signature shall be editable from the identity settings, via a dedicated signature editor screen.
- **FR-3.** A signature shall be fully independent per identity — changing one identity's signature shall not affect any other identity's signature.
- **FR-4.** The signature shall be included in settings export/import, so that it can be backed up and restored alongside other identity configuration.

### 7.2 Signature editor

- **FR-5.** The signature editor shall support a visual rich text editing mode with formatting capabilities (bold, italic, underline, links, images, colors, fonts, etc.), re-using the application's standard rich text editing surface.
- **FR-6.** The signature editor shall support a raw HTML editing mode, allowing the user to directly author or paste HTML source.
- **FR-7.** Switching between visual and raw HTML modes shall preserve the underlying HTML content, with a warning that visual rendering may not represent all HTML constructs faithfully.
- **FR-8.** The signature editor shall provide a full-screen preview of the rendered signature.
- **FR-9.** The signature editor shall support insertion of images, with appropriate handling of image references so that they remain valid when the signature is used in messages.
- **FR-10.** The signature editor shall provide an HTML syntax validation function that reports parsing errors.
- **FR-11.** When raw HTML mode is active, the editor shall display content in a monospaced font for readability.

### 7.3 Insertion triggers

- **FR-12.** The application shall provide three independent boolean settings controlling whether a signature is inserted for: (a) new messages, (b) replies and reply-all, (c) forwards.
- **FR-13.** Each of these three settings shall default to **enabled**.
- **FR-14.** For resend operations, signatures shall **not** be inserted by default.

### 7.4 Reply-once mode

- **FR-15.** The application shall provide a boolean "reply-once" setting.
- **FR-16.** When reply-once is enabled and the "signature on reply" setting is also enabled, the application shall add the signature to a reply **only if** no outbound message has previously been sent in the same conversation thread.
- **FR-17.** If at least one outbound message already exists in the thread, the signature shall be suppressed on subsequent replies.
- **FR-18.** Reply-once shall apply only to reply and reply-all actions, not to new messages or forwards.
- **FR-19.** The reply-once setting shall be disabled (greyed out) in the UI when the "signature on reply" setting is off, since it has no effect in that state.

### 7.5 Signature placement

- **FR-20.** The application shall provide a signature placement setting with three options:
  - **Above the text** — the signature is inserted at the top of the message body, before the user's new text.
  - **Below the text** (default) — the signature is inserted after the user's new text but before any quoted/forwarded material.
  - **At the bottom** — the signature is inserted at the very end of the message, after any quoted/forwarded material.
- **FR-21.** The default placement shall be "below the text".
- **FR-22.** When the user has enabled the "write reply below quote" option and is replying (not forwarding), the placement shall adapt so that the signature appears after the user's reply position, consistent with bottom-posting conventions.

### 7.6 Usenet signature separator

- **FR-23.** The application shall provide a boolean setting to prepend the Usenet signature separator (`-- ` — two dashes followed by a space, on its own line) before the signature.
- **FR-24.** This setting shall default to **disabled**.
- **FR-25.** When the Usenet separator is enabled, the signature placement shall automatically be set to "at the bottom", consistent with the Usenet convention that signatures follow all message content.

### 7.7 Variable placeholders

- **FR-26.** The signature system shall support the following variable placeholders, which are expanded when the signature is inserted into a message:

  | Placeholder | Expands to |
  |---|---|
  | `$name$` | The identity's full display name |
  | `$firstname$` | The first word of the identity's display name |
  | `$lastname$` | The last word of the identity's display name |
  | `$email$` | The identity's email address |
  | `$date$` | The current date, formatted in the system's long date format |
  | `$weekday$` | The current day of the week |

- **FR-27.** Date-related placeholders shall respect the user's locale for formatting.
- **FR-28.** Unrecognized placeholders shall be left as-is (not expanded and not stripped).

### 7.8 Per-message override

- **FR-29.** The compose window shall include a toggle (e.g. checkbox) allowing the user to enable or disable the signature for the current draft, overriding the default behavior.
- **FR-30.** The initial state of this toggle shall be determined by the applicable insertion trigger setting (FR-12) and, if applicable, the reply-once logic (FR-15–FR-18).
- **FR-31.** Changing the toggle shall immediately update the draft (adding or removing the signature).
- **FR-32.** The compose window shall provide a way to preview the content of the current identity's signature.

### 7.9 Identity change in compose

- **FR-33.** When the user changes the selected identity in the compose window, the signature in the draft shall be replaced with the newly selected identity's signature (or removed if the new identity has no signature).
- **FR-34.** The signature toggle state (FR-29) shall be preserved across identity changes — if the user has manually disabled the signature, switching identities shall not re-enable it.

### 7.10 Signature removal from quoted text

- **FR-35.** The application shall provide a boolean setting to automatically remove recognized signatures from quoted text when composing a reply.
- **FR-36.** This setting shall default to **disabled**.
- **FR-37.** The removal shall recognize signatures from at least the following sources: messages composed by this application, and common third-party email clients (e.g. Gmail, Outlook).
- **FR-38.** Removal shall be best-effort — unrecognized signature formats shall be left in the quoted text.

### 7.11 Font inheritance

- **FR-39.** The inserted signature shall inherit the user's chosen compose font, so that the signature and the message body are visually consistent.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Inserting, removing, or repositioning a signature in the compose window shall be near-instantaneous and shall not introduce a visible delay.
- **NFR-2. Fidelity.** The signature HTML authored by the user shall be sent as-is in the outgoing message, even if the visual editor cannot render all of it faithfully during editing. The editor is a best-effort preview; the wire format is the source of truth.
- **NFR-3. Portability.** Signatures shall be included in settings export/import, preserving all content including embedded image references.
- **NFR-4. Accessibility.** All signature-related settings, the signature editor, and the per-message toggle shall be reachable via keyboard, with appropriate labels for screen readers.
- **NFR-5. Privacy.** Variable expansion (e.g. `$date$`) shall occur locally at composition time. No signature content shall be transmitted to any external service for processing.

| AC-3 | With "signature on new" enabled and "signature on reply" disabled, a new message includes the signature and a reply does not. |

## 10. Open Questions

- **OQ-1. Variable placeholders in signatures vs. answers.** The source application clearly implements variable expansion (`$name$`, `$email$`, `$date$`, etc.) for reply templates ("answers"). The code path for signature insertion does not appear to invoke the same variable expansion logic. It is unclear whether this is intentional (signatures are static) or an omission. The high-level feature description explicitly calls out "variables (name, email, date)" for signatures. This should be clarified: should the desktop application support variable expansion in signatures, or only in reply templates? This epic assumes signatures should support variables, consistent with the high-level feature description.

- **OQ-2. Date placeholder format customization.** The `$date$` placeholder expands using the system's long date format. Should users be able to specify a custom date format (e.g. `$date$yyyy-MM-dd$`) within the placeholder syntax, as the answer/template system supports? Or is the system default sufficient?

- **OQ-3. Interaction between "write below quote" and signature placement.** When the user has enabled "write reply below quoted text" and the signature placement is set to "below the text", the placement logic has special-case behavior that depends on whether the message is a reply or a forward, and whether it originated as a forwarded message. The exact interaction is subtle and may surprise users. Consider whether the UI should surface a clearer explanation or a dedicated placement option for bottom-posters.

- **OQ-4. Recognized signature formats for removal.** The set of third-party signature formats that the removal logic recognizes is currently limited (this application's own format, Gmail, Outlook). Should the desktop application expand this set, or is best-effort recognition of the most common formats sufficient?

- **OQ-5. Image handling in signatures.** The source application uses persistent URI permissions to keep signature images accessible. The desktop application will need a different mechanism for ensuring that images embedded in signatures remain valid across sessions and across export/import. The specific approach is a design decision, but the requirement (images must remain valid) is captured in FR-9 and NFR-3.

- **OQ-6. Per-identity vs. global placement setting.** The signature placement setting is currently global (applies to all identities). Some users might want different placement for different identities (e.g. top-posting for work, bottom-posting for personal). Should per-identity placement be considered, or is the global setting sufficient?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Per-identity, not per-account.** The signature is a property of the identity, not the account. This is the correct granularity because a single account may have multiple identities (aliases), and each alias may need a different signature (e.g. a user who sends as both "alice@company.com" with a corporate signature and "alice@company.com (Personal)" with a minimal one). Making signatures per-account would force all aliases to share one signature.

- **N-2. Three-position placement model.** The three placement options ("above text", "below text", "at the bottom") cover the three dominant email culture conventions: top-posting with signature between reply and quote, top-posting with signature at the very end, and a less common "signature first" style. The "below the text" default reflects the most common convention in general-purpose email clients.

- **N-3. Reply-once prevents signature stacking.** In long email threads, every reply adding a signature creates an ever-growing tower of duplicate signatures in the quoted text. Reply-once mode solves this by adding the signature only on the first reply. This is particularly valued in mailing-list and support-ticket contexts. The detection of "already replied in this thread" is based on whether any outbound message exists in the conversation thread.

- **N-4. Usenet separator forces bottom placement.** When the user enables the `-- ` separator, the application automatically moves the signature to the bottom of the message. This is not an arbitrary coupling — RFC 3676 defines the signature separator as appearing after all message content, and placing a separated signature in the middle of a message (between reply text and quote) would violate the convention and confuse clients that attempt to strip signatures from quoted text. The forced placement preserves the semantic contract that the separator implies.

- **N-5. Global send settings, not per-identity.** The insertion trigger settings (new/reply/forward), placement, reply-once, and Usenet separator are all global settings, not per-identity. This keeps the settings surface simple — most users want consistent behavior across all their identities. The per-identity variation comes from the *content* of the signature, not from the *rules* governing when and where it appears.

- **N-6. Compose-time override is a safety valve.** The per-message toggle in the compose window exists because no set of defaults can cover every situation. A user who normally wants a signature on replies may occasionally reply to an internal thread where a signature is inappropriate. Rather than requiring a trip to the settings, the toggle provides a one-click override. The toggle's initial state is always derived from the global settings and reply-once logic, so it "does the right thing" by default and only needs manual intervention in exceptional cases.
