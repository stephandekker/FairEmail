# Epic: Link Confirmation & Sanitization

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.12** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Confirm before opening links; strip known tracking parameters; warn on punycode/spoofed domains; 'open in private browser' option."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[SHOULD]` — important, planned for v1 but acceptable to slip to v1.x if pressed.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email messages routinely contain hyperlinks that may lead to phishing sites, tracking endpoints, or deceptive domains. Users who click links without inspection expose themselves to credential theft, unwanted profiling, and malware. At the same time, legitimate links are a core part of email communication and must remain easy to follow.

The Link Confirmation & Sanitization feature exists to interpose a safety layer between the user and the act of opening a link. It gives the user a moment to inspect, modify, and understand a link before committing to open it — and it automatically strips or neutralizes known tracking and redirection artifacts so that the URL the user actually visits is as clean and direct as possible.

The feature also provides progressive trust: users who have verified a domain once can suppress future confirmation for that domain, while retaining the ability to force confirmation at any time.

## 2. Goals

- **G1.** Prevent accidental or uninformed navigation to malicious, spoofed, or tracking-laden URLs by showing a confirmation dialog before opening any link in a message.
- **G2.** Automatically detect and strip known tracking query parameters (UTM parameters, ad-network click IDs, mailing-list subscriber tokens, etc.) so that the URL the user actually visits carries no unnecessary tracking payload.
- **G3.** Automatically unwrap redirect/wrapper URLs (safe-links services, AMP wrappers, email-marketing click-trackers, base64-encoded redirect payloads) to reveal the true destination.
- **G4.** Warn the user visually when a link's domain uses mixed Unicode scripts (IDN homograph / punycode spoofing), so that deceptive domains are immediately obvious.
- **G5.** Warn the user when a link's domain appears on a known tracker/malware blocklist.
- **G6.** Allow the user to choose how a link is opened: in an embedded browser tab, in an external browser, or via an application chooser — and remember that preference.
- **G7.** Allow the user to suppress confirmation on a per-domain basis ("do not ask again for this host") or globally, while always retaining the ability to force confirmation via a long-press or a global setting.
- **G8.** Support an advanced tracking-parameter removal engine (community-maintained filter lists) as an opt-in alternative to the built-in parameter stripping.

## 3. Non-Goals

- **NG1.** This epic does **not** define how message bodies are rendered or how links are detected within HTML — only what happens when a detected link is activated.
- **NG2.** This epic does **not** cover blocking or warning about tracking *images* (web beacons); that is a separate privacy feature.
- **NG3.** This epic does **not** define URL rewriting at send time (e.g. stripping tracking from links the user composes).
- **NG4.** Real-time safe-browsing lookups against a cloud service are **not** in scope; link safety assessment here relies on locally available data (blocklists, script analysis, structural patterns).
- **NG5.** This epic does **not** prescribe specific UI toolkit choices, animations, or layout — only the information presented and the actions available.

## 4. Glossary

| Term | Definition |
|---|---|
| **Confirmation dialog** | A modal dialog shown before a link is opened, displaying the resolved URL and offering inspection and action controls. |
| **Sanitized URL** | The result of stripping known tracking parameters and unwrapping redirect wrappers from an original URL. |
| **Tracking parameter** | A query-string key (e.g. `utm_source`, `fbclid`, `gclid`) added to a URL solely for analytics/attribution purposes, with no effect on the destination content. |
| **Redirect wrapper** | A URL that does not lead to the intended content directly but instead redirects (via HTTP redirect, JavaScript, or an embedded parameter) to the real destination. Examples: safe-links services, AMP cache URLs, email-marketing click-trackers. |
| **IDN homograph attack** | A spoofing technique where a domain is registered using characters from multiple Unicode scripts that visually resemble Latin characters, making a malicious domain look like a legitimate one. |
| **Mixed-script domain** | A domain whose characters span more than one Unicode script (excluding well-known script combinations such as Han + Hiragana + Katakana in Japanese). |
| **Punycode** | The ASCII-compatible encoding of an internationalized domain name (IDN). Displaying punycode makes non-Latin characters explicit. |
| **Embedded browser tab** | A browser view that appears integrated within the application window (as opposed to launching a separate browser process). |
| **External browser** | A standalone browser application launched independently of the email client. |
| **Tracker blocklist** | A curated, periodically updated list of domains known to serve tracking, advertising, or malicious content, used to warn the user about link destinations. |
| **Filter list (advanced)** | A community-maintained rule set (e.g. from a content-filtering project) that specifies which URL parameters to strip, supporting more aggressive or frequently updated tracking removal than the built-in list. |
| **Per-domain trust** | A user preference that suppresses the confirmation dialog for a specific domain after the user has explicitly opted out of future prompts for that domain. |
| **DNS-based blocklist (DBL)** | A blocklist that can be queried to determine whether a domain is associated with spam, phishing, or malware. |

## 5. Personas

- **P1. The cautious user.** Wants to see every link before opening it; values the confirmation dialog and never disables it. May not be technically sophisticated but appreciates clear warnings about suspicious domains.
- **P2. The privacy-conscious user.** Primarily interested in stripping tracking parameters and unwrapping redirects; wants clean URLs with no analytics payload reaching their browser.
- **P3. The power user.** Clicks many links daily; wants confirmation suppressed for trusted domains but retains the ability to force-inspect any link via long-press. Configures an advanced filter list for maximum tracking removal.
- **P4. The targeted user.** Receives phishing attempts; relies on IDN/punycode warnings and blocklist checks to catch spoofed domains before clicking through.
- **P5. The casual user.** Finds confirmation dialogs annoying; disables them globally but still benefits from automatic sanitization when confirmation is bypassed for a remembered domain.

## 6. User Stories

### 6.1 Confirmation before opening

> **US-1.** As any user, when I tap/click a link in a message body, I want a confirmation dialog to appear showing me the full URL before anything is opened, so that I can inspect the destination and decide whether to proceed.

> **US-2.** As any user, I want the confirmation dialog to clearly distinguish between HTTP and HTTPS links (with a visual warning for non-secure links), so that I can assess the security of the connection.

> **US-3.** As any user, I want to be able to edit the URL directly in the confirmation dialog before opening it, so that I can manually remove suspicious parameters or correct the scheme.

> **US-4.** As any user, I want a toggle in the confirmation dialog to switch between HTTP and HTTPS for the link, so that I can upgrade (or downgrade) the connection security before opening.

### 6.2 Tracking parameter removal

> **US-5.** As a privacy-conscious user, I want the confirmation dialog to offer a "Remove tracking parameters" option that strips known analytics/attribution query parameters from the URL, so that I can visit the destination without being tracked.

> **US-6.** As a privacy-conscious user, when the sanitized URL differs from the original, I want the confirmation dialog to show both versions (or clearly indicate that parameters were removed), so that I can verify nothing essential was stripped.

> **US-7.** As a privacy-conscious user, I want a global setting that automatically applies tracking-parameter removal to all links (even when confirmation is suppressed for a domain), so that I get consistent tracking removal without per-link effort.

> **US-8.** As a power user, I want to opt into an advanced, community-maintained filter list for tracking removal that covers more parameters and is updated more frequently than the built-in list, so that I get the most thorough cleaning available.

### 6.3 Redirect unwrapping

> **US-9.** As any user, when a link is wrapped in a redirect service (safe-links, marketing click-tracker, AMP cache, URL-defense wrapper, or similar), I want the confirmation dialog to automatically extract and display the true destination URL, so that I can see where I will actually end up.

> **US-10.** As any user, I want redirect unwrapping to handle multiple known wrapper formats — including those that encode the real URL as a query parameter, as a base64-encoded value, or as a path segment — so that the feature works across the variety of wrappers encountered in real email.

### 6.4 Domain spoofing detection

> **US-11.** As any user, when a link's domain contains characters from multiple Unicode scripts (indicating a possible IDN homograph attack), I want the confirmation dialog to display a prominent warning and show the punycode (ASCII) form of the domain, so that I am not deceived by look-alike characters.

> **US-12.** As any user, when the visible link text (the anchor text in the message) points to a different domain than the actual hyperlink URL, I want the confirmation dialog to highlight this discrepancy, so that I notice when the displayed text is misleading.

> **US-13.** As any user, when a link's domain appears on a known tracker/malware blocklist, I want a visible warning in the confirmation dialog, so that I can reconsider opening the link.

### 6.5 Choosing how to open

> **US-14.** As any user, I want the confirmation dialog to let me choose which application or browser to use for opening the link (embedded browser tab, external browser, or a specific installed application), so that I have control over my browsing context.

> **US-15.** As any user, I want the application to remember my preferred browser/application choice across sessions, so that I do not have to re-select it every time.

> **US-16.** As any user, I want a "Browse" / "Open with…" action in the confirmation dialog that hands the link to the system's application chooser, so that I can open links in apps other than browsers when appropriate.

### 6.6 Per-domain and global trust

> **US-17.** As a power user, I want a checkbox in the confirmation dialog labeled with the current domain that suppresses future confirmation for that specific domain, so that I am not repeatedly prompted for domains I trust.

> **US-18.** As any user, I want a global setting to disable link confirmation entirely, so that links open immediately without any dialog.

> **US-19.** As any user, even when confirmation is globally disabled or suppressed for a domain, I want to be able to force the confirmation dialog for a specific link by long-pressing it, so that I always have access to the inspection and sanitization tools.

> **US-20.** As any user, I want a "Reset questions" action in the application settings that clears all per-domain trust decisions, so that I can return to a fully-prompted state after a security concern.

> **US-21.** As any user, I want a global "never ask again" option in the confirmation dialog that disables confirmation entirely in one step, so that I have a quick way to opt out if I find the prompts unnecessary.

### 6.7 Additional inspection tools

> **US-22.** As a cautious user, I want a "Check owner" or "Lookup" action in the confirmation dialog that resolves the link's host to an IP address and displays the registered owner/organization, so that I can verify who controls the domain.

> **US-23.** As a cautious user, I want a "WHOIS" action in the confirmation dialog that retrieves and displays WHOIS registration data for the link's root domain, so that I can investigate newly registered or suspicious domains.

> **US-24.** As any user, I want a "Copy" action in the confirmation dialog that copies the (possibly sanitized) URL to the clipboard, so that I can paste it elsewhere without opening it.

> **US-25.** As any user, I want a "Share" action in the confirmation dialog that sends the URL to another application via the system share mechanism, so that I can forward the link without opening it in a browser.

### 6.8 Tracker blocklist integration

> **US-26.** As a privacy-conscious user, I want the application to maintain a locally cached copy of a well-known tracker protection list, updated periodically, so that link warnings are based on current data.

> **US-27.** As a privacy-conscious user, I want a setting to enable automatic periodic updates of the tracker blocklist, so that my protection stays current without manual intervention.

> **US-28.** As any user, when a link's domain is found on the tracker blocklist, I want the confirmation dialog to display the tracker category (advertising, analytics, social, etc.), so that I understand the nature of the tracking.

### 6.9 Scheme guessing and edge cases

> **US-29.** As any user, when a link in a message has no explicit scheme (e.g. bare `example.com`), I want the application to infer the most appropriate scheme (defaulting to HTTPS for web URLs, `mailto:` for email addresses, `tel:` for phone numbers), so that the link still works correctly.

> **US-30.** As any user, when a link uses the `mailto:` scheme, I want the confirmation dialog to display the recipient address and offer a "Search" action to find messages from that address, in addition to the standard open action.

## 7. Functional Requirements

### 7.1 Confirmation dialog display

- **FR-1.** When a user activates (clicks/taps) a hyperlink in a message body, the application shall display a confirmation dialog before navigating to the URL, unless confirmation has been suppressed for that domain or disabled globally.
- **FR-2.** The confirmation dialog shall display the full resolved URL in an editable text field.
- **FR-3.** The confirmation dialog shall visually distinguish secure (HTTPS) from insecure (HTTP) links via color, icon, or text styling.
- **FR-4.** The confirmation dialog shall provide at minimum three actions: confirm/open, cancel, and "open with" (system chooser).
- **FR-5.** Long-pressing a link shall always display the confirmation dialog regardless of per-domain or global suppression settings.

### 7.2 Tracking parameter stripping

- **FR-6.** The application shall maintain a built-in list of known tracking query parameters (UTM parameters, ad-network click IDs, mailing-platform subscriber tokens, and similar) that are stripped during sanitization.
- **FR-7.** When the sanitized URL differs from the original, the confirmation dialog shall offer a "Remove tracking parameters" checkbox (or equivalent control) that toggles between the original and sanitized URLs.
- **FR-8.** The built-in tracking parameter list shall include at minimum: UTM parameters (`utm_*`), Google (`gclid`, `gclsrc`, `dclid`, `gbraid`, `wbraid`), Facebook (`fbclid`), Microsoft (`msclkid`), MailChimp (`mc_cid`, `mc_eid`), HubSpot (`_hsenc`, `__hssc`, `__hstc`, `__hsfp`, `hsCtaTracking`), and common others documented in well-known tracking-parameter registries.
- **FR-9.** For specific high-traffic domains (e.g. a major social network), the application shall apply domain-specific query-parameter whitelisting rules — stripping all parameters except those known to be functional for content retrieval.
- **FR-10.** The application shall optionally support an advanced, externally maintained filter-list engine (opt-in via settings) that provides more comprehensive parameter stripping rules, downloaded and cached locally.
- **FR-11.** When the advanced filter list is enabled and the auto-update setting is active, the application shall periodically refresh the filter list from its upstream source.

### 7.3 Redirect unwrapping

- **FR-12.** The sanitization engine shall detect and unwrap URLs from the following categories of redirect wrappers:
  - Safe-links services (e.g. Outlook Safe Links, Trend Micro)
  - AMP cache URLs
  - Marketing click-tracker URLs that embed the real destination as a query parameter or base64-encoded value
  - URL-defense/rewriting services
  - Generic redirects using `redirect`, `redirectUrl`, or `url` query parameters
  - AWS click-tracking URLs
  - Brave-style debounce rules (from a locally cached rule set)
  - Sophos-style base64-encoded wrappers
  - Base64-encoded last-path-segment patterns
- **FR-13.** When a redirect is unwrapped, the sanitized result shall be the innermost destination URL, and it shall be a valid hyperlink. If the unwrapped result is not a valid hyperlink, the original URL shall be preserved.
- **FR-14.** Redirect unwrapping and tracking-parameter stripping shall be applied together: first unwrap, then strip parameters from the unwrapped result.

### 7.4 Domain spoofing warnings

- **FR-15.** The confirmation dialog shall detect mixed-script domains (IDN homograph candidates) by analyzing the Unicode scripts present in the domain name. If multiple scripts are detected (excluding well-known benign combinations such as Han/Hiragana/Katakana in Japanese contexts), a "suspicious" warning shall be displayed prominently.
- **FR-16.** When a domain contains non-ASCII characters, the confirmation dialog shall display the punycode (ASCII-encoded) form of the domain so that the user can see the underlying encoding.
- **FR-17.** When the anchor text of a link (as displayed in the message) contains a URL pointing to a different domain than the actual hyperlink destination, the confirmation dialog shall display a visual indicator (e.g. a separate "title" line and/or an "open title URL" control) so that the user is aware of the mismatch.

### 7.5 Blocklist warnings

- **FR-18.** When the "check links against blocklist" setting is enabled, the application shall query a locally maintained DNS-based blocklist to determine if the link's domain is flagged as malicious/spam.
- **FR-19.** If the domain is flagged, the confirmation dialog shall display a "suspicious" warning (the same or similar visual treatment as the mixed-script warning).
- **FR-20.** The application shall query a locally cached tracker-protection list (sourced from a well-known open-source tracking-protection project) to determine if the link's domain is a known tracker.
- **FR-21.** If the domain is found on the tracker-protection list, the confirmation dialog shall display a tracker warning along with the category classification(s) of the tracker.
- **FR-22.** The tracker-protection list shall be downloadable on demand and optionally auto-updated on a periodic schedule.

### 7.6 Browser/application selection

- **FR-23.** The confirmation dialog shall enumerate installed applications capable of handling the link's URL scheme and present them as a selectable list.
- **FR-24.** For each browser that supports embedded tab integration, the list shall offer two entries: one for embedded tab mode and one for standalone/external mode.
- **FR-25.** The user's selection shall be persisted and re-applied as the default for future link opens.
- **FR-26.** A "system chooser" / "select app" entry shall always be available, delegating application selection to the operating system.

### 7.7 Per-domain and global confirmation settings

- **FR-27.** The confirmation dialog shall offer a "Do not ask again for [domain]" checkbox. When checked and the user confirms, future activations of links to that domain shall bypass the confirmation dialog.
- **FR-28.** When confirmation is suppressed for a domain, the user's sanitization preference for that domain (sanitize or not) shall also be remembered and applied automatically.
- **FR-29.** The confirmation dialog shall offer a "Never ask again" checkbox that, when checked, disables confirmation globally (equivalent to turning off the global setting).
- **FR-30.** A global "Confirm opening links" setting (default: enabled) shall control whether the confirmation dialog is shown. When disabled, links open immediately using the user's remembered application preference.
- **FR-31.** A "Reset questions" action in settings shall clear all per-domain confirmation suppressions, restoring the fully-prompted state.
- **FR-32.** The "Never ask again" and "Do not ask again for [domain]" checkboxes shall not be shown when confirmation was forced (e.g. via long-press or when sanitization altered the URL), ensuring the user cannot accidentally bypass confirmation in security-relevant contexts.

### 7.8 Inspection tools

- **FR-33.** The confirmation dialog shall offer a "Check owner" action that resolves the link's hostname to an IP address and looks up the organization, city, region, and country of the IP, displaying the results inline.
- **FR-34.** The confirmation dialog shall offer a "WHOIS" action that retrieves WHOIS data for the link's root domain and displays it in a secondary view.
- **FR-35.** The confirmation dialog shall offer a "Copy" action that copies the currently displayed URL (including any sanitization applied) to the system clipboard.
- **FR-36.** The confirmation dialog shall offer a "Share" action that sends the currently displayed URL to another application via the system's share/send mechanism.

### 7.9 Scheme inference

- **FR-37.** When a link has no explicit scheme, the application shall infer the scheme: HTTPS for patterns matching a web URL or IP address, `mailto:` for email addresses, `tel:` for phone numbers.
- **FR-38.** The confirmation dialog shall handle `mailto:` links by displaying the recipient address and offering a "Search" action to find messages involving that address, in addition to the standard compose action.

### 7.10 Settings surface

- **FR-39.** All link-related privacy settings shall be accessible from a dedicated section of the application's privacy settings: confirm links (on/off), remove tracking parameters by default (on/off), advanced filter list (on/off + update), check links against blocklist (on/off), tracker-protection list (on/off + update).
- **FR-40.** A link from within the confirmation dialog shall navigate the user directly to the relevant settings section, so that settings are discoverable in context.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** The confirmation dialog shall appear within 300 ms of the user activating a link. Sanitization (parameter stripping + redirect unwrapping) shall complete within the same window for the vast majority of URLs.
- **NFR-2. Offline operation.** All sanitization logic (parameter stripping, redirect unwrapping, mixed-script detection) shall operate entirely locally with no network access. Owner lookup and WHOIS are explicitly network-dependent and are user-initiated.
- **NFR-3. Blocklist freshness.** When auto-update is enabled, blocklists and filter lists shall be refreshed at least weekly. The timestamp of the last update shall be visible to the user.
- **NFR-4. No data leakage.** The confirmation and sanitization process itself shall not make any network requests unless the user explicitly invokes an inspection action (owner lookup, WHOIS). Displaying the dialog must not fetch resources from the link's domain.
- **NFR-5. Accessibility.** All controls in the confirmation dialog shall be keyboard-navigable and labeled for screen readers. Warning text shall use sufficient contrast and not rely solely on color.
- **NFR-6. Extensibility of parameter lists.** The built-in tracking-parameter list shall be structured so that adding new parameters requires only data changes, not logic changes.
- **NFR-7. Graceful degradation.** If sanitization encounters an unparseable URL (opaque URIs, unusual schemes), it shall fall back to displaying the original URL unchanged rather than failing or showing an error.
- **NFR-8. Scale.** The built-in tracking-parameter list, public-suffix list, and debounce rule set shall each support thousands of entries without perceptible delay in the confirmation dialog.

## 10. Open Questions

- **OQ-1. Scope of "always confirm" triggers.** The source application has an `always_confirm` flag passed in certain contexts (e.g. links in specific UI areas). The complete set of contexts where confirmation is forced regardless of user preference is not fully enumerated — the triggering conditions should be documented.
- **OQ-2. Advanced filter list update frequency.** The optimal auto-update interval for the advanced filter list is not specified. Weekly may be too infrequent for rapidly evolving tracker landscapes; daily may consume unnecessary resources.
- **OQ-3. Interaction between sanitization and per-domain bypass.** When a user has suppressed confirmation for a domain but the link would benefit from sanitization (e.g. new tracking parameters added after the suppression was set), should the application silently sanitize, or should it re-prompt? The source application silently applies the user's saved sanitization preference without re-prompting; this should be confirmed as the desired behavior.
- **OQ-4. WHOIS availability.** WHOIS data is increasingly restricted due to GDPR redactions. The value of the WHOIS action may be limited for many domains. Whether to retain it, replace it with RDAP, or remove it is a design decision.
- **OQ-5. DNS blocklist selection.** The source application uses a DNS-based blocklist to check link domains. The specific list(s) used and whether the user should be able to select or disable individual lists for link checking (as they can for email sender checking) is not fully resolved.
- **OQ-6. Embedded browser privacy.** The source application offers an embedded browser tab mode. Whether this mode should include privacy enhancements (e.g. no cookies from the email client session, no history retention — effectively a "private browsing" mode) or whether that is left to the browser's own settings is a design decision. The high-level feature description mentions "open in private browser" but the source application delegates this to the external browser's capabilities rather than enforcing it.
- **OQ-7. Mixed-script detection tuning.** The current heuristic exempts Han/Hiragana/Katakana combinations from the mixed-script warning. Whether additional script-combination exemptions are needed (e.g. Latin + common diacritical scripts) should be validated with internationalization testing.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Confirmation is on by default.** The global "confirm links" setting defaults to enabled. This is a deliberate safe-by-default posture: users must actively choose to reduce their protection. This is the correct default for an application whose user base includes non-technical users who are prime phishing targets.
- **N-2. Per-domain, not per-link.** Trust is granted at the domain level (specifically, the full hostname for HTTPS links or the full email address for mailto links). This granularity balances convenience (not re-prompting for every page on a trusted site) with safety (a subdomain of a trusted domain does not inherit trust).
- **N-3. Sanitization is optional but visible.** The "Remove tracking parameters" checkbox in the confirmation dialog defaults to unchecked unless the user has enabled the "sanitize links by default" global setting. This avoids surprising users by silently modifying URLs, while making the option highly discoverable.
- **N-4. Redirect unwrapping is always attempted.** Unlike tracking-parameter removal (which is opt-in), redirect unwrapping is always performed to populate the sanitized URL candidate. The user sees the unwrapped result and can choose whether to use it. This asymmetry reflects the different risk profiles: an unwrapped URL is strictly more transparent (the user sees the real destination), whereas parameter stripping modifies what the destination server receives.
- **N-5. Multiple warning signals stack.** A link can simultaneously trigger a mixed-script warning, a blocklist warning, and a title-mismatch indicator. These are not mutually exclusive; all applicable warnings are shown together. This avoids a false sense of security from showing only the "most important" warning.
- **N-6. Browser choice is persistent, not per-domain.** The user's preferred browser/tab mode applies to all links, not per-domain. This keeps the mental model simple and avoids a combinatorial explosion of per-domain preferences.
- **N-7. The "private browser" concept is delegated.** The source application does not itself implement private/incognito browsing. It offers embedded tabs and external browser launch — whether those operate in a private mode is a function of the chosen browser's configuration. The "open in private browser" aspiration from the high-level feature list is realized indirectly: by offering the application chooser, users can select a browser configured for privacy (or a dedicated private-browsing shortcut if their OS provides one).
