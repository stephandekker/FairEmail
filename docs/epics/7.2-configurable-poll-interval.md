# Epic: Configurable Poll Interval

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **7.2** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Configurable poll interval (per account and per folder), independent of IDLE."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email clients retrieve new messages from the server through two complementary mechanisms:

1. **Push (IMAP IDLE):** The server notifies the client in real time when new mail arrives. This is efficient but requires a persistent connection per folder, and most servers limit the number of simultaneous IDLE connections (typically to a handful of folders).
2. **Polling:** The client periodically connects to the server, compares its local state with the remote state, and downloads any new or changed messages. This works for every folder and every protocol (IMAP and POP3 alike) but is more resource-intensive because each poll cycle requires a full comparison.

In practice, a user's Inbox may use push for immediacy, while secondary folders (Sent, Drafts, Archive, user-created folders) use polling on a slower cadence because real-time awareness is less important there. Different accounts may also warrant different frequencies: a busy work account might poll every five minutes, while a seldom-used personal account might poll once a day.

The Configurable Poll Interval feature gives the user fine-grained control over how often each account and each folder is checked for new messages, independent of whether push is also active. It also provides a global override for situations where the user wants a single polling cadence across all accounts (e.g. to conserve battery on a laptop running on battery power), and an auto-tuning mechanism that adjusts the connection keep-alive cadence based on observed server behavior.

## 2. Goals

- **G1.** Allow the user to set a polling / keep-alive interval independently for each account, with a sensible default that balances responsiveness and resource usage.
- **G2.** Allow the user to control, per folder, whether that folder uses push or polling, and to apply a multiplier to the account's base interval for folders that need less frequent checking.
- **G3.** Provide a global polling interval that can override per-account settings when the user wants uniform behavior or wants to force periodic polling across all accounts.
- **G4.** Make the interaction between push (IDLE) and polling transparent: the user should understand when their account is using push, when it is polling, and what the interval controls in each case.
- **G5.** Warn the user when a chosen interval is likely to cause excessive resource consumption.
- **G6.** Support automatic tuning of the keep-alive interval so that the application adapts to servers that prematurely close idle connections, without requiring manual intervention.
- **G7.** Respect network conditions: allow the user to enable or disable periodic polling based on whether the network connection is metered or unmetered.

## 3. Non-Goals

- **NG1.** This epic does **not** define the push (IMAP IDLE) mechanism itself; that is covered by epic 7.1. This epic addresses only the polling cadence and the keep-alive interval that coexists with push.
- **NG2.** This epic does **not** define schedule-based sync windows (e.g. "sync only during work hours"); that is a separate concern (feature 7.3). However, accounts may be exempt from schedule restrictions, and that exemption interacts with polling.
- **NG3.** This epic does **not** prescribe power-management integration beyond user-visible warnings and network-awareness toggles. OS-level battery optimization and power profiles are external concerns.
- **NG4.** This epic does **not** cover connection timeouts, TCP keep-alive, DNS settings, or TLS tuning — those are connection-tuning concerns (feature 7.8).
- **NG5.** This epic does **not** define the synchronization depth (days to sync, days to keep) — that is a per-folder sync setting (feature 2.3).

## 4. Glossary

| Term | Definition |
|---|---|
| **Poll interval** | The number of minutes between consecutive checks of a mailbox for new or changed messages. A poll cycle involves connecting (or reusing a connection), comparing local and remote state, and downloading any differences. |
| **Keep-alive interval** | For accounts using push (IMAP IDLE), the number of minutes between periodic "still here" signals sent to the server to prevent it from closing the idle connection. Functionally, this is the same setting as the poll interval — the meaning changes depending on whether push is active. |
| **Push (IMAP IDLE)** | A server-driven notification mechanism where the server informs the client of changes in real time over a persistent connection. |
| **Polling** | A client-driven mechanism where the client periodically initiates a check for changes. |
| **Folder poll mode** | A per-folder setting that determines whether the folder uses push (when available) or falls back to periodic polling. |
| **Poll factor** | A per-folder multiplier applied to the account's base poll interval. A factor of 2 means the folder is polled half as often as the account's base cadence. |
| **Global poll interval** | An application-wide setting that, when enabled, forces periodic polling at a fixed cadence across all accounts, regardless of per-account settings. |
| **Metered network** | A network connection with usage limits or costs (e.g. mobile data, tethered connection). |
| **Unmetered network** | A network connection without usage limits (e.g. home Wi-Fi, wired Ethernet). |
| **On-demand account** | An account configured to synchronize only when the user explicitly requests it; automatic polling is disabled. |
| **Auto-tune** | An automatic mechanism that adjusts the keep-alive interval based on observed server behavior (connection drops, premature IDLE termination). |

## 5. Personas

- **P1. The "just works" user.** Has one or two accounts, accepts defaults, and expects new mail to appear promptly without configuring anything. Relies on the default 15-minute keep-alive interval and push for the Inbox.
- **P2. The battery-conscious laptop user.** Uses the application on a laptop that frequently runs on battery. Wants to reduce polling frequency globally or disable polling on non-essential accounts to extend battery life.
- **P3. The multi-account power user.** Has 5+ accounts with different urgency levels. Wants their primary work account polling every 5 minutes and their archive/newsletter accounts polling once a day.
- **P4. The POP3 user.** Uses a POP3 account where push is not available. Polling is the only mechanism for checking new mail; the interval setting directly controls how quickly new messages are discovered.
- **P5. The server-constrained user.** Uses a mail provider that limits simultaneous IDLE connections to 2–3 folders. Needs secondary folders to poll rather than use push, and wants control over how often they poll.

## 6. User Stories

### 6.1 Account-level interval

> **US-1.** As any user, when I configure an IMAP account, I want to set a keep-alive / poll interval (in minutes), so that I can control how frequently the application checks or refreshes the connection for that account.

> **US-2.** As any user, when I configure a POP3 account, I want to set a poll interval (in minutes), so that I can control how often the application checks for new messages on that account.

> **US-3.** As any user, I want the default interval to be a reasonable value (15 minutes) so that the application works well out of the box without me having to configure anything.

> **US-4.** As a power user, I want to choose from a range of intervals — from as frequent as every minute up to once a day — so that I can balance responsiveness against resource usage for each account.

> **US-5.** As any user, I want to set the interval to a "push only" / "always connected" mode (effectively zero polling) for IMAP accounts that support push, so that I do not waste resources on periodic polling when push is handling real-time delivery.

### 6.2 Folder-level polling control

> **US-6.** As any user, I want to configure whether each folder uses push or periodic polling, so that I can reserve push connections for the folders I care about most (typically the Inbox) and let less important folders poll on a schedule.

> **US-7.** As a server-constrained user, I want the application to default system folders (Sent, Drafts, Trash, Spam, Archive) to polling mode, since most servers only allow push for a handful of folders.

> **US-8.** As a power user, I want to set a poll factor (multiplier) on individual folders, so that a folder can poll less frequently than the account's base interval — e.g. a factor of 2 means the folder is polled every 30 minutes when the account interval is 15 minutes.

> **US-9.** As any user, I want to be informed that most servers limit the number of simultaneous push connections, so that I understand why some folders default to polling.

### 6.3 Bulk folder configuration

> **US-10.** As a power user, I want to apply poll settings to all system folders or all user folders at once, so that I do not have to configure each folder individually.

> **US-11.** As any user, when I enable polling for system folders in bulk, I want the application to automatically cap the account's interval at a reasonable maximum (e.g. 15 minutes) if it is currently set higher, so that the folders are actually checked at a useful frequency.

### 6.4 Global poll interval

> **US-12.** As a battery-conscious user, I want a single global poll interval setting that forces all accounts to poll at the same cadence, so that I can quickly switch to a conservative polling mode.

> **US-13.** As any user, I want to be warned when I set a global poll interval that is very frequent (e.g. less than 15 minutes), since frequent polling consumes significantly more resources than push.

> **US-14.** As any user, I want the application to explain (via a hint or informational text) that periodic polling compares local and remote messages every cycle, which is resource-intensive compared to push, so that I make an informed choice.

### 6.5 Network-aware polling

> **US-15.** As a battery-conscious user, I want to enable or disable periodic polling based on network type (metered vs. unmetered), so that the application does not consume mobile data or battery for polling when I am on an expensive connection.

> **US-16.** As a power user, I want to exempt specific accounts from network-based polling restrictions, so that my most important account always polls regardless of network type.

### 6.6 Auto-tuning

> **US-17.** As any user, I want the application to automatically reduce the keep-alive interval if it detects that the server is prematurely closing idle connections, so that I do not have to manually troubleshoot connection drops.

> **US-18.** As any user, I want the auto-tuning to have a sensible floor (e.g. 9 minutes) so that it does not reduce the interval to an unreasonably aggressive value.

> **US-19.** As any user, when I manually change the poll interval for an account, I want any previously auto-tuned state to be reset, so that the application re-learns the server's behavior from the new baseline.

### 6.7 Interaction with push

> **US-20.** As any user, I want to understand that on accounts using push (IMAP IDLE), the "interval" setting controls how often the application sends a keep-alive signal — not how often it polls — and that push delivers new mail in real time regardless of this setting.

> **US-21.** As any user, when push is active, I want the application to optionally trigger a sync check alongside each keep-alive signal, so that any changes missed by the push channel are caught.

> **US-22.** As any user, if push is not supported by the server or if I set the interval below the keep-alive threshold, I want the application to automatically fall back to periodic polling, so that I still receive new mail.

### 6.8 On-demand and schedule interaction

> **US-23.** As any user, I want accounts marked as "on-demand" (manual sync only) to never poll automatically, regardless of any interval setting, so that I have full control over when those accounts check for mail.

> **US-24.** As any user, I want accounts that are exempt from the global sync schedule to continue polling at their configured interval even during schedule blackout periods, so that my critical accounts are always up to date.

## 7. Functional Requirements

### 7.1 Account-level poll interval

- **FR-1.** Each account shall have a configurable poll interval, expressed in minutes.
- **FR-2.** The default poll interval for a new account shall be **15 minutes**.
- **FR-3.** The available interval choices shall be: 0 (push only / always connected), 1, 2, 5, 10, 15, 30, 60, 120, 240, 480, and 1440 minutes.
- **FR-4.** The minimum enforced polling interval (when polling is enabled) shall be **1 minute**. Any value below 1 minute (other than 0, which means push-only) shall be clamped to 1.
- **FR-5.** For IMAP accounts with push capability, an interval of 0 shall mean "use push only, no periodic polling." The application shall maintain a persistent connection and rely on the server to notify of changes.
- **FR-6.** For IMAP accounts with push capability and a non-zero interval at or above the keep-alive threshold (9 minutes), the interval shall control the keep-alive signal frequency rather than a full poll cycle. Push remains the primary delivery mechanism.
- **FR-7.** For IMAP accounts with push capability and a non-zero interval below the keep-alive threshold, or for accounts where push is not supported, the interval shall control a full periodic poll cycle (connect, compare, download).
- **FR-8.** For POP3 accounts (where push is never available), the interval shall always control a full periodic poll cycle.
- **FR-9.** The interval setting shall be presented in the account configuration screen with a label and hint that explains its dual role: "Frequency of refreshing the connection for push messages, or frequency of checking for new messages."

### 7.2 Folder-level poll control

- **FR-10.** Each folder shall have an independent boolean setting controlling whether it uses push (when available) or periodic polling.
- **FR-11.** The Inbox folder shall default to push mode (polling disabled).
- **FR-12.** System folders other than the Inbox (Drafts, Sent, Trash, Spam, Archive) shall default to polling mode (push disabled), reflecting the practical limitation that most servers support push for only a small number of folders.
- **FR-13.** User-created folders shall default to push mode (polling disabled), but this shall be configurable.
- **FR-14.** Each folder shall have a configurable poll factor (integer multiplier, default 1) that is applied to the account's base poll interval. The effective poll interval for the folder is: `account interval × folder poll factor`.
- **FR-15.** The folder configuration screen shall display an informational note explaining that most servers limit push to a handful of folders.

### 7.3 Bulk folder configuration

- **FR-16.** The application shall provide a bulk-apply operation that can set the polling mode for all system folders or all user folders of an account in one action.
- **FR-17.** When the user enables polling for system folders in bulk and the account's poll interval is greater than 15 minutes, the application shall automatically reduce the account's poll interval to 15 minutes and inform the user, so that newly polling folders are checked at a useful frequency.

### 7.4 Global poll interval

- **FR-18.** The application shall provide a global poll interval setting in the synchronization preferences, with the same range of choices as the account-level setting (0–1440 minutes).
- **FR-19.** When the global poll interval is set to a non-zero value, it shall override per-account intervals for the purpose of scheduling periodic sync cycles.
- **FR-20.** When the global poll interval is set to 0, per-account intervals shall govern.
- **FR-21.** When the global poll interval is set to a value greater than 0 and less than 15 minutes, the application shall display a prominent warning: "This can cause high battery usage!"
- **FR-22.** The application shall display an informational hint near the global poll interval explaining that periodic checking compares local and remote messages every cycle, which is a resource-intensive operation.

### 7.5 Network-aware polling

- **FR-23.** The application shall provide two independent toggles in synchronization preferences: one to enable periodic polling on metered networks, and one to enable periodic polling on unmetered networks.
- **FR-24.** When a network-type toggle is disabled, no periodic polling shall occur while on that type of network — accounts shall rely on push (if available) or wait until the network type changes.
- **FR-25.** Each account shall have an independent "exempt from network restrictions" flag. When enabled, the account shall poll at its configured interval regardless of the network-type toggles.

### 7.6 Auto-tuning

- **FR-26.** The application shall provide an "automatically tune the keep-alive interval" setting (per account or global, default enabled).
- **FR-27.** When auto-tuning is active and the application detects that the server prematurely terminated an idle connection, it shall reduce the account's keep-alive interval by a step (2 minutes) and retry.
- **FR-28.** Auto-tuning shall not reduce the keep-alive interval below a floor of **9 minutes**.
- **FR-29.** The application shall track the number of consecutive keep-alive successes and failures per account to inform tuning decisions.
- **FR-30.** When the user manually changes an account's poll interval, all auto-tuning state for that account (success/failure counters, tuned status) shall be reset, and tuning shall restart from the new baseline.
- **FR-31.** When auto-tuning detects that a server is unreliable for push (e.g. sends premature responses within an abnormally short window), it may recommend switching the account to a polling-only mode at a recommended interval (15 minutes) rather than continuing to fight with push.

### 7.7 Interaction with push

- **FR-32.** Polling and push shall be independent mechanisms. An account may use both simultaneously: push for real-time delivery on selected folders, and polling as a periodic consistency check or for folders that cannot use push.
- **FR-33.** An optional "poll on keep-alive" setting shall allow the application to trigger a sync check each time it sends a keep-alive signal on a push connection, catching any changes that the push channel may have missed.
- **FR-34.** When an account's interval is set to 0 (push only) and push is not supported by the server, the application shall automatically fall back to polling at the default interval (15 minutes) and inform the user.

### 7.8 On-demand and schedule interaction

- **FR-35.** Accounts configured as on-demand (manual sync) shall never initiate automatic polling, regardless of any interval setting. Polling shall occur only when the user explicitly triggers a sync.
- **FR-36.** Accounts with an "ignore schedule" exemption shall continue to poll at their configured interval even during global sync schedule blackout periods.
- **FR-37.** Changes to the poll interval shall take effect on the next scheduled poll cycle; the application shall not require a restart.

### 7.9 Periodic folder-list sync

- **FR-38.** The application shall optionally synchronize the folder list itself on a periodic basis (in addition to syncing the folder list when first connecting to an account), so that server-side folder changes are detected between connection cycles.

## 8. Non-Functional Requirements

- **NFR-1. Resource efficiency.** A poll cycle shall be as lightweight as possible: only messages changed since the last sync should be compared and downloaded. The application shall not re-download the entire mailbox on each poll.
- **NFR-2. Battery and CPU impact.** The application shall document (via UI hints) that frequent polling (< 15 minutes) is significantly more resource-intensive than push, and that intervals of 30–60+ minutes are recommended for non-critical accounts.
- **NFR-3. Reliability.** If a poll cycle fails (network error, server timeout, authentication failure), the application shall retry on the next scheduled cycle without crashing or losing state. Transient failures shall not permanently disable polling.
- **NFR-4. Transparency.** The user shall be able to determine, for any given account, whether it is currently using push or polling, and what the effective interval is. This information should be visible in the account's status or the operations log.
- **NFR-5. Responsiveness.** Changing a poll interval shall take effect promptly (within one poll cycle); the user shall not need to restart the application or the sync service.
- **NFR-6. Scale.** The polling scheduler shall handle at least 20 accounts, each with up to 50 folders in polling mode, without degraded performance or excessive resource contention.
- **NFR-7. Offline resilience.** If the network is unavailable, poll cycles shall be skipped gracefully and resume when connectivity returns.
- **NFR-8. Privacy.** No polling telemetry or interval data shall be transmitted to any third party. All configuration is local.

## 10. Open Questions

- **OQ-1. Keep-alive threshold.** The source application uses 9 minutes as the threshold below which a non-zero interval triggers full polling instead of IDLE with keep-alive. Is this the right threshold for the desktop application, or should it be adjustable? The value appears to be derived from practical server behavior rather than any RFC requirement (RFC 2177 suggests 29 minutes as a reasonable IDLE timeout).
- **OQ-2. Auto-tune step size.** Auto-tuning currently reduces the interval by 2 minutes per failure. Should the step be configurable, or should a more adaptive algorithm (e.g. binary search) be used to converge faster?
- **OQ-3. Auto-optimization recommendation.** The source application can automatically recommend switching an account from push to polling at 15 minutes if the server is persistently unreliable. Should this recommendation be automatic (with a notification), or should it require explicit user confirmation?
- **OQ-4. Global vs. per-account network-awareness.** The metered/unmetered toggles are currently global. Should per-account network-awareness be supported, or is the per-account exemption flag sufficient?
- **OQ-5. Folder poll factor range.** The source application allows a per-folder poll factor but does not clearly document the maximum useful value. Should a practical cap be enforced (e.g. factor of 10, yielding a maximum effective interval of 10 × 1440 = 10 days)?
- **OQ-6. NOOP vs. IDLE keep-alive.** The source application offers a per-account option to send NOOP commands instead of using IDLE for keep-alive, as a workaround for servers with broken IDLE support. Should this be exposed as a user-facing option, or should it be handled transparently by auto-tuning?
- **OQ-7. Folder-list sync cadence.** The periodic folder-list sync (FR-38) is mentioned as an option in the source application's settings. Should this be tied to the poll interval, or should it have its own independent cadence?
- **OQ-8. Interaction with "poll on keep-alive."** The "poll on keep-alive" option (FR-33) triggers a sync check alongside each IDLE keep-alive signal. This effectively doubles the work done during keep-alive. Should this be on by default, off by default, or conditionally enabled based on server reliability?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Dual-purpose interval.** The source application uses a single numeric setting per account (`poll_interval`) for two purposes: the keep-alive cadence when IMAP IDLE is active, and the polling cadence when IDLE is not in use. The UI presents it differently depending on context — "keep-alive interval" for IMAP, "poll interval" for POP3 — but it is the same underlying value. This simplification avoids asking the user to configure two separate timers per account, at the cost of conflating two subtly different concerns.

- **N-2. Folder poll factor as a multiplier.** Rather than giving each folder its own independent interval (which would create a combinatorial explosion of timers), the source application uses a single multiplier relative to the account's base interval. This keeps the scheduling simple: all folders of an account share the same timer, and the factor determines how many cycles a folder skips between actual checks.

- **N-3. Default folder modes reflect server reality.** The default of "push for Inbox, poll for everything else" reflects the empirical reality that most IMAP servers support only a small number of simultaneous IDLE connections. The application does not attempt to detect the server's IDLE limit; instead, it conservatively defaults non-Inbox folders to polling and lets the user opt individual folders into push.

- **N-4. Battery warning threshold at 15 minutes.** The source application warns when the global interval is set below 15 minutes. This threshold is not arbitrary: a 15-minute interval roughly aligns with the OS-level alarm scheduling granularity for background tasks, and intervals shorter than this require more aggressive scheduling that can conflict with power-saving modes.

- **N-5. Auto-tuning rationale.** Servers vary widely in how long they keep IDLE connections open. RFC 2177 does not mandate a minimum, and some servers close IDLE after as little as 5 minutes. Auto-tuning allows the application to adapt to each server without the user needing to diagnose connection-drop logs. The 9-minute floor was chosen as a balance between connection stability and resource usage: below 9 minutes, the overhead of maintaining IDLE approaches the cost of periodic polling, so the application switches to polling outright.

- **N-6. Global interval as a power-user override.** The global poll interval is not a "default for new accounts" — it is an override that forces a cadence on all accounts simultaneously. This design serves the use case of a user who temporarily wants to reduce sync frequency (e.g. during travel or on battery) without reconfiguring every account individually.

- **N-7. On-demand accounts are exempt from all automatic polling.** An on-demand account is, by definition, one that the user wants to check manually. No global or per-account interval setting should override this intent. The user must explicitly trigger sync via a refresh action or menu command.
