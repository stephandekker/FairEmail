# Backend Implementation — Architectural Decisions

Companion to [0.0-backend-implementation.md](0.0-backend-implementation.md). These decisions resolve the open questions raised in §10 of the parent epic and govern every user story under it.

> Decisions recorded 2026-05-05. Revisit when a slice surfaces evidence that one of these is wrong; update this file in place rather than starting parallel docs.

---

## D-1. Async runtime — `tokio` (multi-threaded)

Resolves: **OQ-1**.

The sync engine runs on a `tokio` multi-threaded runtime hosted on a dedicated worker thread. UI ↔ engine messaging uses `tokio::sync::mpsc` channels; change notifications cross to the GTK main loop via a `glib::MainContext` idle source that drains a `tokio` broadcast receiver.

**Why:** `async-imap`, `lettre`, `rustls`, `notify-rust` (D-Bus path), and the rest of the 2026 mail-crate ecosystem target `tokio`. `async-std` is effectively unmaintained. A `glib`-only runtime would lose access to those crates and force hand-rolled protocol code.

**Implications:**

- The GTK thread never `.await`s anything that touches the network or disk.
- Engine code is `async fn` end-to-end; storage primitives are sync (SQLite is sync; we run them via `tokio::task::spawn_blocking` where they sit on hot paths).
- Add `tokio` with `rt-multi-thread`, `macros`, `sync`, `time`, `net`.

## D-2. Keychain — `secret-service` crate

Resolves: **OQ-2**.

The `CredentialStore` trait's default implementation uses the [`secret-service`](https://crates.io/crates/secret-service) crate, which speaks the freedesktop Secret Service API (the protocol `libsecret` implements).

**Why:** Architecture.md explicitly names libsecret. The Secret Service API's attribute-based search makes per-account enumeration on delete trivial (`{ "fairmail-account-id": "<uuid>" }`). v1 is Linux-only, so `keyring`'s cross-platform abstraction has no payoff yet.

**Implications:**

- Secrets are stored with the attributes `{ "application": "fairmail", "fairmail-account-id": "<uuid>", "role": "imap-password" | "smtp-password" | "oauth-refresh-token" }`.
- A future macOS/Windows port replaces this implementation behind the `CredentialStore` trait without disturbing callers.
- Tests run against an in-memory mock; CI does not need a live D-Bus session bus.

## D-3. Body-text extraction for FTS — `mail-parser` + `html2text`

Resolves: **OQ-3**.

Headers are parsed with `mail-parser` (the de facto Rust mail-parsing crate). The `body_text` field indexed by FTS5 is derived as: prefer the first `text/plain` part if present, otherwise strip the first `text/html` part with `html2text`, otherwise empty.

**Why:** Both crates are mature, narrow-purpose, and replaceable. The renderer is local to the FTS pipeline — swapping it later only requires rebuilding the FTS index, not re-fetching content (the `.eml` files are intact).

## D-4. Schema — pragmatic JMAP-flavored

Resolves: **OQ-4**.

The schema follows the FR-4 layout in the parent epic: multi-folder per message via the `message_folders` join table (JMAP-shaped), but parsed headers stay as flat columns on `messages` for query speed. No separate `email_addresses` / `mailboxes` / `threads` tables until a downstream feature actually needs them.

**Why:** Closer to JMAP than to raw IMAP without paying full-normalization cost up-front for hypothetical future needs. A later migration can split out address tables when contacts/search features demand them.

## D-5. Threading — derived (JWZ), with override column

Resolves: **OQ-5**.

`messages.thread_id TEXT` is populated by a JWZ-style derivation from `References`, `In-Reply-To`, and `Message-ID`. A nullable `messages.server_thread_id TEXT` column carries server-supplied identifiers (Gmail `X-GM-THRID`, JMAP `threadId`) when available; the application prefers the server value where present and falls back to the derived value otherwise.

**Why:** Works on every IMAP server with no Gmail dependency. The override column accommodates Gmail's authoritative threading without re-shaping the schema. Population logic is epic 3.2's concern; this epic only declares the columns.

## D-6. Content-store cleanup — synchronous reference-count delete

Resolves: **OQ-6**.

Deleting a `messages` row triggers an immediate reference-count check; if no other row references the same `content_hash`, the `.eml` file is deleted in the same operation. No periodic GC pass for v1.

**Why:** Simpler, fewer moving parts. The reference count is one indexed query against `messages.content_hash`. Defer GC-as-optimization until measurement shows synchronous delete is a problem in practice.

## D-7. Database layout — single shared SQLite file

Resolves: **OQ-7**.

One `fairmail.db` file under `$XDG_DATA_HOME/fairmail/` serves all accounts. WAL mode permits concurrent UI reads while the sync engine writes. The sync engine remains the single writer and processes accounts sequentially in v1; concurrent per-account sync is a later optimization.

**Why:** Cross-account moves and queries stay atomic. Backups are one file. WAL mode mitigates the single-writer cost in practice. A per-account database would complicate cross-account features (unified inbox, cross-account search) that are explicitly on the roadmap.

## D-8. JSON-import compatibility window — two minor releases

Resolves: **OQ-8**.

The `accounts.json` / `settings.json` / `order.json` import path stays in place for two minor releases past the one that introduces the SQLite backend, then is removed. The `*.migrated` recovery files remain on users' disks indefinitely (we never delete them).

**Why:** Two releases gives users a reasonable upgrade window without dragging the import path forever. The recovery files are tiny and never read at runtime.

## D-9. Test infrastructure for IMAP/SMTP — trait mocks for v1

Resolves: **OQ-9**.

v1 ships with trait mocks (the existing `MockImapChecker`, `MockSmtpChecker`, etc.) used for unit tests. Integration tests against a real Dovecot/Postfix instance in CI are deferred to a future QA epic.

**Why:** Trait-mock coverage is sufficient to validate the wiring this epic introduces. A real-server CI rig adds significant operational cost and is best designed against actual end-to-end features rather than infrastructure scaffolding.

## D-10. POP3 deferral — confirmed

Resolves: **OQ-10**.

POP3 wizard fields continue to be saved (the `accounts.protocol` column accepts both `imap` and `pop3`), but POP3 message fetching is not implemented in this epic. A follow-up epic adds POP3 sync. Epic 1.4 manual-configuration POP3 settings round-trip through the new schema unchanged.
