# Epic: Desktop Notifications

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **8.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Desktop notifications (libnotify) for new mail, sync errors, send confirmations."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

A desktop email client that synchronizes mail in the background must keep the user informed about events that require attention — new messages, synchronization failures, and the status of outgoing mail — without forcing the user to keep the application window open and focused. On a Linux desktop the standard mechanism for this is the system notification daemon (the freedesktop Notifications specification), which provides transient pop-up banners, a persistent notification tray, action buttons, sounds, and urgency levels.

FairEmail's Android counterpart implements a rich, deeply configurable notification subsystem spanning eight distinct notification channels, per-account and per-folder and per-sender customization, notification actions (reply, archive, delete, snooze, etc.), grouping and summary notifications, privacy controls, rate limiting, quiet hours, and launcher badge counts. This epic translates those behaviors to their desktop equivalents, dropping concepts that have no meaningful desktop counterpart (LED light, vibration, screen wake, car-mode suppression) and mapping the rest onto the freedesktop notification and system-tray conventions.

The notification subsystem is the primary mechanism by which the application communicates with the user when the main window is not in focus. It must be reliable, timely, respectful of the user's attention, and deeply configurable — from "notify me about everything" down to "only notify me when this specific sender writes to this specific folder".

## 2. Goals

- **G1.** Deliver timely, non-intrusive desktop notifications for new mail arriving in user-selected folders across all synchronized accounts.
- **G2.** Surface synchronization errors, authentication failures, and connection problems as high-urgency notifications so the user is aware of disruptions without checking the application.
- **G3.** Provide clear feedback on outgoing mail status — queued, sending, sent, or failed — via notifications and the persistent system-tray icon.
- **G4.** Allow fine-grained per-account, per-folder, and per-sender notification configuration: whether to notify, with what urgency, with what sound, and with what content.
- **G5.** Offer actionable notifications — the user should be able to triage mail (mark read, archive, delete, reply) directly from the notification without opening the main window.
- **G6.** Respect the user's attention: support grouping/summarization, rate limiting, quiet hours, and "only notify from known contacts" to prevent notification fatigue.
- **G7.** Integrate cleanly with the desktop environment: honor system Do Not Disturb, display unread counts on the system tray icon or launcher badge, and use the system's notification sound infrastructure.

## 3. Non-Goals

- **NG1.** This epic does **not** define the persistent background daemon or system-tray icon itself — those are covered by epic 8.9. This epic defines *what* should be communicated via notifications, not the lifecycle of the process that sends them.
- **NG2.** This epic does **not** define quiet hours (8.7), per-account/per-folder/per-sender notification rules (8.2), notification actions (8.3), quick reply from notification (8.4), summary/grouped notifications (8.5), sender avatars in notifications (8.6), custom notification sounds (8.8), or autostart (8.10). Those are separate epics under the Notifications feature group. Where this epic references those capabilities, it defines the *interface contract* — not the detailed behavior, which belongs in the respective epic.
- **NG3.** Vibration and LED notification light have no meaningful desktop counterpart and are out of scope.
- **NG4.** Screen-wake behavior (turning on a phone display) has no desktop equivalent and is out of scope.
- **NG5.** Car-mode and in-call suppression are mobile-specific and out of scope.
- **NG6.** This epic does not prescribe the transport mechanism for notifications (e.g. which IPC protocol, which daemon), only the user-visible behavior.

## 4. Glossary

| Term | Definition |
|---|---|
| **Notification** | A transient or persistent message shown by the desktop notification daemon, typically as a pop-up banner and/or an entry in the notification center. |
| **Notification category** | A named, user-configurable channel through which the application routes notifications. Each category has independent urgency, sound, and display settings. The application defines a fixed set of categories. |
| **Urgency** | The importance level of a notification (low, normal, critical), which determines how prominently the desktop environment displays it and whether it may bypass Do Not Disturb. |
| **Summary notification** | A single notification that aggregates multiple individual events (e.g. "5 new messages") instead of showing one notification per event. |
| **Grouped notifications** | A set of individual notifications visually grouped together, with a summary header, so the user can expand them to see details or collapse them to save space. |
| **Notification action** | A button embedded in the notification that the user can click to perform an operation (mark read, archive, reply, etc.) without opening the main window. |
| **System tray icon** | A persistent icon in the desktop panel's notification area (system tray / indicator area) that shows the application's state — connection status, unread count, etc. |
| **Do Not Disturb (DND)** | A system-wide mode that suppresses or silences non-critical notifications. The application must respect it, but critical notifications (errors, authentication failures) may bypass it. |
| **Rate limiting** | A mechanism to prevent rapid-fire notification sounds when many messages arrive in quick succession. |
| **Known contact** | A sender whose address exists in the application's local contact store or the system address book. |
| **Notify-enabled folder** | A folder whose per-folder notification setting is turned on, meaning new messages in that folder should produce notifications. |

## 5. Personas

- **P1. The always-connected worker.** Uses the email client all day with the main window open. Wants subtle new-mail notifications that do not interrupt their flow — a brief banner, a badge count update, and the ability to triage from the notification if it looks urgent.
- **P2. The background monitor.** Keeps the email client running in the system tray but rarely has the main window open. Relies heavily on notifications to know when something needs attention. Wants rich previews, sender information, and one-click actions.
- **P3. The quiet professional.** Has multiple accounts but only wants notifications from specific folders and specific senders. Configures fine-grained rules and expects the application to respect them precisely — no stray notifications from mailing lists.
- **P4. The newcomer.** Has just set up their first account and expects notifications to "just work" for their Inbox without any configuration.

## 6. User Stories

### 6.1 New mail notifications

> **US-1.** As a newcomer, when I finish adding my first account and new mail arrives in my Inbox, I want to see a desktop notification automatically, so that I know the application is working without configuring anything.

> **US-2.** As any user, when new mail arrives in a folder that has notifications enabled, I want to see a desktop notification containing the sender's name, the message subject, and optionally a preview of the message body, so that I can decide whether to act on it without switching to the application.

> **US-3.** As any user, I want each new-mail notification to identify which account and folder the message belongs to (e.g. via a subtitle or color), so that I can distinguish work mail from personal mail at a glance.

> **US-4.** As any user, when multiple new messages arrive in quick succession, I want the notifications to be grouped or summarized rather than producing a barrage of individual pop-ups, so that my desktop is not overwhelmed.

> **US-5.** As a background monitor, I want individual notifications within a group to be expandable so that I can see each sender and subject without opening the application.

> **US-6.** As any user, when I have already seen a notification and the application is brought to the foreground, I want the corresponding notifications to be automatically cleared (optionally), so that stale notifications do not accumulate.

> **US-7.** As any user, when I mark a message as read (whether from within the application or from a notification action), I want the corresponding notification to be dismissed, so that the notification state stays in sync with the mailbox state.

> **US-8.** As any user, I want new-mail notifications to respect my per-folder notification setting: folders with notifications disabled should never produce a notification, regardless of whether they are in the Unified Inbox or not.

> **US-9.** As a quiet professional, I want an option to receive notifications only from known contacts (senders already in my address book or local contact store), so that I am not disturbed by cold emails or mailing lists.

> **US-10.** As any user, I want the notification content to respect my privacy settings: when a privacy mode is active, notifications should show only a count or a generic "new mail" message without revealing sender, subject, or preview text.

### 6.2 Sync error and warning notifications

> **US-11.** As any user, when a synchronization or connection error occurs (authentication failure, connection timeout, server error), I want to see a notification with the affected account name and a human-readable description of the problem, so that I can take corrective action.

> **US-12.** As any user, I want error notifications to indicate whether the application will retry automatically and, if so, how many retries remain, so that I know whether I need to intervene.

> **US-13.** As any user, I want sync error and authentication failure notifications to be treated as high-urgency so they are prominently displayed and may bypass Do Not Disturb, because a broken email connection is something I want to know about even during focus time.

> **US-14.** As any user, I want error notifications to be auto-dismissed when the underlying problem is resolved (e.g. reconnection succeeds), so that I am not left with stale error banners.

### 6.3 Send status notifications

> **US-15.** As any user, when I send a message, I want the system-tray icon or a transient notification to indicate that a send operation is in progress (e.g. "Sending 2 messages"), so that I have confidence the message is being dispatched.

> **US-16.** As any user, when a send operation fails, I want a notification with the recipient address and the error reason, so that I can diagnose and retry.

> **US-17.** As any user, I want send-failure notifications to be high-urgency, because a failed send is an event I need to know about promptly.

> **US-18.** As any user, I want send progress to be reflected in the system-tray icon tooltip or a transient notification, showing the number of queued/unsent messages, so that I have visibility into the outbox without opening the application.

### 6.4 Notification configuration

> **US-19.** As any user, I want to configure notification behavior globally — enable/disable all notifications, choose whether to show subject, preview text, and account/folder subtitles — so that I have a single place to control the notification experience.

> **US-20.** As any user, I want the application to define distinct notification categories (new mail, warnings, errors, send status, background service) with independent urgency and sound defaults, so that I can customize each type separately through the system's notification settings.

> **US-21.** As a power user, I want per-account notification settings that override the global defaults, so that I can silence a noisy account without affecting my primary one.

> **US-22.** As a power user, I want per-folder notification settings that override the account defaults, so that I can enable notifications for a specific high-priority folder while leaving the rest of the account silent.

> **US-23.** As a power user, I want per-sender notification settings that override folder and account defaults, so that I can ensure I am always notified when a specific person writes to me regardless of which folder the message lands in.

> **US-24.** As any user, I want the precedence order for notification settings to be clear and predictable: a rule-based override takes priority, then sender-specific, then folder-specific, then account-specific, then global default.

> **US-25.** As any user, I want a rate-limiting option so that when many messages arrive at once, only the first notification produces a sound and subsequent ones arrive silently for a configurable interval.

> **US-26.** As any user, I want the option to suppress new-mail notifications while the application window is in the foreground, since I can already see the mail arriving in the message list.

### 6.5 Notification content and display

> **US-27.** As any user, I want to control what content appears in notifications: sender name (always), subject (optional), message body preview (optional), account and folder name (optional).

> **US-28.** As any user, I want the notification to display the sender's contact photo or avatar when available, so that I can visually identify the sender at a glance.

> **US-29.** As any user, I want notifications to show the received timestamp, so that I can tell how recently the message arrived.

> **US-30.** As any user, I want an option to transliterate or normalize non-Latin scripts in notification text, for compatibility with notification daemons or external displays that have limited character support.

### 6.6 Launcher badge / unread count

> **US-31.** As any user, I want the system-tray icon and/or launcher entry to display an unread message count (badge), so that I can see at a glance whether I have unread mail without reading each notification.

> **US-32.** As any user, I want to choose whether the badge count reflects the number of active new-message notifications or the total number of unread messages across notify-enabled folders, since these are different numbers and different users prefer different semantics.

### 6.7 Notification dismissal and lifecycle

> **US-33.** As any user, I want to dismiss individual or grouped notifications by the standard desktop gesture (click dismiss / swipe away), and I want the application to record that the notification was dismissed so it does not re-display it.

> **US-34.** As any user, I want the ability to dismiss all new-mail notifications for a group (account or unified) in one action.

> **US-35.** As any user, when I restart the application or the notification daemon, I want existing unread-mail notifications to be restored so that I do not lose awareness of messages I haven't acted on.

## 7. Functional Requirements

### 7.1 Notification categories

- **FR-1.** The application shall define the following distinct notification categories, each with independent default urgency and user-configurable properties:
  - **Service / monitoring**: persistent low-priority notification indicating the background sync daemon is running.
  - **New mail**: normal-to-high urgency notifications for incoming messages.
  - **Send status**: normal urgency notifications for outgoing mail progress.
  - **Warning**: high urgency notifications for retryable errors (connection failures, transient server errors).
  - **Error**: high urgency notifications for permanent failures (authentication failures, unrecoverable errors).
  - **Alerts**: high urgency notifications for server-side alerts.
- **FR-2.** Each category shall be independently configurable by the user for: sound, urgency, and whether it may bypass Do Not Disturb.
- **FR-3.** Warning, Error, and Alert categories shall bypass Do Not Disturb by default.

### 7.2 New mail notifications

- **FR-4.** When a new (previously unseen) message is received in a folder whose notification setting is enabled, and the owning account's notification setting is also enabled, the application shall emit a desktop notification.
- **FR-5.** By default, only the Inbox folder of each account shall have notifications enabled. Sent, Drafts, Trash, Spam, and Archive folders shall have notifications disabled by default.
- **FR-6.** The notification shall include at minimum: the sender's display name. Optionally (per user configuration): the message subject, a body preview (plain text, up to a reasonable length), the account name, and the folder name.
- **FR-7.** If a contact photo or avatar is available for the sender, the notification shall display it as the notification icon or image.
- **FR-8.** The notification shall include the message's received timestamp.

### 7.3 Notification grouping and summarization

- **FR-9.** When multiple new messages are pending notification, the application shall group them. The grouping strategy shall be configurable:
  - **Unified** (default): all new-mail notifications across all accounts are grouped together under a single summary.
  - **Per-account**: notifications are grouped by account, with a separate summary per account.
  - **Per-folder**: notifications are grouped by folder.
- **FR-10.** Each group shall have a summary notification showing the total count of new messages and a list of sender names (with subjects if enabled).
- **FR-11.** Individual message notifications within a group shall be viewable when the group is expanded.
- **FR-12.** The application shall cap the number of individual notifications displayed per group (e.g. 10), with additional messages represented only in the summary count, to prevent notification overflow.
- **FR-13.** The user shall be able to opt for summary-only mode, where only the summary notification is shown and no individual message notifications are displayed.

### 7.4 Notification sound and rate limiting

- **FR-14.** The application shall play the configured notification sound when a new-mail notification is emitted, subject to rate limiting.
- **FR-15.** A configurable rate-limiting interval shall be provided. When multiple notifications arrive within the interval, only the first shall produce a sound; subsequent notifications within the interval shall be silent.
- **FR-16.** An "alert once" option shall be provided: when enabled, only the first notification in a group produces a sound; updates to the group (additional messages) are silent.
- **FR-17.** The notification sound shall be configurable globally, per-account, per-folder, and per-sender, with the most specific setting taking precedence.

### 7.5 Sync error and warning notifications

- **FR-18.** When a synchronization, connection, or authentication error occurs, the application shall emit a notification on the Warning (retryable) or Error (permanent) category.
- **FR-19.** The error notification shall include: the affected account name, a human-readable error description, and (for retryable errors) the number of remaining retries and whether automatic retry is scheduled.
- **FR-20.** Error notifications shall be auto-dismissed when the error condition clears (e.g. successful reconnection).
- **FR-21.** Error notifications shall be dismissible by the user without affecting the retry mechanism.

### 7.6 Send status notifications

- **FR-22.** While one or more messages are queued for sending, the application shall indicate the outbound queue status — either via the system-tray icon tooltip, a persistent low-priority notification, or both.
- **FR-23.** The send status indication shall show: the number of queued messages and, during active transmission, a progress indication.
- **FR-24.** When a send operation fails, the application shall emit a Warning or Error notification including the recipient address and the error reason.
- **FR-25.** Send-failure notifications shall be high-urgency.

### 7.7 Per-account, per-folder, and per-sender configuration

- **FR-26.** Each account shall have an independent notification enable/disable toggle that overrides the global setting for all folders in that account.
- **FR-27.** Each folder shall have an independent notification enable/disable toggle that overrides the account setting.
- **FR-28.** The user shall be able to assign per-sender notification overrides. A per-sender override shall take precedence over the folder and account settings.
- **FR-29.** The effective precedence order shall be, from highest to lowest: (1) rule-based override, (2) per-sender setting, (3) per-folder setting, (4) per-account setting, (5) global default.
- **FR-30.** Creating or deleting per-account, per-folder, or per-sender notification configurations shall be accessible from the relevant account, folder, or message context.

### 7.8 Privacy and content controls

- **FR-31.** When a privacy mode is active (e.g. the application requires authentication to unlock), notification content shall be redacted: only a count or generic label shall be shown, with no sender, subject, or preview.
- **FR-32.** The user shall be able to independently enable or disable each content field in notifications: sender name, subject, body preview, account/folder subtitle.
- **FR-33.** An option to transliterate or restrict notification text to ASCII shall be provided, for compatibility with limited-capability notification displays.

### 7.9 Notification lifecycle and dismissal

- **FR-34.** Notifications shall be auto-cleared when the corresponding message is marked as read, whether from within the application, from a notification action, or from a remote client.
- **FR-35.** The user shall have the option to auto-clear all new-mail notifications when the application window is brought to the foreground.
- **FR-36.** Dismissed notifications shall not be re-displayed for the same message unless the message is marked unread again and a new sync cycle detects it.
- **FR-37.** On application restart, the application shall restore notifications for messages that were pending notification at the time of shutdown, so that no unread messages are silently lost.
- **FR-38.** A "clear all" action on a group summary shall dismiss all individual notifications in that group.

### 7.10 Unread badge

- **FR-39.** The application shall expose an unread message count to the desktop environment's badge/indicator mechanism (system tray icon badge, launcher count, or equivalent).
- **FR-40.** The user shall be able to choose whether the badge count reflects: (a) the number of active new-message notifications, or (b) the total number of unread messages in notify-enabled folders.
- **FR-41.** The badge count shall update within a few seconds of a notification being emitted or dismissed.

### 7.11 Suppression and filtering

- **FR-42.** The user shall be able to suppress new-mail notifications when the application window is in the foreground.
- **FR-43.** The user shall be able to restrict new-mail notifications to known contacts only (senders in the local contact store or system address book).
- **FR-44.** Filter rules (defined in the Rules & Automation feature group) shall be able to suppress or silence notifications for messages matching the rule.

### 7.12 Independence from other features

- **FR-45.** A folder's notification setting and its Unified Inbox membership shall be independently configurable. Changing one shall never modify the other.
- **FR-46.** A folder's notification setting and its synchronization setting shall be independently configurable. A folder that is synchronized but has notifications disabled shall never produce a new-mail notification.

## 8. Non-Functional Requirements

- **NFR-1. Timeliness.** A new-mail notification shall be emitted within a few seconds of the message being detected by the synchronization subsystem. The user-perceived delay between message arrival on the server and notification display shall be bounded by the sync latency, not the notification subsystem.
- **NFR-2. Reliability.** Notifications shall not be silently dropped under normal operation. If the notification daemon is unavailable, the application shall queue or retry, and the system-tray icon shall still reflect the unread state.
- **NFR-3. Responsiveness.** Notification actions (mark read, archive, delete) shall complete and update the notification state within one second under normal conditions.
- **NFR-4. Scale.** The notification subsystem shall handle at least 100 new messages arriving in a single sync cycle without crashing, hanging, or overwhelming the desktop notification daemon.
- **NFR-5. Privacy.** Notification content shall never be transmitted to any third-party server. All notification rendering happens locally.
- **NFR-6. Accessibility.** Notification text shall be available to screen readers. Notification actions shall be keyboard-accessible where the desktop environment supports it.
- **NFR-7. Resource efficiency.** The notification subsystem shall not cause excessive CPU or memory usage when processing large batches of new messages. Notifications beyond the per-group cap shall be represented in the summary count only.
- **NFR-8. Desktop integration.** The application shall respect the desktop environment's Do Not Disturb / Focus mode. Notifications categorized as critical (errors, authentication failures) may bypass DND if the category is configured to do so.

## 10. Open Questions

- **OQ-1. Notification daemon capabilities.** The freedesktop Notifications specification defines a baseline, but desktop environments vary in their support for grouping, inline actions, images, and persistence. Should the application degrade gracefully (e.g. fall back to individual notifications if grouping is unsupported), or should a minimum feature set be required? The source application relies heavily on platform notification grouping.
- **OQ-2. Inline reply from notification.** The source application supports direct reply from the notification where the platform supports it. On Linux, notification daemon support for inline text input is inconsistent. Should inline reply be attempted where possible (and gracefully hidden where not), or deferred to a separate epic (8.4)?
- **OQ-3. Per-sender notification channel creation.** The source application creates a per-sender notification channel when the user configures per-sender notification overrides. On the desktop, the mechanism for per-sender customization (separate notification category vs. application-internal routing) needs a design decision.
- **OQ-4. Badge count mechanism.** Linux desktops have multiple, incompatible badge/count mechanisms (Unity launcher API, KDE StatusNotifierItem, GNOME extension APIs, freedesktop StatusNotifierItem). Which mechanisms should be supported, and should the application fall back to tooltip-only on unsupported environments?
- **OQ-5. Notification restoration on daemon restart.** If the notification daemon crashes and restarts, should the application detect this and re-post active notifications? The source application restores notifications on its own restart, but daemon restarts are a distinct scenario on Linux.
- **OQ-6. High-importance message escalation.** The source application uses higher-priority notifications for messages flagged as high importance by the sender. Should this behavior be replicated, and should it be independently configurable?
- **OQ-7. Service/monitoring notification.** The source application shows a persistent low-priority notification indicating the background sync service is running. On the desktop, the system-tray icon serves this purpose. Is a separate persistent notification still needed, or is the tray icon sufficient?
- **OQ-8. Sound file format and location.** The source application delegates sound to the platform. On Linux, should the application ship default notification sounds, or rely entirely on the desktop environment's configured notification sound? How should per-account/per-folder/per-sender custom sounds reference files?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Three independent flags.** The source application treats synchronization, notification, and Unified Inbox membership as three entirely independent per-folder settings. All eight combinations are meaningful and supported. This independence is critical: a user may want a folder synchronized and visible in the Unified Inbox but silent (no notifications), or a folder that notifies but is not in the Unified Inbox. The notification subsystem must never assume anything about a folder's sync or Unified Inbox state.

- **N-2. Inbox-only default.** Only the Inbox has notifications enabled by default. This is a deliberate anti-noise measure: enabling notifications for Sent, Drafts, Trash, or user-created folders by default would cause notifications the user did not expect. The user must opt in to notifications for any other folder.

- **N-3. Precedence chain.** The notification configuration precedence (rule > sender > folder > account > global) mirrors the source application's behavior and reflects the principle of most-specific-wins. A user who sets a per-sender override expects it to dominate, regardless of what the folder or account default says. Rule-based overrides sit above all others because rules represent explicit user intent with conditions.

- **N-4. Rate limiting vs. grouping.** Rate limiting (silence sounds after the first in a burst) and grouping (collapse many notifications into a summary) solve different problems. Rate limiting prevents audio annoyance; grouping prevents visual clutter. Both are needed and are configured independently. The source application implements both: an `alert_once` flag and a configurable rate-limit interval for sound, plus a maximum individual notification count per group for visual grouping.

- **N-5. Privacy mode redaction.** When the source application has biometric/PIN protection enabled, notification content is hidden — only a count is shown. The desktop equivalent is application-lock or keyring-protected access. The principle is the same: if the application requires authentication to view content, notifications must not leak that content. This is a hard constraint, not a preference.

- **N-6. Notification as state, not event.** The source application tracks a per-message "notifying" state and uses it to manage the notification lifecycle — dismissing notifications when messages are read, restoring them on restart, and avoiding re-notification of already-dismissed messages. This stateful model is essential for correctness: without it, restarting the application would either re-notify everything (annoying) or silently drop notifications for unread messages (dangerous). The desktop implementation must maintain equivalent state.

- **N-7. Error notifications bypass DND.** The source application configures warning and error notification channels to bypass Do Not Disturb. The rationale is that a broken email connection or authentication failure is something the user needs to know about even during focus time — unlike new mail, which can wait. This is a sensible default but should be user-overridable.
