# Epic: Auto-Save Drafts

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **4.8** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Drafts saved on paragraph break, on punctuation, or on a timer; revision history kept locally with rollback."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Composing an email can span minutes or hours. Network drops, application crashes, accidental window closures, and user mistakes all threaten in-progress work. A robust auto-save system must protect the user from data loss without demanding explicit "save" actions, while also giving the user the ability to step backward through earlier states of a draft when a line of editing turns out to be wrong.

The Auto-Save Drafts feature exists to:

1. **Prevent data loss** by automatically persisting the current state of a draft at frequent, natural editing boundaries — paragraph breaks, sentence-ending punctuation, and when the compose window loses focus — so that no more than a few seconds of work is ever at risk.
2. **Provide a revision history** that captures the draft at each auto-save point, letting the user undo and redo across saved snapshots rather than only across character-level edits.
3. **Synchronize drafts to the mail server** (when the user opts in) so that drafts are accessible from other devices and survive local data loss.

The feature operates silently in the background. The user should never have to think about saving a draft — it should already be saved.

## 2. Goals

- **G1.** Automatically save the draft at semantically meaningful editing boundaries (paragraph breaks, sentence-ending punctuation) so that saves align with natural pauses in writing.
- **G2.** Automatically save the draft whenever the compose window loses focus or is backgrounded, so that switching away from the editor never risks data loss.
- **G3.** Maintain a local revision history of saved draft states, so that the user can step backward and forward through prior versions of the draft body.
- **G4.** Provide clear, always-visible undo/redo controls that navigate the revision history, so that recovery from unwanted edits is discoverable and immediate.
- **G5.** Optionally synchronize the draft to the server-side Drafts folder, so that drafts are available across devices.
- **G6.** Allow the user to configure which auto-save triggers are active and whether revision history is kept, so that the feature adapts to different editing styles.

## 3. Non-Goals

- **NG1.** This epic does **not** define a periodic timer-based auto-save. Saves are triggered by editing events and lifecycle transitions, not by a clock. (See §11, N-1.)
- **NG2.** This epic does **not** prescribe how draft content or revision snapshots are stored on disk — only that they are durable, recoverable, and cleaned up when no longer needed.
- **NG3.** Character-level undo/redo (the standard text-editing undo stack) is a separate concern from revision-level undo/redo. This epic covers revision history; character-level undo is part of the rich text editor (feature 4.1).
- **NG4.** Conflict resolution between local and server-side draft versions is out of scope. The local version is authoritative during editing; server synchronization is a one-way push.
- **NG5.** Collaborative or multi-device real-time draft editing is not a goal. Only one client edits a draft at a time.
- **NG6.** Auto-save of attachments is not covered here. Attachments are persisted at the point of attachment, not as part of the body auto-save cycle.

## 4. Glossary

| Term | Definition |
|---|---|
| **Draft** | A message that the user has begun composing but has not yet sent. A draft has a body (rich text or plain text), recipients, subject, attachments, and metadata. |
| **Auto-save** | The act of persisting the current state of the draft body to local storage without an explicit user action. |
| **Save trigger** | An editing event or lifecycle transition that causes an auto-save to occur. |
| **Revision** | A numbered snapshot of the draft body captured at a save point. Revisions are stored locally and form a linear sequence. |
| **Revision history** | The ordered collection of all revisions for a given draft, from the initial state (revision 1) through the most recent save. |
| **Current revision** | The revision that the user is currently viewing and editing. This is normally the latest revision, but may be an earlier one if the user has used undo. |
| **Dirty state** | A flag indicating that the draft body has been modified since the last save. Auto-save triggers only fire when the draft is dirty. |
| **Silent save** | An auto-save that occurs in the background without showing a progress indicator or interrupting the user's typing. |
| **Server-side draft** | A copy of the draft stored in the account's IMAP Drafts folder, synchronized from the local draft. |
| **Paragraph break** | A newline character inserted into the draft body, representing the end of a paragraph. |
| **End character** | A punctuation character that signals the end of a clause or sentence: period (`.`), full-width period (`。`), colon (`:`), semicolon (`;`), question mark (`?`), exclamation mark (`!`), or comma (`,`). |

## 5. Personas

- **P1. The fast typist.** Writes quickly in long paragraphs; relies on auto-save at paragraph breaks to capture work. Wants saves to be invisible — no lag, no interruption.
- **P2. The careful composer.** Writes slowly and revises extensively; values the ability to step back to a previous version of the draft when a round of edits goes wrong.
- **P3. The multi-device user.** Starts a draft on one machine and expects to continue it on another; needs server-side draft synchronization.
- **P4. The minimal-interruption user.** Finds any visible save indicator distracting; wants auto-save to be completely silent.
- **P5. The power user.** Wants fine-grained control over when saves happen and whether revision history is kept; may disable punctuation-triggered saves to reduce save frequency.

## 6. User Stories

### 6.1 Automatic saving on paragraph break

> **US-1.** As a fast typist, when I press Enter to start a new paragraph, I want the application to automatically save my draft, so that if the application crashes I lose at most the current paragraph.

> **US-2.** As any user, I want auto-save on paragraph break to be enabled by default, so that I am protected from data loss without configuring anything.

> **US-3.** As a power user, I want to be able to disable auto-save on paragraph break in the send settings, so that I can control when saves happen.

### 6.2 Automatic saving on punctuation

> **US-4.** As a careful composer writing long sentences, I want the option to auto-save whenever I type sentence-ending punctuation (period, question mark, exclamation mark, colon, semicolon, comma, or full-width period), so that saves happen more frequently than paragraph breaks alone.

> **US-5.** As any user, I want punctuation-triggered auto-save to be disabled by default, so that it does not create excessive save points for users who do not want them.

> **US-6.** As a power user, I want to enable punctuation-triggered auto-save in the send settings independently of paragraph-break auto-save.

### 6.3 Automatic saving on loss of focus

> **US-7.** As any user, when the compose window loses focus (I switch to another window, minimize the application, or navigate away from the editor), I want the application to immediately save the current draft, so that my work is not lost if I do not return.

> **US-8.** As any user, I want the loss-of-focus save to happen unconditionally — it should not be possible to disable it — so that there is always a safety net.

### 6.4 Revision history and rollback

> **US-9.** As a careful composer, I want the application to keep a history of prior saved states of my draft body, so that I can return to an earlier version if my recent edits were a mistake.

> **US-10.** As any user, I want visible undo and redo controls in the compose window that step backward and forward through the revision history, so that I can navigate between saved snapshots without guessing.

> **US-11.** As any user, when I undo to a previous revision, I want the draft body to be replaced with the content of that revision, and the undo/redo controls to update to reflect my position in the history.

> **US-12.** As any user, I want the undo control to be available only when there is a previous revision to return to, and the redo control to be available only when there is a later revision to advance to, so that I am never offered an action that does nothing.

> **US-13.** As any user, I want revision history to be enabled by default, so that I have rollback protection without configuring anything.

> **US-14.** As a power user, I want to be able to disable revision history in the send settings, so that each save overwrites the previous snapshot and no history accumulates.

> **US-15.** As any user, when revision history is disabled, I want undo and redo controls to be hidden, so that the interface is not cluttered with non-functional controls.

### 6.5 Revision indicator

> **US-16.** As a careful composer, I want to see which revision number I am currently viewing (e.g. in the title bar or a status area), so that I know where I am in the history.

> **US-17.** As any user, I want a visual indicator (such as an asterisk or other marker) when the current content has been modified since the last save, so that I know whether my latest changes have been captured.

### 6.6 Server-side draft synchronization

> **US-18.** As a multi-device user, I want the application to synchronize my draft to the server-side Drafts folder after each save, so that I can continue editing on another device.

> **US-19.** As any user, I want to be able to disable server-side draft synchronization via a toggle in the compose window, so that I can keep a draft local-only when I choose.

> **US-20.** As any user, I want the default for server-side draft synchronization to be enabled, so that drafts are backed up to the server without extra configuration.

### 6.7 Silent operation

> **US-21.** As a fast typist, I want auto-saves to happen silently in the background without showing a progress spinner, blocking my input, or causing visible lag, so that the saving process is imperceptible during normal editing.

> **US-22.** As any user, I want auto-saves triggered by editing events (paragraph break, punctuation) to avoid triggering encryption dialogs or other interactive prompts, so that the auto-save does not interrupt my flow. Encryption should only be applied at final send time or on an explicit manual save.

### 6.8 Cleanup

> **US-23.** As any user, I want revision history files to be cleaned up automatically when the draft is sent or discarded, so that old snapshots do not accumulate and waste disk space indefinitely.

> **US-24.** As any user, I want a periodic background cleanup process to remove orphaned revision files (for drafts that no longer exist), so that storage remains tidy even after unexpected termination.

## 7. Functional Requirements

### 7.1 Save triggers

- **FR-1.** The application shall auto-save the draft body when the user inserts a newline character, provided the preceding character was not also a newline. (This captures single paragraph breaks but avoids redundant saves on consecutive blank lines.)
- **FR-2.** The application shall optionally auto-save the draft body when the user types an end character (period, full-width period, colon, semicolon, question mark, exclamation mark, or comma), provided the preceding character was not also an end character. (This avoids redundant saves on sequences like `...` or `?!`.)
- **FR-3.** The application shall auto-save the draft body whenever the compose window loses focus or is backgrounded, regardless of trigger settings.
- **FR-4.** Auto-save shall only occur when the draft body is in a dirty state (i.e. has been modified since the last save). If the body is unchanged, no save shall be performed.
- **FR-5.** An explicit user-initiated save action (e.g. a "Save" menu item or keyboard shortcut) shall also be available and shall always save, regardless of dirty state.

### 7.2 Save behavior

- **FR-6.** Auto-saves triggered by editing events (paragraph break, punctuation) shall be silent: no progress indicator, no blocking of user input, and no interactive prompts (such as encryption dialogs).
- **FR-7.** Auto-saves triggered by loss of focus shall also be silent but shall complete before the compose window fully relinquishes control, so that the saved state is durable even if the application is terminated immediately afterward.
- **FR-8.** Each save shall persist the full current state of the draft body to local storage.

### 7.3 Revision history

- **FR-9.** When revision history is enabled, each save shall create a new numbered revision snapshot of the draft body, incrementing the revision counter. The previous revision(s) shall be retained.
- **FR-10.** The first save of a new draft shall create revision 1.
- **FR-11.** When revision history is disabled, each save shall overwrite the single stored snapshot. No prior versions shall be retained, and the revision counter shall not advance.
- **FR-12.** Revision snapshots shall be stored locally and shall not be synchronized to the server. Only the current draft body is synchronized to the server-side Drafts folder.

### 7.4 Undo and redo

- **FR-13.** The compose window shall provide an undo control that loads the previous revision (current revision number minus one) and replaces the draft body with that revision's content.
- **FR-14.** The compose window shall provide a redo control that loads the next revision (current revision number plus one) and replaces the draft body with that revision's content.
- **FR-15.** The undo control shall be visible only when the current revision is greater than 1 (i.e. there exists a prior revision).
- **FR-16.** The redo control shall be visible only when the current revision is less than the total number of revisions (i.e. there exists a later revision).
- **FR-17.** Navigating to a different revision via undo or redo shall update the revision indicator and the visibility of the undo/redo controls immediately.
- **FR-18.** When revision history is disabled, the undo and redo controls shall be hidden.

### 7.5 Revision indicator

- **FR-19.** The compose window shall display the current revision number in a visible location (e.g. title bar or status area).
- **FR-20.** When the draft body has been modified since the last save (dirty state), the revision indicator shall show a distinguishing mark (e.g. an asterisk) alongside the revision number.

### 7.6 Server-side synchronization

- **FR-21.** When server-side draft saving is enabled, the application shall push the current draft body to the account's Drafts folder on the mail server after each save that changes content.
- **FR-22.** Server-side draft saving shall be independently toggleable per compose session via a control in the compose window.
- **FR-23.** Silent auto-saves (paragraph break, punctuation) shall update the server-side draft. However, if the draft requires encryption, the server push shall be deferred until the user explicitly triggers a send or manual save that includes encryption.
- **FR-24.** The server-side draft shall reflect the current (latest or undo-navigated) revision only — revision history itself is not synchronized.

### 7.7 Settings

- **FR-25.** The following settings shall be user-configurable in the send preferences:

| Setting | Default | Description |
|---|---|---|
| Auto-save on paragraph break | Enabled | Trigger auto-save when a newline is entered |
| Auto-save on punctuation | Disabled | Trigger auto-save when an end character is entered |
| Save revision history | Enabled | Keep numbered snapshots for undo/redo |
| Save drafts to server | Enabled | Synchronize draft to the IMAP Drafts folder |

- **FR-26.** Changes to these settings shall take effect immediately for all open and future compose sessions.

### 7.8 Cleanup

- **FR-27.** When a draft is sent or discarded (deleted), all local revision files associated with that draft shall be eligible for cleanup.
- **FR-28.** A periodic background cleanup process shall remove orphaned revision files — those whose associated draft no longer exists or whose content has been purged.
- **FR-29.** Revision files shall not be retained indefinitely. The cleanup process shall remove stale files after a reasonable retention period.

## 8. Non-Functional Requirements

- **NFR-1. Latency.** An auto-save triggered by a paragraph break or punctuation character shall complete its local persistence within a timeframe imperceptible to the user (well under one second) so that typing is never visibly interrupted.
- **NFR-2. Concurrency.** Auto-save shall not block the compose editor. The user shall be able to continue typing while the save is being persisted in the background.
- **NFR-3. Durability.** A completed auto-save shall survive application crash, unexpected termination, and system reboot. If the application is killed immediately after a loss-of-focus save, the draft shall be recoverable on next launch.
- **NFR-4. Scale.** The revision history mechanism shall handle drafts with dozens of revisions without noticeable performance degradation in undo/redo navigation.
- **NFR-5. Storage.** Revision snapshots shall be stored efficiently. The cleanup process shall prevent unbounded growth of revision storage.
- **NFR-6. Consistency.** After an undo or redo operation, the displayed draft body shall exactly match the content that was saved at that revision — no partial state, no data corruption.
- **NFR-7. Accessibility.** Undo and redo controls shall be reachable via keyboard shortcut as well as pointer interaction, with appropriate labels for screen readers.
- **NFR-8. Privacy.** Revision snapshots are local-only and shall never be transmitted to any server. Only the current draft body is synchronized to the mail server when server-side draft saving is enabled.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|
| AC-1 | With auto-save on paragraph break enabled (default), typing a paragraph of text and pressing Enter causes the draft to be saved. Killing the application immediately afterward and restarting recovers the draft including the paragraph just completed. |
| AC-2 | With auto-save on punctuation enabled, typing a sentence and ending it with a period causes the draft to be saved. The save is silent — no spinner or dialog appears. |
| AC-3 | Typing multiple consecutive newlines does not trigger multiple redundant saves — only the first newline after non-newline content triggers a save. |
| AC-4 | Typing multiple consecutive end characters (e.g. `...` or `?!`) does not trigger multiple redundant saves — only the first end character after a non-end character triggers a save. |
| AC-5 | Switching away from the compose window (e.g. to another application window) triggers an auto-save of the current draft, even if no paragraph break or punctuation was entered. |
| AC-6 | With revision history enabled (default), after making three edits separated by paragraph breaks, the undo control steps back through all three saved states, and the redo control steps forward again. |
| AC-7 | The undo control is hidden when viewing revision 1 (the initial state). The redo control is hidden when viewing the latest revision. |
| AC-8 | The compose window displays the current revision number, and shows a dirty indicator when the body has been modified since the last save. |
| AC-9 | With revision history disabled, undo and redo controls are not visible, and each auto-save overwrites the previous saved state. |
| AC-10 | With server-side draft saving enabled (default), a draft that has been auto-saved locally also appears in the account's Drafts folder on the server. |
| AC-11 | Toggling server-side draft saving off in the compose window prevents subsequent saves from pushing to the server, but local auto-saves continue. |
| AC-12 | Disabling auto-save on paragraph break in settings prevents newlines from triggering saves, but loss-of-focus saves still occur. |
| AC-13 | After a draft is sent, its local revision files are cleaned up (either immediately or by the next periodic cleanup cycle). |
| AC-14 | After a draft is discarded (deleted), its local revision files are cleaned up. |
| AC-15 | Auto-saves that occur while composing an encrypted message do not trigger encryption prompts or dialogs. Encryption is deferred to the explicit send action. |
| AC-16 | All four settings (paragraph auto-save, punctuation auto-save, revision history, server draft saving) are independently configurable and persist across application restarts. |

## 10. Open Questions

- **OQ-1. Timer-based auto-save.** The high-level feature description mentions drafts saved "on a timer," but the source application does not implement a periodic timer-based auto-save. Saves are triggered only by editing events (paragraph break, punctuation) and lifecycle transitions (loss of focus). Should a timer-based fallback be added for the desktop application (e.g. save every N seconds if no other trigger has fired), or is the current trigger set considered sufficient?
- **OQ-2. Revision branching.** If the user undoes to revision 3 (out of 5) and then makes a new edit, should the new edit create revision 6 (appending to the linear history) or revision 4 (discarding revisions 4–5 and branching)? The source application appends to the linear history — the old revisions 4–5 remain accessible via redo. This should be confirmed as the desired behavior, as it differs from the "discard forward history on edit" convention used by many undo systems.
- **OQ-3. Revision storage limits.** There is no cap on the number of revisions a single draft can accumulate. For a long editing session with frequent paragraph breaks, this could result in dozens or hundreds of revision files. Should a maximum revision count or a total storage budget be introduced?
- **OQ-4. Quoted text in revisions.** The source application stores quoted/reply text separately from the draft body. Revisions capture only the user-composed body, not the quoted text. This means that undo/redo does not restore changes to quoted text (e.g. if the user trims the quoted block). Should revision snapshots include quoted text as well?
- **OQ-5. Signature in revisions.** Signatures are applied dynamically and are not stored in revision snapshots. Toggling the signature on or off triggers a save, but the revision captures the body without the signature. This means undoing past a signature toggle does not restore the previous signature state. Is this acceptable, or should the revision capture the full rendered content including signature?
- **OQ-6. Character-level undo interaction.** The source application offers an optional character-level undo/redo mechanism in the text editor (disabled by default), separate from the revision-level undo/redo. The interaction between these two undo systems (character-level vs. revision-level) may be confusing to users. Should they be unified, or should they remain independent?
- **OQ-7. Server sync frequency.** Currently, every local auto-save that changes content also triggers a server-side draft sync (when enabled). For users typing quickly, this could result in many server round-trips. Should server syncs be debounced or rate-limited relative to local saves?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Event-driven, not timer-driven.** The source application deliberately chose editing-event triggers (paragraph break, punctuation, loss of focus) over a periodic timer. This means saves align with natural pauses in writing rather than interrupting at arbitrary intervals. The tradeoff is that a user who types a very long paragraph without pressing Enter or using sentence-ending punctuation will not have an intermediate save until they pause or switch away. The loss-of-focus save acts as the safety net for this case.
- **N-2. Paragraph break is on by default; punctuation is off.** This reflects a judgment that paragraph breaks are a universal, low-frequency save trigger that all users benefit from, while punctuation-triggered saves are higher frequency and suit only users who want very aggressive saving. Keeping punctuation off by default avoids creating an excessive number of revisions for typical users.
- **N-3. Consecutive-character deduplication.** Both triggers include a guard that the *previous* character must not be the same type (newline-after-newline, end-char-after-end-char). This prevents rapid-fire saves when the user types `...`, `?!`, or inserts multiple blank lines. The granularity is deliberately coarse — one save per transition from non-trigger to trigger character.
- **N-4. Revision history is append-only.** Revisions form a simple linear sequence. Undoing does not discard later revisions — they remain accessible via redo. This is simpler than a branching undo tree and matches the expectation that "undo" is reversible. The cost is that the revision count only grows, never shrinks (until cleanup).
- **N-5. Revisions capture body only.** Quoted text and signatures are excluded from revision snapshots. This keeps revisions small and focused on the user's original composition. The tradeoff is that undo/redo does not restore changes to quoted text or signature state.
- **N-6. Silent auto-save skips encryption.** When a message is being composed with encryption enabled, auto-saves persist the plaintext body locally but do not trigger the encryption workflow. Encryption is applied only at explicit send time or manual save. This prevents encryption dialogs from interrupting the user's typing flow.
- **N-7. Server sync is coupled to local save.** Each content-changing local save also pushes to the server (when enabled). This keeps the server draft closely in sync with the local state. If this proves too chatty for some server environments, debouncing may be needed (see OQ-7).
