# Epic: Two-Way IMAP Synchronization

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **2.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Two-way IMAP synchronization. Local actions (move, flag, delete, mark read) propagate to the server and vice versa."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email client that only downloads messages but never pushes changes back to the server is fundamentally broken for modern multi-device workflows. A user who reads, flags, moves, or deletes a message on one device expects that change to appear everywhere — on the server, on their webmail, and on every other client connected to the same account. Conversely, when another client (phone, webmail, shared device) makes a change on the server, the desktop client must pick it up without the user having to intervene.

Two-way IMAP synchronization is the foundational mechanism that keeps the local mailbox and the remote server in a consistent, converged state. It ensures that every user action taken locally is propagated to the server, and every change observed on the server is reflected locally — continuously, reliably, and without data loss.

This feature underpins virtually all other mailbox operations. Without it, move, delete, flag, read/unread, keywords, and folder management would be purely local illusions.

## 2. Goals

- **G1.** Propagate every user-initiated local action (move, copy, delete, flag, mark read/unread, set keyword, mark answered) to the IMAP server reliably and in order.
- **G2.** Detect and incorporate server-side changes (new messages, removed messages, flag changes, keyword changes, folder structure changes) into the local state promptly.
- **G3.** Resolve conflicts between local pending actions and incoming server state without data loss or user confusion.
- **G4.** Operate gracefully offline: queue all local actions and replay them when connectivity returns, without user intervention.
- **G5.** Minimize unnecessary network traffic through incremental synchronization — only fetch what has actually changed.
- **G6.** Support the full range of IMAP server implementations, gracefully degrading when optional capabilities are absent.
- **G7.** Provide the user with visibility into pending operations and their status (queued, executing, failed).

## 3. Non-Goals

- **NG1.** This epic does **not** define the user interface for reading, composing, or organizing messages — only the synchronization contracts that keep local and remote state consistent.
- **NG2.** This epic does **not** cover POP3 synchronization, which is inherently one-way (download only) with limited local emulation of folder operations.
- **NG3.** This epic does **not** cover JMAP synchronization, which uses a fundamentally different state-change protocol.
- **NG4.** This epic does **not** define push notification behavior (IMAP IDLE / NOTIFY); those are covered under connectivity (feature 7.1). This epic assumes that *some* mechanism triggers a sync cycle and defines what happens during that cycle.
- **NG5.** This epic does **not** specify scheduling policies (poll intervals, work-hours restrictions, network-type awareness). It defines what a sync cycle does, not when or how often it runs.
- **NG6.** This epic does **not** define cross-account operations (moving a message between two different IMAP accounts), which involve download-then-upload semantics rather than pure IMAP synchronization.

## 4. Glossary

| Term | Definition |
|---|---|
| **Local action** | Any user-initiated change to a message or folder performed in the client UI (mark read, flag, move, delete, set keyword, etc.). |
| **Server state** | The authoritative state of messages and folders as held on the IMAP server. |
| **Operation** | A discrete unit of work representing a local action that must be propagated to the server (e.g. "set Seen flag on message X"). |
| **Operation queue** | An ordered, persistent list of operations awaiting execution against the server. |
| **Sync cycle** | A single pass of the synchronization engine over a folder, during which server changes are fetched and pending operations are executed. |
| **Incremental sync** | A sync cycle that fetches only changes since the last sync, rather than re-downloading the full folder state. |
| **Full sync** | A sync cycle that re-evaluates the entire folder state within the configured sync window, used when incremental sync is not possible or when forced by the user. |
| **Sync window** | The configurable number of days of messages that the application actively monitors for changes. Messages older than this window may still be kept locally but are not checked on every cycle. |
| **Keep window** | The configurable number of days after which locally-cached messages are removed from the device to reclaim storage. |
| **UID validity** | A server-assigned value that, if it changes, indicates that all previously known message UIDs are no longer valid and a full resync is required. |
| **Modification sequence (MODSEQ)** | A per-message version number provided by servers that support conditional operations; enables the client to ask "what changed since sequence N?" |
| **Permanent flags** | The set of flags that the server guarantees it will persist across sessions (e.g. Seen, Flagged, Answered, Deleted, and optionally user-defined keywords). |
| **Expunge** | The IMAP operation that permanently removes messages marked with the Deleted flag from the server. |
| **Conflict** | A situation where a local pending operation targets a message whose server state has changed since the operation was queued. |

## 5. Personas

- **P1. The multi-device user.** Uses the desktop client, a phone, and webmail interchangeably. Expects all three to show the same mailbox state within seconds of a change on any device.
- **P2. The offline worker.** Triages email on a laptop during commutes or flights with no connectivity. Expects all their read/flag/move/delete actions to be applied seamlessly when they reconnect.
- **P3. The power user with rules.** Has server-side rules (Sieve) that move, flag, or delete messages. Expects those server-side changes to appear locally without manual intervention.
- **P4. The shared-mailbox user.** Shares a mailbox with colleagues. Expects that when a colleague marks a message read or moves it, the change is reflected in their own view.
- **P5. The cautious user.** Worries about accidentally losing email. Expects that delete means "move to Trash" unless they explicitly choose permanent deletion, and that mistakes can be undone before the next sync.

## 6. User Stories

### 6.1 Local-to-server propagation

> **US-1.** As a multi-device user, when I mark a message as read in the desktop client, I want that message to appear as read on the server and on all my other devices, so that I do not have to re-read messages I have already seen.

> **US-2.** As any user, when I flag/star a message, I want the flag to persist on the server, so that it appears starred in webmail and on my phone.

> **US-3.** As any user, when I move a message from my Inbox to a folder, I want the message to appear in the destination folder on the server and disappear from the Inbox on the server, so that the move is real — not just a local illusion.

> **US-4.** As any user, when I delete a message (move to Trash), I want it to appear in the server's Trash folder and disappear from its original location on the server.

> **US-5.** As any user, when I permanently delete a message from Trash, I want it to be irrecoverably removed from the server.

> **US-6.** As any user, when I mark a message as answered or forwarded, I want that flag to persist on the server so other clients can display the replied/forwarded indicator.

> **US-7.** As a power user, when I assign a custom keyword or label to a message, I want that keyword to be stored on the server (if the server supports user-defined flags), so that it is visible from other clients that understand keywords.

> **US-8.** As any user, when I copy a message to another folder, I want the copy to appear on the server in the destination folder while the original remains in place.

### 6.2 Server-to-local propagation

> **US-9.** As a multi-device user, when I read a message on my phone, I want the desktop client to show that message as read on its next sync cycle, without me doing anything.

> **US-10.** As any user, when a new message arrives on the server, I want it to appear in my local folder automatically.

> **US-11.** As any user, when a message is moved or deleted on the server (by another client, by a server-side rule, or by webmail), I want the desktop client to reflect that change — the message should disappear from the old folder and appear in the new one (or disappear entirely if deleted).

> **US-12.** As a power user, when a server-side rule adds a keyword or flag to a message, I want that keyword/flag to appear locally on the next sync.

> **US-13.** As a shared-mailbox user, when a colleague changes a message's flags or moves it, I want to see those changes reflected in my local view.

### 6.3 Offline operation

> **US-14.** As an offline worker, when I have no network connectivity, I want to be able to mark messages as read, flag them, move them, and delete them, with those actions applied to my local view immediately.

> **US-15.** As an offline worker, when connectivity returns, I want all my queued actions to be executed against the server automatically, in the order I performed them, without me having to remember or re-do anything.

> **US-16.** As an offline worker, I want to see a visual indication that operations are pending (not yet confirmed by the server), so that I know my actions have not yet propagated.

### 6.4 Conflict handling

> **US-17.** As a multi-device user, if I flag a message on the desktop while my phone simultaneously unflags it, I want the application to resolve this deterministically without losing data or entering an inconsistent state.

> **US-18.** As any user, if a message I have queued a move for is deleted on the server by another client before my move executes, I want the application to handle this gracefully — not crash, not leave ghost entries, and ideally inform me that the operation could not complete.

### 6.5 Error visibility and recovery

> **US-19.** As any user, I want to see a list of pending operations and their status (queued, in progress, failed), so that I can understand what the application is doing with my mailbox.

> **US-20.** As any user, when an operation fails (e.g. the server rejects a move because the destination folder no longer exists), I want to be informed clearly about what failed and why.

> **US-21.** As any user, I want failed operations to be retried automatically a reasonable number of times before giving up, so that transient network errors do not require my intervention.

> **US-22.** As any user, I want the ability to manually retry or cancel a failed operation, so that I am not stuck with a permanently-failing queue entry.

### 6.6 Incremental and efficient sync

> **US-23.** As any user, I want synchronization to be fast — fetching only what has changed rather than re-downloading my entire mailbox on every cycle.

> **US-24.** As any user, I want the ability to force a full resynchronization of a folder when I suspect the local state has drifted, without having to remove and re-add the account.

> **US-25.** As any user with a large mailbox, I want to configure how far back the application synchronizes (sync window), so that I can balance completeness against storage and bandwidth.

### 6.7 UID validity and mailbox reconstruction

> **US-26.** As any user, if my mail server reconstructs a mailbox (invalidating all previously known message identifiers), I want the application to detect this and perform a full resync automatically, so that I do not end up with stale or mismatched messages.

### 6.8 Expunge control

> **US-27.** As a cautious user, I want "delete" to mean "move to Trash" by default, not permanent removal, so that I have a safety net.

> **US-28.** As a power user, I want the option to permanently expunge deleted messages immediately (or on a schedule), so that my Trash does not accumulate indefinitely.

> **US-29.** As any user, I want expunging to be precise — only the messages I chose to delete should be removed, not unrelated messages that happen to carry a Deleted flag from another client.

## 7. Functional Requirements

### 7.1 Operations propagated to the server

- **FR-1.** The application shall propagate the following local actions to the IMAP server: mark seen/unseen, flag/unflag, mark answered, set/remove keywords, move to folder, copy to folder, delete (move to Trash), permanently delete (expunge), and append (save draft, file sent message).
- **FR-2.** Each propagated action shall be represented as a discrete, persistent operation in a queue that survives application restart.
- **FR-3.** Operations shall be executed in a deterministic order that preserves user intent (e.g. a move followed by a flag on the moved message must execute in that order).
- **FR-4.** Where multiple operations of the same type target different messages in the same folder, the application shall batch them into a single server command where the protocol allows, to reduce round-trips.

### 7.2 Server-to-local synchronization

- **FR-5.** On each sync cycle for a folder, the application shall detect: new messages added to the folder, messages removed from the folder, and changes to message flags/keywords.
- **FR-6.** New messages detected on the server shall be added to the local store with their full envelope, flags, and (depending on configuration) body and attachments.
- **FR-7.** Messages removed from the server shall be removed from the local store (or marked as server-deleted, depending on retention policy).
- **FR-8.** Flag and keyword changes detected on the server shall be applied to the local message state, subject to conflict resolution (see §7.4).
- **FR-9.** If the server reports that UID validity has changed for a folder, the application shall discard all local UID mappings for that folder and perform a full resynchronization.

### 7.3 Incremental synchronization

- **FR-10.** When the server supports modification-sequence tracking, the application shall use it to request only messages that have changed since the last known sequence, avoiding a full folder scan.
- **FR-11.** When the server does not support modification-sequence tracking, the application shall fall back to comparing the full message list within the sync window against the local state.
- **FR-12.** A sync cycle triggered within a short interval of a previous successful sync (e.g. within 30 seconds) shall perform a full comparison rather than relying on incremental state, to catch changes made by other clients during the brief window.
- **FR-13.** The user shall be able to force a full resynchronization of any folder at any time.

### 7.4 Conflict resolution

- **FR-14.** When a server-side flag/keyword change is detected for a message that has a pending local operation of the same type (e.g. both local and server changed the Seen flag), the local pending operation shall take precedence — the server change shall not overwrite the user's intent.
- **FR-15.** When a server-side flag/keyword change is detected for a message that has **no** pending local operation of the same type, the server state shall be accepted as authoritative.
- **FR-16.** When a pending operation targets a message that no longer exists on the server (e.g. deleted by another client), the operation shall be cancelled gracefully, the local message removed, and the user optionally informed.
- **FR-17.** Conflict resolution shall never result in data loss. In ambiguous cases, the application shall prefer to keep the message rather than delete it.

### 7.5 Offline operation and queue replay

- **FR-18.** When the application has no network connectivity, all user-initiated actions shall be applied to the local state immediately and queued as operations for later execution.
- **FR-19.** The operation queue shall persist across application restarts and system reboots.
- **FR-20.** When connectivity is restored, the application shall automatically begin processing the operation queue without user intervention.
- **FR-21.** Messages with pending (unconfirmed) operations shall be visually distinguishable from messages whose state is confirmed by the server (e.g. dimmed, italicized, or badged).

### 7.6 Move operations

- **FR-22.** When moving a message, the application shall use the server's atomic move capability if available, which moves the message and removes it from the source folder in a single operation.
- **FR-23.** When the server does not support atomic move, the application shall fall back to copy-to-destination followed by delete-from-source, ensuring the copy succeeds before the delete is issued.
- **FR-24.** After a move or copy, the application shall locate the message in the destination folder (by message identifier) and update the local UID mapping, so that subsequent operations on the moved message target the correct server object.
- **FR-25.** When moving a message to the Junk/Spam folder, the application shall set appropriate junk-indicator keywords (if supported by the server) and remove any not-junk keywords. The reverse shall apply when moving out of Junk.

### 7.7 Delete and expunge

- **FR-26.** By default, "delete" shall mean "move to the account's configured Trash folder." The message shall remain recoverable from Trash until explicitly expunged or auto-cleaned.
- **FR-27.** "Permanently delete" (from Trash or via explicit action) shall set the Deleted flag and issue an expunge command to the server.
- **FR-28.** When the server supports per-UID expunge, the application shall use it to expunge only the targeted messages, not all Deleted-flagged messages in the folder.
- **FR-29.** When the server does not support per-UID expunge, the application shall either use a folder-wide expunge (with appropriate user warning) or defer expunge to a batch operation.
- **FR-30.** The user shall have a configurable option to control whether expunge happens immediately on permanent delete, or is deferred to a manual or scheduled batch operation.

### 7.8 Flag and keyword synchronization

- **FR-31.** The application shall synchronize all permanent flags supported by the server (minimally: Seen, Flagged, Answered, Deleted, Draft).
- **FR-32.** The application shall synchronize user-defined keywords/labels if the server advertises support for user flags.
- **FR-33.** If a folder is read-only (server reports it as such), the application shall not attempt to write flag changes back to the server for that folder.
- **FR-34.** If the server does not persist a particular flag (not in the permanent-flags set), the application shall store it locally only and not attempt to sync it.
- **FR-35.** When multiple instances of the same message exist across folders (as with label-based systems), a flag change on one instance shall be propagated to all instances on the server.

### 7.9 Operation retry and error handling

- **FR-36.** Transient errors (network timeout, temporary server unavailability) shall trigger automatic retry with exponential backoff, up to a configurable maximum number of attempts.
- **FR-37.** Permanent errors (folder does not exist, message not found, quota exceeded, permission denied) shall cause the operation to be marked as failed and removed from the queue after notifying the user.
- **FR-38.** The application shall distinguish between transient and permanent errors based on the server's response codes.
- **FR-39.** If a batched operation fails, the application shall retry individual operations within the batch separately, to avoid one bad message blocking the entire batch.
- **FR-40.** The user shall be able to view the operation queue, see the status and error message of each entry, and manually retry or dismiss failed operations.

### 7.10 Sync window and retention

- **FR-41.** The user shall be able to configure, per folder, how many days of messages the application actively synchronizes (sync window).
- **FR-42.** The user shall be able to configure, per folder, how many days of messages are retained locally (keep window). The keep window must be greater than or equal to the sync window.
- **FR-43.** Messages outside the sync window shall not be checked for flag changes on routine sync cycles, but shall remain accessible locally if within the keep window.
- **FR-44.** Messages outside the keep window shall be removed from local storage. This removal is local only and shall not delete messages from the server.

### 7.11 Provider compatibility

- **FR-45.** The application shall detect server capabilities at connection time and adapt its behavior accordingly (e.g. presence or absence of atomic move, per-UID expunge, modification sequences, user flags).
- **FR-46.** The application shall handle known provider-specific quirks (e.g. providers that advertise a capability but implement it incorrectly) by maintaining a compatibility profile that can disable specific optimizations per provider.
- **FR-47.** The application shall not require any optional IMAP extension to function. All extensions (atomic move, modification sequences, per-UID expunge) shall be used opportunistically when available and gracefully skipped when absent.

## 8. Non-Functional Requirements

- **NFR-1. Consistency.** After a sync cycle completes without errors, the local state of all flags and folder membership for messages within the sync window shall match the server state, except where a pending local operation intentionally overrides it.
- **NFR-2. Durability.** No user action shall be lost due to application crash, forced quit, or system reboot. The operation queue shall be persisted to stable storage before the user-facing state is updated.
- **NFR-3. Ordering.** Operations targeting the same message shall be executed in the order the user performed them. Operations targeting different messages may be reordered or parallelized for efficiency.
- **NFR-4. Latency.** Under normal connectivity, a local action should be propagated to the server within seconds (not minutes). The user should not have to wait for a scheduled poll cycle.
- **NFR-5. Bandwidth efficiency.** Incremental sync shall transfer only changed metadata (flags, UIDs) for already-known messages. Full message bodies shall only be re-downloaded if the local copy is missing or corrupted.
- **NFR-6. Scale.** The synchronization engine shall handle folders containing at least 100,000 messages and operation queues of at least 1,000 pending entries without degradation.
- **NFR-7. Resilience.** A single failed operation shall not block the processing of unrelated operations in the queue. The queue shall be partitioned such that independent operations proceed in parallel where possible.
- **NFR-8. Observability.** The user shall have access to a log of recent synchronization activity, including errors, retries, and completion times, to aid in diagnosing connectivity or server issues.
- **NFR-9. Idempotency.** Replaying an operation that has already been applied (e.g. due to ambiguous server response) shall not produce duplicate effects (e.g. two copies of a moved message).
- **NFR-10. Privacy.** The synchronization engine shall not transmit any data to any server other than the user's configured mail server. No telemetry, no proxies, no third-party services.

## 10. Open Questions

- **OQ-1. Draft synchronization semantics.** Drafts are mutable messages that change content (not just flags) over time. Should draft updates be treated as "delete old + append new" (which is how IMAP handles it), or should the application maintain a local revision history with periodic server-side replacement? The source application appears to use delete-and-re-append, but the optimal user-facing contract is unclear.
- **OQ-2. Provider-specific move quirks.** At least one major provider is known to advertise atomic move support but implement it incorrectly. Should the compatibility profile be user-editable, or purely maintained by the application developers? The source application hard-codes provider exceptions.
- **OQ-3. Expunge timing for non-UID-EXPUNGE servers.** When the server does not support per-UID expunge, a folder-wide expunge could remove messages flagged as Deleted by other clients. Should the application warn the user, defer expunge entirely, or offer a "careful mode" that avoids folder-wide expunge? The source application appears to support both immediate and deferred expunge.
- **OQ-4. Keyword/label sync for label-based providers.** Providers that use labels instead of folders (where a message can appear in multiple "folders" simultaneously) have different move/delete semantics. Should the application expose label management as a first-class concept, or map it transparently onto folder semantics? The source application handles this as a special case with dedicated label operations.
- **OQ-5. Operation queue size limits.** Should there be a cap on how many operations can be queued offline? If a user goes offline for a week and performs thousands of actions, is there a point at which the queue becomes impractical to replay? No explicit limit was found in the source.
- **OQ-6. Sync window interaction with search.** If a message is outside the sync window but matches a search query, should its flags be considered potentially stale? Should search results from outside the sync window trigger a targeted flag refresh?
- **OQ-7. Handling of vanished messages during batch operations.** When a batch of flag-set operations is sent and some messages in the batch have been expunged by another client mid-operation, should the entire batch fail and retry individually, or should partial success be accepted? The source application retries individually on batch failure.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Dual-state fields for conflict resolution.** The source application maintains two parallel states for key flags (e.g. "confirmed server state" and "user-intended state"). When a sync cycle runs, it compares the server state against the confirmed state — if they differ and no pending operation exists, the server wins. If a pending operation exists, the user-intended state wins because it will be propagated on the next operation execution. This two-field approach is the simplest correct conflict resolution for disconnected operation without requiring vector clocks or CRDTs.

- **N-2. Operation priority ordering.** Not all operations are equally urgent. Content-fetching operations (download body, fetch headers) are prioritized over state-change operations (set flag, move) because the user is likely waiting to read a message. Within state-change operations, destructive operations (delete, move) are prioritized over non-destructive ones (flag, seen) because they affect what the user sees in their folder list. This priority scheme reduces perceived latency.

- **N-3. Batching similar operations.** When multiple messages need the same flag change (e.g. "mark 50 messages read"), sending 50 individual commands is wasteful. The source application batches these into chunked commands (e.g. groups of 20 UIDs per command). The chunk size is a trade-off: larger chunks are more efficient but risk larger failures if one message in the batch has been expunged.

- **N-4. Move is not just copy+delete.** Atomic move (when available) is preferred because it is faster, generates less server-side I/O, preserves the message's internal date and flags without explicit re-application, and avoids a window where the message exists in both folders simultaneously. The copy+delete fallback is only used when the server lacks the capability or when known provider bugs make the atomic operation unsafe.

- **N-5. Per-UID expunge is critical for safety.** Without per-UID expunge, a folder-wide expunge command removes *all* messages carrying the Deleted flag — including ones deleted by other clients that the user has not seen. The source application strongly prefers per-UID expunge and treats its absence as a reason to defer expunge rather than risk collateral deletion.

- **N-6. UID validity as a reset signal.** UID validity is the IMAP server's way of saying "forget everything you knew about this folder." The source application treats this as an unconditional trigger for full resynchronization. There is no attempt to heuristically match old and new UIDs — the risk of misidentification is too high.

- **N-7. Modification sequences as an optimization, not a requirement.** The application works correctly without modification-sequence support (by doing date-window-based full comparisons), but with it, routine sync cycles become dramatically cheaper. This is why it is detected and used opportunistically rather than required.

- **N-8. Local-first, server-authoritative.** The application applies changes locally first (for immediate UI responsiveness) but treats the server as the ultimate source of truth. If an operation fails permanently (e.g. server rejects it), the local state is rolled back to match the server. This gives the user instant feedback while maintaining long-term consistency.
