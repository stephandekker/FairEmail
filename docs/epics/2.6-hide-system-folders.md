# Epic: Hide System Folders

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **2.6** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Hide system folders. Choose which folders appear in the folder list."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[SHOULD]` — important, planned for v1 but acceptable to slip to v1.x if pressed.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

A mail account's folder tree typically contains many folders that serve infrastructure roles — Sent, Drafts, Trash, Spam, Archive — as well as an unbounded number of user-created folders. Not all of these are useful in every context. A user triaging their Inbox rarely needs to see Trash or Spam in the folder list, and a user with deeply nested server-side folders may want to reduce visual clutter to just the handful of folders they actively work with.

The "Hide System Folders" feature gives the user control over which folders appear in the main folder list and the navigation panel. It provides three complementary mechanisms:

1. **Explicit hide** — a per-folder toggle that removes a folder from the folder list entirely (the folder remains on the server and continues to synchronize; it simply becomes invisible in the UI).
2. **Conditional hide (hide when read)** — a per-folder toggle that automatically hides a folder when it contains no unread messages, and reveals it again when new unread mail arrives.
3. **Navigation panel pinning** — a per-folder toggle that promotes a folder to the application's primary navigation panel for quick access, independently of whether it is hidden or shown in the full folder list.

Together these mechanisms let the user sculpt a minimal, relevant folder view without losing access to any underlying data. A "show hidden folders" toggle in the folder list's menu allows the user to temporarily reveal everything, inspect or change hidden folders, and return to their curated view.

## 2. Goals

- **G1.** Allow the user to hide any folder — system or user-created — from the main folder list, without deleting it, unsubscribing from it, or stopping synchronization.
- **G2.** Allow the user to auto-hide folders that have no unread messages, so that the folder list dynamically reflects where attention is needed.
- **G3.** Allow the user to pin any folder to the navigation panel for one-click access, independently of hide state.
- **G4.** Provide a discoverable way to reveal all hidden folders temporarily, so the user can always find and manage folders they previously hid.
- **G5.** Ensure that hiding a folder has no side effects on synchronization, notifications, unified-inbox membership, or any other folder property — it is purely a display concern.

## 3. Non-Goals

- **NG1.** This epic does **not** define folder subscription management (subscribing/unsubscribing IMAP folders on the server). Hiding is a local display preference; subscription is a server-side state. They are independent.
- **NG2.** This epic does **not** define folder creation, renaming, or deletion. Those are separate folder-management operations.
- **NG3.** This epic does **not** define the unified-inbox membership toggle. A folder's unified-inbox membership and its hidden state are independent properties (see §11, N-1).
- **NG4.** This epic does **not** define folder synchronization controls (sync on/off, poll interval, days to keep). Hiding a folder does not affect sync.
- **NG5.** Account-level "hide all folders" or "show all folders" bulk toggles are not a goal. The governing granularity is per-folder, with a convenience bulk action for child folders of a parent (see §7.4).

## 4. Glossary

| Term | Definition |
|---|---|
| **Folder** | A container of messages within an account. May be a system folder or a user-created folder. |
| **System folder** | A folder with a recognized role: Inbox, Sent, Drafts, Trash, Spam, Archive, Outbox. |
| **User folder** | Any folder that is not a system folder — created by the user or by server-side rules. |
| **Hidden folder** | A folder whose explicit-hide flag is enabled. It does not appear in the folder list unless the user activates "show hidden folders". |
| **Conditionally hidden folder** | A folder whose hide-when-read flag is enabled. It disappears from the folder list when it has zero unread messages (including in child folders) and reappears when unread messages arrive. |
| **Navigation panel** | The application's primary side panel providing quick access to accounts, folders, and top-level views (Unified Inbox, etc.). |
| **Pinned folder** | A folder whose navigation flag is enabled, causing it to appear in the navigation panel for quick access. |
| **Folder list** | The full list of folders displayed when viewing a specific account. Distinct from the navigation panel. |

## 5. Personas

- **P1. The minimalist.** Has one or two accounts; hides Trash, Spam, and other noise folders so the folder list shows only Inbox and a few active project folders. Wants a clean, short folder list.
- **P2. The attention-driven user.** Has several accounts with many folders. Uses hide-when-read so that folders only surface when they demand attention — new unread mail makes a folder visible; reading all messages makes it disappear again.
- **P3. The power navigator.** Has a complex folder hierarchy but wants three or four key folders always one click away in the navigation panel, regardless of where they sit in the folder tree.
- **P4. The occasional auditor.** Normally runs with a curated, minimal folder list but periodically needs to see everything — e.g. to check an old archive folder or to verify that a rule is filing mail into the right place.

## 6. User Stories

### 6.1 Hiding a folder

> **US-1.** As any user, I want to hide a folder from the folder list via a context action (e.g. right-click or long-press equivalent), so that I can remove clutter without leaving the folder list.

> **US-2.** As any user, I want to hide a folder from the folder's properties/edit screen, so that I have a discoverable settings-based way to control visibility.

> **US-3.** As any user, I want to hide any folder regardless of its role — system folders (Sent, Drafts, Trash, Spam, Archive) and user-created folders alike — so that I am not forced to see folders I never use.

> **US-4.** As any user, when I hide a folder, I want it to disappear from the folder list immediately, so that my change takes effect without delay.

> **US-5.** As any user, when I hide a folder, I want all of its other properties (synchronization, notifications, unified-inbox membership, unread counts) to remain unchanged, so that hiding is purely cosmetic and non-destructive.

### 6.2 Showing hidden folders

> **US-6.** As any user, I want a menu option in the folder list to "show hidden folders", so that I can temporarily reveal all folders I have hidden.

> **US-7.** As any user, when "show hidden folders" is active, I want hidden folders to appear visually distinct from normal folders (e.g. dimmed or with reduced opacity), so that I can tell at a glance which folders are hidden.

> **US-8.** As any user, I want to toggle "show hidden folders" off to return to my curated view, so that revealing hidden folders is temporary and reversible.

### 6.3 Conditional hiding (hide when read)

> **US-9.** As an attention-driven user, I want to set a folder to "hide when all messages are read", so that it only appears in the folder list when it contains unread messages.

> **US-10.** As any user, when a conditionally-hidden folder has zero unread messages (including in its child folders), I want it to disappear from the folder list automatically, without manual intervention.

> **US-11.** As any user, when new unread messages arrive in a conditionally-hidden folder, I want it to reappear in the folder list automatically, so that I notice the new mail.

> **US-12.** As any user, I want the "show hidden folders" toggle to also reveal conditionally-hidden folders that are currently hidden due to having no unread messages, so that I can always find every folder.

> **US-13.** As any user, I want the hide-when-read behavior to apply in the account folder list and the unified folder list, but not to suppress folders in other contexts where I am actively selecting a folder (e.g. move-to dialogs), so that I can still target a hidden folder when I need to.

### 6.4 Navigation panel pinning

> **US-14.** As any user, I want to add a folder to the navigation panel via a context action or from the folder's properties screen, so that I can access it with one click from anywhere in the application.

> **US-15.** As any user, I want to remove a folder from the navigation panel the same way, so that pinning is easily reversible.

> **US-16.** As any user, I want the navigation panel pin to be independent of the hide flag — I can pin a folder to the navigation panel *and* hide it from the main folder list, or show it in the folder list without pinning it to the navigation panel.

### 6.5 Bulk operations on child folders

> **US-17.** As a power user with hierarchical folders, when I act on a parent folder I want to be able to add or remove all of its child folders from the navigation panel in one operation, so that I do not have to toggle each child individually.

### 6.6 Persistence

> **US-18.** As any user, I want my hide, hide-when-read, and navigation-pin choices to persist across application restarts and across server folder re-discovery, so that I configure each folder once.

## 7. Functional Requirements

### 7.1 Per-folder hide flag

- **FR-1.** Every folder shall have an independent, persistent **hide** flag (boolean). When enabled, the folder shall not appear in the folder list under normal viewing.
- **FR-2.** The hide flag shall be available on all folders regardless of role: system folders and user-created folders alike.
- **FR-3.** The hide flag shall be togglable from a context action on the folder in the folder list (applicable to non-selectable / non-Inbox system folders).
- **FR-4.** The hide flag shall also be togglable from the folder's properties/edit screen.
- **FR-5.** Changing the hide flag shall take effect immediately in the folder list without requiring a restart or manual refresh.
- **FR-6.** The hide flag shall have **no effect** on: synchronization, notifications, unified-inbox membership, unread counts, or any other folder property. It is purely a display concern.

### 7.2 Per-folder hide-when-read flag

- **FR-7.** Every folder shall have an independent, persistent **hide-when-read** flag (boolean). When enabled, the folder shall be automatically hidden from the account folder list and unified folder list whenever it has zero unread messages, including unread messages in descendant folders.
- **FR-8.** When unread messages arrive (or a message is marked unread), a conditionally-hidden folder shall reappear in the folder list automatically.
- **FR-9.** The hide-when-read flag shall be togglable from the folder's properties/edit screen.
- **FR-10.** The hide-when-read behavior shall NOT apply in selection contexts (e.g. "move to folder" dialogs), so that the user can always target any folder regardless of its unread state.

### 7.3 Show-hidden-folders toggle

- **FR-11.** The folder list shall provide a menu option to "show hidden folders" (a togglable checkbox).
- **FR-12.** When "show hidden folders" is active, all folders that are hidden (by the explicit hide flag) or conditionally hidden (by the hide-when-read flag with zero unread) shall be displayed.
- **FR-13.** Folders revealed by "show hidden folders" shall be visually distinguished from normally-visible folders — rendered at reduced opacity or with a similar dimming treatment — so the user can tell which folders are hidden.
- **FR-14.** The "show hidden folders" preference shall default to **off** on each session, so the user starts with their curated view.

### 7.4 Navigation panel pinning

- **FR-15.** Every folder shall have an independent, persistent **navigation** flag (boolean). When enabled, the folder shall appear in the application's navigation panel.
- **FR-16.** Only folders with the navigation flag enabled shall appear in the navigation panel (when not in a settings/configuration view).
- **FR-17.** The navigation flag shall be togglable from the folder's properties/edit screen and from a context action on the folder.
- **FR-18.** The navigation flag shall be independent of the hide flag and the hide-when-read flag. A folder can be pinned to the navigation panel and hidden from the folder list, or visible in the folder list but not pinned to the navigation panel.
- **FR-19.** When acting on a parent folder, the application shall offer a bulk action to enable or disable the navigation flag for all descendant folders in one operation.

### 7.5 Independence of concerns

- **FR-20.** The three folder-visibility flags (hide, hide-when-read, navigation) shall be independent of each other and of: synchronization on/off, notification on/off, unified-inbox membership, folder subscription state, and folder color.
- **FR-21.** Changing any visibility flag shall never modify any other flag or folder property.

### 7.6 Defaults

- **FR-22.** When a new folder is discovered (on account creation or on subsequent server sync), the hide flag shall default to **disabled** (folder visible).
- **FR-23.** When a new folder is discovered, the hide-when-read flag shall default to **disabled** (folder always visible regardless of unread count).
- **FR-24.** When a new folder is discovered, the navigation flag shall default to **disabled** (folder not pinned to the navigation panel).

### 7.7 Persistence

- **FR-25.** All three flags (hide, hide-when-read, navigation) shall survive application restart, re-login, and re-discovery of the server folder list.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Toggling a folder's hide flag shall update the folder list in well under one second under normal conditions.
- **NFR-2. Scale.** The feature shall remain usable with accounts containing hundreds of folders, where many may be hidden simultaneously.
- **NFR-3. Discoverability.** The "show hidden folders" option shall be in the folder list's menu, clearly labeled. The context action for hiding shall be available on long-press or right-click, consistent with other folder context actions.
- **NFR-4. Visual clarity.** Hidden folders shown via the "show hidden" toggle shall be rendered with clearly reduced opacity (or equivalent visual treatment) so they are unmistakably distinct from visible folders.
- **NFR-5. Privacy.** Folder visibility state shall be stored locally only. The feature shall introduce no network traffic and shall not modify any server-side folder state.
- **NFR-6. Accessibility.** All controls (hide toggle, show-hidden toggle, navigation pin) shall be reachable via keyboard, with screen-reader labels, and shall respect the application-wide theme.

## 10. Open Questions

- **OQ-1. Hide on Inbox.** The source application restricts the explicit-hide context action to non-selectable folders, which in practice means Inbox cannot be hidden via the context menu (though it may be hideable via the properties screen). Should the desktop application allow hiding Inbox, or should Inbox be protected from hiding? This is a UX safety decision.
- **OQ-2. Session vs. persistent "show hidden" state.** The source application resets "show hidden folders" to off at the start of each session. Should this be a persistent preference instead, or is the session-scoped reset the right default? Users who frequently audit hidden folders might prefer persistence.
- **OQ-3. Auto-add new folders to navigation.** The source application has a preference (`auto_folder_nav`) that appears to control whether newly created folders are automatically added to the navigation panel. The scope and default of this preference is unclear. Should newly created folders ever be auto-pinned to the navigation panel, and if so, under what conditions?
- **OQ-4. Hide-when-read scope.** The source application's hide-when-read applies "only in the account and unified folder list" (per its UI hint). The exact contexts where hide-when-read does and does not apply (e.g. search results, folder pickers, notification folder lists) should be enumerated during design.
- **OQ-5. Interaction with folder collapsing.** Folders can be collapsed (hiding their children in the tree view). The interaction between collapse state, the hide flag, and the hide-when-read flag on parent vs. child folders needs clarification — e.g. if a parent is collapsed and a hidden child gets unread mail, does the child's reappearance force the parent to expand?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Three independent display concerns.** The source application treats hide (explicit), hide-when-read (conditional), and navigation-panel pinning as three fully independent boolean flags. Combined with synchronization on/off, notification on/off, and unified-inbox membership, each folder carries six orthogonal boolean properties. All combinations are meaningful. For example: a folder can be hidden from the folder list, excluded from the unified inbox, but still synchronized and still generating notifications — useful for a folder the user monitors by notification only, without wanting it to occupy screen space.

- **N-2. Hide vs. unsubscribe.** Hiding a folder is a local display preference. Unsubscribing from an IMAP folder is a server-side operation that may cause the server to stop reporting the folder entirely. The two are completely separate. Hiding is lightweight, instant, and trivially reversible; unsubscribing may have permanent side effects depending on the server. This distinction is important because users coming from other email clients may conflate "I don't want to see this folder" with "unsubscribe."

- **N-3. Hide-when-read includes child unread counts.** The conditional-hide logic considers not just a folder's own unread count but the sum of unread messages across all its descendants. This prevents a parent folder from disappearing while a child folder still has unread mail — which would be confusing in a hierarchical folder tree.

- **N-4. Show-hidden resets per session.** The source application defaults "show hidden folders" to off each time the folder list is entered. This is a deliberate choice: the curated view is the user's "home state," and revealing hidden folders is a temporary inspection mode. If the toggle persisted, a user who forgot they turned it on would see clutter indefinitely, defeating the purpose of hiding.

- **N-5. Navigation panel as a promotion, not a filter.** The navigation panel flag is additive — it promotes a folder to a prominent location for quick access. It does not replace or override the folder's presence in the full account folder list. A folder can appear in both places, or in only one, giving the user layered navigation: the navigation panel for frequent destinations, the full folder list for everything else.
