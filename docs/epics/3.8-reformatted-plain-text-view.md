# Epic: Reformatted Plain-Text View

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.8** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Reformatted plain-text view. Optional re-flow into a phishing-resistant plain layout."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[COULD]` — nice to have; ship if time allows, otherwise post-v1.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email messages arrive in a wide range of formats — plain text, richly styled HTML with embedded CSS, or both as multipart alternatives. Original HTML messages may include hidden content, invisible tracking pixels, misleading CSS that conceals phishing links, custom fonts and colors designed to confuse, background images that mask text, JavaScript (in rare clients), and other elements that create security and privacy risks.

The Reformatted Plain-Text View is the application's **default** rendering mode for message bodies. Rather than handing raw HTML to a full browser engine, the application re-processes the message through a strict sanitization pipeline that:

1. Strips all scripts, unsafe CSS, hidden content, tracking pixels, and remote resources.
2. Normalizes font families, sizes, colors, and margins to the application's own theme-consistent defaults.
3. Blocks external images and remote content until the user explicitly allows them.
4. Presents the result as styled text rendered by the application's own text component — not a browser widget.

The effect is a display that looks like a neatly reflowed plain-text message while preserving basic structure (paragraphs, lists, tables, blockquotes, inline images when allowed). Because the rendering is controlled end-to-end by the application rather than delegated to a browser, phishing techniques that rely on CSS trickery, hidden overlays, or spoofed link targets are neutralized or made visible.

Users who need the full original styling — for example to view complex newsletter layouts or QR codes — can switch to the Original Message View (feature 3.7) on a per-message basis, with appropriate confirmation prompts.

## 2. Goals

- **G1.** Provide a phishing-resistant, privacy-preserving default rendering for every email message, requiring zero configuration from the user.
- **G2.** Sanitize HTML content by removing dangerous, tracking, and deceptive elements while preserving readability and basic document structure.
- **G3.** Normalize visual presentation (font, size, color, margins, backgrounds) to the application's theme, so that sender-controlled styling cannot deceive the user.
- **G4.** Block external images and remote content by default, giving the user explicit per-message, per-sender, and per-domain control over when to allow them.
- **G5.** Render plain-text-only messages faithfully, preserving whitespace structure, quote levels, and optionally using a monospaced font for preformatted content.
- **G6.** Allow the user to switch between the reformatted view and the original view at any time, with a discoverable toggle and an optional confirmation dialog.
- **G7.** Support a force-light-background mode so that messages remain legible when the application is in a dark theme.

## 3. Non-Goals

- **NG1.** This epic does **not** define how messages are fetched, stored, or synchronized — only how the stored body is processed for display.
- **NG2.** This epic does **not** cover the Original Message View (feature 3.7), which renders raw HTML in a browser widget. It only defines the boundary and switching contract between the two views.
- **NG3.** This epic does **not** cover remote-image confirmation dialogs or tracker-image detection in depth — those are features 3.10 and 3.11 respectively. It defines only that the reformatted view blocks remote content by default and integrates with those features.
- **NG4.** This epic does **not** prescribe the sanitization algorithm or which HTML tags survive — only the observable behavioral contract of what the user sees and does not see.
- **NG5.** Composing in plain-text mode (feature 4.2) is out of scope; this epic concerns the **reading** experience only.

## 4. Glossary

| Term | Definition |
|---|---|
| **Reformatted view** | The application's default message rendering mode, which re-processes the message body through a sanitization pipeline and displays the result using the application's own text-rendering component, not a browser engine. |
| **Original view** | An alternative rendering mode that displays the message's raw HTML in an embedded browser widget, preserving the sender's original styling. Less safe than the reformatted view. |
| **Sanitization** | The process of stripping or neutralizing dangerous, tracking, and deceptive elements from an HTML message body before display. |
| **Plain-text-only message** | A message whose MIME structure contains only a `text/plain` part and no `text/html` alternative. |
| **Multipart/alternative message** | A message that provides both a plain-text and an HTML version of its body. The application normally uses the HTML part (after sanitization) but can be configured to prefer the plain-text part. |
| **Tracking pixel** | A tiny or invisible remote image embedded in a message for the purpose of detecting whether and when the recipient opened the message. |
| **Remote content** | Any resource (image, font, stylesheet) referenced by URL in the message body that must be fetched from an external server to display. |
| **Force-light mode** | A per-message toggle that forces a light background and dark text in the reformatted view, overriding the application's dark theme, to improve readability of messages that assume a light background. |
| **Quote level** | The depth of nested email quoting, conventionally indicated by `>` prefixes in plain text (RFC 3676 §4.5) or `<blockquote>` elements in HTML. |
| **Preformatted text** | Text whose whitespace (spaces, tabs, line breaks) is structurally significant and should be preserved exactly as authored — for example, ASCII art or code. |

## 5. Personas

- **P1. The security-conscious user.** Prioritizes safety above appearance. Wants the application to neutralize phishing tricks by default and only views the original message when absolutely necessary. Appreciates the reformatted hint banner.
- **P2. The casual reader.** Reads newsletters and promotional email. Expects messages to look "good enough" in the reformatted view and occasionally switches to the original view for complex layouts. May choose to always show original messages for trusted senders.
- **P3. The plain-text purist.** Sends and receives primarily plain-text email. Values monospaced rendering, preserved whitespace, and faithful quote-level display. Never needs the original HTML view.
- **P4. The dark-theme user.** Runs the application in a dark or black theme. Needs force-light mode for messages whose embedded colors become unreadable against a dark background.
- **P5. The power user.** Has fine-tuned per-sender and per-domain preferences for when to show the original HTML view and when to load remote images. Wants those preferences to persist and to be overridable.

## 6. User Stories

### 6.1 Default reformatted display

> **US-1.** As any user, when I open a message, I want to see a reformatted, sanitized version of its body by default, so that I am protected from phishing, tracking, and deceptive styling without having to take any action.

> **US-2.** As any user, I want the reformatted view to preserve the basic structure of the message — paragraphs, headings, lists, tables, blockquotes, horizontal rules, and inline formatting (bold, italic, underline) — so that the message remains readable and organized.

> **US-3.** As any user, I want font families, font sizes, and text colors in the reformatted view to be normalized to the application's theme defaults, so that sender-chosen styling cannot obscure or disguise content.

> **US-4.** As any user, I want margins, padding, and background colors to be stripped or normalized in the reformatted view, so that the message text fills the available width without awkward whitespace or hidden regions.

### 6.2 Sanitization behavior

> **US-5.** As a security-conscious user, I want all scripts, form elements, event handlers, and unsafe CSS to be removed before the message is displayed, so that no executable content can run in the reformatted view.

> **US-6.** As a security-conscious user, I want tracking pixels to be detected and blocked by default in the reformatted view, so that senders cannot silently confirm that I opened their message.

> **US-7.** As any user, I want external images to be blocked by default in the reformatted view, with placeholder indicators showing where images would appear, so that I can choose whether to load them.

> **US-8.** As any user, I want remote fonts and external CSS to be stripped in the reformatted view, so that no external network requests are made merely by reading a message.

### 6.3 Plain-text-only messages

> **US-9.** As any user, when I open a message that has only a plain-text body (no HTML alternative), I want the reformatted view to display it with proper line breaks, preserved whitespace alignment, and nested blockquotes derived from `>` quoting prefixes, so that the text is easy to read and quote levels are visually distinct.

> **US-10.** As a plain-text purist, I want an option to render plain-text-only messages in a monospaced font as preformatted text, so that ASCII art, code snippets, and tabular data display correctly.

> **US-11.** As any user, I want tabs in plain-text messages to be expanded to consistent column-aligned spaces, so that indented or tabular content lines up as the sender intended.

### 6.4 Switching between reformatted and original views

> **US-12.** As any user, I want a clearly visible toggle button in the message toolbar area that lets me switch between the reformatted view and the original (HTML) view for the current message, so that I can see the sender's intended layout when needed.

> **US-13.** As a security-conscious user, I want the application to show a confirmation dialog the first time I switch to the original view, warning me that the original view is less safe, so that I make an informed choice.

> **US-14.** As a casual reader, I want the option to suppress the confirmation dialog globally, so that I am not asked every time I view an original message.

> **US-15.** As any user, I want to be able to remember my "show original" choice per sender address or per sender domain, so that messages from trusted senders automatically display in the original view without prompting.

> **US-16.** As any user, when a message is in the spam/junk folder, I want the confirmation dialog to always appear before switching to the original view (regardless of my per-sender or global suppression settings), so that I am extra cautious with suspicious messages.

> **US-17.** As any user, I want the toggle button icon to change to indicate which view is currently active, so that I always know whether I am seeing the reformatted or the original message.

### 6.5 Reformatted-view hint banner

> **US-18.** As a new user, I want a dismissible information banner displayed above the reformatted message body explaining that this is the safe reformatted view and how to switch to the original, so that I understand what I am seeing and how to get to the original if needed.

> **US-19.** As any user, once I dismiss the hint banner (by switching to the original view at least once), I want it to never appear again, so that it does not become a permanent nuisance.

### 6.6 Force-light mode

> **US-20.** As a dark-theme user, I want a toggle (button or menu item) that forces a light background and appropriate text color for the reformatted view of the current message, so that messages with embedded dark-on-dark or inverted colors become readable.

> **US-21.** As a dark-theme user, I want an option to default force-light mode to "on" for all messages, so that I do not have to toggle it per message.

> **US-22.** As a user not using a dark theme, I do not want to see the force-light toggle, since it has no effect in a light theme.

### 6.7 Formatting fidelity preferences

> **US-23.** As a power user, I want individual toggles controlling whether the reformatted view preserves the sender's text colors, text sizes, font faces, text alignment, and background colors, so that I can choose the trade-off between fidelity and safety for my own workflow.

> **US-24.** As any user, I want an option to show or hide image placeholder indicators in the reformatted view when remote images are blocked, so that I can choose between a cleaner layout and awareness of where images exist.

> **US-25.** As any user, I want an option to show or hide horizontal-rule separators between message sections in the reformatted view, so that I can control visual density.

### 6.8 Prefer plain-text part

> **US-26.** As a security-conscious user, when a message provides both HTML and plain-text alternatives, I want an option to always use the plain-text part instead of the HTML part, so that I never process HTML at all.

### 6.9 Text selection and quoting

> **US-27.** As any user, I want to be able to select text in the reformatted view and copy it, so that I can paste message content elsewhere.

> **US-28.** As any user, when I select text in the reformatted view and then choose "Reply", I want only the selected text to be quoted in my reply, so that I can quote precisely without manual editing.

### 6.10 Large messages

> **US-29.** As any user, when a message body is very large, I want the reformatted view to truncate it at a reasonable limit and show an indication that the message was truncated, along with a link to view the full original, so that the application remains responsive.

## 7. Functional Requirements

### 7.1 Default rendering mode

- **FR-1.** The reformatted view shall be the default rendering mode for all messages. No user configuration shall be required for it to be active.
- **FR-2.** When the reformatted view is active, the message body shall be rendered by the application's own text-display component, **not** by an embedded browser engine.
- **FR-3.** When the original view is active, the message body shall be rendered by an embedded browser widget. The two rendering paths shall be mutually exclusive for a given message at a given time.

### 7.2 HTML sanitization

- **FR-4.** The sanitization pipeline shall remove or neutralize at minimum: all script elements, all event-handler attributes, all form elements, all embedded objects/applets, all unsafe CSS properties (e.g. `position: absolute`, overlays, opacity tricks intended to hide content), and all external stylesheet references.
- **FR-5.** The sanitization pipeline shall enforce an allowlist of safe HTML elements and attributes. Elements not on the allowlist shall be removed (their text content may be preserved).
- **FR-6.** When the "preserve text color" preference is disabled (or when using a black-and-white theme), sender-specified text colors shall be stripped and replaced with the application's theme text color.
- **FR-7.** When the "preserve text size" preference is disabled, sender-specified font sizes shall be stripped and replaced with the application's default message text size, adjusted by the user's zoom preference.
- **FR-8.** When the "preserve font face" preference is disabled, sender-specified font families shall be stripped and replaced with the application's configured display font.
- **FR-9.** When the "preserve text alignment" preference is disabled, sender-specified text alignment (center, right, justify) shall be stripped.
- **FR-10.** When the "preserve background color" preference is disabled (the default), background colors on all elements shall be stripped.
- **FR-11.** Background images shall be converted to regular inline images (visible to the user as normal images rather than hidden behind text).
- **FR-12.** Microsoft conditional comments and proprietary namespace markup shall be normalized or removed so they do not produce rendering artifacts.

### 7.3 Remote content blocking

- **FR-13.** All references to remote images shall be blocked by default in the reformatted view. Blocked images shall be represented by a placeholder (when the image-placeholder preference is enabled) or omitted entirely (when disabled).
- **FR-14.** Remote fonts and remote CSS shall always be stripped in the reformatted view, regardless of user preference.
- **FR-15.** Embedded (inline/CID) images shall be displayed if the user's inline-image preference is enabled; they are not considered remote content.
- **FR-16.** When the "disable tracking images" preference is enabled (the default), the sanitization pipeline shall detect likely tracking pixels (using bundled blocklists) and block them, reporting the detection to the user.

### 7.4 Plain-text message handling

- **FR-17.** When a message has only a plain-text body, the application shall convert it to a structured display format by: escaping any HTML-significant characters, converting line breaks to visible line breaks, expanding tab characters to spaces aligned on 4-character boundaries, and converting `>`-prefixed quote lines into visually indented blockquote regions following RFC 3676 conventions.
- **FR-18.** When the "monospaced preformatted" preference is enabled, plain-text-only messages shall be rendered in a monospaced font with whitespace preserved exactly (preformatted mode).
- **FR-19.** The application shall record whether a message is plain-text-only, HTML-only, or multipart/alternative, and make this status visible to the user (e.g. via a "plain text only" label in the message metadata).

### 7.5 Prefer-plain-text mode

- **FR-20.** A global preference ("download plain text only") shall cause the application to use the plain-text MIME part of multipart/alternative messages instead of the HTML part. When this preference is enabled, such messages shall be treated as plain-text-only for display purposes.

### 7.6 View switching

- **FR-21.** A toggle control shall be displayed in the message toolbar area whenever a message body is shown. The control's icon shall visually indicate whether the reformatted view (default) or the original view is active.
- **FR-22.** Clicking the toggle shall switch to the other view. When switching from reformatted to original, the confirmation dialog shall appear if the "confirm showing original messages" preference is enabled (the default) **and** the user has not previously suppressed the dialog for this message, sender, or domain.
- **FR-23.** For messages in the spam/junk folder, the confirmation dialog shall always appear when switching to the original view, regardless of any suppression preferences.
- **FR-24.** The confirmation dialog shall offer the following suppression options (except for junk-folder messages, where all suppression options are hidden):
  - "Do not ask again for this sender address"
  - "Do not ask again for this sender domain"
  - "Do not ask again" (global suppression)
  - "Never ask again" (disables the preference entirely)
- **FR-25.** The confirmation dialog shall include a reference to a help/FAQ entry explaining why viewing original messages can be unsafe.
- **FR-26.** When the user has previously chosen "always show original" for a sender or domain, messages from that sender or domain shall automatically display in the original view without prompting.
- **FR-27.** A global preference ("show reformatted messages by default") shall control whether the confirmation dialog is active. When disabled, all messages open in the original view by default (except junk-folder messages, which always require confirmation).
- **FR-28.** The user shall be able to reset all per-sender and per-domain view preferences via a "reset questions" action in settings.

### 7.7 Reformatted-view hint banner

- **FR-29.** When a message is displayed in the reformatted view and the hint has not been permanently dismissed, an informational banner shall appear explaining: (a) that this is the safe reformatted message view, (b) which icon to tap to switch to the original view, and (c) that reformatting takes time proportional to message size.
- **FR-30.** The hint banner shall be permanently dismissed the first time the user switches to the original view. Once dismissed, it shall never reappear for that user.
- **FR-31.** The hint banner shall only appear when the application environment supports the original view (i.e. an embedded browser engine is available). If no browser engine is available, the banner is unnecessary since the original view is inaccessible.

### 7.8 Force-light mode

- **FR-32.** When the application is using a dark theme, a force-light toggle shall be available for both the reformatted and original views. In the reformatted view, activating force-light shall switch the text-display background to white and the text color to the application's light-theme text color.
- **FR-33.** When the application is using a light theme, the force-light toggle shall be hidden.
- **FR-34.** A global preference ("default force light") shall allow the user to set force-light mode to "on" by default for all newly opened messages when in a dark theme.
- **FR-35.** Force-light state is per-message and does not persist across application sessions.

### 7.9 Zoom and text size

- **FR-36.** The reformatted view shall respect the application's global message-zoom preference, applying the configured zoom percentage to the base text size.
- **FR-37.** The user shall be able to adjust text size for an individual message (e.g. via pinch or buttons), and the per-message size shall be remembered for the duration of the current session.

### 7.10 Quote handling

- **FR-38.** In the reformatted view, quoted sections (blockquotes) shall be collapsible. When the "collapse quotes" preference is enabled, top-level quotes shall be collapsed by default and replaced with a clickable expand indicator.
- **FR-39.** Expanding or collapsing a quote block shall be instantaneous and shall not reload the message.

### 7.11 Text selection and quoting

- **FR-40.** Text selection (select, select-all, copy) shall be fully supported in the reformatted view.
- **FR-41.** When the user selects text in the reformatted view and initiates a reply, only the selected text shall be included as the quoted text in the reply draft.

### 7.12 Large message handling

- **FR-42.** When a message body exceeds the application's configured maximum reformatted-text size, the body shall be truncated at that limit in the reformatted view. A visible truncation notice and a link to open the full original view shall be appended.
- **FR-43.** When a message body exceeds the maximum full-text size (in the original view), it shall also be truncated with a notice and an option to open the message in an external browser.

### 7.13 Long-press options

- **FR-44.** Long-pressing the view-toggle button (when the original view is active) shall offer a context menu with at least: (a) open in full-screen / external viewer, (b) toggle "fit to width" mode, (c) toggle "monospaced preformatted" mode.

## 8. Non-Functional Requirements

- **NFR-1. Performance.** The reformatted view shall render a typical message (under 100 KB of HTML) in well under one second on a modern system. The application shall remain responsive during sanitization of larger messages.
- **NFR-2. Truncation safety.** The large-message truncation threshold shall be configurable or at least have a sensible default that prevents the application from becoming unresponsive on extremely large messages (multi-megabyte HTML bodies).
- **NFR-3. Theme consistency.** The reformatted view shall respect the application's current theme (light, dark, black, sepia) for all normalized text colors, backgrounds, and accent colors. Theme changes shall take effect on the next message open.
- **NFR-4. Accessibility.** All reformatted-view text shall be selectable, zoomable, and screen-reader accessible. The view-toggle, force-light toggle, and all toolbar controls shall have accessible labels and be keyboard-reachable.
- **NFR-5. Privacy.** The reformatted view shall make zero network requests unless the user explicitly allows remote images. No remote resources — images, fonts, CSS, scripts — shall be fetched by default.
- **NFR-6. Robustness.** If the sanitization pipeline encounters malformed HTML, excessively deep nesting, or an out-of-memory condition, it shall degrade gracefully (e.g. display a plain-text fallback or an error message) rather than crash.
- **NFR-7. Idempotency.** Toggling between reformatted and original views and back shall produce the same reformatted output as the initial display — no content shall be lost through round-tripping.

## 10. Open Questions

- **OQ-1. Sanitization allowlist scope.** The exact set of HTML tags and CSS properties that survive sanitization is a design-time decision. The current implementation uses a generous allowlist (most formatting tags, tables, lists, images, links). Should the allowlist be documented as a stable contract, or is it acceptable for it to evolve between releases?

- **OQ-2. Heading preservation.** The "text titles" preference (which controls whether HTML headings are rendered as headings versus normal text) defaults to off. Should it default to on for better structural readability, or does stripping headings serve a security purpose (e.g. preventing visually dominant phishing headers)?

- **OQ-3. Hidden-content handling.** The codebase contains provisions for displaying content hidden via CSS (`display_hidden` preference), but this is currently gated behind a debug/experimental flag. Should there be a user-facing option to reveal hidden content, or is the security risk too high?

- **OQ-4. Background-image conversion.** Background images are converted to inline images in the reformatted view. This makes hidden background images visible, which is good for transparency, but may produce unexpected visual artifacts. Should there be an option to suppress this conversion entirely?

- **OQ-5. Truncation threshold configurability.** The maximum reformatted-text size is currently a fixed internal value. Should it be exposed as a user-configurable preference, or is a fixed default sufficient?

- **OQ-6. Per-message view preference persistence.** The current behavior stores "show original" and "show images" preferences per sender address and per sender domain, but not per individual message. When a user toggles the view for a single message, it does not persist across sessions. Should per-message persistence be supported for power users?

- **OQ-7. Quote collapsing interaction with text selection.** When quotes are collapsed, the collapsed text is not selectable. Should the "select text and reply" feature (US-28) automatically expand collapsed quotes before initiating the selection, or is it acceptable that collapsed text cannot be quoted?

- **OQ-8. Force-light persistence.** Force-light state is currently per-message and per-session only. Should it persist across sessions for individual messages, or is the global "default force light" preference sufficient?

- **OQ-9. CSS class parsing.** The sanitization pipeline optionally parses CSS classes from `<style>` elements and inlines them. This improves fidelity but increases processing time and attack surface. Should class parsing be on by default, or should it be an opt-in power-user preference?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Reformatted is the default, not an alternative.** Unlike most email clients that render HTML natively and offer a "view source" or "view as text" alternative, this application inverts the model: the sanitized/reformatted view is the default, and the original HTML is the alternative. This is a deliberate security posture — the user must opt in to risk, not opt out of safety.

- **N-2. Two rendering engines, mutually exclusive.** The reformatted view uses the application's own text-rendering component; the original view uses an embedded browser widget. They are never both active for the same message. This clean separation ensures that the reformatted view's security guarantees are not diluted by browser-engine behavior.

- **N-3. Junk-folder hardening.** Messages in the spam/junk folder always require confirmation before the original view is shown, even if the user has globally disabled confirmations or has a per-sender exception. This is a defense-in-depth measure: if a malicious message is misclassified as spam and the user explores it, the extra friction reduces the chance of accidental exposure.

- **N-4. Sanitization-preference granularity.** The five formatting-preservation toggles (text color, text size, font face, text alignment, background color) let the user progressively trade security for fidelity. The secure defaults (most preservation disabled, especially background color) mean that a user who never visits settings gets the safest experience. Power users who trust their mail can re-enable specific aspects without switching entirely to the original view.

- **N-5. Plain-text wrapping via HTML.** Even plain-text-only messages are internally converted to a minimal HTML structure (a single wrapper element with the plain-text content escaped and line-broken). This allows the same rendering pipeline to handle both HTML and plain-text messages, simplifying quote handling, text selection, and theme integration.

- **N-6. Monospaced-preformatted is opt-in.** The monospaced-preformatted preference defaults to off because most plain-text email is prose, not code or ASCII art. Enabling it globally would make proportional-font prose look worse. Users who frequently receive preformatted content can opt in.

- **N-7. The hint banner as onboarding.** The reformatted-view hint banner serves the same purpose as a first-run tooltip: it educates the user about the existence of two views and how to switch. Once the user demonstrates awareness by switching views at least once, the banner is permanently dismissed. It is not shown at all if the environment lacks a browser engine (since in that case the original view is inaccessible and the banner's instructions would be misleading).

- **N-8. Force-light as a dark-theme safety net.** Many HTML emails assume a white background and use dark text colors. In a dark application theme, after sanitization strips background colors, this can result in dark text on a dark background. Force-light mode is the escape hatch: it overrides the theme for the message area only, restoring legibility without forcing the user to switch their entire application theme.

- **N-9. "Download plain text only" is a connection-level preference.** The option to prefer the plain-text MIME part over the HTML part is classified as a connection/download preference rather than a display preference because it affects which part of the message is fetched and stored, not just how it is rendered. Once the plain-text part is stored, the message is treated as plain-text-only for all display purposes.
