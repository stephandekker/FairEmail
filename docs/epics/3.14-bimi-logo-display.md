# Epic: BIMI Logo Display

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.14** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"BIMI logo display with optional VMC certificate verification."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Brand Indicators for Message Identification (BIMI) is a DNS-based standard that allows domain owners to publish a brand logo alongside their outgoing email. A sending domain publishes a DNS TXT record at `<selector>._bimi.<domain>` containing a reference to an SVG logo and, optionally, a Verified Mark Certificate (VMC) that cryptographically proves the domain's right to use that logo.

For the end user, BIMI solves two problems:

1. **Visual brand recognition.** Instead of a generic placeholder avatar or an unreliable address-book photo, the user sees the sender's official logo next to the message — immediately recognizable and consistent across every message from that brand.
2. **Sender authenticity signal.** When a VMC is present and valid, the application can display a "verified sender" badge that gives the user confidence the message genuinely comes from the claimed organization. This complements (and depends on) the email authentication results (DKIM, SPF, DMARC) already evaluated by the application.

BIMI logos participate in the application's existing avatar-source priority chain. They are fetched on demand, cached locally, and never sent to third-party services. The feature is disabled by default and must be explicitly opted into by the user, consistent with the application's privacy-first posture.

## 2. Goals

- **G1.** Display the sender's BIMI-published SVG logo as the message avatar when the user has enabled BIMI and no higher-priority avatar source (e.g. local contact photo) is available.
- **G2.** Optionally verify the VMC certificate accompanying a BIMI record and, when valid, display a distinct "verified sender" badge on the message.
- **G3.** Allow the user to require VMC verification, so that unverified BIMI logos are suppressed entirely when the user prefers only cryptographically attested brand indicators.
- **G4.** Gate BIMI logo fetches on the message's DMARC authentication result when the user has enabled the "require DMARC for external images" option, so that logos are not fetched for unauthenticated mail.
- **G5.** Ensure BIMI logos appear in every surface where avatars are shown: message list, conversation view, notifications, and desktop widgets.
- **G6.** Never fetch BIMI logos for messages in the Spam/Junk folder, to avoid giving visual legitimacy to junk mail.

## 3. Non-Goals

- **NG1.** This epic does **not** define the broader avatar-source priority chain (contact photos, Gravatars, Libravatars, favicons, identicons, etc.). It only specifies how the BIMI source integrates into whatever chain exists.
- **NG2.** This epic does **not** define how DKIM, SPF, DMARC, or ARC authentication results are obtained or stored. It only consumes the DMARC pass/fail result already available on the message.
- **NG3.** Animated SVG logos or non-SVG image formats are not a goal. The BIMI specification mandates SVG Tiny Portable/Secure; the application renders SVG to a raster bitmap at a fixed size.
- **NG4.** BIMI logo management by the sender (publishing DNS records, obtaining VMCs) is entirely outside the scope of this application.
- **NG5.** This epic does **not** cover the broader DNS lookup infrastructure (caching, resolution method, DNS-over-HTTPS). It only specifies what records to look up and how to interpret them.

## 4. Glossary

| Term | Definition |
|---|---|
| **BIMI** | Brand Indicators for Message Identification — a DNS-based standard (published by the BIMI Group) for associating a brand logo with a sending domain. |
| **BIMI DNS record** | A DNS TXT record at `<selector>._bimi.<domain>` containing key-value pairs: `v` (version, must be `BIMI1`), `l` (HTTPS URL to an SVG logo), and `a` (HTTPS URL to a PEM-encoded VMC certificate chain). |
| **BIMI-Selector header** | An email header (`BIMI-Selector: v=BIMI1; s=<selector>;`) that allows a sender to specify a non-default selector for BIMI DNS lookup. |
| **Selector** | A label used to partition BIMI records for a domain, analogous to DKIM selectors. Defaults to `default` when not specified in the message header. |
| **VMC** | Verified Mark Certificate — an X.509 certificate with the BIMI extended key usage OID, issued by an accredited Certificate Authority, binding a domain to a specific logo. The certificate's Logotype extension (RFC 3709) embeds or references the logo image. |
| **Verified sender** | A message whose BIMI logo was obtained from a valid, chain-verified VMC certificate and whose sending domain passes DMARC with a policy of `quarantine` or `reject` at 100% coverage. |
| **Avatar-source priority chain** | The ordered list of sources the application consults when resolving an avatar for a sender address: contact photo, BIMI, Gravatar, Libravatar, favicon, generated icon, identicon, etc. |
| **DMARC** | Domain-based Message Authentication, Reporting & Conformance — an email authentication protocol. BIMI verification requires DMARC to pass with an enforcement policy. |

## 5. Personas

- **P1. The security-conscious professional.** Wants visual confirmation that messages from banks, vendors, and enterprise partners are genuine. Enables BIMI with VMC verification required, trusting only cryptographically attested logos.
- **P2. The visual organizer.** Appreciates brand logos as visual anchors in a busy inbox. Enables BIMI without requiring VMC verification — any published logo is better than a generic icon.
- **P3. The privacy-first user.** Prefers no external network requests for avatars. Leaves BIMI disabled (the default) and relies on local contact photos or generated icons.
- **P4. The power user.** Enables BIMI alongside other avatar sources (Gravatar, favicons) and fine-tunes the "require DMARC for external images" gate to suppress logos from unauthenticated senders.

## 6. User Stories

### 6.1 Enabling and disabling BIMI

> **US-1.** As any user, I want BIMI logo display to be disabled by default, so that no external network requests are made for avatar images until I explicitly opt in.

> **US-2.** As any user, I want a toggle in the display settings to enable or disable BIMI logo display, so that I can control whether the application fetches brand logos from DNS-referenced URLs.

> **US-3.** As any user, I want a separate toggle to require VMC certificate verification, so that I can choose between "show any published BIMI logo" and "show only cryptographically verified logos".

> **US-4.** As any user, when I disable BIMI, I want the VMC-required toggle to become inactive (greyed out), so that it is clear the sub-option has no effect when the parent feature is off.

### 6.2 BIMI logo resolution

> **US-5.** As any user with BIMI enabled, when I view a message whose sender's domain publishes a BIMI DNS record containing a logo URL, I want the application to fetch and display that logo as the sender's avatar, so that I see the brand's official image.

> **US-6.** As any user, when the message contains a `BIMI-Selector` header, I want the application to use the specified selector for the DNS lookup instead of the default, so that senders who partition their BIMI records by mail stream are supported.

> **US-7.** As any user, when no BIMI record is found for the sending domain, I want the application to fall back to the parent domain (organizational domain) before giving up, so that subdomains that inherit the parent's BIMI policy are handled correctly.

> **US-8.** As any user, I want BIMI logos to be rendered from SVG to a raster image at a consistent size matching other avatar sources, so that the message list looks uniform.

> **US-9.** As any user, I want BIMI logo fetches to occur only over HTTPS, so that the integrity and confidentiality of the fetch are protected.

### 6.3 VMC certificate verification

> **US-10.** As a security-conscious user with VMC verification enabled, when the BIMI DNS record includes a certificate URL, I want the application to fetch the PEM-encoded certificate chain, validate it against a set of trusted root CAs (both bundled VMC-specific roots and the system trust store), and confirm that the certificate carries the BIMI extended key usage and that its subject matches the sending domain, so that I see a "verified sender" badge only when the chain is valid.

> **US-11.** As any user, when VMC verification succeeds, I want the application to extract the logo embedded in the certificate's Logotype extension (RFC 3709) and prefer it over the logo referenced by the `l` tag in the DNS record, so that the displayed logo is the one actually attested by the certificate.

> **US-12.** As any user, when VMC verification succeeds, I additionally want the application to confirm that the sender's domain has a DMARC policy of `quarantine` or `reject` with 100% coverage (`pct=100` or absent), so that the "verified" badge is only shown for domains that enforce authentication.

> **US-13.** As a security-conscious user who has enabled "require VMC", when a BIMI logo is found but VMC verification fails or no certificate URL is present, I want the logo to be suppressed entirely, so that I only see logos that are cryptographically verified.

> **US-14.** As a visual-organizer user who has **not** enabled "require VMC", when a BIMI logo is found but VMC verification fails, I still want the logo to be displayed (without a verified badge), so that I benefit from brand recognition even without cryptographic attestation.

### 6.4 Verified sender badge

> **US-15.** As any user, when a message's avatar comes from a verified BIMI/VMC source and the message also passes full email authentication (DKIM, SPF, and DMARC all pass, or the aggregate authentication result is positive), I want a distinct visual badge (icon and/or color) displayed next to the avatar indicating "verified sender", so that I can distinguish verified from unverified messages at a glance.

> **US-16.** As any user, when a message's avatar comes from BIMI but VMC verification failed or was not attempted, I want either no badge or a visually distinct "unverified" indicator (in debug/advanced mode), so that I am not misled about the verification status.

> **US-17.** As any user, I want the verified/unverified badge to have an accessible label (tooltip or screen-reader text) that says "Verified sender" or "Unverified sender" respectively, so that the information is available without relying on color or iconography alone.

### 6.5 Integration with other avatar sources

> **US-18.** As any user, I want the BIMI logo to be used only when no local contact photo is available for the sender, so that my personal address book photos take precedence over brand logos.

> **US-19.** As any user, I want BIMI logos to take precedence over Gravatars, Libravatars, favicons, and generated icons in the avatar-source priority chain, so that an official brand logo is preferred over generic web-scraped icons.

> **US-20.** As any user with the "require DMARC for external images" option enabled, I want BIMI fetches to be suppressed for messages that did not pass DMARC, so that external image fetches are gated on authentication consistently across all avatar sources.

### 6.6 Caching

> **US-21.** As any user, I want resolved BIMI logos (including the verified/unverified status) to be cached locally for a reasonable duration, so that repeated views of the same sender do not trigger redundant DNS lookups and image fetches.

> **US-22.** As any user, I want the BIMI cache to be cleared when I toggle the BIMI or VMC settings, so that changed settings take effect on the next view without stale cached results.

### 6.7 Spam/junk exclusion

> **US-23.** As any user, I want BIMI logo fetches to be completely suppressed for messages in the Spam/Junk folder, so that spam messages do not benefit from brand-logo visual credibility.

### 6.8 Surfaces

> **US-24.** As any user, I want BIMI logos to appear in every surface where sender avatars are displayed: the message list, the conversation/thread view, desktop notifications, and home-screen widgets, so that the brand indicator experience is consistent.

### 6.9 Privacy and discoverability

> **US-25.** As any user, I want to be able to reach an explanation of the privacy implications of BIMI (the fact that enabling it causes the application to make DNS queries and HTTPS requests to sender-controlled URLs) from the BIMI settings area, so that I can make an informed choice.

> **US-26.** As any user, I want a link to the BIMI Group's website from the settings area, so that I can learn more about the standard.

## 7. Functional Requirements

### 7.1 Settings

- **FR-1.** The application shall provide a user-facing toggle to enable or disable BIMI logo display. The default shall be **disabled**.
- **FR-2.** The application shall provide a separate toggle to require VMC certificate verification. This toggle shall be dependent on FR-1: it is only active when BIMI is enabled. The default shall be **disabled** (i.e. unverified logos are shown when BIMI is on).
- **FR-3.** Changing either toggle shall immediately clear the avatar cache so that the new setting takes effect on the next view.
- **FR-4.** The BIMI settings area shall include a link to the BIMI privacy/guidance specification and to the BIMI Group website.

### 7.2 DNS lookup

- **FR-5.** When resolving a BIMI logo for a sender, the application shall perform a DNS TXT lookup at `<selector>._bimi.<domain>`, where `<selector>` is extracted from the message's `BIMI-Selector` header (tag `s`), defaulting to `default` if the header is absent or the tag is missing.
- **FR-6.** If no BIMI record is found at the sending domain, the application shall retry the lookup at the organizational (parent) domain before declaring no BIMI record exists.
- **FR-7.** The BIMI DNS record shall be parsed as semicolon-delimited key-value pairs. The application shall recognize version tag `v` (must equal `BIMI1`), image tag `l`, and certificate tag `a`.
- **FR-8.** Unrecognized tags in the BIMI record shall be ignored (logged for diagnostics, not treated as errors).

### 7.3 Logo retrieval

- **FR-9.** The logo URL referenced by the `l` tag shall be fetched only if it uses the HTTPS scheme. Non-HTTPS URLs shall be rejected.
- **FR-10.** The fetched image shall be an SVG document. The application shall render it to a raster bitmap at the standard avatar size (matching other avatar sources), using a white background fill.
- **FR-11.** Logo fetches shall be subject to connection and read timeouts to avoid blocking the UI on slow or unresponsive servers.
- **FR-12.** Logo fetches shall not be performed for messages in the Spam/Junk folder.

### 7.4 VMC certificate verification

- **FR-13.** The certificate chain URL referenced by the `a` tag shall be fetched only if it uses the HTTPS scheme.
- **FR-14.** The fetched PEM stream shall be parsed into one or more X.509 certificates. The first certificate is the end-entity (leaf) certificate; subsequent certificates are intermediates.
- **FR-15.** The leaf certificate shall be validated against the following criteria:
  - (a) It carries the BIMI-specific extended key usage OID.
  - (b) At least one of its DNS subject alternative names shares the same root/organizational domain as the sending domain.
  - (c) The certificate chain validates to a trusted root CA, where the trust set comprises both a bundled set of VMC-specific CA certificates and the system-wide trust store.
  - (d) The certificate is not expired.
- **FR-16.** Certificate revocation checking is **not** required (may be disabled for performance and reliability).
- **FR-17.** If the leaf certificate contains a Logotype extension (RFC 3709) with an embedded SVG image (data URI), the application shall extract and render that image, preferring it over the `l`-tag logo.
- **FR-18.** After certificate validation, the application shall additionally look up the sender's DMARC DNS record (`_dmarc.<domain>`, falling back to `_dmarc.<parent-domain>`) and confirm that:
  - (a) A DMARC record exists.
  - (b) The policy (`p` tag) is `quarantine` or `reject`.
  - (c) The percentage (`pct` tag), if present, equals `100`.
- **FR-19.** Verification is considered successful only when FR-15, FR-17 (or FR-10 as fallback), and FR-18 all pass.

### 7.5 Display

- **FR-20.** When a BIMI logo is successfully resolved, the application shall use it as the sender's avatar in the message list, conversation view, notifications, and widgets.
- **FR-21.** A BIMI-sourced avatar shall be used only when no higher-priority avatar source (local contact photo) is available. It shall take precedence over Gravatars, Libravatars, favicons, and generated/identicon avatars.
- **FR-22.** When the BIMI logo source is a verified VMC **and** the message passes full email authentication (DKIM + SPF + DMARC all pass, or the aggregate authentication-results header indicates a pass), the application shall display a "verified sender" badge adjacent to or overlaid on the avatar.
- **FR-23.** The verified badge shall use a distinct color and icon level to differentiate it from the unverified state.
- **FR-24.** The badge shall carry an accessible content description: "Verified sender" when verified, "Unverified sender" otherwise.
- **FR-25.** When the "verified sender" badge is shown for a message, the separate authentication-results indicator (SPF/DKIM/DMARC icons) may be suppressed for that message to avoid visual redundancy.

### 7.6 VMC-required mode

- **FR-26.** When the user has enabled "require VMC" and a BIMI logo is resolved but VMC verification fails (or no certificate URL is present), the logo shall be discarded and the application shall fall through to the next avatar source in the priority chain.

### 7.7 DMARC gating

- **FR-27.** When the user has enabled the global "require DMARC for external images" option, BIMI logo fetches shall be suppressed for messages whose DMARC result is not a pass, consistent with the same gating applied to Gravatars, Libravatars, and favicons.

### 7.8 Caching

- **FR-28.** Resolved BIMI results (bitmap, source type, and verified status) shall be cached locally. The cache key shall incorporate the sender's domain. The cache lifetime shall be comparable to other favicon/avatar caches (on the order of days to weeks).
- **FR-29.** Toggling the BIMI or VMC-required settings shall clear the avatar cache.

### 7.9 Build-variant restrictions

- **FR-30.** Certain distribution channels (e.g. those with restrictive store policies that prohibit external avatar fetches) shall disable BIMI at build time, hiding the setting entirely from the user. This is consistent with the same restriction applied to Gravatars and Libravatars.

## 8. Non-Functional Requirements

- **NFR-1. Privacy.** BIMI logo resolution involves DNS queries and HTTPS fetches to URLs controlled by the sender. No user data (email address, IP, etc.) beyond what is inherent in the DNS/HTTPS request is transmitted. The feature shall be opt-in only.
- **NFR-2. Performance.** BIMI DNS lookups and image fetches shall be performed asynchronously and shall not block the rendering of the message list. Timeouts shall be enforced (connection timeout ~10 seconds, read timeout ~15 seconds).
- **NFR-3. Resilience.** A failure to resolve a BIMI logo (DNS timeout, HTTP error, unparseable SVG, invalid certificate) shall be silently absorbed. The application shall fall through to the next avatar source without displaying an error to the user.
- **NFR-4. Scale.** The feature shall handle inboxes with hundreds of distinct sender domains without excessive network traffic, relying on the local cache to amortize repeated lookups.
- **NFR-5. Security.** All BIMI-related network requests shall use HTTPS. The VMC validation chain shall enforce standard PKIX path validation. The SVG renderer shall not execute embedded scripts or external references.
- **NFR-6. Accessibility.** The verified/unverified badge shall be labeled for screen readers. Color shall not be the sole differentiator between verified and unverified states (icon shape/level shall also differ).
- **NFR-7. Consistency.** A BIMI logo resolved for a given sender shall appear identically in every surface (message list, conversation, notification, widget). The verified badge shall appear in the same surfaces where authentication results are displayed.

## 10. Open Questions

- **OQ-1. Certificate revocation.** The current behavior disables revocation checking for VMC certificates (no CRL or OCSP). Should a future version enable revocation checking, and if so, should it be soft-fail (treat unreachable responders as valid) or hard-fail? This has performance and reliability implications.
- **OQ-2. Bundled VMC root CA list.** The application bundles a small set of VMC-specific root CA certificates (currently DigiCert and Entrust). The process for updating this list when new CAs are accredited by the BIMI Group is undefined — should it be tied to application releases, or should there be a mechanism for out-of-band updates?
- **OQ-3. Unverified badge visibility.** The "unverified sender" indicator (shown when the avatar source is BIMI/VMC but verification failed) is currently visible only in debug builds. Should it be surfaced to end users as well, or is the absence of a verified badge sufficient?
- **OQ-4. BIMI in the DMARC/authentication details screen.** The DMARC diagnostic screen fetches the `default._bimi.<domain>` record and displays it. Should this be expanded to also attempt VMC validation and display the result, providing a one-stop diagnostic for authentication and BIMI?
- **OQ-5. SVG security.** The SVG rendering path must not execute scripts, fetch external resources, or process entity expansion attacks. The exact sanitization guarantees of the SVG renderer should be documented and tested as part of implementation.
- **OQ-6. Interaction with "prefer contact photo" setting.** When the user has "prefer contact photo" enabled, the current behavior unconditionally prefers the address-book photo over BIMI. Should there be an option to prefer the BIMI logo even when a contact photo exists, for users who value verified brand logos over potentially outdated contact photos?
- **OQ-7. Cache duration tuning.** The BIMI cache currently shares the same duration as the general favicon cache (approximately two weeks). Should BIMI have a shorter cache to pick up logo changes more promptly, or is the shared duration acceptable?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Opt-in by default.** BIMI is disabled by default because enabling it causes the application to make DNS queries and HTTPS requests to URLs published by the sender's domain. This is consistent with the application's treatment of all external avatar sources (Gravatars, Libravatars, favicons) — none are enabled by default. The user must explicitly opt in, and a privacy notice is provided.

- **N-2. VMC verification is a separate toggle, not mandatory.** Many legitimate senders publish BIMI logos without purchasing a VMC certificate (VMCs are expensive and issued by a small number of CAs). Requiring VMC for all BIMI logos would dramatically reduce the feature's usefulness. The two-toggle design lets security-focused users require VMC while letting others benefit from unverified logos. "Require VMC" defaults to off.

- **N-3. BIMI before Gravatar/favicon in the priority chain.** BIMI logos are fetched before Gravatars, Libravatars, and favicons in the avatar-resolution pipeline. This is intentional: a BIMI logo is an authoritative brand asset published by the domain owner, whereas a Gravatar is a user-chosen image tied to an email hash and a favicon is scraped from a website. The authoritative source should win. The first successful result in the ordered pipeline is used; subsequent sources are not consulted.

- **N-4. DMARC enforcement for VMC verification.** The BIMI specification requires that a domain pass DMARC with an enforcement policy (`quarantine` or `reject`) for VMC-based verification to be valid. The application enforces this by performing a DMARC DNS lookup as part of VMC validation — even after the certificate chain itself validates. A `none` policy or a `pct` value less than 100 causes verification to fail. This prevents an attacker from publishing a valid VMC while running DMARC in report-only mode.

- **N-5. Embedded logo from certificate preferred.** When a VMC contains a Logotype extension with an embedded SVG (as a data URI), the application uses that image rather than the one referenced by the `l` tag in the DNS record. This is correct per the BIMI specification: the logo in the certificate is the one that was attested by the CA during issuance. The DNS `l` tag serves as a fallback or as the logo source when no VMC is present.

- **N-6. Junk folder exclusion.** BIMI lookups are suppressed for messages in the Spam/Junk folder. This prevents spammers from exploiting BIMI to make their messages appear legitimate. The exclusion is unconditional — even a message with a valid VMC in the Junk folder will not display a BIMI logo.

- **N-7. Build-variant gating.** Certain distribution channels prohibit external network requests for contact/avatar images as a matter of store policy. The application conditionally compiles BIMI support out of these builds entirely, replacing the real implementation with a stub that always returns no result. This means the feature — including its settings UI — is invisible on those builds.

- **N-8. Shared DMARC gate.** The "require DMARC for external images" setting is a global gate that applies to all external avatar sources equally (BIMI, Gravatar, Libravatar, favicon, etc.). When enabled, a message that did not pass DMARC will have none of these sources consulted. This is a defense-in-depth measure: it prevents information leakage (the HTTPS fetch to the avatar URL) for unauthenticated mail.
