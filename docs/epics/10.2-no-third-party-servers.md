# Epic: No Third-Party Servers

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **10.2** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"No third-party servers. All data stays local; sync goes only to the user's mail server."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email client occupies a uniquely sensitive position in a user's digital life: it stores credentials, personal correspondence, financial notifications, legal documents, and a social graph of every person the user has ever emailed. The user has already placed trust in their mail provider — they should not be forced to extend that trust to any additional party simply by choosing an email client.

The "No third-party servers" principle exists to guarantee that, in its default configuration, the application communicates exclusively with the user's own mail servers (IMAP, POP3, SMTP) and stores all data locally on the user's device. No message content, metadata, contact information, credentials, usage patterns, or behavioral signals leave the device unless the user takes an explicit, informed action to enable an optional service that requires external communication.

This principle is the foundational privacy contract of the application. Every other privacy feature (tracker blocking, remote image suppression, link sanitization, EXIF stripping) builds on the assumption that the application itself is not a source of data leakage. Without this guarantee, no other privacy feature is credible.

## 2. Goals

- **G1.** In the default, out-of-the-box configuration, the application shall communicate only with the user's configured mail servers and with no other remote endpoint.
- **G2.** Every optional feature that contacts an external service shall be disabled by default, clearly disclosed, and individually controllable by the user.
- **G3.** The user shall be able to use the full core email workflow — reading, composing, searching, organizing, synchronizing — without any data leaving the device except mail-server traffic.
- **G4.** The application shall not embed analytics, telemetry, crash reporting, or advertising that phones home in its default state.
- **G5.** The architecture shall make it structurally difficult for new features to silently introduce third-party network calls without explicit opt-in gating.
- **G6.** Distribution-channel differences (e.g. a community build vs. a vendor-store build) shall not weaken the default privacy posture; they may restrict which optional services are available, but the baseline of "no third-party traffic by default" shall hold across all channels.

## 3. Non-Goals

- **NG1.** This epic does **not** prohibit the application from ever communicating with third-party servers. It constrains the *default* and requires *explicit opt-in* for any exception.
- **NG2.** This epic does **not** define the behavior of any individual optional service (translation, AI summarization, avatar fetching, etc.). Each such service has its own feature scope; this epic defines only the gating contract that all of them must satisfy.
- **NG3.** This epic does **not** cover network-level privacy measures such as VPN binding, Tor routing, or DNS-over-HTTPS configuration. Those are connectivity concerns, not application-level data-flow concerns.
- **NG4.** This epic does **not** prescribe how credentials are stored at rest (that is a security/encryption concern, not a third-party-server concern).
- **NG5.** This epic does **not** address what the user's mail server itself does with data once it receives it. The trust boundary ends at the mail server.

## 4. Glossary

| Term | Definition |
|---|---|
| **Mail server** | The user's configured IMAP, POP3, SMTP, or JMAP endpoint — the only remote system the application contacts by default. |
| **Third-party server** | Any remote endpoint that is not the user's mail server and not required by the mail protocols themselves. |
| **Optional service** | A feature that, when enabled, causes the application to contact a third-party server (e.g. translation, avatar lookup, blocklist download, error reporting). |
| **Opt-in gate** | A user-facing setting, defaulting to off, that must be explicitly enabled before the application will contact the associated third-party server. |
| **Sub-processor** | In the GDPR sense, a third party that processes data on behalf of the application. All sub-processors are gated behind opt-in settings. |
| **Distribution channel** | A build variant distributed through a specific store or mechanism (e.g. community/GitHub build, vendor app store, F-Droid). Channels may differ in which optional services are compiled in, but not in default privacy posture. |
| **Remote content** | Any resource embedded in or referenced by an email message that requires fetching from a server other than the user's mail server — images, fonts, CSS, scripts, tracking pixels. |
| **Core email workflow** | The set of operations a user performs using only standard mail protocols: reading messages, composing, replying, forwarding, searching, organizing into folders, flagging, deleting, and synchronizing. |

## 5. Personas

- **P1. The privacy-conscious user.** Chose this application specifically because of its privacy stance. Expects zero third-party traffic in the default configuration and wants to verify this claim. Will audit settings and network behavior.
- **P2. The default user.** Installed the application, set up accounts, and started using it without changing any settings. Expects their data to be safe without having to understand or configure privacy options.
- **P3. The power user who opts in selectively.** Understands the trade-offs and wants to enable specific optional services (e.g. AI summarization via a local model, or avatar fetching for known contacts) while keeping everything else locked down.
- **P4. The compliance officer.** Evaluating the application for use in an organization that requires no data to leave the device except to approved mail servers. Needs to confirm the default posture and understand every possible exception.

## 6. User Stories

### 6.1 Default privacy posture

> **US-1.** As a default user, when I install the application and configure my mail account, I want the application to communicate only with my mail server, so that no third party learns that I use this application, what I read, or whom I email.

> **US-2.** As a privacy-conscious user, I want the application to contain no analytics, telemetry, or crash-reporting that is active by default, so that I do not have to hunt through settings to disable tracking.

> **US-3.** As a default user, I want the application to not use cloud-based push notification infrastructure by default, so that a third party does not learn when I receive mail or from whom.

> **US-4.** As any user, I want the application to perform no automatic update checks against a third-party server unless I have explicitly enabled that feature, so that my usage of the application is not disclosed to any external service.

### 6.2 Remote content blocking

> **US-5.** As any user, when I open an email containing remote images, fonts, or other externally hosted resources, I want the application to block all such content by default and show me a clear prompt before loading it, so that the sender cannot track whether or when I opened the message.

> **US-6.** As any user, I want the option to allow remote content on a per-message, per-sender, or per-domain basis, so that I can choose convenience for trusted senders without lowering my default protection.

> **US-7.** As a privacy-conscious user, I want the application to detect and flag likely tracking pixels (very small or invisible images) separately from regular remote images, so that I can make an informed decision about what to load.

### 6.3 Link and web content handling

> **US-8.** As any user, when I click a link in an email, I want the application to show me a confirmation dialog with the actual destination URL before opening it, so that I am not silently redirected through tracking services.

> **US-9.** As any user, I want the application to not route link clicks through any proxy or intermediary server operated by the application developer or any third party, so that my browsing intent stays between me and the destination.

> **US-10.** As any user, I want the option to have known tracking parameters stripped from URLs before they are opened, so that the destination site receives less information about how I arrived.

### 6.4 Account setup and provider discovery

> **US-11.** As any user, when I set up a new account using the quick-setup wizard, I want the application to first attempt to discover my provider's settings from a bundled, offline provider database, so that my email domain is not disclosed to any third-party service by default.

> **US-12.** As any user, if my provider is not in the bundled database and the application needs to query an external autoconfiguration service, I want to be informed before that query is made, so that I can choose to enter settings manually instead.

### 6.5 Optional services — opt-in gating

> **US-13.** As any user, I want every feature that contacts a third-party server to be clearly labeled in the settings with a description of what data is sent and to whom, so that I can make informed choices.

> **US-14.** As a power user, I want to be able to enable optional services individually (e.g. enable translation but not avatar fetching, or enable a local AI model but not a cloud one), so that I control exactly which third parties receive my data.

> **US-15.** As any user, when I enable an optional service that contacts a third-party server, I want a one-time confirmation that tells me what data will be shared and with whom, so that I do not enable it accidentally.

> **US-16.** As a privacy-conscious user, I want to be able to disable any optional service I have previously enabled, and have the application immediately stop contacting that service, so that my opt-in is fully reversible.

### 6.6 OAuth and provider authentication

> **US-17.** As a user who authenticates via OAuth (e.g. for a major mail provider), I understand that the OAuth flow contacts my mail provider's authentication servers. I want this to be the only external contact required for authentication, and I want no additional third-party servers to be involved in the OAuth flow beyond the provider itself.

### 6.7 Safe browsing and web view

> **US-18.** As any user, I want any built-in web view used to render email content to have cloud-based safe-browsing services disabled by default, so that the contents of my emails are not sent to a third party for URL checking.

> **US-19.** As a power user, I want the option to enable cloud-based safe-browsing if I choose to, understanding that it involves sending URL data to a third party.

### 6.8 Error reporting

> **US-20.** As any user, I want error and crash reporting to be completely disabled by default, so that no diagnostic data leaves my device without my consent.

> **US-21.** As a user who wants to help improve the application, I want to be able to opt in to error reporting, with a clear explanation of what data will be sent and to whom.

### 6.9 Distribution channel parity

> **US-22.** As any user, regardless of which distribution channel I installed the application from, I want the same default privacy posture: no third-party traffic until I opt in, so that choosing a different install source does not silently weaken my privacy.

> **US-23.** As a user of a community or freedom-respecting build, I want certain optional services that rely on proprietary cloud providers to be absent entirely, so that there is no possibility of accidental enablement.

### 6.10 Transparency and auditability

> **US-24.** As a privacy-conscious user, I want the application to maintain a clear, accessible privacy policy that enumerates every third-party service the application can contact, under what conditions, and what data is sent, so that I can audit the application's behavior.

> **US-25.** As a compliance officer, I want to be able to confirm that the list of possible third-party connections is complete and that each one is gated behind an explicit user action, so that I can approve the application for organizational use.

## 7. Functional Requirements

### 7.1 Default network posture

- **FR-1.** In its default configuration (all settings at their initial values), the application shall initiate network connections only to the user's configured mail servers (IMAP, POP3, SMTP, JMAP) and to DNS resolvers necessary for those connections.
- **FR-2.** The application shall not embed or activate any analytics, telemetry, usage tracking, or behavioral profiling in its default configuration.
- **FR-3.** The application shall not use cloud-based push notification services (e.g. vendor push messaging infrastructure) in its default configuration. Mail arrival notifications shall be driven by direct IMAP IDLE connections or local polling.
- **FR-4.** The application shall not perform automatic update checks against external servers in its default configuration. If an update-check feature exists, it shall be opt-in.

### 7.2 Remote content in messages

- **FR-5.** The application shall block all remotely hosted content in email messages (images, fonts, CSS, iframes, scripts, objects, media) by default.
- **FR-6.** When remote content is blocked, the application shall display a visible, non-modal indicator and offer a one-action control to load remote content for the current message.
- **FR-7.** The application shall support per-sender and per-domain allow-lists so that the user can permanently authorize remote content from trusted sources.
- **FR-8.** The application shall detect likely tracking pixels (images below a configurable size threshold or matching known tracker patterns from bundled blocklists) and report them to the user as distinct from regular images.
- **FR-9.** When the user authorizes remote content for a message, the application shall fetch that content directly from the hosting server — not through any intermediary proxy operated by the application developer or a third party.

### 7.3 Link handling

- **FR-10.** The application shall present a confirmation dialog before opening any link contained in an email message.
- **FR-11.** The confirmation dialog shall display the actual destination URL, clearly showing the domain, and shall warn if the displayed link text differs from the actual URL target.
- **FR-12.** The application shall not route link navigation through any proxy, redirect, or intermediary server.
- **FR-13.** The application shall offer an optional setting to strip known tracking parameters from URLs before opening them. This setting shall use a bundled parameter list and shall not require contacting an external server to function.

### 7.4 Account setup — provider discovery

- **FR-14.** The application shall ship with a bundled, offline provider database containing known server settings for common mail providers.
- **FR-15.** During account setup, the application shall first attempt provider discovery using the bundled database, without any network call to a third party.
- **FR-16.** If the bundled database does not contain the user's provider and the application offers external autoconfiguration lookup, that lookup shall require explicit user consent before executing and shall disclose that the user's email domain will be sent to the autoconfiguration service.
- **FR-17.** Manual server configuration shall always be available as an alternative to any automated discovery, allowing the user to avoid all third-party contact during setup.

### 7.5 Optional service gating

- **FR-18.** Every feature that contacts a third-party server shall be governed by an independent opt-in setting that defaults to **disabled**.
- **FR-19.** Each opt-in setting shall be accompanied by a description visible in the settings interface that names the third-party service, describes what data is transmitted, and links to or references the service's privacy policy.
- **FR-20.** Enabling an optional service shall require an affirmative user action (e.g. toggling a switch, confirming a dialog). The application shall not enable services implicitly as a side effect of another action.
- **FR-21.** Disabling a previously enabled optional service shall take effect immediately: the application shall cease contacting that service and shall not retain any authorization state that could cause it to resume without re-enablement.
- **FR-22.** The application shall maintain a single, consolidated location in its settings where the user can review the state of all optional-service toggles.

### 7.6 Categories of optional services

The following categories of optional services are recognized. Each must satisfy FR-18 through FR-22:

- **FR-23.** *Error/crash reporting*: transmits diagnostic data (stack traces, device information, application state) to a developer-operated error-reporting service. Disabled by default.
- **FR-24.** *Translation services*: transmits message text to a cloud translation provider. Disabled by default; activated only by an explicit user action on a specific message.
- **FR-25.** *AI/LLM services*: transmits message text or compose context to a cloud AI provider (or a user-configured local model endpoint). Disabled by default; requires the user to configure an endpoint and/or API key before use.
- **FR-26.** *Avatar/contact-photo services*: fetches sender profile images from remote avatar services using a hash of the sender's email address. Disabled by default.
- **FR-27.** *Favicon/brand-indicator services*: fetches website icons or brand logos from sender domains. Disabled by default.
- **FR-28.** *Spam blocklist lookups*: queries third-party DNS-based blocklists to check sender reputation. Disabled by default.
- **FR-29.** *Spell-checking/grammar services*: transmits composed text to a cloud language-checking service. Disabled by default.
- **FR-30.** *File/attachment scanning services*: transmits file hashes or file contents to a cloud scanning service. Disabled by default.
- **FR-31.** *Password breach checking*: transmits a partial hash of a password to a breach-checking service. Disabled by default; activated only by an explicit user action.
- **FR-32.** *Blocklist/filter-list updates*: downloads updated tracker blocklists or tracking-parameter lists from a known source. Disabled by default or bundled with application updates; if fetched separately, requires opt-in.
- **FR-33.** *Application update checks*: contacts a release server to check for newer versions. Disabled by default (or absent entirely in builds where the system package manager handles updates).
- **FR-34.** *Cloud-based safe browsing*: sends URLs viewed in the built-in message renderer to a cloud safe-browsing service for phishing/malware checks. Disabled by default.
- **FR-35.** *IP geolocation lookups*: resolves IP addresses found in message headers to geographic locations via a third-party service. Disabled by default; activated only by an explicit user action per lookup.

### 7.7 OAuth authentication

- **FR-36.** OAuth authentication flows shall contact only the user's mail provider's authorization and token endpoints. No additional third-party servers shall be involved in the authentication process.
- **FR-37.** OAuth token refresh shall occur directly between the application and the provider's token endpoint, with no intermediary.

### 7.8 Distribution channel invariants

- **FR-38.** All distribution channels shall ship with the same default privacy posture: no third-party network traffic until the user opts in.
- **FR-39.** A distribution channel may restrict the set of available optional services (e.g. a freedom-respecting build may omit proprietary cloud AI endpoints entirely), but shall not expand the set of services that are enabled by default.
- **FR-40.** If a distribution channel includes platform-mandated services (e.g. vendor billing or vendor backup infrastructure), those services shall be disabled by default within the application's own settings, even if the platform itself may activate them at the OS level.

### 7.9 Data locality

- **FR-41.** All message content, metadata, attachments, contact data, search indexes, drafts, and application settings shall be stored exclusively on the user's local device (and, via standard mail-protocol sync, on the user's mail server). No application data shall be stored on or transmitted to any server operated by the application developer or a third party in the default configuration.
- **FR-42.** If the application offers a cloud backup feature, it shall be opt-in, user-directed (the user chooses the destination), and encrypted before transmission.

### 7.10 Minimizing incidental disclosure

- **FR-43.** The application shall offer an option to use a generic user-agent string when fetching remote content (for cases where the user has opted to load remote images), so that the email client's identity is not disclosed to remote servers.
- **FR-44.** The application shall offer an option to suppress or genericize timezone and locale information in outgoing message headers, to reduce fingerprinting surface.

## 8. Non-Functional Requirements

- **NFR-1. Auditability.** The application shall maintain a complete, user-accessible privacy policy or equivalent document that enumerates every third-party service the application can contact, the conditions under which contact occurs, and what data is transmitted. This document shall be updated whenever a new optional service is added.
- **NFR-2. Verifiability.** The application's source code shall be available for inspection, enabling independent verification that the default configuration produces no third-party network traffic.
- **NFR-3. Structural enforcement.** The architecture should make it straightforward for a reviewer to identify all points where third-party network calls are initiated and confirm that each is gated behind an opt-in check. A new developer adding a feature should find it natural to gate any new external call behind an opt-in setting.
- **NFR-4. Regulatory compliance.** The default posture and opt-in gating shall satisfy GDPR requirements for data minimization (Article 5(1)(c)) and consent (Article 6(1)(a)), as well as comparable provisions in CCPA, VCDPA, LGPD, and similar regulations.
- **NFR-5. Performance.** The absence of third-party calls in the default configuration shall not degrade the user experience. Core email operations shall function at full speed without any external service dependency.
- **NFR-6. Resilience.** The application shall remain fully functional for all core email operations when third-party services are unreachable, whether because they are disabled, the service is down, or the network blocks them. Optional features that depend on third-party services shall degrade gracefully (clear error message, no crash, no retry loop).
- **NFR-7. Transparency of state.** The user shall be able to determine, at any point, which optional services are currently enabled and which third-party connections are active or have recently been made.

## 10. Open Questions

- **OQ-1. Autoconfiguration consent flow.** The source application appears to query the Thunderbird/Mozilla autoconfiguration service during quick setup without a separate, explicit consent step. Should the desktop application require an interstitial consent prompt before this lookup, or is the act of using the quick-setup wizard itself considered sufficient consent? The conservative interpretation (prompt before lookup) is assumed in FR-16, but this should be confirmed.
- **OQ-2. Bundled blocklist freshness.** Tracker blocklists and tracking-parameter lists are bundled with the application but can also be updated from an external source. Should the application offer automatic background updates for these lists (opt-in), or should updates be delivered only through application upgrades? The source application supports both, gated behind a setting.
- **OQ-3. DNS-based blocklist queries as "third-party" traffic.** Spam blocklist lookups (e.g. querying Spamhaus or Spamcop via DNS) use standard DNS rather than HTTPS. These queries are structurally different from HTTP API calls but still contact third-party infrastructure. The current treatment (opt-in, disabled by default) seems correct, but the boundary should be explicitly documented.
- **OQ-4. Platform-mandated telemetry.** Some distribution channels include platform-level telemetry that the application cannot fully disable (e.g. vendor billing infrastructure that phones home). Should the privacy documentation explicitly disclaim these platform-level behaviors, or should the application attempt to suppress them where possible? The source application documents one such case (vendor billing endpoint) as being outside its control.
- **OQ-5. OAuth client registration.** OAuth flows for major providers require the application to register as a client with each provider, embedding client IDs in the application. These client IDs identify the application (not the user) to the provider. Should this be disclosed in the privacy documentation as a form of application identification, even though no user data beyond what the mail protocol already requires is transmitted?
- **OQ-6. Favicon and BIMI domain disclosure.** When the user enables favicon or BIMI fetching, the application contacts the sender's domain directly (not a third-party aggregator). This discloses to the sender's web server that someone at the user's IP address uses this email client. Should this be treated as "third-party server" contact (since the sender's web server is not the user's mail server), or as a distinct category? The source application treats these as opt-in features, which is consistent with the conservative interpretation.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. "Third-party" means anything beyond the user's mail server.** The source application draws the trust boundary tightly: the user's configured IMAP/POP3/SMTP servers are trusted; everything else is third-party, including the application developer's own infrastructure. This is a deliberately conservative definition that maximizes user trust.

- **N-2. Opt-in, not opt-out.** The source application consistently defaults every external service to disabled and requires an affirmative action to enable it. This is the opposite of the industry norm (enable by default, let the user opt out) and is the single most important architectural decision for the privacy posture. Any deviation from this pattern — even for a seemingly benign service — would undermine the feature's credibility.

- **N-3. Per-service granularity, not a global "privacy mode."** The source application does not have a single "strict privacy" toggle that disables everything. Instead, each service has its own independent toggle. This allows the power user to compose a precise set of trade-offs (e.g. enable translation but not avatars). A global toggle might seem simpler but would force an all-or-nothing choice that does not match real-world usage.

- **N-4. Distribution-channel variation is subtractive, not additive.** Different builds of the source application may remove optional services (e.g. the F-Droid build omits cloud AI endpoints entirely), but no build adds services that are enabled by default. The invariant is: the privacy floor is the same everywhere; the ceiling varies by channel.

- **N-5. No intermediary proxies.** The source application never proxies remote content, link clicks, or image fetches through a server operated by the developer. When the user authorizes loading remote images, the application fetches them directly from the originating server. This avoids creating a centralized observation point and avoids the operational burden of running proxy infrastructure.

- **N-6. Bundled data over network fetches.** Where possible, the source application bundles reference data (provider database, tracker blocklists, tracking-parameter lists, CA root list, public suffix list) with the application rather than fetching them at runtime. This eliminates a class of third-party calls entirely. The trade-off is staleness; the source application mitigates this by offering optional, user-initiated updates.

- **N-7. OAuth is provider traffic, not third-party traffic.** The source application treats OAuth endpoints (e.g. Google's OAuth server for Gmail, Microsoft's OAuth server for Outlook) as part of the user's mail provider infrastructure, not as third-party servers. This is a pragmatic distinction: the user has already entrusted their email to this provider, and OAuth is the provider's own authentication mechanism. The application does not involve any additional parties in the OAuth flow.

- **N-8. Disclosure is not just a legal obligation — it is a feature.** The source application's privacy policy exhaustively enumerates every possible third-party connection, including the trigger condition and the data transmitted. This level of transparency is itself a user-facing feature: it enables auditability and builds trust. The desktop application should maintain an equivalent document and keep it synchronized with the actual codebase.
