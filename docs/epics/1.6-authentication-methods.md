# Epic: Authentication Methods

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.6** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"PLAIN, LOGIN, CRAM-MD5, NTLM, OAuth/XOAUTH2, APOP, and EXTERNAL (RFC 4959, e.g. for client-cert auth)."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email client must authenticate with one or more remote servers (incoming mail retrieval and outgoing mail submission) before it can do anything useful. Different mail providers support different authentication mechanisms — some legacy, some modern, some mandated by enterprise policy, some required by provider-specific security models. A client that supports only one or two mechanisms will lock out a meaningful fraction of users.

The Authentication Methods feature exists to give the application a broad, negotiable repertoire of authentication mechanisms so that it can connect to as wide a range of mail servers as possible — from modern OAuth-protected cloud providers down to legacy corporate servers that still require challenge-response or domain-scoped credentials — while keeping the experience seamless for users who neither know nor care which mechanism is in play.

The feature spans three independent axes:

1. **Password-based mechanisms** — where the user supplies a static credential and the client and server negotiate how to present it (PLAIN, LOGIN, CRAM-MD5, NTLM, APOP).
2. **Token-based mechanisms** — where the client obtains a short-lived bearer token through a browser-mediated authorization flow and presents it via the XOAUTH2 protocol extension (OAuth 2.0).
3. **Certificate-based mechanisms** — where authentication is performed at the transport layer by presenting a client certificate during the TLS handshake, with no password or token required (EXTERNAL / RFC 4959).

The application must support all three axes, select the strongest available mechanism automatically, and expose enough control for power users and enterprise administrators to override the selection when necessary.

## 2. Goals

- **G1.** Support all widely deployed IMAP, POP3, and SMTP authentication mechanisms: PLAIN, LOGIN, CRAM-MD5, NTLM, XOAUTH2, APOP, and EXTERNAL (client certificate).
- **G2.** Negotiate the authentication mechanism automatically based on the server's advertised capabilities, preferring stronger mechanisms over weaker ones, so that the user does not have to choose a mechanism manually in the common case.
- **G3.** Provide first-class, browser-mediated OAuth 2.0 sign-in flows for major providers (Gmail, Outlook / Microsoft 365, Yahoo, AOL, and others), with automatic token refresh transparent to the user.
- **G4.** Support client-certificate authentication for environments that use mutual TLS, allowing the user to select an installed certificate and authenticate without a password.
- **G5.** Allow advanced users to enable or disable individual password-based mechanisms globally, so that problematic or unwanted mechanisms can be suppressed without reconfiguring accounts.
- **G6.** Ensure that no password-based credential is ever transmitted in the clear over an unencrypted connection by default.

## 3. Non-Goals

- **NG1.** This epic does **not** define the transport-layer encryption strategy (TLS versions, cipher suites, certificate pinning, DANE). Those are covered by the TLS Hardening feature (7.9). This epic assumes that a secure transport is available and concerns itself only with what happens *after* the connection is established.
- **NG2.** This epic does **not** define the credential storage model (encryption-at-rest, master password, system keyring integration). It only requires that credentials and tokens are available to the authentication subsystem when needed.
- **NG3.** Full SASL framework support (arbitrary SASL mechanisms beyond CRAM-MD5) is **not** a goal. The mail protocol libraries available on the target platform do not expose a general-purpose SASL negotiation API; only the mechanisms listed in G1 are in scope.
- **NG4.** Two-factor authentication flows (TOTP, U2F, SMS codes) are **not** in scope. Where a provider requires 2FA, the expectation is that it is handled inside the OAuth browser flow or through app-specific passwords — not by the email client itself.
- **NG5.** Server-to-server authentication, relay authentication, or SMTP submission authorization beyond what the user's identity requires is out of scope.

## 4. Glossary

| Term | Definition |
|---|---|
| **Authentication mechanism** | A protocol-level method by which the client proves its identity to the server. Examples: PLAIN, LOGIN, CRAM-MD5, NTLM, XOAUTH2, APOP, EXTERNAL. |
| **PLAIN** | A SASL mechanism (RFC 4616) that transmits the username and password in Base64 encoding. Secure only when used over an encrypted transport. |
| **LOGIN** | A legacy mechanism similar to PLAIN that transmits the username and password in two separate Base64-encoded steps. Widely supported but not formally standardized as a SASL mechanism. |
| **CRAM-MD5** | A challenge-response mechanism (RFC 2195) where the server sends a challenge and the client replies with an MD5-keyed hash of the challenge using the password as the key. The password is never sent over the wire. |
| **NTLM** | A Microsoft proprietary challenge-response mechanism used in Windows domain environments. Requires a domain/realm in addition to username and password. |
| **XOAUTH2** | A vendor-originated SASL mechanism that carries an OAuth 2.0 access token instead of a password. Used by Gmail, Outlook, Yahoo, and other providers that support OAuth for mail protocols. |
| **APOP** | Authenticated Post Office Protocol (RFC 1939 §7). A POP3-specific mechanism where the server includes a unique timestamp in its greeting and the client replies with an MD5 hash of that timestamp plus the password. |
| **EXTERNAL** | A SASL mechanism (RFC 4422 §A, RFC 4959) that delegates authentication to an external channel — in practice, the TLS client certificate presented during the handshake. No password is needed. |
| **OAuth 2.0** | An authorization framework (RFC 6749) in which the user authorizes the application via a browser-based flow, and the application receives a short-lived access token and a long-lived refresh token. |
| **Access token** | A short-lived bearer credential obtained via OAuth 2.0, presented to the mail server using the XOAUTH2 mechanism. Typically valid for approximately one hour. |
| **Refresh token** | A long-lived credential obtained alongside the access token, used to obtain new access tokens without re-prompting the user. |
| **Client certificate** | An X.509 certificate installed on the user's machine, presented during the TLS handshake to authenticate the client to the server (mutual TLS). |
| **Realm** | A domain or scope identifier used by CRAM-MD5 and NTLM to qualify the authentication context (e.g. a Windows domain name). |
| **App-specific password** | A provider-generated password intended for use by third-party applications when the provider requires 2FA for browser logins but does not support OAuth for mail protocols. Created by the user in the provider's security settings, not by the email client. |
| **Capability advertisement** | The list of supported features and authentication mechanisms that a mail server announces at connection time (IMAP CAPABILITY, SMTP EHLO, POP3 greeting/CAPA). |

## 5. Personas

- **P1. The mainstream user.** Has one or two accounts with major providers (Gmail, Outlook, Yahoo). Expects to sign in via a familiar browser-based flow and never think about authentication again. Does not know what PLAIN or CRAM-MD5 means.
- **P2. The self-hosted user.** Runs their own mail server or uses a smaller provider. Authenticates with a username and password. May need CRAM-MD5 or a specific mechanism that their server requires.
- **P3. The enterprise user.** Works in a corporate environment with an Exchange/Outlook server. May need NTLM for domain authentication. May have a client certificate issued by their IT department for mutual TLS.
- **P4. The security-conscious user.** Wants OAuth wherever possible, wants to disable weak mechanisms globally, wants assurance that passwords are never sent in the clear.
- **P5. The legacy-server user.** Connects to an older POP3 server that requires APOP or to a server that only advertises LOGIN. Needs the application to accommodate their server's limitations.

## 6. User Stories

### 6.1 Password-based authentication (common path)

> **US-1.** As a mainstream user adding an account with a username and password, I want the application to automatically negotiate the strongest authentication mechanism that both the application and the server support, so that I do not have to understand or choose a mechanism myself.

> **US-2.** As a self-hosted user, when my server requires a specific mechanism (e.g. CRAM-MD5), I want the application to detect that requirement from the server's capability advertisement and use it without my intervention.

> **US-3.** As any user, when I provide a username and password during account setup, I want the application to test the connection and confirm that authentication succeeds before saving the account, so that I am not left with a broken configuration.

### 6.2 OAuth / XOAUTH2

> **US-4.** As a mainstream user adding a Gmail, Outlook, Yahoo, or other supported provider account, I want the application to offer a browser-based sign-in option that lets me authorize access without typing my password into the email client, so that my credentials stay with my provider.

> **US-5.** As any user with an OAuth-authenticated account, I want the application to refresh my access token automatically in the background, so that I am never interrupted by expiration unless my authorization has been genuinely revoked.

> **US-6.** As any user whose OAuth authorization has been revoked or whose refresh token has expired, I want the application to notify me clearly and offer a one-click re-authorization flow, so that I can restore access without deleting and re-adding the account.

> **US-7.** As any user with an OAuth-authenticated account, I want the option to switch to password-based authentication if I prefer (e.g. using an app-specific password), so that I am not locked into OAuth.

> **US-8.** As any user, when a provider supports OAuth, I want to see the provider's privacy policy link during the authorization flow, so that I can make an informed decision.

### 6.3 Client certificate (EXTERNAL)

> **US-9.** As an enterprise user whose organization uses client-certificate authentication, I want to select an installed certificate in the account settings and authenticate without providing a password, so that I can use mutual TLS as my sole authentication method.

> **US-10.** As any user, I want the ability to select a client certificate independently for the incoming-mail connection and the outgoing-mail connection, so that I can use different certificates if my infrastructure requires it.

> **US-11.** As any user who has selected a client certificate, I want to be able to clear the selection and revert to password-based authentication, so that I am not locked in.

### 6.4 NTLM / domain authentication

> **US-12.** As an enterprise user on a Windows domain, I want to be able to specify a domain/realm alongside my username and password, so that NTLM authentication succeeds against my organization's Exchange server.

### 6.5 APOP (POP3-specific)

> **US-13.** As a legacy-server user whose POP3 server requires APOP, I want to enable APOP support and have the application use it for that connection, so that I can connect to my server.

> **US-14.** As any user, I want APOP to be disabled by default, since it relies on a weak hash and is rarely needed, so that it is not attempted unnecessarily against servers that do not support it.

### 6.6 Mechanism control

> **US-15.** As a security-conscious user, I want to be able to disable specific password-based mechanisms (PLAIN, LOGIN, NTLM, CRAM-MD5, APOP) globally in the application's advanced settings, so that the application never attempts a mechanism I consider unacceptable.

> **US-16.** As any user, when I disable a mechanism that my server requires, I want to receive a clear authentication error that helps me understand the cause, so that I can re-enable the mechanism or adjust my server settings.

### 6.7 Security baseline

> **US-17.** As any user, I want the application to refuse to send my password using PLAIN or LOGIN over an unencrypted connection by default, so that my credentials are not exposed on the network.

> **US-18.** As a power user who knowingly connects to a server without encryption (e.g. on a trusted local network), I want an explicit "allow insecure connections" override per account, so that the application does not prevent me from connecting, but I have to opt in.

## 7. Functional Requirements

### 7.1 Supported mechanisms

- **FR-1.** The application shall support the following authentication mechanisms for **incoming mail (IMAP)**: PLAIN, LOGIN, CRAM-MD5, NTLM, XOAUTH2, and EXTERNAL (client certificate).
- **FR-2.** The application shall support the following authentication mechanisms for **incoming mail (POP3)**: PLAIN, LOGIN, CRAM-MD5, NTLM, XOAUTH2, APOP, and EXTERNAL (client certificate).
- **FR-3.** The application shall support the following authentication mechanisms for **outgoing mail (SMTP)**: PLAIN, LOGIN, CRAM-MD5, NTLM, and EXTERNAL (client certificate).
- **FR-4.** APOP shall be available only for POP3 connections. It has no meaning for IMAP or SMTP.

### 7.2 Mechanism negotiation

- **FR-5.** When a password-based account connects, the application shall inspect the server's capability advertisement and select the authentication mechanism automatically, in the following preference order (strongest first): CRAM-MD5, LOGIN, PLAIN, NTLM.
- **FR-6.** When an OAuth-based account connects, the application shall use the XOAUTH2 mechanism exclusively. Password-based fallback shall not be attempted automatically for OAuth accounts.
- **FR-7.** When a client certificate is configured and no password is provided, the application shall rely on the TLS handshake for authentication (EXTERNAL). No password-based mechanism shall be attempted.
- **FR-8.** The negotiated mechanism shall be logged for diagnostic purposes.

### 7.3 OAuth 2.0 / XOAUTH2

- **FR-9.** The application shall maintain a bundled registry of providers that support OAuth, including for each: authorization endpoint, token endpoint, required scopes, and redirect URI.
- **FR-10.** For each registered OAuth provider, the application shall provide a browser-based authorization flow that opens the provider's consent page, receives the authorization code via redirect, and exchanges it for an access token and refresh token.
- **FR-11.** The application shall refresh access tokens automatically before they expire. A minimum interval between refresh attempts shall be enforced to prevent excessive requests.
- **FR-12.** If a refresh attempt fails because the refresh token has been revoked or has expired, the application shall surface a notification to the user with an option to re-authorize.
- **FR-13.** The application shall support provider-specific requirements during the OAuth flow, including but not limited to: prompting for account selection, requesting specific consent prompts, sending additional parameters, and using PKCE (Proof Key for Code Exchange) where required.
- **FR-14.** The access token shall be presented to the mail server using the XOAUTH2 SASL mechanism.
- **FR-15.** The user shall be able to switch an OAuth-authenticated account to password-based authentication, and vice versa, without deleting and re-adding the account.
- **FR-16.** The application shall display a link to the provider's privacy policy during the OAuth authorization flow, where the provider has published one.

### 7.4 Client certificate / EXTERNAL

- **FR-17.** The application shall allow the user to select an installed client certificate for each connection endpoint (incoming server and outgoing server independently).
- **FR-18.** When a client certificate is selected, the application shall present it during the TLS handshake as a client credential. If the server accepts the certificate, no further password-based authentication shall be required.
- **FR-19.** The user shall be able to clear the certificate selection at any time, reverting the account to password-based or OAuth-based authentication.
- **FR-20.** The selected certificate shall be identified by a stable alias. The application shall re-resolve the alias to the current certificate and private key at each connection attempt.

### 7.5 NTLM

- **FR-21.** For accounts that require NTLM authentication, the application shall allow the user to specify a domain/realm value in the account settings.
- **FR-22.** The domain/realm value shall be sent as the NTLM domain during authentication.

### 7.6 APOP

- **FR-23.** APOP shall be disabled by default. The user shall be able to enable it via an advanced setting.
- **FR-24.** When APOP is enabled and the POP3 server's greeting contains a valid timestamp, the application shall authenticate using APOP instead of other password-based mechanisms.

### 7.7 Per-mechanism global toggles

- **FR-25.** The application shall provide global toggle settings for the following mechanisms: PLAIN, LOGIN, NTLM, CRAM-MD5 (via a general SASL toggle), and APOP.
- **FR-26.** When a mechanism is disabled globally, the application shall not attempt it on any connection, regardless of server advertisement.
- **FR-27.** All password-based mechanisms except APOP shall be enabled by default.
- **FR-28.** APOP shall be disabled by default.
- **FR-29.** These toggles shall be accessible from the application's advanced settings area, not from per-account settings, because they govern application-wide behavior of the protocol libraries.

### 7.8 Security constraints

- **FR-30.** When a connection uses no encryption (neither TLS nor STARTTLS), the application shall refuse to authenticate using PLAIN or LOGIN by default, because these mechanisms transmit the password in a recoverable form.
- **FR-31.** An explicit per-account "allow insecure connections" flag shall override FR-30, permitting password-based authentication over unencrypted connections.
- **FR-32.** OAuth tokens and passwords shall be treated as sensitive credentials for storage purposes (see the Credential Storage epic for specifics).

### 7.9 Provider-specific handling

- **FR-33.** The application shall accommodate known provider-specific authentication quirks without requiring user intervention. Examples include: provider-specific greeting identifiers, multi-line authentication sequences for certain POP3 OAuth flows, and provider-mandated request parameters.
- **FR-34.** Provider-specific adaptations shall be keyed to the server hostname, so that they activate only for the relevant provider and do not affect other connections.

### 7.10 Error reporting

- **FR-35.** When authentication fails, the application shall present a clear error message distinguishing between: invalid credentials, unsupported mechanism, expired or revoked token, certificate rejection, and server-side errors.
- **FR-36.** When authentication fails because all enabled mechanisms have been exhausted, the error shall mention the possibility that a required mechanism has been disabled in advanced settings.

## 8. Non-Functional Requirements

- **NFR-1. Transparency.** The user shall never need to understand authentication mechanisms to set up a mainstream account. Mechanism selection shall be invisible in the happy path.
- **NFR-2. Token refresh latency.** Automatic OAuth token refresh shall complete within a reasonable timeframe and shall not visibly delay mail synchronization under normal network conditions.
- **NFR-3. Refresh storm prevention.** The application shall enforce a minimum interval between consecutive token refresh attempts for the same account, to avoid hammering the provider's token endpoint when a refresh fails transiently.
- **NFR-4. Credential isolation.** No authentication credential (password, token, certificate private key) shall be written to log output, crash reports, or diagnostic exports.
- **NFR-5. Offline resilience.** Cached OAuth tokens shall be stored durably so that the application can authenticate immediately after restart without requiring a network round-trip to the token endpoint, as long as the token has not expired.
- **NFR-6. Provider extensibility.** Adding OAuth support for a new provider shall require only a configuration entry (endpoint URLs, scopes, client identifier) — not code changes to the authentication subsystem.
- **NFR-7. Degradation clarity.** When a provider deprecates a mechanism (e.g. Google disabling password-based access), the application's error messaging shall guide the user toward the supported alternative (e.g. OAuth or app-specific passwords).
- **NFR-8. Accessibility.** The OAuth browser flow, certificate selection dialog, and mechanism toggle controls shall all be operable via keyboard and compatible with screen readers.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|
| AC-1 | A new account added with a username and password against a server advertising CRAM-MD5, LOGIN, and PLAIN authenticates using CRAM-MD5 (the highest-priority mechanism). |
| AC-2 | Disabling CRAM-MD5 (SASL) in advanced settings causes the same account to fall back to LOGIN or PLAIN on the next connection attempt. |
| AC-3 | Adding a Gmail account via the OAuth flow completes authorization in the browser, returns to the application, and successfully authenticates using XOAUTH2 without the user typing a password into the email client. |
| AC-4 | An OAuth-authenticated account whose access token has expired refreshes the token automatically and reconnects without user intervention. |

| AC-6 | Switching an OAuth account to password authentication (and back) preserves the account's folders, messages, and settings — only the authentication method changes. |
| AC-7 | Selecting a client certificate for an account's incoming server and leaving the password blank results in successful authentication via the TLS handshake (EXTERNAL), with no password prompt. |
| AC-8 | Clearing the client certificate selection and providing a password results in the account reverting to password-based authentication on the next connection. |
| AC-9 | Specifying a domain/realm value for an account connecting to an NTLM-requiring server allows authentication to succeed, whereas omitting the domain causes authentication to fail. |
| AC-10 | Enabling APOP in advanced settings and connecting to a POP3 server that includes a timestamp in its greeting results in APOP authentication. |
| AC-11 | With APOP disabled (the default), the same POP3 server is authenticated using a standard password-based mechanism instead. |
| AC-12 | Attempting to connect to an unencrypted server with a password and without the "allow insecure" flag results in a connection refusal or clear warning — the password is not sent in the clear. |
| AC-13 | Enabling the "allow insecure" flag for the same unencrypted server permits the connection and authentication to proceed. |
| AC-14 | Disabling PLAIN and LOGIN in advanced settings while connecting to a server that advertises only PLAIN produces a clear error message mentioning that the required mechanism is disabled. |
| AC-15 | OAuth token refresh does not occur more frequently than the configured minimum interval, even if multiple sync cycles trigger within that window. |
| AC-16 | The diagnostic log records which authentication mechanism was used for each connection, but does not record the password, token value, or certificate private key. |

| AC-18 | A provider with OAuth support that is added to the bundled provider registry (configuration only) becomes available in the OAuth sign-in flow without any changes to the authentication logic. |

## 10. Open Questions

- **OQ-1. SASL extensibility.** The current platform constraint limits SASL support to CRAM-MD5. If the application is ported to a platform with full SASL support, should the mechanism preference order be revisited to include SCRAM-SHA-256 or other modern SASL mechanisms? This is deferred to the porting stage.
- **OQ-2. NTLM testing coverage.** NTLM is documented as "untested" in the existing FAQ. It is unclear whether NTLM authentication has been validated against a real Exchange server. The feature is included because the code path exists, but its reliability should be verified before it is advertised as supported.
- **OQ-3. SMTP and XOAUTH2.** The current behavior for SMTP authentication with OAuth accounts is not entirely clear — some providers accept XOAUTH2 for SMTP while others use the OAuth token as a password via PLAIN. The exact per-provider SMTP authentication strategy should be documented.
- **OQ-4. OAuth provider coverage.** The bundled provider registry includes OAuth configurations for a significant number of providers, but the exact list is subject to change as providers add, modify, or deprecate their OAuth endpoints. A process for keeping the registry current should be established.
- **OQ-5. APOP security posture.** APOP relies on MD5, which is cryptographically broken for collision resistance. While APOP's use of MD5 as an HMAC-like construction is not directly vulnerable to known collision attacks, it is weak by modern standards. Should the application display a warning when APOP is enabled, or is the existing "disabled by default" posture sufficient?
- **OQ-6. Certificate selection UX.** The current certificate selection relies on the operating system's certificate picker. If the system certificate store is empty or the desired certificate is not installed, the user experience may be confusing. Should the application provide guidance on how to install a client certificate, or is that out of scope?
- **OQ-7. Microsoft Graph vs. standard OAuth.** The application supports two distinct OAuth paths for Microsoft accounts — standard OAuth and a Graph-API-based variant. The conditions under which each is preferred, and whether both should be exposed to the user as separate options, are not fully clear from the code alone.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Mechanism negotiation is automatic, not manual.** The user never selects "use CRAM-MD5" or "use PLAIN" from a dropdown. Instead, the application reads the server's capability advertisement and picks the best available mechanism from the enabled set. This is the right default — the vast majority of users cannot make an informed choice, and automatic negotiation matches what other mainstream clients do.
- **N-2. CRAM-MD5 is preferred over PLAIN/LOGIN.** Despite CRAM-MD5's reliance on MD5, it has the advantage of never transmitting the password (even in encoded form) over the wire, making it strictly stronger than PLAIN or LOGIN when TLS is not in use. Over TLS, the distinction is marginal, but the preference order is harmless and benefits the rare unencrypted case.
- **N-3. APOP is off by default.** APOP is a POP3-only mechanism that very few servers still require. Enabling it by default would cause the application to attempt APOP negotiation on every POP3 connection, which some servers handle poorly. The opt-in approach avoids breakage while still supporting users who need it.
- **N-4. Global toggles, not per-account.** The mechanism enable/disable toggles are global (application-wide) rather than per-account. This is because the toggles control properties of the underlying protocol libraries, which are configured at the process level. Per-account mechanism selection would require a different architecture and is not warranted given how rarely users need to adjust these settings.
- **N-5. OAuth as a distinct auth type, not a mechanism preference.** An account is either "password-based" or "OAuth-based" — this is a top-level attribute, not a mechanism preference flag. When an account is OAuth-based, the application uses XOAUTH2 exclusively and does not fall back to password mechanisms. This clean separation avoids confusing failure modes where the application might try a password it does not have.
- **N-6. Client certificate selection is per-endpoint.** A certificate can be set independently for the incoming server connection and the outgoing server connection. This supports environments where the IMAP and SMTP servers are managed by different systems with different certificate authorities.
- **N-7. Realm serves dual purpose.** A single realm/domain field in the account settings is used by both CRAM-MD5 (as the SASL realm) and NTLM (as the Windows domain). This simplification works because no real-world scenario requires different values for the two mechanisms on the same account.
- **N-8. Provider-specific workarounds are hostname-keyed.** Quirks such as special greeting identifiers or multi-line POP3 OAuth sequences are activated based on the server hostname in the account configuration. This ensures that workarounds apply narrowly and do not affect unrelated servers.
