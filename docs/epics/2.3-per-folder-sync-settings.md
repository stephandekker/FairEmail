# Epic: Per-Folder Sync Settings

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **2.3** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Sync on/off, poll interval, days-to-keep, days-to-download, auto-classify, notify on new mail, and download attachments per folder."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[SHOULD]` — important, planned for v1 but acceptable to slip to v1.x if pressed.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email accounts accumulate many folders over time: the Inbox, Sent, Drafts, Trash, Spam, Archive, and any number of user-created folders for projects, mailing lists, or team workflows. These folders differ wildly in how urgently the user needs their contents, how much storage they consume, how far back they need to browse, and whether new arrivals should generate notifications.

A single, account-wide synchronization policy cannot satisfy these differences. The Inbox demands near-real-time delivery and notifications; Sent and Archive need periodic checks but rarely urgency; Trash and Spam should keep minimal history; a high-volume mailing-list folder might sync but should stay silent; a rarely-used reference folder might not need to sync at all.

The Per-Folder Sync Settings feature exists to give the user independent control over synchronization behavior for every folder in every account. Each folder carries its own set of sync properties — whether it syncs at all, how often it is checked, how far back messages are fetched, how long they are retained locally, whether message content and attachments are downloaded automatically, whether new messages trigger notifications, and whether the folder participates in automatic message classification. These properties are orthogonal to one another: any combination is valid.

This per-folder granularity lets users balance responsiveness, storage, bandwidth, battery life, and notification noise on a folder-by-folder basis, while sensible defaults ensure that a newly added account works well without any manual tuning.

## 2. Goals

- **G1.** Give every folder an independent, user-editable set of synchronization properties that govern its sync behavior without affecting other folders.
- **G2.** Provide sensible defaults per folder role so that a new account works well out of the box — the Inbox syncs with push and notifications; Sent, Drafts, and Archive sync periodically with content download; Trash and Spam sync periodically without content download; user-created folders default to periodic sync.
- **G3.** Allow fine-grained control over local retention: how many days of messages to fetch from the server and how many days of messages to keep on the device, independently.
- **G4.** Allow the user to disable synchronization for any folder entirely, without deleting the folder or losing its existing local messages.
- **G5.** Allow per-folder control over automatic content download, so that bandwidth and storage can be conserved on folders whose content the user rarely reads.
- **G6.** Allow per-folder notification control, independent of whether the folder is synchronized and independent of whether it appears in the unified inbox.
- **G7.** Allow per-folder participation in automatic message classification (learning from a folder and/or receiving auto-classified messages), so that the classifier can be trained and targeted at the folder level.
- **G8.** Provide a batch-apply mechanism so that users with many folders can change common settings across multiple folders in one operation.

## 3. Non-Goals

- **NG1.** This epic does **not** define the synchronization protocol or transport mechanism (push, poll, connection management). It only constrains which folders are synchronized, at what cadence, and with what retention — not how the sync engine achieves it.
- **NG2.** This epic does **not** define how per-folder settings are persisted or queried — only that they are durable, per-folder, and user-editable.
- **NG3.** Account-level connection settings (server, port, encryption, authentication, timeouts) are out of scope. This epic concerns only the per-folder properties that ride on top of an already-configured account connection.
- **NG4.** Unified-inbox membership is a separate per-folder property covered by the Unified Inbox epic (3.1). This epic acknowledges its existence as an orthogonal setting but does not define its behavior.
- **NG5.** Folder creation, deletion, renaming, and subscription management are covered by separate epics (2.4, 2.6). This epic assumes folders exist and focuses on their sync properties.
- **NG6.** Notification presentation (sounds, badges, grouping, actions, quiet hours) is out of scope. This epic covers only the per-folder toggle that determines whether a folder generates notifications at all.

## 4. Glossary

| Term | Definition |
|---|---|
| **Account** | A configured connection to a single mail provider, with credentials, server settings, and its own folder tree. |
| **Folder** | A container of messages within an account. May be a system folder or a user-created folder. |
| **System folder** | A folder with a well-known role: Inbox, Sent, Drafts, Trash, Spam (Junk), Archive. Detected automatically from the server or assigned by the user. |
| **User folder** | Any folder that is not a system folder — typically created by the user or by server-side rules. |
| **Synchronize (folder)** | The per-folder boolean that controls whether the application actively fetches and monitors this folder for changes. |
| **Push** | A server-initiated notification mechanism (IMAP IDLE) that delivers new-message alerts in near-real-time without periodic polling. |
| **Poll** | A client-initiated check at regular intervals. Used when the server does not support push, or when the user prefers periodic checking for a given folder. |
| **Poll factor** | An integer multiplier applied to the account's base poll interval to determine how often a specific folder is checked. A factor of 2 means the folder is checked every second poll cycle. |
| **Sync window (days to sync / days to download)** | The number of days into the past from which the application fetches messages during a regular sync operation. Messages older than this window are not fetched unless already present locally. |
| **Keep window (days to keep)** | The number of days for which messages are retained on the device. Messages older than this window may be removed from local storage (subject to the auto-delete setting). The keep window must be greater than or equal to the sync window. |
| **Auto-delete** | A per-folder option that, when enabled, automatically removes local messages that have aged past the keep window. |
| **Download (auto-download)** | A per-folder boolean controlling whether message bodies and attachments are fetched automatically during sync, or only on demand when the user opens a message. |
| **Notify** | A per-folder boolean controlling whether new messages arriving in this folder generate desktop notifications. |
| **Auto-classify (source)** | A per-folder boolean indicating that messages in this folder are used as training data for the automatic message classifier. |
| **Auto-classify (target)** | A per-folder boolean indicating that the classifier may automatically move messages into this folder when it determines they belong there. |
| **On-demand account** | An account configured to synchronize only when the user explicitly triggers a sync, not on a schedule. |

## 5. Personas

- **P1. The default user.** Has one or two accounts; never opens folder properties. Relies entirely on defaults. Expects the Inbox to deliver mail promptly with notifications, and everything else to "just work" in the background.
- **P2. The bandwidth-conscious user.** On a metered or slow connection; wants to minimize data transfer. Disables content download for archival folders, reduces sync windows, and turns off sync for folders they rarely check.
- **P3. The power organizer.** Has many accounts with deep folder hierarchies. Wants fine-grained control: different poll intervals for different folders, auto-classification trained on specific folders, notifications only from select folders, and aggressive auto-deletion on transient folders.
- **P4. The notification curator.** Receives mail in many folders but only wants alerts from a handful. Uses per-folder notification toggles to silence everything except their primary Inbox and one or two project folders.
- **P5. The archivist.** Keeps years of mail locally. Sets large keep windows (or "keep all") on important folders, but short sync windows so that regular syncs stay fast.

## 6. User Stories

### 6.1 Viewing and editing folder sync settings

> **US-1.** As any user, I want to access a properties screen for any folder that shows all of its sync-related settings in one place, so that I can understand and adjust how the folder behaves.

> **US-2.** As any user, I want changes to folder sync settings to take effect immediately (on the next sync cycle) without requiring an application restart, so that I can tune settings interactively.

> **US-3.** As any user, I want my folder sync settings to persist across application restarts, account re-syncs, and server folder-list rediscovery, so that I configure each folder once.

### 6.2 Synchronize on/off

> **US-4.** As a bandwidth-conscious user, I want to disable synchronization for a folder entirely, so that the application does not fetch any new messages from it and does not maintain a connection to it.

> **US-5.** As any user, when I disable synchronization for a folder, I want existing local messages in that folder to remain accessible, so that I do not lose what has already been downloaded.

> **US-6.** As any user, when I re-enable synchronization for a folder, I want it to begin fetching messages according to its sync window, picking up where it left off, so that I do not miss messages that arrived while sync was off.

> **US-7.** As any user, when synchronization is disabled for a folder but the account allows on-demand browsing, I want to be able to manually browse the folder's server contents, so that I can access messages without committing to ongoing synchronization.

### 6.3 Poll interval

> **US-8.** As a power organizer, I want to control whether a folder uses push (real-time) or poll (periodic) delivery, so that I can balance responsiveness against resource usage.

> **US-9.** As a power organizer, I want to set a poll factor for a folder that multiplies the account's base poll interval, so that less-important folders are checked less frequently without having to change the account-wide interval.

> **US-10.** As any user, I want the folder properties screen to display the effective poll interval in human-readable terms (e.g. "every 30 minutes"), so that I can understand the consequence of the poll factor I have set.

### 6.4 Sync window (days to download/sync)

> **US-11.** As a bandwidth-conscious user, I want to set the number of days of messages that a folder fetches during each sync, so that I can limit how far back the application reaches.

> **US-12.** As any user, I want the sync window to default to a sensible value (7 days) for most folders, so that a newly added account fetches a useful amount of history without excessive data transfer.

> **US-13.** As any user, if synchronization was not performed for longer than the sync window, I want the next sync to expand its window to cover the entire gap, so that no messages are missed due to a period of disconnection.

### 6.5 Keep window (days to keep)

> **US-14.** As an archivist, I want to set the number of days for which messages are retained on the device, independently of the sync window, so that I can keep older messages available offline even after they fall outside the sync window.

> **US-15.** As an archivist, I want a "keep all" option that retains messages indefinitely on the device, so that I never lose locally stored messages to age-based cleanup.

> **US-16.** As any user, I want the keep window to default to a sensible value (30 days for most folders, 365 days for Drafts), so that local storage does not grow unbounded without the user opting in.

> **US-17.** As any user, I want the application to enforce that the keep window is always greater than or equal to the sync window, so that freshly synced messages are never immediately eligible for deletion.

### 6.6 Auto-delete

> **US-18.** As a power organizer, I want an auto-delete option per folder that automatically removes local messages older than the keep window, so that I do not have to manually clean up transient folders like Trash or Spam.

> **US-19.** As any user, I want auto-delete to be **off** by default, so that I do not lose messages unexpectedly.

> **US-20.** As any user, when auto-delete is enabled on an archive-type folder, I want a warning that this will permanently remove messages that may not exist elsewhere, so that I make an informed choice.

### 6.7 Download content and attachments

> **US-21.** As a bandwidth-conscious user, I want to disable automatic download of message bodies and attachments for a folder, so that only message headers are fetched during sync and full content is loaded on demand.

> **US-22.** As any user, I want the download setting to default to enabled for the Inbox, Drafts, Sent, and Archive, so that messages in commonly read folders are available for offline reading.

> **US-23.** As any user, I want the download setting to default to disabled for Trash and Spam, so that bandwidth is not spent on messages I am unlikely to read.

> **US-24.** As any user, when download is disabled for a folder, I want to be able to read any individual message on demand by opening it, so that the setting affects bulk behavior but does not prevent access.

### 6.8 Notifications

> **US-25.** As a notification curator, I want to enable or disable new-message notifications independently for each folder, so that I am alerted only by the folders I care about.

> **US-26.** As any user, I want the Inbox to have notifications enabled by default and all other system folders to have notifications disabled by default, so that I get useful alerts without noise out of the box.

> **US-27.** As any user, I want per-folder notification settings to be independent of synchronization on/off and independent of unified-inbox membership, so that toggling one does not affect the others.

> **US-28.** As any user, I want per-folder notifications to be subordinate to account-level notification control: if the account has notifications disabled, no folder in that account should notify, regardless of its per-folder setting.

### 6.9 Auto-classify

> **US-29.** As a power organizer, I want to mark a folder as a classification source, so that the message classifier learns from the messages in that folder.

> **US-30.** As a power organizer, I want to mark a folder as a classification target, so that the classifier can automatically move messages into it when it determines they belong there.

> **US-31.** As any user, I want the Inbox and Spam folders to be classification sources by default, and no folder to be a classification target by default, so that the classifier learns from the most important folders without moving messages unexpectedly.

> **US-32.** As any user, I want the classification-target setting to be available only when the classification-source setting is also enabled, so that the relationship between learning and acting is clear.

> **US-33.** As any user, I want the auto-classify settings to be available only when automatic download is enabled for the folder, so that the classifier has access to message content.

### 6.10 Batch operations

> **US-34.** As a power organizer with many folders, I want to apply common sync settings (sync on/off, sync days, keep days, poll mode) to multiple folders at once, so that I do not have to edit each folder individually.

> **US-35.** As any user, when I create a new child folder under an existing user folder, I want the child to inherit sync settings (sync on/off, download, sync days, keep days, notify) from the parent, so that folder hierarchies have consistent defaults.

### 6.11 Interaction with account-level settings

> **US-36.** As any user, I want account-level synchronization to act as a master switch: when the account is disabled, no folder in that account syncs, regardless of per-folder settings. When the account is re-enabled, each folder resumes according to its own settings.

> **US-37.** As any user, I want the account's base poll interval to serve as the foundation for per-folder poll timing, with each folder's poll factor applied as a multiplier, so that I can adjust the account-wide cadence and have all folder intervals scale proportionally.

## 7. Functional Requirements

### 7.1 Per-folder sync properties

- **FR-1.** Every folder shall have the following independently configurable, persistent properties:
  - **Synchronize** (boolean): whether the folder is actively synced.
  - **Poll mode** (boolean): whether to use polling instead of push for this folder.
  - **Poll factor** (positive integer, minimum 1): multiplier for the account's base poll interval.
  - **Sync window** (positive integer, in days): how many days of messages to fetch.
  - **Keep window** (positive integer, in days): how many days of messages to retain locally.
  - **Auto-delete** (boolean): whether to automatically remove messages older than the keep window.
  - **Download** (boolean): whether to automatically download message bodies and attachments.
  - **Notify** (boolean): whether to generate notifications for new messages.
  - **Auto-classify source** (boolean): whether the classifier learns from this folder.
  - **Auto-classify target** (boolean): whether the classifier may move messages into this folder.

- **FR-2.** All properties shall be editable from a folder properties screen accessible via a context action on the folder (e.g. right-click, long-press equivalent) or from within the folder list.

- **FR-3.** All properties shall persist across application restarts, re-login, and server folder-list rediscovery.

- **FR-4.** Changes to any property shall take effect on the next sync cycle without requiring an application restart.

### 7.2 Defaults by folder role

- **FR-5.** When a new account is added, each detected folder shall receive defaults according to its role:

| Property | Inbox | Drafts | Sent | Archive | Trash | Spam | User folders |
|---|---|---|---|---|---|---|---|
| Synchronize | on | on | on | on | on | on | on |
| Poll mode | off (push) | on (poll) | on (poll) | on (poll) | on (poll) | on (poll) | on (poll) |
| Poll factor | 1 | 1 | 1 | 1 | 1 | 1 | 1 |
| Sync window | 7 days | 7 days | 7 days | 7 days | 7 days | 7 days | 7 days |
| Keep window | 30 days | 365 days | 30 days | 30 days | 30 days | 30 days | 30 days |
| Auto-delete | off | off | off | off | off | off | off |
| Download | on | on | on | on | off | off | on |
| Notify | on | off | off | off | off | off | off |
| Classify source | on | off | off | off | off | on | off |
| Classify target | off | off | off | off | off | off | off |

- **FR-6.** Provider-specific folders that the application recognizes as functionally equivalent to the Inbox **may** receive Inbox-like defaults (synchronize, unified, notify). This is a known carve-out and shall be documented per provider.

### 7.3 Constraints and validation

- **FR-7.** The keep window shall always be greater than or equal to the sync window. If the user sets a keep window smaller than the sync window, the application shall silently raise the keep window to match the sync window.

- **FR-8.** The poll factor shall have a minimum value of 1. Values below 1 shall be clamped to 1.

- **FR-9.** For protocols that do not support partial sync (e.g. POP3), the sync window and keep window shall be set to their maximum values and shall not be user-editable.

- **FR-10.** The auto-classify source setting shall only be effective when the download setting is enabled, because classification requires message content. The UI should make this dependency visible.

- **FR-11.** The auto-classify target setting shall only be effective when the auto-classify source setting is also enabled for the same folder. The UI should make this dependency visible.

### 7.4 Sync behavior

- **FR-12.** When a folder's synchronize property is enabled, the application shall fetch messages within the sync window and monitor the folder for new arrivals according to the folder's poll mode and poll factor.

- **FR-13.** When a folder's synchronize property is disabled, the application shall not connect to or monitor that folder. Existing local messages shall remain accessible.

- **FR-14.** If a folder's last successful sync is older than its sync window, the next sync shall expand its effective window to cover the entire gap since the last sync, ensuring no messages are missed.

- **FR-15.** When poll mode is disabled (push) for a folder, the application shall use the server's real-time notification mechanism. If the server does not support push for that folder, the application shall fall back to polling automatically.

- **FR-16.** The effective poll interval for a folder shall be the product of the account's base poll interval and the folder's poll factor.

### 7.5 Retention behavior

- **FR-17.** Messages older than the keep window shall be candidates for local removal. Actual removal occurs only if auto-delete is enabled for the folder.

- **FR-18.** When auto-delete is off, messages older than the keep window remain on the device until manually removed, but are not re-fetched if deleted locally.

- **FR-19.** A "keep all" option shall set the keep window to its maximum value, effectively disabling age-based retention.

- **FR-20.** When auto-delete is enabled on an archive-role folder, the application shall display a warning that messages removed locally from an archive may not be recoverable.

### 7.6 Download behavior

- **FR-21.** When download is enabled for a folder, message bodies and attachments shall be fetched automatically during sync, subject to global size limits and network-type restrictions.

- **FR-22.** When download is disabled for a folder, only message headers (sender, subject, date, flags, size) shall be fetched during sync. Full message content shall be fetched on demand when the user opens the message.

- **FR-23.** The download setting shall not prevent the classifier from accessing message content if classification requires it and the folder is marked as a classification source.

### 7.7 Notification behavior

- **FR-24.** When notify is enabled for a folder and the account's notification setting is also enabled, new messages arriving in the folder during sync shall generate desktop notifications according to the application's notification subsystem.

- **FR-25.** When notify is disabled for a folder, no notifications shall be generated for new messages in that folder, regardless of account-level or global notification settings.

- **FR-26.** The notify setting shall be independent of the synchronize, download, and unified-inbox membership settings. Toggling one shall never modify another.

### 7.8 Auto-classification behavior

- **FR-27.** When a folder is marked as a classification source, messages in that folder shall be used as training data for the automatic message classifier. The classifier learns patterns from the folder to predict where future messages belong.

- **FR-28.** When a folder is marked as a classification target, the classifier may automatically move incoming messages into that folder when its prediction confidence is sufficiently high.

- **FR-29.** Classification shall only occur on folders that are synchronized and whose content has been downloaded (either automatically or on demand).

- **FR-30.** A classified message that has been automatically moved shall be visually distinguishable from a manually filed message, or the move shall be recorded in a way that the user can review and correct.

### 7.9 Batch operations

- **FR-31.** The application shall provide a batch-apply dialog that allows the user to change sync-related settings across multiple folders in one operation. At minimum, the batch dialog shall support: enabling/disabling synchronization, setting sync days, setting keep days, and enabling/disabling poll mode.

- **FR-32.** When a new user-type child folder is discovered under an existing user-type parent folder, the child shall inherit the parent's sync settings (synchronize, download, sync days, keep days, notify) as its defaults.

### 7.10 Account-level interaction

- **FR-33.** Account-level synchronization shall act as a master switch. When the account is disabled, no folder in that account shall sync, regardless of individual folder settings. When the account is re-enabled, each folder shall resume according to its own settings.

- **FR-34.** Account-level notification control shall act as a master switch for notifications. When account notifications are disabled, no folder in that account shall generate notifications, even if the folder's notify property is enabled.

- **FR-35.** The account's base poll interval shall serve as the foundation for per-folder poll timing, with each folder's poll factor applied as a multiplier.

### 7.11 UI conditional visibility

- **FR-36.** The poll factor setting shall be visible and editable only when poll mode is enabled for the folder and the folder is synchronized.

- **FR-37.** The auto-classify source and target settings shall be visible only when the message classifier subsystem is enabled globally.

- **FR-38.** Sync window, keep window, auto-delete, poll mode, and download settings shall be hidden for protocols that do not support them (e.g. POP3).

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Changes to per-folder sync settings shall be reflected in the application's sync behavior within the next sync cycle (typically under one minute for pushed folders, within one poll interval for polled folders).
- **NFR-2. Scale.** The feature shall remain usable with at least 20 accounts and hundreds of folders per account. The folder properties screen shall open without perceptible delay regardless of the total folder count.
- **NFR-3. Offline.** All per-folder settings shall be editable while offline. Changes shall take effect the next time the application connects.
- **NFR-4. Storage efficiency.** Default values should avoid unnecessary storage consumption. Folders with download disabled (Trash, Spam) should consume negligible local storage beyond header metadata.
- **NFR-5. Discoverability.** The folder properties screen shall be reachable from the folder list via a standard context action (right-click, long-press equivalent). FAQ or help text shall explain the meaning of sync window vs. keep window.
- **NFR-6. Privacy.** All per-folder sync settings shall be local user preferences. No sync-setting metadata shall be transmitted to the mail server or any third party.
- **NFR-7. Accessibility.** All controls on the folder properties screen shall be reachable via keyboard, with screen-reader labels, and shall obey the application-wide theme.

## 10. Open Questions

- **OQ-1. Auto-delete vs. server-side delete.** When auto-delete removes a message locally, should it also issue a server-side delete? The source application appears to remove messages locally only. The interaction with server-side retention policies and IMAP EXPUNGE semantics needs clarification.
- **OQ-2. Download setting and classifier interaction.** FR-23 notes that the classifier may need message content even when download is off. Should the classifier silently download content for classification purposes, or should the user be informed that enabling classification on a download-off folder will incur additional data transfer?
- **OQ-3. Batch-apply scope.** The source application's batch-apply dialog distinguishes between system folders and user folders. Should the batch dialog allow applying settings to system folders as well, or should system folders always be configured individually to prevent accidental misconfiguration?
- **OQ-4. Provider-specific overrides.** The source application has provider-specific folder overrides (e.g. certain providers have non-standard folders forced to sync with Inbox-like defaults). Should the desktop application carry these overrides forward, or should they be handled via a user-editable provider configuration?
- **OQ-5. Sync window expansion heuristic.** FR-14 describes expanding the sync window to cover gaps when the application has been offline longer than the sync window. The upper bound of this expansion is unclear — should it be capped at the keep window, or should it be unlimited?
- **OQ-6. "Keep all" and storage growth.** When a user enables "keep all" on a high-volume folder, there is no built-in safeguard against unbounded storage growth. Should the application provide a warning or a soft quota mechanism?
- **OQ-7. Auto-classify target as a premium feature.** In the source application, the auto-classify target feature (except for the Spam folder) is gated behind a premium subscription. Should this gating be preserved, modified, or removed in the desktop application?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Orthogonality of per-folder flags.** Synchronize, notify, download, unified-inbox membership, and auto-classify are five independent concerns. Every combination of on/off for each is meaningful and supported. A folder can be synchronized but not notifying, notifying but not unified, unified but not downloading content, downloading but not classifying, and so on. This orthogonality is fundamental to the feature's utility and should not be collapsed.

- **N-2. Push is the Inbox default; poll is everything else.** Only the Inbox defaults to push (real-time delivery). All other system folders and all user folders default to poll mode. This reflects the reality that only the Inbox benefits meaningfully from sub-minute delivery latency; other folders can tolerate periodic checks. Users who want push on additional folders can disable poll mode per folder.

- **N-3. Poll factor, not absolute interval.** Per-folder polling is expressed as a multiplier of the account's base interval, not as an absolute interval. This means the user can adjust the account-wide cadence (e.g. from 15 to 30 minutes) and have all folder intervals scale proportionally, without editing each folder. The poll factor provides per-folder *relative* control on top of that.

- **N-4. Sync window vs. keep window separation.** The sync window controls how far back the application *reaches* on each sync; the keep window controls how long messages *stay* locally. Separating these lets the archivist keep a deep local archive (keep=365) while keeping regular syncs fast (sync=7). A common misconception is that "sync days" means "days of messages I can see" — it actually means "days of messages I re-check against the server." Messages already downloaded remain accessible until the keep window expires.

- **N-5. Auto-delete off by default.** The source application defaults auto-delete to off for all folders. This is a safety decision: users should not lose messages without explicitly opting in. The keep window, without auto-delete, merely governs when messages stop being re-fetched — they are not actively removed.

- **N-6. Child folder inheritance.** When a new user folder is created under an existing user folder, it inherits the parent's sync settings. This reduces configuration burden for users who organize mail in deep hierarchies (e.g. Projects/Alpha, Projects/Beta). System folders never inherit from other folders; they always use their role-based defaults.

- **N-7. Account as master switch.** The account-level synchronize and notify properties are master switches. Per-folder settings are subordinate: they can restrict further (folder notify=off even though account notify=on) but cannot override upward (folder notify=on has no effect when account notify=off). This two-tier model prevents a situation where a user disables an account's notifications but a single folder keeps buzzing.

- **N-8. Conditional UI visibility.** Several settings are conditionally visible based on other settings or the protocol in use. Poll factor is only shown when poll mode is on. Auto-classify controls are only shown when the classifier subsystem is enabled globally. IMAP-specific settings (sync window, keep window, auto-delete) are hidden for POP3 accounts. This keeps the properties screen simple for users who do not need advanced options, while making them available to power users.
