# Epic: Recipient Autocomplete

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **4.11** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Suggestions drawn from sent mail, received mail, frequency rank, the local address book, or the system address book (Evolution Data Server)."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

When composing an email, the most frequent friction point is addressing: typing a recipient's full email address correctly, every time. A desktop email client that lacks autocomplete forces users to remember exact addresses or to switch context to an address book, both of which slow down one of the most common actions in the application.

Recipient autocomplete solves this by offering real-time suggestions as the user types into the To, Cc, or Bcc fields. Suggestions are drawn from multiple sources — the user's own sending history, their received mail, and any system-level address book (such as GNOME's Evolution Data Server or KDE's KAddressBook) — and ranked so that the most likely intended recipient appears first. The goal is to make addressing a message feel almost instantaneous for contacts the user has corresponded with before, while remaining useful for first-time addresses through integration with the platform's shared contact store.

Beyond simple completion, the feature also drives a related convenience: **automatic identity selection**, where the application can switch the sender identity to match the account most recently associated with the chosen recipient. This transforms autocomplete from a typing aid into a lightweight routing decision.

## 2. Goals

- **G1.** Provide real-time, inline suggestions in every recipient field (To, Cc, Bcc) as the user types, drawn from multiple data sources.
- **G2.** Rank suggestions so that the most likely intended recipient appears first, using a combination of contact frequency, recency, and source priority.
- **G3.** Integrate with the desktop system's shared address book (Evolution Data Server / KAddressBook) so that contacts managed outside the email client are available for completion.
- **G4.** Maintain a local contact store that learns automatically from sent and received mail, so that autocomplete is useful even without access to a system address book.
- **G5.** Allow the user to manage (view, search, delete) locally stored contacts.
- **G6.** Support optional automatic sender-identity selection based on the chosen recipient.
- **G7.** Render selected recipients as visually rich tokens (chips) that convey name, avatar, and encryption status at a glance.

## 3. Non-Goals

- **NG1.** This epic does **not** define how the system address book is synchronized or managed — only how the autocomplete feature reads from it.
- **NG2.** Full contact-management UI (editing contact details, assigning avatars, creating groups) is out of scope. This epic covers only the autocomplete interaction, the display of selected recipients as chips, and basic management of the local contact store (view/search/delete).
- **NG3.** Contact groups (expanding a named group into multiple recipients) are a separate feature (see 4.12 — Insert contact group) and are not covered here, although autocomplete must not conflict with group expansion.
- **NG4.** Server-side address-book protocols (CardDAV, LDAP directory lookups) are out of scope for this epic.
- **NG5.** Recipient validation beyond basic email-address format checking (e.g., MX verification, deliverability checks) is not a goal.

## 4. Glossary

| Term | Definition |
|---|---|
| **Recipient field** | Any of the To, Cc, or Bcc input fields in the compose view. |
| **Suggestion** | A single autocomplete candidate shown in the dropdown as the user types, consisting of a display name (if known) and an email address. |
| **Suggestion dropdown** | The popup list of matching suggestions that appears below the recipient field as the user types. |
| **Local contact store** | An application-managed database of email addresses and display names, built automatically from the user's sent and received mail. Independent of the system address book. |
| **System address book** | The desktop platform's shared contact store (e.g., Evolution Data Server on GNOME, KAddressBook on KDE), accessible to all applications. |
| **Chip** | A compact visual token representing a selected recipient, displaying the contact's name or email, an optional avatar, and an optional encryption-status indicator. |
| **Frequency rank** | A count of how many times the user has corresponded with a given address, used to order suggestions. |
| **Recency rank** | The timestamp of the most recent correspondence with a given address, used as a secondary ordering signal. |
| **Auto-identity selection** | An optional behavior where the application automatically switches the sender identity to match the account that has previously corresponded with the first selected recipient. |
| **Starred contact** | A contact marked as a favorite in the system address book, given elevated ranking in suggestions. |
| **Ignored contact** | A contact explicitly marked by the user as excluded from autocomplete suggestions. |

## 5. Personas

- **P1. The frequent correspondent.** Sends 20–50 emails per day, mostly to the same set of colleagues and clients. Expects autocomplete to learn their most-used addresses and surface them with minimal typing (1–2 characters).
- **P2. The multi-account user.** Has personal and work accounts; needs autocomplete to either scope suggestions to the current sending account or to merge all contacts, depending on preference.
- **P3. The privacy-conscious user.** Does not grant the application access to the system address book; relies entirely on the locally learned contact store.
- **P4. The newcomer.** Has just set up the application; has no local contact history. Relies on the system address book for initial suggestions until the local store builds up.
- **P5. The power user.** Manages a large contact list, uses contact groups, and expects autocomplete to handle hundreds of matches efficiently and with clear disambiguation.

## 6. User Stories

### 6.1 Basic autocomplete

> **US-1.** As any user, when I type two or more characters into any recipient field (To, Cc, or Bcc), I want a dropdown of matching suggestions to appear, so that I can select a recipient without typing the full address.

> **US-2.** As any user, I want suggestions to match against both the display name and the email address, so that I can search by either a person's name or their address.

> **US-3.** As any user, I want matching to be case-insensitive and accent-insensitive, so that typing "jose" matches "José García" and typing "SMITH" matches "smith@example.com".

> **US-4.** As any user, I want matching to be substring-based (not just prefix), so that typing "example" matches "jane@example.org" and not just addresses starting with "example".

### 6.2 Data sources

> **US-5.** As any user, I want suggestions to include addresses I have previously sent mail to, so that my most common recipients are always available.

> **US-6.** As any user, I want the option to include addresses I have received mail from in suggestions, so that I can quickly reply to or address people who have contacted me.

> **US-7.** As any user who has granted system address-book access, I want suggestions to include contacts from the system address book (Evolution Data Server, KAddressBook, or equivalent), so that I can use my existing desktop contacts.

> **US-8.** As a privacy-conscious user who has not granted system address-book access, I want autocomplete to work using only the locally learned contact store, so that the feature remains useful without sharing my address book.

> **US-9.** As any user, when the same email address appears in multiple sources (local store and system address book), I want to see it only once in the suggestion list — deduplicated by email address — so that the dropdown is not cluttered with duplicates.

### 6.3 Ranking and ordering

> **US-10.** As a frequent correspondent, I want the option to rank suggestions by how often and how recently I have corresponded with each address, so that my most-used contacts appear first.

> **US-11.** As any user, when frequency-based ranking is disabled, I want system address-book contacts to be ranked above locally learned contacts, and contacts with known display names above email-only entries, with alphabetical ordering as the fallback, so that the dropdown has a predictable order.

> **US-12.** As any user, I want contacts marked as favorites/starred in the system address book to be ranked very highly in suggestions, so that my important contacts are easy to reach.

### 6.4 Suggestion display

> **US-13.** As any user, I want each suggestion in the dropdown to show the contact's display name and email address, so that I can distinguish between contacts with similar names or addresses.

> **US-14.** As any user, I want each suggestion to show the contact's avatar (photo) if one is available, so that I can visually identify the right person.

### 6.5 Selecting a suggestion

> **US-15.** As any user, when I click or keyboard-select a suggestion, I want the recipient to be inserted into the field and rendered as a chip, so that I can see at a glance who I am addressing.

> **US-16.** As any user, I want to be able to continue typing additional recipients after selecting a suggestion, with each new recipient getting its own autocomplete session, so that I can address multiple people efficiently.

### 6.6 Chip display

> **US-17.** As any user, I want each selected recipient to be displayed as a chip showing the contact's name (or email if no name is known) and an avatar, so that the recipient list is scannable.

> **US-18.** As any user, I want each chip to indicate whether an encryption key (PGP and/or S/MIME) is available for that recipient, so that I can see my encryption options before sending.

> **US-19.** As any user, I want to delete a chip by pressing backspace or by clicking a remove control on the chip, so that I can correct addressing mistakes.

### 6.7 Automatic identity selection

> **US-20.** As a multi-account user, I want the option to have the application automatically switch my sender identity to match the account that has previously corresponded with the first recipient I select, so that I send from the right account without manually switching.

> **US-21.** As any user, when auto-identity selection is enabled, I want it to use the same data sources (sent contacts, received contacts) as autocomplete, so that the identity match is based on actual correspondence history.

### 6.8 Account scoping

> **US-22.** As a multi-account user, I want the option to restrict autocomplete suggestions to only contacts associated with the currently selected sending account, so that I do not accidentally address someone from the wrong account.

> **US-23.** As any user, when account scoping is disabled (the default), I want suggestions to draw from contacts across all accounts, so that I have the broadest possible suggestion pool.

### 6.9 Local contact store management

> **US-24.** As any user, I want the local contact store to be populated automatically as I send and receive mail, without any manual action, so that autocomplete improves over time.

> **US-25.** As any user, I want noreply addresses to be excluded from the local contact store, so that useless addresses do not clutter my suggestions.

> **US-26.** As any user, I want to be able to view, search, and delete individual entries in the local contact store, so that I can remove outdated or unwanted suggestions.

> **US-27.** As any user, I want to be able to mark a contact as "ignored" so that it never appears in autocomplete suggestions, without having to delete it entirely.

> **US-28.** As any user, I want the local contact store to periodically purge old, rarely-contacted entries based on configurable age and frequency thresholds, so that the store does not grow unboundedly with stale data.

### 6.10 Configuration

> **US-29.** As any user, I want to be able to enable or disable each suggestion source independently (sent contacts, received contacts, system address book), so that I control what appears in autocomplete.

> **US-30.** As any user, I want to choose between frequency-based ranking and source-priority-based ranking, so that I can pick the ordering model that suits my workflow.

> **US-31.** As any user, I want to choose whether display names are included in suggestion text and chip labels, or whether only email addresses are shown, so that I can tailor the display to my preference.

## 7. Functional Requirements

### 7.1 Suggestion trigger

- **FR-1.** Autocomplete suggestions shall appear after the user has typed at least two characters into any recipient field (To, Cc, or Bcc).
- **FR-2.** The suggestion dropdown shall update in real time as the user continues typing, narrowing the result set with each additional character.
- **FR-3.** If the user clears the input below the threshold, the dropdown shall dismiss.

### 7.2 Matching

- **FR-4.** Matching shall be performed as a substring search against both the display name and the email address of each candidate contact.
- **FR-5.** Matching shall be case-insensitive.
- **FR-6.** Matching against display names shall be accent-insensitive, so that unaccented input matches accented names (e.g., "jose" matches "José").
- **FR-7.** Special characters in the search input shall be treated literally and shall not be interpreted as pattern metacharacters.

### 7.3 Data sources

- **FR-8.** The autocomplete system shall support three independent data sources, each independently enabled or disabled by the user:
  - (a) **Sent contacts**: addresses the user has sent mail to (enabled by default).
  - (b) **Received contacts**: addresses the user has received mail from (disabled by default).
  - (c) **System address book**: contacts from the desktop platform's shared contact store (available only if the user has granted the application permission to read it).
- **FR-9.** When multiple sources are enabled, results from all enabled sources shall be merged into a single suggestion list.
- **FR-10.** Deduplication shall be performed by email address (case-insensitive). When the same address appears in multiple sources, a single entry shall be shown, preferring the version that has an avatar.

### 7.4 Local contact store

- **FR-11.** The application shall maintain a local contact store that is automatically populated from outgoing and incoming mail.
- **FR-12.** For each contact entry, the store shall track: email address, display name (if known), the number of times the user has corresponded with that address, the timestamp of the most recent correspondence, and a user-assignable state (default, favorite, or ignored).
- **FR-13.** Contacts shall be stored per-account and per-direction (sent-to vs. received-from), so that account-scoped queries and source-specific queries are possible.
- **FR-14.** The store shall exclude noreply addresses automatically.
- **FR-15.** The store shall not learn from messages in Drafts, Archive, Trash, or Spam folders.
- **FR-16.** The store shall support configurable purging of entries older than a user-specified age threshold and entries with fewer than a user-specified number of interactions.
- **FR-17.** The user shall be able to view, search, and delete individual entries in the local contact store via a management screen accessible from settings.
- **FR-18.** The user shall be able to mark an entry as "ignored," which excludes it from all future autocomplete suggestions without deleting the underlying data.

### 7.5 Ranking

- **FR-19.** The application shall support two ranking modes, selectable by the user:
  - (a) **Frequency/recency mode**: suggestions ordered first by number of interactions (descending), then by most recent interaction (descending), then alphabetically.
  - (b) **Source-priority mode** (default): system address-book contacts ranked above local contacts; contacts with a known display name ranked above email-only entries; then alphabetical by name; then alphabetical by email. Starred/favorite system contacts shall be ranked at the top regardless of other factors.
- **FR-20.** Alphabetical ordering in both modes shall be locale-aware and accent-insensitive.

### 7.6 Suggestion display

- **FR-21.** Each entry in the suggestion dropdown shall display:
  - (a) The contact's display name (or a placeholder if no name is known).
  - (b) The contact's email address.
  - (c) The contact's avatar/photo, if available.
- **FR-22.** The dropdown shall appear directly below (or above, if space is constrained) the active recipient field.

### 7.7 Selection and insertion

- **FR-23.** The user shall be able to select a suggestion via click or keyboard navigation (arrow keys + Enter).
- **FR-24.** Upon selection, the address shall be inserted into the recipient field, formatted as an RFC 5322 address (display name + email).
- **FR-25.** After insertion, the cursor shall advance so that the user can immediately begin typing the next recipient.
- **FR-26.** The user shall also be able to type a complete address manually (without selecting a suggestion) and have it accepted.

### 7.8 Chip rendering

- **FR-27.** When chip display is enabled (the default), each recipient in a recipient field shall be rendered as a compact visual chip.
- **FR-28.** Each chip shall display:
  - (a) The contact's display name or email address.
  - (b) An avatar (circular or rounded, following the application's global avatar-shape setting).
  - (c) An encryption-status indicator showing whether a PGP key, an S/MIME certificate, or both are available for that address.
- **FR-29.** Chips shall be deletable by pressing backspace when the cursor is immediately after the chip, or by a direct removal gesture on the chip itself.
- **FR-30.** The chip display mode shall be toggleable by the user (chips on/off); when off, recipients shall be displayed as plain comma-separated text.
- **FR-31.** Chip text direction shall follow the directionality of the contact's name (left-to-right or right-to-left).

### 7.9 Automatic identity selection

- **FR-32.** When enabled, the application shall observe changes to the first recipient in the To field and, upon detecting a known address, switch the sender identity to the account and identity most recently used to correspond with that address.
- **FR-33.** Identity selection shall consult the same local contact store and the same enabled data sources as autocomplete.
- **FR-34.** This behavior shall be disabled by default and shall be independently toggleable.

### 7.10 Account scoping

- **FR-35.** The user shall have the option to restrict autocomplete suggestions to contacts associated with the currently selected sending account only.
- **FR-36.** When account scoping is disabled (the default), suggestions shall be drawn from contacts across all accounts.

### 7.11 System address-book integration

- **FR-37.** When the user has granted address-book read permission, the autocomplete system shall query the system address book for matches in addition to the local store.
- **FR-38.** If address-book permission has not been granted, the system address-book source shall be silently skipped with no error or prompt.
- **FR-39.** The user shall also have the option to invoke the system's native contact picker to select a contact, as an alternative to typing and using autocomplete.

### 7.12 Configuration

- **FR-40.** The following settings shall be exposed to the user, all under the send/compose settings area:
  - (a) **Suggest names**: whether display names are included in suggestions and chips (default: on).
  - (b) **Suggest sent contacts**: whether sent-to addresses appear in suggestions (default: on).
  - (c) **Suggest received contacts**: whether received-from addresses appear in suggestions (default: off).
  - (d) **Suggest frequently used**: whether to rank by frequency/recency rather than source priority (default: off).
  - (e) **Suggest from current account only**: whether to restrict suggestions to the current sending account (default: off).
  - (f) **Auto-select identity**: whether to automatically switch sender identity based on recipient (default: off).
  - (g) **Contact purge age**: how old a contact entry must be (in months) before it is eligible for purging (default: 1 month).
  - (h) **Contact purge frequency**: minimum number of interactions below which an entry is eligible for purging (default: 0, meaning purge by age only).
  - (i) **Show chips**: whether to render recipients as chips or as plain text (default: on).

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** The suggestion dropdown shall appear and update within a perceptibly instant timeframe after the user types a character, even with a local contact store containing tens of thousands of entries.
- **NFR-2. Scale.** Autocomplete shall remain usable with at least 20 accounts, a local contact store of 50,000+ entries, and a system address book of comparable size.
- **NFR-3. Offline.** Autocomplete shall work fully offline; all data sources (local store and cached system address-book data) shall be queryable without a network connection.
- **NFR-4. Privacy.** The local contact store shall be stored locally only and shall never be transmitted to any server. System address-book access shall be gated on explicit user permission and shall be read-only.
- **NFR-5. Graceful degradation.** If the system address book is unavailable or permission is not granted, autocomplete shall function normally using only the local contact store, with no error messages or degraded UI.
- **NFR-6. Accessibility.** The suggestion dropdown shall be navigable by keyboard (arrow keys, Enter to select, Escape to dismiss). Chips and dropdown entries shall have appropriate screen-reader labels. All controls shall respect the application-wide theme and text-direction settings.
- **NFR-7. Internationalization.** Matching, sorting, and display shall handle Unicode correctly, including right-to-left scripts, combining characters, and locale-specific collation.

## 10. Open Questions

- **OQ-1. Accent-insensitive matching scope.** The source application implements accent-insensitive matching for system address-book queries but appears to use simpler case-insensitive matching for local contact store queries. Should accent-insensitive matching be applied uniformly across all data sources?
- **OQ-2. Suggestion list size limit.** The source application does not impose a hard limit on the number of suggestions displayed in the dropdown. For very large contact stores, should a maximum number of visible suggestions be enforced (e.g., 50), with a "show more" affordance?
- **OQ-3. Minimum character threshold configurability.** The threshold before suggestions appear is fixed at two characters. Should this be user-configurable (e.g., 1–3 characters)?
- **OQ-4. Contact learning from which folders.** The source application skips learning contacts from Drafts, Archive, Trash, and Spam folders. Should Archive be included, since archived messages represent completed correspondence that may be worth remembering?
- **OQ-5. Purge defaults.** The default purge age is 1 month, which is relatively aggressive — a contact seen only once more than a month ago would be eligible for removal. Should the default be longer (e.g., 6 or 12 months) for a desktop user who may compose less frequently than a mobile user?
- **OQ-6. Auto-identity selection and multiple recipients.** The auto-identity feature uses only the first recipient in the To field to determine the sender identity. If the user adds a second recipient from a different account, should the application warn or offer to switch?
- **OQ-7. System contact picker integration.** The source application provides a button to open the system's native contact picker as an alternative to typing. The exact integration path on Linux (Evolution Data Server vs. KAddressBook vs. a portal-based picker) needs to be determined during design.
- **OQ-8. Received-contacts default.** The source application disables received-contact suggestions by default, presumably to avoid suggesting addresses of spammers or mailing lists. Is this the right default for a desktop client, or should it be enabled by default with noreply/noreply filtering as sufficient protection?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Two-character threshold, not one.** The autocomplete threshold is set at two characters rather than one. This is a deliberate tradeoff: a single character would produce very long, unfocused suggestion lists for most users, while two characters significantly narrows the candidate set. This threshold applies uniformly to all recipient fields.
- **N-2. Received contacts off by default.** The decision to disable received-contact suggestions by default is a privacy/relevance tradeoff. Received mail includes mailing lists, automated senders, and potential spam — all of which would clutter autocomplete. Users who want this source can enable it explicitly. Noreply addresses are also filtered out from received-contact learning as an additional safeguard.
- **N-3. Deduplication prefers avatars.** When the same email address appears in both the local store and the system address book, the merged entry prefers whichever version has an avatar. This ensures that the richest visual representation is shown to the user, since system address-book entries are more likely to have photos.
- **N-4. Frequency vs. source-priority ranking.** The application offers two distinct ranking philosophies. Frequency/recency ranking rewards habitual correspondence patterns and is ideal for users with a small, stable set of contacts. Source-priority ranking surfaces system address-book contacts (which the user has explicitly curated) above automatically learned contacts, and is better for users who maintain a well-organized address book. The default is source-priority, reflecting the assumption that a curated address book is more trustworthy than raw frequency counts.
- **N-5. Starred contacts as a ranking signal.** System address-book contacts marked as starred/favorite receive an extreme frequency boost in the ranking algorithm, effectively pinning them to the top of suggestions. This respects a curation decision the user has already made in another application.
- **N-6. Chip encryption indicators.** Chips display an encryption-key indicator that distinguishes between PGP-only, S/MIME-only, and both-available states. This gives the user immediate feedback about encryption options before they reach the send button, reducing the chance of accidentally sending unencrypted mail to a recipient they intended to encrypt for.
- **N-7. Per-account contact storage.** Contacts in the local store are tagged with their associated account. This enables the "suggest from current account only" feature without requiring a separate store per account. It also means that the same external person may have multiple entries (one per account they've corresponded through), which is intentional — it preserves the association between a contact and the account used to reach them, which is critical for auto-identity selection.
- **N-8. Independent from notification and sync.** Autocomplete's local contact store is purely an application-level convenience feature. It has no interaction with folder synchronization settings, notification rules, or unified-inbox membership. Adding or removing a contact from the store has no side effects on any other subsystem.
