# Epic: Show Original Headers / Source / Parsed HTML / Alternative Text

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.17** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Show original headers / source / parsed HTML / alternative text for power users."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

By default, an email client reformats and sanitizes every message before displaying it — stripping dangerous scripts, normalizing fonts and colors, collapsing margins, and hiding remote images. This is the right trade-off for the majority of users, but it hides the real structure of the message.

Power users — mail administrators, security analysts, support engineers, and privacy-conscious recipients — regularly need to see what the mail server actually delivered:

1. **Original headers** — the full set of RFC 5322 / RFC 2822 headers, including Received chains, authentication results (SPF, DKIM, DMARC), transport security indicators, and any custom X-headers the sending infrastructure added.
2. **Message source** — the raw HTML (or plain text) body of the message exactly as transmitted, with its original styling, structure, and embedded references visible as code.
3. **Parsed / original HTML view** — the message rendered in a browser-like view using the sender's original HTML, without the client's reformatting, so that layout-dependent content (newsletters, HTML invitations, richly formatted reports) is displayed as the sender intended.
4. **Alternative body part** — when a message is sent as multipart/alternative (containing both an HTML part and a plain-text part), the ability to switch to whichever part is not currently displayed.
5. **Character-encoding override** — when a message declares an incorrect or ambiguous charset, the ability to force re-interpretation of the body under a different encoding.

Together these capabilities form a diagnostic and transparency toolkit that lets the user see exactly what arrived and make informed decisions about trust, formatting fidelity, and content integrity — without leaving the mail client.

## 2. Goals

- **G1.** Provide in-context access to the full, unabridged RFC headers of any message.
- **G2.** Provide a code-level view of the raw message body (HTML or plain text) with syntax highlighting and search.
- **G3.** Provide a rendered view of the sender's original HTML, bypassing the client's reformatting, while still protecting against the most dangerous content (scripts, form actions).
- **G4.** Allow toggling between the HTML part and the plain-text part of a multipart/alternative message.
- **G5.** Allow the user to override the declared character encoding when a message displays garbled text.
- **G6.** Gate privacy-sensitive views (original HTML, remote images) behind clear, per-sender-dismissable confirmation, so that casual users are protected while power users can opt out of repeated prompts.
- **G7.** Support exporting the raw message as an `.eml` file for external analysis or archival.

## 3. Non-Goals

- **NG1.** This epic does **not** cover the reformatted safe-HTML view itself (feature 3.6) or the reformatted plain-text view (feature 3.8). It covers only the mechanisms for switching *away* from those views to the original/raw content and back.
- **NG2.** This epic does **not** prescribe how headers or message bodies are fetched from the server or stored locally — only the user-visible behavior once data is available or while it is being fetched.
- **NG3.** Editing or modifying the message source is not a goal. All views described here are read-only inspections.
- **NG4.** Full MIME tree exploration (viewing individual MIME parts, nested attachments, or signed/encrypted sub-parts as a tree) is out of scope; this epic covers only the top-level body parts (HTML vs. plain text) and the headers.
- **NG5.** Authentication-result interpretation and display (SPF/DKIM/DMARC badges) is covered by feature 3.13, not here. This epic provides the raw header text from which those results are derived.

## 4. Glossary

| Term | Definition |
|---|---|
| **RFC headers** | The set of name-value pairs at the top of an email message as defined by RFC 5322 (and extensions). Includes From, To, Subject, Date, Received, Message-ID, authentication results, and any custom headers. |
| **Message source** | The raw body content of the message — typically HTML or plain text — displayed as code rather than rendered. |
| **Original / full HTML view** | The message body rendered using the sender's original HTML markup and styling, without the client's reformatting or sanitization (except for script and form-action removal for security). |
| **Reformatted view** | The client's default rendering of a message, with normalized fonts, colors, margins, and stripped unsafe elements. This is the baseline view that the features in this epic let users bypass. |
| **Multipart/alternative** | An email structure (RFC 2046) where the same content is provided in two or more formats — typically HTML and plain text — and the client selects one to display. |
| **Alternative body part** | Whichever part of a multipart/alternative message is not currently being displayed. |
| **Character encoding / charset** | The text encoding declared (or inferred) for the message body (e.g. UTF-8, ISO-8859-1, Windows-1252). An incorrect declaration causes garbled text. |
| **`.eml` file** | A standard file format (RFC 5322) representing a complete email message — headers plus body — suitable for import into any compliant mail client. |
| **Privacy confirmation** | A dialog shown before displaying original content that may leak information (e.g. remote images, tracking pixels), requiring explicit user consent. |
| **Tracking pixel** | A small (often 1×1) remote image embedded in a message whose purpose is to signal the sender that the message has been opened. |

## 5. Personas

- **P1. The mail administrator.** Manages mail servers and needs to trace delivery chains via Received headers, diagnose authentication failures (SPF/DKIM), and inspect transport-layer headers.
- **P2. The security analyst.** Investigates phishing or spam; needs the raw source to examine obfuscated HTML, embedded links, and header forgery.
- **P3. The privacy-conscious user.** Wants to inspect what remote resources a message references before allowing them to load, and wants to verify what the sender actually sent versus what the client chose to display.
- **P4. The formatting troubleshooter.** Sends or receives richly formatted HTML email (newsletters, invitations, reports) and needs to see the sender's original layout when the reformatted view looks wrong.
- **P5. The encoding victim.** Receives messages with garbled text because the sender's mail system declared the wrong charset, and needs to manually select the correct one.

## 6. User Stories

### 6.1 Viewing original headers

> **US-1.** As a mail administrator, I want to toggle the display of the full RFC headers of a message within the message view, so that I can inspect delivery chain, authentication results, and custom headers without leaving the reading pane.

> **US-2.** As any user, when headers are displayed, I want to copy all headers to the system clipboard with a single action, so that I can paste them into a support ticket, terminal, or text editor.

> **US-3.** As any user, I want to close the headers area without navigating away from the message, so that the reading view returns to normal.

> **US-4.** As any user, if the headers have not yet been downloaded, I want the application to fetch them from the server automatically when I request them, showing a progress indicator until they arrive.

> **US-5.** As any user, if the message is no longer available on the server (e.g. a POP3 message that was not left on the server), I want the header toggle to be visibly disabled so that I understand why I cannot see them.

### 6.2 Viewing message source

> **US-6.** As a security analyst, I want to open the raw HTML (or plain text) source of a message in a dedicated code viewer, so that I can read the markup with syntax highlighting and search for suspicious patterns.

> **US-7.** As any user viewing the source, I want a text-search function so that I can locate specific tags, URLs, or strings in large messages.

> **US-8.** As any user viewing the source, I want to toggle line numbers on or off, so that I can reference specific lines when discussing the source with others.

> **US-9.** As any user viewing the source, I want to toggle pretty-printing (formatted indentation) on or off, so that I can choose between seeing the source as transmitted and seeing a more readable layout.

> **US-10.** As any user viewing the source, I want to toggle between the original (unsanitized) HTML and the reformatted (sanitized) version the client normally displays, so that I can compare what the sender sent with what the client shows me.

> **US-11.** As any user viewing the source, I want to toggle whether URLs in the source are rendered as clickable links, so that I can choose between safe inspection and convenient navigation.

> **US-12.** As any user viewing the source, I want to validate the HTML and see any parse errors reported, so that I can diagnose rendering issues.

> **US-13.** As any user viewing the source, I want to save the displayed HTML to a file on disk, so that I can analyze it externally or archive it.

> **US-14.** As any user viewing the source in a dark theme, I want an option to force a light background in the code viewer, so that the source is readable regardless of theme.

### 6.3 Viewing original (parsed) HTML

> **US-15.** As a formatting troubleshooter, I want to switch the message display from the client's reformatted view to the sender's original HTML rendering, so that I can see the message as the sender intended.

> **US-16.** As a privacy-conscious user, the first time I request the original HTML view, I want a confirmation dialog warning me that showing the original message may leak privacy-sensitive information (remote images, tracking pixels), so that I make an informed choice.

> **US-17.** As a power user, I want the confirmation dialog to offer a "do not ask again for this sender" option and a "do not ask again for this sender's domain" option, so that I can trust specific senders or domains without disabling the warning globally.

> **US-18.** As a power user, I want a global "do not ask again for any message" option and a further "show original messages by default" option, so that I can disable the confirmation entirely once I understand the risks.

> **US-19.** As any user, I want the ability to reset all per-sender and global confirmation preferences from a settings screen, so that I can restore the safety prompts if my trust model changes.

> **US-20.** As any user, I want the original HTML view to inform me about detected tracking pixels and to offer a checkbox to always show images when viewing originals, so that I can balance convenience against privacy.

> **US-21.** As any user, I want to open the original HTML rendering in a full-screen view, so that I can see wide layouts (newsletters, tables) without the constraints of the normal reading pane.

> **US-22.** As any user viewing the original HTML in full-screen, I want options to fit content to the window width and to use a monospaced font for preformatted text blocks, so that the view adapts to different content types.

> **US-23.** As any user in a dark theme, I want the original HTML view to attempt a dark background rendering when possible, so that bright white newsletters do not blind me — but I understand this depends on rendering-engine capabilities.

### 6.4 Switching to the alternative body part

> **US-24.** As any user reading an HTML-rendered message, I want to switch to the plain-text part of the message (if one exists), so that I can see the text the sender provided for non-HTML clients — which may differ in content from the HTML part.

> **US-25.** As any user reading a plain-text-rendered message, I want to switch to the HTML part of the message (if one exists), so that I can see the richer formatting the sender provided.

> **US-26.** As any user, I want the menu label for this action to reflect the direction of the switch — e.g. "Load plain text" when I am viewing HTML, or "Load HTML" when I am viewing plain text — so that the action is unambiguous.

> **US-27.** As any user, I want this option to be disabled (or hidden) when the message has only one body part, or when the message is encrypted, so that I am not confused by an action that cannot succeed.

### 6.5 Overriding character encoding

> **US-28.** As an encoding victim, I want to select a character encoding from a list of all supported encodings, so that the message body is re-interpreted under the correct charset and the garbled text becomes readable.

> **US-29.** As any user, I want an "Automatic" option at the top of the encoding list, so that I can revert to the client's default charset detection after experimenting.

> **US-30.** As any user, I want the re-interpretation to fetch the body again from the server using the selected encoding, so that the conversion is faithful to the original byte stream — not a re-encoding of already-decoded text.

### 6.6 Exporting the raw message

> **US-31.** As a mail administrator, I want to save the complete raw message (headers + body + attachments) as an `.eml` file to a location I choose, so that I can analyze it in another tool, forward it as evidence, or archive it.

> **US-32.** As any user, I want to forward a message (or an entire conversation thread) as an `.eml` attachment in a new compose window, so that the recipient receives the original message structure intact.

> **US-33.** As any user, if the raw message has not yet been downloaded, I want the application to fetch it from the server before saving, showing a progress indicator, so that the export contains the complete message.

## 7. Functional Requirements

### 7.1 Header display

- **FR-1.** The application shall provide a toggle action (accessible from the message's context or overflow menu) to show or hide the full RFC headers within the message reading view.
- **FR-2.** Headers shall be displayed in a monospaced font, preserving the original line structure, and the text shall be user-selectable.
- **FR-3.** A copy-to-clipboard action shall be provided adjacent to the header display, copying the complete header text.
- **FR-4.** A close/dismiss action shall be provided adjacent to the header display, hiding the headers without navigating away.
- **FR-5.** If headers are not available locally when the user requests them, the application shall fetch them from the server, displaying a progress indicator until they arrive.
- **FR-6.** If fetching is not possible (message no longer on server, account type does not support it), the toggle action shall be visibly disabled and the reason should be apparent from context.
- **FR-7.** The header toggle state shall be remembered per message for the duration of the viewing session.

### 7.2 Message source viewer

- **FR-8.** The application shall provide an action (from the message's overflow menu) to open the raw message body in a dedicated code viewer.
- **FR-9.** The code viewer shall display the HTML (or plain text) with syntax highlighting.
- **FR-10.** The code viewer shall provide an in-content text search with next/previous navigation.
- **FR-11.** The code viewer shall offer toggleable options for: line numbers, clickable links, pretty-printing (formatted indentation), and a sanitized-vs-original toggle.
- **FR-12.** The sanitized-vs-original toggle shall switch between the client's processed version (with computed styles shown) and the raw message as transmitted.
- **FR-13.** The code viewer shall offer an HTML validation action that reports parse errors to the user.
- **FR-14.** The code viewer shall offer a save-to-file action, allowing the user to export the displayed content to a location of their choice.
- **FR-15.** When the application is using a dark theme, the code viewer shall offer a "force light background" toggle so that the source remains readable.

### 7.3 Original HTML rendering

- **FR-16.** The application shall provide a toggle (accessible from the message view) to switch between the reformatted view and the sender's original HTML rendering.
- **FR-17.** The original HTML rendering shall display the message using the sender's original fonts, colors, margins, and layout, with only dangerous elements removed (scripts, form actions, and similar active content).
- **FR-18.** Inline images that have already been downloaded shall be embedded in the original HTML rendering from the local cache. Remote images shall be subject to the application's existing image-blocking policy.
- **FR-19.** The application shall provide a full-screen mode for the original HTML rendering, allowing wide content to be viewed without reading-pane constraints.
- **FR-20.** In full-screen mode, the user shall be able to toggle fit-to-width scaling and monospaced rendering of preformatted text blocks.
- **FR-21.** When the application is using a dark theme and the rendering engine supports it, the original HTML view shall apply dark-background rendering. When the rendering engine does not support it, a notice shall inform the user that dark mode is not available for the original view.

### 7.4 Privacy confirmation

- **FR-22.** Before displaying the original HTML rendering for the first time (per sender), the application shall present a confirmation dialog warning the user that showing the original message may leak privacy-sensitive information.
- **FR-23.** The confirmation dialog shall include:
  - (a) A primary warning message linking to relevant documentation.
  - (b) A checkbox to always show images when viewing original messages.
  - (c) A notice about tracking-pixel risks (when tracking protection is enabled).
  - (d) A "do not ask again for [sender address]" checkbox.
  - (e) A "do not ask again for [@domain]" checkbox (enabled only after the sender checkbox is selected).
  - (f) A "do not ask again for any message" global checkbox.
  - (g) A "show original messages by default" checkbox (enabled only after the global checkbox is selected).
  - (h) A hint explaining where to undo these preferences in settings.
- **FR-24.** Per-sender and per-domain preferences shall persist across sessions.
- **FR-25.** A full-screen original view shall also present a (simpler) confirmation dialog before the first use, with a "do not ask again" option.
- **FR-26.** The application's settings shall provide a way to reset all per-sender, per-domain, and global original-message confirmation preferences.
- **FR-27.** Messages in the spam/junk folder shall bypass certain per-sender trust options (the "do not ask again for sender/domain" checkboxes shall be hidden), since junk senders should not accumulate trust.

### 7.5 Alternative body part

- **FR-28.** When a message contains both an HTML and a plain-text body part (multipart/alternative), the application shall provide a menu action to switch to the currently-hidden part.
- **FR-29.** The menu label shall reflect the target: "Load plain text" when HTML is displayed, "Load HTML" when plain text is displayed.
- **FR-30.** Switching shall re-fetch the body from the server to obtain the alternative part, with a progress indication.
- **FR-31.** This action shall be disabled or hidden when: (a) the message has only one body part, (b) the message is encrypted, or (c) the message is no longer available on the server for re-fetch.

### 7.6 Character-encoding override

- **FR-32.** The application shall provide a menu action to select a character encoding from a list of all encodings supported by the runtime, plus an "Automatic" option.
- **FR-33.** Selecting an encoding shall re-fetch the message body from the server and re-interpret it under the selected charset.
- **FR-34.** The "Automatic" option shall restore the client's default charset detection behavior.
- **FR-35.** This action shall be available only for account types that support re-fetching the body from the server.

### 7.7 Raw message export

- **FR-36.** The application shall provide a menu action to save the complete raw message (headers, body, all MIME parts) as an `.eml` file to a user-chosen location.
- **FR-37.** If the raw message is not yet available locally, the application shall fetch it from the server before saving, with a progress indication.
- **FR-38.** The application shall provide an action to forward a single message — or all messages in the current conversation thread — as `.eml` file attachment(s) in a new compose window.
- **FR-39.** Raw-message export actions shall be available only when the message can be re-fetched from the server (or was previously cached in raw form).

## 8. Non-Functional Requirements

- **NFR-1. Performance.** Opening the code viewer for a large message (up to several megabytes of HTML) shall complete within a few seconds. Syntax highlighting shall not freeze the UI.
- **NFR-2. Responsiveness.** Toggling between reformatted and original HTML views shall feel near-instantaneous for messages whose body is already cached locally.
- **NFR-3. Security.** The original HTML rendering shall never execute scripts, submit forms, or allow navigation away from the message content. The code viewer shall not execute any content it displays.
- **NFR-4. Privacy.** No original-HTML or full-screen view shall load remote resources unless the user has explicitly consented (via the confirmation dialog or a global setting). The confirmation system shall default to maximum privacy.
- **NFR-5. Offline.** Headers and source views shall work fully offline for any message whose headers and body are already cached. Encoding override and alternative-part switching require a server round-trip and shall indicate when offline.
- **NFR-6. Accessibility.** All toggle actions, menu items, copy buttons, and close buttons shall be keyboard-accessible and carry screen-reader labels. The monospaced header text and code viewer content shall respect the system font-size preference.
- **NFR-7. Theme support.** The header display, code viewer, and original HTML renderer shall all respect the application's light/dark/black theme settings, with explicit fallbacks (force-light toggle) where dark rendering is not supported.

## 10. Open Questions

- **OQ-1. Scope of "original" rendering.** The original HTML view removes scripts and form actions but otherwise renders the sender's markup faithfully. The exact boundary between "safe-to-keep" and "unsafe-to-render" elements (e.g. embedded objects, iframes, applets) should be defined during design; the source application applies a relatively aggressive sanitizer even for the "original" view, but the contract is not fully documented.
- **OQ-2. Tracking-pixel indicator visibility.** The confirmation dialog shows a tracking-pixel warning contingent on a separate "disable tracking" preference. If tracking protection is turned off, should the warning still appear (informational) or should it be hidden? The source application hides it; this may merit reconsideration.
- **OQ-3. Charset override persistence.** When the user selects a charset override, is it remembered for that message on subsequent views, or is it session-only? The source application appears to re-fetch each time; a persistent per-message override may be more convenient.
- **OQ-4. Alternative-part content differences.** Some senders include materially different content in their HTML and plain-text parts (e.g. the plain-text part is truncated or says "view in browser"). Should the application warn the user when switching, or is silent switching sufficient? Currently silent.
- **OQ-5. Debug-only features.** The source application exposes additional diagnostic information (TLS details, DKIM status, blocklist status, a thread-structure inspector, and an HTML DOM inspector) only in debug builds. Whether any of these should be promoted to user-facing features (perhaps behind an "advanced diagnostics" setting) is a design decision.
- **OQ-6. Domain trust granularity.** The per-domain trust option uses the full domain of the sender address. For organizations with many subdomains (e.g. `news.example.com`, `alerts.example.com`), should trust apply to the organizational domain (`example.com`) instead? The source application uses the exact domain.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Privacy-first default.** The confirmation dialog before showing original HTML exists because the original view may load remote images (including tracking pixels) that the reformatted view blocks by default. The graduated opt-out (per-sender → per-domain → global → default-on) is designed so that each step requires the previous, preventing accidental global exposure. This is a deliberate friction-for-safety trade-off.

- **N-2. Re-fetch rather than re-decode.** Both encoding override and alternative-part switching re-fetch the message body from the server rather than attempting to re-decode locally cached content. This ensures the conversion operates on the original byte stream, avoiding double-encoding artifacts. The trade-off is that these features require connectivity and a message that still exists on the server.

- **N-3. Separate code viewer.** The message source is displayed in a dedicated viewer (not inline in the reading pane) because raw HTML can be very large and benefits from a specialized environment with syntax highlighting, search, and scrolling that would conflict with the normal message-reading UI.

- **N-4. Junk-folder trust exclusion.** The per-sender and per-domain "do not ask again" checkboxes are hidden when viewing messages in the spam/junk folder. This prevents a scenario where a user casually trusts a spammer's address while triaging junk, which would then suppress the privacy warning for future messages from that sender if they reach the inbox.

- **N-5. Header display is inline, not a separate view.** Headers are shown within the message reading pane (above or below the body) rather than in a separate window. This lets the user cross-reference headers with message content without window-switching. The copy and close buttons keep the area self-contained.

- **N-6. Full-screen as escalation.** The original HTML view is available in two levels: inline (within the reading pane, where it replaces the reformatted view) and full-screen (a dedicated window). The inline toggle is the common case; full-screen is an escalation for content that needs more space. Each level has its own confirmation gate.

- **N-7. POP3 limitations are surfaced, not hidden.** For POP3 accounts that do not leave messages on the server, header fetching, encoding override, alternative-part switching, and raw-message export are all disabled rather than silently failing. The user sees a disabled control and can infer the reason from the account type. This is preferable to showing an action that always errors.
