# Epic: Mark as Read on View, on Scroll, or Never

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.5** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Mark as read on view, on scroll, or never — with an optional delay and an undo grace period."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[SHOULD]` — important, planned for v1 but acceptable to slip to v1.x if pressed.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

When a user opens or scrolls through their email, the question of *when* a message should be considered "read" is surprisingly personal. Some users want messages marked read the instant they expand them; others want a short grace period so that a glance does not lose the "unread" marker; others want full manual control and never want a message auto-marked.

This feature governs the automatic transition of a message's read/unread state based on how the user interacts with it, and provides three complementary safety nets:

1. **A configurable delay** before the automatic mark-read fires, so that briefly glancing at a message does not consume its unread status.
2. **A per-account opt-out**, so that specific accounts can be excluded from auto-marking entirely (the "never" mode).
3. **An undo grace period** on destructive triage actions (archive, trash, delete, move) during which the action can be cancelled, and which interacts with the mark-read timing.

The feature also extends to *implicit* read-marking: an option to automatically mark messages as read when they are moved (archived, trashed, or filed), since the act of filing often signals that the user has dealt with the message.

Together these controls let each user calibrate the read-state behavior to match their triage style — from aggressive auto-read to fully manual — without losing information they care about.

## 2. Goals

- **G1.** Give the user explicit control over *when* expanding a message causes it to be marked as read: immediately, after a configurable delay, or never.
- **G2.** Provide a per-account override so that individual accounts can opt out of automatic mark-as-read entirely, independent of the global delay setting.
- **G3.** Offer an optional behavior that marks messages as read automatically when they are moved, archived, or trashed, so that filing doubles as triage.
- **G4.** Provide a configurable undo grace period for triage actions, during which the user can reverse the action before it is committed.
- **G5.** Ensure that the read/unread state is always synchronized bidirectionally with the mail server — local changes propagate to the server and server changes propagate locally.
- **G6.** Ensure that message bodies can be fetched from the server without implicitly marking the message as read on the server (peek semantics).

## 3. Non-Goals

- **NG1.** This epic does **not** define the visual rendering of read vs. unread messages (font weight, color, badge) — only the transitions between the two states.
- **NG2.** This epic does **not** define notification behavior on read-state changes; notification dismissal is governed by the notification subsystem and is documented separately.
- **NG3.** This epic does **not** cover "mark all as read" as a bulk folder action — that is a folder-operations concern. It does cover the automatic marking of *individual* messages during normal reading.
- **NG4.** This epic does **not** prescribe scroll-based marking as a distinct trigger mode. The source application does not implement automatic mark-as-read on scroll; the "on scroll" in the feature name refers to the user naturally scrolling through an expanded message, during which the delay timer is already running. There is no separate "mark read when scrolled past" behavior.
- **NG5.** This epic does **not** define filter-rule actions that set read state (e.g. "mark read when moving via a rule"). Rule actions are covered by the Rules & Automation epic.

## 4. Glossary

| Term | Definition |
|---|---|
| **Read / Seen** | A message whose read flag is set. Displayed with reduced visual emphasis in the message list. The server-side equivalent is the IMAP `\Seen` flag. |
| **Unread / Unseen** | A message whose read flag is not set. Displayed with heightened visual emphasis. |
| **Expand** | The action of opening a message's body for reading within a conversation view. In a single-message view, equivalent to opening the message. |
| **Auto-mark-read** | The application's behavior of automatically setting the read flag when a message is expanded, subject to delay and per-account settings. |
| **Mark-read delay** | A configurable time interval between the moment a message is expanded and the moment the application automatically sets its read flag. If the message is collapsed before the delay elapses, the flag is not set. |
| **Peek** | A protocol-level mode in which the application fetches message content from the server without causing the server to set the read flag. The application controls read-state transitions exclusively through explicit operations. |
| **Auto-read on move** | An optional behavior that automatically marks a message as read when the user moves, archives, or trashes it. |
| **Undo grace period** | A configurable time window after a triage action (archive, trash, delete, move) during which the action is held locally and can be reversed by the user before being committed to the server. |
| **Account auto-seen** | A per-account boolean setting that controls whether expanding a message in that account triggers auto-mark-read. When disabled, messages in that account are never auto-marked; the user must mark them manually. |

## 5. Personas

- **P1. The speed-triager.** Reads email quickly, expanding and collapsing messages in rapid succession. Wants messages marked read the instant they are expanded so that the unread count drops as they work.
- **P2. The careful reader.** Scans messages briefly before deciding which to read fully. Wants a delay before auto-marking so that a quick glance does not lose the unread indicator.
- **P3. The manual-control user.** Uses read/unread as a personal task queue. Never wants messages auto-marked; marks them manually when they have been actioned.
- **P4. The file-and-forget user.** Archives or files messages as their primary triage action. Wants filing to implicitly mark messages read, saving a separate step.
- **P5. The multi-account user.** Has some accounts where auto-mark-read is useful (personal) and others where it is not (shared team mailbox). Needs per-account control.

## 6. User Stories

### 6.1 Auto-mark-read on expand

> **US-1.** As a speed-triager, when I expand a message, I want it to be marked as read immediately so that my unread count reflects what I have seen.

> **US-2.** As a careful reader, when I expand a message, I want a configurable delay before it is marked as read, so that briefly glancing at a message does not consume its unread status.

> **US-3.** As a careful reader, if I collapse a message before the mark-read delay elapses, I want the message to remain unread, so that glancing and moving on does not lose information.

> **US-4.** As a manual-control user, I want to disable auto-mark-read entirely for a specific account, so that messages in that account are never marked read unless I explicitly do so.

> **US-5.** As any user, I want the global mark-read delay setting to apply to all accounts by default, so that I configure it once and it applies everywhere.

> **US-6.** As a multi-account user, I want each account to independently opt out of auto-mark-read, overriding the global setting, so that I can have different behavior per account.

### 6.2 Peek semantics

> **US-7.** As any user, when I open a message to read its body, I want the application to fetch the content from the server without the server marking the message as read on its own, so that read-state transitions are entirely under the application's (and my) control.

### 6.3 Auto-read on move

> **US-8.** As a file-and-forget user, I want an option to automatically mark messages as read when I move, archive, or trash them, so that filing doubles as marking read and I save a step.

> **US-9.** As any user, I want the auto-read-on-move option to be off by default, so that moving a message does not silently change its read state unless I have opted in.

### 6.4 Undo grace period

> **US-10.** As any user, when I perform a triage action (archive, trash, delete, move), I want a configurable grace period during which I can undo the action before it is committed to the server.

> **US-11.** As any user, I want to be able to set the undo grace period to a value from a reasonable range (including "off" for immediate execution), so that I can balance speed against safety.

> **US-12.** As any user, when the undo grace period is active, I want a visible, transient notification or status indicator that allows me to trigger the undo, so that I can act quickly if I made a mistake.

### 6.5 Manual read/unread toggle

> **US-13.** As any user, I want to manually toggle any message between read and unread at any time, regardless of the auto-mark-read settings, so that I always have final control.

> **US-14.** As any user, I want the manual read/unread toggle to be available from the message view (as a button), from a context menu, from a keyboard shortcut, and as a configurable swipe action, so that it is easy to reach in any workflow.

> **US-15.** As any user, when I manually mark a message as unread while it is still expanded, I want the auto-mark-read timer (if any) to not re-fire for that expansion, so that my manual override is respected.

### 6.6 Threading interaction

> **US-16.** As any user viewing a conversation, when I expand one message and it is auto-marked as read, I want only that specific message to be marked — not the entire thread — so that unread indicators on other messages in the conversation remain accurate.

> **US-17.** As any user, when I manually toggle read/unread on a message within a threaded conversation, I want the toggle to apply to all related messages in the thread (across folders), so that the conversation's read state is consistent.

### 6.7 Interaction with close-on-unread

> **US-18.** As any user, I want an option that automatically closes (collapses) a conversation when I manually mark a message as unread, so that the message returns to the list as an unread item I can revisit later.

### 6.8 Server synchronization

> **US-19.** As any user, when a message's read state changes locally (by auto-mark, manual toggle, or auto-read-on-move), I want the change to be propagated to the mail server so that other clients see the same state.

> **US-20.** As any user, when a message's read state changes on the server (by another client or a webmail interface), I want the local state to update to match, so that the application always reflects the true server state.

## 7. Functional Requirements

### 7.1 Global mark-read delay

- **FR-1.** The application shall provide a global setting — "Delay automatically marking messages as read" — that controls the time between expanding a message and automatically setting its read flag.
- **FR-2.** The available delay values shall be: Off (0 — immediate), 1.25 s, 2.5 s, 5 s, 7.5 s, 10 s, 15 s, 30 s, and 60 s.
- **FR-3.** The default value shall be Off (0), meaning messages are marked read immediately on expand.
- **FR-4.** When the delay is set to Off and the account's auto-seen setting is enabled, expanding a message shall mark it as read immediately.
- **FR-5.** When the delay is greater than zero and the account's auto-seen setting is enabled, expanding a message shall start a timer. If the message is still expanded when the timer elapses, it shall be marked as read. If the message is collapsed or navigated away from before the timer elapses, it shall remain unread.

### 7.2 Per-account auto-seen override

- **FR-6.** Each account shall have an independent boolean setting — "Mark messages read on expanding" — that controls whether expanding a message in that account triggers auto-mark-read.
- **FR-7.** This setting shall default to enabled (auto-mark-read active).
- **FR-8.** When this setting is disabled for an account, no message in that account shall be auto-marked as read on expand, regardless of the global delay setting.
- **FR-9.** The global settings screen shall display a hint indicating that auto-mark-read can be disabled per account in the account's advanced settings.

### 7.3 Peek semantics

- **FR-10.** The application shall fetch message bodies from the mail server using peek semantics (i.e. without causing the server to set the read flag as a side effect of the fetch).
- **FR-11.** All read-state transitions shall be performed by the application through explicit flag-change operations, never as an implicit side effect of content retrieval.

### 7.4 Auto-read on move

- **FR-12.** The application shall provide a global boolean setting — "Automatically mark messages read on moving messages" — that controls whether moving, archiving, or trashing a message also marks it as read.
- **FR-13.** This setting shall default to disabled (off).
- **FR-14.** When enabled, any user-initiated move, archive, or trash action shall set the read flag on the affected message(s) as part of the same operation.

### 7.5 Undo grace period

- **FR-15.** The application shall provide a global setting — "Undo timeout" — that controls the grace period for triage actions.
- **FR-16.** The available undo timeout values shall be: Off (0 — immediate execution), 1.25 s, 2.5 s, 5 s, 7.5 s, 10 s, 15 s, 30 s, and 60 s.
- **FR-17.** The default value shall be 5 seconds.
- **FR-18.** When the undo timeout is greater than zero, triage actions (archive, trash, delete, move) shall be held locally for the duration of the grace period before being committed.
- **FR-19.** During the grace period, the application shall display a transient, dismissible indicator (e.g. a status bar message or toast) offering an "Undo" action.
- **FR-20.** If the user triggers the undo action within the grace period, the triage action shall be reversed and the message(s) shall return to their prior state (including their prior read/unread state).
- **FR-21.** If the undo timeout is set to Off, triage actions shall be committed immediately with no undo opportunity.

### 7.6 Manual read/unread toggle

- **FR-22.** The user shall be able to manually toggle any message's read/unread state at any time.
- **FR-23.** The manual toggle shall be available from: (a) a button in the message view (optionally shown via button customization), (b) a context menu action, (c) a configurable keyboard shortcut, and (d) a configurable swipe action in the message list.
- **FR-24.** Swipe-to-toggle-read shall be available as one of the configurable swipe directions.
- **FR-25.** When a message is manually marked as unread while it is still expanded, the auto-mark-read timer (if running) shall be cancelled and shall not re-fire for the current expansion.

### 7.7 Threading interaction

- **FR-26.** When a message is auto-marked as read (via the expand-and-delay mechanism), only that specific message shall have its read flag changed. Other messages in the same conversation shall not be affected.
- **FR-27.** When a message is manually toggled between read and unread by the user, the toggle shall apply to all related messages in the same conversation thread, across all folders where those messages reside.

### 7.8 Close-on-unread

- **FR-28.** The application shall provide a boolean setting — "Close conversations on marking messages as unread" — that, when enabled, automatically closes (collapses) the current conversation view when the user manually marks a message as unread.
- **FR-29.** This setting shall default to disabled.

### 7.9 Server synchronization

- **FR-30.** Any local change to a message's read flag (whether by auto-mark, manual toggle, or auto-read-on-move) shall be queued for synchronization to the mail server.
- **FR-31.** Read-flag changes received from the server shall be reflected locally, updating the message's display state without user intervention.
- **FR-32.** If the application is offline when a read-state change occurs locally, the change shall be queued and synchronized when connectivity is restored.

### 7.10 Rule integration surface

- **FR-33.** When a filter rule action moves, hides, or snoozes a message, the rule shall offer an optional "Mark read" modifier that sets the read flag as part of the rule action.
- **FR-34.** This modifier shall be independent of the global auto-read-on-move setting and shall be configured per rule.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** The auto-mark-read operation (when the delay has elapsed) shall complete and update the message's visual state within 200 ms under normal load.
- **NFR-2. Timer accuracy.** The mark-read delay timer shall fire within ±250 ms of the configured delay value under normal system load.
- **NFR-3. No data loss.** The peek-fetch requirement (FR-10) shall be upheld for every supported protocol (IMAP, POP3, JMAP), so that viewing a message never irreversibly changes its state on the server.
- **NFR-4. Offline resilience.** Read-state changes made while offline shall survive application restarts and shall be replayed in order when connectivity returns.
- **NFR-5. Consistency.** A message's read/unread state as displayed in any view (folder list, conversation, unified inbox, search results) shall always reflect the most recent local state, with no stale indicators.
- **NFR-6. Accessibility.** The manual read/unread toggle shall be reachable via keyboard and shall carry a screen-reader label indicating the current state and the action it will perform. The undo indicator shall be announced to assistive technology.
- **NFR-7. Privacy.** No read-state telemetry or tracking shall be introduced. Read-state changes shall only travel between the application and the user's own mail server.

## 10. Open Questions

- **OQ-1. "On scroll" semantics.** The high-level feature name includes "on scroll", but the source application does not implement a distinct scroll-based auto-mark-read trigger. The "on scroll" behavior in practice is that the mark-read delay timer runs while the user reads (and possibly scrolls through) an expanded message. Should a future implementation introduce a true "mark read when scrolled past in the list" mode, or is the current expand-plus-delay model sufficient?
- **OQ-2. Manual toggle threading asymmetry.** The source application marks all thread messages as read/unread on a manual toggle, but only marks the single expanded message on auto-mark-read. This asymmetry may surprise users. Should the manual toggle also be single-message, or is the current behavior intentional for batch-triage purposes?
- **OQ-3. Undo scope.** The undo grace period currently applies to triage actions (move, archive, trash, delete) but not to auto-mark-read itself. Should there be an explicit undo mechanism for "this message was just auto-marked as read" (beyond the ability to manually toggle it back)?
- **OQ-4. Interaction between auto-read-on-move and undo.** If auto-read-on-move is enabled and the user undoes a move during the grace period, the message should presumably revert to its original read state. The source application appears to handle this, but edge cases (e.g. message was unread, moved, auto-marked read, then undone) should be explicitly tested.
- **OQ-5. Per-folder auto-seen override.** The source application provides per-account but not per-folder auto-seen overrides. Would a per-folder override be valuable (e.g. auto-mark in Inbox but not in a "Review" folder within the same account)?
- **OQ-6. Delay timer and rapid expand/collapse.** If a user expands a message, collapses it before the delay, then re-expands it, should the timer restart from zero? The source application appears to restart the timer, but this should be confirmed and documented.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Peek-first architecture.** The application always fetches message bodies in peek mode at the protocol level, ensuring the server never marks a message as read as a side effect of display. This is critical: without peek mode, any delay or "never" setting would be undermined by the server marking the message read the moment its body is fetched. Peek mode is the foundation that makes all other auto-seen controls meaningful.
- **N-2. Delay and auto-seen are orthogonal.** The global delay setting and the per-account auto-seen toggle are independent axes. The delay controls *how long to wait* before auto-marking; the per-account toggle controls *whether to auto-mark at all*. When auto-seen is disabled, the delay is irrelevant. When auto-seen is enabled and the delay is zero, marking is immediate. This two-axis model avoids a single dropdown with confusing combined states.
- **N-3. Auto-read-on-move is separate from auto-seen.** The "mark read on move" option is a distinct setting from "mark read on expand". A user who wants manual control of read state during reading may still want filing to imply reading. Conversely, a user who auto-marks on expand may not want trashing to mark a message as read (since they may be trashing without reading). Keeping these independent respects both workflows.
- **N-4. Default-off for auto-read-on-move.** The source application defaults this to off, which is the conservative choice: moving a message does not silently change a flag the user did not ask to change. Users who discover the option find it a welcome convenience; users who never discover it lose nothing.
- **N-5. Undo grace period is shared.** The undo timeout value is shared across all undoable triage actions (archive, trash, delete, move) rather than being configurable per action type. This keeps the settings surface small and predictable — one timeout value, one mental model.
- **N-6. Close-on-unread as a workflow accelerator.** The "close conversation on marking unread" option exists because a common workflow is: expand a message, realize it needs attention later, mark it unread, and return to the list. Without this option, the user must mark unread *and then* manually close the conversation — two steps. The option collapses them into one.
- **N-7. Thread toggle vs. single-message auto-mark.** Auto-mark-read affects only the single expanded message because the user has only *seen* that message. Manual toggle affects the whole thread because when a user explicitly marks a conversation as read or unread, they typically intend the action to apply to the entire conversation, not just the message their cursor happens to be on. This asymmetry is intentional and mirrors the convention in most email clients.
