# Epic: External-Image Confirmation

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.10** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Block remote images by default; one-click 'show images' with optional 'always allow this sender / domain'."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email messages frequently reference images hosted on remote servers — everything from inline logos and marketing banners to invisible single-pixel tracking beacons. When the mail client fetches those images, the remote server learns that the recipient opened the message, from which IP address, at what time, and often which client they used. For spam and phishing messages, this confirms a live address; for marketing and surveillance senders, it feeds a behavioral profile.

A privacy-respecting email client must therefore **block remote images by default** and give the user an explicit, informed choice before any remote fetch occurs. At the same time, the blocking must not be so burdensome that users disable it entirely: a single confirmation click per message, with the option to permanently whitelist a trusted sender or domain, keeps the feature practical for everyday use.

This feature covers the full lifecycle of that interaction: the default-blocked state, the per-message "show images" action, the confirmation dialog with its privacy warning and whitelist options, the per-sender and per-domain persistence of the whitelist, and the global settings that govern the behavior. It also defines the boundary with the separate tracker-image detection feature (3.11), which operates as an additional layer on top of the image-blocking mechanism described here.

## 2. Goals

- **G1.** Block all remotely hosted images in email messages by default, preventing any network request for image content until the user explicitly opts in.
- **G2.** Provide a single, discoverable "show images" action on every message that has blocked images, so that displaying them is always one click away.
- **G3.** Before loading images, present a confirmation dialog that warns the user about the privacy implications, unless the user has opted out of confirmations for that sender, domain, or globally.
- **G4.** Allow the user to whitelist a specific sender address so that future messages from that sender load images automatically without a confirmation dialog.
- **G5.** Allow the user to whitelist a specific sender domain so that future messages from any address at that domain load images automatically without a confirmation dialog.
- **G6.** Allow the user to disable the confirmation dialog globally (show images for all messages without asking), while keeping the option reversible.
- **G7.** Preserve embedded (inline / data-URI) images at all times — only images requiring a network fetch are subject to blocking.
- **G8.** Integrate cleanly with the tracker-image detection feature (3.11): even when the user chooses to show images, images identified as tracking pixels remain suppressed unless the user has disabled tracker detection.

## 3. Non-Goals

- **NG1.** This epic does **not** define how tracking images are detected or reported — that is the scope of feature 3.11 (Tracker-image detection). This epic only specifies that tracker detection, when enabled, acts as a second gate that survives the "show images" action.
- **NG2.** This epic does **not** cover the "show original / full HTML" confirmation, which is a parallel but distinct mechanism. Although the two share a similar dialog pattern and per-sender persistence model, they protect against different risks and are configured independently.
- **NG3.** This epic does **not** prescribe how the HTML content is sanitized or rendered — only that remote image references must not trigger network fetches until the user has authorized them.
- **NG4.** Proxying images through an intermediary server (as some providers do) is out of scope. The feature blocks or allows direct fetches; proxy behavior is a separate concern.
- **NG5.** This epic does **not** define image resizing, format conversion, or EXIF stripping for outgoing images — those are compose-side features.

## 4. Glossary

| Term | Definition |
|---|---|
| **Remote image** | An image referenced in a message body via a URL that requires a network fetch (typically `http:` or `https:`) to retrieve the content. Also called an "external image". |
| **Embedded image** | An image whose content is already present in the message — either attached and referenced by a content identifier, or encoded inline as a data URI. Embedded images do not require a network fetch. |
| **Tracking image** | A remote image (typically a single transparent pixel or very small graphic) placed in a message specifically to detect when, where, and by whom the message is opened. Detected by the tracker-image detection feature (3.11). |
| **Sender address** | The email address of the message's author as displayed to the user (typically the `From` header value). Used as the key for per-sender whitelisting. |
| **Sender domain** | The domain part of the sender address (everything after the `@`). Used as the key for per-domain whitelisting. |
| **Image whitelist** | The set of sender addresses and domains for which the user has chosen to allow remote images without a confirmation dialog. Persisted locally as user preferences. |
| **Confirmation dialog** | The modal dialog presented when the user requests to show images for a message that has not been whitelisted, warning about privacy implications and offering whitelist options. |
| **Show-images action** | The user-facing control (button or menu item) that triggers the display of remote images for the current message. |

## 5. Personas

- **P1. The privacy-conscious user.** Leaves all defaults in place; reviews the confirmation dialog each time; selectively whitelists only senders they truly trust. Values the privacy warning and tracking-image note.
- **P2. The convenience-oriented user.** Finds per-message confirmation tedious; whitelists frequently contacted senders and domains early on, or eventually disables the confirmation globally while relying on tracker detection as a safety net.
- **P3. The power user.** Understands the technical implications; uses per-domain whitelisting strategically (e.g. whitelists their employer's domain, their bank); keeps confirmation enabled for unknown senders.
- **P4. The newcomer.** Does not yet understand why images are blocked; needs the confirmation dialog's explanatory text to make an informed decision; benefits from the "this can be undone in settings" hint.

## 6. User Stories

### 6.1 Default blocking

> **US-1.** As any user, when I open an email message that contains remote images, I want those images to be blocked by default so that no network request is made on my behalf without my knowledge.

> **US-2.** As any user, when remote images are blocked, I want embedded images (attached inline or encoded as data URIs) to still display normally, so that the message remains readable for content the sender included directly.

> **US-3.** As any user, when remote images are blocked, I want a clear visual indicator that the message contains blocked images, so that I know images are available if I choose to show them.

### 6.2 One-click show images

> **US-4.** As any user, I want a single, discoverable action (button or toolbar control) on the message view that loads all remote images for the current message, so that I can see the full message with one click.

> **US-5.** As any user, after I have loaded images for a message, I want to be able to toggle them back off (re-block) for that message, so that I can undo the action without navigating away.

> **US-6.** As any user, when I switch between viewing messages, I want each message's image-display state to be independent, so that showing images on one message does not affect other messages.

### 6.3 Confirmation dialog

> **US-7.** As any user, when I click the show-images action and image confirmation is enabled, I want to see a dialog that warns me about the privacy implications of loading remote images, so that I can make an informed decision.

> **US-8.** As any user, in the confirmation dialog, I want to see a note that images identified as tracking images will remain blocked even if I proceed, so that I understand the layered protection.

> **US-9.** As any user, I want the confirmation dialog to offer a "do not ask again for [sender address]" option, so that I can whitelist a trusted sender without leaving the message.

> **US-10.** As any user, I want the confirmation dialog to offer a "do not ask again for [@domain]" option (available after selecting the sender option), so that I can whitelist an entire domain in one step.

> **US-11.** As any user, I want the confirmation dialog to offer a "do not ask again" (global) option, so that I can disable the confirmation entirely if I find it unnecessary.

> **US-12.** As any user, when I select the global "do not ask again" option, I want an additional "show images by default" option to appear, so that I can choose between merely skipping the dialog and also removing the image-blocking default.

> **US-13.** As any user, I want the confirmation dialog to include a hint that the choice can be reversed in settings, so that I feel safe experimenting with the options.

> **US-14.** As any user, I want the confirmation dialog to offer a shortcut to the privacy settings screen, so that I can adjust related settings without hunting for them.

### 6.4 Sender and domain whitelisting

> **US-15.** As any user, after I whitelist a sender address, I want future messages from that exact address to load images automatically without showing a confirmation dialog, so that my trust decision persists.

> **US-16.** As any user, after I whitelist a sender domain, I want future messages from any address at that domain to load images automatically without showing a confirmation dialog, so that one decision covers the whole organization.

> **US-17.** As any user, I want sender-level and domain-level whitelist entries to be independent, so that removing one does not affect the other.

> **US-18.** As any user, I want to be able to review and clear all my whitelist entries from the privacy settings screen, so that I can start fresh if I have been too permissive.

### 6.5 Global settings

> **US-19.** As any user, I want a master toggle in privacy settings that controls whether remote images are blocked by default, so that I have a single place to enable or disable the feature.

> **US-20.** As any user, I want a separate toggle that controls whether the confirmation dialog is shown (independently of whether blocking is active), so that I can block images but skip the dialog if I prefer a simple toggle-button workflow.

> **US-21.** As any user, I want a setting that automatically shows images whenever I switch to viewing the original / full HTML rendering of a message, so that the "show original" action implies "show everything".

> **US-22.** As any user, when I disable the "confirm before showing images" setting, I want all previously saved sender and domain whitelist entries to be cleared, so that stale per-sender data does not accumulate.

### 6.6 Interaction with junk / spam

> **US-23.** As any user, when I view a message in the Junk / Spam folder, I want the show-images action to work without a confirmation dialog regardless of the global setting, so that I can quickly inspect suspicious messages without extra friction.

### 6.7 Interaction with tracker detection

> **US-24.** As any user, when I choose to show images and tracker detection is enabled, I want images identified as tracking pixels to remain blocked and to be replaced with a visual indicator, so that I see the legitimate images without also leaking tracking data.

> **US-25.** As any user, I want the confirmation dialog's tracking-image note to be visible only when tracker detection is actually enabled, so that the dialog does not make promises the application cannot keep.

### 6.8 Auto-show when viewing original HTML

> **US-26.** As any user, when the "always show images in original view" setting is enabled and I switch to the original / full HTML rendering, I want images to load automatically for that message without a separate confirmation, so that viewing the original is a single action.

> **US-27.** As any user, when images are auto-shown because of the original-view setting, I want the show-images toggle to reflect that images are currently displayed, and I want to be able to re-block them for that message if I choose.

### 6.9 Reset

> **US-28.** As any user, I want a way to reset all image-related confirmation choices (including all sender and domain whitelist entries) back to defaults, so that I can recover from overly permissive decisions.

## 7. Functional Requirements

### 7.1 Default state

- **FR-1.** Remote images in all messages shall be blocked by default. No network request for image content shall be initiated until the user explicitly authorizes it for that message, sender, domain, or globally.
- **FR-2.** Embedded images (content-ID references resolved to local attachments, and data-URI images) shall always be displayed regardless of the remote-image blocking state.
- **FR-3.** The image-blocking default shall be represented by a persistent user preference, initially set to "block" (enabled).

### 7.2 Show-images action

- **FR-4.** Every message view shall expose a show-images action (button, toolbar item, or equivalent) that is visually present whenever the message contains or may contain remote images.
- **FR-5.** The show-images action shall function as a toggle: activating it loads remote images; activating it again re-blocks them for that message.
- **FR-6.** The visual state of the toggle shall clearly indicate whether images are currently shown or blocked for the current message.
- **FR-7.** Each message's show/block state shall be independent of every other message's state within the same session.

### 7.3 Confirmation dialog

- **FR-8.** When the user activates the show-images action and the confirmation preference is enabled, the application shall display a modal confirmation dialog before loading any images.
- **FR-9.** The confirmation dialog shall display a privacy warning explaining that showing images can leak privacy-sensitive information.
- **FR-10.** When tracker detection is enabled, the dialog shall display a note that images recognized as tracking images will not be shown even if the user proceeds.
- **FR-11.** When tracker detection is disabled, the tracking-image note shall be hidden.
- **FR-12.** The dialog shall offer a per-sender checkbox: "Do not ask this again for [sender address]".
- **FR-13.** The dialog shall offer a per-domain checkbox: "Do not ask this again for [@domain]". This checkbox shall be disabled by default and shall become enabled only when the per-sender checkbox is checked.
- **FR-14.** The dialog shall offer a global checkbox: "Do not ask this again". Checking this disables the confirmation dialog for all future messages.
- **FR-15.** When the global "do not ask again" checkbox is checked, a nested "Show images by default" checkbox shall become enabled, allowing the user to also disable blocking entirely.
- **FR-16.** The dialog shall display a hint that the choices can be undone in the privacy settings screen.
- **FR-17.** The dialog shall offer a button or link that navigates directly to the privacy settings screen.
- **FR-18.** Confirming the dialog (positive action) shall load images for the current message and persist any whitelist or global choices the user made.
- **FR-19.** Cancelling the dialog shall leave images blocked for the current message and shall not change any preferences.

### 7.4 Whitelist persistence

- **FR-20.** When the user checks the per-sender option and confirms, the application shall persistently record that the sender address is whitelisted for images.
- **FR-21.** When the user checks the per-domain option and confirms, the application shall persistently record that the sender domain is whitelisted for images.
- **FR-22.** Whitelist entries shall survive application restarts.
- **FR-23.** When a message is opened and its sender address or sender domain matches a whitelist entry, images for that message shall be loaded automatically without showing the confirmation dialog.
- **FR-24.** Domain matching shall use the domain part of the sender address (after the `@`). Subdomain handling shall follow the same rules as the rest of the application's domain processing.
- **FR-25.** Sender-level and domain-level whitelist entries shall be independent: removing one shall not affect the other; either one matching shall be sufficient to auto-load images.

### 7.5 Global preferences

- **FR-26.** The application shall expose the following image-related settings in the privacy settings screen:
  - (a) A master toggle for blocking remote images by default (on = block, off = allow).
  - (b) A toggle for showing the confirmation dialog before loading images (dependent on the master toggle being enabled).
  - (c) A toggle for automatically showing images when the user switches to the original / full HTML view.
- **FR-27.** Setting (b) shall only be configurable when setting (a) is enabled. If the user disables (a), (b) becomes moot.
- **FR-28.** When the user disables the confirmation dialog via setting (b), all previously saved per-sender and per-domain whitelist entries shall be cleared.

### 7.6 Junk / spam folder exception

- **FR-29.** When a message resides in a folder identified as Junk or Spam, activating the show-images action shall load images immediately without presenting the confirmation dialog, regardless of the global confirmation setting.

### 7.7 Interaction with original / full HTML view

- **FR-30.** When the "always show images in original view" setting is enabled and the user switches to the original / full HTML rendering of a message, images shall be loaded automatically for that message instance.
- **FR-31.** If the user subsequently toggles back to the safe/reformatted view, the image-display state shall revert to whatever it was before the original-view switch.

### 7.8 Interaction with tracker detection

- **FR-32.** When images are shown for a message (whether by user action, whitelist match, or global setting) and tracker detection is enabled, images that the tracker-detection subsystem identifies as tracking images shall remain blocked and shall be replaced with a visual indicator.
- **FR-33.** The tracker-detection gate shall operate independently of the image-confirmation gate: disabling image confirmation does not disable tracker detection, and vice versa.

### 7.9 Reset

- **FR-34.** The application shall provide a mechanism (in settings or via a general "reset asked questions" action) to clear all image-related preferences — the master toggle, the confirmation toggle, the original-view toggle, and all per-sender and per-domain whitelist entries — returning them to their factory defaults.

## 8. Non-Functional Requirements

- **NFR-1. Zero leakage by default.** When image blocking is enabled (the default), no network request for remote image content shall be made, under any code path, without the user having explicitly authorized it. This is the feature's primary security invariant.
- **NFR-2. Responsiveness.** Loading images after the user confirms shall begin immediately (within the responsiveness threshold of the message viewer) with no perceptible delay beyond the network fetch itself.
- **NFR-3. Persistence durability.** Whitelist entries and global preferences shall persist across application restarts, crashes, and system reboots.
- **NFR-4. Scalability.** The whitelist shall remain performant with at least several hundred sender and domain entries.
- **NFR-5. Discoverability.** The show-images action shall be visible without scrolling or opening menus when a message with blocked images is displayed. The confirmation dialog's options shall be self-explanatory to a non-technical user.
- **NFR-6. Accessibility.** All controls — the show-images toggle, the confirmation dialog and its checkboxes, and the privacy settings — shall be reachable via keyboard, carry appropriate screen-reader labels, and respect the application-wide theme.
- **NFR-7. Privacy.** Whitelist data (sender addresses and domains) shall be stored locally only and shall never be transmitted to any external service.
- **NFR-8. Independence.** The image-confirmation preference shall be independent of the "show original / full HTML" confirmation preference. Changing one shall not affect the other.

## 10. Open Questions

- **OQ-1. Subdomain whitelisting granularity.** When a user whitelists `@example.com`, should messages from `user@sub.example.com` also match? The source application stores the literal domain from the sender address; there is no evidence of parent-domain matching. Whether this is intentional or an oversight should be confirmed.
- **OQ-2. Whitelist review UI.** The source application clears all whitelist entries when the confirmation toggle is disabled and provides a bulk reset, but does not appear to offer a UI to view or selectively remove individual whitelist entries. Whether a list/management screen is desirable for the desktop application is an open design question.
- **OQ-3. Per-domain checkbox gating.** The source application enables the per-domain checkbox only when the per-sender checkbox is also checked, meaning a user cannot whitelist a domain without also whitelisting the specific sender. Whether this coupling is the best UX for the desktop application should be evaluated — a user might want to trust a domain without trusting every individual address they have seen from it.
- **OQ-4. Secure-connections-only for images.** The CHANGELOG mentions an option to "download images via secure connections only". Whether the desktop application should expose a similar toggle (block images served over plain HTTP even when the user has chosen to show images) is undecided.
- **OQ-5. Interaction with encrypted messages.** The high-level feature list (11.10) mentions "block external content in encrypted messages". Whether the image-confirmation feature should enforce stricter defaults for encrypted messages (e.g. never auto-load images even if the sender is whitelisted) is a cross-cutting concern not fully resolved in the source.
- **OQ-6. Session persistence of per-message state.** In the source application, a message's show/block state does not persist across sessions — reopening the message re-evaluates the whitelist but does not remember a one-time "show" choice. Whether the desktop application should persist one-time choices (per message, in the local store) is a design question.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Blocking is the default, not a preference the user must discover.** The most important property of this feature is that a new installation blocks remote images before the user has touched any setting. This is a privacy-by-default posture. The confirmation dialog then serves a dual purpose: it gates the network request *and* educates the user about why blocking exists. Removing or weakening the default would undermine the feature's core value.
- **N-2. Three-tier escalation: per-message → per-sender → per-domain → global.** The dialog is designed so that the user naturally escalates trust. The first time they see a message from a sender, they confirm once. If they trust the sender, they check the per-sender box. If they trust the whole organization, they enable the per-domain box (which requires the per-sender box first — a deliberate gate to prevent accidental broad whitelisting). If they trust all senders, they use the global toggle. Each tier is strictly broader than the previous, and each is independently reversible.
- **N-3. Domain checkbox is gated behind sender checkbox.** This prevents a user from accidentally whitelisting an entire domain on a first encounter. The intent is that per-domain trust is a conscious escalation from per-sender trust, not a shortcut.
- **N-4. Junk folder bypasses confirmation.** In junk/spam folders, the user is already in a suspicious-content context and may need to inspect the full message to evaluate whether it is truly spam. Requiring a confirmation dialog here adds friction without adding protection, because the user has already navigated to a folder they know contains untrusted mail.
- **N-5. Tracker detection is a separate gate.** Even when the user says "show images", tracking pixels remain blocked (when tracker detection is enabled). This means the user's "show images" action means "show the images I care about" rather than "leak my read receipt". The two features compose rather than override each other.
- **N-6. Clearing whitelist on disabling confirmation.** When the user turns off the confirmation dialog, the application clears all per-sender and per-domain whitelist entries. This prevents stale whitelist data from accumulating invisibly. If the user later re-enables confirmation, they start with a clean slate rather than inheriting decisions they may have forgotten.
- **N-7. Independence from the "show original" confirmation.** The image-confirmation dialog and the "show original / full HTML" confirmation dialog are parallel features that share a similar UI pattern but protect against different risks. Showing original HTML risks script injection and unsafe styling; showing images risks privacy leakage. They are configured, persisted, and triggered independently so that a user can, for example, always view original HTML but still be prompted about images, or vice versa.
- **N-8. Original-view auto-show as a convenience bridge.** The "always show images when viewing original" setting exists because users who request the original HTML rendering are typically doing so to see the message as the sender intended — and that almost always includes images. Without this setting, the user would need two separate actions (show original + show images) every time, which is unnecessarily tedious for this use case.
