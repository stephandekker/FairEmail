# Epic: Folder Subscription Management

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **2.4** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Subscribe/unsubscribe IMAP folders; create, rename, delete folders from the UI."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

IMAP servers expose a potentially large tree of folders (mailboxes) per account. Not every folder is relevant to every user: shared folders, legacy folders, and automatically generated folders often clutter the view. The IMAP protocol provides a **subscription** mechanism (RFC 3501 §6.3.6) that allows a client to mark which folders it cares about, and the server remembers this across sessions.

Beyond subscription, users need to shape the folder tree itself: creating new folders for organizing mail, renaming folders when a naming scheme evolves, and deleting folders that are no longer needed. These are standard IMAP operations (CREATE, RENAME, DELETE) that an email client should surface through the UI so the user is not forced to resort to a webmail interface or command-line tool.

This epic covers all four operations — subscribe, unsubscribe, create, rename, and delete — as a single cohesive feature, because they share the same context (acting on the folder tree), the same preconditions (IMAP connectivity), and the same interaction patterns (folder context menus and folder property screens).

## 2. Goals

- **G1.** Allow users to subscribe to and unsubscribe from any IMAP folder, with the change propagated to the server so that subscription state is consistent across all clients that connect to the same account.
- **G2.** Allow users to create new folders (including nested sub-folders) from within the application, without requiring an external webmail interface.
- **G3.** Allow users to rename user-created folders from within the application.
- **G4.** Allow users to delete user-created folders from within the application, with appropriate safeguards against accidental data loss.
- **G5.** Provide a "subscribed only" view filter that hides unsubscribed folders from the folder list, so that users who rely on subscription can keep their folder view clean.
- **G6.** Keep subscription management gated behind a global opt-in setting, so that users who do not use IMAP subscriptions are not burdened with extra UI controls.

## 3. Non-Goals

- **NG1.** This epic does **not** cover POP3 accounts. POP3 has no concept of server-side folders, subscriptions, or folder operations. Folder creation under POP3 (if supported at all) is a local-only concern not addressed here.
- **NG2.** This epic does **not** define synchronization behavior (poll intervals, IDLE, days to keep). Per-folder sync settings are covered by feature 2.3.
- **NG3.** This epic does **not** define how folders appear in the navigation pane, how they are colored, or how unread counters are displayed. Those are covered by features 2.6 and 2.7.
- **NG4.** This epic does **not** cover the Unified Inbox membership toggle (feature 3.1) or notification settings (feature 8.2), even though both appear on the same folder properties screen.
- **NG5.** Batch operations on child folders (e.g., "subscribe all children", "unsubscribe all children") are **not** a goal. Subscription is toggled one folder at a time.
- **NG6.** Moving or copying entire folders (as opposed to renaming) is not a goal.

## 4. Glossary

| Term | Definition |
|---|---|
| **Account** | A configured connection to a single mail provider, with credentials, server settings, and its own folder tree. |
| **Folder** | A container of messages within an account. Corresponds to an IMAP mailbox. Folders may be system folders or user-created folders. |
| **System folder** | A folder with a known role: Inbox, Sent, Drafts, Trash, Spam (Junk), Archive. These are auto-detected or user-assigned and receive special treatment. |
| **User folder** | Any folder that is not a recognized system folder. User folders are eligible for rename and delete. |
| **Subscription** | An IMAP concept (RFC 3501 §6.3.6). A subscribed folder is one the user has expressed interest in; the server maintains this list. Subscription state is independent of whether the folder is synchronized locally. |
| **Subscribed-only filter** | A view-level filter in the folder list that, when active, hides unsubscribed non-Inbox IMAP folders from display. |
| **Subscription management (global setting)** | An application-wide preference that controls whether subscription-related UI controls (subscribe toggle, subscribed-only filter) are shown at all. Disabled by default. |
| **Folder separator** | The character used by the IMAP server to delimit hierarchy levels in folder names (commonly `/` or `.`). Determined per-account from the server. |
| **Inferiors** | An IMAP folder attribute indicating whether the folder can contain child folders. A folder without the inferiors capability cannot be used as a parent for new sub-folders. |
| **Selectable** | An IMAP folder attribute indicating whether the folder can contain messages. Non-selectable folders serve only as namespace containers. |
| **Read-only folder** | A folder whose flags, messages, or attributes cannot be modified by the client (server-enforced). Subscribe/unsubscribe, rename, and delete are all blocked on read-only folders. |
| **Pending operation** | A queued action (move, flag, delete, etc.) targeting messages within a folder. Pending operations must drain before the folder itself can be deleted. |

## 5. Personas

- **P1. The default user.** Uses the application with default settings; has never interacted with IMAP subscriptions. The subscription UI is hidden from this user entirely. Folder create, rename, and delete are still available.
- **P2. The subscription-aware user.** Has enabled the global subscription management setting because their IMAP server hosts a large shared namespace or many legacy folders. Uses subscribe/unsubscribe to curate which folders appear, and the "subscribed only" filter to keep the list clean.
- **P3. The folder organizer.** Regularly creates, renames, and deletes folders to maintain a tidy mailbox hierarchy. Expects these operations to propagate to the server so that other clients see the same structure.
- **P4. The multi-client user.** Uses this application alongside webmail or another desktop client. Expects subscription changes made here to be visible in the other client, and vice versa.

## 6. User Stories

### 6.1 Subscription management opt-in

> **US-1.** As a default user, I want subscription-related controls to be hidden unless I explicitly opt in, so that the folder UI is not cluttered with concepts I do not use.

> **US-2.** As a subscription-aware user, I want a global setting in the synchronization preferences that enables subscription management, so that I can opt in once and have the controls appear throughout the application.

### 6.2 Subscribing and unsubscribing

> **US-3.** As a subscription-aware user, I want to toggle subscription on or off for any IMAP folder via a context action on the folder list, so that I can manage subscriptions without leaving the folder view.

> **US-4.** As a subscription-aware user, I want the subscribe/unsubscribe action to propagate to the IMAP server, so that other clients connecting to the same account see the same subscription state.

> **US-5.** As a subscription-aware user, I want to see a visual indicator on each folder showing whether it is currently subscribed, so that I can tell at a glance which folders are part of my subscription set.

> **US-6.** As a subscription-aware user, I do not want to be offered the subscribe/unsubscribe action on read-only folders, since the server will reject such changes.

### 6.3 Subscribed-only folder filter

> **US-7.** As a subscription-aware user, I want a "subscribed only" toggle in the folder list's menu that, when active, hides all unsubscribed IMAP folders from the list, so that I can focus on the folders I care about.

> **US-8.** As a subscription-aware user, when the "subscribed only" filter is active, I still want the Inbox and any folder that has visible children to remain displayed regardless of subscription state, so that the navigation hierarchy remains coherent.

> **US-9.** As a subscription-aware user, I want the "subscribed only" filter to apply only to IMAP accounts, so that folders from other account types (POP3, local) are unaffected.

> **US-10.** As a subscription-aware user, I want the "subscribed only" filter to be remembered across sessions, so that I do not have to re-enable it every time I launch the application.

### 6.4 Subscription synchronization

> **US-11.** As a multi-client user, I want the application to detect subscription changes made on the server (e.g., by webmail) during folder synchronization, so that the subscription status displayed in the application stays accurate.

### 6.5 Creating a folder

> **US-12.** As a folder organizer, I want to create a new top-level folder via a prominent action in the folder list view, so that folder creation is easily discoverable.

> **US-13.** As a folder organizer, I want to create a sub-folder under any existing folder (that supports children) via a context action on the parent folder, so that I can build a nested hierarchy.

> **US-14.** As a folder organizer, when creating a folder I want to specify at minimum: the folder name, display name, color, synchronization on/off, notification on/off, unified-inbox membership, and navigation-pane visibility, so that the new folder is ready to use immediately without a follow-up edit.

> **US-15.** As any user, I want the application to validate that the folder name is not empty and does not duplicate an existing folder at the same level, and to display a clear error if validation fails, so that I am not left guessing why the creation failed.

> **US-16.** As any user, I want newly created folders to be automatically subscribed on the server, so that other clients see them immediately.

> **US-17.** As any user, I expect that creating a sub-folder correctly uses the account's folder separator to construct the full hierarchical name, so that the folder appears at the correct level of nesting on the server.

### 6.6 Renaming a folder

> **US-18.** As a folder organizer, I want to rename any user-created folder through the folder properties screen, so that I can correct naming mistakes or reorganize without deleting and recreating.

> **US-19.** As any user, I want the rename to propagate to the IMAP server, so that other clients see the new name.

> **US-20.** As any user, I want the application to validate that the new name is not empty and does not duplicate an existing folder, and to display a clear error if validation fails.

> **US-21.** As any user, I want the rename to preserve the folder's subscription status — if the folder was subscribed before rename, it should remain subscribed under its new name.

> **US-22.** As any user, I do not want to be able to rename system folders (Inbox, Sent, Drafts, Trash, Spam, Archive), since renaming these could break server-side conventions and other clients.

### 6.7 Deleting a folder

> **US-23.** As a folder organizer, I want to delete any user-created folder via a context action on the folder list or from the folder properties screen, so that I can remove folders I no longer need.

> **US-24.** As any user, I want a confirmation dialog before deletion, warning that this action is irreversible and that all messages in the folder will be lost, so that I do not delete a folder by accident.

> **US-25.** As any user, I do not want to be able to delete a folder that still has child folders, so that I do not accidentally orphan or lose an entire sub-tree of mail.

> **US-26.** As any user, I do not want to be able to delete a folder while it has pending operations (queued moves, flags, etc.), so that those operations are not silently lost.

> **US-27.** As any user, I want the deletion to propagate to the IMAP server (unsubscribing first, then deleting), so that the folder is truly removed and not just hidden locally.

> **US-28.** As any user, I do not want to be able to delete system folders (Inbox, Sent, Drafts, Trash, Spam, Archive), since these are essential to normal mail flow.

> **US-29.** As any user, I do not want to be able to delete read-only folders, since the server will reject the operation.

### 6.8 Error handling

> **US-30.** As any user, if a folder operation fails on the server (quota exceeded, permission denied, namespace error, network failure), I want a clear error message displayed in the UI, so that I understand what went wrong and can take corrective action.

> **US-31.** As any user, if a creation or deletion fails on the server, I want the local state to be rolled back (the "to be created" or "to be deleted" flag cleared), so that the folder list does not show a phantom or half-deleted folder.

## 7. Functional Requirements

### 7.1 Global subscription management setting

- **FR-1.** The application shall provide a global preference (in the synchronization settings area) that enables or disables subscription management UI controls. The default shall be **disabled**.
- **FR-2.** When subscription management is disabled, no subscribe/unsubscribe toggle and no "subscribed only" filter shall be visible anywhere in the application.
- **FR-3.** When subscription management is enabled, the subscribe/unsubscribe toggle shall appear in the context menu of every non-read-only IMAP folder, and the "subscribed only" filter shall appear in the folder list's option menu.

### 7.2 Subscription state

- **FR-4.** Every IMAP folder shall have a persistent **subscribed** property that tracks whether the folder is subscribed on the server. The property may be null (unknown), true, or false.
- **FR-5.** When the user toggles subscription on a folder, the application shall queue a subscription operation that propagates the new state to the IMAP server.
- **FR-6.** The subscription toggle in the folder context menu shall be a checkable item that reflects the current subscription state.
- **FR-7.** A visual indicator (icon or badge) in the folder list shall distinguish subscribed folders from unsubscribed ones.

### 7.3 Subscription synchronization

- **FR-8.** During each folder synchronization cycle, the application shall query the server for the current subscription list and update the local subscription state of each folder to match.
- **FR-9.** If the server reports a subscription change (e.g., made by another client), the local state shall be updated without requiring user intervention.

### 7.4 Subscribed-only filter

- **FR-10.** The folder list shall provide a "subscribed only" toggle in its option menu, visible only when the global subscription management setting is enabled.
- **FR-11.** When "subscribed only" is active, the folder list shall hide all IMAP folders that are unsubscribed, **except**:
  - (a) The Inbox (always shown regardless of subscription).
  - (b) Any folder that has at least one visible child (to preserve hierarchy).
  - (c) Folders belonging to non-IMAP accounts (unaffected).
- **FR-12.** The "subscribed only" preference shall persist across application restarts.
- **FR-13.** Changes to the "subscribed only" filter shall take effect immediately on the displayed folder list.

### 7.5 Creating a folder

- **FR-14.** The folder list view shall provide a prominent action (e.g., a floating button) to create a new top-level folder for the current account. This action shall only be available for IMAP accounts.
- **FR-15.** The folder context menu shall provide a "Create sub-folder" action for any folder that supports child folders (inferiors = true). The action shall be disabled (greyed) for folders that do not support children.
- **FR-16.** The folder creation form shall accept at minimum: folder name (required), display name (optional), color (optional), synchronize on/off, poll on/off, download messages on/off, auto-classify on/off, notification on/off, unified-inbox membership on/off, navigation-pane visibility on/off, sync-days, keep-days, and auto-delete on/off.
- **FR-17.** The folder name shall be validated: it must not be empty, and no folder with the same full name (including hierarchy prefix) may already exist in the same account. A clear error shall be shown on validation failure.
- **FR-18.** When creating a sub-folder, the application shall construct the full folder name by concatenating the parent's name, the account's folder separator, and the user-provided name.
- **FR-19.** The creation shall be queued locally (marked as "to be created") and executed on the server during the next synchronization cycle.
- **FR-20.** Upon successful server-side creation, the folder shall be automatically subscribed on the server.
- **FR-21.** Upon successful server-side creation, the "to be created" flag shall be cleared and a folder synchronization triggered so that the new folder appears in the list with its server-assigned properties.
- **FR-22.** If server-side creation fails (quota exceeded, namespace error, permission denied), the error shall be recorded and displayed, and the "to be created" flag cleared.

### 7.6 Renaming a folder

- **FR-23.** The folder properties screen shall allow the user to edit the folder name for user-created folders only. System folder names shall not be editable.
- **FR-24.** The new name shall be validated: it must not be empty, and no other folder in the same account may already have the target name. A clear error shall be shown on validation failure.
- **FR-25.** The rename shall be queued locally (the target name stored as a "rename" pending value) and executed on the server during the next synchronization cycle.
- **FR-26.** The rename operation shall preserve the folder's subscription state: if the folder was subscribed, the old name shall be unsubscribed and the new name re-subscribed.
- **FR-27.** Upon successful server-side rename, the local folder record shall be updated to reflect the new name, and the "rename" pending value cleared.
- **FR-28.** A rename shall trigger a full account reload so that any child folders or dependent references are updated.

### 7.7 Deleting a folder

- **FR-29.** The folder context menu and the folder properties screen shall offer a delete action for user-created folders only. The action shall not be available for: (a) system folders, (b) read-only folders, (c) folders that have child folders.
- **FR-30.** Activating the delete action shall present a confirmation dialog that names the folder and warns the user that deletion is permanent.
- **FR-31.** If the folder has pending operations in the queue, deletion shall be refused with an error message indicating the number of pending operations.
- **FR-32.** Upon confirmation, the folder shall be marked "to be deleted" locally and the deletion executed on the server during the next synchronization cycle.
- **FR-33.** The server-side deletion shall first unsubscribe the folder, then delete it.
- **FR-34.** Upon successful server-side deletion, the folder shall be removed from the local database.
- **FR-35.** If the folder no longer exists on the server at the time of deletion, the operation shall succeed silently (idempotent).
- **FR-36.** If server-side deletion fails (quota error, permission denied), the error shall be recorded and displayed, and the "to be deleted" flag cleared so the folder remains in the local list.

### 7.8 Interaction with other folder properties

- **FR-37.** Subscription state shall be independent of: synchronization on/off, unified-inbox membership, notification on/off, navigation visibility, and hide/show state. Changing subscription shall not alter any of these properties, and vice versa.
- **FR-38.** Newly created folders shall have their initial sync, notification, and unified-inbox settings determined by the values the user chose in the creation form — not influenced by subscription state.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Subscription toggle, folder creation, rename, and delete shall appear to complete locally within one second; server propagation happens asynchronously during the next sync cycle.
- **NFR-2. Consistency.** After a folder operation completes on the server, any other client connecting to the same account shall see the result (created folder, new name, deletion, subscription change).
- **NFR-3. Offline.** Folder operations shall be queued locally when offline. They shall execute automatically when the account next connects. The user shall see the queued state (e.g., the folder exists locally with a pending flag) until sync completes.
- **NFR-4. Error resilience.** A failed server-side operation shall not leave the local database in an inconsistent state. Pending flags shall be cleared on failure, and the error reported.
- **NFR-5. IMAP-only.** All subscription, create, rename, and delete operations shall be limited to IMAP accounts. The UI shall not offer these actions for POP3 or local-only accounts (except that POP3/local accounts are unaffected by the subscribed-only filter).
- **NFR-6. Accessibility.** All controls — context menu items, property-screen fields, confirmation dialogs, filter toggles — shall be reachable via keyboard, with screen-reader labels, and shall obey the application-wide theme.
- **NFR-7. Safety.** Destructive operations (delete) shall always require explicit confirmation. Non-destructive operations (subscribe, create, rename) shall not.

## 10. Open Questions

- **OQ-1. Batch subscription operations.** The source application supports batch operations on child folders for synchronization, notification, and unified-inbox membership, but not for subscription. Should batch subscribe/unsubscribe for all children of a parent folder be supported? The source application does not offer this.
- **OQ-2. Subscription default for newly discovered folders.** When the server reports a new folder during sync, should the application respect the server's subscription list to set the initial local subscription state, or should it default to unsubscribed? The source application reads the server's subscription list and mirrors it.
- **OQ-3. Rename of folders with children.** The source application allows renaming a folder that has child folders (the delete restriction on children does not apply to rename). The IMAP RENAME command is supposed to rename child folders as well. Should the application explicitly verify that child folders were renamed, or trust the server?
- **OQ-4. Shared/public namespace folders.** Some IMAP servers expose shared or public namespace folders. Can these be subscribed/unsubscribed? The source application treats them like any other folder for subscription purposes, but create/rename/delete may fail depending on server permissions. Should the UI pre-emptively hide create/rename/delete for shared namespaces?
- **OQ-5. Folder quota.** Some servers enforce a maximum number of folders. The source application surfaces the server error when creation fails due to quota, but does not proactively check quota before offering the create action. Should a proactive check be added?
- **OQ-6. Separator discovery failure.** If the account's folder separator is unknown (null) at the time of sub-folder creation, the source application silently refuses to create the sub-folder. Should a user-facing error message be shown instead?
- **OQ-7. Subscription management setting name.** The global setting is called "subscriptions" internally. For a desktop application, should it be surfaced as "Manage IMAP subscriptions", "Show subscription controls", or something else? This is a UX naming decision.

## 11. Design Notes & Rationale

- **N-1. Subscription management is opt-in.** The subscription controls are hidden by default because most users never interact with IMAP subscriptions — their client simply shows all folders. Surfacing subscribe/unsubscribe by default would confuse users who do not know what IMAP subscriptions are. The opt-in setting in the synchronization preferences is a deliberate gate.
- **N-2. Newly created folders are auto-subscribed.** When a user creates a folder through the UI, the intent is clearly that the folder should be visible. Auto-subscribing after creation ensures that "subscribed only" filtering and other subscription-aware clients do not immediately hide the folder the user just created.
- **N-3. Rename preserves subscription.** The IMAP RENAME command does not define behavior for subscriptions. The source application explicitly unsubscribes the old name and re-subscribes the new name (if the folder was subscribed and selectable), because some servers do not automatically transfer subscription state on rename.
- **N-4. Delete unsubscribes first.** Deleting a folder without unsubscribing first can leave orphan entries in the server's subscription list. The source application defensively unsubscribes before deleting.
- **N-5. Delete blocked by children.** Allowing deletion of a folder with children would require either recursive deletion (dangerous) or orphaning the children (confusing). The source application takes the safe route: the user must delete children first, bottom-up.
- **N-6. Delete blocked by pending operations.** If operations (message moves, flag changes) are queued for a folder and the folder is deleted, those operations would fail in confusing ways. The source application requires the operation queue to drain before allowing deletion, surfacing a message that indicates how many operations remain.
- **N-7. Subscribed-only filter always shows Inbox.** Even if the Inbox is technically unsubscribed (unusual but possible), hiding it would render the application's primary mail view inaccessible. The filter unconditionally includes the Inbox.
- **N-8. Subscribed-only filter shows parents of visible children.** If a parent folder is unsubscribed but one of its children is subscribed (and therefore visible), the parent must also be shown to preserve the visual hierarchy. Without this rule, subscribed child folders would appear to float at the wrong level.
- **N-9. Create and delete use deferred execution.** Folder create and delete are not performed synchronously during the user interaction. Instead, a local flag is set ("to be created" or "to be deleted") and the actual server operation runs during the next synchronization cycle. This design allows offline operation and avoids blocking the UI on network latency.
- **N-10. System folders are protected.** System folders (Inbox, Sent, Drafts, Trash, Spam, Archive) cannot be renamed or deleted because they have server-side semantic meaning. Other clients, server-side rules, and the application's own logic depend on their existence and names.
