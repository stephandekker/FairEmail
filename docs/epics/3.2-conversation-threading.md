# Epic: Conversation Threading

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.2** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"RFC-compliant threading using `In-Reply-To` / `References`, plus Gmail-style and subject-fallback modes; configurable per-account."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email conversations naturally form trees: a message is sent, one or more replies follow, and so on. Threading is the operation of reconstructing these trees from the metadata that messages carry — primarily the `Message-ID`, `In-Reply-To`, and `References` headers defined by RFC 5322 — so that an email client can present related messages together rather than as an unordered stream.

In practice, RFC-header-based threading alone does not cover every case. Some providers embed proprietary thread identifiers (e.g. Gmail's `X-GM-THRID`). Some messages arrive with missing or corrupt headers. Some users expect messages with the same subject from the same sender to group together even when no reply chain exists. A robust threading implementation must support multiple strategies, let the user choose which ones to enable, and constrain each strategy with sensible time bounds so that unrelated messages are not spuriously merged.

The Conversation Threading feature exists to:

1. Group related messages into conversations so that the user sees a coherent thread rather than scattered individual messages.
2. Offer multiple threading modes — standard RFC headers, provider-specific thread identifiers, and a subject-based fallback — so that the user can match threading behavior to the mail sources they use.
3. Consolidate thread membership dynamically as new messages arrive and reveal previously missing links in a reply chain.
4. Present threaded conversations in a compact, navigable form in every message-list view, including the Unified Inbox.

## 2. Goals

- **G1.** Group messages into conversations using RFC 5322 `Message-ID`, `In-Reply-To`, and `References` headers as the primary threading signal.
- **G2.** Support provider-specific thread identifiers (Gmail thread index) as an optional, higher-priority grouping signal when the user enables them.
- **G3.** Provide an optional subject-based fallback that groups messages by sender and subject within a bounded time window, for use with mail sources that produce poor or missing threading headers.
- **G4.** Allow the user to enable or disable conversation threading globally, so that users who prefer a flat message list can have one.
- **G5.** Constrain all threading strategies with a configurable time range to prevent spurious grouping of unrelated messages that happen to share identifiers across long spans.
- **G6.** Dynamically consolidate threads as new messages arrive — when an incoming message reveals a missing link, update all affected messages to belong to the same thread.
- **G7.** Display threaded conversations with clear visual indicators of message count, unread count, and per-message expand/collapse, so that the user can triage and navigate efficiently.

## 3. Non-Goals

- **NG1.** This epic does **not** define how messages are fetched, stored, or synchronized; it only constrains how threading consumes whatever the synchronization subsystem produces.
- **NG2.** This epic does **not** prescribe the internal representation of thread identifiers — only their user-visible behavior and the contracts between threading strategies.
- **NG3.** Per-account selection of threading mode (e.g. "use Gmail thread index only for this account") is **not** a goal. Threading mode selection is global. (The high-level feature list mentions "configurable per-account"; in practice the source application applies all threading settings globally, with provider-specific strategies activating only when the message originates from the relevant provider.)
- **NG4.** Full implementation of Outlook Thread-Index threading is **not** a goal for initial delivery. The source application has experimental support for this strategy but does not expose it to end users outside of debug builds. It is documented here for completeness and deferred to a future iteration.
- **NG5.** Smart or AI-based threading (e.g. grouping by semantic similarity) is out of scope.
- **NG6.** Cross-account threading is **not** a goal of this epic. Threads are scoped to a single account. (Cross-account grouping in the Unified Inbox is addressed by the Unified Inbox epic's conversation grouping requirement.)

## 4. Glossary

| Term | Definition |
|---|---|
| **Message-ID** | A globally unique identifier assigned to a message by the originating mail system, carried in the `Message-ID` header (RFC 5322 §3.6.4). |
| **In-Reply-To** | A header containing the Message-ID(s) of the message(s) to which this message is a direct reply (RFC 5322 §3.6.4). |
| **References** | A header containing an ordered list of Message-IDs tracing the ancestry of the conversation, from the root message to the immediate parent (RFC 5322 §3.6.4). |
| **Thread / Conversation** | A set of messages that the application has determined to be related, based on one or more threading strategies. Displayed as a single grouped entry in the message list. |
| **Thread identifier** | An opaque, application-internal identifier assigned to every message; messages sharing the same thread identifier belong to the same conversation. |
| **Threading strategy** | A method for computing thread membership. The application supports: RFC-header-based, Gmail thread index, common-reference, and subject-based strategies. |
| **Thread range** | A configurable time window (expressed as a number of days) within which the application will search for related messages when computing thread membership. Messages outside this window are not considered, even if their identifiers match. |
| **Subject-based threading** | A fallback strategy that groups messages by matching sender address and subject line within a bounded time window (default: 48 hours). |
| **Gmail thread index** | A proprietary thread identifier (`X-GM-THRID`) provided by Gmail's IMAP extensions, representing Gmail's server-side conversation grouping. |
| **Outlook Thread-Index** | A proprietary header (`Thread-Index`) used by Microsoft Exchange/Outlook to group messages. (Experimental; not exposed to end users.) |
| **Common-reference threading** | A strategy that uses the first entry in a message's `References` header (typically the root message of the conversation) as the thread identifier, linking messages that share a common ancestor even when intermediate messages are missing. |
| **Thread consolidation** | The process of merging two previously separate threads into one when a newly arrived message reveals a link between them. |
| **Forward-as-new-thread** | A user preference controlling whether forwarded messages start a new conversation or remain linked to the original thread via the `X-Forwarded-Message-Id` header. |

## 5. Personas

- **P1. The standard user.** Uses one or two accounts with well-behaved mail servers (Gmail, Outlook, etc.) that produce correct RFC headers. Expects threading to "just work" with no configuration. Default settings should produce correct conversations.
- **P2. The mailing-list power user.** Subscribes to high-volume mailing lists and discussion forums where long threads span weeks. Needs the time-range constraint to be generous enough to keep threads together, but not so wide that unrelated messages merge.
- **P3. The multi-provider user.** Uses accounts from providers with varying header quality — some produce correct `References` chains, others produce incomplete ones, and some use proprietary identifiers. Benefits from enabling provider-specific strategies alongside the RFC-based default.
- **P4. The flat-list user.** Prefers to see every message as an individual item, not grouped. Wants to disable threading entirely and have the message list reflect chronological order without conversation grouping.
- **P5. The subject-matcher.** Receives mail from systems (ticketing platforms, legacy listservs, transactional senders) that do not set `In-Reply-To` or `References` at all. Relies on subject-based fallback to get some semblance of threading.

## 6. User Stories

### 6.1 Default threading behavior

> **US-1.** As a standard user, when I view my Inbox, I want messages that are part of the same email conversation to appear as a single grouped entry, so that I can see each conversation as a unit rather than as scattered individual messages.

> **US-2.** As a standard user, I want threading to work out of the box without any configuration, using the standard `Message-ID`, `In-Reply-To`, and `References` headers, so that replies and their parents are grouped automatically.

> **US-3.** As any user, when a new message arrives that is a reply to an existing conversation, I want it to be added to that conversation immediately, so that I see the conversation grow in real time.

### 6.2 Enabling and disabling threading

> **US-4.** As a flat-list user, I want to disable conversation threading globally through a display setting, so that every message appears as an individual row in the message list.

> **US-5.** As any user, when I disable threading, I want the change to take effect immediately across all views (Inbox, Unified Inbox, folder views, search results), so that I do not have to restart or refresh manually.

> **US-6.** As any user, I want my threading preference to persist across application restarts.

### 6.3 Gmail thread index

> **US-7.** As a multi-provider user with a Gmail account, I want the option to use Gmail's native thread grouping instead of (or in addition to) RFC-header-based threading, so that my conversations match what I see in the Gmail web interface.

> **US-8.** As any user, I want the Gmail thread index option to be off by default and clearly labeled as applying only to newly received messages, so that I understand it will not retroactively regroup older messages and may alter existing groupings.

### 6.4 Subject-based fallback threading

> **US-9.** As a subject-matcher, I want an optional mode that groups messages by sender and subject line within a recent time window, so that messages from systems with poor threading headers are still grouped into conversations.

> **US-10.** As any user, I want subject-based threading to be off by default, so that it does not spuriously merge unrelated messages that happen to share a subject line.

> **US-11.** As any user, I want subject-based threading to be bounded to a short, fixed time window (e.g. 48 hours), so that old messages with the same subject are not pulled into new conversations.

### 6.5 Thread time range

> **US-12.** As a mailing-list power user, I want to configure the time window within which the application searches for related messages when building threads, so that I can widen it for long-lived discussions or narrow it to avoid spurious grouping.

> **US-13.** As any user, I want a sensible default time range (e.g. 128 days) that works for typical personal and business email without configuration.

### 6.6 Common-reference threading

> **US-14.** As a power user, I want an option for the application to use the root message of a References chain as a shared thread anchor, so that messages sharing a common ancestor are grouped together even when intermediate messages in the chain are missing from my mailbox.

### 6.7 Forwarded messages and new threads

> **US-15.** As any user, I want the option to have forwarded messages start a new conversation thread rather than being linked to the original thread, so that forwarded discussions do not clutter the original conversation.

> **US-16.** As any user, when the "forward as new thread" option is disabled, I want forwarded messages to remain linked to the original conversation, so that I can trace the full history.

### 6.8 Thread consolidation

> **US-17.** As any user, when a message arrives that bridges two previously separate threads (e.g. a reply that references messages from both), I want the application to merge those threads into one, so that the full conversation is visible in a single view.

> **US-18.** As any user, I want thread consolidation to happen transparently — I should see the merged conversation without having to take any action.

### 6.9 Conversation view

> **US-19.** As any user, when I open a conversation, I want to see all messages in the thread displayed in chronological order, with each message individually expandable and collapsible, so that I can navigate long threads efficiently.

> **US-20.** As any user, when I open a conversation, I want one message to be auto-expanded according to sensible rules: the single message if there is only one; the single unread message if there is exactly one; the single starred message if there are no unread messages; otherwise, the most recent message.

> **US-21.** As any user, I want the conversation view to show trashed messages and drafts within the thread context, so that I can find accidentally deleted messages and resume drafts in the context where they belong.

> **US-22.** As any user, I want the ability to permanently delete a message from a conversation, removing it from the thread entirely.

### 6.10 Thread-level display indicators

> **US-23.** As any user, in the message list I want each conversation entry to show the total message count and the unread message count, so that I can see at a glance how large and how current a thread is.

> **US-24.** As any user, I want an option to show the number of unread messages within each conversation entry, so that I can prioritize threads with unread content.

> **US-25.** As any user, I want an option to visually indent sent and received messages differently within the conversation view (e.g. left-align received, right-align sent), so that the flow of the conversation is easier to follow.

### 6.11 Thread-level actions

> **US-26.** As any user, when I move a conversation (e.g. to archive, trash, or another folder), I want the option to move all messages in the thread, not just the message I selected, so that the entire conversation is handled as a unit.

> **US-27.** As any user, I want a separate option to include or exclude sent messages when moving a conversation, so that my Sent folder is not disrupted by triage actions on received mail.

### 6.12 Strategy mutual exclusivity

> **US-28.** As any user, I want subject-based threading to be automatically disabled when I enable Gmail thread index (or vice versa), so that conflicting strategies do not produce confusing results.

## 7. Functional Requirements

### 7.1 Threading strategies

- **FR-1.** The application shall support the following threading strategies, listed in descending priority:
  1. **Gmail thread index**: use the provider-supplied thread identifier from Gmail's IMAP extensions.
  2. **RFC-header-based**: compute thread membership from `Message-ID`, `In-Reply-To`, and `References` headers per RFC 5322.
  3. **Common-reference**: use the first (root) entry in the `References` header as a shared thread anchor.
  4. **Subject-based fallback**: match messages by sender address and subject line within a bounded time window.

- **FR-2.** Each strategy shall be independently enableable or disableable via a user-facing setting, except for the RFC-header-based strategy, which is always active when threading is enabled.

- **FR-3.** When multiple strategies are enabled, they shall be evaluated in priority order. The first strategy that produces a thread identifier shall be used; lower-priority strategies shall not override it for that message.

- **FR-4.** Subject-based threading shall be mutually exclusive with Gmail thread index in the user interface: enabling one shall disable the other.

### 7.2 RFC-header-based threading

- **FR-5.** The application shall extract the `Message-ID`, `In-Reply-To`, and `References` headers from every incoming message.

- **FR-6.** The application shall process `References` in reverse order (most recent parent first) and `In-Reply-To` as the highest-priority reference, consistent with RFC 5322 §3.6.4 semantics.

- **FR-7.** When building the reference list for a message, the application shall also consider the `X-Forwarded-Message-Id` header (if present and if the "forward as new thread" preference is disabled), so that forwarded messages can be linked to their original thread.

- **FR-8.** The application shall impose a maximum limit on the number of references processed per message (at least 450) to avoid exceeding query-size constraints.

### 7.3 Gmail thread index

- **FR-9.** When the Gmail thread index strategy is enabled and a message originates from Gmail, the application shall extract the provider-supplied thread identifier and use it as the thread identifier for that message.

- **FR-10.** The Gmail thread index strategy shall be disabled by default.

- **FR-11.** The user interface shall inform the user that enabling Gmail thread index applies only to newly received messages and may alter existing conversation groupings.

### 7.4 Outlook Thread-Index (experimental / deferred)

- **FR-12.** The application may support extracting thread identifiers from the Outlook `Thread-Index` header. This strategy is experimental and shall not be exposed in the user interface outside of debug/developer builds until it is validated.

### 7.5 Subject-based fallback

- **FR-13.** When enabled, subject-based threading shall group messages that share the same sender address and the same subject line, received within a fixed time window (default: 48 hours).

- **FR-14.** Subject-based threading shall be disabled by default.

- **FR-15.** Subject-based threading shall not apply to delivery status notifications or other automated report messages.

- **FR-16.** The time window for subject-based threading shall be fixed (not user-configurable) to limit the risk of spurious grouping.

### 7.6 Common-reference threading

- **FR-17.** When enabled, common-reference threading shall use the first entry in the `References` header (typically the root message of the conversation) as the thread identifier, linking all messages that share the same root ancestor.

- **FR-18.** Common-reference threading shall be enabled by default.

- **FR-19.** Common-reference threading shall not use the message's own `Message-ID` as the common reference (to avoid self-linking).

### 7.7 Thread time range

- **FR-20.** The application shall constrain all threading lookups (except subject-based, which has its own window) to a configurable time range centered on the received date of the message being threaded.

- **FR-21.** The default time range shall be 128 days (approximately four months).

- **FR-22.** The user shall be able to adjust the time range through an advanced/miscellaneous setting. The range shall be expressed as a power-of-two number of days, adjustable from 1 day to a very large upper bound.

- **FR-23.** Messages received outside the configured time range shall not be considered for thread matching, even if their identifiers match.

### 7.8 Thread consolidation

- **FR-24.** When a newly arrived message references (via `In-Reply-To`, `References`, or `X-Forwarded-Message-Id`) messages that belong to different existing threads, the application shall consolidate those threads into a single thread.

- **FR-25.** Consolidation shall update all affected messages — both those sent before the arriving message (that reference it) and those sent after (that it references) — to share the same thread identifier.

- **FR-26.** Consolidation shall respect the thread time range: only messages within the configured time window shall be candidates for consolidation.

### 7.9 Forwarded messages

- **FR-27.** The application shall provide a user-facing send preference ("forward as new thread") that controls whether forwarded messages start a new conversation or remain linked to the original.

- **FR-28.** When "forward as new thread" is enabled, the application shall not include the `X-Forwarded-Message-Id` header in outgoing forwarded messages and shall not use that header for threading on incoming messages.

- **FR-29.** When "forward as new thread" is disabled (the default for new installations), forwarded messages shall be threaded with the original conversation.

### 7.10 Global threading toggle

- **FR-30.** The application shall provide a global display setting to enable or disable conversation threading.

- **FR-31.** When threading is enabled, the message list shall group messages by their thread identifier, showing one entry per conversation.

- **FR-32.** When threading is disabled, the message list shall show every message as an individual entry, ignoring thread identifiers.

- **FR-33.** Threading shall be enabled by default.

- **FR-34.** The threading toggle shall take effect immediately across all message-list views without requiring a restart.

### 7.11 Conversation view

- **FR-35.** When the user opens a conversation, the application shall display all messages in the thread in chronological order, with each message individually expandable and collapsible.

- **FR-36.** Messages shall be ordered primarily by received date, with a secondary sort by folder role (Inbox before Outbox before Drafts before Sent before Trash before Spam before other system folders before user folders before Archive).

- **FR-37.** Trashed messages and drafts shall be included in the conversation view, so that the user can find accidentally deleted messages and resume drafts in context.

- **FR-38.** The user shall be able to permanently delete a message from a conversation, removing it from the thread.

- **FR-39.** When navigating to a conversation, exactly one message shall be auto-expanded according to these rules (evaluated in order):
  1. If there is exactly one message in the conversation, expand it.
  2. If there is exactly one unread message, expand it.
  3. If there are no unread messages and exactly one starred message, expand it.
  4. Otherwise, expand the most recent message.

- **FR-40.** Messages shall be automatically marked as read upon expansion, unless the user has disabled auto-mark-read in account settings.

### 7.12 Thread-level display

- **FR-41.** Each conversation entry in the message list shall display the total number of messages in the thread.

- **FR-42.** The application shall provide an optional display setting to show the number of unread messages within each conversation entry. This setting shall be disabled by default.

- **FR-43.** The application shall provide an optional display setting to visually indent sent and received messages differently within the conversation view (e.g. left-indent received, right-indent sent). This setting shall be disabled by default and shall apply only when using the card-style message display.

### 7.13 Thread-level move operations

- **FR-44.** The application shall provide a behavior setting ("move all messages when moving conversations") that, when enabled, causes a move/archive/trash action on a conversation to apply to all messages in the thread, not just the currently viewed or selected message.

- **FR-45.** The application shall provide a separate behavior setting ("also move sent messages when moving a conversation") that, when enabled, includes sent messages in thread-level move operations. This setting shall be independently toggleable but shall be implicitly enabled when "move all messages" is enabled.

- **FR-46.** Both thread-level move settings shall be disabled by default.

### 7.14 Outgoing message headers

- **FR-47.** When composing a reply, the application shall set the `In-Reply-To` header to the `Message-ID` of the message being replied to, and the `References` header to the replied-to message's `References` value followed by its `Message-ID`, per RFC 5322.

- **FR-48.** The application shall limit the length of the `References` header on outgoing messages to prevent excessively large headers on deeply nested threads.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Thread computation for an incoming message shall complete within a time that does not noticeably delay message display under normal load (hundreds of threads, thousands of messages).

- **NFR-2. Scale.** Threading shall remain correct and performant with at least 20 accounts and tens of thousands of messages per account.

- **NFR-3. Consistency.** Thread membership shall be deterministic: given the same set of messages and the same settings, the same threads shall be produced regardless of the order in which messages were received.

- **NFR-4. Offline.** Thread computation shall work entirely on locally stored data. No network round-trips shall be required to determine thread membership.

- **NFR-5. Idempotency.** Re-processing a message that has already been threaded (e.g. after a re-sync) shall produce the same thread assignment, not create a duplicate thread.

- **NFR-6. Privacy.** Thread identifiers and threading metadata shall be local to the application. No threading-related data shall be transmitted to any server beyond the standard outgoing `In-Reply-To` and `References` headers on messages the user sends.

- **NFR-7. Graceful degradation.** When threading headers are missing or malformed, the application shall fall back to lower-priority strategies or assign the message its own thread. It shall never crash or produce an error visible to the user.

- **NFR-8. Accessibility.** Conversation entries, expand/collapse controls, message counts, and all threading-related display elements shall be reachable via keyboard and shall carry appropriate screen-reader labels.

## 10. Open Questions

- **OQ-1. Outlook Thread-Index readiness.** The source application contains experimental support for threading via the Outlook `Thread-Index` header, but it is gated behind a debug flag and not exposed to users. Should this be promoted to a user-facing setting, or should it remain experimental until further validation with real-world Outlook mail?

- **OQ-2. Per-account threading mode.** The high-level feature list describes threading as "configurable per-account," but the source application applies all threading settings globally. The Gmail thread index strategy activates per-message based on the message's provider origin, which creates a de facto per-account effect for Gmail accounts. Is true per-account threading mode selection (e.g. "use subject-based for Account A, RFC-only for Account B") a desired future capability, or is the current global-with-provider-detection model sufficient?

- **OQ-3. Thread range and subject-based threading interaction.** The configurable thread time range governs RFC-header-based and common-reference threading, while subject-based threading has its own fixed 48-hour window. Should these be unified into a single time-range control, or does the deliberately shorter subject window serve as a necessary safety constraint?

- **OQ-4. Retroactive re-threading.** Currently, changing a threading setting (e.g. enabling Gmail thread index) affects only newly received messages. Should the application offer a "re-thread all" action that reprocesses existing messages under the new settings? This could be expensive for large mailboxes.

- **OQ-5. Cross-account threading.** Threads are scoped to a single account. The Unified Inbox groups threads from different accounts that share the same conversation (per the Unified Inbox epic's AC-16). The boundary between per-account thread computation and cross-account display grouping should be explicitly defined during design.

- **OQ-6. FAQ vs. implementation divergence on subject threading.** The FAQ (entry 121) states that FairEmail "does not group on other criteria, like the subject," but the source application does support optional subject-based threading. The FAQ appears to describe the default behavior rather than the full capability. This should be clarified in user-facing documentation.

- **OQ-7. Common-reference default.** Common-reference threading is enabled by default in non-app-store builds but disabled in app-store builds. Should the desktop application follow the non-app-store default (enabled) or the app-store default (disabled)?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Priority cascade, not composition.** The threading strategies form a priority cascade: Gmail thread index (if enabled and applicable) takes precedence over RFC-header-based threading, which takes precedence over common-reference, which takes precedence over subject-based. A message is assigned to exactly one thread by exactly one strategy. This avoids the complexity of merging conflicting signals from multiple strategies.

- **N-2. Time-bounded lookups.** All threading lookups (except subject-based, which has its own shorter window) are bounded by a configurable time range (default 128 days). This is a deliberate tradeoff: it prevents ancient messages with reused or colliding `Message-ID` values from being pulled into new threads, at the cost of occasionally failing to thread very old reply chains. The exponential scale (2^n days) gives the user a wide adjustment range without requiring fine-grained numeric input.

- **N-3. Thread consolidation is eager.** When a new message arrives that bridges two existing threads, the application immediately merges them. This is correct behavior: the two threads were always part of the same conversation; they were separate only because the linking message had not yet arrived. The merge is irreversible within the current threading pass — splitting a thread requires either deleting the bridging message or re-threading from scratch.

- **N-4. Subject-based threading is intentionally conservative.** The 48-hour window and the requirement that both sender and subject must match are deliberate constraints. Subject-only matching (without sender) would merge unrelated messages from mailing lists where many participants use the same subject. A longer time window would merge messages from automated systems that reuse subjects (e.g. daily reports). The combination of sender + subject + short window is the least-bad heuristic.

- **N-5. Trashed messages and drafts in conversations.** The source application deliberately shows trashed messages and drafts within the conversation view. This is counterintuitive but practical: individual messages are rarely trashed intentionally (it is usually accidental), and showing them in context makes recovery easy. Similarly, drafts shown in conversation context let the user see the messages they are responding to while composing. Permanent deletion is the mechanism for truly removing a message from a conversation.

- **N-6. Forward-as-new-thread default.** Newer installations default to "forward as new thread" enabled, meaning forwarded messages start their own conversation. This matches user expectations: forwarding a message to a different person is conceptually a new discussion, not a continuation of the original. Older installations that were migrated retain the previous default (disabled) for continuity.

- **N-7. Global, not per-account.** Although the high-level feature list says "configurable per-account," the source application's threading settings are global. The practical reason is that threading is computed at message-insertion time and stored as a persistent identifier; per-account settings would require either storing multiple thread identifiers per message or recomputing threads when switching views. The Gmail thread index achieves a per-account effect implicitly: it only activates for messages from Gmail, so enabling it changes behavior for Gmail accounts without affecting others.
