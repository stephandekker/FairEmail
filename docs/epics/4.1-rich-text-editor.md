# Epic: Rich Text Editor

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **4.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Rich text editor with formatting toolbar: bold, italic, underline, strikethrough, sub/super-script, font, size, color, highlight, bullet/numbered lists, indentation, alignment, headings, links, horizontal rule, clear formatting."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email is inherently a rich-text medium. While some users prefer plain text, the majority expect to compose messages with visual formatting — bold headings, colored text, bulleted lists, inline links — that conveys emphasis, structure, and brand in the same way a word processor does.

A desktop email client's compose window must therefore provide an integrated rich text editing surface with a discoverable formatting toolbar, enabling users to apply character-level styling (bold, italic, color, size, font), paragraph-level formatting (lists, indentation, alignment, block quotes), and structural elements (headings, horizontal rules, links) — all while producing clean, standards-compliant HTML that renders faithfully in any recipient's mail client.

At the same time, the editor must support a graceful fallback to plain-text-only composition for users who prefer it, and must handle the messy realities of pasting content from external sources, round-tripping through HTML, and interoperating with reply/forward quoting.

## 2. Goals

- **G1.** Provide a toolbar-driven rich text editor as the default compose experience, enabling users to format messages without knowledge of HTML.
- **G2.** Support the full range of common email formatting: character styles, paragraph styles, structural elements, and embedded links.
- **G3.** Produce clean, portable HTML output that renders consistently across major email clients and webmail providers.
- **G4.** Allow seamless switching between rich text and plain-text-only composition modes within a single message.
- **G5.** Handle paste operations intelligently — preserving formatting from rich sources, stripping it when requested, and never introducing unsafe or broken markup.
- **G6.** Provide undo/redo capability so that formatting experiments are risk-free.
- **G7.** Support power-user workflows: keyboard shortcuts, keep-selection mode, quick code formatting, and password-protected content embedding.

## 3. Non-Goals

- **NG1.** This epic does **not** define how the composed HTML is transmitted (MIME structure, multipart boundaries, character encoding negotiation) — only that the editor produces well-formed HTML suitable for transmission.
- **NG2.** This epic does **not** cover inline image insertion, attachment management, or image editing — those are separate compose-surface features that happen to coexist with the text editor.
- **NG3.** Full page-layout capabilities (tables, columns, CSS grid) are **not** a goal. The editor targets email-safe formatting, not desktop publishing.
- **NG4.** Markdown-as-source editing mode is **not** in scope here; it is a separate compose option that may reuse the same underlying conversion pipeline but has its own UX.
- **NG5.** Spell-checking and grammar-checking integrations are out of scope for this epic, though the editor surface must not interfere with external language-tool services.
- **NG6.** Signature editing is **not** in scope. Although the signature editor shares some formatting capabilities, it has its own entry point and constraints.

## 4. Glossary

| Term | Definition |
|---|---|
| **Rich text mode** | The default compose mode in which the user edits visually formatted text via a toolbar; the application stores and transmits the message as HTML. |
| **Plain text mode** | An alternative compose mode in which no formatting toolbar is shown, the editing surface uses a monospace font, and the message is transmitted as plain text only (no HTML part). |
| **Formatting toolbar** | A horizontal bar of icon buttons displayed above or below the compose area, offering one-tap access to formatting actions. |
| **Character style** | A formatting property that applies to a contiguous run of characters within a paragraph: bold, italic, underline, strikethrough, subscript, superscript, font family, font size, text color, background/highlight color. |
| **Paragraph style** | A formatting property that applies to an entire paragraph (block of text between line breaks): alignment, list membership, indentation level, block quote, heading level. |
| **Structural element** | A non-text visual element inserted into the message flow: horizontal rule (line separator), hyperlink. |
| **Keep-selection mode** | A user preference that controls whether the text cursor remains at its original position after a formatting action, rather than moving to the end of the affected range. |
| **Clear formatting** | An action that strips character and paragraph styles from the selected text (or the entire message), restoring it to default appearance. |
| **Paste as plain text** | A paste variant that inserts clipboard content without any formatting, regardless of source richness. |
| **Paste as quote** | A paste variant that wraps clipboard content in a block quote before insertion. |

## 5. Personas

- **P1. The casual composer.** Writes short replies and forwards; occasionally bolds a word or adds a bulleted list. Expects formatting to "just work" like a word processor.
- **P2. The visual communicator.** Composes newsletters, announcements, or styled correspondence with headings, colors, fonts, and horizontal dividers. Uses the toolbar extensively.
- **P3. The plain-text purist.** Prefers plain text for philosophical or compatibility reasons. Wants to opt out of rich text entirely and compose in a monospace font.
- **P4. The power user.** Uses keyboard shortcuts, undo/redo, keep-selection mode, and quick-code formatting. Values precise control over formatting scope and clipboard behavior.
- **P5. The privacy-conscious sender.** Occasionally needs to embed password-protected content within an otherwise normal message.

## 6. User Stories

### 6.1 Basic character formatting

> **US-1.** As a casual composer, I want to select text and tap a Bold button to make it bold, so that I can add emphasis without knowing HTML.

> **US-2.** As a casual composer, I want to apply italic, underline, and strikethrough to selected text via toolbar buttons, so that I have the standard set of character emphasis tools.

> **US-3.** As a visual communicator, I want to apply subscript or superscript to selected text, so that I can write chemical formulas, footnote markers, or mathematical notation.

> **US-4.** As any user, when I tap a character-style button with no text selected, I want the formatting to apply to the word under the cursor (auto-expanding to word boundaries), so that I do not have to manually select a single word before formatting it.

> **US-5.** As any user, when I apply a character style that is already active on the selection, I want the style to be removed (toggle behavior), so that I can undo formatting with the same button that applied it.

### 6.2 Font, size, and color

> **US-6.** As a visual communicator, I want to choose a font family for the selected text from a curated list of email-safe fonts, so that I can give my messages a distinct typographic voice.

> **US-7.** As a visual communicator, I want to choose a text size from a set of predefined sizes (extra-small, small, medium/default, large, extra-large), so that I can create visual hierarchy without guessing pixel values.

> **US-8.** As a visual communicator, I want to pick a foreground text color from a color chooser, so that I can color-code or emphasize specific words.

> **US-9.** As a visual communicator, I want to pick a background highlight color for selected text, so that I can simulate a highlighter pen effect.

> **US-10.** As any user, I want a "mark" action that applies a standard yellow highlight with black text, so that I have a one-tap highlighter without opening a color picker.

> **US-11.** As any user, I want a reset/remove option within the color picker to remove a previously applied color, returning text to the default.

### 6.3 Paragraph formatting

> **US-12.** As a visual communicator, I want to set paragraph alignment to left, center, or right, so that I can lay out headings, signatures, and callouts.

> **US-13.** As any user, I want to convert selected paragraphs into a bullet (unordered) list, so that I can present items as a scannable list.

> **US-14.** As any user, I want to convert selected paragraphs into a numbered (ordered) list, with support for multiple numbering styles (decimal, lower-alpha, upper-alpha, lower-roman, upper-roman), so that I can create sequenced instructions or outlines.

> **US-15.** As any user, when I press Enter at the end of a list item, I want a new list item to be created automatically, and when I press Enter on an empty list item, I want the list to end and normal paragraph mode to resume, so that list editing feels natural.

> **US-16.** As a power user, I want to increase or decrease the nesting level of a list item (indent/outdent), so that I can create multi-level outlines.

> **US-17.** As any user, I want to indent or outdent paragraphs that are not part of a list, so that I can visually offset quoted or secondary content.

> **US-18.** As any user, I want to wrap selected paragraphs in a block quote (with a visible left-border style), so that I can visually distinguish quoted material from my own writing.

> **US-19.** As a visual communicator, when I press Enter inside a block quote, I want the quote to split so that I can interleave my own commentary between quoted sections.

### 6.4 Structural elements

> **US-20.** As a visual communicator, I want to insert a horizontal rule (visual line separator) at the cursor position, so that I can divide my message into distinct sections.

> **US-21.** As any user, I want to insert or edit a hyperlink on selected text, specifying a URL and optionally a display title, so that I can link to resources without exposing raw URLs.

> **US-22.** As any user, when I select text that is already a hyperlink and invoke the link action, I want to edit the existing URL rather than creating a nested link, so that link editing is non-destructive.

### 6.5 Headings

> **US-23.** As a visual communicator, I want to apply heading styles (at least two distinct levels) to a paragraph, so that I can create document-like structure within a longer message.

### 6.6 Clear formatting

> **US-24.** As any user, I want a "clear formatting" action that removes all character and paragraph styles from the current selection, returning it to default appearance, so that I can start over on a badly-formatted region.

> **US-25.** As any user, I want a "clear all formatting" action that strips all styles from the entire message body, so that I can reset the whole message at once.

> **US-26.** As any user, I expect clear-formatting to preserve hyperlinks and embedded images, so that structural content is not accidentally destroyed.

### 6.7 Plain text mode

> **US-27.** As a plain-text purist, I want to toggle a "plain text only" mode that hides the formatting toolbar and switches the editing surface to a monospace font, so that I compose in the mode I prefer without distraction.

> **US-28.** As any user, I want a preference that makes plain-text mode the default for all new compositions, so that I do not have to toggle it every time.

> **US-29.** As any user, when replying to a message that was sent as plain text only, I want the editor to default to plain-text mode for my reply, so that my reply matches the original's format.

### 6.8 Clipboard and paste behavior

> **US-30.** As any user, when I paste content from an external application, I want the editor to preserve the source formatting (bold, lists, links, etc.) by default, so that copy-paste from a web page or document works as expected.

> **US-31.** As any user, I want a "paste as plain text" action (accessible from a context menu or keyboard shortcut) that inserts clipboard content with all formatting stripped, so that I can paste clean text from messy sources.

> **US-32.** As any user, I want a "paste as quote" action that wraps the pasted content in a block quote, so that I can quote external material with one action.

> **US-33.** As any user, I want pasted content to be sanitized so that potentially dangerous or unsupported markup (scripts, iframes, external tracking pixels) is never introduced into my compose surface.

### 6.9 Undo and redo

> **US-34.** As a power user, I want undo and redo actions available via keyboard shortcut and/or context menu, so that I can reverse accidental formatting or text changes.

> **US-35.** As any user, I want undo/redo to be optionally enabled via a preference, so that users who do not need it are not confused by unexpected behavior.

### 6.10 Selection behavior and keep-selection mode

> **US-36.** As a power user, I want a "keep selection" preference that, when enabled, preserves my text selection after applying a formatting action, so that I can apply multiple styles in rapid succession without re-selecting.

> **US-37.** As any user, when "keep selection" is disabled (default), I want the cursor to move to the end of the formatted range after applying a style, matching standard word-processor behavior.

### 6.11 Quick-code formatting

> **US-38.** As a power user, I want a one-tap "code" formatting action that applies a small monospace font to the selected text, so that I can mark inline code without manually choosing font and size separately.

### 6.12 Password-protected content

> **US-39.** As a privacy-conscious sender, I want to select text and encrypt it with a password, replacing the visible text with a decryption link, so that only the intended recipient (who knows the password) can read it.

> **US-40.** As a privacy-conscious sender, I want to be warned if the selected content exceeds the maximum encryptable size (approximately 1,500 characters including formatting), so that I do not accidentally truncate my protected content.

### 6.13 Convenience text actions

> **US-41.** As any user, when I have text selected, I want quick actions to wrap the selection in brackets or quotation marks (toggling: if already wrapped, unwrap), so that I can add common punctuation without precise cursor placement.

## 7. Functional Requirements

### 7.1 Editor surface

- **FR-1.** The compose window shall provide a rich text editing surface as the default mode for new compositions, replies, and forwards.
- **FR-2.** The editing surface shall render formatting in real time (WYSIWYG): the user sees bold text as bold, colored text in color, lists with bullets/numbers, etc.
- **FR-3.** The editing surface shall support free-form text entry, selection (mouse/keyboard/touch), and cursor navigation.

### 7.2 Formatting toolbar

- **FR-4.** A formatting toolbar shall be displayed when the editor is in rich text mode, providing one-tap access to all supported formatting actions.
- **FR-5.** The toolbar shall be horizontally scrollable to accommodate all buttons without truncation.
- **FR-6.** The toolbar shall be hideable/showable via a user action (e.g. a toggle in the compose menu).
- **FR-7.** The toolbar may be optionally fixed in position or scrollable with the message content, configurable by the user.

### 7.3 Character styles

- **FR-8.** The editor shall support the following character styles, each toggleable on the current selection: bold, italic, underline, strikethrough, subscript, superscript.
- **FR-9.** Character styles shall be toggleable: applying a style to text that already has it shall remove that style.
- **FR-10.** When no text is selected, applying a character style shall auto-expand the affected range to the word boundaries surrounding the cursor.
- **FR-11.** Multiple character styles may be combined on the same text run (e.g. bold + italic + underline simultaneously).

### 7.4 Font family

- **FR-12.** The editor shall offer a font family selector presenting a curated list of fonts known to render reliably in email.
- **FR-13.** The font list shall include at least: a default proportional font, a monospace font, a serif font, a sans-serif font, and optionally bundled specialty fonts (e.g. a dyslexia-friendly font).
- **FR-14.** The user shall be able to configure (via preferences) whether bundled fonts and narrow font variants are included in the font list.

### 7.5 Font size

- **FR-15.** The editor shall offer at least five distinct font sizes (e.g. extra-small, small, medium/default, large, extra-large) selectable from a menu or picker.
- **FR-16.** Sizes shall be expressed as relative proportions, not absolute pixel values, so that they scale appropriately with the recipient's display settings.

### 7.6 Text color and highlight

- **FR-17.** The editor shall provide a foreground color picker allowing the user to set the text color of the selection to any color.
- **FR-18.** The editor shall provide a background color picker allowing the user to set a highlight/background color on the selection.
- **FR-19.** Both color pickers shall offer a visual color wheel or palette, a text input for hex color values, and a reset button to remove the applied color.
- **FR-20.** The editor shall provide a one-tap "mark" action that applies a fixed yellow background with black foreground, independent of the color pickers.

### 7.7 Paragraph alignment

- **FR-21.** The editor shall support paragraph alignment: left (start), center, and right (end).
- **FR-22.** Alignment shall apply to complete paragraphs. If the selection spans multiple paragraphs, all shall be aligned.

### 7.8 Lists

- **FR-23.** The editor shall support unordered (bullet) lists and ordered (numbered) lists.
- **FR-24.** Ordered lists shall support multiple numbering styles: decimal, lower-alpha, upper-alpha, lower-roman, upper-roman.
- **FR-25.** Lists shall support nesting (multiple indent levels), with each level maintaining its own numbering sequence.
- **FR-26.** Pressing Enter within a list item shall create a new list item at the same level.
- **FR-27.** Pressing Enter on an empty list item (or pressing a dedicated "exit list" action) shall terminate the list and return to normal paragraph mode.
- **FR-28.** Inserting or removing list items shall automatically renumber subsequent items in an ordered list.
- **FR-29.** The user shall be able to increase or decrease the indent level of a list item.

### 7.9 Indentation (non-list)

- **FR-30.** The editor shall support increasing and decreasing paragraph indentation independently of list membership.
- **FR-31.** Indentation shall be visually represented as a left margin offset.

### 7.10 Block quotes

- **FR-32.** The editor shall support wrapping paragraphs in a block quote, visually indicated by a colored left border with increased left margin.
- **FR-33.** Pressing Enter within a block quote shall split the quote, allowing the user to interleave non-quoted text between quoted sections.
- **FR-34.** Block quotes shall be nestable (a quote inside a quote).

### 7.11 Headings

- **FR-35.** The editor shall support at least two heading levels, visually distinct from body text in size and weight.
- **FR-36.** Heading styles shall apply to complete paragraphs.

### 7.12 Hyperlinks

- **FR-37.** The editor shall allow the user to create a hyperlink on selected text by specifying a URL.
- **FR-38.** Selecting existing linked text and invoking the link action shall allow editing or removing the URL.
- **FR-39.** Links shall be visually indicated in the editing surface (e.g. underline + color).

### 7.13 Horizontal rule

- **FR-40.** The editor shall allow inserting a horizontal rule (visual line separator) at the cursor position.
- **FR-41.** The horizontal rule shall be rendered as a visible line spanning the width of the message body.

### 7.14 Clear formatting

- **FR-42.** The editor shall provide a "clear formatting on selection" action that removes all character styles and paragraph styles from the selected range.
- **FR-43.** The editor shall provide a "clear all formatting" action that strips all styles from the entire message body.
- **FR-44.** Clear-formatting actions shall preserve hyperlinks, embedded images, and other structural/non-style content.

### 7.15 Plain text mode

- **FR-45.** The editor shall provide a toggleable "plain text only" mode that: (a) hides the formatting toolbar, (b) displays the editing surface in a monospace font, (c) causes the message to be sent without an HTML part.
- **FR-46.** A user preference shall control whether plain-text mode is the default for new compositions.
- **FR-47.** A user preference shall control whether replies to plain-text-only messages automatically use plain-text mode.
- **FR-48.** Switching from rich text to plain text shall strip all formatting from the message content (with appropriate warning if the message contains formatting).

### 7.16 Clipboard and paste

- **FR-49.** Pasting content from the system clipboard in rich text mode shall preserve source formatting by default, converting it to the editor's supported style set.
- **FR-50.** A "paste as plain text" action shall be available (context menu and/or keyboard shortcut) that inserts clipboard content with all formatting stripped.
- **FR-51.** A "paste as quote" action shall be available that wraps pasted content in a block quote.
- **FR-52.** All pasted content shall be sanitized to remove potentially dangerous markup (scripts, event handlers, iframes, object/embed elements, external resource references) before insertion.
- **FR-53.** Pasting into plain-text mode shall always insert as plain text regardless of clipboard content type.

### 7.17 Undo and redo

- **FR-54.** The editor shall support undo and redo operations, reversing or re-applying text and formatting changes.
- **FR-55.** Undo/redo shall be accessible via standard keyboard shortcuts (Ctrl+Z / Ctrl+Shift+Z or equivalent).
- **FR-56.** Undo/redo shall be optionally enabled via a user preference.

### 7.18 Selection behavior

- **FR-57.** A "keep selection" preference shall control whether the user's text selection is preserved after a formatting action is applied.
- **FR-58.** When "keep selection" is disabled, the cursor shall move to the end of the formatted range after applying a style.
- **FR-59.** The "keep selection" toggle shall be accessible from the formatting toolbar or its overflow menu.

### 7.19 Quick-code formatting

- **FR-60.** The editor shall provide a one-tap "code" action that applies small-size monospace formatting to the selection, suitable for marking inline code snippets.

### 7.20 Password-protected content

- **FR-61.** The editor shall allow the user to select text and encrypt it with a user-supplied password, replacing the selected content with an encrypted payload or link that the recipient can decrypt.
- **FR-62.** The maximum size of content that can be password-protected shall be approximately 1,500 characters (including any formatting markup); the editor shall warn the user if the selection exceeds this limit.
- **FR-63.** Decryption of password-protected content shall be available to any recipient without requiring a paid license.

### 7.21 HTML output

- **FR-64.** When the message is sent or saved as a draft, the editor's content shall be serialized to well-formed HTML suitable for email transmission.
- **FR-65.** The HTML output shall use inline styles rather than external stylesheets, for maximum compatibility with email clients that strip `<style>` blocks.
- **FR-66.** The HTML output shall correctly represent all supported formatting: character styles as appropriate tags or inline CSS, lists as `<ul>`/`<ol>`/`<li>`, block quotes as `<blockquote>`, alignment as CSS text-align, etc.
- **FR-67.** The HTML output shall handle bidirectional text correctly, applying appropriate directionality attributes when the content includes right-to-left text.

### 7.22 Convenience text actions

- **FR-68.** When text is selected, the editor shall offer quick actions to wrap/unwrap the selection in parentheses or quotation marks (toggle behavior).

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Applying any formatting action shall produce a visible result within 100 milliseconds under normal conditions, so that the editor feels instantaneous.
- **NFR-2. Stability.** The editor shall not crash or corrupt message content when handling edge cases: empty selections, selections spanning multiple paragraph types, deeply nested lists, or very large messages.
- **NFR-3. Fidelity.** A round-trip from the editor's internal representation to HTML and back (e.g. saving a draft and re-opening it) shall preserve all user-applied formatting without degradation.
- **NFR-4. Interoperability.** The HTML produced shall render acceptably (preserving semantic intent if not pixel-perfect appearance) in at least the top five email clients by market share.
- **NFR-5. Accessibility.** All toolbar buttons shall have descriptive labels readable by screen readers. The toolbar shall be navigable via keyboard. Formatting actions shall be operable without a pointing device.
- **NFR-6. Discoverability.** The toolbar shall use universally recognized iconography (B for bold, I for italic, etc.) with tooltips on hover/long-press.
- **NFR-7. Graceful degradation.** If a paste operation contains markup that the editor cannot represent (e.g. tables, complex CSS), the editor shall make a best-effort conversion to supported formatting rather than silently dropping content.
- **NFR-8. Performance at scale.** The editor shall remain responsive when composing messages of at least 10,000 words with mixed formatting, without perceptible lag on text entry or scrolling.

## 10. Open Questions

- **OQ-1. Heading levels.** The source application's toolbar exposes heading formatting, but the exact number of distinct heading levels supported (and whether they map directly to HTML `<h1>`–`<h6>` or use size/weight combinations) is ambiguous. Design should clarify how many levels are needed for email use cases.
- **OQ-2. Font list governance.** The set of available fonts is partially preference-controlled (bundled fonts, narrow variants). Should the user be able to add custom fonts, or is the curated list fixed? What happens when a chosen font is not available on the recipient's system — is a fallback stack specified?
- **OQ-3. Password-protection size limit.** The source application limits encrypted content to approximately 1,500 characters (or approximately 5,000 in some code paths). The discrepancy should be investigated and a single authoritative limit established.
- **OQ-4. Default text color preference.** The source application supports a user-configured default compose text color. Should this be a property of the editor (applying to all new compositions) or per-account (e.g. different color for work vs. personal)?
- **OQ-5. Toolbar placement.** The source application offers the toolbar in a fixed position or scrolling with content. For a desktop application with more screen real estate, should the toolbar always be fixed (more discoverable) or should the user choice be preserved?
- **OQ-6. Block-quote visual style.** The source application renders block quotes with a colored left border. The specific color, width, and gap are configurable. Should the desktop application match these defaults or adopt platform-native quote styling?
- **OQ-7. List numbering style selection.** The source application supports five numbering styles (decimal, lower-alpha, upper-alpha, lower-roman, upper-roman). How should the user select between these — a sub-menu, a cycling button, or a dialog? The source application's UI for this is not entirely clear from code alone.
- **OQ-8. Interaction with reply quoting.** When replying to a message, the original message is typically quoted. Should formatting toolbar actions be able to modify the quoted portion, or should it be treated as read-only?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Word-boundary auto-expansion.** When no text is selected, character-style actions expand to word boundaries rather than applying to a zero-width insertion point. This is a deliberate UX choice: it prevents the confusing state where formatting is "on" but invisible (because no characters are styled yet), and it matches the most common intent (styling an entire word).

- **N-2. Paragraph-style scope enforcement.** Paragraph-level actions (alignment, lists, indentation, block quotes) always expand their effect to full paragraph boundaries, regardless of the selection. This prevents visually incoherent partial-paragraph formatting that no email client could render correctly.

- **N-3. Sanitization on paste.** The editor aggressively sanitizes pasted HTML. This is not merely a security measure (though it prevents XSS in the compose surface) — it also ensures that the user does not accidentally embed markup that will render differently or be stripped by the recipient's email client. The sanitization boundary is at paste time, not at send time, so the user always sees what the recipient will get.

- **N-4. Keep-selection as opt-in.** The default behavior (cursor moves to end of formatted range) matches mainstream text editors and word processors. "Keep selection" is provided for power users who apply multiple styles in rapid sequence, but is opt-in to avoid confusing the majority.

- **N-5. Relative sizing over absolute.** Font sizes are expressed as relative multipliers rather than absolute point/pixel values. This ensures that the message respects the recipient's base font size preference and scales appropriately on different display densities, which is critical for email where the sender has no control over the rendering environment.

- **N-6. Independence of formatting and transmission.** The editor's rich text capabilities are purely about what the user sees while composing and what HTML is produced. Whether the application also includes a plain-text alternative part in the MIME message, how it handles recipients whose clients cannot render HTML, and how it negotiates format preferences — these are transmission-layer concerns outside this feature.

- **N-7. Password-protection as editor feature, not transport encryption.** The password-protection feature encrypts a *portion* of the visible message content and replaces it with a decryptable payload. It is not end-to-end message encryption (that would be PGP/S/MIME, which is a separate feature). It exists for the use case of sending a single sensitive datum (a password, an account number) within an otherwise normal message.

- **N-8. One toolbar, not contextual ribbons.** The source application uses a single scrollable toolbar rather than context-sensitive ribbon tabs. This reduces cognitive overhead for the email use case where messages are short and formatting needs are modest — the user never has to discover which tab holds a control.
