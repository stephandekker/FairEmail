# Epic: Unlimited Accounts

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Multiple IMAP, POP3 and JMAP accounts; each with its own connection settings, color, name, avatar, and category grouping."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email client that only supports a single account forces the user to choose which mailbox to manage — or to run multiple instances. Most people today have at least two email accounts (personal and work), and power users may have many more across different providers and protocols.

The Unlimited Accounts feature removes this constraint entirely: users may add as many mail accounts as they need, each with its own server connection, credentials, visual identity, and organizational grouping. There is no hard ceiling on the number of accounts.

Each account is a self-contained unit with its own protocol (IMAP or POP3), server endpoints, authentication, folder tree, identities, synchronization cadence, and notification preferences. At the same time, accounts share a common visual and organizational framework — colors, names, avatars, and category labels — so that the user can tell them apart at a glance in every surface where account-specific information appears: message lists, the navigation pane, compose windows, notifications, and desktop widgets.

The feature also provides the scaffolding that every other multi-account feature depends on: the Unified Inbox, cross-account move/copy, per-account rules, and per-account notification channels all presuppose that "account" is a first-class, user-manageable entity with a rich set of properties.

## 2. Goals

- **G1.** Allow the user to configure an unlimited number of mail accounts, with no application-imposed ceiling.
- **G2.** Support IMAP and POP3 protocols, each with their full set of protocol-specific settings, behind a common account abstraction.
- **G3.** Give every account a distinct visual identity — user-chosen color, display name, and avatar image — that propagates consistently to every surface where account context matters.
- **G4.** Provide a category-based grouping mechanism so that users with many accounts can organize them into logical clusters (e.g. "Work", "Personal", "Newsletters").
- **G5.** Allow accounts to be reordered, copied, exported, and imported so that the user's multi-account configuration is portable and manageable.
- **G6.** Ensure that each account is independently configurable for synchronization, notifications, and connection behavior, so that the user can tailor each account to its purpose without affecting the others.
- **G7.** Designate exactly one account as the **primary** account, used as the default when no specific account context is implied.

## 3. Non-Goals

- **NG1.** This epic does **not** define how messages are fetched, stored, or synchronized within an account — only that each account carries the configuration needed for those subsystems to operate.
- **NG2.** This epic does **not** define identities (From addresses, signatures, reply-to) within an account; that is covered by the Identities epic (feature 1.2).
- **NG3.** This epic does **not** define the quick-setup wizard or provider auto-detection; those are covered by features 1.3 and 1.7 respectively. This epic covers only the properties and lifecycle of an account once it exists.
- **NG4.** This epic does **not** define the Unified Inbox, which consumes account data but is a separate feature (3.1).
- **NG5.** This epic does **not** define per-folder settings (synchronization scope, notification rules, poll intervals per folder); those belong to the Folders epic (feature 2.x).
- **NG6.** JMAP protocol support is **not** a goal of this epic. Although the high-level feature list mentions JMAP, the source application does not implement it. If JMAP is added in the future, it should integrate into the same account abstraction defined here.

## 4. Glossary

| Term | Definition |
|---|---|
| **Account** | A configured connection to a single mail provider, with its own credentials, server settings, folder tree, identities, visual identity, and synchronization state. A user may have one or many. |
| **Protocol** | The mail-retrieval protocol used by an account: IMAP or POP3. Each protocol implies a different set of capabilities and settings. |
| **Primary account** | The single account designated as the default for operations where no specific account is implied (e.g. composing from the Unified Inbox). Only one account may be primary at a time. |
| **Account color** | A user-chosen color associated with an account, displayed as a stripe or badge alongside messages, in the navigation pane, in notifications, in the compose window, and in desktop widgets. |
| **Account avatar** | A user-chosen image associated with an account, displayed alongside the account name in the account list. |
| **Account category** | A user-defined text label used to group accounts in the navigation pane. Categories are implicit — they exist as long as at least one account carries that label. |
| **Account order** | A user-defined sort position that overrides the default alphabetical ordering of accounts in lists and the navigation pane. |
| **On-demand account** | An account that synchronizes only when the user explicitly requests it, rather than on a schedule or via push. |
| **Synchronized account** | An account whose synchronization is enabled and whose messages are being kept up to date by the application. |
| **Connection settings** | The set of properties that define how the application connects to the account's mail server: host, port, encryption mode, authentication method, credentials, and optional security features. |
| **Quota** | The server-reported storage usage and limit for an account's mailbox, displayed to the user when the server provides this information. |

## 5. Personas

- **P1. The two-account user.** Has one personal and one work email account. Wants both configured with minimal effort, visually distinguishable, and independently manageable.
- **P2. The multi-account power user.** Has five or more accounts across multiple providers and protocols. Needs categories, custom ordering, and per-account tuning to keep things organized.
- **P3. The POP3 holdout.** Uses a provider or corporate server that only offers POP3. Needs the POP3-specific settings (leave on server, download caps) to be discoverable and correctly defaulted.
- **P4. The migrator.** Is setting up a new machine and wants to export their entire multi-account configuration from one installation and import it into another without re-entering credentials and preferences.
- **P5. The newcomer.** Is adding their first account and expects the process to result in a working, ready-to-use mailbox with sensible defaults.

## 6. User Stories

### 6.1 Adding and removing accounts

> **US-1.** As any user, I want to add a new mail account by providing my server connection details, so that I can begin receiving and sending mail through that account.

> **US-2.** As any user, I want to add accounts using either the IMAP or POP3 protocol, so that I can connect to any mail server regardless of which protocol it supports.

> **US-3.** As any user, I want to add as many accounts as I need without hitting an application-imposed limit, so that the application scales with my needs.

> **US-4.** As any user, I want to delete an account permanently, with a confirmation prompt, so that I can remove accounts I no longer use — understanding that all associated data (folders, messages, identities, rules, contacts) will be removed with it.

> **US-5.** As any user, I want to duplicate an existing account's configuration into a new account, so that I can set up a similarly-configured account (e.g. on the same provider) without re-entering all settings.

### 6.2 Connection settings

> **US-6.** As any user, I want to configure the server hostname, port, and encryption mode (SSL/TLS, STARTTLS, or none) for each account independently, so that I can connect to any server.

> **US-7.** As any user, I want to authenticate using a username and password, or via an OAuth2 flow for supported providers, so that I can use whichever authentication method my provider requires.

> **US-8.** As a security-conscious user, I want to enable DNSSEC validation and DANE certificate verification per account, so that I can harden the connection to servers that support these standards.

> **US-9.** As a user connecting to a server with a self-signed or otherwise non-standard certificate, I want to pin a specific certificate fingerprint or allow insecure connections for that account only, so that I am not blocked from connecting while other accounts remain strictly validated.

> **US-10.** As a user whose server requires client-certificate authentication, I want to select a client certificate from my system keystore for a specific account, so that mutual TLS authentication works.

> **US-11.** As any user, I want to test the connection to a server before saving the account, so that I can verify my settings are correct.

### 6.3 Visual identity

> **US-12.** As any user, I want to assign a display name to each account, so that I can identify it by a meaningful label rather than a server address.

> **US-13.** As any user, I want to assign a color to each account, so that messages, folders, and notifications from that account are visually distinguishable from those of other accounts.

> **US-14.** As any user, I want the account color to appear consistently as a color stripe or badge on messages in the message list, in the navigation pane, in notifications, in the compose window, and in desktop widgets, so that account attribution is always visible at a glance.

> **US-15.** As any user, I want to assign an avatar image to each account, so that the account is visually identifiable in the account list.

> **US-16.** As any user, I want to clear or change the color or avatar at any time, so that my visual choices are always reversible.

### 6.4 Categories and ordering

> **US-17.** As a multi-account user, I want to assign a category label to each account (e.g. "Work", "Personal"), so that accounts are grouped under category headers in the navigation pane.

> **US-18.** As a multi-account user, I want category labels to be free-form text with autocomplete from existing categories, so that I can create new categories simply by typing a new name and reuse existing ones easily.

> **US-19.** As any user, I want to reorder accounts via drag-and-drop, so that I can arrange them in the order that makes sense to me rather than being forced into alphabetical order.

> **US-20.** As any user, I want to reset the custom ordering back to the default (primary first, then alphabetical), so that I can undo a reordering I no longer want.

> **US-21.** As any user, I want category grouping in the navigation pane to be togglable via a setting, so that I can choose whether accounts are grouped by category or shown in a flat list.

### 6.5 Primary account

> **US-22.** As any user, I want to designate one account as the primary account, so that the application knows which account to use by default for new compositions and other ambiguous-account situations.

> **US-23.** As any user, I want the primary account to be visually indicated (e.g. with a star or similar marker) in the account list, so that I can see at a glance which account is primary.

> **US-24.** As any user, I want only synchronized accounts to be eligible for primary designation, so that the primary account is always one that is actively receiving mail.

> **US-25.** As any user, when I set an account as primary, I want the previously-primary account to be automatically demoted, so that there is always exactly one primary account.

### 6.6 Synchronization and connection behavior

> **US-26.** As any user, I want to enable or disable synchronization for each account independently, so that I can temporarily or permanently stop syncing an account without deleting it.

> **US-27.** As any user, I want to mark an account as "on-demand", so that it synchronizes only when I explicitly request it rather than continuously or on a schedule.

> **US-28.** As any user, I want to set the polling / keep-alive interval per account, so that accounts with different responsiveness needs can sync at different rates.

> **US-29.** As any user, I want to constrain an account to synchronize only on unmetered networks or only when a VPN is active, so that I can control data usage and security on a per-account basis.

> **US-30.** As any user, I want to exempt a specific account from the global synchronization schedule, so that a critical account continues to sync even during off-hours.

### 6.7 POP3-specific settings

> **US-31.** As a POP3 user, I want to choose whether downloaded messages are left on the server, so that I can either preserve them for access from other clients or remove them to save server space.

> **US-32.** As a POP3 user, I want to choose whether messages deleted on the device are also deleted from the server, so that I control the relationship between local and server-side state.

> **US-33.** As a POP3 user, I want to choose whether messages deleted from the server are kept on the device, so that I retain a local archive even if the server copy is removed.

> **US-34.** As a POP3 user, I want to set a maximum number of messages to download, so that I can limit storage and bandwidth consumption.

> **US-35.** As a POP3 user, I want to be clearly informed of the limitations of POP3 compared to IMAP (no server-side folders, no server-side search, no remote flag synchronization, local-only sent/drafts/trash), so that I set appropriate expectations.

### 6.8 Per-account defaults

> **US-36.** As any IMAP user, I want to designate which server folder serves as my Drafts, Sent, Archive, Trash, and Junk folder for each account, so that the application files messages into the correct folders.

> **US-37.** As any user, I want to configure default swipe actions and a default "move-to" folder per account, so that quick-triage gestures target the right destination for each account.

### 6.9 Notifications

> **US-38.** As any user, I want to enable or disable notifications independently for each account, so that noisy accounts do not generate alerts while important ones do.

> **US-39.** As any user, I want a separate notification channel per account (where the desktop notification system supports it), so that I can customize sound, priority, and behavior per account at the system level.

### 6.10 Quota and diagnostics

> **US-40.** As any user, I want to see my mailbox quota usage (used / limit) for each account when the server reports it, with a visual warning when usage is high, so that I can manage my storage proactively.

> **US-41.** As any user, I want to see the current connection state of each account (connected, connecting, disconnected, error with detail, backing off until a specific time), so that I can diagnose connectivity problems.

> **US-42.** As any user, I want to view the connection log for a specific account, so that I can troubleshoot server-side issues.

### 6.11 Export and import

> **US-43.** As any user, I want to export all account configurations (including identities, folder mappings, rules, and contacts) to a portable file, optionally password-protected, so that I can back up my setup or transfer it to another machine.

> **US-44.** As any user, I want to import accounts from a previously exported file, with duplicate detection (matching by unique identifier), so that I can restore or migrate my setup without creating duplicates.

> **US-45.** As any user, I want to selectively choose which accounts and which categories of data to include in an export or import, so that I have fine-grained control over what is transferred.

## 7. Functional Requirements

### 7.1 Account entity

- **FR-1.** The application shall support an unlimited number of user-configured mail accounts. No application-level ceiling shall be enforced.
- **FR-2.** Each account shall be uniquely identified by a persistent, globally-unique identifier that survives export, import, and re-creation.
- **FR-3.** Each account shall store the following connection properties: protocol (IMAP or POP3), server host, server port, encryption mode (SSL/TLS, STARTTLS, or none), authentication method, username, and password or token.
- **FR-4.** Each account shall store the following optional security properties: DNSSEC enforcement flag, DANE enforcement flag, insecure-connection flag, server certificate fingerprint (for pinning), client certificate reference, and authentication realm.
- **FR-5.** Each account shall store the following display properties: user-defined name, user-defined color, user-defined avatar image (as a persistent reference), and user-defined category label.
- **FR-6.** Each account shall store the following behavioral properties: synchronization enabled/disabled flag, on-demand flag, primary flag, notification enabled flag, notification summary flag, browse-folders flag, and polling/keep-alive interval.
- **FR-7.** Each account shall store optional network conditions governing when synchronization is permitted: unmetered-only flag, VPN-only flag, and schedule-exemption flag.

### 7.2 Protocol support

- **FR-8.** The application shall support the IMAP protocol with full two-way folder and flag synchronization.
- **FR-9.** The application shall support the POP3 protocol with the following protocol-specific settings: leave-on-server, client-delete-after-download, leave-deleted-on-server, leave-deleted-on-device, and maximum-messages-to-download.
- **FR-10.** POP3 accounts shall present a fixed set of local-only folders (Inbox, Drafts, Sent, Trash) rather than a server-derived folder tree, and the application shall inform the user of this limitation.
- **FR-11.** POP3 accounts shall be visually distinguished from IMAP accounts in the account list (e.g. by a different text color or label), so that users are aware of the protocol in use.

### 7.3 Visual identity

- **FR-12.** The user shall be able to set, change, and clear a color for any account at any time.
- **FR-13.** The user shall be able to set, change, and clear an avatar image for any account at any time.
- **FR-14.** The account color shall be displayed consistently in the following surfaces: message-list color stripe or badge, navigation pane folder icons, account settings list, compose window, notifications, and desktop widgets.
- **FR-15.** When multiple levels of color are available (account, folder, identity), the most specific color shall take precedence: identity color overrides folder color, which overrides account color.
- **FR-16.** The account name shall be displayed wherever account attribution is needed, and shall be user-editable at any time.

### 7.4 Categories and ordering

- **FR-17.** Each account shall have an optional category label (free-form text).
- **FR-18.** When category display is enabled in the navigation pane, accounts shall be grouped under their category headers, sorted alphabetically by category, then by custom order within each category.
- **FR-19.** When category display is disabled, accounts shall be sorted by custom order, then by primary status (primary first), then alphabetically by name.
- **FR-20.** The user shall be able to reorder accounts via drag-and-drop. The custom order shall persist across restarts.
- **FR-21.** The user shall be able to reset the custom order, reverting to the default sort.
- **FR-22.** Categories shall be implicitly created when assigned to an account and implicitly deleted when no account carries that label. There is no separate category-management interface.
- **FR-23.** When editing an account's category, the application shall offer autocomplete suggestions from existing categories.

### 7.5 Primary account

- **FR-24.** Exactly one account may be designated as primary at any time.
- **FR-25.** Only a synchronized (synchronization-enabled) account may be designated as primary.
- **FR-26.** When a new account is set as primary, the previously-primary account shall be automatically demoted.
- **FR-27.** The primary account shall be visually indicated in the account list (e.g. with a star icon).
- **FR-28.** If the first account is added and no primary account exists, that account shall be designated primary by default.

### 7.6 Account lifecycle

- **FR-29.** The user shall be able to delete an account. Deletion shall be preceded by a confirmation dialog that names the account.
- **FR-30.** Deleting an account shall remove all associated data: folders, messages, identities, pending operations, rules, and contacts belonging to that account.
- **FR-31.** The user shall be able to duplicate an account's configuration into a new account. The duplicate shall open in an editor pre-filled with the source account's settings, allowing modification before saving.
- **FR-32.** Disabling synchronization for an account shall clear any active error or warning state for that account and revoke its primary designation if applicable.

### 7.7 Connection testing

- **FR-33.** The user shall be able to test the connection to the configured server before saving an account, receiving clear success or failure feedback.
- **FR-34.** Connection testing shall verify both the incoming (IMAP/POP3) and outgoing (SMTP) servers when applicable.

### 7.8 Per-account folder defaults (IMAP)

- **FR-35.** For IMAP accounts, the user shall be able to designate which server folder serves each system role: Drafts, Sent, Archive, Trash, and Junk.
- **FR-36.** The application shall auto-detect system folders from server metadata when available, and allow user overrides.

### 7.9 Per-account swipe and move defaults

- **FR-37.** Each account shall store configurable default swipe-left and swipe-right target folders (or actions), so that quick-triage gestures are account-specific.
- **FR-38.** Each account shall store a configurable default "move-to" folder, so that the move action has a per-account default destination.

### 7.10 Notifications

- **FR-39.** Per-account notification enablement shall be independent of synchronization enablement and independent of any folder's notification setting.
- **FR-40.** When per-account notifications are enabled, the application shall create a dedicated notification channel for that account (where the notification system supports channels), allowing the user to customize notification behavior at the system level.
- **FR-41.** Disabling or deleting an account shall remove its notification channel.

### 7.11 Quota display

- **FR-42.** When the server reports mailbox quota information, the application shall display current usage and limit in the account settings view.
- **FR-43.** When quota usage exceeds a high threshold (e.g. 95%), the application shall display a visual warning.

### 7.12 Connection state and diagnostics

- **FR-44.** Each account shall display its current connection state (connected, connecting, disconnected, closing, backing off) via a status indicator in the account list.
- **FR-45.** When an account is in an error state, the error detail shall be displayed in the account settings view.
- **FR-46.** The user shall be able to view a per-account connection log for diagnostic purposes.

### 7.13 Export and import

- **FR-47.** The application shall support exporting account configurations (including nested identities, folder mappings, rules, and contacts) to a portable file format.
- **FR-48.** The export shall support optional password-based encryption.
- **FR-49.** The application shall support importing accounts from a previously-exported file, with duplicate detection based on the account's unique identifier.
- **FR-50.** The user shall be able to selectively include or exclude specific accounts and data categories during export and import.

### 7.14 Advanced connection tuning

- **FR-51.** Each account shall expose the following advanced fetch settings: partial-fetch mode, raw-fetch mode, ignore-size-limits flag, date-header preference (server time, Date header, or Received header), and unicode/UTF-8 support flag.
- **FR-52.** Each account shall expose the following keep-alive settings: polling interval and use-NOOP-instead-of-IDLE flag.
- **FR-53.** Advanced settings shall be hidden behind an expandable "Advanced" section to avoid overwhelming new users.

## 8. Non-Functional Requirements

- **NFR-1. No hard account limit.** The application shall remain functional and responsive with at least 20 simultaneously configured accounts.
- **NFR-2. Independent failure isolation.** A connection failure, authentication error, or server outage on one account shall not prevent other accounts from synchronizing or being used.
- **NFR-3. Atomic account operations.** Creating or updating an account (including its folder mappings, swipe actions, and related settings) shall be transactional — either all changes are persisted or none are.
- **NFR-4. Responsive account list.** The account list, navigation pane, and category grouping shall render without perceptible delay with at least 20 accounts.
- **NFR-5. Secure credential storage.** Account passwords and tokens shall be stored in a manner that is not trivially readable by other processes or by filesystem inspection.
- **NFR-6. Offline account management.** Account configuration (creation, editing, deletion, reordering, color/avatar changes) shall be possible while offline. Connection-dependent operations (test connection, folder discovery) shall fail gracefully with a clear message.
- **NFR-7. Accessibility.** All account-management controls (add, edit, delete, reorder, color picker, avatar picker, category field, primary toggle) shall be reachable via keyboard, with screen-reader labels, and shall respect the application-wide theme.
- **NFR-8. Consistent color rendering.** The account color shall render identically (same hue, same saturation) across all surfaces — message list, navigation pane, compose window, notifications, and widgets — regardless of theme (light, dark, or high-contrast).

## 10. Open Questions

- **OQ-1. JMAP support.** The high-level feature list includes JMAP alongside IMAP and POP3, but the source application does not implement JMAP, noting that few providers offer it and library support is lacking. Should JMAP be included in the initial scope, deferred to a future epic, or dropped?
- **OQ-2. Account-level avatars vs. contact avatars.** The source application supports both a per-account avatar (a user-chosen image for the account itself) and per-message contact photos (Gravatar, Libravatar, BIMI, identicons, etc.). The interaction between these two — whether the account avatar ever overrides or supplements contact photos in the message list — is not clearly defined. This should be clarified during design.
- **OQ-3. Pro-feature gating.** In the source application, account color, avatar, and per-account notification channels are gated behind a paid "Pro" license. The desktop application needs to decide whether to retain a tiered model or include all features in a single edition.
- **OQ-4. Category management UX.** Categories are currently implicit (created by typing a name, deleted when no account uses them). Should there be an explicit category-management screen for renaming, reordering, or color-coding categories themselves? The source application does not provide one.
- **OQ-5. POP3 limitations disclosure.** The source application shows inline warning text on the POP3 setup screen. The optimal way to communicate POP3 limitations on a desktop (inline warnings, a dedicated help panel, a pre-setup dialog) should be decided during design.
- **OQ-6. Account-level schedule exemption.** The source application allows individual accounts to be exempted from the global sync schedule. Whether this per-account exemption belongs in the account settings or in a centralized scheduling UI is a design decision.
- **OQ-7. OAuth provider list.** The set of providers that support OAuth sign-in (Gmail, Outlook, Yahoo, AOL, etc.) is maintained in a bundled provider database. The initial provider list for the desktop application and the mechanism for updating it should be decided as part of the setup-wizard epic (1.3/1.5), but the account model must accommodate OAuth credentials and token refresh.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. No account limit.** The source application imposes no ceiling on the number of accounts. The database uses auto-incrementing identifiers, and no guard in the setup flow rejects a new account based on count. This is intentional — the application positions itself as suitable for power users with many accounts.
- **N-2. Protocol stored per account, not per server.** Each account is bound to exactly one protocol (IMAP or POP3). If a user wants to connect to the same server via both IMAP and POP3 (unusual but possible), they create two accounts. This keeps the account model simple and avoids protocol-switching complexity.
- **N-3. Color precedence hierarchy.** When displaying a message, the most specific color wins: identity color (if set) overrides folder color, which overrides account color. This allows a user who wants one color per account to set it at the account level and be done, while a user who wants fine-grained coloring can override at the folder or identity level. The account color is the broadest brush.
- **N-4. Categories are strings, not entities.** Categories are stored as plain text on each account, not as separate entities with their own identifiers. This makes them lightweight (no separate management screen, no orphan cleanup) at the cost of being case-sensitive and lacking their own metadata (no category color or icon). The source application mitigates this with autocomplete from existing category names.
- **N-5. Primary requires synchronization.** An account that is not synchronizing cannot be primary. This prevents the situation where the default account for composition is one that cannot send or receive mail. When synchronization is disabled on the primary account, the primary designation is automatically revoked.
- **N-6. POP3 visual warning.** POP3 accounts are displayed in a distinct (warning) color in the account list. This is a deliberate design choice to remind users that POP3 accounts have significant limitations compared to IMAP, and to prompt consideration of whether IMAP is available from the same provider.
- **N-7. Account deletion is permanent.** There is no "archive" or "disable" intermediate state for deletion. Disabling synchronization is the soft alternative; deletion is the hard one. The confirmation dialog and the cascade (all folders, messages, identities, rules, contacts deleted) make the consequences clear.
- **N-8. Duplicate detection on import.** When importing accounts from a backup file, the application matches by the account's unique identifier. If an account with the same identifier already exists, the import skips it (or updates it, depending on the flow). This prevents accidental duplication when restoring from backup.
