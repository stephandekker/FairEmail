# Epic: Original (Raw) Message View

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.7** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Original (raw) message view. Switch to the original HTML/plain rendering if needed."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email messages arrive in a wide range of formats: rich HTML with embedded styling, simple plain text, or a mix of both. For privacy and security reasons, a modern email client defaults to a **reformatted** (sanitized) view that strips tracking content, disables scripts, neutralizes unsafe CSS, and standardizes layout. This protects users from phishing, fingerprinting, and unintended remote resource loading.

However, some messages — particularly marketing emails, newsletters, HTML receipts, or messages with bespoke formatting — are only fully legible in their original form. The reformatting may strip colors, break table layouts, remove custom fonts, or collapse intentional spacing. Users need a way to **opt into** viewing the message exactly as the sender authored it, with full awareness of the trade-offs.

The Original Message View feature provides that escape hatch: a one-click toggle that switches from the safe, reformatted rendering to the sender's original HTML (or plain text), displayed faithfully in a browser-style rendering surface. The feature is intentionally friction-ful — it requires confirmation by default — because viewing original content carries privacy implications (remote image loading, tracking pixels, JavaScript in some clients).

## 2. Goals

- **G1.** Allow any user to view a message in its original, unsanitized form when the reformatted view is insufficient.
- **G2.** Make the toggle between reformatted and original views instant, reversible, and discoverable within the message reading pane.
- **G3.** Protect user privacy by defaulting to the reformatted view and requiring explicit opt-in (with optional confirmation) before showing original content.
- **G4.** Allow users to establish trust at the sender or domain level so that familiar senders' messages are shown in original form automatically.
- **G5.** Provide a full-screen original-message view for messages whose layout requires more space than the inline reading pane provides.
- **G6.** Support light-theme rendering of original messages even when the application is in dark mode, to preserve readability of dark-on-light email designs.

## 3. Non-Goals

- **NG1.** This epic does **not** cover the display of raw RFC 2822 headers or MIME source; that is a separate power-user feature (see feature 3.17).
- **NG2.** This epic does **not** define the reformatted/sanitized view itself (feature 3.6) — only the mechanism for bypassing it.
- **NG3.** This epic does **not** prescribe how remote images are handled once original view is active; external-image confirmation (feature 3.10) and tracker-image detection (feature 3.11) remain independent concerns that may layer on top of original view rendering.
- **NG4.** This epic does **not** cover saving or exporting the raw `.eml` file — that is a distinct action available from the message menu.
- **NG5.** Editing, annotating, or modifying the original message content is not in scope.

## 4. Glossary

| Term | Definition |
|---|---|
| **Reformatted view** | The default rendering of a message after sanitization: scripts removed, unsafe CSS neutralized, remote content blocked, layout standardized for readability and safety. |
| **Original view** | A rendering of the message's HTML (or plain text) body exactly as authored by the sender, displayed in a browser-grade rendering surface with full CSS and layout fidelity. |
| **Confirmation dialog** | A user-facing prompt displayed before switching to original view, warning that the action may expose privacy-sensitive information (tracking pixels, remote resource fetches). |
| **Sender trust** | A per-sender or per-domain preference that suppresses the confirmation dialog and/or auto-enables original view for messages from that origin. |
| **Full-screen original view** | A maximized rendering surface (separate window or panel) that displays the original message HTML at full width, outside the constraints of the message reading pane. |
| **Force-light rendering** | An option to render the original message with a light background regardless of the application's current theme, preventing dark-on-dark readability issues. |
| **Fit-to-width** | A zoom/scaling mode that shrinks wide original messages to fit the available viewport width. |

## 5. Personas

- **P1. The privacy-conscious reader.** Keeps the reformatted view as their default; rarely switches to original view; appreciates the confirmation dialog and leaves it enabled.
- **P2. The newsletter reader.** Subscribes to many HTML-rich newsletters and marketing emails; frequently needs original view to see intended layouts; wants to trust specific senders so the switch happens automatically.
- **P3. The power user.** Wants full control: toggle freely between views, open full-screen for wide emails, choose their rendering theme, and never be interrupted by confirmation dialogs.
- **P4. The troubleshooter.** Occasionally needs original view to diagnose rendering issues, verify what was actually sent, or confirm that a suspicious message looks intentional.

## 6. User Stories

### 6.1 Switching to original view

> **US-1.** As any user, when reading a message in the reformatted view, I want to switch to the original view with a single action (button or keyboard shortcut), so that I can see the message as the sender intended.

> **US-2.** As any user, when I switch to original view, I want to be warned (via a confirmation dialog) that this may leak privacy-sensitive information, so that I make an informed choice.

> **US-3.** As a power user, I want to disable the confirmation dialog globally, so that switching to original view is instantaneous without interruption.

### 6.2 Switching back to reformatted view

> **US-4.** As any user, when viewing the original message, I want to switch back to the reformatted view with the same toggle control, so that the action is reversible and symmetric.

> **US-5.** As any user, I want the toggle state to be remembered for the current message across navigation (leaving and returning to the message), so that I do not have to re-enable original view each time I revisit a message.

### 6.3 Sender and domain trust

> **US-6.** As a newsletter reader, when the confirmation dialog appears, I want an option to "always show original for this sender" so that future messages from the same address bypass the dialog and display in original view automatically.

> **US-7.** As a newsletter reader, I want an option to "always show original for this domain" so that all messages from a particular organization are shown in original view without repeated confirmations.

> **US-8.** As any user, I want to be able to reset sender/domain trust decisions, so that I can revoke automatic original-view display if my preferences change.

### 6.4 Full-screen original view

> **US-9.** As any user, I want to open the original message in a full-screen or maximized panel, so that wide HTML layouts (tables, multi-column designs) are legible without horizontal scrolling.

> **US-10.** As a power user, I want the full-screen view to support fit-to-width scaling, so that even very wide emails are visible without manual zooming.

> **US-11.** As a power user, I want the option to use a monospaced font in the full-screen view for preformatted text blocks, so that ASCII art, code, and aligned text display correctly.

### 6.5 Theme and rendering options

> **US-12.** As any user viewing original messages in a dark application theme, I want to force a light background for the original message content, so that emails designed for white backgrounds remain readable.

> **US-13.** As any user, I want a global preference to default to light-background rendering for all original message views, so that I do not have to toggle it per message.

> **US-14.** As any user, I want the original view to auto-scale text for readability when the content would otherwise render at an uncomfortably small size, so that I can read the message without manual zooming.

### 6.6 Discoverability

> **US-15.** As a new user, when I first open a message in the reformatted view, I want a brief, dismissible hint explaining that I can tap/click the full-screen icon to view the original message, so that I learn the feature exists.

> **US-16.** As any user, I want the toggle button's icon to change state to reflect whether I am currently viewing the reformatted or original version, so that the current mode is always visually clear.

### 6.7 Inline trigger for large messages

> **US-17.** As any user, when a message is too large to display fully in reformatted mode and has been truncated, I want a visible link or button at the truncation point that switches to original view, so that I can see the full content without hunting for the toggle.

### 6.8 Privacy integration

> **US-18.** As a privacy-conscious user, I want the original view to respect my tracker-blocking and image-blocking preferences independently, so that switching to original view does not automatically load all remote resources unless I separately allow them.

> **US-19.** As any user, I want the confirmation dialog to include an option to "always show images when showing original messages", so that I can opt into full-fidelity rendering in one step.

## 7. Functional Requirements

### 7.1 View toggle

- **FR-1.** Every message displayed in the reading pane shall have a toggle control (button, icon, or keyboard shortcut) that switches between the reformatted view and the original view.
- **FR-2.** The toggle shall be bidirectional: activating it while in reformatted view switches to original; activating it while in original view switches back to reformatted.
- **FR-3.** The toggle state for a given message shall be persisted so that returning to the message (within the same session or across restarts) restores the last-chosen view mode.
- **FR-4.** The toggle control shall display a distinct icon or visual state for each mode, making the current rendering mode unambiguous.

### 7.2 Confirmation dialog

- **FR-5.** By default, switching to original view shall present a confirmation dialog before rendering the original content.
- **FR-6.** The confirmation dialog shall communicate that viewing the original message may leak privacy-sensitive information (e.g. tracking pixels, remote resource fetches).
- **FR-7.** The confirmation dialog shall offer at minimum the following response options:
  - (a) Show original this time only.
  - (b) Always show original for this sender.
  - (c) Always show original for this sender's domain.
  - (d) Cancel (remain in reformatted view).
- **FR-8.** The confirmation dialog shall offer a checkbox or option to "always show images when showing original messages", controlling whether remote images are also unblocked.
- **FR-9.** A global preference shall allow the user to disable the confirmation dialog entirely, making the toggle immediate.
- **FR-10.** When the dialog is disabled (globally or via sender/domain trust), switching to original view shall be immediate with no intermediate prompt.

### 7.3 Sender and domain trust

- **FR-11.** When a user selects "always show original for this sender" or "always show original for this domain", the application shall store that preference durably.
- **FR-12.** On subsequent messages from a trusted sender or domain, the application shall display the message in original view automatically (bypassing the toggle and dialog).
- **FR-13.** A global setting shall exist to reset all sender/domain trust decisions at once.
- **FR-14.** Individual sender/domain trust decisions shall be revocable (the mechanism — per-sender settings, a list view, or equivalent — is a design decision).

### 7.4 Original view rendering

- **FR-15.** The original view shall render the message's HTML body with full CSS support, table layout, embedded images (subject to image-blocking preferences), and custom fonts.
- **FR-16.** The original view shall render plain-text messages faithfully, preserving whitespace and line breaks.
- **FR-17.** The original view shall not execute scripts or permit form submissions, even though it renders the full visual fidelity of the HTML.
- **FR-18.** Links within the original view shall be clickable and shall invoke the application's standard link-handling behavior (confirmation, sanitization, external browser).
- **FR-19.** The original view shall support text auto-sizing to ensure minimum legibility regardless of the author's font-size choices.

### 7.5 Full-screen original view

- **FR-20.** A separate action (distinct from the inline toggle) shall open the original message in a full-screen or maximized rendering surface.
- **FR-21.** The full-screen view shall support a fit-to-width mode that scales the message content to fill the available viewport width.
- **FR-22.** The full-screen view shall support an optional monospaced font mode for preformatted text.
- **FR-23.** The full-screen view shall be dismissible, returning the user to the reading pane with its prior state intact.
- **FR-24.** The full-screen view shall respect the same force-light and image-blocking preferences as the inline original view.

### 7.6 Theme and rendering preferences

- **FR-25.** A global preference shall control whether original views default to a light background (force-light mode) or follow the application theme.
- **FR-26.** A per-message override shall allow the user to toggle force-light rendering on or off for the current message.
- **FR-27.** When the application is in dark mode and force-light is **not** active, the rendering surface shall apply an algorithmic darkening transform to the original content. If algorithmic darkening is not supported (platform limitation), the application shall inform the user and recommend force-light mode.
- **FR-28.** A global preference shall control whether original messages are scaled to fit the viewport width (fit-to-width / overview mode).

### 7.7 Large-message trigger

- **FR-29.** When a message body exceeds the application's configured display-size threshold and is truncated in reformatted view, a visible action (link or button) shall appear at the truncation point offering to show the full original message.
- **FR-30.** Activating the truncation-point action shall behave identically to the standard original-view toggle (including confirmation dialog if applicable).

### 7.8 Discoverability

- **FR-31.** On first encountering a reformatted message (or on first use), the application shall display a brief, dismissible hint directing the user to the original-view toggle.
- **FR-32.** The hint shall not re-appear once dismissed.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Switching between reformatted and original view shall complete rendering within one second for messages up to 500 KB of HTML body.
- **NFR-2. Memory.** Rendering very large original HTML messages (up to several MB) shall not cause the application to crash or become unresponsive; graceful degradation (e.g. truncation with a warning) is acceptable.
- **NFR-3. Privacy by default.** The reformatted view shall remain the default for all messages unless overridden by sender/domain trust or a global "always show original" preference. No original content shall be rendered without explicit user action.
- **NFR-4. Security.** Script execution and form submission shall be blocked in the original view under all circumstances.
- **NFR-5. Offline.** Original view shall work offline for messages whose body has been downloaded; no network fetch shall be required to switch views.
- **NFR-6. Accessibility.** The toggle control, confirmation dialog, full-screen view, and all rendering preferences shall be operable via keyboard and compatible with screen readers.
- **NFR-7. Consistency.** Switching to original view shall not alter the message's flags, read state, or any server-side property.
- **NFR-8. Independence.** The original view feature shall operate independently of remote-image blocking, tracker detection, and link sanitization — those features layer on top and have their own controls.

## 10. Open Questions

- **OQ-1. Trust granularity.** The source application supports both per-sender and per-domain trust. Should there be a UI to browse and manage these individual entries, or is a global "reset all" sufficient?
- **OQ-2. Interaction with "always show images".** When a user selects "always show images when showing original messages", does this override their per-sender image blocking preferences, or does it only apply to messages where original view is active? The boundary between this feature and feature 3.10 needs clarification.
- **OQ-3. Algorithmic darkening quality.** The source application notes that displaying original messages on a dark background "is not possible" on older platforms because dark text/images become invisible. What is the minimum acceptable platform capability, and what should the fallback behavior be on systems that lack darkening support?
- **OQ-4. Full-screen as separate window.** On a desktop application, should "full-screen original view" open in a new window, a maximized panel within the main window, or a tab? The source application uses a dialog overlay; the desktop equivalent needs a design decision.
- **OQ-5. Persistence scope of toggle state.** The toggle state is persisted per message. Should this survive indefinitely (stored alongside the message), or should it be session-scoped and reset on next launch? The source application persists it permanently.
- **OQ-6. Monospaced font scope.** The monospaced font option is available in the full-screen view. Should it also be available in the inline original view, or is the inline view too space-constrained for this to be useful?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Privacy-first default.** The reformatted view is the default because most email-borne privacy threats (tracking pixels, fingerprinting via CSS, remote resource enumeration) require the original HTML to be rendered. This is a deliberate friction: the original view is one click away, but never zero clicks (unless the user explicitly opts in via sender trust or global preference). The confirmation dialog is the primary mechanism for informed consent.

- **N-2. Confirmation dialog as teaching moment.** The dialog is not merely a gate — it educates the user about the privacy implications. This is why it is enabled by default and requires a conscious decision to disable. The per-sender/domain trust options in the dialog serve dual duty: they reduce future friction *and* they make the user aware that trust is a per-origin decision.

- **N-3. Separate full-screen action.** The inline toggle and the full-screen action are intentionally distinct. The inline toggle switches rendering mode in-place (quick, low-commitment). The full-screen action opens a dedicated surface for wide/complex emails. Conflating them would either force all original views into full-screen (disruptive) or deny the user a full-width rendering (limiting).

- **N-4. Force-light as explicit override.** Many HTML emails are designed with a white background. When the email client applies dark-mode transforms, dark text on the sender's assumed-white background becomes dark-on-dark and invisible. Force-light rendering exists specifically for this failure mode. It is not about user preference for light themes — it is about preserving the sender's intended contrast ratios.

- **N-5. Toggle state persistence.** The source application stores the original-view preference per message permanently. This means "I chose to view this newsletter in original view" is remembered forever. The rationale is that viewing an already-rendered message is not a new privacy decision — the user already consented. Re-prompting on every revisit would be friction without benefit.

- **N-6. Large-message trigger.** The "show full message" link at the truncation point is a secondary entry point to original view. It exists because truncation is the most visible symptom of the reformatting engine reaching its limits, and at that moment the user is most likely to want the original. It intentionally shares the same confirmation flow to avoid creating an unguarded path to original rendering.
