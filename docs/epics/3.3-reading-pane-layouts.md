# Epic: Reading-Pane Layouts

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.3** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Single, two-column or three-column pane layouts that adapt to window width."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

A desktop email client needs to make efficient use of the available window space. On a narrow window or a small display, showing one thing at a time (folder list → message list → message reader) is the only practical approach. On a wide window or a large display, the user should be able to see the message list and the message reader side by side — and on a very wide window, the navigation panel, the message list, and the message reader can all be visible simultaneously.

The Reading-Pane Layouts feature provides three distinct arrangements — **single-pane**, **two-pane**, and **three-pane** — and lets the user control which arrangement is active for each window orientation, with optional minimum-size thresholds so the split layouts are never forced onto windows too small to support them.

The feature also supports two different split directions for the two-pane mode: a **horizontal split** (message list on the left, reader on the right — the classic "columns" layout) and a **vertical split** (message list on top, reader on the bottom — the "rows" layout). Which direction is used depends on the orientation and the user's preference.

## 2. Goals

- **G1.** Provide three layout tiers — single-pane, two-pane, and three-pane — that give progressively more information density as window width increases.
- **G2.** Let the user choose which layout is active for each window orientation (portrait and landscape) independently.
- **G3.** Allow both horizontal (side-by-side columns) and vertical (stacked rows) split directions in the two-pane mode.
- **G4.** Provide a sensible default configuration so that the user gets a useful two-column layout in landscape and a single-pane layout in portrait without configuring anything.
- **G5.** Allow the user to control the relative size of the panes (e.g. the percentage of width devoted to the message list versus the reader).
- **G6.** Make the layout adapt gracefully when the window is resized or the orientation changes, preserving the user's reading context wherever possible.

## 3. Non-Goals

- **NG1.** This epic does **not** define the content of the navigation panel, the message list, or the message reader — only how the three regions are arranged and sized.
- **NG2.** This epic does **not** prescribe how the layout decision is persisted — only that it is durable and per-orientation.
- **NG3.** Arbitrary user-defined pane counts (e.g. four panes) are out of scope.
- **NG4.** Side-by-side multi-window with external applications (i.e. opening a link or attachment in a second window next to the mail client) is a separate concern and is not covered here.
- **NG5.** This epic does **not** define the visual content of the separator between panes (handle appearance, drag behavior for resizing) — only that a visible separator exists and that the pane proportions are configurable via a setting.

## 4. Glossary

| Term | Definition |
|---|---|
| **Single-pane mode** | A layout in which only one content region is visible at a time: the user navigates sequentially from the folder/account list to the message list to the message reader, using forward and back navigation. |
| **Two-pane mode** | A layout in which two content regions are visible simultaneously. The message list occupies one region and the message reader occupies the other. The user selects a message in the list and its content appears in the adjacent pane. |
| **Three-pane mode** | A layout in which three content regions are visible simultaneously: the navigation panel (accounts and folders), the message list, and the message reader. Achieved by pinning the navigation panel open while a two-pane split is active. |
| **Horizontal split (columns)** | A two-pane arrangement where the message list is on the left and the message reader is on the right, separated by a vertical divider. |
| **Vertical split (rows)** | A two-pane arrangement where the message list is on top and the message reader is on the bottom, separated by a horizontal divider. |
| **Navigation panel** | The sidebar that lists accounts, folders, and application-level menu entries. In single-pane and two-pane modes it is shown as a retractable drawer; in three-pane mode it is pinned open. |
| **Pinned navigation** | A state in which the navigation panel is locked open alongside the content panes, effectively adding a third visible column. |
| **Reader pane** | The content region that displays the currently selected conversation or message. In two-pane and three-pane modes it is persistently visible; in single-pane mode it replaces the message list on navigation. |
| **Column width** | The user-configurable ratio that determines how much of the available space is allocated to the message-list pane versus the reader pane in a split layout. |
| **Minimum size threshold** | A per-orientation setting that specifies the smallest window/screen size category at which a split layout will be activated. Below this threshold the application falls back to single-pane mode. |
| **Orientation** | Whether the window is taller than it is wide (portrait) or wider than it is tall (landscape). |

## 5. Personas

- **P1. The laptop user.** Works on a typical 13–15″ display in landscape. Wants the classic two-column layout (message list + reader side by side) and expects it to be the default.
- **P2. The large-monitor power user.** Has a wide display (24″+). Wants all three regions visible at once — navigation panel, message list, and reader — so they never have to open or close the sidebar.
- **P3. The narrow-window user.** Runs the email client in a narrow or half-screen window. Expects a clean single-pane experience where the message list and reader take turns occupying the full window.
- **P4. The tiled-window-manager user.** Frequently resizes the window. Expects the layout to adapt when the window crosses a size threshold, ideally without losing the currently-open message.

## 6. User Stories

### 6.1 Default layout behavior

> **US-1.** As a laptop user opening the application for the first time in a landscape-oriented window, I want to see a two-column layout (message list on the left, reader on the right) by default, so that I can read mail efficiently without configuring anything.

> **US-2.** As any user opening the application in a portrait-oriented or narrow window, I want a single-pane layout by default, so that each view has enough room to be readable.

> **US-3.** As any user on a very small display, I want the application to force single-pane mode even if I have enabled a split layout, so that the panes are never too narrow to be usable.

### 6.2 Choosing a layout mode

> **US-4.** As any user, I want a display setting that lets me enable or disable the two-column (horizontal split) layout for landscape orientation independently of my portrait setting.

> **US-5.** As any user, I want a display setting that lets me choose between single-pane, horizontal split (two columns), or vertical split (two rows) for portrait orientation, so that I can pick the arrangement that best fits my window shape.

> **US-6.** As any user, I want to set a minimum window-size threshold per orientation, so that the split layout only activates when the window is large enough to support it.

### 6.3 Three-pane mode via pinned navigation

> **US-7.** As a large-monitor user, I want to pin the navigation panel open so that it stays visible alongside the message list and reader panes, giving me a three-column layout.

> **US-8.** As any user, I want a visible pin/unpin control on the navigation panel, so that I can switch between two-pane and three-pane modes without visiting the settings.

> **US-9.** As any user, I want the pinned state of the navigation panel to be remembered separately for portrait and landscape orientations, so that I can have the panel pinned in landscape but retractable in portrait.

> **US-10.** As any user, if the window is too narrow to support both an expanded navigation panel and the content panes, I want the application to automatically unpin or collapse the navigation panel rather than rendering unusable, cramped panes.

### 6.4 Pane proportions

> **US-11.** As any user, I want a setting that controls the width (or height) ratio between the message-list pane and the reader pane, so that I can give more room to whichever pane I use more.

> **US-12.** As any user, I want a sensible default ratio (e.g. roughly two-thirds for the message list, one-third for the reader) so that the split is usable without adjustment.

### 6.5 Reader pane visibility when empty

> **US-13.** As any user, when no message is currently selected in a split layout, I want a setting to control whether the reader pane collapses (giving the full width to the message list) or stays visible as an empty placeholder, so that the layout does not jump unexpectedly.

> **US-14.** As any user who prefers the reader pane to stay visible, when I close a conversation I want the reader pane to show an empty state (not the previous message), so that stale content does not linger.

### 6.6 Navigation within the reader pane

> **US-15.** As any user in a split layout, when I select a different message in the message list, I want the reader pane to update to show the newly selected message without affecting the message list.

> **US-16.** As any user in a split layout, when I use "next message" or "previous message" navigation within the reader pane, I want the message-list selection to update to match, so that the two panes stay in sync.

> **US-17.** As any user in a split layout, I want the currently-displayed message to be visually highlighted in the message list, so that I can always see which message I am reading.

### 6.7 Orientation and size changes

> **US-18.** As any user, when the window orientation changes (e.g. I resize from landscape to portrait proportions), I want the layout to switch to my configured mode for the new orientation automatically.

> **US-19.** As any user viewing a conversation in the reader pane when the layout changes from split to single-pane (due to a resize or orientation change), I want the conversation to remain open and visible — not discarded — so that I do not lose my reading context.

> **US-20.** As any user viewing a conversation in single-pane mode when the layout changes to split mode, I want the conversation to move into the reader pane and the message list to reappear in the list pane, so that both become visible without re-navigating.

### 6.8 Navigation panel behavior in split modes

> **US-21.** As any user in two-pane mode (navigation not pinned), I want to be able to open the navigation panel as a temporary overlay (drawer) by swiping or clicking a menu button, and have it close automatically when I select an item or tap outside it.

> **US-22.** As any user in three-pane mode (navigation pinned), I want the navigation panel to push the content area to the right rather than overlaying it, so that all three regions are fully visible without overlap.

### 6.9 Foldable and dual-screen displays

> **US-23.** As a user with a foldable or dual-screen display, I want the two-pane split to align with the physical fold or hinge, so that each pane occupies one screen half cleanly.

> **US-24.** As a user with a foldable display, I want the default pane ratio to be 50/50 and the reader pane to remain visible when empty, so that the layout naturally maps to the two physical halves of the device.

## 7. Functional Requirements

### 7.1 Layout modes

- **FR-1.** The application shall support three layout modes: **single-pane**, **two-pane horizontal split** (columns), and **two-pane vertical split** (rows).
- **FR-2.** A **three-pane** mode shall be achievable by pinning the navigation panel open while any two-pane split is active. The three visible regions shall be: navigation panel, message list, and reader.
- **FR-3.** Each orientation (portrait and landscape) shall have its own independently configurable layout mode.

### 7.2 Defaults

- **FR-4.** The default layout for **landscape** orientation shall be the two-pane horizontal split (columns).
- **FR-5.** The default layout for **portrait** orientation shall be single-pane.
- **FR-6.** On very small displays (below a "normal" size category), the landscape split shall be disabled by default.
- **FR-7.** The default pane ratio shall allocate approximately two-thirds of the available space to the message-list pane and one-third to the reader pane.
- **FR-8.** On foldable or dual-screen displays, the default pane ratio shall be 50/50.

### 7.3 User-configurable settings

- **FR-9.** The application shall provide a display setting to enable or disable the two-column horizontal split for landscape orientation.
- **FR-10.** The application shall provide a display setting to choose among single-pane, horizontal split (two columns), and vertical split (two rows) for portrait orientation. The horizontal and vertical split options shall be mutually exclusive.
- **FR-11.** The application shall provide a per-orientation minimum-size threshold setting. When the window or display size is below the threshold, the application shall fall back to single-pane mode regardless of the split setting.
- **FR-12.** The application shall provide a setting to control the pane ratio (percentage of space allocated to the message-list pane vs. the reader pane), expressed as a value between 10 % and 90 %.
- **FR-13.** The application shall provide a setting to control whether the reader pane collapses (hides) or remains visible as an empty placeholder when no conversation is open.
- **FR-14.** All layout settings shall persist across application restarts.

### 7.4 Navigation panel pinning

- **FR-15.** The navigation panel shall have a user-togglable **pin** control that locks it open alongside the content panes.
- **FR-16.** When pinned, the navigation panel shall not overlay the content area; instead, the content area shall be inset to make room for the navigation panel.
- **FR-17.** When unpinned, the navigation panel shall behave as a retractable drawer that overlays the content area when opened and auto-closes after a selection.
- **FR-18.** The pinned/unpinned state shall be stored independently per orientation.
- **FR-19.** If the window is too narrow for the pinned navigation panel to be at least a usable minimum width alongside the content area, the application shall automatically unpin the panel.
- **FR-20.** The pinned navigation panel width shall be capped so that it never consumes more than half the window width, and shall never be narrower than a defined minimum (enough to display at least icons and short labels).

### 7.5 Pane behavior

- **FR-21.** In single-pane mode, navigating from the message list to a message shall replace the message list with the reader view. The back action shall return to the message list.
- **FR-22.** In any two-pane or three-pane mode, selecting a message in the message list shall display it in the reader pane without replacing the message list.
- **FR-23.** In any split mode, the message-list pane shall visually highlight the currently-displayed message, and shall update the highlight when the user navigates to a different message from within the reader pane.
- **FR-24.** In any split mode, all within-reader navigation (next/previous message, opening a related conversation) shall occur within the reader pane, leaving the message-list pane undisturbed except for updating the selection highlight.
- **FR-25.** A visible separator shall be drawn between adjacent panes (vertical line for horizontal splits, horizontal line for vertical splits).

### 7.6 Foldable and dual-screen display support

- **FR-26.** On foldable or dual-screen displays, the separator between panes shall align with the physical fold or hinge.
- **FR-27.** On foldable or dual-screen displays, the "collapse reader when empty" setting shall default to off (reader pane stays visible), so that each physical half always contains content.

### 7.7 Orientation and resize handling

- **FR-28.** When the window orientation changes or the window is resized across a layout-mode boundary, the application shall switch to the layout mode configured for the new state.
- **FR-29.** When transitioning from a split layout to single-pane mode while a conversation is open in the reader pane, the application shall preserve the open conversation in the single-pane view so that the user's reading context is not lost.
- **FR-30.** When transitioning from single-pane mode to a split layout while a conversation is open, the application shall move the conversation into the reader pane and restore the message list in the list pane.

### 7.8 Interaction with other features

- **FR-31.** The layout mode shall not affect which messages are displayed, how they are sorted, filtered, or searched, or how message actions behave. All message-list and message-reader features shall operate identically regardless of layout mode.
- **FR-32.** The layout mode shall apply uniformly to all message-list contexts: individual folders, the unified inbox, search results, and any other list view.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Switching between layout modes (whether triggered by a setting change, a resize, or an orientation change) shall complete in well under one second, with no visible blank frame or flicker.
- **NFR-2. Context preservation.** The user's scroll position in the message list and the currently-open conversation shall survive layout transitions wherever technically feasible.
- **NFR-3. Accessibility.** All layout-related controls (pin button, settings, separator) shall be keyboard-accessible and carry screen-reader labels. The pane that has focus shall be clearly indicated.
- **NFR-4. Theming.** The pane separator, the empty-reader-pane placeholder, and the pinned-navigation panel shall respect the application-wide theme (light, dark, high-contrast).
- **NFR-5. Minimum pane size.** No pane shall be rendered narrower than a usable minimum. If the window cannot support the configured split at the configured ratio without violating this minimum, the application shall either fall back to single-pane mode or clamp the ratio.
- **NFR-6. Performance.** The split layout shall introduce no measurable increase in memory usage or rendering latency beyond what the individual pane contents already require.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|
| AC-1 | In a landscape-oriented window of normal size or larger, the application defaults to a two-column layout with the message list on the left and the reader on the right. |
| AC-2 | In a portrait-oriented window, the application defaults to a single-pane layout where tapping a message replaces the list with the reader. |
| AC-3 | Enabling the "two columns" setting for portrait orientation causes a horizontal split layout to appear in portrait mode. |
| AC-4 | Enabling the "two rows" setting for portrait orientation causes a vertical split layout (list on top, reader on bottom) to appear in portrait mode. |
| AC-5 | The "two columns" and "two rows" portrait settings are mutually exclusive — enabling one disables the other. |
| AC-6 | Setting the minimum size threshold to "large" for landscape causes the landscape split to be suppressed on normal-sized displays, falling back to single-pane. |
| AC-7 | Adjusting the pane-ratio setting visibly changes the relative sizes of the message-list and reader panes. |
| AC-8 | Pinning the navigation panel in a two-pane layout produces a three-region layout where navigation, message list, and reader are all visible simultaneously without overlap. |
| AC-9 | Unpinning the navigation panel returns it to drawer (overlay) behavior. |
| AC-10 | The navigation panel's pinned state persists independently for portrait and landscape: pinning in landscape does not pin in portrait, and vice versa. |
| AC-11 | On a window too narrow for the pinned navigation panel plus content, the application automatically unpins or collapses the navigation panel. |
| AC-12 | With "collapse reader when empty" enabled, closing a conversation in a split layout causes the reader pane to disappear and the message list to expand to fill the space. |
| AC-13 | With "collapse reader when empty" disabled, closing a conversation leaves the reader pane visible but empty. |
| AC-14 | Selecting a message in the message list in a split layout highlights that message in the list and displays it in the reader pane. Selecting a different message updates both the highlight and the reader. |
| AC-15 | Using "next/previous message" navigation within the reader pane updates the selection highlight in the message list to match. |
| AC-16 | Rotating the window from landscape (split) to portrait (single-pane) while reading a message preserves the open message in the single-pane view. |
| AC-17 | Rotating from portrait (single-pane, message open) to landscape (split) moves the message into the reader pane and restores the message list in the list pane. |
| AC-18 | On a foldable or dual-screen display, the pane separator aligns with the physical hinge, and the default ratio is 50/50. |
| AC-19 | All layout settings (landscape split, portrait split mode, minimum sizes, pane ratio, collapse-reader, navigation pin state) persist across application restarts. |
| AC-20 | The layout mode has no effect on message sorting, filtering, searching, or per-message actions — all behave identically in single-pane, two-pane, and three-pane modes. |

## 10. Open Questions

- **OQ-1. Drag-to-resize panes.** The source application uses a fixed ratio controlled by a slider in settings. Should the desktop version also support dragging the pane separator at runtime to resize panes interactively, or is the settings-based approach sufficient?
- **OQ-2. Independent pane ratios per orientation.** The source application stores a single pane-ratio value. Should the desktop version allow different ratios for portrait and landscape (e.g. 60/40 in landscape but 50/50 in portrait)?
- **OQ-3. Keyboard focus and tab order.** In a three-pane layout, what should the tab order be (navigation → list → reader)? Should a keyboard shortcut cycle focus between panes?
- **OQ-4. Navigation-panel width when pinned.** The source application caps the pinned navigation panel at 300 dp or half the screen width, whichever is smaller. Is this appropriate for desktop, or should the pinned width also be user-configurable?
- **OQ-5. Vertical split in landscape.** The source application only offers horizontal split (columns) for landscape orientation and allows either columns or rows for portrait. Should the desktop version also offer a vertical split (rows) option for landscape, or is that unlikely to be useful on a wide window?
- **OQ-6. Layout persistence scope.** Should layout preferences be global (all windows share the same layout) or per-window, for a potential multi-window desktop scenario?
- **OQ-7. Collapsed vs. icon-only navigation panel.** When the navigation panel is not expanded but is pinned, it shows a narrow column of icons. Should this "collapsed-pinned" state be treated as a distinct layout tier, or is the current behavior (icon strip + full content area) sufficient?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Two-pane is the workhorse, three-pane is opt-in.** The source application defaults to two-pane in landscape (message list + reader) with the navigation panel as a retractable drawer. Three-pane mode requires the user to explicitly pin the navigation panel. This keeps the default simple — most users only need two panes — while giving power users on large displays the density they want.
- **N-2. Portrait and landscape are configured independently.** Each orientation has its own split-mode toggle, its own minimum-size threshold, and its own navigation-pin state. This recognizes that what works in a wide window often does not work in a tall one, and vice versa.
- **N-3. Two split directions serve different window shapes.** The horizontal split (columns) is the natural choice for wide windows. The vertical split (rows) exists for tall, narrow windows in portrait mode where a side-by-side split would make each column too narrow. Offering both directions lets the user pick the arrangement that fits their actual window proportions.
- **N-4. Pane ratio is a single global value.** The source application uses one ratio setting for all split modes (both horizontal and vertical, both orientations). This simplifies configuration at the cost of some flexibility. Whether to split this into per-orientation or per-direction values is an open question (see OQ-2).
- **N-5. Collapsing the reader pane is a deliberate trade-off.** When the reader pane collapses on close, the message list gets the full width, which is visually cleaner. When it stays visible, the layout is stable and the user always knows where the reader will appear. The source application defaults to collapsing (except on foldable devices), matching the expectation that most users prefer a clean list view when they are not actively reading a message.
- **N-6. Fragment re-homing on layout change.** When the layout transitions between single-pane and split modes (due to resize or orientation change), the source application detects that the currently-displayed conversation fragment is in the wrong container and moves it to the correct one. This is the mechanism behind context preservation during layout transitions and is essential for a seamless resize experience.
- **N-7. Foldable/dual-screen alignment is physical, not logical.** On devices with a physical hinge, the separator between panes is widened to match the hinge width and positioned to align with it. The pane ratio defaults to 50/50. This is a hardware-specific adaptation, not a general layout mode — on a non-foldable display the same 50/50 ratio can be achieved via the pane-ratio setting without any special alignment.
