# Epic: Per-Message Actions

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **3.16** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Snooze, hide, pin, flag with color, set importance, edit subject, edit notes, manage IMAP keywords/labels, move, copy, delete, delete attachments."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email client's value is measured largely by how efficiently a user can triage and organize their mail. Beyond simply reading messages, users need a rich set of per-message actions that let them defer, prioritize, annotate, categorize, relocate, and dispose of individual messages — all without leaving their current view.

The Per-Message Actions feature defines the complete repertoire of operations a user can perform on one or more messages. These actions span the full lifecycle of a message after it arrives: temporary deferral (snooze, hide), prioritization (flag with color, set importance), annotation (edit subject, edit notes), categorization (IMAP keywords/labels), organization (move, copy), and disposal (delete, delete attachments). Each action has well-defined behavior regarding server synchronization, undo support, thread scope, and interaction with filters and notifications.

The actions are surfaced across multiple interaction surfaces — toolbar buttons, context menus, keyboard shortcuts, swipe gestures, and bulk-selection mode — and the set of visible actions is user-configurable. This configurability is essential because different users have radically different triage workflows, and a one-size-fits-all toolbar would either overwhelm casual users or under-serve power users.

## 2. Goals

- **G1.** Provide a complete, consistent set of per-message actions available from every message view context: single-message view, conversation/thread view, message list, and the unified inbox.
- **G2.** Let users configure which actions are visible in the toolbar, which are accessible via context menu, and which are bound to swipe gestures or keyboard shortcuts — so that the most-used actions are always one interaction away.
- **G3.** Support bulk application of every action across multiple selected messages in a single operation.
- **G4.** Ensure that actions which modify server state (flag, move, delete, keywords) propagate reliably to the mail server and survive offline/queued scenarios.
- **G5.** Ensure that actions which are local-only (hide, notes, local importance) are clearly communicated as such, so that users do not expect cross-device synchronization.
- **G6.** Provide undo for destructive or hard-to-reverse actions (move, delete) within a configurable grace period.
- **G7.** Respect thread context: where appropriate, let the user choose whether an action applies to a single message or to the entire conversation thread.

## 3. Non-Goals

- **NG1.** This epic does **not** define how messages are fetched, stored, or synchronized; it only constrains how actions interact with whatever the synchronization subsystem provides.
- **NG2.** This epic does **not** define the compose, reply, forward, or redirect workflows — those are covered by the Conversation Actions feature (3.15).
- **NG3.** This epic does **not** define the filter rule engine or automatic classification — those are covered by Rules & Automation (6.x). However, it does note where per-message actions overlap with rule actions (e.g. rules can snooze, flag, set keywords).
- **NG4.** This epic does **not** define notification actions (e.g. "snooze from notification") — those are covered by the Notifications feature (8.x). However, it does note where per-message actions interact with notification state.
- **NG5.** This epic does **not** prescribe the visual layout of action buttons, toolbars, or menus — only the behavioral contracts of each action and the requirement that actions be configurable and accessible from multiple surfaces.

## 4. Glossary

| Term | Definition |
|---|---|
| **Per-message action** | Any operation a user can perform on one or more selected messages that changes the message's state, location, metadata, or visibility. |
| **Snooze** | Temporarily hiding a message until a user-chosen future time, at which point it reappears as if newly arrived. |
| **Hide** | Indefinitely suppressing a message from normal views without moving or deleting it. Reversible. |
| **Pin** | Creating a persistent system-level shortcut (e.g. desktop shortcut, panel launcher) that opens a specific message or conversation directly. |
| **Flag / Star** | Marking a message with the IMAP `\Flagged` flag, optionally with an associated color. Synced to the server. |
| **Flag color** | An RGB color associated with a flagged message, displayed as a tinted star icon. May map to server-side color keywords where supported. |
| **Importance** | A three-level priority indicator (high, normal, low) that can be set by the sender via headers or by the recipient locally. |
| **Keyword** | An IMAP keyword (user-defined flag) attached to a message on the server. Used for categorization, filtering, and labeling. |
| **Label** | A provider-specific categorization tag (e.g. provider-managed labels). Treated separately from standard IMAP keywords. |
| **Note** | A user-written plain-text annotation attached to a message, stored locally only. |
| **Undo grace period** | A configurable window (default 5 seconds) after a destructive action during which the user can reverse it before it is committed to the server. |
| **Soft delete** | Marking a message with the IMAP `\Deleted` flag without expunging it. The message remains on the server and can be recovered. |
| **Hard delete** | Marking a message as deleted and immediately expunging it from the server. Irreversible. |
| **Action surface** | Any UI element through which a user can invoke an action: toolbar button, context menu item, swipe gesture, keyboard shortcut, or bulk-selection button. |

## 5. Personas

- **P1. The inbox-zero triager.** Processes 50–200 messages per day. Relies heavily on swipe gestures and keyboard shortcuts to archive, delete, and snooze messages rapidly. Needs undo for mistakes.
- **P2. The color-coding organizer.** Uses flag colors and keywords to categorize messages visually. Wants quick access to the color picker and keyword manager without deep menu navigation.
- **P3. The note-taker.** Annotates messages with private notes for context that only they need to see. Expects notes to survive application restarts and to be searchable by presence.
- **P4. The multi-account professional.** Manages messages across several accounts. Needs cross-account move/copy and expects actions in the unified inbox to apply to the correct source account transparently.
- **P5. The cautious user.** Wants confirmation dialogs before destructive actions and a generous undo window. May disable swipe gestures entirely to avoid accidental operations.

## 6. User Stories

### 6.1 Snooze

> **US-1.** As a triager, I want to snooze a message until a specific future date and time, so that it disappears from my view now and reappears when I am ready to deal with it.

> **US-2.** As a triager, I want quick-access snooze presets (e.g. "1 hour", "tomorrow morning", "this weekend", "next week", custom date/time), so that I can snooze without manually picking a time for common deferral patterns.

> **US-3.** As any user, when a snoozed message's wake-up time arrives, I want it to reappear as an unread message and optionally trigger a notification (respecting the folder's notification setting), so that it behaves like newly arrived mail.

> **US-4.** As any user, I want to cancel a snooze at any time, making the message immediately visible again, so that I can change my mind.

> **US-5.** As any user, I want snoozed messages to be hidden from normal views by default, but I want a filter toggle that lets me see all currently snoozed messages, so that I can review my deferred queue.

> **US-6.** As any user, when I snooze a message that is part of a conversation, I want the entire thread to be snoozed together, so that I do not see partial conversations.

> **US-7.** As any user, I want an optional setting to automatically flag (star) messages when snoozing and to automatically unflag them when they wake up, so that snoozed messages are visually distinguished.

> **US-8.** As any user, I want an optional setting to automatically mark woken-up messages as high importance, so that they stand out when they reappear.

> **US-9.** As any user, I want an optional setting to cancel a snooze automatically when I manually move a snoozed message to another folder, so that relocation implies I have dealt with it.

### 6.2 Hide

> **US-10.** As any user, I want to hide a message indefinitely from my normal message views without deleting or moving it, so that I can declutter without losing the message.

> **US-11.** As any user, I want to unhide a previously hidden message, restoring it to normal visibility, so that hiding is reversible.

> **US-12.** As any user, I want hidden messages to be excluded from notifications, so that hiding truly silences them.

> **US-13.** As any user, I want hidden messages to share the same filter toggle as snoozed messages, so that I can review everything I have suppressed in one place.

### 6.3 Pin

> **US-14.** As a power user, I want to pin a message or conversation to a persistent system-level shortcut (e.g. a desktop shortcut or panel launcher entry), so that I can jump directly to an important thread without navigating through the application.

> **US-15.** As a power user, when I activate a pinned shortcut, I want the application to open directly to that conversation, so that the shortcut saves me navigation time.

### 6.4 Flag with color

> **US-16.** As any user, I want to flag (star) a message with a single action, toggling the flag on or off, so that I can mark messages for follow-up.

> **US-17.** As an organizer, I want to associate a color with a flagged message using a color picker, so that I can visually categorize flagged messages (e.g. red for urgent, blue for reference, green for personal).

> **US-18.** As any user, I want the flag and its color to be synchronized to the mail server, so that the flag is visible from other mail clients.

> **US-19.** As any user, I want to filter and sort by flagged/unflagged status, so that I can quickly find my flagged messages.

> **US-20.** As an organizer, when the mail server supports named color keywords (e.g. predefined color label sets), I want the application to recognize them and display the corresponding colors, so that color assignments made in other clients are honored.

### 6.5 Set importance

> **US-21.** As any user, I want to set the importance of an existing message to high, normal, or low, so that I can prioritize messages after receiving them.

> **US-22.** As any user, I want the importance level to be displayed as a visual indicator (e.g. an up-arrow for high, a down-arrow for low) in the message list, so that priority is visible at a glance.

> **US-23.** As any user, I want to sort messages by importance, so that high-priority messages appear first.

> **US-24.** As any user, I want the importance I set locally to optionally propagate to the server as IMAP keywords, so that other clients can see my priority assignments.

> **US-25.** As any user, when I receive a message whose sender set a priority via standard headers, I want the application to recognize and display that priority, so that sender intent is honored.

### 6.6 Edit subject

> **US-26.** As a power user, I want to edit the subject line of a received message, so that I can correct mis-labeled threads or add context for my own organization.

> **US-27.** As a power user, I understand that editing the subject creates a replacement message on the server (the original is removed and a new copy with the modified subject is stored), and I want the conversation threading of the replacement to be preserved, so that the edited message remains part of its original thread.

> **US-28.** As a power user, I want subject editing to be available only when the server supports it (IMAP with write access) and the folder is not read-only, so that I am not offered an action that would fail.

### 6.7 Edit notes

> **US-29.** As a note-taker, I want to attach a plain-text note to any message, so that I can record private context or reminders alongside the message.

> **US-30.** As a note-taker, I want my notes to be displayed inline in the message list (below the preview text), so that I can see my annotations without opening each message.

> **US-31.** As a note-taker, I want to assign a color to each note, so that I can visually distinguish different types of annotations.

> **US-32.** As any user, I want to be clearly informed that notes are local-only and not synchronized to the mail server, so that I do not expect them to appear on other devices.

> **US-33.** As any user, when a message exists in multiple folders (e.g. via server-side copy), I want a note added to one copy to appear on all copies within the same account, so that I do not have to annotate duplicates separately.

> **US-34.** As any user, I want to filter the message list to show only messages that have notes, so that I can find my annotated messages.

### 6.8 Manage IMAP keywords / labels

> **US-35.** As a power user, I want to view, add, and remove IMAP keywords on any message via a keyword management dialog, so that I can categorize messages using server-synced tags.

> **US-36.** As a power user, I want to create new custom keywords by typing a name, so that I am not limited to a predefined set.

> **US-37.** As a power user, I want to assign a display color to each keyword, so that keywords are visually distinguishable in the message list.

> **US-38.** As a power user, I want to assign a display alias (friendly name) to each keyword, so that cryptic keyword identifiers can be shown with human-readable labels.

> **US-39.** As a power user, I want keywords to be synchronized bidirectionally with the mail server, so that keywords I set are visible from other clients and keywords set elsewhere appear in my client.

> **US-40.** As a power user, I want to search for messages by keyword, both locally and via server-side search, so that keywords function as a usable tagging system.

> **US-41.** As a power user, when the server hosts provider-specific labels (e.g. provider-managed label systems), I want them displayed alongside IMAP keywords but managed through the provider's own protocol, so that label operations work correctly.

> **US-42.** As any user, I want internal system keywords (e.g. forwarded, delivery-receipt-sent, filtered) to be hidden from the keyword management UI, so that I only see keywords relevant to my workflow.

> **US-43.** As a power user, I want to define a set of global keywords that are available for quick selection on any message in any folder, so that I do not have to recreate my tagging vocabulary per folder.

### 6.9 Move

> **US-44.** As any user, I want to move a message to any folder within the same account via a folder picker, so that I can organize my mail.

> **US-45.** As a multi-account user, I want to move a message to a folder in a different account, with a confirmation step acknowledging the cross-account nature of the operation, so that I can consolidate mail across accounts when needed.

> **US-46.** As any user, after moving a message, I want an undo option available for a configurable grace period, so that I can reverse accidental moves.

> **US-47.** As any user, I want an optional setting to move the entire conversation thread when I move one message, so that threads stay together.

> **US-48.** As any user, I want optional settings to automatically mark a message as read and/or remove its flag when moving it, so that moving doubles as triage.

> **US-49.** As any user, I want the archive action to behave as a move to the account's designated archive folder, inheriting all move behaviors (undo, thread scope, auto-read), so that archiving is just a specialized move.

### 6.10 Copy

> **US-50.** As any user, I want to copy a message to another folder within the same account, leaving the original in place, so that I can have a message in two locations.

> **US-51.** As a multi-account user, I want to copy a message to a folder in a different account, so that I can duplicate important messages across accounts.

### 6.11 Delete

> **US-52.** As any user, I want to delete a message, which by default moves it to the Trash folder, so that deletion is recoverable.

> **US-53.** As any user, I want a "delete permanently" action that immediately and irrecoverably removes a message from the server, so that I can dispose of messages I never want to see again.

> **US-54.** As a cautious user, I want a confirmation dialog before permanent deletion, with an option to suppress future confirmations, so that I am protected from mistakes but not nagged indefinitely.

> **US-55.** As any user, when I delete a message from the Trash folder, I want it to be permanently expunged, since there is no deeper trash.

> **US-56.** As any user, I want an undo option for non-permanent deletions within a configurable grace period, so that I can reverse accidental deletions.

> **US-57.** As any user, I want an optional setting to choose between soft delete (mark as deleted but retain on server) and hard delete (mark and expunge immediately), so that I can match my server's retention policy.

### 6.12 Delete attachments

> **US-58.** As a power user, I want to selectively delete individual attachments from a message, so that I can reclaim storage on the server without losing the message body.

> **US-59.** As a power user, when deleting attachments, I want a dialog listing all attachments with checkboxes so I can choose which to remove, so that I have granular control.

> **US-60.** As a power user, I understand that deleting attachments creates a replacement message on the server (the original is removed and a new copy with placeholder stubs replacing the deleted attachments is stored), and I want the conversation threading to be preserved.

> **US-61.** As a power user, I want attachment deletion to be available only when the server supports it (IMAP with write access), so that I am not offered an action that would fail.

### 6.13 Action surfaces and configurability

> **US-62.** As any user, I want to configure which action buttons appear in the message toolbar, so that my most-used actions are immediately accessible and rarely-used ones are hidden.

> **US-63.** As any user, I want all actions to remain accessible via the context menu regardless of toolbar configuration, so that hiding a button never removes the capability.

> **US-64.** As any user, I want to bind per-message actions to left-swipe and right-swipe gestures in the message list, choosing from the full action set, so that I can triage without opening messages.

> **US-65.** As any user, I want keyboard shortcuts for the most common actions (archive, delete, toggle flag, toggle selection), so that I can triage with the keyboard.

> **US-66.** As any user, I want to select multiple messages and apply any per-message action to all of them in one operation, so that bulk triage is efficient.

> **US-67.** As a cautious user, I want an option to disable swipe gestures entirely, so that I do not accidentally trigger actions.

## 7. Functional Requirements

### 7.1 Snooze

- **FR-1.** The application shall allow the user to snooze one or more selected messages until a user-chosen future date and time.
- **FR-2.** The snooze dialog shall offer the following contextual presets, suppressing options that refer to a time already past:
  - Quick presets: 1 hour, 1 day.
  - Today: this afternoon (12:00), this evening (18:00) — shown only if the current time is before those hours.
  - Tomorrow: morning (08:00), afternoon (12:00).
  - Day after tomorrow: morning (08:00), afternoon (12:00).
  - Weekend: Saturday morning (08:00) — hidden if Saturday is today or tomorrow.
  - Next work week: Monday morning (08:00) — hidden if Monday is today or tomorrow.
  - Next week: current day + 7 days.
  - Custom: full date and time picker.
- **FR-3.** The default snooze duration (used for quick-snooze from notifications and as the initial value in the picker) shall be user-configurable (1–24 hours, default 1 hour).
- **FR-4.** When a message is snoozed, it shall be immediately hidden from normal message views, marked as read, and excluded from notifications.
- **FR-5.** When a message is part of a conversation, snoozing it shall snooze all messages in the thread together.
- **FR-6.** When the wake-up time arrives, the snoozed message shall reappear as an unread message. If the containing folder has notifications enabled, a new notification shall be generated.
- **FR-7.** The application shall enforce a maximum number of concurrently snoozed messages (at least 300). Attempts to exceed this limit shall produce an error.
- **FR-8.** The user shall be able to cancel a snooze at any time via the same menu entry (which toggles to "Reset" or equivalent), immediately restoring visibility.
- **FR-9.** The following optional behaviors shall be independently configurable:
  - Automatically flag (star) a message when it is snoozed.
  - Automatically unflag (unstar) a message when it wakes up.
  - Automatically mark a woken-up message as high importance.
  - Automatically cancel snooze when a snoozed message is moved to another folder.

### 7.2 Hide

- **FR-10.** The application shall allow the user to hide a message indefinitely from normal message views without moving or deleting it. The message shall remain in its original folder.
- **FR-11.** Hide shall be a toggle: invoking it on a hidden message shall unhide it, restoring normal visibility.
- **FR-12.** Hidden messages shall be excluded from notifications.
- **FR-13.** Hide state shall be local-only; it shall not be synchronized to the mail server.
- **FR-14.** Hidden and snoozed messages shall share a single filter toggle that, when enabled, reveals all suppressed messages in the current view.

### 7.3 Pin

- **FR-15.** The application shall allow the user to create a persistent system-level shortcut (desktop shortcut, panel launcher entry, or equivalent) for a specific message or conversation thread.
- **FR-16.** The shortcut shall include a user-editable label (defaulting to the message subject) and an icon derived from the sender's avatar or a generic mail icon.
- **FR-17.** Activating the shortcut shall open the application directly to the pinned conversation.

### 7.4 Flag with color

- **FR-18.** The application shall allow the user to toggle the flagged (starred) state of a message with a single action.
- **FR-19.** The application shall allow the user to associate an arbitrary RGB color with a flagged message via a color picker.
- **FR-20.** The flag state shall be synchronized bidirectionally with the mail server using the standard IMAP `\Flagged` flag.
- **FR-21.** The flag color shall be stored locally and displayed as a tinted star icon in the message list and message view.
- **FR-22.** The application shall recognize server-side color keywords from known conventions (e.g. Open-Xchange `$cl_1`–`$cl_10` color set, provider label keywords `$label1`–`$label5`) and display their predefined colors:
  - Open-Xchange: red, blue, green, grey, purple, light green, orange, pink, light blue, yellow.
  - Provider labels: red (Important), orange (Work), green (Personal), blue (To do), purple (Later).
- **FR-23.** Flagging a message shall not affect its read/unread state, its importance, or its folder membership.

### 7.5 Set importance

- **FR-24.** The application shall allow the user to set the importance of an existing message to one of three levels: high, normal, or low.
- **FR-25.** High importance shall be indicated by an upward-arrow icon in warning color; low importance by a downward-arrow icon in subdued color; normal importance shall have no indicator.
- **FR-26.** The application shall recognize incoming importance from standard message headers (`Priority`, `X-Priority`, `X-MSMail-Priority`) and display it accordingly:
  - Header values 1–2 → high; 3 → normal; 4–5 → low.
- **FR-27.** Locally set importance shall be independent of the sender's header-based priority. Both may coexist; the local setting governs display.
- **FR-28.** The user shall be able to sort messages by importance, with high-importance messages appearing first.
- **FR-29.** An optional setting shall allow locally set importance to be propagated to the server as IMAP keywords (`$HighImportance`, `$LowImportance`), and shall recognize those keywords when synchronizing from the server.

### 7.6 Edit subject

- **FR-30.** The application shall allow the user to edit the subject line of a message, provided the account uses IMAP and the folder is not read-only.
- **FR-31.** Editing the subject shall create a replacement message on the server with the new subject, preserving the conversation thread linkage (via references to the original message identifiers). The original message shall be removed.
- **FR-32.** The edited message shall be re-indexed for search.
- **FR-33.** Subject editing shall be gated behind an "experimental features" preference, as it is a destructive server operation.

### 7.7 Edit notes

- **FR-34.** The application shall allow the user to attach a plain-text, multi-line note to any message.
- **FR-35.** Notes shall be stored locally only. A clear disclaimer shall be shown in the editing dialog stating that notes are not synchronized to the mail server.
- **FR-36.** Notes shall be displayed inline in the message list, below the preview text, in bold, using the note's assigned color (if any).
- **FR-37.** Each note shall have an optional associated color, selectable via a color picker, with a default color preference remembered across notes.
- **FR-38.** When a message exists as multiple copies (same message identifier) across folders within the same account, a note added to one copy shall be applied to all copies.
- **FR-39.** The user shall be able to filter the message list to show only messages that have notes.
- **FR-40.** The user shall be able to clear a note and its color via a reset action in the editing dialog.

### 7.8 Manage IMAP keywords / labels

- **FR-41.** The application shall provide a keyword management dialog that lists all keywords currently applied to a selected message (or the union of keywords across multiple selected messages).
- **FR-42.** Each keyword shall be presented with a checkbox (checked = applied, unchecked = not applied). For multi-message selections where a keyword is present on some but not all messages, the checkbox shall show an indeterminate state.
- **FR-43.** The user shall be able to add a new custom keyword by name. The application shall sanitize the input to conform to IMAP keyword format restrictions (removing invalid characters).
- **FR-44.** The application shall enforce a maximum of 32 keywords per message.
- **FR-45.** The user shall be able to assign a display color and a display alias (friendly name) to any keyword. These display attributes shall be stored locally and shall not affect the keyword's server-side identity.
- **FR-46.** Keyword add/remove operations shall be synchronized bidirectionally with the mail server, provided the server supports user-defined flags.
- **FR-47.** For accounts that use a provider-specific label system, label operations shall use the provider's own protocol rather than standard IMAP keywords.
- **FR-48.** System-internal keywords (e.g. forwarded, delivery-receipt-sent, filtered, classified) shall be hidden from the keyword management UI. Only user-relevant keywords shall be displayed.
- **FR-49.** The user shall be able to define a set of global keywords (available across all folders and accounts) for quick selection.
- **FR-50.** The user shall be able to search for messages by keyword, both locally and via server-side IMAP search.

### 7.9 Move

- **FR-51.** The application shall allow the user to move one or more messages to any folder within the same account, selected via a folder picker dialog.
- **FR-52.** The application shall allow cross-account moves. A cross-account move shall fetch the raw message from the source and add it to the target account. A confirmation dialog shall warn the user that this is a cross-account operation.
- **FR-53.** After a move, an undo action shall be available for a configurable grace period (default 5 seconds). If the user does not undo within this period, the operation shall be committed to the server.
- **FR-54.** The following optional settings shall apply to moves:
  - Move entire conversation thread (all messages in the thread, not just the selected one).
  - Include sent-folder messages when moving a thread.
  - Automatically mark moved messages as read.
  - Automatically remove the flag from moved messages.
  - Automatically reset importance on moved messages (except moves to trash/archive/junk).
  - Automatically cancel snooze on moved messages.
- **FR-55.** The archive action shall behave as a move to the account's designated Archive folder, inheriting all move behaviors.
- **FR-56.** The "move to junk" action shall behave as a move to the account's Junk/Spam folder, with an optional setting to automatically block the sender.

### 7.10 Copy

- **FR-57.** The application shall allow the user to copy one or more messages to any folder within the same account, leaving the original in place.
- **FR-58.** The application shall allow cross-account copies, using the same fetch-and-add mechanism as cross-account moves but without removing the original.

### 7.11 Delete

- **FR-59.** The application shall support two deletion modes, selectable via a global preference:
  - **Soft delete:** marks the message with the IMAP `\Deleted` flag but does not expunge. The message remains on the server and can be recovered.
  - **Hard delete (default):** marks the message as deleted and immediately expunges it from the server. Irreversible.
- **FR-60.** A "delete to trash" action shall move the message to the account's Trash folder rather than expunging it. This is the default "delete" behavior in most contexts.
- **FR-61.** A "delete permanently" action shall hard-delete the message regardless of the soft/hard preference.
- **FR-62.** Deleting a message that is already in the Trash folder shall permanently expunge it.
- **FR-63.** A confirmation dialog shall be shown before permanent deletion. The user shall be able to suppress future confirmations via a "don't ask again" option.
- **FR-64.** An undo action shall be available for non-permanent deletions (move-to-trash and soft delete) within the configured grace period.
- **FR-65.** For POP3 accounts, the application shall offer a "leave on server" option: deleted messages can be marked locally but retained on the server, or fully removed from the server, as the user configures.

### 7.12 Delete attachments

- **FR-66.** The application shall allow the user to selectively delete individual attachments from a message, provided the account uses IMAP and the folder is not read-only.
- **FR-67.** A dialog shall list all attachments with checkboxes (all pre-selected by default). The user may uncheck attachments they wish to keep.
- **FR-68.** Deleting attachments shall create a replacement message on the server with the removed attachments replaced by empty placeholder stubs retaining the original filename (prefixed to indicate deletion) and content type. The original message shall be removed.
- **FR-69.** Conversation threading of the replacement message shall be preserved.
- **FR-70.** This operation is destructive and not undoable once committed to the server.

### 7.13 Action surfaces and configurability

- **FR-71.** The application shall provide the following action surfaces for per-message actions:
  - **Toolbar buttons:** a configurable set of action buttons displayed in the message view, with an option for a second row of extra buttons.
  - **Context menu:** a grouped overflow menu accessible from every message, containing all actions regardless of toolbar configuration.
  - **Swipe gestures:** left-swipe and right-swipe on message list items, each independently bindable to any action from the full set (archive, trash, delete permanently, junk, move-to, snooze, hide, flag, mark read/unread, set importance, reply, forward, text-to-speech, summarize, or "ask me").
  - **Keyboard shortcuts:** at minimum, shortcuts for archive, delete/trash, toggle flag/star, and toggle selection.
  - **Bulk-selection buttons:** a floating action panel shown when multiple messages are selected, offering all applicable actions.
- **FR-72.** Each toolbar button shall be independently toggleable (show/hide) via a configuration dialog. The configuration shall persist across restarts.
- **FR-73.** Swipe actions shall be configurable per account, with sensible defaults (left-swipe → trash, right-swipe → archive).
- **FR-74.** The user shall be able to adjust swipe sensitivity or disable swipe gestures entirely.
- **FR-75.** Actions that are inapplicable in the current context shall be hidden or disabled. For example:
  - IMAP-only actions (edit subject, delete attachments, copy, move) shall be hidden for POP3 accounts.
  - Write-dependent actions shall be hidden for read-only folders.
  - Experimental actions shall be hidden unless the experimental-features preference is enabled.

### 7.14 Cross-cutting concerns

- **FR-76.** Every per-message action available in a single-folder view shall also be available in the Unified Inbox, with the action applied to the message in its original folder and account.
- **FR-77.** Every per-message action shall be available in bulk-selection mode with the same semantics as the single-message version.
- **FR-78.** Actions that modify server state shall be queued locally and replayed when the connection is available, supporting full offline operation.
- **FR-79.** The context menu shall group actions into logical categories: operations (seen, snooze, hide, flag, importance, subject, move, copy, delete), properties (keywords, notes), sharing (share, pin, print), and advanced (headers, raw view, diagnostics).

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Toggle actions (flag, read/unread, hide) shall reflect in the UI within 100 milliseconds. Actions requiring a dialog (snooze, move, keywords) shall open the dialog within 300 milliseconds.
- **NFR-2. Offline resilience.** All actions shall be queueable offline. The queue shall survive application restart and replay in order when connectivity returns.
- **NFR-3. Consistency.** An action performed on a message in the Unified Inbox and the same action performed on the same message in its original folder shall produce identical results. No action shall behave differently based on the view from which it was invoked.
- **NFR-4. Undo reliability.** The undo grace period shall be guaranteed: an action shall not be committed to the server until the grace period expires or the user explicitly confirms. If the application is terminated during the grace period, the action shall be committed on next launch.
- **NFR-5. Configurability persistence.** All user choices about action surface layout (toolbar buttons, swipe bindings, sensitivity) shall persist across application restarts.
- **NFR-6. Accessibility.** All action surfaces shall be reachable via keyboard and compatible with screen readers. Toolbar buttons shall have descriptive labels. Swipe gestures shall have keyboard-accessible equivalents.
- **NFR-7. Scale.** Bulk actions shall handle at least 500 selected messages without timeout or UI freeze. Progress indication is recommended for operations on more than 50 messages.
- **NFR-8. Privacy.** Local-only state (notes, hide, local importance) shall never be transmitted to the mail server or any third party.

## 10. Open Questions

- **OQ-1. Pin as desktop concept.** The source application implements "pin" as creating a home-screen launcher shortcut, which is a mobile-specific concept. The desktop equivalent (desktop shortcut, panel launcher entry, or bookmark) needs design work to determine the most natural mapping. Whether pinning should additionally maintain an in-application "pinned messages" list (virtual folder) is an open design question.
- **OQ-2. Flag color server sync granularity.** The source application stores flag colors locally and recognizes server-side color keywords (Open-Xchange, provider labels), but does not write arbitrary RGB colors back as keywords. Should the desktop application attempt to map user-chosen colors to the nearest known server keyword, or should it remain a local-only enhancement? The answer may depend on the server's keyword support.
- **OQ-3. Subject editing safety.** Subject editing is gated behind an "experimental" flag in the source application, implying the developers considered it risky. The operation replaces the message on the server, which can confuse other clients. Should the desktop application show an explicit warning dialog explaining the server-side implications, or is the experimental gate sufficient?
- **OQ-4. Note searchability.** The source application allows filtering by note presence ("has notes") but does not support full-text search within note content (the indexing is disabled). Should the desktop application index note content for search, or is presence-filtering sufficient?
- **OQ-5. Snooze alarm limit.** The source application enforces a maximum of 300 concurrently snoozed messages due to a platform alarm-scheduling constraint. The desktop platform may not have this constraint. Should the limit be removed, raised, or retained as a safeguard?
- **OQ-6. Cross-account move semantics.** Cross-account moves in the source application fetch the raw message and re-add it to the target account as a new message (new server-side identifier). Should the desktop application warn the user that the moved message is effectively a copy-then-delete with a new server identity, or is this an acceptable implementation detail to hide?
- **OQ-7. Hide vs. archive mental model.** Hide and archive are conceptually similar (both remove a message from the active view) but mechanically different (hide is local and non-destructive; archive moves to a server folder). User research may be needed to determine whether the distinction is clear enough or whether "hide" should be renamed or repositioned.
- **OQ-8. Attachment deletion undo.** Attachment deletion is destructive and not undoable in the source application. Should the desktop application implement an undo grace period (holding the original locally before committing the server replacement), or is a confirmation dialog sufficient?
- **OQ-9. Keyword limit enforcement.** The source application enforces a maximum of 32 keywords per message. This may be a server-imposed limit (some IMAP servers cap permanent flags). Should the desktop application detect the server's actual limit and use that, or enforce a fixed cap?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Snooze is thread-scoped.** The source application snoozes all messages in a conversation together, not just the selected message. This prevents the confusing situation where half a thread is visible and half is snoozed. The wake-up alarm is set only on the primary snoozed message; companion messages are marked with an indefinite-hide sentinel that clears when the primary wakes.

- **N-2. Hide reuses the snooze infrastructure.** Hidden messages are internally represented as snoozed with an indefinite wake-up time (effectively "never"). This means they share the same filter toggle and the same query exclusion logic. The advantage is simplicity; the disadvantage is that "show snoozed" also shows hidden messages, which may confuse users who think of these as distinct concepts.

- **N-3. Pin is not star.** The source application carefully separates "pin" (create a system shortcut to a conversation) from "flag/star" (mark a message with IMAP `\Flagged`). These serve different purposes: pin is about navigation convenience; flag is about message metadata. They share no state and do not interact.

- **N-4. Flag color is local-enriched, server-recognized.** The application stores flag colors locally (any RGB value the user picks) and recognizes known server keywords (Open-Xchange colors, provider labels) when syncing from the server. It does not write arbitrary colors back to the server. This avoids polluting the server with non-standard keywords while still honoring colors set by other clients that use known conventions.

- **N-5. Two importance systems coexist.** The source application maintains two separate importance fields: one derived from the sender's headers (recognized on receipt and used for outgoing messages during compose) and one set locally by the recipient. The local importance governs the display indicator and sort order. This means a user can override the sender's priority without modifying the original message.

- **N-6. Subject editing is deliberately gated.** The operation replaces the message on the server (IMAP APPEND of a modified copy + deletion of the original). This is inherently risky: it changes the message's server-side identifier, it can confuse other clients, and it is irreversible. Gating it behind an "experimental" preference ensures that only users who explicitly opt in encounter this behavior.

- **N-7. Notes are deliberately local-only.** The source application stores notes in the local database and explicitly disclaims server synchronization. This is a conscious trade-off: simplicity and privacy (notes never leave the device) over cross-device availability. The application compensates by applying notes to all local copies of the same message (same message identifier across folders).

- **N-8. Keyword display is curated.** The source application maintains a blacklist of internal system keywords that are hidden from the user-facing keyword management dialog. This prevents users from accidentally removing keywords that the application or server relies on for internal bookkeeping (e.g. forwarded-tracking, delivery-receipt-sent, rule-filtered markers).

- **N-9. Delete behavior is preference-driven.** The choice between soft and hard delete is a global preference, not a per-action decision. This simplifies the mental model: "delete" always means the same thing, and the user configures once how aggressive that is. The "delete permanently" action bypasses this preference for explicit, intentional permanent removal.

- **N-10. Attachment deletion creates server-side stubs.** Rather than simply removing attachment MIME parts (which would change the message's structure unpredictably), the source application replaces each deleted attachment with an empty placeholder that preserves the original filename (prefixed) and content type. This ensures that the message structure remains valid and that recipients of the message (if forwarded) can see that attachments were intentionally removed.

- **N-11. Undo is time-boxed, not modal.** The source application uses a non-blocking undo mechanism: after a destructive action, a transient notification offers an undo button for a configurable number of seconds. If the user does nothing, the action commits. This is less disruptive than a modal confirmation for every action, while still providing a safety net for the most common mistake (accidental swipe or tap).
