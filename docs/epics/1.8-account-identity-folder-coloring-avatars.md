# Epic: Account / Identity / Folder Coloring & Avatars

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.8** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"User-defined colors and image avatars per account, identity and folder, used throughout the UI."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[SHOULD]` — important, planned for v1 but acceptable to slip to v1.x if pressed.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Users who manage multiple email accounts, identities, and folder structures need a fast, glanceable way to tell at a glance which account or identity a message belongs to, and to visually distinguish folders that share similar names across accounts. Color-coding and avatar imagery are the two most effective channels for this: color provides an instant, pre-attentive cue in dense lists, while avatars give accounts and contacts a recognizable face.

This feature gives every account, every identity, and every folder an optional user-chosen color, and gives every account an optional user-chosen image avatar. These visual markers are then propagated consistently throughout the application — message lists, navigation pane, conversation view, action bars, widgets, and notifications — so that the user's color and avatar choices form a coherent, application-wide visual language.

The feature also encompasses contact-level avatar resolution: for each message sender, the application assembles an avatar from a prioritized cascade of sources (address-book photo, domain favicon, generated icon, etc.), providing a visual identity even for senders the user has never explicitly configured.

## 2. Goals

- **G1.** Allow the user to assign a color to any account, identity, or folder, and to clear that color at any time.
- **G2.** Allow the user to assign an image avatar to any account, and to clear that avatar at any time.
- **G3.** Propagate assigned colors consistently across every surface where the owning entity appears: message lists, navigation pane, conversation action bar, message headers, widgets, and notifications.
- **G4.** Provide a clear, deterministic precedence order when multiple colors could apply to the same message (identity, folder, account), so the user always knows which color they are seeing and why.
- **G5.** For message senders, resolve an avatar automatically from a cascade of sources with user-controllable toggles, so that most messages display a meaningful sender image without manual configuration.
- **G6.** Offer display-level controls over how colors are rendered (stripe vs. dot, stripe width, saturation and brightness of generated icons) so the user can tune visual density to their preference.

## 3. Non-Goals

- **NG1.** This epic does **not** define application-wide theming (light / dark / accent color). Theme selection is a separate concern; this epic addresses per-entity colors that overlay the theme.
- **NG2.** This epic does **not** define colored stars / flag colors on individual messages. Flag coloring is a per-message property and is covered under message actions.
- **NG3.** This epic does **not** prescribe how colors or avatars are persisted or queried — only that they are durable, per-entity, and observable at runtime.
- **NG4.** Contact-level colors, aliases, and notes (feature 12.4 in the high-level feature list) are a separate contact-management concern. This epic covers only the avatar-resolution cascade for sender display, not the full contact record.
- **NG5.** BIMI logo display (feature 3.14) is documented in its own right. This epic references BIMI only as one source in the avatar cascade.

## 4. Glossary

| Term | Definition |
|---|---|
| **Account** | A configured connection to a single mail provider, with its own credentials, server settings, folder tree, color, and avatar. |
| **Identity** | A "From" persona within an account — display name, email address, signature, and optionally its own color. An account may have many identities. |
| **Folder** | A container of messages within an account, which may carry its own color independently of the account color. |
| **Entity color** | A user-assigned color on an account, identity, or folder. Nullable — absence means "no color assigned". |
| **Color stripe** | A narrow vertical bar rendered at the leading edge of a message row, filled with the resolved color for that message. |
| **Color dot** | A small circular indicator rendered near a message row, filled with the resolved color for that message. An alternative to the stripe. |
| **Account avatar** | A user-chosen image associated with an account, displayed alongside the account name in the account list and (optionally) in the navigation pane. |
| **Sender avatar** | An image displayed alongside a message in a list or header, representing the sender. Resolved automatically from a cascade of sources. |
| **Avatar cascade** | The ordered list of sources the application consults to find a sender avatar. The first source that produces an image wins. |
| **Generated icon** | A synthetic avatar created from the sender's name or address — either a letter on a colored background or a geometric pattern (identicon). |
| **Identicon** | A symmetric, deterministic geometric pattern derived from a hash of the sender's address, used as a fallback avatar. |
| **Favicon** | The small icon published by a domain's website, used as a fallback sender avatar when no contact photo or third-party avatar is available. |

## 5. Personas

- **P1. The multi-account triager.** Has 3–5 accounts (work, personal, side-project). Relies on color stripes to instantly identify which account a message belongs to without reading the header. Assigns a distinct color to each account and expects those colors to appear everywhere.
- **P2. The multi-identity user.** Uses several identities within a single account (personal alias, team alias, role-based alias). Assigns a different color to each identity so they can tell which identity received a message.
- **P3. The folder organizer.** Has a deep folder hierarchy and wants key folders (e.g. "Urgent", "Clients") to pop visually via folder color in the navigation pane and in message lists.
- **P4. The visual reader.** Prefers a rich, image-heavy message list where every sender has a photo or icon. Enables multiple avatar sources and tunes saturation/brightness to match their theme.
- **P5. The privacy-conscious user.** Wants sender avatars from local sources only (address book, generated icons) and disables all network-fetched avatar sources (Gravatar, Libravatar, favicons, third-party icon services).

## 6. User Stories

### 6.1 Assigning colors

> **US-1.** As any user, I want to assign a color to an account from the account settings screen, so that all messages and folders belonging to that account carry a visual marker.

> **US-2.** As any user, I want to assign a color to an identity from the identity settings screen, so that messages matched to that identity are visually distinct from other identities in the same account.

> **US-3.** As any user, I want to assign a color to a folder from the folder settings screen, so that the folder stands out in the navigation pane and in message lists.

> **US-4.** As any user, I want to clear (remove) a previously assigned color from any account, identity, or folder, restoring the default uncolored appearance.

> **US-5.** As any user, I want to edit an account's color directly from a shortcut in the folder list (without navigating to the full account settings), so that quick color changes are convenient.

### 6.2 Color precedence

> **US-6.** As a multi-identity user, when a message is matched to an identity that has its own color, I want the identity color to take precedence over the folder color and the account color for the message-list color indicator, so that I see the most specific color.

> **US-7.** As a folder organizer, when a folder has its own color and the message is not matched to a colored identity, I want the folder color to take precedence over the account color for the message-list color indicator.

> **US-8.** As a multi-account triager, when neither the matched identity nor the folder has a color, I want the account color to be used as the fallback for the message-list color indicator.

### 6.3 Color display modes

> **US-9.** As any user, I want to choose how account/identity/folder colors appear in the message list — as a vertical stripe, as a small dot, or not at all — so that I can control the visual weight of the color indicator.

> **US-10.** As any user, when I choose the stripe display mode, I want to select the stripe width (narrow, medium, or wide), so that I can balance visibility against space usage.

> **US-11.** As any user, I want the chosen color display mode and stripe width to apply consistently in both the main message list and in any message-list widget.

### 6.4 Color propagation across the UI

> **US-12.** As a multi-account triager, I want the account or folder color to tint the folder icon in the navigation pane, so that the navigation pane reflects my color scheme.

> **US-13.** As any user, when viewing a conversation, I want the account name and folder name in the message header to be rendered in their respective colors, so that I can identify the source at a glance.

> **US-14.** As any user, I want the option to use the account color as the background color of the conversation action bar, so that the entire reading experience is tinted to the account.

> **US-15.** As any user, when the action bar background is set to a custom color, I want the icons and text on the action bar to automatically adjust (light on dark, dark on light) so they remain legible.

> **US-16.** As a multi-identity user, I want the option to use the identity color as the background of the bottom action bar when composing, so that I always know which identity I am writing as.

> **US-17.** As any user, when I move a message to another account via a context menu, I want the account names in the menu to be rendered in their respective colors, so that I can visually distinguish the target accounts.

### 6.5 Default folder colors

> **US-18.** As any user, I expect the Trash and Spam/Junk folders to display a warning-level color by default (without me manually assigning one), so that dangerous-destination folders are visually distinct out of the box.

> **US-19.** As any user, I want to be able to override a folder's default color with a custom one, or clear it back to the default.

### 6.6 Account avatars

> **US-20.** As any user, I want to set a custom image as the avatar for an account, selected from any image file on my system, so that the account has a recognizable face in the account list.

> **US-21.** As any user, I want to clear a previously set account avatar, reverting to the default (no image or a generic placeholder).

> **US-22.** As any user, I want account avatars to appear in the account list view, scaled to a consistent size, so that accounts are visually distinguishable at a glance.

### 6.7 Sender avatar cascade

> **US-23.** As a visual reader, I want each message in the message list to display a sender avatar resolved automatically from a prioritized set of sources, so that most messages have a meaningful image without manual configuration.

> **US-24.** As any user, I want to control which avatar sources are active via individual toggles in the display settings, so that I can enable or disable sources according to my privacy and aesthetic preferences.

> **US-25.** As a privacy-conscious user, I want all network-fetched avatar sources (Gravatar, Libravatar, favicons, third-party icon services) to be independently togglable and all disabled by default, so that no avatar-related network traffic occurs unless I opt in.

> **US-26.** As any user, I want the avatar cascade to try sources in a well-defined priority order, and to stop at the first source that produces an image, so that the display is deterministic and efficient.

> **US-27.** As any user, I want the option to display avatars as round (circular-cropped) or square, so that the visual style matches my preference.

### 6.8 Avatar source details

> **US-28.** As any user with a populated address book, I want the sender's address-book contact photo to be the highest-priority avatar source, so that people I know always show their real photo.

> **US-29.** As a visual reader, I want the application to fetch the sender's domain favicon as a fallback avatar when no contact photo exists, so that messages from companies and services show a recognizable brand icon.

> **US-30.** As a visual reader, I want the option to enable Gravatar and/or Libravatar as avatar sources, so that senders who have registered a Gravatar or Libravatar show their chosen image.

> **US-31.** As any user, I want the option to enable a third-party icon service (e.g. a privacy-respecting icon lookup) as an additional favicon source, understanding that this sends the sender's domain to the service.

> **US-32.** As any user, when no external avatar source produces an image, I want the application to generate a synthetic icon — either a letter-on-colored-background or a geometric identicon — so that every message has some visual marker.

> **US-33.** As any user, I want to choose between generated letter icons and geometric identicons as the fallback style.

### 6.9 Generated icon tuning

> **US-34.** As a visual reader, I want to adjust the color saturation of generated icons via a slider, so that the generated colors harmonize with my chosen theme.

> **US-35.** As a visual reader, I want to adjust the brightness of generated icons via a slider, so that they are neither washed-out in a light theme nor invisible in a dark theme.

> **US-36.** As any user, I want the letter color on generated icons to automatically contrast against the background color (light letter on dark background, dark letter on light background), with a configurable luminance threshold, so that letters are always legible.

### 6.10 Favicon refinements

> **US-37.** As any user, I want the favicon fetcher to try the sender's exact domain first, then fall back to parent domains, so that subdomains with their own favicon are respected.

> **US-38.** As any user, I want the option to scan only the initial portion of a web page for favicon references (partial scan), so that favicon resolution does not consume excessive bandwidth.

> **US-39.** As any user, I want the option to scan web-app manifests for higher-quality icons, so that I get the best available favicon.

> **US-40.** As a security-conscious user, I want the option to require successful DMARC verification before displaying any externally fetched image (favicon, Gravatar, etc.) for a sender, so that spoofed senders do not get a legitimate-looking avatar.

### 6.11 Avatar caching

> **US-41.** As any user, I want fetched and generated avatars to be cached locally, so that the same avatar does not need to be re-fetched or re-generated on every view.

> **US-42.** As any user, I want stale avatar cache entries to be cleaned up automatically over time, so that the cache does not grow unboundedly and outdated images are eventually refreshed.

### 6.12 Widget and external surface support

> **US-43.** As any user, I want message-list widgets to support the same color stripe/dot display as the main message list, so that my color scheme carries over to external surfaces.

> **US-44.** As any user, I want message-list widgets to optionally display sender avatars, so that the widget provides the same visual richness as the main list.

## 7. Functional Requirements

### 7.1 Entity color assignment

- **FR-1.** Each account, identity, and folder shall have an optional **color** property, expressed as a full-spectrum color value (not limited to a fixed palette). The property shall be nullable — null means "no color assigned".
- **FR-2.** The user shall be able to set or clear the color of an account from the account settings screen.
- **FR-3.** The user shall be able to set or clear the color of an identity from the identity settings screen.
- **FR-4.** The user shall be able to set or clear the color of a folder from the folder settings screen.
- **FR-5.** A shortcut to edit the account color shall be available from the folder list context menu, without navigating to the full account settings.
- **FR-6.** Color assignment shall be persisted durably and shall survive application restart, re-login, and server folder re-discovery.
- **FR-7.** The color picker shall offer a color wheel (or equivalent full-spectrum selector), a lightness/brightness slider, and a text field for entering exact color values.
- **FR-8.** The color picker shall include a "reset" action that clears the color, returning the entity to the uncolored default.

### 7.2 Color precedence

- **FR-9.** When displaying a color indicator for a message in the message list, the application shall resolve the color using the following precedence (highest to lowest):
  1. The color of the matched identity (if the message was received by an identity that has a color assigned).
  2. The color of the folder containing the message (if the folder has a color assigned).
  3. The color of the account owning the message (if the account has a color assigned).
  4. The application's default separator/background color (if none of the above are assigned).
- **FR-10.** This precedence shall be documented to the user (e.g. a hint on the identity color setting that identity colors take precedence over folder and account colors).

### 7.3 Color display modes in the message list

- **FR-11.** The application shall offer three display modes for per-message color indicators in the message list:
  - **None:** no color indicator displayed.
  - **Stripe:** a vertical color bar at the leading edge of the message row.
  - **Dot:** a small colored circle/badge near the message row.
- **FR-12.** When the stripe mode is selected, the user shall be able to choose from at least three stripe widths (e.g. narrow, medium, wide).
- **FR-13.** The selected display mode and stripe width shall be persisted and shall apply to both the main message list and any message-list widget.

### 7.4 Color propagation across surfaces

- **FR-14.** In the navigation pane, the icon for each folder or account entry shall be tinted with the entity's resolved color (folder color if set, otherwise account color), when a color is assigned.
- **FR-15.** In the message header area of a conversation view, the account name shall be rendered in the account's color, and the folder name shall be rendered in the folder's color (if colors are assigned).
- **FR-16.** The application shall provide an option to use the account color as the background color of the conversation action bar. When enabled, icon and text colors on the bar shall automatically adjust for legibility (light content on a dark background, dark content on a light background).
- **FR-17.** The application shall provide an option to use the identity color as the background color of the compose action bar. The same automatic legibility adjustment shall apply.
- **FR-18.** When presenting account names in context menus (e.g. "move to account" lists), each account name shall be rendered in its assigned color.

### 7.5 Default folder colors

- **FR-19.** Trash and Spam/Junk folders shall display a warning-level color (typically red or the theme's error color) by default, without requiring user configuration.
- **FR-20.** The user shall be able to override a folder's default color with a custom color, or clear it back to the default.
- **FR-21.** All other folder types shall have no default color (null).

### 7.6 Account avatars

- **FR-22.** Each account shall have an optional **avatar** property, accepting a user-chosen image file.
- **FR-23.** The user shall be able to set the avatar from the account settings screen by selecting any image file from the local filesystem.
- **FR-24.** The user shall be able to clear the avatar (e.g. via a long-press or a dedicated clear action), reverting to no image.
- **FR-25.** Account avatars shall be displayed in the account list, scaled to a consistent standard size.
- **FR-26.** The application shall maintain persistent read access to the selected image file across restarts.

### 7.7 Sender avatar cascade

- **FR-27.** For each message displayed in a list or header, the application shall resolve a sender avatar by consulting sources in the following priority order, stopping at the first source that produces a valid image:
  1. Address-book contact photo (local).
  2. BIMI logo (if the BIMI feature is enabled).
  3. Gravatar (if enabled by the user).
  4. Libravatar (if enabled by the user).
  5. Domain favicon (if enabled by the user).
  6. Third-party icon service (if enabled by the user).
  7. Custom favicon service URI (if configured by the user).
  8. Generated icon (letter icon or identicon, depending on user preference).
- **FR-28.** Each source in the cascade shall be independently togglable via a display setting.
- **FR-29.** All network-fetched sources (items 2–7 above) shall be disabled by default; only address-book photos and generated icons shall be active out of the box.
- **FR-30.** The user shall be able to choose between two generated-icon styles: letter-on-colored-background, or geometric identicon.
- **FR-31.** The user shall be able to choose between round (circular-cropped) and square avatar shapes.

### 7.8 Generated icon tuning

- **FR-32.** The application shall provide a saturation slider (0–100%) that controls the color saturation of generated icons. Default: 100%.
- **FR-33.** The application shall provide a brightness slider (0–100%) that controls the brightness/value of generated icons. Default: 100%.
- **FR-34.** The letter or symbol overlaid on a generated icon shall automatically use a contrasting color (light on dark background, dark on light background), determined by a luminance threshold.
- **FR-35.** The luminance threshold shall be configurable via a slider, allowing the user to tune the crossover point between light and dark letter colors.

### 7.9 Favicon behavior

- **FR-36.** The favicon fetcher shall attempt the sender's exact domain first, then fall back to progressively higher-level parent domains.
- **FR-37.** The application shall provide an option to limit the portion of a web page scanned for favicon references (partial scan), to control bandwidth usage.
- **FR-38.** The application shall provide an option to scan web-app manifests for higher-resolution icons.
- **FR-39.** The application shall provide an option to require successful DMARC verification before displaying any externally fetched avatar image for a sender.

### 7.10 Avatar caching

- **FR-40.** Fetched and generated avatars shall be cached locally (in memory and/or on disk) to avoid redundant fetches or regeneration.
- **FR-41.** The cache shall automatically evict stale entries after a reasonable period, so that updated favicons and contact photos are eventually refreshed.

### 7.11 Widget and external surface support

- **FR-42.** Message-list widgets shall support the same three color display modes (none, stripe, dot) as the main message list, using the same user preference.
- **FR-43.** Message-list widgets shall provide an option to display sender avatars.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Assigning or clearing a color shall reflect in all visible surfaces within the next view refresh, with no perceptible delay.
- **NFR-2. Consistency.** A given message shall display the same resolved color in every surface where it appears (main list, widget, notification, conversation header). No surface shall use a different precedence rule.
- **NFR-3. Privacy.** No avatar-related network request shall be made unless the user has explicitly enabled the corresponding avatar source. The default configuration shall produce zero network traffic for avatar resolution.
- **NFR-4. Performance.** Avatar resolution shall not block or visibly delay message-list scrolling. Avatars for off-screen messages shall be resolved lazily.
- **NFR-5. Scale.** The avatar cache and color system shall remain performant with at least 20 accounts, 200 folders, and tens of thousands of cached sender avatars.
- **NFR-6. Accessibility.** Color indicators shall not be the sole means of conveying account/identity/folder information — textual labels shall always be available alongside color. Generated icons shall respect the user's contrast and legibility settings.
- **NFR-7. Durability.** All color assignments and avatar selections shall survive application restart, account re-authentication, and server folder re-synchronization.

## 10. Open Questions

- **OQ-1. Color stripe source in message list.** The source application's message-list color stripe is driven by the account color only (not the full identity → folder → account cascade) in some code paths, while the FAQ states identity color takes precedence. The full cascade (identity → folder → account) may only apply in certain view configurations. The exact scope of the precedence rule should be confirmed during design.
- **OQ-2. Account avatar vs. sender avatar.** Account avatars (user-uploaded images per account) and sender avatars (automatically resolved per sender) are distinct concepts that coexist. It is unclear whether the account avatar ever appears in a message-list context (e.g. as a fallback for messages sent *by* the user from that account) or only in the account-list and navigation views. This boundary should be clarified.
- **OQ-3. Favicon URI customization.** The source application allows users to configure a custom URI template for favicon resolution (substituting the domain into a pattern). The set of known-working services and the exact URI templating syntax should be documented for the user.
- **OQ-4. Distribution-channel differences for avatar sources.** In the source application, certain avatar sources (Gravatar, Libravatar, BIMI) are available only in some distribution channels due to store policies. Whether similar restrictions apply to a desktop distribution should be decided during design.
- **OQ-5. BIMI position in the cascade.** BIMI is listed as a separate feature (3.14) but participates in the avatar cascade. The boundary between "BIMI as an avatar source" (this epic) and "BIMI as an authentication indicator" (feature 3.14) should be clarified to avoid duplication.
- **OQ-6. Generated icon hue derivation.** The source application derives the hue of generated icons from a hash of the sender's email address. Whether this derivation should be specified as a requirement (ensuring deterministic colors across devices) or left as an implementation choice is an open question.
- **OQ-7. Saved search colors.** The source application allows assigning colors to saved searches. Whether this falls under this epic or under the search feature epic should be decided.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Nullable colors, not a fixed palette.** Entity colors are stored as arbitrary color values (full spectrum), not as indices into a predefined palette. This gives users maximum flexibility but means the color picker must be a full-spectrum selector (wheel or similar), not a simple swatch grid. The source application uses a color-wheel picker with a lightness slider and an RGB text-input field.
- **N-2. Identity color overrides folder and account.** The documented precedence is identity → folder → account → default. This lets a user who has three identities on one account see distinct colors per identity, while a user who only colors accounts sees account colors everywhere. The precedence is "most specific wins", which is the least surprising rule.
- **N-3. Trash and Junk get a default warning color.** These two folder types (and only these two) receive an automatic color (the theme's error/warning color) without user intervention. This is a safety measure — a user glancing at the navigation pane can immediately see if they are about to drag something into Trash or Junk. All other folder types default to no color.
- **N-4. All network avatar sources off by default.** The privacy-first posture means that a fresh installation generates no avatar-related network traffic. The user opts in to each source individually. This is consistent with the application's broader philosophy of no third-party server contact without explicit consent.
- **N-5. Saturation and brightness affect only generated icons.** The saturation and brightness sliders control the HSV parameters of the algorithmically generated letter and identicon avatars. They do not alter contact photos, favicons, or any externally sourced image. This is intentional — external images should be displayed faithfully; only synthetic images are tunable.
- **N-6. Automatic text contrast on colored backgrounds.** Wherever a color is used as a background (action bar, generated icon, color stripe with text overlay), the foreground color is computed automatically from the background's luminance. This prevents illegible text on user-chosen colors and removes the need for the user to think about contrast.
- **N-7. Persistent image access for account avatars.** When a user selects an image file as an account avatar, the application retains persistent read access to that file across restarts. If the file is later deleted or moved, the avatar silently degrades to "no avatar". The application does not copy the image into its own storage (in the source application this is handled via persistent URI permissions).
