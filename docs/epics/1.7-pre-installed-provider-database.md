# Epic: Pre-installed Provider Database

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **1.7** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Bundled list of providers with known IMAP/SMTP/OAuth settings, available offline."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Configuring an email account in any standards-based client requires the user to supply four non-trivial values beyond their address and password: the IMAP host, IMAP port, SMTP host, and SMTP port — plus encryption mode and username format for each. Most users do not know these values and should never need to. They simply want to type their email address and password, click "Next", and start reading mail.

The Pre-installed Provider Database exists to bridge this gap. It is an offline-first, bundled catalogue of well-known email provider configurations — roughly 150–170 entries covering the majority of consumer, ISP, hosting, and enterprise mail services worldwide. When the user enters their email address, the application matches the domain against this catalogue and, on a hit, fills in every server setting automatically. The user never sees a host, port, or encryption selector.

Because the catalogue ships with the application itself, this lookup requires no network connectivity, no DNS queries, and no trust in external services. It is the fastest and most reliable discovery path, taking priority over all network-based alternatives (DNS SRV, ISPDB, Microsoft Autodiscovery, port scanning). When the catalogue does not contain a match, the application transparently falls back to those network-based discovery mechanisms — but the bundled database is the first line of defence against misconfiguration.

The catalogue also encodes provider-specific behavioural tuning — connection quirks, authentication requirements, OAuth credentials, polling intervals, and known limitations — that no generic autodiscovery protocol can express. This makes it not just a convenience for users but a correctness mechanism: the application behaves differently for providers with known idiosyncrasies, and does so from the first connection attempt.

## 2. Goals

- **G1.** Enable zero-configuration account setup for every major email provider: the user supplies only their email address and password (or OAuth consent), and the application derives all server settings from the bundled catalogue.
- **G2.** Provide this setup experience offline, with no dependency on network-based discovery services.
- **G3.** Encode provider-specific quirks, workarounds, and authentication requirements so that the application behaves correctly from the first connection attempt, without user intervention or troubleshooting.
- **G4.** Serve as the highest-trust discovery source in a multi-stage discovery pipeline, ahead of DNS, ISPDB, Autodiscovery, and port scanning.
- **G5.** Support OAuth-based authentication for providers that offer it, with all necessary OAuth parameters bundled in the catalogue.
- **G6.** Provide a browsable, searchable list of known providers for users who prefer to select their provider explicitly rather than relying on automatic detection.
- **G7.** Allow the catalogue to be extended or overridden by locally imported provider profiles, so that administrators or power users can add internal or niche providers without waiting for an application update.

## 3. Non-Goals

- **NG1.** This epic does **not** define the quick-setup wizard UI itself — only the data source it consumes. The wizard is a separate concern that happens to be the primary consumer of the provider database.
- **NG2.** This epic does **not** define the network-based discovery fallback mechanisms (DNS SRV, ISPDB, Microsoft Autodiscovery, port scanning). Those are downstream consumers of the same provider data model but have their own discovery logic and trust levels.
- **NG3.** This epic does **not** prescribe how provider data is stored, parsed, or indexed at runtime — only the contract that the data model must fulfil.
- **NG4.** Automatic over-the-air updates to the provider catalogue (e.g. fetching new entries from a server between application releases) are **not** a goal. The catalogue is updated with the application itself.
- **NG5.** Support for proprietary, non-standard-protocol email services (e.g. providers that do not expose IMAP/POP3/SMTP) is **not** a goal. The application explicitly rejects such providers with a clear error message.

## 4. Glossary

| Term | Definition |
|---|---|
| **Provider** | A single entry in the bundled catalogue, representing one email service (e.g. Gmail, Outlook, Fastmail) with all of its server settings, authentication parameters, and behavioural tuning. |
| **Provider catalogue** | The complete, ordered collection of all bundled provider entries, shipped with the application and available offline. |
| **Domain pattern** | A regular-expression pattern associated with a provider, used to match the domain portion of a user's email address to the correct provider entry (e.g. `gmail\.com`, `yahoo\..*`). |
| **MX pattern** | A regular-expression pattern associated with a provider, used to match the MX (mail-exchanger) DNS records of a domain to the correct provider entry. Used as a secondary matching strategy when domain patterns do not match (e.g. a custom domain hosted on a known provider). |
| **Discovery pipeline** | The ordered sequence of strategies the application uses to determine server settings for an email address: (1) bundled catalogue lookup, (2) DNS namespace lookup, (3) DNS SRV records, (4) ISPDB / Thunderbird autoconfig, (5) Microsoft Autodiscovery, (6) port scanning. |
| **Trust score** | A numeric value assigned to each discovered server configuration indicating confidence in its correctness. Bundled catalogue entries receive the highest score. |
| **Quick setup** | The streamlined account-creation flow where the user provides only an email address and password/OAuth consent, and the application fills in all server settings automatically. |
| **OAuth profile** | The subset of a provider entry that defines OAuth 2.0 parameters: client credentials, scopes, endpoints, redirect URI, and any custom parameters. |
| **App password** | A provider-specific application password (distinct from the user's main account password) required by some providers when two-factor authentication is enabled. |
| **Provider quirk** | A behavioural override encoded in a provider entry that adjusts the application's default connection or protocol behaviour to accommodate a known provider limitation or non-standard behaviour. |
| **Imported profile** | A provider entry loaded from a user-supplied file rather than from the bundled catalogue, allowing local extensions without modifying the application. |

## 5. Personas

- **P1. The mainstream user.** Has a Gmail, Outlook, or Yahoo account. Expects to type their email address and password, click a button, and be reading mail within seconds. Has no idea what IMAP or SMTP means and should never encounter those terms.
- **P2. The multi-provider user.** Has accounts with 3–5 different providers (e.g. Gmail for personal, a company Exchange server for work, Fastmail for a side project). Needs each provider's quirks handled correctly without manual tuning.
- **P3. The custom-domain user.** Owns a personal domain hosted on a well-known provider (e.g. Google Workspace, Fastmail, Hetzner). The domain itself does not appear in the provider catalogue, but the underlying hosting provider does — the application should still match via MX records or DNS.
- **P4. The ISP email user.** Uses an email address from a regional internet service provider (e.g. Comcast, T-Online, Free.fr, Ziggo). Needs the correct settings without hunting through the ISP's support site.
- **P5. The IT administrator.** Wants to prepare a provider profile for an internal mail server so that users in their organisation get the same zero-configuration experience as mainstream provider users, without waiting for the profile to be accepted into the upstream catalogue.

## 6. User Stories

### 6.1 Automatic provider detection

> **US-1.** As a mainstream user, when I enter my email address in the quick setup flow, I want the application to automatically detect my provider and fill in all server settings, so that I never have to look up IMAP/SMTP hostnames, ports, or encryption modes.

> **US-2.** As a custom-domain user, when my domain is not in the provider catalogue but my hosting provider is, I want the application to identify the hosting provider (e.g. via MX records) and apply the correct settings, so that my custom domain works without manual configuration.

> **US-3.** As any user, when the application detects my provider, I want the detection to happen instantly and offline (no network round-trips), so that setup works even without an internet connection.

### 6.2 Browsable provider list

> **US-4.** As any user, I want to browse a list of all known providers sorted in a sensible order (popular providers first, then alphabetical), so that I can find my provider visually if I prefer not to rely on automatic detection.

> **US-5.** As any user, when I select a provider from the list, I want the application to pre-fill all server settings and show me any provider-specific guidance (e.g. "you need to enable IMAP access on this provider's website first"), so that I can proceed without guesswork.

> **US-6.** As any user, when my provider is not in the list, I want a clear "Other provider" or manual-configuration option, so that I am not stuck.

### 6.3 OAuth integration

> **US-7.** As a user of a provider that supports OAuth (e.g. Gmail, Outlook, Yahoo), I want the application to offer an OAuth sign-in flow instead of asking for my password, so that I never have to create or manage an app-specific password.

> **US-8.** As a user of a provider that supports OAuth, I want the OAuth credentials and endpoints to be bundled in the application, so that the OAuth flow works offline and without fetching configuration from the provider at setup time.

> **US-9.** As a user of a provider that requires tenant-specific OAuth (e.g. a corporate Microsoft 365 tenant), I want the application to prompt me for my tenant identifier and adjust the OAuth endpoints accordingly, so that I can authenticate against my organisation's tenant.

### 6.4 Provider-specific behaviour

> **US-10.** As a user of a provider with known connection quirks (e.g. unreliable IDLE support, partial-fetch incompatibility, TLS version restrictions), I want the application to apply the correct workarounds automatically based on the provider profile, so that I do not experience mysterious connection failures.

> **US-11.** As a user of a provider that requires an app-specific password (e.g. when two-factor authentication is enabled), I want the application to tell me this at setup time — ideally with a link to the provider's app-password page — so that I know what to do before my first connection attempt fails.

> **US-12.** As a user of a provider where IMAP access must be manually enabled on the provider's website (e.g. certain German providers), I want the application to inform me of this requirement at setup time, so that I can enable it before attempting to connect.

> **US-13.** As a user of a provider that uses a non-standard username format (e.g. local-part only instead of full email address, or a fixed domain suffix), I want the application to derive the correct username automatically from the provider profile, so that I do not have to guess the format.

### 6.5 Provider documentation

> **US-14.** As any user, when I select or am matched to a provider, I want to see any provider-specific setup instructions or links to the provider's documentation, so that I can resolve issues without contacting support.

> **US-15.** As a non-English-speaking user, I want provider-specific documentation to be displayed in my language when a translation is available, falling back to English otherwise.

### 6.6 Proprietary provider rejection

> **US-16.** As a user who enters an email address from a provider that does not support standard protocols (e.g. a provider that uses only proprietary encryption and does not expose IMAP/SMTP), I want the application to tell me clearly that this provider is not supported and why, so that I do not waste time troubleshooting a connection that can never work.

### 6.7 Imported / custom profiles

> **US-17.** As an IT administrator, I want to import a custom provider profile from a file, so that users in my organisation get the same zero-configuration experience as users of mainstream providers.

> **US-18.** As an IT administrator, I want imported profiles to coexist with the bundled catalogue (not replace it), so that mainstream providers continue to work alongside my custom entries.

### 6.8 Registration links

> **US-19.** As a prospective user browsing the provider list, when a provider offers a registration/sign-up link, I want to see that link in the provider's entry, so that I can create an account with that provider directly from the setup flow.

## 7. Functional Requirements

### 7.1 Catalogue content and structure

- **FR-1.** The application shall ship with a bundled catalogue of at least 150 email provider profiles, covering the major consumer, ISP, hosting, and enterprise email services worldwide.
- **FR-2.** Each provider entry shall specify, at minimum: a display name, and one or more server configurations (IMAP and/or POP3 for receiving, SMTP for sending), each with host, port, and encryption mode (SSL/TLS or STARTTLS).
- **FR-3.** Each provider entry may optionally specify: one or more domain-matching patterns, one or more MX-matching patterns, a unique identifier, a descriptive subtitle, a sort-order priority, a documentation URL, a registration URL, inline setup documentation (with locale-specific variants), an OAuth profile, a Microsoft Graph profile, and a set of behavioural overrides (quirks).
- **FR-4.** The catalogue shall be usable entirely offline, with no network dependency for reading or matching entries.

### 7.2 Domain and MX matching

- **FR-5.** When the user provides an email address, the application shall extract the domain and attempt to match it against the domain patterns of all enabled providers in the catalogue, using regular-expression matching.
- **FR-6.** Domain matching shall be case-insensitive.
- **FR-7.** Domain patterns shall support wildcards and alternation sufficient to cover multi-TLD providers (e.g. a single provider entry matching `yahoo.com`, `yahoo.co.uk`, `yahoo.fr`, etc.).
- **FR-8.** If no domain pattern matches, the application shall (when network is available) look up the MX records for the user's domain and attempt to match them against the MX patterns of all enabled providers, to identify custom domains hosted on known providers.
- **FR-9.** A match on a bundled catalogue entry shall take priority over any network-derived configuration (DNS SRV, ISPDB, Autodiscovery, port scan) in the discovery pipeline. Bundled entries shall receive the highest trust score.

### 7.3 Provider list presentation

- **FR-10.** The application shall present a browsable list of all enabled providers, sorted by a provider-defined priority (popular providers first), with ties broken by locale-aware alphabetical name comparison.
- **FR-11.** Providers marked as disabled shall not appear in the browsable list or in automatic matching.
- **FR-12.** Providers marked as debug-only shall appear only when the application is running in a debug or development mode.
- **FR-13.** Providers marked as alternative variants shall be visually distinguished or grouped with their primary variant.
- **FR-14.** The list shall include a fallback entry (e.g. "Other provider") that routes the user to manual server configuration.

### 7.4 Server configuration

- **FR-15.** Each server entry (IMAP, SMTP, POP3) in a provider profile shall specify: hostname, port number, and encryption mode (SSL/TLS on connect, or STARTTLS upgrade from plain).
- **FR-16.** A provider may define multiple server entries of the same type (e.g. two IMAP servers) to express alternatives or regional variants.
- **FR-17.** When a provider profile is applied during account setup, the server settings shall be pre-filled into the account configuration without requiring user modification.

### 7.5 Username format

- **FR-18.** Each provider entry shall specify the expected username format: full email address (default), local part only, or a custom template pattern.
- **FR-19.** When a provider profile is applied, the application shall derive the correct username from the user's email address according to the provider's username format and pre-fill it.

### 7.6 OAuth profiles

- **FR-20.** A provider entry may include an OAuth profile specifying: client identifier, optional client secret, required scopes, authorization endpoint URL, token endpoint URL, redirect URI, optional privacy policy URL, optional prompt behaviour, and optional custom parameters.
- **FR-21.** When a provider with an enabled OAuth profile is matched, the application shall offer OAuth-based sign-in as the preferred authentication method.
- **FR-22.** OAuth profiles may support tenant placeholders in endpoint URLs. When such placeholders are present, the application shall prompt the user for a tenant identifier and substitute it into the URLs before initiating the OAuth flow.
- **FR-23.** A provider entry may include a separate Graph API profile (with the same attribute set as an OAuth profile) for providers that support REST-based mail sending alongside IMAP-based receiving.
- **FR-24.** OAuth and Graph profiles may each be independently enabled, disabled, or restricted to debug mode.

### 7.7 Behavioural overrides (quirks)

- **FR-25.** A provider entry may specify a polling/keepalive interval (in minutes) to override the application's default, for providers with unreliable push-notification (IDLE) support.
- **FR-26.** A provider entry may specify that the application should use periodic no-operation commands instead of IDLE for connection keepalive, for providers that drop IDLE connections prematurely.
- **FR-27.** A provider entry may disable partial/incremental message fetching for providers that do not support it correctly.
- **FR-28.** A provider entry may restrict the maximum TLS version used for connections, for providers with known TLS compatibility issues.
- **FR-29.** A provider entry may disable IP-address-based connections (requiring hostname-based connections only), for providers whose certificates do not cover IP addresses.
- **FR-30.** A provider entry may flag that the provider requires the user to manually enable IMAP/SMTP access on the provider's website before the application can connect.
- **FR-31.** A provider entry may flag that the provider requires an app-specific password when two-factor authentication is enabled on the user's account.
- **FR-32.** All behavioural overrides encoded in a provider entry shall be applied automatically when the provider is used for account creation, without requiring user awareness or manual adjustment.

### 7.8 Provider documentation

- **FR-33.** A provider entry may include a link to external documentation (e.g. the provider's setup guide or the application's own FAQ entry for that provider).
- **FR-34.** A provider entry may include inline setup documentation as formatted text, with locale-specific variants (e.g. English, German, French). The application shall display the variant matching the user's locale, falling back to the default if no match exists.
- **FR-35.** A provider entry may include a registration/sign-up URL. When present, this shall be surfaced to the user in the provider's detail view or selection UI.

### 7.9 Proprietary provider rejection

- **FR-36.** The application shall maintain a list of known email domains that use proprietary protocols and do not support standard IMAP/POP3/SMTP access.
- **FR-37.** When the user enters an email address matching a proprietary-only provider, the application shall display a clear, non-technical error message explaining that this provider does not support standard email protocols and cannot be used with the application.
- **FR-38.** The rejection shall occur early — before any connection attempt — so that the user receives immediate feedback.

### 7.10 Imported profiles

- **FR-39.** The application shall support importing provider profiles from a user-supplied file, using the same data model as the bundled catalogue.
- **FR-40.** Imported profiles shall be merged with the bundled catalogue at load time, not replace it. Both bundled and imported providers shall be available for matching and browsing.
- **FR-41.** When an imported profile and a bundled profile match the same domain, the conflict resolution behaviour (imported wins, bundled wins, or user chooses) shall be deterministic and documented.

### 7.11 Catalogue maintenance

- **FR-42.** The provider catalogue shall be updated as part of the application's regular release cycle. There is no automatic over-the-air update mechanism for the catalogue alone.
- **FR-43.** Adding, modifying, or removing a provider entry shall require only a change to the catalogue data file and a new application release — no code changes shall be required for routine catalogue maintenance (unless a new quirk type is introduced).

## 8. Non-Functional Requirements

- **NFR-1. Offline availability.** The entire provider catalogue shall be readable and matchable with zero network connectivity. No provider lookup shall require a network round-trip.
- **NFR-2. Lookup speed.** Matching an email domain against the catalogue shall complete in well under one second on a modern desktop, even with 200+ provider entries.
- **NFR-3. Extensibility.** The provider data model shall accommodate new optional fields (e.g. new quirk types, new OAuth parameters) without breaking existing entries or requiring migration of the catalogue format.
- **NFR-4. Locale awareness.** Provider names shall be sorted using locale-aware collation rules. Inline documentation shall support locale-specific variants.
- **NFR-5. Coverage.** The bundled catalogue should cover the providers serving the vast majority of consumer email users worldwide, plus major regional ISPs and popular hosting providers. The exact count will grow over time.
- **NFR-6. Correctness.** Provider entries shall be verified against the provider's current server settings before each application release. Stale or incorrect entries degrade the user experience more than missing entries (a missing entry falls back to autodiscovery; an incorrect entry causes a failed connection attempt with a confusing error).
- **NFR-7. Security.** OAuth client secrets bundled in the catalogue are, by definition, not truly secret (they ship in the application binary). The application shall use OAuth flows appropriate for public clients (e.g. PKCE) where the provider supports them, and shall not rely on client-secret confidentiality for security.

## 10. Open Questions

- **OQ-1. Conflict resolution for imported profiles.** When an imported provider profile defines a domain pattern that overlaps with a bundled provider, which one takes priority? The source application loads imported profiles alongside bundled ones, but the precedence behaviour when both match the same domain is not explicitly documented. This should be specified.
- **OQ-2. Catalogue staleness detection.** Provider server settings change over time (hosts renamed, ports retired, OAuth endpoints rotated). There is currently no mechanism to detect or warn about stale entries. Should the application track a "last verified" date per entry and warn when entries exceed a certain age?
- **OQ-3. OAuth client credential rotation.** OAuth client IDs and secrets are bundled in the catalogue. If a provider revokes or rotates credentials, all users on older application versions are affected. Should there be a fallback (e.g. fetching updated OAuth credentials from a trusted server) or is the current "update the application" approach sufficient?
- **OQ-4. Provider logo/icon support.** The current catalogue does not include provider logos or icons. Should the data model support an optional icon per provider for richer presentation in the provider list?
- **OQ-5. Community-contributed providers.** Is there a process for third parties (ISPs, hosting providers, community members) to submit new provider entries for inclusion in the bundled catalogue? If so, what verification is required before inclusion?
- **OQ-6. Provider-specific folder name mappings.** Some providers use non-standard names for system folders (e.g. a localised "Sent" folder name). The provider catalogue does not currently encode folder-name mappings. Should it, or is auto-detection of folder roles (a separate feature) sufficient?
- **OQ-7. Multiple server alternatives.** The data model supports multiple server entries per provider, but the selection logic when multiple IMAP or SMTP servers are defined (e.g. regional variants) is not well documented. Should the user be prompted to choose, or should the application try each in order?
- **OQ-8. Debug-only providers.** Several provider entries are marked as debug-only. The purpose of these entries (testing? beta rollout? experimental OAuth?) should be clarified so that the debug flag's semantics are well-understood.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Offline-first, not online-first.** The bundled catalogue is deliberately the *first* discovery strategy, not a cache of previously fetched results. This is a design choice with real trade-offs: the catalogue can go stale between releases, but in exchange the user gets instant, deterministic, network-independent provider detection. Network-based discovery (DNS SRV, ISPDB, Autodiscovery, port scanning) serves as a fallback for providers not in the catalogue, not as the primary path. This ordering means that even a user setting up their email on a train with no connectivity gets a working configuration for any major provider.

- **N-2. Trust scoring creates a merge strategy, not a replacement strategy.** The discovery pipeline does not stop at the first match. Each discovery method assigns a trust score to its results, and the bundled catalogue receives the highest score. This means that if both the catalogue and ISPDB return configurations for the same provider, the catalogue's configuration wins — but if the catalogue has no entry and ISPDB does, the ISPDB result is used. The scoring approach avoids a brittle "if/else" chain and makes it straightforward to add new discovery methods in the future.

- **N-3. Quirks belong in the catalogue, not in provider-specific code branches.** Connection tuning (keepalive intervals, NOOP-vs-IDLE, partial-fetch disabling, TLS version caps) is expressed as data in the provider entry rather than as conditional logic scattered through the connection code. This means that adding a new quirk for a new provider is a catalogue change, not a code change — reducing the risk of regression and keeping provider-specific knowledge in one place.

- **N-4. MX-based matching extends the catalogue's reach to custom domains.** Many users have personal domains hosted on a handful of major providers (Google Workspace, Fastmail, Hetzner, etc.). Domain-pattern matching alone cannot cover these because the domain is unique per user. MX-pattern matching solves this: the application looks up the domain's MX records and matches them against known provider MX patterns. This is the one place where the "offline-first" principle is relaxed — MX lookup requires a network call — but the trade-off is worthwhile because it dramatically expands the catalogue's effective coverage without requiring an entry for every custom domain in the world.

- **N-5. Proprietary providers are rejected, not silently ignored.** The application maintains an explicit blocklist of domains belonging to providers that use only proprietary protocols. When a user enters one of these domains, the application shows a clear error explaining *why* the provider is unsupported (no standard IMAP/SMTP access), rather than letting the connection attempt fail with a cryptic timeout or authentication error. This is a deliberate UX choice: a fast, honest rejection is better than a slow, confusing failure.

- **N-6. OAuth profiles are bundled, not fetched.** OAuth client IDs, scopes, and endpoint URLs are embedded directly in the provider catalogue. This means they work offline and do not depend on provider-side discovery mechanisms (like OpenID Connect discovery documents). The downside is that credential rotation requires an application update, but the upside — instant, deterministic OAuth setup with no network dependency beyond the OAuth flow itself — is considered more valuable for the typical user.

- **N-7. The catalogue is flat, not hierarchical.** Each provider entry is self-contained. There is no inheritance (e.g. "Outlook variants inherit from a base Outlook profile"). When multiple variants of a provider exist (e.g. Outlook with different OAuth configurations, or regional variants of a hosting provider), each is a separate, complete entry. This trades some duplication for simplicity and independence: modifying one variant cannot accidentally break another.
