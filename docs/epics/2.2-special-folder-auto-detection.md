# Epic: Special-Folder Auto-Detection

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **2.2** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Automatically detects Inbox, Sent, Drafts, Trash, Spam, Archive, plus user overrides."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[MUST]` — required for v1 to be a credible, usable desktop email client. v1 cannot ship without it.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

When a user adds a new email account to a mail client, the server presents a flat or hierarchical list of folders. Among those folders are several that serve well-known roles: Inbox (where new mail arrives), Sent (copies of outgoing messages), Drafts (in-progress compositions), Trash (deleted messages), Spam/Junk (messages flagged as unsolicited), and Archive (long-term storage). The user expects the mail client to recognize these roles automatically so that operations such as "move to trash", "save draft", and "file sent copy" work correctly without manual configuration.

Unfortunately, there is no universal guarantee that a mail server will clearly advertise which folder serves which role. Some servers provide standardized metadata attributes; others rely on conventional folder names in English or the user's locale; and some do neither. A robust mail client must therefore employ a multi-layered detection strategy that handles the common case seamlessly, gracefully degrades for non-standard servers, and always allows the user to correct mistakes manually.

The Special-Folder Auto-Detection feature exists to:

1. Automatically assign semantic roles to server-side folders at account setup time, minimizing user effort.
2. Continue to detect or correct role assignments when the folder list changes (e.g. a folder is renamed on the server).
3. Give the user full authority to override any assignment, ensuring that even the most unusual server configuration can be made to work correctly.

## 2. Goals

- **G1.** Correctly identify the Inbox, Sent, Drafts, Trash, Spam, and Archive folders for the vast majority of providers and locales without user intervention.
- **G2.** Support a multi-tiered detection approach: prefer server-advertised role metadata, fall back to folder-name heuristics, and ultimately defer to user assignment.
- **G3.** Ensure that exactly one folder per account is assigned to each role at any time (except where no suitable candidate exists), preventing ambiguity.
- **G4.** Allow the user to manually assign or reassign any folder to any role through the account settings interface.
- **G5.** Persist role assignments across application restarts, folder re-synchronization, and server-side folder renames.
- **G6.** Ensure that the assignment of a role to a folder triggers appropriate default behaviors (synchronization, polling, retention) without requiring additional user action.

## 3. Non-Goals

- **NG1.** This epic does **not** define how mail is synchronized, fetched, or stored — only how the application determines *which* folder serves each role.
- **NG2.** This epic does **not** cover creation of local-only folders (e.g. a local drafts or outbox folder that exists only on the client). It concerns server-side folder classification.
- **NG3.** This epic does **not** define how folders are displayed, sorted, or navigated — only how roles are assigned. Display conventions (e.g. system folders appearing above user folders) belong to the folder-list feature.
- **NG4.** This epic does **not** address folder creation, renaming, or deletion on the server — only classification of folders that already exist.
- **NG5.** Handling of provider-specific quirks beyond role detection (e.g. provider-specific polling requirements) is out of scope; those belong to per-provider integration epics.

## 4. Glossary

| Term | Definition |
|---|---|
| **Account** | A configured connection to a single mail server, with credentials, server settings, and its own folder tree. |
| **Folder** | A named container of messages on the mail server. |
| **Folder role / type** | The semantic meaning assigned to a folder: Inbox, Sent, Drafts, Trash, Spam (Junk), Archive, System, or User (no special role). |
| **System folder** | Any folder assigned one of the well-known roles (Inbox, Sent, Drafts, Trash, Spam, Archive). |
| **User folder** | A folder with no assigned special role — a general-purpose folder created by the user or the provider. |
| **Role metadata** | Server-advertised attributes (per RFC 6154 / IMAP SPECIAL-USE extension) that declare a folder's intended role. |
| **Name-based heuristic** | A fallback strategy that infers a folder's role from its name by matching against known folder-name patterns in multiple languages. |
| **User override** | A manual assignment by the user that takes precedence over any automatic detection. |
| **Detection priority** | The order in which strategies are tried: (1) role metadata from the server, (2) name-based heuristic, (3) user override; with user override being authoritative at all times. |
| **Confidence score** | An internal ranking used during name-based heuristic matching to select the best candidate when multiple folders partially match. |

## 5. Personas

- **P1. The typical user.** Adds one or two accounts from well-known providers. Expects everything to "just work" — the right folders are detected automatically and operations like "delete" or "save draft" target the correct folder without configuration.
- **P2. The multilingual user.** Uses a provider whose folder names are in a non-English language, or has a mixed-language folder tree. Expects the application to recognize common folder names in their language.
- **P3. The self-hosted user.** Runs their own mail server with non-standard or custom folder names. Auto-detection may fail; this user needs a clear, accessible path to manually assign folder roles.
- **P4. The multi-account user.** Has accounts across multiple providers, each with different folder naming conventions. Expects each account's roles to be detected independently and correctly.
- **P5. The corporate user.** Uses a provider with shared or organizational folders alongside personal folders. Expects the application to distinguish personal system folders from shared/namespace folders.

## 6. User Stories

### 6.1 Automatic detection at account setup

> **US-1.** As a typical user, when I finish adding a new account, I want the application to have automatically identified my Inbox, Sent, Drafts, Trash, Spam, and Archive folders, so that I can immediately perform common operations (send, delete, archive) without any manual folder configuration.

> **US-2.** As a multilingual user, when my mail server uses folder names in my language (e.g. "Papierkorb" for Trash, "Envoyés" for Sent), I want the application to recognize those names and assign the correct roles, so that detection is not limited to English folder names.

> **US-3.** As any user, when the server advertises folder roles through standard metadata, I want the application to trust those role declarations as the primary signal, so that detection is reliable even if the folder names are unusual or ambiguous.

### 6.2 Inbox as guaranteed minimum

> **US-4.** As any user, I want the application to always identify an Inbox for every account, even if the server does not explicitly advertise one, so that incoming mail always has a home.

> **US-5.** As any user, if the server's folder list does not contain any folder that can be identified as the Inbox, I want the application to synthesize or assume a default Inbox entry, so that account setup can still complete successfully.

### 6.3 Uniqueness of role assignments

> **US-6.** As any user, I want each account to have at most one folder assigned to each role (one Inbox, one Sent, one Drafts, one Trash, one Spam, one Archive), so that there is never ambiguity about which folder receives drafts, deleted messages, etc.

> **US-7.** As any user, if the server presents multiple folders that could serve the same role (e.g. two folders with Trash-like attributes), I want the application to choose the best candidate and leave the other as a regular user folder, so that I am not confused by duplicate roles.

### 6.4 Fallback when metadata is absent

> **US-8.** As a self-hosted user whose server does not advertise role metadata, I want the application to attempt to infer folder roles from folder names using common patterns, so that I am not forced to configure everything manually.

> **US-9.** As any user, when name-based inference yields ambiguous or low-confidence results, I want the application to prefer not assigning a role over assigning the wrong one, so that I can trust the assignments that are made.

### 6.5 User overrides

> **US-10.** As any user, I want to be able to manually designate which folder serves as my Drafts, Sent, Archive, Trash, and Spam folder for each account through the account settings screen, so that I can correct detection errors or accommodate unusual configurations.

> **US-11.** As a self-hosted user, when auto-detection has failed to identify my Sent folder, I want to select it from a list of all available folders in account settings, so that sent messages are filed correctly.

> **US-12.** As any user, when I manually assign a folder to a role that was previously assigned to a different folder, I want the previous folder to revert to a regular user folder, so that there is no conflict.

> **US-13.** As any user, I want my manual role assignments to be durable — they should persist across application restarts, re-synchronization of the folder list, and even server-side folder renames — so that I do not have to re-configure after every sync.

### 6.6 Re-detection on folder list changes

> **US-14.** As any user, when a folder is renamed on the server and the application re-synchronizes the folder list, I want the application to track the renamed folder and preserve its role assignment, so that a server-side rename does not break my configuration.

> **US-15.** As any user, if a system folder disappears from the server (deleted or unsubscribed) and a new folder appears that matches the role, I want the application to re-assign the role to the new folder (or prompt me), so that operations continue to work.

> **US-16.** As any user, when a new folder appears on the server during synchronization, I want the application to classify it but not override an existing role assignment, so that my manual choices remain stable.

### 6.7 Default behaviors triggered by role

> **US-17.** As a typical user, when a folder is assigned the Inbox role, I want it to be automatically added to the unified inbox, have notifications enabled, and be synchronizing, so that the most important folder is immediately active without extra steps.

> **US-18.** As any user, when a folder is assigned any system role (Sent, Drafts, Archive, Trash, Spam), I want it to be automatically set to synchronize with sensible defaults for polling frequency and retention, so that system folders are immediately operational.

> **US-19.** As any user, I want the application to assign longer default retention to the Drafts folder than to other folders, so that in-progress work is less likely to be pruned.

### 6.8 Provider-specific behavior

> **US-20.** As a user of a provider with non-standard folder semantics (e.g. a provider that stores all mail in an "All Messages" folder that serves as Archive), I want the application to correctly map provider-specific conventions to the standard roles, so that features like "archive" work as expected.

> **US-21.** As a user of a provider where a specific folder serves a dual purpose (e.g. functions as both inbox and unknown-sender filter), I want the application to handle it appropriately and inform me if special configuration is in place, so that I am not confused by unexpected behavior.

### 6.9 Setup-time review

> **US-22.** As any user, during account setup, I want to be shown the detected system folder assignments before finalizing, so that I can review and correct any misdetection before the account becomes active.

> **US-23.** As a self-hosted user, if the application was unable to detect one or more required system folders (e.g. no Drafts folder found), I want to be informed during setup so that I can select one, rather than discovering the gap later when an operation fails.

## 7. Functional Requirements

### 7.1 Role taxonomy

- **FR-1.** The application shall recognize the following folder roles: Inbox, Sent, Drafts, Trash, Spam (Junk), Archive, System (important/flagged — non-assignable by user), and User (no special role).
- **FR-2.** Each account shall have at most one folder assigned to each of the user-assignable roles (Inbox, Sent, Drafts, Trash, Spam, Archive) at any time.
- **FR-3.** The Inbox role shall always be assigned for every account. If no folder can be identified as the Inbox, the application shall assume or synthesize one.

### 7.2 Detection strategy — Tier 1: Server-advertised metadata

- **FR-4.** When the server advertises folder role attributes conforming to the IMAP SPECIAL-USE extension (RFC 6154) or equivalent, the application shall use those attributes as the primary signal for role assignment.
- **FR-5.** The following server-advertised attributes shall be mapped to roles:
  - "All" or "Archive" → Archive
  - "Drafts" → Drafts
  - "Trash" → Trash
  - "Junk" → Spam
  - "Sent" → Sent
  - "Important" → System (non-assignable)
  - "Flagged" → System (non-assignable)
- **FR-6.** Folders whose attributes include "NoSelect" or "NonExistent" shall not be assigned any role and shall not be presented as selectable to the user.
- **FR-7.** Any folder whose name matches the canonical "INBOX" (case-insensitive) shall be assigned the Inbox role, regardless of attributes.

### 7.3 Detection strategy — Tier 2: Name-based heuristic

- **FR-8.** For any folder that was not classified by Tier 1 (i.e. remains a "User" folder), the application shall attempt to infer its role by matching its name (case-insensitively, substring match) against a built-in dictionary of known folder names.
- **FR-9.** The dictionary shall cover at minimum the following languages: English, German, French, Russian, Italian, Polish, Dutch, and Norwegian, with common folder names for Drafts, Trash, Spam, Sent, and Archive in each.
- **FR-10.** Each dictionary entry shall carry a confidence score. When multiple folders match the same role, the application shall select the candidate with the highest confidence. Ties shall be broken by preferring shallower (less deeply nested) folders.
- **FR-11.** Name-based heuristic matching shall only assign a role to a folder if no other folder in the same account already holds that role (from Tier 1 or a prior heuristic match).
- **FR-12.** Certain dictionary entries may be scoped to a specific server host, so that provider-specific folder names do not produce false positives on other servers.

### 7.4 Detection strategy — Tier 3: User override

- **FR-13.** The user shall be able to manually assign any selectable folder to any of the assignable roles (Sent, Drafts, Trash, Spam, Archive) through the account settings interface.
- **FR-14.** A user override shall take immediate, authoritative precedence over any prior automatic detection for that role.
- **FR-15.** When the user assigns a role to a folder, any other folder that previously held that role shall revert to "User" type.
- **FR-16.** User overrides shall persist across application restarts, folder re-synchronization, and server-side renames, and shall not be overwritten by subsequent automatic detection runs.

### 7.5 Behavior on re-synchronization

- **FR-17.** When the application re-synchronizes the folder list from the server, it shall not change the role of a folder that already has an assigned role, except in the following cases:
  - A folder transitions between "User" and "System" types based on updated server attributes.
  - A folder gains or loses selectability.
  - The folder previously holding the Inbox role no longer exists, and a new folder claims the Inbox role via server attributes.
- **FR-18.** Newly discovered folders (not previously seen) shall be classified using the full detection strategy (Tier 1 then Tier 2), but shall not displace an existing role assignment.
- **FR-19.** If a system folder is renamed on the server and the old name no longer exists, the application shall update its records to reflect the new name while preserving the role assignment.

### 7.6 Default properties triggered by role assignment

- **FR-20.** When a folder is assigned the Inbox role, it shall automatically be configured with: synchronization enabled, unified-inbox membership enabled, notifications enabled, and classification-source enabled.
- **FR-21.** When a folder is assigned any system role (Sent, Drafts, Trash, Spam, Archive), it shall automatically be configured with: synchronization enabled and polling/download settings appropriate to the role.
- **FR-22.** The Drafts role shall trigger a longer default retention period than other system folders.
- **FR-23.** The Spam role shall automatically be configured as a classification-training source.
- **FR-24.** These default properties shall only be set at the time of initial role assignment. Once set, the user is free to change them, and subsequent re-synchronization shall not reset them.

### 7.7 Provider-specific rules

- **FR-25.** The application shall support a mechanism to define provider-specific folder rules that trigger additional default behaviors for specific folders on specific servers (e.g. adding a particular folder to the unified inbox or overriding its display name).
- **FR-26.** Provider-specific rules shall be applied after standard detection, as a supplementary layer, and shall not interfere with user overrides.

### 7.8 Setup-time presentation

- **FR-27.** During account setup, after the folder list has been fetched and roles detected, the application shall present the detected system folder assignments to the user for review before finalizing the account.
- **FR-28.** The user shall be able to change any system folder assignment from the setup screen before completing account creation.
- **FR-29.** If the application cannot identify a required system folder (Drafts, Sent, Trash), it shall warn the user during setup and allow them to select one manually.

## 8. Non-Functional Requirements

- **NFR-1. Speed.** Folder detection (Tier 1 + Tier 2) shall complete within a negligible fraction of the overall account setup time. The user shall not perceive any delay attributable to detection logic.
- **NFR-2. Accuracy.** For servers that advertise standard role metadata, detection shall be 100% accurate. For servers that rely on name-based heuristics, detection shall be correct for at least the top 50 email providers by market share and for all languages in the dictionary.
- **NFR-3. Stability.** Once a role is assigned (whether automatically or manually), it shall remain stable across unlimited re-synchronization cycles unless the underlying server state genuinely changes.
- **NFR-4. Extensibility.** The name-based heuristic dictionary shall be straightforward to extend with additional languages or provider-specific patterns without requiring changes to the detection logic itself.
- **NFR-5. Transparency.** The user shall always be able to see which folder currently holds each role for any account, and shall always be able to change it.
- **NFR-6. Graceful degradation.** If the server fails to return a folder list (e.g. network error during LIST), the application shall retain previously detected assignments and retry on next synchronization, rather than clearing assignments.
- **NFR-7. No false promotion.** The application shall never assign a system role to a folder with low confidence if doing so would be more harmful than leaving the role unassigned. (E.g., misidentifying an important user folder as Trash would cause data loss.)

## 10. Open Questions

- **OQ-1. Incomplete language coverage.** The current name-based dictionary covers approximately eight languages. Should additional languages be prioritized, and if so, which ones? (The code shows that some providers in regions without dictionary coverage may require users to manually assign folders.)
- **OQ-2. Confidence threshold for assignment.** When a name-based heuristic match has a low confidence score, should the application assign the role tentatively (and allow the user to confirm/reject during setup), or should it leave the role unassigned and inform the user? The current behavior assigns regardless of score if it is the best candidate.
- **OQ-3. Handling of the "All Mail" folder.** Some providers (notably Gmail) expose an "All Mail" folder that contains every message. This is mapped to the Archive role, but its semantics differ from a traditional archive (it is additive, not a move target). Should the epic distinguish between "All Mail" style archives and "move-to" style archives?
- **OQ-4. Multiple folders matching a role with equal confidence.** When two folders match the same role with identical confidence scores, the current behavior prefers shallower (less nested) folders and non-read-only folders. Is this heuristic sufficient, or should additional tie-breaking criteria be specified?
- **OQ-5. Re-detection trigger.** Currently, re-detection occurs only during folder list synchronization. Should there also be a user-triggered "re-detect" action that reruns the full detection strategy (useful after server-side reorganization)?
- **OQ-6. Role metadata vs. user override conflict.** If a user manually assigns the Sent role to folder A, and the server later begins advertising folder B as Sent via role metadata, should the user override still take absolute precedence? (Current behavior: yes, user override wins. This seems correct but should be explicitly confirmed.)
- **OQ-7. Provider-specific rule maintenance.** The application contains a small number of hardcoded provider-specific rules. Should these be externalized into a user-editable or community-maintained configuration, or is the current approach sufficient?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Three-tier detection is sequential, not parallel.** The detection strategy applies Tier 1 (server metadata) to all folders first, then Tier 2 (name heuristics) only to folders that remain unclassified. This ensures that server-advertised roles are never contradicted by name heuristics. Tier 3 (user override) operates independently and takes precedence at all times.

- **N-2. Name heuristic uses substring containment, not exact match.** The dictionary matching checks whether a folder's name *contains* a known pattern (case-insensitively), rather than requiring an exact match. This handles hierarchical names like "INBOX/Drafts" or "Mail/Sent Items" gracefully, but means the dictionary entries must be chosen carefully to avoid false positives (e.g. "all" is commented out to avoid matching unrelated folder names like "Install").

- **N-3. Duplicate role prevention.** If the server already has a folder detected as (say) Trash via metadata attributes, then no second folder will be assigned Trash via the name heuristic — even if the second folder's name is a perfect match. This prevents confusing duplicate assignments. The same principle applies in reverse: a name-detected assignment blocks a second name-detected assignment.

- **N-4. Provider-scoped dictionary entries.** Some folder names are ambiguous across providers — a name that means "Sent" on one provider might mean something else on another. The dictionary supports scoping entries to a specific server hostname to avoid cross-provider false positives.

- **N-5. Preference for shallower folders in tie-breaking.** When two folders match the same role with equal confidence, the shallower (less deeply nested) folder is preferred. The rationale is that system folders are typically top-level, while nested folders with similar names are more likely to be user-created sub-organizations.

- **N-6. Inbox is always guaranteed.** The application treats the Inbox as non-optional. If no folder in the server's list resolves to Inbox (neither via metadata, nor via the canonical "INBOX" name match), the application creates a synthetic Inbox entry. This ensures that all account-level operations that require an Inbox (receiving mail, unified inbox membership) always have a target.

- **N-7. Role assignment triggers default properties, not ongoing enforcement.** When a role is first assigned to a folder, sensible defaults are applied (sync enabled, appropriate retention, etc.). However, these are one-time defaults, not ongoing constraints. The user is free to disable synchronization on their Drafts folder, for example, and re-synchronization will not force it back on. This respects user autonomy while providing a good out-of-box experience.

- **N-8. User override survives type transitions.** During re-synchronization, only transitions between "User" and "System" (a non-assignable role like Important/Flagged) are allowed to change a folder's type. A user-assigned role (Sent, Drafts, etc.) is never overwritten by re-detection — even if the server stops advertising the corresponding attribute. The user's explicit choice is authoritative.
