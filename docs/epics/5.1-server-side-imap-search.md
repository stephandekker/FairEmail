# Epic: Server-Side IMAP Search

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **5.1** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Server-side IMAP search with date range, sender, recipient, subject, body, flags."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

An email client that stores only a sliding window of messages locally (the most recent N days) leaves a large and growing body of mail accessible only on the server. A user who needs to find an old invoice, a conversation from last year, or every flagged message from a particular sender cannot rely on local search alone — the messages they need may never have been downloaded.

Server-side IMAP search solves this by delegating the query to the mail server, which has access to the full mailbox. The server evaluates the search criteria against all messages it holds and returns matching results, regardless of whether those messages exist in the local cache.

Because IMAP search capabilities vary between providers (some support full-text body search, others do not; some handle non-ASCII characters, others do not; Gmail exposes proprietary query extensions), the feature must gracefully negotiate what the server actually supports and fall back or inform the user when a particular criterion cannot be honored.

The feature complements — but does not replace — local search. Users should be able to start with a fast local search and then seamlessly escalate to a server search when local results are insufficient, or choose server search directly when they know the message predates their local window.

## 2. Goals

- **G1.** Allow users to search for messages on the mail server using a rich set of criteria: date range, sender, recipient, subject, body text, and flags.
- **G2.** Provide a clear, discoverable path from local search to server search, so that a user whose local results are incomplete can continue searching on the server without re-entering their query.
- **G3.** Negotiate server capabilities transparently: attempt the full query, degrade gracefully when the server rejects a criterion, and inform the user what was excluded.
- **G4.** Support structured search expressions (prefix-based criteria such as `from:`, `to:`, `cc:`, `bcc:`, `keyword:`) that let power users target specific fields without opening the advanced search dialog.
- **G5.** For Gmail accounts, expose the provider's native query language so that users who know it can leverage Gmail-specific operators.
- **G6.** Present server search results in the same message-list experience as local results, with identical sort, filter, threading, and action capabilities.

## 3. Non-Goals

- **NG1.** This epic does **not** define local full-text search indexing — that is covered by feature 5.2.
- **NG2.** This epic does **not** define saved searches / virtual folders — that is covered by feature 5.3.
- **NG3.** This epic does **not** define the quick filter bar — that is covered by feature 5.4.
- **NG4.** Multi-folder server search in a single IMAP command is not a goal. The IMAP protocol does not support searching across multiple folders simultaneously; the application works within this constraint.
- **NG5.** POP3 server-side search is not a goal. The POP3 protocol provides no search facility; POP3 accounts rely entirely on local search.
- **NG6.** Searching by attachment presence, encryption status, local notes, hidden/snoozed state, or filename on the server is not a goal — these are local-only metadata not represented in standard IMAP search.

## 4. Glossary

| Term | Definition |
|---|---|
| **Server-side search** | A search where the query is sent to the IMAP server for evaluation against all messages the server holds, returning matches regardless of local cache state. |
| **Local search** | A search evaluated against the locally downloaded message store only. Faster but limited to messages that have been fetched. |
| **Search criterion** | A single condition in a search query (e.g. "sender contains X", "received after date Y", "body contains Z"). Multiple criteria are combined with AND logic by default. |
| **Search expression prefix** | A structured token such as `from:`, `to:`, `cc:`, `bcc:`, or `keyword:` that targets a specific message field in the query text. |
| **Charset negotiation** | The process of requesting that the server interpret the search query in a specific character encoding (typically UTF-8), with fallback to ASCII if the server does not support it. |
| **Gmail raw search** | A provider-specific extension that allows Gmail's native query operators to be passed directly to the server, bypassing standard IMAP search term construction. |
| **Capability degradation** | Automatic retry of a search with reduced criteria when the server rejects the original query (e.g. removing body search if the server reports that full-text search is unsupported). |
| **Boundary loading** | Incremental fetching of search results in pages as the user scrolls, rather than loading the full result set at once. |
| **IMAP keyword** | A user-defined or server-defined label/flag on a message (distinct from the standard flags like Seen and Flagged). |

## 5. Personas

- **P1. The archive diver.** Has years of email on the server but only the most recent weeks cached locally. Frequently needs to find old messages — receipts, contracts, travel confirmations — and knows roughly when they arrived or who sent them.
- **P2. The quick-filter user.** Uses the search dialog primarily to find recent messages by a combination of sender and subject. Expects results to appear quickly and does not distinguish between local and server results.
- **P3. The power searcher.** Uses structured prefixes (`from:`, `to:`, `keyword:`) and advanced criteria (date ranges, flag filters, size filters) to construct precise queries. May use Gmail raw operators if on Gmail.
- **P4. The multi-account user.** Has several accounts and wants to search across all of them, understanding that cross-account search requires per-folder server queries or local search, and is willing to narrow scope when needed.

## 6. User Stories

### 6.1 Initiating a search

> **US-1.** As any user, I want to open a search dialog from the message list (whether in a single folder, an account view, or the Unified Inbox), so that I can search for messages from wherever I am in the application.

> **US-2.** As any user, I want to type a free-text query and have the application search across the fields I have selected (sender, recipient, subject, body, keywords), so that a simple query covers the most common search needs.

> **US-3.** As any user, I want the search dialog to remember my most recent queries (at least the last three) and present them as quick-access buttons, so that I can repeat frequent searches without retyping.

### 6.2 Choosing search scope

> **US-4.** As any user, when I initiate a search from a specific folder, I want the option to search on the server for that folder, so that I can find messages that are not in my local cache.

> **US-5.** As any user, when I initiate a search from the Unified Inbox or an account-level view, I want the search to run locally across all relevant folders by default, since the IMAP protocol does not support multi-folder server search in a single operation.

> **US-6.** As any user, after a local search returns results, I want a clearly labeled action to "continue searching on the server" that lets me pick a specific folder and re-run the same query on the server, so that I can escalate to server search when local results are incomplete.

> **US-7.** As any user, when I have selected a specific folder, I want a toggle or checkbox that lets me choose whether to search on the device first or go directly to the server, so that I can control the search strategy.

### 6.3 Search criteria

> **US-8.** As any user, I want to expand an "advanced options" section in the search dialog where I can enable or disable individual search fields: senders, recipients, subject, keywords, message body, so that I can control exactly which parts of the message are searched.

> **US-9.** As any user, I want to specify a date range (after date, before date, or both) for my search, so that I can narrow results to a specific time period.

> **US-10.** As any user, I want to filter search results by message flags: unread only, flagged/starred only, so that I can combine text search with flag-based filtering.

> **US-11.** As any user, I want to filter search results by minimum message size, so that I can find large messages (e.g. those with big attachments).

> **US-12.** As any user, I want to choose whether to include messages in the Trash and Junk/Spam folders in my search results, so that deleted or spam messages do not pollute my results by default but are findable when I need them.

### 6.4 Structured search expressions

> **US-13.** As a power user, I want to type prefix-based search expressions directly in the query field (e.g. `from:alice@example.com`, `to:bob@example.com`, `cc:carol@example.com`, `bcc:dave@example.com`, `keyword:important`), so that I can target specific fields without opening the advanced options panel.

> **US-14.** As a Gmail user, I want to use a `raw:` prefix to pass Gmail's native query operators directly to the server, so that I can leverage Gmail-specific search features (e.g. `raw:larger:10M`, `raw:has:attachment`).

> **US-15.** As a Gmail user, I want the `raw:` search to work from the Archive folder (and from the Unified Inbox when only one Gmail account is configured), so that it can search across Gmail's full mail corpus.

### 6.5 Capability negotiation and fallback

> **US-16.** As any user, when the server does not support full-text body search, I want the application to automatically retry my search without the body criterion and notify me that body search was excluded, so that I still get partial results rather than a blank failure.

> **US-17.** As any user, when the server does not support keyword search, I want the application to automatically retry without the keyword criterion and notify me, so that the search still returns results on the remaining criteria.

> **US-18.** As any user, when my query contains non-ASCII characters and the server does not support UTF-8 search, I want the application to attempt a best-effort transliteration to ASCII and retry, so that searches for names with accents or non-Latin characters degrade gracefully rather than failing outright.

### 6.6 Search results

> **US-19.** As any user, I want server search results to appear in the same message-list format as any other folder view, with the same display options (density, columns, preview text, account color, threading), so that the search experience is consistent with the rest of the application.

> **US-20.** As any user, I want an indicator in the search results view that tells me whether I am seeing local results or server results, so that I understand the scope of what is displayed.

> **US-21.** As any user, I want search results to load incrementally as I scroll, rather than forcing me to wait for the entire result set, so that I can start reviewing matches immediately.

> **US-22.** As any user, I want to be able to perform all standard message actions on search results (read, reply, forward, move, delete, flag, archive, etc.), so that I can act on found messages without navigating away from the results.

### 6.7 Remembering preferences

> **US-23.** As any user, I want the search dialog to remember my last-used settings (which fields were enabled, whether trash/junk was included, whether device-first was selected), so that I do not have to reconfigure the dialog each time I search.

> **US-24.** As any user, I want my recent search queries to persist across application restarts, so that I can quickly repeat searches from a previous session.

## 7. Functional Requirements

### 7.1 Search initiation

- **FR-1.** The application shall provide a search action accessible from any message-list context: a specific folder, an account-level view, or the Unified Inbox.
- **FR-2.** The search dialog shall present a text input field for the query, quick-access buttons for the most recent queries (up to three), and an expandable section for advanced criteria.
- **FR-3.** The search dialog shall provide quick-filter buttons for common flag-based searches (flagged, unread, hidden, with attachments, with calendar invitations, with notes) that can be activated without typing a text query.

### 7.2 Search scope

- **FR-4.** When the user has selected a specific folder, the search dialog shall offer a toggle to search on the device first or to search directly on the server.
- **FR-5.** When no specific folder is selected (Unified Inbox or account-level view), search shall execute locally across all applicable folders. The device-first toggle shall be disabled in this context.
- **FR-6.** After any search completes, the application shall offer an action to "continue searching on the server" that allows the user to select a specific account and folder for a server-side search using the same query.

### 7.3 Supported server-side search criteria

- **FR-7.** The following criteria shall be supported for server-side IMAP search, all combined with AND logic:

  | Criterion | Description |
  |---|---|
  | **Sender** | Matches messages whose From header contains the search text. |
  | **Recipient** | Matches messages whose To, CC, or BCC headers contain the search text. |
  | **Subject** | Matches messages whose Subject header contains the search text. |
  | **Body** | Matches messages whose body text contains the search text. |
  | **Keywords** | Matches messages bearing a specified IMAP keyword/label. |
  | **Flagged** | Matches messages with the Flagged (starred) flag set. |
  | **Unseen** | Matches messages without the Seen (read) flag set. |
  | **Date — on or after** | Matches messages received on or after a specified date. |
  | **Date — on or before** | Matches messages received on or before a specified date. |
  | **Size — greater than** | Matches messages larger than a specified size. |

- **FR-8.** Each text-based criterion (sender, recipient, subject, body, keywords) shall be individually toggleable by the user in the search dialog.
- **FR-9.** The date-range criteria shall be selectable via a date picker that allows the user to specify a start date, an end date, or both.
- **FR-10.** The application shall allow the user to include or exclude messages in Trash and Junk/Spam folders from the search scope.

### 7.4 Structured search expressions

- **FR-11.** The application shall recognize the following prefix expressions in the query text and translate them into targeted server-side search criteria:

  | Prefix | Target field |
  |---|---|
  | `from:` | Sender (From header) |
  | `to:` | To recipient |
  | `cc:` | CC recipient |
  | `bcc:` | BCC recipient |
  | `keyword:` | IMAP keyword/label |

- **FR-12.** Prefix expressions shall be treated as AND conditions combined with any remaining free-text query and any criteria selected in the advanced options.

### 7.5 Gmail raw search

- **FR-13.** For Gmail accounts, the application shall support a `raw:` prefix that passes the remainder of the query directly to the server using Gmail's native query extension.
- **FR-14.** Gmail raw search shall be available when searching the Archive folder, or when searching from the Unified Inbox and only one Gmail account is configured.
- **FR-15.** When the `raw:` prefix is used with a non-Gmail account, the application shall ignore the prefix and perform a standard search, or inform the user that raw search is only available for Gmail.

### 7.6 Charset negotiation

- **FR-16.** When constructing the server search command, the application shall request UTF-8 charset encoding if the server advertises support for it.
- **FR-17.** If the server rejects the UTF-8-encoded search, the application shall retry with ASCII encoding, transliterating non-ASCII characters to their closest ASCII equivalents (e.g. ß → ss, ø → o, accented characters → base characters).
- **FR-18.** If both UTF-8 and ASCII attempts fail, the search shall fail with an error message that identifies the server and the nature of the failure.

### 7.7 Capability degradation

- **FR-19.** If the server rejects a search that includes a body-text criterion with an error indicating that full-text search is not supported, the application shall:
  1. Automatically retry the search with the body criterion removed.
  2. Notify the user that body search was excluded from the results.

- **FR-20.** If the server rejects a search that includes a keyword criterion with an error indicating invalid search criteria, the application shall:
  1. Automatically retry the search with the keyword criterion removed.
  2. Notify the user that keyword search was excluded from the results.

- **FR-21.** Capability degradation shall be transparent: the user shall always be informed when a criterion has been dropped, so they understand why results may be broader than expected.

### 7.8 Result handling

- **FR-22.** Server search results shall be presented in the standard message-list view with all display options (density, columns, preview, threading, account indicator) available.
- **FR-23.** Results shall be loaded incrementally (boundary/paged loading) as the user scrolls, with an upper bound on the total number of results fetched per search (at least 100,000 messages).
- **FR-24.** Results shall be sorted by received date (newest first) by default. The user shall be able to change the sort order using the same sort options available in any message list.
- **FR-25.** The application shall display a clear indicator of whether the current results are from a local search or a server search.
- **FR-26.** All per-message actions available in a regular folder view (reply, forward, move, delete, flag, mark read/unread, archive, etc.) shall be available on messages in the search results.

### 7.9 Preference persistence

- **FR-27.** The search dialog shall persist the user's last-used settings for each toggleable criterion (senders, recipients, subject, keywords, body, notes, filenames, include trash, include junk, device-first) across sessions.
- **FR-28.** The application shall persist the user's most recent search queries (at least three) across sessions and present them as quick-access buttons in the search dialog.

### 7.10 Local-only search criteria

- **FR-29.** The following criteria shall be available for local (device) search only, and shall not be sent to the server:
  - Search in local notes
  - Search in attachment filenames
  - Filter by encryption status
  - Filter by attachment presence
  - Filter by hidden/snoozed state
  - Search in message headers (available only in debug mode)
  - Search in raw HTML content (available only in debug mode)

- **FR-30.** When the user has enabled any local-only criterion and also requests a server search, the application shall either apply the local-only filter as a post-filter on server results or clearly indicate that the criterion applies only to local results.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** The search dialog shall open and be ready for input in well under one second. Server search results shall begin appearing as soon as the server returns the first page of matches.
- **NFR-2. Scale.** The feature shall handle result sets of up to 100,000 messages without crashing or becoming unresponsive, using incremental loading.
- **NFR-3. Degradation transparency.** Every automatic capability fallback (body search removed, keyword search removed, charset downgrade) shall produce a user-visible notification explaining what happened.
- **NFR-4. Offline behavior.** If the application is offline, server search shall not be offered. The search dialog shall still allow local search. If connectivity is lost mid-search, the application shall display an appropriate error and retain any partial results already loaded.
- **NFR-5. Provider compatibility.** The feature shall work with any standard IMAP server. Provider-specific extensions (Gmail raw search) shall be activated only when the server advertises the relevant capability.
- **NFR-6. Privacy.** Search queries shall be sent only to the user's own mail server. No search telemetry, query logging, or third-party query forwarding shall occur.
- **NFR-7. Accessibility.** All search controls (text input, toggles, date pickers, quick filters, result actions) shall be reachable via keyboard, with screen-reader labels, and shall obey the application-wide theme.

## 10. Open Questions

- **OQ-1. Cross-folder server search.** The IMAP protocol limits server search to one folder at a time. When a user initiates a search from the Unified Inbox, should the application sequentially issue server searches against each member folder (potentially slow with many folders), or should it rely solely on local search in this context? The source application appears to use local-only search for multi-folder contexts, with a manual "continue on server" escalation path scoped to a single folder.
- **OQ-2. Case sensitivity.** IMAP search behavior regarding case sensitivity is server-dependent and not standardized. Should the application document or indicate to the user that case-sensitivity behavior may vary between providers?
- **OQ-3. Partial vs. whole-word matching.** Similarly, whether IMAP search performs substring matching or whole-word matching varies by server. The application cannot control this. Should this be documented to the user, or should the application attempt to detect and report the server's behavior?
- **OQ-4. Gmail raw search scope.** The source application restricts `raw:` search to the Archive folder (which in Gmail semantics means "All Mail"). Should this restriction be documented as a hard requirement, or should future implementations attempt to support `raw:` in other Gmail folders?
- **OQ-5. Result cap.** The source application caps server search results at 100,000 messages. Is this limit appropriate for all use cases, or should it be user-configurable? Should the user be informed when the cap is reached?
- **OQ-6. Local-only criteria + server search interaction.** When a user enables both server-searchable criteria (e.g. sender) and local-only criteria (e.g. has-attachment), the current behavior is ambiguous. Should the application post-filter server results using local-only criteria, or should it warn the user that some criteria will not be applied to server results?
- **OQ-7. Saved search integration.** Feature 5.3 covers saved searches. How tightly should server search integrate with saved searches — can a saved search specify "always search on server" as part of its persisted configuration?
- **OQ-8. Complex expression support.** The source application supports query modifiers (`+word` for AND, `-word` for NOT, `?word` for OR) in local search only, and requires a specific folder to be selected for complex expressions. Should server search support these modifiers as well, or is this inherently a local-search concern?

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Local-first, server-second.** The default search path is local, with server search as an explicit escalation. This is deliberate: local search is fast, does not require a network round-trip, and covers the common case (recent mail). Server search is slower, subject to provider quirks, and limited to one folder at a time. Making local search the default optimizes for the majority of searches while keeping server search available for the minority that need it.
- **N-2. AND-only server logic.** All server-side search criteria are combined with AND logic. OR combinations and NOT modifiers are not sent to the server. This is a pragmatic choice: IMAP's OR and NOT support is inconsistent across servers, and AND-only queries produce predictable, narrowing results. More complex boolean logic is deferred to local search where the application has full control.
- **N-3. Automatic capability degradation.** Rather than probing server capabilities in advance, the application attempts the full query and retries with reduced criteria if the server rejects it. This "try and fall back" approach avoids unnecessary round-trips to probe capabilities, works with servers that do not accurately report their capabilities, and ensures the user always gets the broadest results the server can provide.
- **N-4. Charset fallback with transliteration.** The two-stage charset strategy (UTF-8 first, then ASCII with transliteration) reflects the reality that many IMAP servers still do not support UTF-8 search. Transliteration is lossy but preferable to a hard failure: a search for "Müller" that matches "Muller" is more useful than an error message.
- **N-5. Gmail raw search as an escape hatch.** The `raw:` prefix is not a general-purpose feature but a targeted accommodation for Gmail's rich server-side query language. It is scoped to Gmail accounts and the Archive folder (Gmail's "All Mail") because Gmail's native operators are meaningful only in that context. Exposing it elsewhere would create confusion about what operators are available where.
- **N-6. Quick-filter buttons as search shortcuts.** The search dialog offers one-tap buttons for common flag-based searches (flagged, unread, hidden, with attachments, invitations, notes). These are not separate features but pre-built search configurations that reduce the number of taps for the most frequent search patterns. They populate the same criteria model as the advanced options.
- **N-7. Preference persistence as friction reduction.** The search dialog remembers the user's last-used toggles because search behavior is highly personal and repetitive. A user who always searches senders and subjects but never body text should not have to reconfigure this each time. The three saved recent queries serve the same purpose for the query text itself.
