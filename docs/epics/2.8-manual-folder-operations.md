# Epic: Manual Folder Operations

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **2.8** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Empty trash, empty spam, expunge, mark all read, force re-sync, force send."*
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.

---

## 1. Background & Purpose

Email folders accumulate messages that users want to dispose of in bulk — spam messages pile up, trash fills over time, deleted-but-not-yet-purged messages linger on the server, and entire folders grow stale between sync intervals. In each of these cases the user needs a single, deliberate action to bring a folder to a clean state.

Manual folder operations exist to give users **explicit, on-demand control** over six common housekeeping and recovery tasks that are too consequential to perform automatically without user intent:

1. **Empty Trash** — permanently delete all trashed messages.
2. **Empty Spam** — permanently delete all spam/junk messages.
3. **Expunge** — permanently remove messages already marked for deletion on the server.
4. **Mark All Read** — dismiss unread indicators across an entire folder or view.
5. **Force Re-sync** — discard cached sync state and re-synchronize a folder from the server with maximum priority.
6. **Force Send** — retry delivery of all queued outgoing messages immediately.

These are intentionally manual because each carries a risk of data loss or network cost that the user should consciously accept. They complement the application's automatic housekeeping (auto-expunge, scheduled sync, retry-on-reconnect) by giving the user a way to override timing and force immediate resolution.

## 2. Goals

- **G1.** Provide one-action bulk disposal of all messages in the Trash folder, with a clear warning that the action is permanent and irreversible.
- **G2.** Provide the same for the Spam/Junk folder.
- **G3.** Provide a manual expunge action that permanently removes server-side messages marked for deletion, for users who have disabled automatic expunge.
- **G4.** Provide a one-action "mark all read" that clears unread state across an entire folder or the unified view, respecting any active filters.
- **G5.** Provide a force re-sync that bypasses normal sync throttling and coalescing, for recovery from stale or corrupted local state.
- **G6.** Provide a force send that immediately retries delivery of all queued outgoing messages in the outbox.
- **G7.** Ensure each destructive operation requires explicit user confirmation before execution.

## 3. Non-Goals

- **NG1.** This epic does **not** define automatic or scheduled variants of these operations (e.g. "auto-empty trash after N days"). Those belong to the auto-clean / housekeeping feature.
- **NG2.** This epic does **not** define per-message delete, move-to-trash, or mark-as-spam actions — only the bulk folder-level operations.
- **NG3.** This epic does **not** define the synchronization protocol or connection management — only the user-facing trigger that forces a sync cycle.
- **NG4.** This epic does **not** define the outbox queuing model or retry strategy — only the user-facing trigger that forces immediate send attempts.
- **NG5.** Selective expunge (expunge only certain messages) is out of scope; expunge acts on all messages currently flagged for deletion in the folder.

## 4. Glossary

| Term | Definition |
|---|---|
| **Trash folder** | The folder designated as the account's trash/deleted-items receptacle. Each account has at most one. |
| **Spam folder** | The folder designated as the account's junk/spam receptacle. Each account has at most one. |
| **Expunge** | The act of permanently removing messages that have been flagged for deletion on the mail server. Until expunged, such messages may still be visible to other clients or recoverable. |
| **Marked for deletion** | A server-side flag indicating a message is logically deleted but not yet physically removed. Corresponds to the IMAP `\Deleted` flag. |
| **Auto-expunge** | A setting that causes the application to expunge automatically during normal sync operations, removing the need for manual expunge. |
| **Outbox** | A local pseudo-folder holding composed messages that are queued for delivery but have not yet been successfully transmitted to the server. |
| **Force sync** | A user-initiated sync that clears pending sync operations, ignores throttle windows, and re-synchronizes the folder from scratch. |
| **Purge** | The act of permanently deleting all messages in a folder (marking all as deleted, then expunging). Used for empty-trash and empty-spam. |
| **Sync throttle** | A minimum interval between consecutive user-initiated syncs to prevent excessive server load. |
| **POP3 account** | An account using the POP3 protocol, which has no server-side folder structure and limited remote operations. |

## 5. Personas

- **P1. The tidy user.** Periodically clears out trash and spam manually rather than relying on auto-clean. Wants a single action that empties the folder completely.
- **P2. The storage-conscious user.** Needs to reclaim server-side storage by expunging messages flagged for deletion, especially on providers with tight quotas.
- **P3. The inbox-zero practitioner.** Uses "mark all read" as a deliberate triage reset — declaring bankruptcy on a backlog of unread messages.
- **P4. The troubleshooter.** Suspects the local state is stale or corrupted. Wants to force a full re-sync without restarting the application or removing the account.
- **P5. The impatient sender.** Has messages stuck in the outbox (due to prior connectivity issues or transient errors) and wants to force immediate retry.

## 6. User Stories

### 6.1 Empty Trash

> **US-1.** As a tidy user, I want to permanently delete all messages in the Trash folder of a specific account with a single action, so that I can reclaim storage and keep my mailbox clean.

> **US-2.** As any user, before the trash is emptied, I want a confirmation dialog that clearly states the action is permanent and affects both local and server copies, so that I do not accidentally destroy messages I intended to recover.

> **US-3.** As a multi-account user viewing the unified trash, I want the option to empty trash across all accounts at once, so that I do not have to repeat the action for each account individually.

> **US-4.** As any user, I want the empty-trash action to be available both from within the trash folder view and from the folder's context menu in the folder list, so that I can access it from wherever I happen to be.

### 6.2 Empty Spam

> **US-5.** As a tidy user, I want to permanently delete all messages in the Spam folder of a specific account with a single action, so that I can remove unwanted messages efficiently.

> **US-6.** As any user, before spam is emptied, I want a confirmation dialog that clearly states the action is permanent, so that I can abort if I realize I need to check spam for false positives first.

> **US-7.** As a multi-account user viewing the unified spam folder, I want the option to empty spam across all accounts at once.

> **US-8.** As any user, I want the empty-spam action to be available both from within the spam folder view and from the folder's context menu in the folder list.

### 6.3 Expunge

> **US-9.** As a storage-conscious user who has disabled auto-expunge, I want a manual expunge action that permanently removes all messages currently flagged for deletion in a folder, so that I control exactly when deletion becomes irreversible.

> **US-10.** As any user, before expunge executes, I want a confirmation dialog warning that all messages marked for deletion will be permanently removed, so that I can abort if I have second thoughts.

> **US-11.** As a user with auto-expunge enabled, I do not want to see the manual expunge action in my menus, since it would be redundant — the application handles expunge automatically.

> **US-12.** As a power user, I want the expunge action available from the folder's context menu in the folder list as well as from within the folder view.

### 6.4 Mark All Read

> **US-13.** As an inbox-zero practitioner, I want to mark all messages in the current folder as read with a single action, so that I can reset my unread state without opening each message individually.

> **US-14.** As a user viewing the unified inbox, I want "mark all read" to apply across all folders contributing to the unified view, so that I can clear my combined unread count in one step.

> **US-15.** As any user, the first time I use "mark all read" I want a confirmation dialog, so that I do not accidentally dismiss important unread indicators. I also want the option to suppress this confirmation for future uses.

> **US-16.** As any user with active filters (e.g. showing only flagged messages, or only messages in a specific language), I want "mark all read" to respect those filters and only mark the currently visible subset as read, so that I do not accidentally mark messages I cannot see.

> **US-17.** As any user, I want the mark-all-read action available from both the folder view and the folder's context menu in the folder list.

### 6.5 Force Re-sync

> **US-18.** As a troubleshooter, I want to force a complete re-synchronization of a folder that bypasses normal throttling and coalescing, so that I can recover from stale or inconsistent local state without removing and re-adding the account.

> **US-19.** As any user, I want force re-sync to execute immediately without a confirmation dialog, since it is non-destructive (it only refreshes data from the server).

> **US-20.** As a user viewing the unified inbox, I want force re-sync to re-synchronize all contributing folders, not just one.

> **US-21.** As any user, after triggering a force re-sync, I want a brief visual indication (e.g. a progress indicator or status message) that the operation has been initiated, so that I know my action was received.

### 6.6 Force Send

> **US-22.** As an impatient sender, I want to force immediate retry of all messages queued in the outbox, so that I do not have to wait for the next automatic retry cycle.

> **US-23.** As any user, I want the force-send action to be available only when viewing the outbox, since it has no meaning in other folders.

> **US-24.** As any user, after triggering force send, I want to see that the queued messages are being processed (e.g. status changes from "queued" to "sending"), so that I have confidence the retry is happening.

## 7. Functional Requirements

### 7.1 Empty Trash

- **FR-1.** The application shall provide an "Empty Trash" action that permanently deletes all messages in the trash folder.
- **FR-2.** The action shall be available when viewing the trash folder (in the message list toolbar/menu) and from the trash folder's context menu in the folder list.
- **FR-3.** The action shall be disabled or hidden when one or more messages are individually selected, to prevent confusion between "empty entire folder" and "delete selected messages".
- **FR-4.** Before executing, the application shall display a confirmation dialog that:
  - States that the action will permanently delete all trashed messages.
  - States that deletion applies to both local and server copies.
  - Offers Cancel and Confirm actions.
- **FR-5.** When invoked from a unified (cross-account) trash view, the confirmation shall indicate that all accounts' trash folders will be emptied, and the operation shall apply to every configured account's trash folder.
- **FR-6.** On confirmation, the application shall immediately hide all affected messages from the user interface and queue a server-side purge operation.
- **FR-7.** For accounts using protocols that support server-side deletion, the purge shall mark all messages as deleted on the server and then expunge them.
- **FR-8.** For accounts using protocols without server-side folder operations (e.g. POP3), the purge shall remove messages from local storage only.

### 7.2 Empty Spam

- **FR-9.** The application shall provide an "Empty Spam" action that permanently deletes all messages in the spam/junk folder. Its behavior, confirmation, and availability rules shall be identical to Empty Trash (FR-1 through FR-8) except that it targets the spam folder instead of the trash folder.

### 7.3 Expunge

- **FR-10.** The application shall provide a manual "Expunge" action that permanently removes all messages currently flagged for deletion in the target folder.
- **FR-11.** The expunge action shall be visible only when automatic expunge is disabled (or in a diagnostic/debug mode). When auto-expunge is enabled, the action shall be hidden since it is redundant.
- **FR-12.** The expunge action shall be available from the folder view menu and from the folder's context menu in the folder list.
- **FR-13.** Before executing, the application shall display a confirmation dialog warning that all messages marked for deletion will be permanently removed.
- **FR-14.** On confirmation, the application shall queue a server-side expunge operation.
- **FR-15.** If the server supports per-message expunge (removing only specific messages rather than all deleted messages), the application should prefer that mechanism when enabled by the user, to avoid unintended removal of messages flagged by other clients.
- **FR-16.** The expunge action shall only be available for accounts using protocols that support the expunge concept. It shall not appear for protocol types where it is meaningless.

### 7.4 Mark All Read

- **FR-17.** The application shall provide a "Mark All Read" action that sets the read/seen flag on all unread messages in the current scope.
- **FR-18.** The action shall be available from the message list menu and from the folder's context menu in the folder list.
- **FR-19.** When active filters are applied (e.g. "show only flagged", "show only messages in language X"), the mark-all-read action shall apply only to messages matching the current filter criteria, not to the entire folder contents.
- **FR-20.** When invoked from the unified inbox or a unified view, the action shall apply across all contributing folders.
- **FR-21.** On first use, the application shall display a confirmation dialog. The dialog shall include an option to suppress the confirmation for future uses. Once suppressed, subsequent invocations shall execute immediately.
- **FR-22.** The seen/read flag shall be propagated to the server for each affected message (not local-only).

### 7.5 Force Re-sync

- **FR-23.** The application shall provide a "Force Sync" action that initiates an immediate, high-priority re-synchronization of the target folder(s).
- **FR-24.** A force sync shall differ from a normal sync in that it:
  - Clears any pending sync operations for the folder (avoids conflict with stale queued work).
  - Bypasses the normal minimum interval between user-initiated syncs (the throttle window).
  - Triggers a full folder state refresh from the server.
- **FR-25.** The action shall execute without a confirmation dialog, since it is non-destructive.
- **FR-26.** When invoked from the unified inbox or a unified view, the force sync shall apply to all contributing folders.
- **FR-27.** After invocation, the application shall provide immediate visual feedback (e.g. a progress indicator, status text, or refresh animation) indicating that the sync has been initiated.
- **FR-28.** If an outbox with pending messages exists at the time of a force sync, the application shall also trigger the send subsystem to process queued outgoing messages.

### 7.6 Force Send

- **FR-29.** The application shall provide a "Force Send" action that immediately retries delivery of all messages currently queued in the outbox.
- **FR-30.** The action shall be available only when the user is viewing the outbox folder. It shall not appear in any other folder context.
- **FR-31.** The action shall execute without a confirmation dialog, since it merely retries an already-intended operation.
- **FR-32.** After invocation, the application shall trigger the send subsystem immediately, processing all queued messages regardless of any retry backoff or scheduling that might otherwise delay them.

### 7.7 General

- **FR-33.** All six operations shall provide feedback upon completion or failure (e.g. a brief status notification or toast indicating success, or an error message if the operation could not be completed).
- **FR-34.** All six operations shall be available via the folder list context menu (long-press / right-click on a folder) where applicable, in addition to the in-folder menu.
- **FR-35.** Destructive operations (empty trash, empty spam, expunge) shall not be undoable once confirmed. The confirmation dialog is the sole safeguard.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** After confirmation, the UI shall immediately reflect the operation (e.g. messages hidden from view) without waiting for server-side completion. Server-side work proceeds asynchronously.
- **NFR-2. Atomicity.** Each operation shall either complete fully or fail gracefully with an error report. Partial completion (e.g. half the trash emptied) shall not leave the user in an ambiguous state without notification.
- **NFR-3. Offline behavior.** Destructive operations (empty, expunge) shall be queueable offline: messages are hidden locally immediately, and the server-side portion executes when connectivity is restored. Non-destructive operations (mark all read) shall also queue offline. Force sync shall require connectivity and shall inform the user if the network is unavailable.
- **NFR-4. Scale.** Empty trash/spam and mark-all-read shall remain performant on folders containing thousands of messages. The UI shall not freeze or become unresponsive during processing.
- **NFR-5. Concurrency safety.** If new messages arrive in a folder while an empty or expunge operation is in flight, those new messages shall not be inadvertently deleted. Only messages present at the time the operation was initiated shall be affected.
- **NFR-6. Accessibility.** All actions shall be reachable via keyboard shortcuts and shall have screen-reader-accessible labels. Confirmation dialogs shall be focus-trapped and keyboard-dismissible.
- **NFR-7. Idempotency.** Invoking an operation on an already-empty folder or a folder with no unread messages shall succeed silently (no error) — it is a no-op.

## 9. Acceptance Criteria

| # | Criterion |
|---|---|
| AC-1 | Invoking "Empty Trash" on a trash folder containing messages, then confirming, results in all messages being removed from the local view immediately and from the server within the next sync cycle. |
| AC-2 | Invoking "Empty Trash" from a unified trash view empties the trash folder of every configured account. |
| AC-3 | Cancelling the empty-trash confirmation leaves all messages untouched. |
| AC-4 | Invoking "Empty Spam" behaves identically to AC-1 through AC-3 but targets the spam folder. |
| AC-5 | With auto-expunge disabled, the "Expunge" action appears in the folder menu. With auto-expunge enabled, it does not appear. |
| AC-6 | Invoking "Expunge" on a folder with messages flagged for deletion permanently removes those messages from the server. Messages not flagged for deletion remain unaffected. |
| AC-7 | Invoking "Mark All Read" on a folder with 100 unread messages marks all 100 as read, both locally and on the server. |
| AC-8 | Invoking "Mark All Read" with a filter active (e.g. "show flagged only") marks only the filtered subset as read; messages outside the filter remain unread. |
| AC-9 | After checking "Don't ask again" on the mark-all-read confirmation dialog, subsequent invocations execute immediately without a dialog. |
| AC-10 | Invoking "Force Sync" triggers a re-synchronization that updates the local folder state from the server, even if a normal sync was performed less than 30 seconds ago. |
| AC-11 | Invoking "Force Sync" from the unified inbox triggers re-sync for all unified folders, not just one. |
| AC-12 | Invoking "Force Send" from the outbox causes all queued messages to attempt delivery immediately, regardless of any retry backoff. |
| AC-13 | The "Force Send" action is not visible or accessible from any folder other than the outbox. |
| AC-14 | The "Empty Trash" and "Empty Spam" actions are not visible when one or more messages are individually selected. |
| AC-15 | On a POP3 account, "Empty Trash" removes messages from local storage without attempting server-side operations. |
| AC-16 | All destructive operations (empty, expunge) display a confirmation dialog before executing. Force sync and force send do not require confirmation. |
| AC-17 | Triggering "Empty Trash" while offline hides messages locally; server-side purge executes automatically when connectivity is restored. |

## 10. Open Questions

- **OQ-1. Per-UID expunge availability.** The application supports a "UID expunge" mode that removes only specific messages rather than all deleted messages in a folder. Should this be exposed as a separate user-facing option, or should it remain an advanced/hidden setting? The current implementation makes it a secondary toggle dependent on server capability.
- **OQ-2. Confirmation fatigue for mark-all-read.** The "don't ask again" preference for mark-all-read is global. Should it instead be per-folder or per-view (e.g. always confirm in unified inbox but not in individual folders)?
- **OQ-3. Force sync scope ambiguity.** When force sync is invoked from the unified inbox, should it also re-sync folders that are *not* members of the unified inbox (e.g. sent, drafts), or strictly only the unified-member folders?
- **OQ-4. Empty trash/spam and ongoing operations.** If a message in the trash folder has a pending move operation (e.g. "move back to inbox" that hasn't executed yet), should the empty-trash operation skip that message or override the pending move? The current implementation excludes messages with recent pending operations ("busy" messages) from purge.
- **OQ-5. Auto-empty on exit.** The FAQ explicitly states that the application does not support automatically emptying trash when the user exits, citing risk of accidental data loss. Should this remain a firm non-goal, or should it be revisited as an opt-in preference?
- **OQ-6. Force send feedback granularity.** Should force send provide per-message delivery status feedback (sending, sent, failed), or is a folder-level "processing" indicator sufficient?
- **OQ-7. Expunge in unified context.** Should expunge be available as a cross-account operation (expunge all folders across all accounts), or is it inherently a per-folder operation? The current implementation is per-folder only.

## 11. Design Notes & Rationale

- **N-1. Explicit over automatic for destructive actions.** The application deliberately requires manual invocation for empty-trash and empty-spam rather than auto-emptying on a timer or on exit. This reflects a design philosophy that irreversible data destruction should always be a conscious user choice, never a side effect of routine use. Auto-clean (a separate feature) provides the timed alternative for users who want it.

- **N-2. Auto-expunge as the default.** Automatic expunge is enabled by default because most users expect "delete" to mean "gone". The manual expunge action exists as an escape hatch for users who disable auto-expunge — typically those on shared mailboxes or those who want to review deleted messages from other clients before final removal.

- **N-3. Filter-aware mark-all-read.** The decision to scope mark-all-read to the currently filtered view (rather than always applying to the entire folder) prevents a common mistake: the user filters to "flagged only", sees 5 messages, marks all read, and unintentionally marks 500 hidden messages as read too. The filter-respecting behavior makes the action predictable relative to what the user can see.

- **N-4. Force sync is non-destructive.** Force sync never deletes local data — it re-fetches from the server and reconciles. This is why it requires no confirmation. It is the safest recovery action a user can take when they suspect inconsistency.

- **N-5. Force send in outbox only.** Restricting force send to the outbox view is both a discoverability choice (the user goes to where stuck messages live) and a safety choice (preventing accidental send triggers from unrelated contexts).

- **N-6. Hiding before purging.** For empty-trash and empty-spam, messages are hidden from the UI immediately upon confirmation, before the server operation completes. This gives the user instant visual feedback and avoids a confusing "I confirmed but they're still there" state. The server-side purge happens asynchronously. If the server operation fails, error handling must surface this to the user.

- **N-7. Throttle bypass for force sync.** Normal user-initiated syncs are throttled (minimum ~30 seconds apart) to prevent excessive server load from repeated pull-to-refresh. Force sync bypasses this throttle because the user is explicitly signaling that they need fresh data *now*, likely because they believe the local state is wrong.
