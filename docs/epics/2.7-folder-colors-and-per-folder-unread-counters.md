# Epic: Folder Colors and Per-Folder Unread Counters

> Reverse-engineered from the FairEmail Android application. Maps to high-level feature **2.7** in [/reverse-engineering/high-level-features.md](../../reverse-engineering/high-level-features.md): *"Folder colors and per-folder unread counters."*
>
> **MoSCoW priority for v1 (Linux desktop):** `[SHOULD]` — important, planned for v1 but acceptable to slip to v1.x if pressed.
>
> This document captures **what** the feature must do, not **how**. It describes user-visible behavior and contract-level expectations, and is intended to drive design and implementation work in subsequent stages without prescribing them.
>
> **Relationship to other epics:** The mechanics of assigning a color to a folder (color picker, color persistence, color precedence with account and identity colors, color display modes) are already defined in the [Account / Identity / Folder Coloring & Avatars](1.8-account-identity-folder-coloring-avatars.md) epic. This epic focuses on (a) the **folder-list-specific** rendering of folder colors — the color stripe on each row in the folder list and the color-tinted icon in the navigation pane — and (b) the **per-folder unread counter** system: how unread counts are calculated, displayed, aggregated for parent folders, included in account-level badges, and controlled by the user.

---

## 1. Background & Purpose

A user managing many folders across multiple accounts needs two fast, glanceable cues when scanning the folder list or navigation pane:

1. **Which folder am I looking at?** — Color provides a pre-attentive visual marker that distinguishes folders faster than reading their names, especially when the same folder name (e.g. "Inbox", "Sent") appears under several accounts.
2. **Where is my unread mail?** — A numeric count appended to the folder name, combined with bold/highlighted styling, tells the user immediately which folders need attention and roughly how much attention each one needs.

Together, these two cues transform a flat text list into a prioritized, color-coded triage surface. The user can assign colors to the folders they care most about, and the application ensures unread counts are always visible, always accurate, and always responsive to user actions (marking as read, new mail arriving, collapsing parent folders).

The per-folder unread counter system also feeds into higher-level indicators: account-level badge counts in the navigation pane, application-level launcher badges, and the auto-hide behavior for folders that have no unread mail.

## 2. Goals

- **G1.** Display a color indicator on each folder row in the folder list, using the folder's assigned color (or the account's color as a fallback for the Inbox), so that users can visually identify folders at a glance.
- **G2.** Display each folder's unread message count inline in the folder list, with clear visual emphasis (bold text, highlighted color) when the count is nonzero.
- **G3.** Aggregate child-folder unread counts into the parent when a parent folder is collapsed, so that collapsing a subtree does not hide the presence of unread mail.
- **G4.** Allow the user to control whether a folder's unread count contributes to the account-level total shown in the navigation pane badge.
- **G5.** Allow the user to hide a folder from the folder list automatically when all its messages have been read, reducing visual noise.
- **G6.** Provide application-level display settings that govern how unread counts appear in the navigation pane: whether to show counts at all, whether to show them for pinned folders, whether to distinguish "newly arrived unread" from "old unread", and whether to show total message counts alongside or instead of unread counts.
- **G7.** Provide a folder-list sort option that floats folders with unread messages to the top, so the user's attention is drawn to where it is needed.

## 3. Non-Goals

- **NG1.** This epic does **not** define the mechanics of assigning, clearing, or persisting colors on folders — that is covered in the Account / Identity / Folder Coloring & Avatars epic (feature 1.8). This epic addresses only the folder-list and navigation-pane rendering of those colors.
- **NG2.** This epic does **not** define the color precedence cascade (identity → folder → account) used in the **message** list. That cascade is defined in the coloring epic. In the **folder** list, the simpler rule (folder color, or account color for the Inbox) applies.
- **NG3.** This epic does **not** define notification badge behavior or desktop launcher badge behavior in full; it defines the per-folder "count unread" flag that controls whether a folder's unread messages contribute to badge totals, but the badge display itself is part of the notifications feature (8.x).
- **NG4.** This epic does **not** define the message-list highlighting of unread messages (bold subject, highlight color). It covers only folder-level counters and styling.
- **NG5.** Widget unread-count display is out of scope; widgets are a separate concern that consumes the same underlying count data.

## 4. Glossary

| Term | Definition |
|---|---|
| **Folder** | A container of messages within an account. May be a system folder (Inbox, Sent, Drafts, Trash, Spam, Archive, Outbox) or a user-created folder. |
| **Folder color** | A user-assigned color on a folder, displayed as a vertical stripe in the folder list and as a tint on the folder icon in the navigation pane. Nullable — absence means "no color assigned". |
| **Account color** | A user-assigned color on an account. Used as a fallback color for the Inbox folder when the Inbox has no folder color of its own. |
| **Unread count** | The number of messages in a folder that have not been marked as read and are not hidden. Synonymous with "unseen count". |
| **Unexposed count** | The subset of unread messages that arrived after the user last viewed the folder. Distinguishes "newly arrived unread" from "older unread". |
| **Child-folder aggregation** | When a parent folder is collapsed in the folder list, the unread counts of all its descendant folders are summed and added to the parent's displayed count. |
| **Count-unread flag** | A per-folder boolean setting that controls whether the folder's unread messages are included in the account-level badge total displayed in the navigation pane. |
| **Hide-when-read** | A per-folder boolean setting that causes the folder to be hidden from the folder list when its unread count (including collapsed children) is zero. |
| **Navigation folder** | A folder that the user has chosen to display as a standalone entry in the navigation pane, outside the full folder tree. |
| **Sort unread atop** | A folder-list display option that reorders folders so those with unread messages appear before those without. |

## 5. Personas

- **P1. The inbox-zero practitioner.** Processes mail aggressively and wants the folder list to clearly show where unread mail remains. Relies on bold folder names and numeric counts to spot unprocessed folders instantly.
- **P2. The folder organizer.** Has a deep, hierarchical folder tree. Assigns colors to important folders and collapses branches they rarely use. Needs child-count aggregation so collapsing a branch does not mask unread mail.
- **P3. The multi-account user.** Has several accounts and navigates primarily via the navigation pane. Wants the navigation pane badge on each account entry to reflect the true unread total — but only for folders that matter (not Spam, not Trash).
- **P4. The minimal-noise user.** Wants folders with no unread mail to disappear from the list entirely, leaving only folders that need attention visible.
- **P5. The power user.** Wants to see total message counts alongside unread counts, wants to distinguish "newly arrived" from "old unread", and wants folders with unread mail sorted to the top of the list.

## 6. User Stories

### 6.1 Folder color display in the folder list

> **US-1.** As any user, when I view the folder list, I want each folder row to show a vertical color stripe on its leading edge, filled with the folder's assigned color, so that I can distinguish folders by color at a glance.

> **US-2.** As a multi-account user, when a folder is the Inbox and has no explicit folder color assigned, I want the color stripe to use the account's color as a fallback, so that the Inbox row always carries the account's visual identity.

> **US-3.** As any user, when a folder has no color assigned and is not the Inbox, I want the color stripe to be absent (transparent), so that uncolored folders do not display a misleading default color.

> **US-4.** As any user, I want the folder color stripe to be visible only when I have the pro/paid tier of the application, so that color features are gated behind the appropriate license.

### 6.2 Folder color display in the navigation pane

> **US-5.** As any user, when a folder is shown in the navigation pane (either as part of an account's entry or as a standalone navigation folder), I want the folder's icon to be tinted with its color, so that my color scheme carries into the navigation pane.

> **US-6.** As a multi-account user, when the navigation pane shows an account entry (with no specific folder selected), I want the icon to use the account's color; when the entry represents a specific folder, I want the icon to use the folder's color.

### 6.3 Per-folder unread count display

> **US-7.** As any user, when a folder has one or more unread messages, I want the unread count appended to the folder's display name in the folder list (e.g. "Inbox (3)"), so that I can see the count without opening the folder.

> **US-8.** As any user, when a folder has unread messages, I want the folder name to be rendered in bold and in a distinct highlight color, so that folders with unread mail stand out visually from read-only folders.

> **US-9.** As any user, when a folder has zero unread messages, I want the folder name to be rendered in the default (non-bold) style, so that the absence of emphasis communicates "nothing to do here".

> **US-10.** As a power user, I want the option to display a "newly arrived" indicator (a star or similar mark) next to folders that have unread messages which arrived after I last viewed the folder, so that I can distinguish fresh mail from older unread mail.

### 6.4 Total message count display

> **US-11.** As any user, I want to see the total number of messages in each folder alongside or instead of the unread count (e.g. "downloaded / total"), so that I know the size of each folder.

> **US-12.** As any user, when in a compact display mode, I want the total count to be suppressed (showing only the total without the downloaded breakdown), so that the folder list remains clean.

### 6.5 Child-folder aggregation on collapse

> **US-13.** As a folder organizer, when I collapse a parent folder that has child folders, I want the parent's displayed unread count to include the sum of unread messages from all its descendants, so that collapsing a subtree does not hide the presence of unread mail.

> **US-14.** As a folder organizer, when a collapsed parent is showing an aggregated count that includes child-folder unseen mail, I want a visual indicator (e.g. a small marker before the count) that signals the count includes children, not just the parent folder itself.

> **US-15.** As a folder organizer, when I expand a parent folder, I want the parent's displayed count to revert to its own unread count only (no longer including children), since the children are now independently visible.

### 6.6 Count-unread contribution to account badge

> **US-16.** As any user, I want a per-folder setting ("add unread count to account total") that controls whether the folder's unread messages are included in the account-level badge/total displayed in the navigation pane.

> **US-17.** As a multi-account user, I want the count-unread flag to default to enabled for all folders, so that the account badge reflects the full unread picture out of the box.

> **US-18.** As a multi-account user, I want to disable the count-unread flag for specific folders (e.g. Spam, Trash, a noisy mailing-list folder), so that their unread messages do not inflate the account's badge.

### 6.7 Hide folder when all read

> **US-19.** As a minimal-noise user, I want a per-folder setting ("hide folder when all messages are read") that removes the folder from the folder list when its unread count (including collapsed children) drops to zero.

> **US-20.** As a minimal-noise user, I want hidden-when-read folders to reappear automatically when new unread mail arrives, without any manual intervention.

> **US-21.** As any user, I want the hide-when-read behavior to apply only in the account folder list and the unified folder list, not in other contexts (e.g. folder selection dialogs, move-to menus).

### 6.8 Navigation pane count settings

> **US-22.** As any user, I want a global display setting to show the unread count on each entry in the navigation pane, so that I can see at a glance how many unread messages each account or navigation folder has.

> **US-23.** As any user, I want a display setting to show the number of locally stored messages (total count) in the navigation pane instead of or alongside the unread count.

> **US-24.** As any user, I want a display setting to show the unread count on pinned navigation entries even when the navigation pane is collapsed, so that I can see counts without expanding the pane.

> **US-25.** As any user, when a navigation pane entry is collapsed (showing only the icon), I want a small badge overlay on the icon showing the unread count (capped at a reasonable display maximum such as "99+"), so that the count is visible even in the compact view.

> **US-26.** As any user, I want a display setting that controls whether the drafts folder shows its total message count (all drafts) rather than its unread count, since all drafts are by nature "unread" and the total is more useful.

### 6.9 Sort folders by unread

> **US-27.** As a power user, I want a folder-list sort option that floats folders with unread messages to the top of the list (within each level of the hierarchy), so that I do not have to scroll through empty folders to find one with unread mail.

> **US-28.** As a power user, I want the "sort unread atop" option to apply hierarchically — at each level of nesting, folders with unread mail sort above those without — so that the tree structure is preserved while still surfacing unread folders.

## 7. Functional Requirements

### 7.1 Folder color rendering in the folder list

- **FR-1.** Each folder row in the folder list shall display a vertical color stripe at the leading edge of the row, filled with the folder's assigned color.
- **FR-2.** The color stripe width shall respect the application-wide "color stripe width" setting (defined in the coloring epic).
- **FR-3.** For folders of type Inbox that have no explicit folder color assigned, the color stripe shall use the owning account's color as a fallback.
- **FR-4.** For all other folder types with no color assigned, the color stripe shall be transparent (not rendered).
- **FR-5.** The color stripe shall be visible only when the user has the pro/paid license. When the license is absent, the stripe area shall be hidden.

### 7.2 Folder color rendering in the navigation pane

- **FR-6.** In the navigation pane, each folder or account entry shall tint its icon with the resolved color: the folder's own color if the entry represents a specific folder, or the account's color if the entry represents the account at large.
- **FR-7.** When the entry has no resolved color, the icon shall use its default untinted appearance.

### 7.3 Default folder colors

- **FR-8.** Trash and Spam/Junk folders shall display a warning-level color (the application theme's error color) by default, without user intervention.
- **FR-9.** The user may override a Trash or Spam/Junk folder's default color with a custom color, or clear it back to the default.
- **FR-10.** All other folder types shall have no default color.

### 7.4 Per-folder unread count display

- **FR-11.** When a folder has one or more unread (unseen, non-hidden) messages, the folder's display name in the folder list shall be suffixed with the unread count in a parenthetical or equivalent format.
- **FR-12.** When a folder has unread messages, the folder name shall be rendered in bold typeface and in a configurable "unread highlight" color.
- **FR-13.** When a folder has zero unread messages, the folder name shall be rendered in the default (non-bold) typeface and in the secondary text color.
- **FR-14.** The "unread highlight" color shall be configurable via the display settings; a default shall be provided.

### 7.5 Newly-arrived ("unexposed") indicator

- **FR-15.** The application shall track, per folder, the timestamp of the user's last view of that folder.
- **FR-16.** The application shall maintain a count of unread messages that arrived after the last-view timestamp ("unexposed" count).
- **FR-17.** When the "show unexposed" display setting is enabled, folders with unexposed messages shall display an additional indicator (e.g. a star symbol) next to the unread count.
- **FR-18.** The "show unexposed" setting shall default to off.

### 7.6 Total message count display

- **FR-19.** Each folder row shall display the total message count alongside the unread count.
- **FR-20.** The total count display shall show "downloaded / total" in normal mode, or just "total" in compact mode.

### 7.7 Child-folder unread aggregation

- **FR-21.** When a parent folder is collapsed in the folder list, the displayed unread count for the parent shall equal the parent's own unread count plus the sum of unread counts from all its descendant folders.
- **FR-22.** When a collapsed parent's displayed count includes child-folder unseen messages, a visual marker (e.g. a down-pointing triangle) shall be prepended to the count to signal aggregation.
- **FR-23.** When the parent is expanded, its displayed count shall revert to its own unread count only.

### 7.8 Count-unread contribution flag

- **FR-24.** Each folder shall have a persistent, user-editable boolean property ("count unread") that controls whether the folder's unread messages are included in the account-level total displayed in the navigation pane.
- **FR-25.** The count-unread flag shall default to **enabled** for all folders.
- **FR-26.** The count-unread flag shall be editable from the folder settings screen.
- **FR-27.** Changing the count-unread flag shall take effect immediately and shall update the account's navigation-pane badge without requiring restart.
- **FR-28.** The count-unread flag shall be independent of the folder's notification setting, synchronization setting, and unified-inbox membership.

### 7.9 Hide folder when all read

- **FR-29.** Each folder shall have a persistent, user-editable boolean property ("hide when read") that causes the folder to be removed from the folder list when its unread count (including collapsed children) drops to zero.
- **FR-30.** The hide-when-read flag shall default to **disabled** for all folders.
- **FR-31.** When a hidden-when-read folder receives new unread mail, it shall reappear in the folder list automatically on the next refresh.
- **FR-32.** The hide-when-read behavior shall apply only in the account folder list view and the unified folder list view. It shall **not** apply in folder-selection dialogs, move-to menus, or other contexts where the full folder tree must be visible.

### 7.10 Navigation pane count display

- **FR-33.** The application shall provide a display setting ("show unread count in navigation menu") that controls whether each account and navigation-folder entry in the navigation pane shows its unread count. Default: off.
- **FR-34.** The application shall provide a display setting ("show message count in navigation menu") that shows the total number of locally stored messages instead of (or in addition to) the unread count. Default: off.
- **FR-35.** The application shall provide a display setting ("show count on pinned navigation") that shows the unread count as a small overlay badge on pinned/collapsed navigation entries. Default: off.
- **FR-36.** The application shall provide a display setting ("show drafts total") that causes the Drafts folder in the navigation pane to display its total message count rather than its unread count. Default: off.
- **FR-37.** When a navigation entry is collapsed (icon only), a badge overlay shall display the unread count, formatted with a display cap (e.g. "99+"). The badge shall be hidden when the count is zero or when the navigation entry is expanded.
- **FR-38.** Navigation pane entries with unread messages shall render their text in bold and in the unread highlight color, matching the folder-list convention.

### 7.11 Sort unread atop

- **FR-39.** The application shall provide a folder-list display option ("sort unread atop") that reorders folders so those with unread messages appear above those without, at each level of the folder hierarchy.
- **FR-40.** The sort shall be applied after the standard folder ordering (by type, then by name), moving only the unread/no-unread boundary.
- **FR-41.** The "sort unread atop" option shall default to off.

## 8. Non-Functional Requirements

- **NFR-1. Responsiveness.** Unread counts shall update in the folder list within one second of a message being marked as read or unread, or of new mail arriving. The user shall never see a stale count persist across a view refresh.
- **NFR-2. Accuracy.** Unread counts shall be derived from the authoritative message store and shall never diverge from the true count of unseen, non-hidden messages. Rounding, estimation, or sampling are not acceptable.
- **NFR-3. Efficiency.** Unread count calculation shall not cause perceptible lag in the folder list, even with 200+ folders and tens of thousands of messages. Counts shall be computed reactively (triggered by changes) rather than by polling.
- **NFR-4. Deduplication.** When the same message appears in multiple folders (e.g. via aliasing or cross-posting), the navigation-pane badge shall deduplicate by message identity (hash or message ID) to avoid inflating the account total.
- **NFR-5. Offline.** Unread counts shall be fully functional offline, reflecting the state of locally stored messages. Count updates from user actions (mark as read/unread) shall apply immediately, even without a server connection.
- **NFR-6. Persistence.** All per-folder settings (count-unread flag, hide-when-read flag, color) shall survive application restart, account re-authentication, and server folder re-synchronization.
- **NFR-7. Accessibility.** Folder names, unread counts, and "newly arrived" indicators shall be readable by screen readers. Color shall not be the sole means of conveying unread status — bold text and numeric counts provide redundant cues.
- **NFR-8. Theme compliance.** The unread highlight color, default Trash/Junk color, and navigation-pane badge styling shall adapt to the active theme (light, dark, black, sepia).

## 10. Open Questions

- **OQ-1. Scope boundary with the coloring epic.** The coloring epic (1.8) defines folder color assignment, persistence, and the full color precedence cascade. This epic defines the folder-list and navigation-pane rendering of those colors. The boundary is clear in intent but some requirements (e.g. FR-1 through FR-5 here vs. FR-4, FR-14 in the coloring epic) overlap. During implementation, these should be reconciled into a single source of truth to avoid contradictory specifications.
- **OQ-2. Default Inbox color fallback.** The source application falls back to the account color only for the Inbox folder type when no folder color is set. Whether this fallback should also apply to other "primary" folder types (e.g. Drafts, Sent) is a design decision deferred to implementation. The current behavior (Inbox only) is documented as-is.
- **OQ-3. Hide-when-read and unified folder list.** The hide-when-read behavior is documented as applying to the account folder list and the unified folder list. Whether "unified folder list" refers to the unified inbox's folder grouping or to a separate unified view of all folders across accounts should be confirmed.
- **OQ-4. Badge display cap.** The navigation-pane badge formats counts with a cap (e.g. "99+"). The exact cap value is a display detail, but whether it should be user-configurable or fixed is an open question.
- **OQ-5. Count-unread flag interaction with Outbox.** The source application explicitly excludes the Outbox folder from the account-level unread total regardless of the count-unread flag. Whether this is a hard rule or a default that the user should be able to override is unclear.
- **OQ-6. Sort-unread-atop and conversation threading.** When "sort unread atop" is enabled and a folder uses conversation threading, it is not clear whether a thread with a mix of read and unread messages counts as "unread" for sorting purposes. The source application sorts at the folder level (not the thread level), so this is a folder-list-only concern, but the interaction should be confirmed.

## 11. Design Notes & Rationale

These notes record decisions implicit in the source application that future designers should understand before changing them.

- **N-1. Unread count, not notification count.** The folder-list unread count reflects all unseen, non-hidden messages — not just those that have triggered a notification. A message can be unread but already acknowledged (e.g. the user saw it in the unified inbox but did not open it in the source folder). The count is a triage cue, not a notification indicator.
- **N-2. Child aggregation uses a visual marker.** When a collapsed parent's count includes children, a small triangle glyph is prepended to the count. This is a subtle but important disambiguation: without it, a user might think the parent folder itself has that many unread messages, when in fact the count is spread across descendants.
- **N-3. Hide-when-read is opt-in, not the default.** Most users expect their folder tree to be stable. Folders appearing and disappearing based on unread state can be disorienting. The feature exists for power users who want a minimal, attention-driven folder list, but it is disabled by default to avoid confusion.
- **N-4. Count-unread vs. notification independence.** The "count unread" flag (whether a folder's unread messages count toward the account badge) is deliberately independent of the folder's notification setting. A folder can notify but not count (e.g. a noisy mailing list that the user wants alerts for but does not want inflating their badge), or count but not notify (e.g. a low-priority folder the user wants reflected in the badge but does not want audible alerts for). All four combinations are meaningful.
- **N-5. Drafts count as messages, not unread.** Draft messages are inherently "unread" (the user is the author, not a reader), so showing an unread count for Drafts is semantically misleading. The source application provides a setting to show the total draft count instead, which better reflects the user's intent ("how many drafts do I have in progress?").
- **N-6. Deduplication in navigation badge.** The navigation-pane badge query deduplicates messages by a content hash or message identifier. This prevents the same message from being counted multiple times when it exists in multiple folders (e.g. a message in both Inbox and a label/tag folder). The folder-list counts themselves are per-folder and do not deduplicate, because in that context each folder's count should reflect its own contents faithfully.
- **N-7. Pro-gating of color display.** The folder color stripe is visible only to users with the pro/paid tier. Unread counts and all count-related settings are available to all users regardless of tier. This reflects the source application's monetization boundary: color is a premium visual enhancement, while unread counts are core functionality.
